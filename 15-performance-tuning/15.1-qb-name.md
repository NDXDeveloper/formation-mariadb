# 15.15.1 Nommage des blocs de requête (QB_NAME) et noms implicites 🆕

Pour cibler un hint sur une partie précise d'une requête — une sous-requête particulière, une table dérivée, une branche d'un `UNION` —, encore faut-il pouvoir **désigner** le bloc concerné. C'est le rôle du nommage des blocs de requête. Tout bloc reçoit un nom implicite, et l'on peut lui attribuer un nom explicite au moyen du hint `QB_NAME`. Ce mécanisme d'adressage est le préalable au ciblage fin décrit au §15.15 : sans lui, un hint ne saurait s'appliquer qu'au bloc dans lequel il est écrit.

## Qu'est-ce qu'un bloc de requête ?

Un **bloc de requête** est un `SELECT`, ou la construction de plus haut niveau d'une instruction `UPDATE` ou `DELETE`. Une même instruction peut en contenir plusieurs : le `SELECT` principal, ses sous-requêtes, ses tables dérivées, les différentes branches d'un `UNION`. Dès lors qu'on veut appliquer un hint à l'un de ces blocs en particulier — et à la bonne table à l'intérieur de celui-ci —, il devient nécessaire de le nommer et de le référencer.

## Les noms implicites : select#n

Tout bloc de requête se voit automatiquement attribuer un **nom implicite** de la forme `select#n` : `select#1` pour le bloc principal, puis `select#2`, `select#3`, etc., pour les blocs imbriqués ou frères. Ces noms sont visibles en exécutant `EXPLAIN EXTENDED` (suivi de `SHOW WARNINGS`), qui restitue la requête réécrite par l'optimiseur en annotant chaque bloc de son nom implicite.

On peut référencer un bloc par son nom implicite dans un hint, exactement comme on le ferait avec un nom explicite. Dans l'exemple suivant, le hint cible deux tables situées dans deux blocs différents, désignés par leurs noms implicites :

```sql
SELECT /*+ NO_BKA(t2@select#2, t3@select#1) */ SUM(t3.a)  
FROM (SELECT t1.b  
      FROM t1 JOIN t2 ON t1.a = t2.a) AS dt
JOIN t3 ON t3.a = dt.b;
```

Ici, la table dérivée `dt` correspond au bloc `select#2` et contient `t2`, tandis que `t3` appartient au bloc externe `select#1`. La notation `table@bloc` permet d'adresser chaque table dans son bloc respectif.

## Nommer explicitement un bloc : QB_NAME

Le hint `QB_NAME(nom)` attribue un **nom explicite** au bloc de requête dans lequel il est écrit. Il se place donc dans le commentaire de hints propre à ce bloc, immédiatement après son `SELECT` :

```sql
SELECT /*+ QB_NAME(bloc_principal) */ liste_de_colonnes  
FROM ... ;  
```

La portée du nom est l'**ensemble de l'instruction** : une fois défini, il peut être référencé depuis d'autres endroits de la requête. Il est en revanche **interdit de réutiliser un même nom** pour deux blocs distincts ; une telle duplication est signalée par un avertissement, le second `QB_NAME` étant ignoré. Les noms de blocs sont des identifiants ordinaires : un nom comportant des espaces doit être encadré de accents graves, aussi bien à la définition qu'à la référence.

```sql
SELECT /*+ BKA(@`mon bloc`) */ ...  
FROM (SELECT /*+ QB_NAME(`mon bloc`) */ ...) AS dt ... ;  
```

À noter qu'un bloc nommé explicitement conserve par ailleurs son nom implicite `select#n` : les deux désignations restent utilisables pour le référencer.

## Référencer un bloc : la notation @nom

Une fois un bloc nommé — explicitement ou implicitement —, les autres hints le désignent au moyen d'un nom préfixé de `@`. Deux situations se présentent.

Pour un hint **de niveau bloc**, le nom du bloc suffit : `BKA(@bloc)` applique le hint à l'ensemble du bloc désigné. Pour un hint **de niveau table**, deux syntaxes équivalentes coexistent : le nom du bloc peut précéder la table, sous la forme `hint(@bloc table …)`, ou bien être accolé à la table, sous la forme `hint(table@bloc …)`. Si aucun `@bloc` n'est précisé, le hint s'applique au bloc dans lequel il est écrit.

L'exemple suivant, qui imbrique trois blocs nommés, illustre l'adressage croisé :

```sql
SELECT /*+ QB_NAME(qb1)
           MRR(@qb1 t1)
           BKA(@qb2)
           NO_MRR(@qb3 t1 idx1, idx2) */ ...
FROM (SELECT /*+ QB_NAME(qb2) */ ...
      FROM (SELECT /*+ QB_NAME(qb3) */ ... FROM ...) ...) ... ;
```

Ici, `MRR(@qb1 t1)` s'applique à la table `t1` du bloc `qb1` ; `BKA(@qb2)` s'applique à l'ensemble du bloc `qb2` ; et `NO_MRR(@qb3 t1 idx1, idx2)` vise les index `idx1` et `idx2` de la table `t1` dans le bloc `qb3`. On notera que tous ces hints sont rassemblés dans le commentaire du bloc principal, mais chacun est dirigé vers le bloc voulu grâce à son nom.

## Les règles de portée des références

MariaDB encadre précisément la **direction** dans laquelle une référence à un bloc peut s'effectuer. Depuis un bloc donné, on peut référencer un bloc :

- vers le bas, dans une sous-requête ;
- vers le bas, dans une table dérivée ;
- vers le haut, vers le bloc parent ;
- vers un frère situé à **droite** dans un `UNION`.

En revanche, on **ne peut pas** référencer un frère situé à **gauche** dans un `UNION`. Cette asymétrie est à garder à l'esprit lorsqu'on conçoit des hints portant sur les branches d'une union : un hint placé dans une branche ne peut désigner que les branches qui la suivent, jamais celles qui la précèdent.

## Limitation : les vues

Le nommage des blocs de requête ne fonctionne pas avec les vues. Les hints ne sont pas pris en charge à l'intérieur des vues : on ne peut ni les employer dans la définition d'une vue, ni piloter le plan d'exécution à l'intérieur d'une vue non fusionnée. Cette limitation tient à la mécanique interne — la liaison des noms de blocs (`QB_NAME`) est réalisée à un stade précoce de l'analyse, avant que le serveur ne sache que certaines tables sont en réalité des vues.

## En conclusion

Le nommage des blocs de requête constitue le mécanisme d'**adressage** sur lequel repose l'ensemble des hints ciblés : noms implicites `select#n` pour une désignation immédiate, `QB_NAME` pour des noms parlants et stables, et notation `@nom` pour diriger un hint vers le bloc — et la table — voulus, dans le respect des règles de portée. Muni de cette capacité de désignation, on peut désormais appliquer des hints concrets à des tables, des index, des jointures et des sous-requêtes précis : c'est l'objet des sous-sections suivantes, à commencer par les hints de table et d'index (§15.15.2).
