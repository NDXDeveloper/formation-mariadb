# 15.9.2 Partitionnement par liste (LIST)

Là où le partitionnement par intervalles (`RANGE`) s'appuie sur l'ordre naturel des valeurs, le partitionnement par liste (`LIST`) s'appuie sur leur **appartenance à un ensemble discret et énumérable**. Chaque partition est associée à une liste explicite de valeurs, et une ligne y est rangée si la valeur de sa colonne de partitionnement figure dans cette liste. C'est la méthode de prédilection lorsque les données se répartissent en catégories fermées sans relation d'ordre : régions, statuts, codes pays, types d'entité, identifiants de tenant.

## Principe et syntaxe de base

Un partitionnement `LIST` se déclare au moyen de partitions introduites par `VALUES IN`, chacune énumérant les valeurs qu'elle accueille. Comme pour `RANGE`, la forme classique exige une **expression renvoyant un entier**.

```sql
CREATE TABLE ventes_regionales (
    id        BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    region_id TINYINT NOT NULL,
    montant   DECIMAL(10,2),
    PRIMARY KEY (id, region_id)
) ENGINE = InnoDB
PARTITION BY LIST (region_id) (
    PARTITION p_nord  VALUES IN (1, 2, 3),
    PARTITION p_sud   VALUES IN (4, 5, 6),
    PARTITION p_est   VALUES IN (7, 8),
    PARTITION p_ouest VALUES IN (9, 10)
);
```

Une ligne dont `region_id` vaut `5` est dirigée vers `p_sud`, une ligne valant `7` vers `p_est`, et ainsi de suite. On retrouve la règle structurante du partitionnement : la colonne `region_id` faisant partie de l'expression, elle doit appartenir à la clé primaire, d'où la clé composite `(id, region_id)`.

Une contrainte propre à `LIST` mérite d'être soulignée : les listes doivent être **mutuellement exclusives**, une même valeur ne pouvant figurer dans deux partitions. Une déclaration qui violerait cette règle est rejetée à la création de la table.

```sql
-- ÉCHEC : la valeur 5 figure dans deux partitions
PARTITION BY LIST (region_id) (
    PARTITION p_a VALUES IN (1, 2, 5),
    PARTITION p_b VALUES IN (3, 4, 5)
);
```

Contrairement au partitionnement par intervalles, `LIST` ne couvre que les valeurs explicitement énumérées : il n'existe pas de notion d'intervalle « jusqu'à », et donc rien n'est implicitement couvert entre les valeurs listées.

## Valeurs hors liste et partition DEFAULT

Que se passe-t-il pour une valeur qui n'apparaît dans aucune liste ? Comme pour un `RANGE` sans `MAXVALUE`, l'insertion échoue :

```sql
INSERT INTO ventes_regionales (region_id, montant) VALUES (11, 100.00);
-- ERROR 1526 (HY000): Table has no partition for value 11
```

Le partitionnement par liste ne dispose pas de la borne `MAXVALUE`, qui est propre au `RANGE` et n'aurait aucun sens sur un ensemble non ordonné. Son équivalent est la **partition `DEFAULT`**, introduite dans MariaDB 10.2, qui recueille toutes les valeurs ne correspondant à aucune autre liste. Une seule partition `DEFAULT` est autorisée par table.

```sql
PARTITION BY LIST (region_id) (
    PARTITION p_nord   VALUES IN (1, 2, 3),
    PARTITION p_sud    VALUES IN (4, 5, 6),
    PARTITION p_est    VALUES IN (7, 8),
    PARTITION p_ouest  VALUES IN (9, 10),
    PARTITION p_autres DEFAULT
);
```

Avec ce schéma, une insertion avec `region_id = 11` est désormais acceptée et stockée dans `p_autres`. La partition `DEFAULT` joue ainsi pour `LIST` le rôle de filet de sécurité que `MAXVALUE` joue pour `RANGE` : elle protège des erreurs d'insertion, mais introduit la même contrainte de maintenance, détaillée plus bas, puisqu'on ne peut pas ajouter une partition explicite tant que le fourre-tout doit demeurer.

## LIST COLUMNS : types non entiers et colonnes multiples

À l'image de `RANGE COLUMNS`, la variante `LIST COLUMNS` lève la contrainte de l'expression entière. Elle prend en argument une ou plusieurs **colonnes nues** (aucune expression n'est permise) et accepte les types entiers, les types chaîne (`CHAR`, `VARCHAR`…), ainsi que `DATE` et `DATETIME` — `TIMESTAMP` et les types à virgule en étant, là encore, exclus.

C'est la forme naturelle pour partitionner sur des catégories textuelles, par exemple un code pays :

```sql
CREATE TABLE clients (
    id   BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    pays CHAR(2) NOT NULL,
    nom  VARCHAR(100),
    PRIMARY KEY (id, pays)
) ENGINE = InnoDB
PARTITION BY LIST COLUMNS (pays) (
    PARTITION p_europe    VALUES IN ('FR', 'DE', 'ES', 'IT'),
    PARTITION p_ameriques VALUES IN ('US', 'CA', 'BR'),
    PARTITION p_asie      VALUES IN ('JP', 'CN', 'IN'),
    PARTITION p_autres    DEFAULT
);
```

La logique d'affectation repose ici sur l'**égalité** : une ligne est placée dans la partition dont la liste contient exactement sa valeur (ou, en cas de colonnes multiples, dont la liste contient exactement le tuple de ses valeurs). C'est une différence conceptuelle avec `RANGE COLUMNS`, qui raisonne en termes de comparaison « inférieur à » sur des tuples ordonnés lexicographiquement, tandis que `LIST COLUMNS` recherche une correspondance stricte.

## Traitement des valeurs NULL

Le comportement de `LIST` face aux valeurs `NULL` diffère nettement de celui de `RANGE`, et constitue un point d'attention. En `RANGE`, on l'a vu au §15.9.1, `NULL` est traité comme inférieur à toute valeur et atterrit dans la partition la plus basse. En `LIST`, `NULL` est au contraire considéré comme une **valeur à part entière qui doit être explicitement listée** pour être acceptée.

Concrètement, pour stocker des lignes dont la colonne de partitionnement vaut `NULL`, il faut inclure `NULL` dans la liste d'une partition :

```sql
PARTITION BY LIST (statut) (
    PARTITION p_inconnu VALUES IN (NULL, 0),
    PARTITION p_actif   VALUES IN (1),
    PARTITION p_inactif VALUES IN (2)
);
```

Si `NULL` n'est listé nulle part et qu'aucune partition `DEFAULT` n'existe, l'insertion d'une valeur nulle est rejetée avec l'erreur habituelle « Table has no partition for value NULL ». En présence d'une partition `DEFAULT`, les valeurs `NULL` non explicitement listées y sont en revanche dirigées. Lorsque la colonne peut contenir `NULL`, mieux vaut donc soit prévoir une partition la couvrant, soit s'en remettre à `DEFAULT` de façon assumée.

## LIST ou RANGE : comment choisir

Le choix entre `LIST` et `RANGE` découle directement de la nature des valeurs de partitionnement. On retient `RANGE` lorsque les valeurs sont **ordonnées et continues**, et que les requêtes raisonnent en termes d'intervalles — typiquement les données chronologiques. On retient `LIST` lorsque les valeurs forment un **ensemble discret et fini** sans ordre pertinent, et que les requêtes ciblent des catégories précises.

`LIST` présente par ailleurs un usage que `RANGE` ne permet pas aisément : la **consolidation** de valeurs éparses au sein d'une même partition. En regroupant plusieurs codes dans une seule liste, on équilibre la répartition des lignes et l'on réunit dans une même partition des entités que l'on traitera ensemble — par exemple toutes les régions d'un même continent, tous les statuts considérés comme « terminés ». Cette souplesse fait de `LIST` un bon outil de modélisation lorsque la logique de regroupement est métier plutôt que numérique.

Lorsque la colonne ne possède qu'un petit nombre de valeurs possibles mais qu'aucun regroupement particulier n'est recherché, et que l'on souhaite simplement une répartition uniforme, le partitionnement par hachage (`HASH` ou `KEY`), présenté à la section suivante, peut s'avérer plus simple à maintenir, car il n'exige pas de redéfinir les listes lorsque de nouvelles valeurs apparaissent.

## Maintenance des partitions LIST

À la différence du motif de fenêtre glissante du `RANGE`, qui suit l'écoulement du temps, les partitions `LIST` reflètent une **nomenclature** relativement stable : on les fait évoluer surtout lorsque l'ensemble des catégories change.

La suppression d'une catégorie et de ses données se fait, comme pour `RANGE`, par `DROP PARTITION`, opération quasi instantanée et destructrice :

```sql
ALTER TABLE ventes_regionales DROP PARTITION p_ouest;
```

L'ajout d'une nouvelle catégorie dépend de la présence d'une partition `DEFAULT`. Sans `DEFAULT`, on ajoute simplement une partition avec sa liste :

```sql
-- Possible uniquement s'il n'existe pas de partition DEFAULT
ALTER TABLE ventes_regionales ADD PARTITION (
    PARTITION p_outremer VALUES IN (11, 12)
);
```

En présence d'une partition `DEFAULT`, `ADD PARTITION` échoue — exactement comme `ADD` échoue après une borne `MAXVALUE` en `RANGE`. On extrait alors les nouvelles valeurs du fourre-tout au moyen de `REORGANIZE PARTITION`, ce qui redistribue physiquement les lignes concernées tout en conservant `DEFAULT` :

```sql
-- Sortir une nouvelle catégorie du fourre-tout
ALTER TABLE ventes_regionales REORGANIZE PARTITION p_autres INTO (
    PARTITION p_outremer VALUES IN (11, 12),
    PARTITION p_autres   DEFAULT
);
```

`REORGANIZE PARTITION` sert aussi à fusionner des partitions ou à redéfinir la répartition des valeurs entre elles, lorsque la nomenclature évolue ou que l'équilibrage doit être revu.

## Bonnes pratiques et limitations

Les recommandations générales du §15.9 s'appliquent pleinement au partitionnement par liste. Le partitionnement n'apporte un bénéfice tangible qu'à partir d'un volume significatif (de l'ordre du million de lignes), et il ne se substitue pas à une indexation correcte : sur une recherche par valeur unique, un index sur une table non partitionnée est souvent aussi rapide. Il est par ailleurs conseillé de garder un nombre de partitions raisonnable — quelques dizaines plutôt que des centaines — car un nombre élevé alourdit l'ouverture de la table et les vérifications d'état sans gain proportionnel.

Comme toujours, l'efficacité réelle du partitionnement se mesure à l'élagage qu'il permet : une requête filtrant sur la colonne de partitionnement ne consultera que la ou les partitions dont la liste contient les valeurs recherchées. Ce mécanisme et son diagnostic sont détaillés au §15.9.4.
