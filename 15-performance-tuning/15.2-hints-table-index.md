# 15.15.2 Hints de table et d'index 🆕

Une fois acquis le mécanisme d'adressage des blocs de requête (§15.15.1), on peut appliquer des hints concrets aux tables et aux index. Cette section couvre trois grandes catégories : les hints qui contrôlent les **algorithmes de jointure** (BKA, BNL), ceux qui pilotent les **stratégies d'accès aux index** (ICP, MRR), et ceux qui déterminent **quels index** l'optimiseur peut employer (la famille `INDEX`), auxquels s'ajoute le contrôle de l'optimisation par intervalle (`NO_RANGE_OPTIMIZATION`). Tous peuvent cibler une table, un index ou un bloc précis grâce à la notation `@bloc` vue précédemment. Un point commun important : ces hints désignent des **noms d'index**, jamais des noms de colonnes, et la clé primaire s'y nomme `PRIMARY`.

## Niveau table et niveau index

Les hints de cette section se répartissent selon leur granularité. Les hints **de niveau table** (BKA, BNL) agissent sur la mise en mémoire tampon des jointures pour une ou plusieurs tables. Les hints **de niveau index** (NO_ICP, MRR, famille `INDEX`, `NO_RANGE_OPTIMIZATION`) agissent sur la façon dont des index précis sont utilisés.

Une subtilité mérite d'être signalée pour les hints de jointure de niveau table : ils s'appliquent à la table qui **reçoit** des lignes des tables précédentes — celle contre laquelle on sonde —, et non à la table émettrice. Pour la syntaxe d'adressage (même bloc, ou avec `@bloc`), on se reportera au §15.15.1.

## Les hints d'algorithme de jointure : BNL et BKA

Ces deux hints contrôlent les stratégies de mise en tampon des jointures, qui visent à réduire les lectures répétées de la table interne. Ils s'expriment au niveau d'une table ou de tout un bloc, et **priment** sur les réglages globaux correspondants de `optimizer_switch`.

### BNL et NO_BNL (Block Nested Loop)

La jointure par blocs imbriqués met en tampon un bloc de lignes de la table externe et les traite ensemble contre la table interne, plutôt que de parcourir cette dernière ligne à ligne pour chaque ligne externe. On réduit ainsi le nombre de parcours de la table interne. Le hint `BNL()` relève en pratique le niveau de cache de jointure (`join_cache_level`) jusqu'à 4 pour les tables visées, ce qui active sa variante hachée (BNL-H) ; `NO_BNL()` le désactive.

```sql
-- Désactiver le block nested loop pour t1
SELECT /*+ NO_BNL(t1) */ ...  
FROM t1 JOIN t2 ON t1.ref = t2.id;  
```

### BKA et NO_BKA (Batched Key Access)

L'accès par clé groupée intervient lorsque la jointure s'appuie sur un index de la table interne : il met en tampon les clés issues de la table externe et recourt à la lecture multi-intervalle (MRR) pour récupérer les lignes correspondantes dans un ordre favorable au disque, réduisant les accès aléatoires. Le hint `BKA()` **active aussi MRR** — indispensable à son fonctionnement — ainsi que la variante hachée BKAH ; c'est une différence notable avec l'approche par variables de session, où MRR doit être activé séparément. `NO_BKA()` désactive le mécanisme.

```sql
-- Forcer le batched key access sur t2
SELECT /*+ BKA(t2) */ ...  
FROM t1 JOIN t2 ON t1.ref = t2.id;  
```

## Les hints d'accès aux index : NO_ICP et MRR

### NO_ICP (Index Condition Pushdown)

Le pushdown de condition d'index permet au moteur de stockage d'évaluer les conditions portant sur des colonnes indexées **directement sur les entrées d'index**, avant de récupérer la ligne complète — ce qui évite de coûteux accès aux lignes pour des entrées qui seront finalement rejetées. Cette optimisation s'applique automatiquement lorsqu'elle est bénéfique : il n'existe donc pas de hint positif pour la forcer, seulement `NO_ICP(table index…)` pour la désactiver sur les index nommés, dans les rares cas où son surcoût dépasserait son gain.

```sql
-- Désactiver l'ICP sur un index de t1
SELECT /*+ NO_ICP(t1 idx_statut) */ * FROM t1 WHERE statut = 'actif' AND montant > 100;
```

### MRR et NO_MRR (Multi-Range Read)

La lecture multi-intervalle collecte les identifiants de lignes issus d'un parcours d'index, les trie, puis récupère les lignes dans un ordre plus séquentiel, réduisant les accès aléatoires au disque — un gain particulièrement net sur les grandes tables. `MRR(table index…)` la force, `NO_MRR(table index…)` la désactive.

```sql
SELECT /*+ MRR(commandes idx_date) */ *  
FROM commandes WHERE date_cmd BETWEEN '2025-01-01' AND '2025-03-31';  
```

## Les hints de choix d'index : la famille INDEX

Cette famille de hints remplace, de façon plus fine, les anciens `FORCE INDEX` et `IGNORE INDEX` : elle permet de restreindre les index considérés par l'optimiseur, et de le faire **par portée** — accès aux lignes, regroupement ou tri. Le tableau suivant donne les correspondances.

| Hint | Équivalent hérité | Portée |
|------|-------------------|--------|
| `INDEX` / `NO_INDEX` | `FORCE INDEX` / `IGNORE INDEX` | Toutes (jointure, `GROUP BY`, tri) |
| `JOIN_INDEX` / `NO_JOIN_INDEX` | `FORCE`/`IGNORE INDEX FOR JOIN` | Accès aux lignes (range, ref…) |
| `GROUP_INDEX` / `NO_GROUP_INDEX` | `FORCE`/`IGNORE INDEX FOR GROUP BY` | Parcours d'index pour `GROUP BY` |
| `ORDER_INDEX` / `NO_ORDER_INDEX` | `FORCE`/`IGNORE INDEX FOR ORDER BY` | Tri des lignes (`ORDER BY`) |

Le hint `INDEX` agit comme la combinaison de `JOIN_INDEX`, `GROUP_INDEX` et `ORDER_INDEX` : il s'applique à toutes les portées. Les formes positives se comportent comme `FORCE INDEX` (et non comme `USE INDEX`) : le balayage de table est réputé très coûteux, de sorte que l'index nommé est fortement privilégié.

```sql
-- Forcer un index, toutes portées (équivalent FORCE INDEX)
SELECT /*+ INDEX(commandes idx_client) */ * FROM commandes WHERE client_id = 42;

-- Ignorer un index, toutes portées (équivalent IGNORE INDEX)
SELECT /*+ NO_INDEX(commandes idx_date) */ * FROM commandes WHERE statut = 'clos';

-- Restreindre la portée : index imposé uniquement pour le tri
SELECT /*+ ORDER_INDEX(commandes idx_date) */ * FROM commandes ORDER BY date_cmd;

-- Plusieurs index candidats ; la clé primaire se nomme PRIMARY
SELECT /*+ JOIN_INDEX(t1 PRIMARY, idx_ref) */ ... FROM t1 JOIN t2 ...;
```

Ces hints d'index occupent une place particulière dans l'ordre de priorité. Ils **priment sur les `FORCE`/`IGNORE INDEX`** correspondants — qu'ils rendent inopérants —, et plus largement sur **tous les autres hints d'optimiseur**, y compris les autres hints de cette section : les autres hints ne s'appliquent alors qu'aux index que cette famille autorise. Un comportement à connaître : un hint d'index positif restreint l'ensemble des index candidats, mais parmi ceux-ci l'optimiseur conserve son choix ; si aucun index autorisé ne se révèle utile, le hint peut être de fait ignoré, sans avertissement. À l'inverse, combiner deux hints contradictoires sur une même table, comme `INDEX(t1) JOIN_INDEX(t1)`, déclenche un avertissement de conflit.

Ces hints sont destinés à **supplanter** les anciens `USE`/`FORCE`/`IGNORE INDEX` ; pour une nouvelle requête, on les leur préférera.

## NO_RANGE_OPTIMIZATION

Ce hint **désactive l'accès par intervalle** (range) sur les index nommés, contraignant l'optimiseur à envisager d'autres méthodes d'accès. Il est utile lorsqu'un balayage par intervalle portant sur de très nombreuses plages — une longue liste `IN`, une accumulation de conditions `OR` — s'avère plus coûteux qu'une alternative. La clé primaire s'y désigne là encore par `PRIMARY`.

```sql
-- Interdire l'optimisation par intervalle sur la clé primaire de t1
UPDATE /*+ NO_RANGE_OPTIMIZATION(t1 PRIMARY) */ t1  
SET traite = 1 WHERE id IN (12, 45, 78, 102, ...);  
```

## Combiner et cibler les hints

Tous ces hints se combinent dans un même commentaire et se dirigent vers le bloc voulu au moyen de la notation `@bloc` (§15.15.1). L'exemple suivant désactive le block nested loop pour une table située dans une sous-requête nommée :

```sql
DELETE /*+ NO_BNL(t2@qb1) */ FROM t1  
WHERE a IN (SELECT /*+ QB_NAME(qb1) */ a FROM t2);  
```

On rappellera que, en cas de hints dupliqués ou contradictoires dans un même commentaire, c'est le premier qui est retenu, le suivant étant signalé par l'avertissement `4219` (§15.15).

## Bonnes pratiques

Comme pour tout hint, la démarche commence par un diagnostic à l'`EXPLAIN` (§5.7) et se conclut par une vérification de l'effet, là encore à l'`EXPLAIN`. Les hints d'index supposent une connaissance exacte des index de la table — on n'oubliera pas que la clé primaire se désigne par `PRIMARY` et que ce sont bien des noms d'index, non de colonnes, qui sont attendus. Les formes positives de la famille `INDEX` étant puissantes (de type `FORCE`), elles peuvent figer un plan devenu inadapté si la distribution des données évolue : il convient de les réexaminer périodiquement. Enfin, on privilégiera systématiquement ces hints nouvelle génération aux anciens `USE`/`FORCE`/`IGNORE INDEX`.
