# 15.9.3 Partitionnement par hachage (HASH et KEY)

Les méthodes `RANGE` et `LIST` reposent sur une **sémantique** : les partitions ont un sens métier (une période, une catégorie), et c'est au concepteur de définir explicitement les bornes ou les listes. Le partitionnement par hachage adopte une logique radicalement différente : il vise une **répartition mécanique et équilibrée** des lignes, sans qu'aucune partition ne porte de signification particulière. C'est le serveur qui décide, à partir d'une fonction de hachage, dans quelle partition placer chaque ligne. On y recourt lorsqu'aucune logique d'intervalle ni d'énumération ne se dégage, mais que l'on souhaite répartir uniformément le volume — par exemple pour étaler la charge d'écriture et éviter qu'une partition ne devienne un point chaud.

MariaDB propose deux variantes de ce principe, `HASH` et `KEY`, ainsi que leurs déclinaisons linéaires. La série 12.x introduit par ailleurs de **nouveaux algorithmes de hachage pour `PARTITION BY KEY`**, traités plus bas.

## Partitionnement par hachage : HASH

Le partitionnement `HASH` répartit les lignes selon une **expression fournie par l'utilisateur**, qui doit renvoyer un entier non constant et déterministe. Le serveur applique le modulo du nombre de partitions à la valeur de cette expression : une ligne est dirigée vers la partition d'indice `MOD(expression, nombre_de_partitions)`.

```sql
CREATE TABLE sessions (
    id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    utilisateur BIGINT UNSIGNED NOT NULL,
    debut       DATETIME NOT NULL,
    PRIMARY KEY (id, utilisateur)
) ENGINE = InnoDB
PARTITION BY HASH (utilisateur)  
PARTITIONS 8;  
```

Ici, les lignes sont réparties entre huit partitions selon `MOD(utilisateur, 8)`. La clause `PARTITIONS n` suffit à fixer le nombre de partitions : contrairement à `RANGE` et `LIST`, il est inutile (et impossible) de nommer ou de borner chaque partition individuellement, puisqu'elles n'ont pas de contenu sémantique.

Deux recommandations encadrent l'usage de `HASH`. D'une part, l'expression étant évaluée à chaque insertion et à chaque mise à jour, elle doit rester **simple** : une expression coûteuse pénaliserait toutes les écritures. D'autre part, il est conseillé de hacher sur **une seule colonne** dont la valeur évolue de façon cohérente, car c'est la condition pour que l'optimiseur puisse pratiquer un élagage ; faire intervenir plusieurs colonnes dans l'expression de hachage est généralement déconseillé.

## Partitionnement par clé : KEY

Le partitionnement `KEY` poursuit le même objectif d'équilibrage, mais avec une différence essentielle : au lieu d'une expression définie par l'utilisateur, il s'appuie sur une **fonction de hachage interne fournie par le serveur**, appliquée à une liste de colonnes. On ne lui passe donc pas une expression, mais des noms de colonnes.

```sql
CREATE TABLE documents (
    ref     CHAR(36) NOT NULL,   -- identifiant UUID
    contenu TEXT,
    PRIMARY KEY (ref)
) ENGINE = InnoDB
PARTITION BY KEY (ref)  
PARTITIONS 8;  
```

Cette approche présente deux avantages notables par rapport à `HASH`. D'abord, les colonnes utilisées ne sont **pas limitées aux types entiers** : on peut hacher directement sur une chaîne, un UUID ou tout autre type, ce que `HASH` ne permet pas sans transformation préalable. Ensuite, si aucune colonne n'est précisée, `KEY` utilise automatiquement la **clé primaire** de la table (ou, à défaut, une clé unique `NOT NULL`) :

```sql
CREATE TABLE evenements (
    id      BIGINT UNSIGNED NOT NULL,
    libelle VARCHAR(100),
    PRIMARY KEY (id)
) ENGINE = InnoDB
PARTITION BY KEY()  
PARTITIONS 4;  
```

Si la table ne possède ni clé primaire ni clé unique `NOT NULL`, l'omission des colonnes échoue avec l'erreur `1488 (HY000): Field in list of fields for partition function not found in table`. Par ailleurs, `KEY` ne prend pas en charge les préfixes d'index : une colonne déclarée avec un préfixe (par exemple `a(5)`) est silencieusement ignorée dans la clé de partitionnement, et si toutes les colonnes en utilisent un, la création échoue avec l'erreur `1503`.

## Nouveaux algorithmes de PARTITION BY KEY (MariaDB 12.3)

Avant la version 12.3, le choix de la fonction de hachage de `KEY` se réduisait à la lignée historique héritée de MySQL : la seule clause disponible était `ALGORITHM={1|2}` (où `1` désigne l'algorithme de MySQL 5.1 et `2`, le défaut, celui de MySQL 5.5), prévue avant tout pour **préserver le placement des lignes lors d'une mise à niveau** — et non pour donner accès à une fonction de hachage moderne. À partir de la **12.3**, cette clause `ALGORITHM` s'enrichit de **noms explicites**, dont des algorithmes récents, et permet de choisir réellement la fonction de hachage :

```sql
PARTITION BY KEY
  [ALGORITHM={MYSQL51 | MYSQL55 | BASE31 | CRC32C | XXH32 | XXH3}]
  ([colonnes])
  [PARTITIONS (nombre)]
```

Ces algorithmes se répartissent en trois familles. `MYSQL51` et `MYSQL55` sont les **algorithmes historiques** ; `MYSQL55` reste l'algorithme par défaut, et c'était déjà le cas avant la 12.3. `CRC32C`, `XXH32` et `XXH3` reposent sur des fonctions de hachage établies du même nom et constituent les **algorithmes recommandés** pour les nouvelles tables, car plus rapides et mieux répartis. Enfin, `BASE31` s'appuie sur une représentation en base 31 des octets ; il sert de référence simple, déjà plus uniformément distribuée que `MYSQL51` ou `MYSQL55` sur des données séquentielles simples.

```sql
CREATE TABLE documents (
    ref     CHAR(36) NOT NULL,
    contenu TEXT,
    PRIMARY KEY (ref)
) ENGINE = InnoDB
PARTITION BY KEY ALGORITHM=XXH3 (ref)  
PARTITIONS 8;  
```

L'intérêt de ces nouveaux algorithmes est qualitatif : les fonctions historiques répartissent assez mal les données séquentielles régulières (identifiants auto-incrémentés, dates contiguës), ce qui peut produire des partitions déséquilibrées et concentrer la charge. Les fonctions modernes de la famille xxHash ou `CRC32C` corrigent ce défaut tout en étant plus performantes à calculer.

Un point mérite une attention particulière, et relève directement de la migration (voir §19.10). L'algorithme de hachage **détermine la partition dans laquelle réside chaque ligne**. Modifier l'algorithme d'une table déjà peuplée force donc une **redistribution complète des lignes**, équivalente à une reconstruction de la table : ce n'est pas une opération anodine. C'est précisément pour cette raison que le défaut demeure `MYSQL55` : une mise à niveau depuis une version antérieure ne redistribue ainsi pas silencieusement les tables `KEY` existantes, qui conservent leur algorithme d'origine. Le choix d'un algorithme moderne se fait donc à la création des nouvelles tables, ou de façon délibérée et planifiée sur les tables existantes.

## Variantes linéaires : LINEAR HASH et LINEAR KEY

Les variantes `LINEAR HASH` et `LINEAR KEY` emploient un **algorithme en puissances de deux** au lieu du simple modulo. Le compromis est clairement défini : l'ajout, la suppression, la fusion et la division de partitions y sont **nettement plus rapides**, mais la répartition des données est **moins uniforme** qu'avec les variantes non linéaires.

```sql
PARTITION BY LINEAR KEY (ref)  
PARTITIONS 8;  
```

Le choix entre la forme classique et la forme linéaire dépend donc de l'usage. Si la table est statique en nombre de partitions, on privilégie `HASH` ou `KEY` pour une distribution optimale. Si l'on prévoit de redimensionner fréquemment l'ensemble des partitions et que l'on peut tolérer un certain déséquilibre, la variante linéaire réduit considérablement le coût de ces opérations.

## Maintenance : ADD et COALESCE

Le partitionnement par hachage n'ayant pas de notion de période ni de catégorie, sa maintenance ne consiste pas à ajouter ou retirer des partitions nommées, mais à **ajuster leur nombre**. On augmente le nombre de partitions avec `ADD PARTITION PARTITIONS n` et on le réduit avec `COALESCE PARTITION n` :

```sql
-- Porter le nombre de partitions de 8 à 12
ALTER TABLE documents ADD PARTITION PARTITIONS 4;

-- Le ramener de 12 à 8
ALTER TABLE documents COALESCE PARTITION 4;
```

Ces opérations ont un coût important qu'il faut anticiper. Comme la partition cible dépend du modulo (ou de l'algorithme linéaire), changer le nombre de partitions modifie l'affectation d'une grande partie des lignes, qui doivent alors être **physiquement redéplacées**. C'est sur ce terrain que les variantes linéaires reprennent l'avantage, en limitant l'ampleur de cette redistribution.

## Un élagage limité aux égalités

Le partitionnement par hachage souffre d'une limite déterminante par rapport à `RANGE` : son élagage ne fonctionne que pour les **conditions d'égalité** sur la clé de partitionnement (`col = valeur`), ainsi que pour les listes de valeurs (`col IN (…)`). Il ne fonctionne pas sur les intervalles.

La raison est intrinsèque au procédé : le hachage détruit l'ordre des valeurs. Deux valeurs voisines sont volontairement dispersées dans des partitions différentes, de sorte qu'un intervalle `BETWEEN` ou une comparaison `>` ne peut pas être traduit en un sous-ensemble contigu de partitions. Une requête par plage devra donc consulter **toutes les partitions**, annulant le bénéfice de l'élagage.

```sql
-- Élagage possible : recherche par égalité sur la clé de hachage
SELECT * FROM documents WHERE ref = '8f14e45f-...';

-- Aucun élagage : recherche par intervalle sur la clé de hachage
SELECT * FROM sessions WHERE utilisateur BETWEEN 100 AND 200;
```

Cette caractéristique oriente fortement le cas d'usage : `HASH` et `KEY` conviennent aux accès ponctuels répartis uniformément, et non aux balayages par plage, pour lesquels `RANGE` reste imbattable. Le mécanisme d'élagage et son diagnostic font l'objet du §15.9.4.

## Quand choisir HASH ou KEY

Le partitionnement par hachage s'impose lorsqu'aucune dimension d'intervalle ou de catégorie ne structure les données, mais que l'on cherche à **répartir équitablement** le volume et la charge — typiquement pour prévenir les points chauds en écriture sur une table très sollicitée. Entre les deux variantes, on retient `KEY` lorsque l'on souhaite hacher directement sur la clé primaire ou sur des colonnes non entières sans rédiger d'expression, et `HASH` lorsque l'on dispose d'une expression entière précise et que l'on vise un élagage sur égalité.

Pour toute nouvelle table partitionnée par clé sous MariaDB 12.3, il est par ailleurs recommandé de spécifier explicitement un algorithme moderne (`CRC32C` ou `XXH3`), plus rapide et mieux distribué que le défaut historique `MYSQL55`, conservé pour des raisons de compatibilité ascendante.
