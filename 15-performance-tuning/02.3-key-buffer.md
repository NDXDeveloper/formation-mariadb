# 15.2.3 Key buffer (MyISAM)

Le key buffer — ou *key cache* — est, pour MyISAM, l'équivalent du buffer pool d'InnoDB : un cache mémoire destiné à accélérer les lectures. Mais deux particularités le rendent aujourd'hui largement marginal sur une instance MariaDB moderne. Il ne met en cache que les index, et il ne sert que les tables MyISAM — un moteur devenu rare. Le connaître reste utile pour les systèmes hérités, et surtout pour ne pas lui réserver inutilement de la mémoire.

## Ce que met (et ne met pas) en cache le key buffer

Le paramètre `key_buffer_size` fixe la taille du cache des **blocs d'index** des tables MyISAM, partagé par tous les threads. La nuance est capitale : contrairement au buffer pool InnoDB, qui met en cache à la fois les pages de données et d'index, le key buffer **ne met en cache que les index** (fichiers `.MYI`). Les données des tables MyISAM (`.MYD`) ne disposent d'aucun cache dédié dans le serveur : elles reposent entièrement sur le cache de fichiers du système d'exploitation.

Autre point essentiel : le key buffer ne concerne **que MyISAM**. Le moteur Aria — successeur de MyISAM, utilisé notamment pour les tables temporaires internes — possède son propre cache, réglé par `aria_pagecache_buffer_size` (voir le [chapitre 7](../07-moteurs-de-stockage/04-aria.md)) ; quant à InnoDB, il s'appuie sur son buffer pool. Régler `key_buffer_size` n'a donc aucun effet sur les tables Aria ou InnoDB. Confondre ces deux caches est une erreur fréquente.

## Une pertinence devenue marginale

InnoDB est le moteur par défaut depuis longtemps, et MyISAM relève désormais de l'héritage. Dans MariaDB moderne, même les tables système ont migré vers Aria : la plupart des installations à partir de la 10.4 n'utilisent plus aucune table MyISAM de manière explicite. Concrètement, **si vous ne créez pas vous-même de tables MyISAM, le key buffer ne sert à rien.**

Or il a longtemps été fixé à **128 Mo par défaut** — autant de mémoire immobilisée sans usage. La recommandation officielle est donc claire : en l'absence de tables MyISAM, on ramène `key_buffer_size` à une valeur symbolique, de l'ordre de 64 Ko.

```ini
[mariadb]
# Aucune table MyISAM applicative → on libère la RAM
key_buffer_size = 64K
```

## Le dimensionner si vous utilisez encore MyISAM

Dans le cas — minoritaire — où des tables MyISAM sont réellement exploitées, le principe reste le même que pour le buffer pool (voir [§15.2.1](02.1-innodb-buffer-pool.md)) : le cache doit contenir le jeu d'index actif, sans le dépasser. On additionne les tailles d'index (`Index_length`, visible via `SHOW TABLE STATUS`) de toutes les tables MyISAM et l'on dimensionne `key_buffer_size` au plus à cette valeur. Sur un serveur historiquement bâti autour de MyISAM, la règle empirique consiste à lui consacrer environ 25 % de la RAM. Le paramètre est dynamique : il s'ajuste par `SET GLOBAL` sans redémarrage.

La validation passe, là encore, par un taux de succès, à l'aide de deux compteurs de `SHOW GLOBAL STATUS` :

```sql
SHOW GLOBAL STATUS LIKE 'Key_read%';
```

`Key_read_requests` compte les demandes de lecture d'un bloc d'index, et `Key_reads` celles qui ont dû descendre jusqu'au disque. Le rapport `Key_reads` / `Key_read_requests` doit être le plus faible possible : 1 pour 1000 est un bon objectif, 1 pour 100 reste acceptable, tandis que 1 pour 10 traduit un cache nettement sous-dimensionné.

Pour les déploiements MyISAM exigeants, le moteur offre des réglages plus fins — caches nommés multiples (`CACHE INDEX`, `LOAD INDEX INTO CACHE`) permettant de dédier un cache à certaines tables, paramètres `key_cache_block_size`, `key_cache_division_limit` et `key_cache_age_threshold`, ou encore key cache segmenté pour réduire la contention. Ces réglages ne sont quasiment plus nécessaires aujourd'hui ; l'équivalent segmenté côté Aria est traité en [§7.4.1](../07-moteurs-de-stockage/04.1-aria-segmented-key-cache.md).

## En résumé

Le key buffer (`key_buffer_size`) est le cache d'index de MyISAM : il ne couvre que les index (les données passant par le cache de l'OS) et ne s'applique qu'à MyISAM, à l'exclusion d'Aria (qui utilise `aria_pagecache_buffer_size`) et d'InnoDB. Sur une instance MariaDB actuelle, où InnoDB règne et où les tables système sont en Aria, il est le plus souvent inutile : on le réduit alors à ~64 Ko pour récupérer la mémoire. Lorsqu'on exploite réellement MyISAM, on le dimensionne au volume d'index utilisé et on contrôle le rapport `Key_reads` / `Key_read_requests`. Ce dernier réglage clôt le tour d'horizon de la [configuration mémoire](02-configuration-memoire.md) ; la section suivante explique pourquoi le query cache, lui, est déprécié.
