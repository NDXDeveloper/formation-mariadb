# 15.7.1 Slow query log

Comme introduit au §15.7, le *slow query log* est le moyen historique et universel de capturer les requêtes lentes dans un fichier (ou une table) en vue de leur analyse. Cette sous-section détaille son activation, le réglage du seuil, le choix de ce qui est journalisé, les statistiques étendues propres à MariaDB, le format d'une entrée, et la manière de lire le journal.

Un point de vocabulaire d'abord : **MariaDB 10.11 a renommé** les variables du *slow query log* pour les uniformiser sous le préfixe `log_slow_*`. Les noms historiques restent disponibles comme **alias**, mais les exemples ci-dessous mentionnent les deux formes.

| Nom historique | Nom depuis MariaDB 10.11 |
|---|---|
| `slow_query_log` | `log_slow_query` |
| `slow_query_log_file` | `log_slow_query_file` |
| `long_query_time` | `log_slow_query_time` |
| `min_examined_row_limit` | `log_slow_min_examined_row_limit` |

---

## Activer le slow query log

Le journal est **désactivé par défaut**. On l'active avec `slow_query_log` (alias `log_slow_query`) et l'on choisit éventuellement le fichier cible avec `slow_query_log_file` (alias `log_slow_query_file`) ; en l'absence de chemin, le fichier prend par défaut le nom `<hôte>-slow.log` dans le `datadir`.

La destination est gouvernée par `log_output`, dont la valeur par défaut sous MariaDB est `FILE`. Elle peut aussi valoir `TABLE` (les requêtes sont alors écrites dans la table `mysql.slow_log`) ou les deux (`FILE,TABLE`). La gestion générale des journaux et leur rotation sont traitées au §11.4.

Ces variables sont dynamiques (modifiables par `SET GLOBAL`) et se fixent durablement dans le fichier de configuration :

```ini
[mariadb]
log_output            = FILE  
log_slow_query        = ON  
log_slow_query_file   = /var/log/mysql/mariadb-slow.log  
log_slow_query_time   = 2.0  
```

---

## Définir le seuil : `long_query_time`

Le seuil de durée au-delà duquel une requête est considérée comme lente est fixé par `long_query_time` (alias `log_slow_query_time`). Sa **valeur par défaut est de 10 secondes**, ce qui est élevé pour la plupart des usages. Lorsque `log_output = FILE`, il accepte une valeur décimale jusqu'à la microseconde (par exemple `0.5`) ; avec `log_output = TABLE`, seule une valeur entière (résolution à la seconde) est admise. La valeur `0` journalise **toutes** les requêtes.

Ce paramètre étant dynamique et réglable par session, il est fréquent de l'abaisser temporairement pour capturer largement pendant une fenêtre d'analyse. Mais il faut garder à l'esprit la limite vue au §15.7 : un seuil élevé **ignore les requêtes rapides mais fréquentes**, qui sont souvent les vrais responsables. Pour une vision complète et continue à faible coût, le Performance Schema (§15.8) constitue une alternative.

---

## Choisir ce qu'on journalise

Au-delà du seuil de durée, plusieurs leviers affinent ce qui est consigné :

- **`log_queries_not_using_indexes`** (désactivé par défaut) journalise les requêtes n'utilisant aucun index — ou réalisant un balayage d'index qui ne réduit pas le nombre de lignes —, **quelle que soit leur durée**. C'est un outil puissant pour débusquer les index manquants (cf. §5.5). Depuis la 10.3.1, ce comportement correspond à `not_using_index` dans `log_slow_filter`.
- **`min_examined_row_limit`** (alias `log_slow_min_examined_row_limit`, `0` par défaut) ne journalise que les requêtes ayant examiné au moins *N* lignes, ce qui écarte les requêtes triviales.
- **`log_slow_filter`** restreint la journalisation à certains **types d'événements** : par exemple `filesort`, `filesort_on_disk`, `tmp_table`, `tmp_table_on_disk`, `full_join`, `full_scan`, `not_using_index`, `query_cache`, `query_cache_miss`, ou `admin` ; la valeur `all` active tous les filtres. Cela permet de cibler une catégorie précise d'inefficacité.

Par défaut, le journal **ne consigne pas les instructions administratives** (`ALTER TABLE`, `ANALYZE TABLE`, `CHECK TABLE`, `CREATE INDEX`, `DROP INDEX`, `OPTIMIZE TABLE`, `REPAIR TABLE`, ainsi que `ALTER SEQUENCE`). La variable `log_slow_admin_statements` qui servait à les inclure est **dépréciée depuis MariaDB 11.0** : la méthode actuelle consiste à ajouter `admin` à `log_slow_filter`. Enfin, `log_slow_slave_statements` (variante `replica`) permet de journaliser les requêtes lentes appliquées par la réplication.

---

## Maîtriser le volume : l'échantillonnage

À seuil bas, ou avec `log_queries_not_using_indexes` activé sur un serveur chargé, le journal peut **grossir très vite** et engendrer un surcoût d'E/S non négligeable — d'autant que le journal est écrit par le thread qui exécute la requête, ce qui ajoute de la latence.

Pour limiter ce volume, `log_slow_rate_limit` (valeur `1` par défaut, soit toutes les requêtes) ne journalise qu'environ **une requête lente sur *N***, ce qui conserve un échantillon représentatif tout en réduisant la pression sur le disque — c'est le levier d'échantillonnage de MariaDB. (Attention pour qui vient de MySQL : la variable `log_throttle_queries_not_using_indexes`, qui y borne par minute la journalisation des requêtes sans index, **n'existe pas sous MariaDB** ; c'est `log_slow_rate_limit` qui remplit ce rôle d'échantillonnage.) La rotation du fichier (`FLUSH SLOW LOGS`, ou un outil comme `logrotate`) est traitée au §11.4.

---

## Enrichir le diagnostic : `log_slow_verbosity`

Par défaut, chaque entrée ne contient que des informations sommaires. La variable `log_slow_verbosity` ajoute des éléments de diagnostic ; en MariaDB 12.3, ses valeurs possibles (combinables par des virgules) sont **`query_plan`, `innodb`, `explain`, `engine`, `warnings`, `all` et `full`** (les valeurs `warnings` et `all` ayant été ajoutées en 10.6.16). Elle n'est **pas prise en charge** lorsque `log_output = 'TABLE'`.

Les options les plus utiles sont :

- **`explain`** : insère la sortie `EXPLAIN` de la requête. Particularité MariaDB : cette sortie est générée en interne via l'instruction `ANALYZE`, si bien qu'elle inclut les colonnes `r_rows` et `r_filtered` — les **statistiques réelles d'exécution** en regard des estimations (voir §5.7 pour leur interprétation).
- **`query_plan`** : les informations sur le plan d'exécution.
- **`innodb`** : les statistiques du moteur InnoDB pour la requête.
- **`warnings`** : les erreurs, avertissements et notes associés.

Pour une session d'analyse, une combinaison comme `query_plan,explain` (éventuellement `query_plan,innodb,explain`) est recommandée :

```sql
SET GLOBAL log_slow_verbosity = 'query_plan,explain';
```

## Format d'une entrée

Une entrée écrite dans un fichier se présente typiquement ainsi :

```
# Time: 240607 14:23:05
# User@Host: app[app] @ localhost []
# Thread_id: 42  Schema: boutique  QC_hit: No
# Query_time: 3.241560  Lock_time: 0.000120  Rows_sent: 12  Rows_examined: 1840552
SET timestamp=1717763785;  
SELECT * FROM commandes WHERE YEAR(date_commande) = 2024 ORDER BY total DESC;  
```

Les champs déterminants sont `Query_time` (durée totale), `Lock_time` (temps passé en attente de verrous), et surtout le couple `Rows_examined` / `Rows_sent`. Dans l'exemple ci-dessus, la requête examine plus de 1,8 million de lignes pour n'en renvoyer que 12 : c'est la signature d'un balayage complet et d'une mauvaise sélectivité (ici aggravée par l'usage de `YEAR(date_commande)`, non *sargable*, qui empêche l'utilisation d'un index). Ce rapport est, comme rappelé au §15.7, le signal le plus parlant. Avec une verbosité accrue, l'entrée contient en outre le plan d'exécution et la sortie `EXPLAIN`/`ANALYZE`.

---

## Lire et analyser le journal

Lorsqu'il est écrit dans un fichier, le journal se consulte avec n'importe quel éditeur de texte pour une inspection ponctuelle, mais son exploitation se fait surtout avec des outils :

- **`mariadb-dumpslow`** (anciennement `mysqldumpslow`), fourni avec le serveur, produit un résumé agrégé rapide ;
- **`pt-query-digest`** du Percona Toolkit, traité au §15.7.2, fournit l'analyse agrégée la plus complète — c'est l'outil recommandé.

Lorsque la destination est `TABLE`, le journal s'interroge directement en SQL sur `mysql.slow_log` (par exemple `ORDER BY query_time DESC`), au prix d'une agrégation moins commode qu'avec `pt-query-digest`. Dans tous les cas, on se rappellera la règle du §15.7 : **classer les requêtes par temps total consommé**, et non par leur seule lenteur individuelle.

---

## Bonnes pratiques

En synthèse : plutôt que de laisser un seuil à `0` en permanence, on abaisse temporairement `long_query_time` pour des fenêtres d'analyse ciblées ; sur un serveur chargé, on recourt à l'échantillonnage (`log_slow_rate_limit`) et l'on surveille la croissance du fichier ; on active `log_slow_verbosity` le temps d'un diagnostic. Pour un suivi continu à faible surcoût, le Performance Schema (§15.8) reste préférable au *slow query log* permanent.

---

## Points clés à retenir

- Le *slow query log* journalise les requêtes dépassant **`long_query_time`** (défaut **10 s**, décimal possible avec `FILE`, `0` = tout) ; il est **désactivé par défaut** et écrit dans un fichier (`log_output=FILE`) ou dans `mysql.slow_log` (`TABLE`).
- MariaDB 10.11 a **renommé** les variables (`log_slow_query`, `log_slow_query_file`, `log_slow_query_time`…), les anciens noms restant des alias.
- On cible le contenu avec **`log_queries_not_using_indexes`**, `min_examined_row_limit` et **`log_slow_filter`** ; les instructions administratives ne sont pas journalisées par défaut (`log_slow_admin_statements` est **déprécié depuis 11.0**, remplacé par `admin` dans `log_slow_filter`).
- On maîtrise le volume avec **`log_slow_rate_limit`** (échantillonnage), indispensable sur un serveur chargé.
- **`log_slow_verbosity`** (`query_plan`, `explain`, `innodb`, `warnings`, `all`, `full`) enrichit le diagnostic ; l'option `explain` fournit un `EXPLAIN` avec statistiques réelles (`r_rows`, `r_filtered`).
- Le champ clé d'une entrée est le rapport **`Rows_examined` / `Rows_sent`** ; l'analyse se fait avec `mariadb-dumpslow` ou, mieux, **`pt-query-digest`** (§15.7.2).
