# 15.15.3 Hints d'ordre de jointure 🆕

L'**ordre dans lequel les tables sont jointes** est l'une des décisions les plus lourdes de conséquences de l'optimiseur : sur une requête à plusieurs tables, un mauvais ordre peut multiplier le coût d'exécution par plusieurs ordres de grandeur. Jusqu'à la série 12.x, le seul moyen de forcer cet ordre était le mot-clé `STRAIGHT_JOIN`, un mécanisme tout-ou-rien. La série 12.x — à partir de la 12.0, et listée parmi les nouveautés de la 12.3 — a introduit **quatre hints d'ordre de jointure** qui permettent de contraindre cet ordre de façon totale, partielle, ou seulement à son début ou à sa fin : `JOIN_FIXED_ORDER`, `JOIN_ORDER`, `JOIN_PREFIX` et `JOIN_SUFFIX`.

## Pourquoi contrôler l'ordre de jointure ?

L'optimiseur détermine le meilleur ordre de jointure en estimant le coût des différentes combinaisons. Mais l'espace de recherche croît très vite avec le nombre de tables, et une estimation faussée — données déséquilibrées, statistiques périmées — peut conduire à un ordre catastrophique. Le mécanisme hérité, `STRAIGHT_JOIN`, impose l'**ordre complet du `FROM`** : c'est tout ou rien, et cela suppose que l'ordre écrit dans le `FROM` corresponde précisément à celui que l'on souhaite. Les nouveaux hints offrent au contraire un contrôle granulaire, sans avoir à réécrire la requête.

## Les quatre hints

### JOIN_FIXED_ORDER

`JOIN_FIXED_ORDER` force l'optimiseur à joindre les tables dans l'ordre où elles apparaissent dans la clause `FROM` — c'est l'exact équivalent de `STRAIGHT_JOIN`. Particularité syntaxique : contrairement aux autres hints d'ordre, **il n'accepte aucune liste de tables** dans son corps (tout au plus un nom de bloc de requête).

```sql
-- Joindre dans l'ordre du FROM : t2 puis t1,
-- même si l'ordre t1 → t2 paraît plus prometteur
SELECT /*+ JOIN_FIXED_ORDER() */ f1, f2  
FROM t2 JOIN t1 ON t1.id = t2.id  
ORDER BY f1, f2;  
```

### JOIN_ORDER

`JOIN_ORDER` impose l'ordre **relatif** des tables nommées, sans figer la position des autres : l'optimiseur peut placer les tables non citées n'importe où, y compris **entre** les tables nommées. C'est l'outil le plus souple lorsqu'on veut garantir qu'une table en précède une autre, sans contraindre tout le plan.

```sql
-- Imposer l'ordre relatif c → a → b (les tables sont désignées par leurs alias)
SELECT /*+ JOIN_ORDER(c, a, b) */ a.*  
FROM table_a a  
JOIN table_b b ON a.id = b.a_id  
JOIN table_c c ON b.id = c.b_id;  
```

### JOIN_PREFIX

`JOIN_PREFIX` impose que les tables nommées soient les **premières** du plan de jointure, dans l'ordre indiqué ; toutes les autres tables sont placées après, par l'optimiseur.

```sql
-- t1 puis t2 en tête de jointure ; le reste ensuite
SELECT /*+ JOIN_PREFIX(t1, t2) */ ...  
FROM t1 JOIN t2 ON ... JOIN t3 ON ... JOIN t4 ON ...;  
```

### JOIN_SUFFIX

`JOIN_SUFFIX`, symétrique du précédent, impose que les tables nommées soient les **dernières** du plan, dans l'ordre indiqué ; toutes les autres sont placées avant.

```sql
-- t4 en fin de jointure ; le reste avant
SELECT /*+ JOIN_SUFFIX(t4) */ ...  
FROM t1 JOIN t2 ON ... JOIN t3 ON ... JOIN t4 ON ...;  
```

## JOIN_FIXED_ORDER face à STRAIGHT_JOIN

`JOIN_FIXED_ORDER` et `STRAIGHT_JOIN` produisent le même effet — l'ordre du `FROM` —, le premier étant la forme « nouvelle génération ». Mais la famille tout entière apporte deux avantages sur l'ancien mot-clé. D'une part, elle est **plus simple à écrire** : exprimer `JOIN_ORDER(A, B, C)` est bien plus commode que de réécrire une requête en « A STRAIGHT_JOIN B STRAIGHT_JOIN C », opération qui obligerait à remanier les expressions `ON`, les jointures externes, etc. D'autre part, là où `STRAIGHT_JOIN` n'offre que l'ordre complet du `FROM`, les hints `JOIN_ORDER`, `JOIN_PREFIX` et `JOIN_SUFFIX` autorisent un ordonnancement **partiel ou personnalisé**, différent de l'ordre d'écriture. L'ensemble du contrôle de l'ordre se règle ainsi depuis un seul endroit : le commentaire de hints.

## Désigner les tables : alias et @bloc

Les hints d'ordre de jointure désignent les tables par leur **alias** lorsqu'elles en possèdent un — c'est l'alias, et non le nom de base de la table, qu'il faut alors employer ; à défaut d'alias, on emploie le nom de la table. Une difficulté connue tient à la fusion des sous-requêtes : un alias peut devenir ambigu lorsqu'une sous-requête est fusionnée dans le bloc englobant.

Pour les tables issues de sous-requêtes, on recourt à la notation `@bloc` (§15.15.1). Lorsqu'une sous-requête `WHERE … IN (SELECT …)` est convertie en semi-jointure, ses tables sont fusionnées dans le bloc externe, et c'est le hint d'ordre placé dans ce bloc externe qui les contrôle, en les désignant par `table@bloc` :

```sql
SELECT /*+ JOIN_PREFIX(t2, t5@subq2) JOIN_SUFFIX(t3) */ COUNT(*)  
FROM t1 JOIN t2 JOIN t3  
WHERE t1.a IN (SELECT /*+ QB_NAME(subq1) */ a FROM t4)  
  AND t2.a IN (SELECT /*+ QB_NAME(subq2) */ a FROM t5);
```

## Combiner les hints d'ordre : les règles de résolution

Plusieurs hints d'ordre peuvent coexister dans un même commentaire, selon des règles précises résumées ci-dessous.

| Combinaison | Résultat |
|-------------|----------|
| Deux `JOIN_PREFIX` (ou deux `JOIN_SUFFIX`) | Le second est ignoré, avec un avertissement |
| Plusieurs `JOIN_ORDER` | Tous appliqués (pas d'avertissement) |
| `JOIN_PREFIX` + `JOIN_SUFFIX` | Tous deux appliqués (compatibles) |
| Dépendance circulaire entre hints | Le hint en conflit est ignoré |

Ainsi, un seul `JOIN_PREFIX` et un seul `JOIN_SUFFIX` sont retenus, tandis que `JOIN_ORDER` peut être répété. Le cas de la **dépendance circulaire** mérite un mot : avec `/*+ JOIN_ORDER(t1, t2) JOIN_PREFIX(t2, t1) */`, le `JOIN_ORDER` rend `t2` dépendante de `t1`, ce qui interdit au `JOIN_PREFIX` d'imposer l'inverse ; ce dernier est donc ignoré.

## Quand un hint d'ordre est ignoré

Plusieurs situations conduisent l'optimiseur à écarter un hint d'ordre. Les **tables constantes** (const) sont toujours placées en tête du plan, et aucun hint ne peut changer leur position : une référence à une table const dans un hint d'ordre est ignorée, le reste du hint restant applicable. Un hint qui **contredit le type de jointure** — par exemple un ordre incompatible avec une jointure externe `LEFT JOIN` — est ignoré, et ce **sans avertissement**. Enfin, les dépendances circulaires évoquées plus haut entraînent l'abandon du hint en conflit.

Un point de vérification est précieux ici : les hints d'ordre **ignorés n'apparaissent pas** dans la sortie de `EXPLAIN` étendu, qui n'affiche que les hints effectivement retenus. C'est donc le moyen le plus sûr de contrôler qu'un hint a bien été pris en compte.

## Bonnes pratiques

La démarche reste celle de tous les hints. On diagnostique d'abord l'ordre de jointure courant avec `EXPLAIN` (§5.7), puis l'on vérifie, après ajout du hint, que l'ordre souhaité figure bien dans l'`EXPLAIN` étendu — un hint absent de cette sortie ayant été ignoré. On choisira la contrainte **minimale** suffisante : `JOIN_PREFIX` pour ne fixer que le début du plan, `JOIN_SUFFIX` pour n'en fixer que la fin, `JOIN_ORDER` pour imposer un ordre relatif précis, et `JOIN_FIXED_ORDER` uniquement lorsqu'on veut réellement l'ordre complet du `FROM`. Comme tout hint, un ordre forcé peut devenir inadapté à mesure que les données évoluent : il convient de le réexaminer périodiquement (§15.15).
