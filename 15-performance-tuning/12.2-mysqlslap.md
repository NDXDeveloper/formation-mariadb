# 15.12.2 mysqlslap

**mysqlslap** — renommé **`mariadb-slap`** depuis MariaDB 10.5 — est un outil d'émulation de charge livré directement avec MariaDB. Plus simple que sysbench et ne nécessitant aucune installation séparée, il convient aux tests de charge rapides, aux charges générées automatiquement et à la comparaison de moteurs de stockage. C'est l'option intégrée évoquée au §15.12, complément naturel de sysbench.

## Présentation et nommage

mysqlslap émule **plusieurs connexions concurrentes**, exécute un jeu de requêtes un certain nombre de fois, et rapporte le temps de chaque étape. Étant fourni avec le serveur, il est immédiatement disponible sans déploiement préalable.

Depuis MariaDB 10.5, le client porte le nom `mariadb-slap`, conformément à la convention de renommage des outils MariaDB (comme `mariadb-dump`, §12.2.1). Le nom historique `mysqlslap` reste accessible via un lien symbolique sous Linux, ou un binaire alternatif sous Windows ; les deux invoquent le même programme. On emploiera ici le nom moderne `mariadb-slap`.

## Comment fonctionne mariadb-slap

Un test se déroule en trois étapes logiques. Une étape de **préparation** crée éventuellement un schéma et une table, puis y insère des données — cette étape utilise une connexion unique. Une étape d'**exécution** lance les requêtes avec le nombre de clients demandé. Une étape de **nettoyage** supprime le schéma à la fin. Par défaut, l'outil crée et utilise un schéma nommé `mysqlslap`.

Deux façons de fournir la charge s'offrent à l'utilisateur : laisser mariadb-slap **générer le SQL automatiquement**, ou lui **fournir ses propres instructions** de création et de requête. Le premier mode permet un test immédiat sans aucune préparation ; le second, plus représentatif, permet d'éprouver son schéma et ses requêtes réels.

## Les deux modes : SQL généré ou SQL fourni

### SQL généré automatiquement

L'option `--auto-generate-sql` (forme courte `-a`) demande à l'outil de fabriquer lui-même les instructions. Le type de charge se choisit avec `--auto-generate-sql-load-type`, dont les valeurs possibles sont `read` (balayage de tables), `write` (insertions), `key` (lecture par clé primaire), `update` (mise à jour par clé primaire) et `mixed` (moitié insertions, moitié lectures par balayage), cette dernière étant la valeur par défaut.

La table générée se façonne avec `--number-int-cols` et `--number-char-cols`, qui fixent le nombre de colonnes entières et de colonnes de caractères. D'autres options affinent la génération, comme `--auto-generate-sql-write-number` (nombre d'insertions par thread, 100 par défaut) ou `--auto-generate-sql-add-autoincrement` (ajout d'une colonne auto-incrémentée). On obtient ainsi une charge synthétique en une seule commande.

### SQL fourni : --create et --query

Pour un test plus fidèle, on fournit ses propres instructions au moyen de `--create` (création et chargement) et `--query` (requêtes à mesurer), chacune acceptant soit une chaîne, soit un fichier. Lorsque plusieurs instructions sont regroupées, l'option `--delimiter` indique le séparateur — ce qui permet, dans un fichier, de placer plusieurs instructions ou une instruction sur plusieurs lignes. On notera que mariadb-slap ne comprend pas les commentaires dans ces fichiers.

Ce mode est le plus utile en pratique, car il met à l'épreuve le schéma et les requêtes effectivement utilisés par l'application, et non une charge générique.

## Les options de charge

Le **niveau de concurrence** se règle par `--concurrency`, qui fixe le nombre de clients simultanés. Cette option accepte une **liste** de valeurs séparées par des virgules : le test est alors rejoué à chaque niveau, ce qui produit en une commande un balayage de concurrence — un moyen commode d'observer l'évolution des performances à mesure que la charge augmente.

Le nombre de **répétitions** se fixe par `--iterations` : le test entier est exécuté ce nombre de fois, ce qui permet d'en moyenner les résultats — application directe du principe de répétition du §15.12. L'option `--number-of-queries` limite chaque client à un nombre approximatif de requêtes, le décompte tenant compte du délimiteur.

Une force distinctive de mariadb-slap est l'option `--engine`, qui accepte plusieurs moteurs de stockage : le même test est alors exécuté contre chacun, fournissant une **comparaison directe** entre, par exemple, InnoDB et Aria. S'y ajoutent des options pratiques comme `--create-schema` (schéma de travail), `--no-drop` (conserver le schéma à la fin), `--commit` (valider toutes les N instructions) et `--detach` (rouvrir la connexion après N requêtes), ainsi que les options de connexion habituelles (`--host`, `--port`, `--user`, `--password`, `--socket`).

## Exemple de déroulé

Les commandes suivantes illustrent les principaux usages.

```bash
# Charge générée automatiquement : 50 clients, 100 itérations, profil mixte
mariadb-slap \
  --user=bench --password=secret \
  --auto-generate-sql \
  --auto-generate-sql-load-type=mixed \
  --number-int-cols=3 --number-char-cols=2 \
  --concurrency=50 --iterations=100

# Balayage de concurrence : le test est rejoué à 10, 50 puis 100 clients
mariadb-slap \
  --user=bench --password=secret \
  --auto-generate-sql --auto-generate-sql-load-type=read \
  --concurrency=10,50,100 --iterations=50

# Schéma et requêtes fournis (sous forme de chaînes)
mariadb-slap \
  --user=bench --password=secret \
  --delimiter=";" \
  --create="CREATE TABLE t (a INT); INSERT INTO t VALUES (5)" \
  --query="SELECT * FROM t" \
  --concurrency=40 --iterations=100

# Comparaison de deux moteurs sur la même charge
mariadb-slap \
  --user=bench --password=secret \
  --auto-generate-sql --auto-generate-sql-load-type=write \
  --engine=innodb,aria \
  --concurrency=20 --iterations=50
```

## Lire les résultats

mariadb-slap produit un bloc de résultats synthétique pour chaque configuration testée. En voici un exemple (chiffres illustratifs) :

```
Benchmark
    Average number of seconds to run all queries: 0.234 seconds
    Minimum number of seconds to run all queries: 0.213 seconds
    Maximum number of seconds to run all queries: 0.267 seconds
    Number of clients running queries: 50
    Average number of queries per client: 200
```

L'indicateur principal est le **temps moyen** pour exécuter l'ensemble des requêtes : plus il est bas, meilleures sont les performances. Les valeurs **minimale et maximale** renseignent sur la variabilité d'une itération à l'autre — un écart important signale une instabilité qui mérite d'être expliquée. Lorsqu'on indique plusieurs niveaux de concurrence ou plusieurs moteurs, un bloc distinct est produit pour chaque combinaison, ce qui rend les comparaisons immédiates.

## mariadb-slap face à sysbench : que choisir ?

Les deux outils répondent à des besoins différents. mariadb-slap a pour lui d'être **intégré** (aucune installation), **rapide** à mettre en œuvre, capable de générer une charge en une commande, de comparer aisément des moteurs et d'exécuter directement vos propres requêtes. En contrepartie, il ne rapporte qu'un **temps total moyen** assorti de ses bornes, sans percentiles de latence ; sa charge générée est moins représentative d'une véritable charge OLTP transactionnelle, et il n'offre pas la souplesse de scripts Lua.

sysbench (§15.12.1), à l'inverse, est l'outil de référence pour un benchmark OLTP **rigoureux** : conscient des percentiles, extensible en Lua, mais nécessitant une installation et davantage de préparation. La règle de choix se résume ainsi : mariadb-slap pour une vérification rapide intégrée, une comparaison de moteurs ou un test de charge sur ses propres requêtes ; sysbench pour un benchmark OLTP reproductible et attentif à la latence de traîne. Dans les deux cas, c'est la **méthodologie** du §15.12 — volume réaliste, montée en température, répétitions, maîtrise de l'effet de l'observateur — qui conditionne la valeur des résultats, bien davantage que l'outil retenu.
