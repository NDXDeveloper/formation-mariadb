# 15.15.5 MAX_EXECUTION_TIME et limitation du temps de requête 🆕

> **Nouveauté de la série 12.x.** Le *hint* `MAX_EXECUTION_TIME` fait partie des **Expanded (New-Style) Optimizer Hints** introduits dans MariaDB **12.0 / 12.1** et consolidés dans **12.3 LTS**. Il complète — sans le remplacer — le mécanisme historique de MariaDB fondé sur la variable système `max_statement_time` (présente depuis la 10.1). Il s'agit avant tout d'un apport de **compatibilité avec MySQL**.

---

## Pourquoi limiter le temps d'une requête ?

Une requête « folle » — mauvais plan d'exécution, jointure cartésienne involontaire, prédicat oublié, statistiques périmées, sous-requête corrélée coûteuse — peut monopoliser le CPU, gonfler le *Buffer Pool*, saturer l'espace temporaire et dégrader les temps de réponse de **toute l'instance**. Plutôt que de la subir puis de tuer manuellement la session avec `KILL`, on peut fixer **a priori** une borne de temps : au-delà, le serveur interrompt lui-même la requête et rend la main à l'application avec une erreur de dépassement de délai.

C'est un garde-fou particulièrement utile pour :

- les **applications interactives** (API, pages web) où une requête lente bloque un thread applicatif et propage la latence à l'utilisateur final ;
- les **environnements multi-tenants** ou de *reporting* ad hoc, où des requêtes imprévisibles sont émises par des humains ou des outils tiers ;
- la **protection d'une réplique de lecture** contre des requêtes analytiques trop lourdes.

MariaDB 12.3 propose **deux outils complémentaires** pour cela :

| Outil | Unité | Granularité | Requêtes visées | Origine |
|-------|-------|-------------|-----------------|---------|
| Hint `MAX_EXECUTION_TIME(N)` | **millisecondes** | une requête (le `SELECT` qui le porte) | `SELECT` en lecture seule uniquement | Compatibilité MySQL (12.x) |
| Variable `max_statement_time` | **secondes** (DOUBLE) | globale / session / utilisateur / requête | toute requête (hors procédures stockées) | Natif MariaDB (depuis 10.1.1) |

La distinction d'**unité** entre les deux (millisecondes contre secondes) est la principale source d'erreurs : voir l'encadré dédié plus bas.

---

## Le hint `MAX_EXECUTION_TIME`

### Syntaxe

Le hint s'insère dans un commentaire spécial `/*+ ... */` placé immédiatement après le mot-clé `SELECT` :

```sql
SELECT /*+ MAX_EXECUTION_TIME(N) */ ...
```

où `N` est le délai maximal **en millisecondes**. Au-delà, le serveur tente d'interrompre la requête.

```sql
-- Ce SELECT est interrompu s'il dépasse 2000 ms (soit 2 secondes)
SELECT /*+ MAX_EXECUTION_TIME(2000) */
       client_id, SUM(montant) AS total
FROM   commandes  
WHERE  statut = 'EN_ATTENTE'  
GROUP  BY client_id;  
```

### Placement dans les instructions complexes

Pour une instruction comportant **plusieurs mots-clés `SELECT`** (une `UNION`, une requête avec sous-requêtes), le hint :

- s'applique à **l'instruction entière**, pas seulement au bloc qui le porte ;
- doit être placé **après le tout premier `SELECT`**.

```sql
-- Le hint se place sur le premier SELECT et borne toute l'UNION
SELECT /*+ MAX_EXECUTION_TIME(1500) */ id FROM t1  
UNION  
SELECT id FROM t2;  
```

---

## Portée et requêtes concernées

Le hint `MAX_EXECUTION_TIME` est volontairement restreint :

- il n'est **autorisé que pour les `SELECT` en lecture seule**. Sont considérées comme non-lecture-seule les requêtes qui appellent une fonction stockée modifiant des données comme effet de bord ;
- il est **ignoré (avec un avertissement)** lorsqu'il apparaît dans une **sous-requête** ;
- il est **ignoré** dans les `SELECT` situés à l'intérieur d'un **programme stocké** : procédures, fonctions, *triggers* et *events*.

Cette portée correspond exactement à celle du hint homonyme de MySQL, ce qui facilite la migration des applications. Pour borner *toute* requête (y compris `UPDATE`, `DELETE`, `INSERT … SELECT`), il faut recourir à la variable `max_statement_time` décrite ci-dessous, qui n'a pas cette limitation côté MariaDB.

---

## Une interruption « au mieux », pas une garantie stricte

Point essentiel à comprendre avant de s'appuyer sur ces mécanismes en production : **l'arrêt n'est pas instantané**.

L'interruption repose sur la vérification périodique d'un indicateur (« killed flag ») par le thread d'exécution. Le serveur ne contrôle le dépassement qu'à **certains points** du traitement. Concrètement :

- une requête peut s'exécuter **un peu au-delà** du délai annoncé avant d'être effectivement détectée puis stoppée ;
- si la requête entre dans une phase longue où l'indicateur n'est pas testé (certaines opérations internes au moteur de stockage, gros calculs), elle peut dépasser **significativement** la limite ;
- ce n'est donc **pas un rempart fiable** contre l'épuisement des ressources (par exemple le remplissage de l'espace temporaire). Pour cela, on combine d'autres garde-fous : `max_tmp_session_space_usage` / `max_tmp_total_space_usage` (voir §11.8), dimensionnement mémoire, et limites côté pool de connexions.

Lorsqu'une requête est effectivement avortée, l'application reçoit une **erreur de dépassement de délai** (statement timeout) et le compteur d'état `max_statement_time_exceeded` est incrémenté.

---

## La variable `max_statement_time` (mécanisme natif MariaDB)

Introduite dans MariaDB 10.1.1, `max_statement_time` est le mécanisme historique et le plus complet côté MariaDB. Caractéristiques :

- valeur exprimée **en secondes**, de type `DOUBLE` : les **délais sous-seconde** sont possibles (par exemple `0.01` = 10 ms) ;
- valeur par défaut `0` = **aucune limite** ;
- elle peut **arrêter n'importe quelle requête** (et pas seulement les `SELECT`), à l'exception des procédures stockées ;
- définissable à **plusieurs niveaux** : globale, session, par utilisateur, ou par requête.

```sql
-- Niveau session (en SECONDES) : 2 s pour toutes les requêtes de la session
SET SESSION max_statement_time = 2;

-- Niveau requête, sans modifier la session, via SET STATEMENT … FOR …
SET STATEMENT max_statement_time = 0.5 FOR
  SELECT * FROM grande_table ORDER BY col;

-- Niveau utilisateur, via GRANT (limite de ressource attachée au compte)
GRANT SELECT ON app.* TO 'reporting'@'%'
  WITH MAX_STATEMENT_TIME 5;
```

Éléments connexes :

- **Réplicas** : la variable n'affecte pas les répliques. Depuis MariaDB 10.10, `slave_max_statement_time` joue le même rôle, mais côté réplique uniquement.
- **Suivi** : la variable d'état `max_statement_time_exceeded` compte les requêtes ayant dépassé le délai ; une colonne `MAX_STATEMENT_TIME_EXCEEDED` est aussi exposée dans les tables `CLIENT_STATISTICS` et `USER_STATISTICS` d'`INFORMATION_SCHEMA`.

> **À noter :** la syntaxe `SELECT MAX_STATEMENT_TIME = N ...` (clause directement sur le `SELECT`) **n'est pas valide** dans MariaDB. Pour limiter une requête individuelle via la variable, on utilise `SET STATEMENT max_statement_time = N FOR …`. Pour une approche par hint compatible MySQL, on utilise `MAX_EXECUTION_TIME(N)`.

### ⚠️ Piège classique : millisecondes contre secondes

C'est l'erreur la plus fréquente lorsqu'on jongle entre le hint et la variable :

| Mécanisme | `… = 2` signifie | `… = 1000` signifie |
|-----------|------------------|---------------------|
| Hint `MAX_EXECUTION_TIME(N)` | 2 **millisecondes** (quasi immédiat !) | 1 **seconde** |
| Variable `max_statement_time` | 2 **secondes** | 1000 **secondes** (≈ 16 min) |

Un `MAX_EXECUTION_TIME(2)` ne laisse que 2 ms à la requête et l'avortera presque systématiquement ; pour viser 2 secondes via le hint, il faut écrire `MAX_EXECUTION_TIME(2000)`.

---

## Comportement de la valeur 0

Le `0` n'a pas la même signification selon le mécanisme :

- pour la **variable** `max_statement_time`, `0` signifie **pas de limite** ;
- pour le **hint** `MAX_EXECUTION_TIME(0)`, `0` signifie **« ignorer ce hint »** (et donc retomber sur la limite définie par la variable de session / globale), et **non** « désactiver tout délai ».

En conséquence, pour exempter ponctuellement une requête d'un délai par ailleurs imposé au niveau session, on n'écrit **pas** `MAX_EXECUTION_TIME(0)` : on emploie une valeur volontairement très élevée, ou l'on agit sur `max_statement_time`. En l'absence de hint (ou si `N = 0`), c'est la valeur de `max_statement_time` qui gouverne la requête.

---

## Compatibilité MySQL et migration

Le hint `MAX_EXECUTION_TIME` existe principalement pour **faciliter la migration depuis MySQL** (5.7+, 8.x) et l'usage de codes/ORM qui émettent déjà ce hint. Les principales différences de modèle entre les deux SGBD :

- MySQL pilote ce comportement par une **variable système** `max_execution_time`, exprimée **en millisecondes**, qui ne s'applique qu'aux `SELECT`.
- MariaDB conserve sa variable historique `max_statement_time`, exprimée **en secondes**, capable d'arrêter **tout type de requête** (hors procédures stockées).
- Le **hint** `MAX_EXECUTION_TIME(N)` (en millisecondes) est désormais commun aux deux, ce qui rend les requêtes annotées **portables**.

Concrètement, lors d'une migration MySQL → MariaDB 12.3 : les requêtes contenant `/*+ MAX_EXECUTION_TIME(N) */` fonctionnent telles quelles, mais toute configuration globale/session reposant sur la variable `max_execution_time` (ms) de MySQL doit être retranscrite vers `max_statement_time` (s) côté MariaDB — en convertissant les unités.

---

## Limites connues

- `max_statement_time` **ne fonctionne pas** dans les serveurs *embedded*.
- Dans un cluster **Galera**, `max_statement_time` ne s'applique pas aux instructions `COMMIT` (voir MDEV-18673).
- Le délai n'étant contrôlé qu'à certains points d'exécution, des requêtes bloquées dans une opération où le test n'est pas atteint ne s'arrêteront qu'au prochain point de contrôle — d'où des dépassements possibles de la valeur fixée (voir la section « interruption au mieux » ci-dessus).

---

## Quand utiliser quoi ?

- **Borne ponctuelle sur une requête `SELECT` précise, code portable MySql/MariaDB** → hint `MAX_EXECUTION_TIME(N)` (millisecondes).
- **Politique par défaut sur une session, un compte de *reporting* ou toute l'instance, sur tous types de requêtes** → variable `max_statement_time` (secondes), au niveau session, `GRANT … WITH MAX_STATEMENT_TIME`, ou globale.
- **Protéger les répliques de lecture** → `slave_max_statement_time`.
- **Limiter le coût indépendamment du temps** (par nombre de lignes examinées) → mécanisme complémentaire `LIMIT ROWS EXAMINED` (hors du périmètre de cette section).

Dans tous les cas, ces délais sont une **mesure de sécurité**, non un substitut à l'optimisation : une requête qui frôle régulièrement la limite doit d'abord être analysée (`EXPLAIN` / `ANALYZE`, §5.7) et corrigée (index, réécriture, hints d'ordre de jointure).

---

## À retenir

- `MAX_EXECUTION_TIME` est un **optimizer hint** apparu avec les *new-style hints* de la série **12.x** (12.0/12.1), présent dans **12.3 LTS**.
- Syntaxe : `SELECT /*+ MAX_EXECUTION_TIME(N) */ …`, où **`N` est en millisecondes**.
- Il ne vise que les **`SELECT` en lecture seule** ; il est **ignoré** dans les sous-requêtes et les programmes stockés.
- La variable native **`max_statement_time`** est en **secondes**, agit à plusieurs niveaux (global/session/utilisateur/requête) et peut arrêter **toute requête** sauf les procédures stockées.
- ⚠️ **Unités différentes** : millisecondes (hint) vs secondes (variable) — vérifier systématiquement.
- L'interruption est **approximative** (vérifiée par intervalles) et ne doit pas être l'unique protection contre l'épuisement des ressources.
- Pour le hint, `0` = **ignorer** (et non « pas de limite ») ; pour la variable, `0` = **pas de limite**.
