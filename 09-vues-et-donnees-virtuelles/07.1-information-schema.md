# 9.7.1 · `INFORMATION_SCHEMA`

> **Chapitre 9 — Vues et Données Virtuelles** · Niveau : Avancé  
> Version de référence : **MariaDB 12.3 LTS**

---

## Un dictionnaire de données interrogeable et standard

`INFORMATION_SCHEMA` est une base **virtuelle**, **en lecture seule** et définie par le **standard SQL** (ANSI/ISO), qui décrit la totalité de la structure d'une instance MariaDB : les bases, les tables, les colonnes, les vues, les index, les contraintes, les routines, les déclencheurs, les moteurs, les jeux de caractères, les privilèges… C'est la **face interrogeable du dictionnaire de données** : tout ce que le serveur sait de lui-même y est exposé sous forme de tables que l'on consulte avec de simples `SELECT`.

Deux propriétés découlent de sa nature standard. D'une part, elle est largement **portable** : les tables fondamentales (`TABLES`, `COLUMNS`, `VIEWS`…) existent et se nomment de la même façon dans d'autres SGBD relationnels. D'autre part, son contenu est **filtré par les privilèges** : un utilisateur n'y voit que les objets sur lesquels il détient au moins un droit — deux comptes différents obtiennent donc des résultats différents.

## Les tables essentielles

`INFORMATION_SCHEMA` compte des dizaines de tables. Les plus utiles s'organisent par finalité :

| Table | Contenu |
|-------|---------|
| `SCHEMATA` | Les bases de données (une ligne par base) |
| `TABLES` | Tables et vues : type, moteur, taille, nombre de lignes estimé, dates… |
| `COLUMNS` | Colonnes de tous les objets : type, nullabilité, valeur par défaut, clé… |
| `VIEWS` | Définition des vues : `VIEW_DEFINITION`, `IS_UPDATABLE`, `ALGORITHM`, `CHECK_OPTION`, `DEFINER`, `SECURITY_TYPE` |
| `STATISTICS` | Index : une ligne par colonne d'index, avec cardinalité et unicité |
| `TABLE_CONSTRAINTS` | Contraintes : `PRIMARY KEY`, `UNIQUE`, `FOREIGN KEY`, `CHECK` |
| `KEY_COLUMN_USAGE` | Colonnes participant aux clés et contraintes (dont les références de clés étrangères) |
| `REFERENTIAL_CONSTRAINTS` | Détail des clés étrangères : table référencée, règles `ON UPDATE`/`ON DELETE` |
| `ROUTINES` | Procédures et fonctions stockées |
| `TRIGGERS` | Déclencheurs |
| `EVENTS` | Événements planifiés |
| `PARTITIONS` | Partitions des tables |
| `ENGINES` | Moteurs de stockage disponibles et leur support |
| `PROCESSLIST` | Threads et connexions en cours (équivalent de `SHOW PROCESSLIST`) |
| `CHARACTER_SETS`, `COLLATIONS` | Jeux de caractères et collations |
| `*_PRIVILEGES` | Privilèges (`USER_`, `SCHEMA_`, `TABLE_`, `COLUMN_PRIVILEGES`) |

Dans `TABLES`, la colonne `TABLE_TYPE` distingue les `BASE TABLE` des `VIEW` (ainsi que des `SEQUENCE`, §18.1, et des `SYSTEM VIEW` propres aux espaces système).

> **Spécificité MariaDB.** Contrairement à MySQL 8, MariaDB conserve dans `INFORMATION_SCHEMA` les tables `GLOBAL_VARIABLES`, `SESSION_VARIABLES`, `GLOBAL_STATUS` et `SESSION_STATUS`. MariaDB y ajoute aussi de nombreuses tables non standard (préfixées `INNODB_`, les statistiques `userstat`, `ALL_PLUGINS`…), commodes mais **non portables**.

## `INFORMATION_SCHEMA` face aux commandes `SHOW`

Les commandes `SHOW` (`SHOW TABLES`, `SHOW COLUMNS`, `SHOW INDEX`, `SHOW CREATE TABLE`…) offrent des raccourcis pratiques — et beaucoup sont d'ailleurs implémentées **par-dessus** `INFORMATION_SCHEMA`. Mais elles renvoient un affichage figé que l'on ne peut ni filtrer finement, ni joindre, ni agréger.

`INFORMATION_SCHEMA`, lui, se manipule avec **tout le langage SQL**. Là où `SHOW TABLES` se contente de lister, une requête sur `TABLES` peut filtrer, trier, joindre et calculer :

```sql
-- L'équivalent de SHOW TABLES, mais qui devient programmable
SELECT TABLE_NAME
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'ma_base'
  AND TABLE_TYPE = 'BASE TABLE'
ORDER BY TABLE_NAME;
```

C'est cette capacité à **traiter les métadonnées comme des données** qui fait toute la puissance de l'espace.

## Exemples d'interrogation

Quelques requêtes représentatives, sur la base `ma_base` utilisée dans ce chapitre.

**Taille des tables**, triées par encombrement (données + index) :

```sql
SELECT TABLE_NAME, ENGINE,
       ROUND((DATA_LENGTH + INDEX_LENGTH) / 1024 / 1024, 2) AS taille_mo,
       TABLE_ROWS AS lignes_estimees
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'ma_base'
  AND TABLE_TYPE = 'BASE TABLE'
ORDER BY (DATA_LENGTH + INDEX_LENGTH) DESC;
```

**Structure d'une table**, colonne par colonne :

```sql
SELECT COLUMN_NAME, COLUMN_TYPE, IS_NULLABLE, COLUMN_DEFAULT, COLUMN_KEY
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_SCHEMA = 'ma_base'
  AND TABLE_NAME = 'employes'
ORDER BY ORDINAL_POSITION;
```

**Clés étrangères pointant vers une table** — précieux avant de modifier ou supprimer celle-ci :

```sql
SELECT TABLE_NAME, COLUMN_NAME, CONSTRAINT_NAME,
       REFERENCED_TABLE_NAME, REFERENCED_COLUMN_NAME
FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
WHERE TABLE_SCHEMA = 'ma_base'
  AND REFERENCED_TABLE_NAME = 'departements';
```

**Index d'une table**, avec leur composition :

```sql
SELECT INDEX_NAME, SEQ_IN_INDEX, COLUMN_NAME, NON_UNIQUE, CARDINALITY
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = 'ma_base'
  AND TABLE_NAME = 'employes'
ORDER BY INDEX_NAME, SEQ_IN_INDEX;
```

**Inventaire des vues** — le retour du fil rouge de ce chapitre :

```sql
SELECT TABLE_NAME, IS_UPDATABLE, ALGORITHM, CHECK_OPTION, SECURITY_TYPE
FROM INFORMATION_SCHEMA.VIEWS
WHERE TABLE_SCHEMA = 'ma_base';
```

## Générer du SQL avec `INFORMATION_SCHEMA`

Un usage particulièrement fécond consiste à faire produire par `INFORMATION_SCHEMA` les **instructions SQL** à exécuter — « du SQL qui génère du SQL ». Plutôt que d'écrire à la main une commande par table, on la fabrique pour toutes en une requête :

```sql
SELECT CONCAT('ANALYZE TABLE ', TABLE_SCHEMA, '.', TABLE_NAME, ';') AS commande
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'ma_base'
  AND TABLE_TYPE = 'BASE TABLE';
```

Le résultat est une liste d'instructions `ANALYZE TABLE …` prêtes à être copiées et exécutées. Le même principe sert à générer des `OPTIMIZE TABLE`, des `DROP`, des octrois de privilèges, des recherches de colonnes à modifier en masse, etc. C'est l'un des outils les plus efficaces de l'administration courante et de l'automatisation.

## Caractéristiques et précautions

Quelques points à garder à l'esprit pour un usage sûr et performant :

- **Lecture seule.** `INFORMATION_SCHEMA` ne se modifie pas : toute tentative d'`INSERT`, d'`UPDATE` ou de `DELETE` échoue. On agit sur la structure par les instructions DDL habituelles (`CREATE`, `ALTER`, `GRANT`…), et l'espace en reflète aussitôt l'état.
- **Résultats dépendants des privilèges.** Puisque le contenu est filtré par les droits de l'utilisateur, un compte peu privilégié obtient une vue **incomplète** — un point à ne pas négliger dans les scripts d'automatisation.
- **Des estimations, pas des comptages.** Pour les tables InnoDB, `TABLE_ROWS` (comme certaines mesures de taille) est une **valeur estimée**, issue des statistiques de l'optimiseur. Pour un décompte exact, il faut recourir à `COUNT(*)`.
- **Attention au coût.** Interroger `TABLES` ou `COLUMNS` **sans filtre** sur une instance comportant des milliers de tables peut être lent, car le serveur doit parcourir de nombreuses définitions. La bonne pratique est de **toujours restreindre** par `TABLE_SCHEMA` (et `TABLE_NAME` si possible).
- **Standard contre extensions.** Les tables fondamentales sont portables ; les ajouts propres à MariaDB (`INNODB_*`, `userstat`…) ne le sont pas. À considérer si le code doit fonctionner sur plusieurs SGBD.

## En résumé

`INFORMATION_SCHEMA` est le **dictionnaire de données** de MariaDB : une base virtuelle, en lecture seule et conforme au standard SQL, qui décrit toute la structure de l'instance et se laisse interroger avec la pleine puissance du langage. Plus riche que les commandes `SHOW`, il permet de filtrer, joindre, agréger les métadonnées — et même de **générer du SQL**. Ses pièges sont connus : résultats filtrés par les privilèges, statistiques estimées, et requêtes à toujours restreindre par schéma pour rester performantes.

Là où `INFORMATION_SCHEMA` décrit la **structure** du serveur, l'espace suivant en mesure le **comportement** à l'exécution. C'est l'objet du **§9.7.2 — `PERFORMANCE_SCHEMA`**.
