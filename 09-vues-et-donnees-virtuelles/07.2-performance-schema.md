# 9.7.2 · `PERFORMANCE_SCHEMA`

> **Chapitre 9 — Vues et Données Virtuelles** · Niveau : Avancé  
> Version de référence : **MariaDB 12.3 LTS**

---

## Mesurer l'exécution du serveur

Là où `INFORMATION_SCHEMA` décrit la **structure** de l'instance (§9.7.1), `PERFORMANCE_SCHEMA` en mesure le **comportement à l'exécution**. C'est une facilité d'**instrumentation** : des sondes intégrées au cœur du serveur enregistrent en continu ce qui s'y passe — quelles requêtes s'exécutent et combien de temps, où le temps est passé en attente (entrées/sorties, verrous, mutex), quelles étapes traverse chaque requête, comment la mémoire est consommée.

Contrairement à `INFORMATION_SCHEMA`, cet espace est **spécifique à MariaDB/MySQL** (il ne fait pas partie du standard SQL). Ses tables sont servies par un moteur de stockage dédié, **en mémoire**, et alimentées par l'instrumentation. On les **lit** comme des tables ordinaires pour analyser les métriques ; quelques tables de configuration sont, elles, **modifiables** pour piloter ce que l'on collecte.

## Activer `PERFORMANCE_SCHEMA`

Première particularité à connaître : chez MariaDB, `PERFORMANCE_SCHEMA` est **désactivé par défaut** (pour économiser la mémoire), alors qu'il est actif par défaut chez MySQL. Son activation se fait par la variable système `performance_schema`, qui n'est **pas modifiable à chaud** : il faut la déclarer dans la configuration et **redémarrer** le serveur.

```ini
# my.cnf
[mariadb]
performance_schema = ON
```

Une fois l'instrumentation globalement active, on en **affine la portée** via des tables de configuration (les tables `setup_*`), modifiables à l'exécution, pour n'activer que les sondes et les destinations utiles (voir plus bas).

## Comment c'est organisé : instruments, consommateurs, événements

Trois notions structurent l'espace :

- les **instruments** (*instruments*) sont les points de mesure dans le code du serveur. Leurs noms sont **hiérarchiques**, séparés par des `/` : par exemple `statement/sql/select`, `wait/io/file/innodb/innodb_data_file`, ou `stage/sql/Sorting result`. On peut activer ou désactiver des familles entières par motif.
- les **consommateurs** (*consumers*) sont les destinations qui stockent les événements collectés. On choisit, par exemple, de conserver l'historique des requêtes ou seulement leurs agrégats.
- les **événements** (*events*) sont les mesures enregistrées. Ils se déclinent en plusieurs **types** — attentes (*waits*), étapes (*stages*), requêtes (*statements*), transactions — et, pour chacun, en plusieurs niveaux de détail : tables `_current` (en cours), `_history` / `_history_long` (historique récent), et surtout les tables de **synthèse** (`_summary_*`), qui pré-agrègent les mesures et constituent l'essentiel de l'usage pratique.

## Les tables les plus utiles

| Table | Contenu |
|-------|---------|
| `events_statements_summary_by_digest` | **La plus précieuse** : statistiques agrégées par requête normalisée (nombre d'exécutions, latence totale/moyenne/max, lignes examinées/renvoyées, scans complets…) |
| `events_waits_summary_global_by_event_name` | Où le serveur passe son temps en **attente** (E/S, verrous, mutex) |
| `file_summary_by_instance` | Statistiques d'**entrées/sorties** par fichier |
| `table_io_waits_summary_by_table` | Activité d'E/S **par table** |
| `events_statements_current` / `_history` | Requêtes récentes, **par thread** |
| `threads` | Tous les threads du serveur, et s'ils sont instrumentés |
| `metadata_locks` | Verrous de **métadonnées** (utile pour diagnostiquer des `ALTER` ou des requêtes bloquées) |
| `setup_instruments`, `setup_consumers` | **Configuration** de ce qui est collecté |

## Exemples d'interrogation

Avant tout, **une unité à retenir** : les colonnes de durée (`*_TIMER_WAIT`) sont exprimées en **picosecondes** (10⁻¹² seconde). Pour obtenir des secondes, on divise par 10¹² ; pour des millisecondes, par 10⁹.

**Les requêtes les plus coûteuses**, classées par temps total cumulé — l'équivalent vivant d'un `pt-query-digest` (§15.7) :

```sql
SELECT DIGEST_TEXT,
       COUNT_STAR                       AS executions,
       ROUND(SUM_TIMER_WAIT / 1e12, 3)  AS total_sec,
       ROUND(AVG_TIMER_WAIT / 1e9, 3)   AS moyenne_ms,
       SUM_ROWS_EXAMINED, SUM_ROWS_SENT
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;
```

**Où le serveur attend le plus** (verrous, E/S…) :

```sql
SELECT EVENT_NAME,
       COUNT_STAR,
       ROUND(SUM_TIMER_WAIT / 1e12, 3) AS total_sec
FROM performance_schema.events_waits_summary_global_by_event_name
WHERE SUM_TIMER_WAIT > 0
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;
```

**Les requêtes qui n'utilisent pas d'index** — pistes d'optimisation prioritaires (en lien direct avec §5 et §9.6) :

```sql
SELECT DIGEST_TEXT, COUNT_STAR, SUM_NO_INDEX_USED, SUM_SELECT_SCAN
FROM performance_schema.events_statements_summary_by_digest
WHERE SUM_NO_INDEX_USED > 0
ORDER BY SUM_NO_INDEX_USED DESC
LIMIT 10;
```

## Configurer ce qui est collecté

Les tables `setup_*` se modifient **à chaud**, par exemple pour activer l'instrumentation des requêtes et leur conservation :

```sql
UPDATE performance_schema.setup_instruments
SET ENABLED = 'YES', TIMED = 'YES'
WHERE NAME LIKE 'statement/%';

UPDATE performance_schema.setup_consumers
SET ENABLED = 'YES'
WHERE NAME LIKE 'events_statements_%';
```

Attention : ces réglages sont **perdus au redémarrage**. Pour les rendre permanents, on emploie les options de démarrage correspondantes dans `my.cnf` (`performance-schema-instrument`, `performance-schema-consumer-…`).

## Le schéma `sys` : la version lisible

Interroger directement `PERFORMANCE_SCHEMA` est puissant mais peu confortable — noms de colonnes verbeux, durées en picosecondes, agrégats bruts. Le schéma **`sys`** (§15.8) répond à ce besoin : c'est un ensemble de **vues** bâties par-dessus `PERFORMANCE_SCHEMA` (et `INFORMATION_SCHEMA`) qui présentent les mêmes informations sous une forme directement exploitable — latences formatées en unités lisibles, vues prêtes à l'emploi comme l'analyse des requêtes ou des entrées/sorties. C'est, du reste, une belle illustration du chapitre : `sys` n'est rien d'autre qu'une **couche de vues** posée sur des données d'instrumentation.

## Caractéristiques et précautions

- **Activation au démarrage.** `PERFORMANCE_SCHEMA` doit être activé via la configuration puis un redémarrage ; il consomme de la mémoire, proportionnellement aux sondes activées.
- **Durées en picosecondes.** Toujours convertir (ou s'appuyer sur `sys`) pour obtenir des valeurs interprétables.
- **Un coût, faible mais réel.** L'instrumentation n'est pas gratuite : plus on active d'instruments, plus l'empreinte mémoire et le surcoût augmentent. On active donc **sélectivement** ce dont on a besoin.
- **Des données volatiles.** Tout étant en mémoire, les mesures sont **perdues au redémarrage** et ne couvrent que l'activité depuis celui-ci. On peut par ailleurs **réinitialiser** une table de synthèse avec `TRUNCATE TABLE` (par exemple pour repartir d'une base propre avant un test).
- **Configuration à chaud vs permanente.** Les écritures sur les tables `setup_*` ne survivent pas au redémarrage ; seules les options de `my.cnf` sont durables.
- **Une implémentation plus restreinte que celle de MySQL.** Le `PERFORMANCE_SCHEMA` de MariaDB dérive d'une version antérieure et propose **moins de tables et d'instruments** que les versions récentes de MySQL ; certaines analyses (verrous de données, instrumentation mémoire étendue) y sont plus limitées.
- **Privilèges.** La lecture requiert le droit `SELECT` sur `performance_schema`, et la configuration le droit `UPDATE` sur les tables `setup_*` — droits généralement réservés à un compte de supervision dédié.

## En résumé

`PERFORMANCE_SCHEMA` est l'espace d'**instrumentation** de MariaDB : spécifique au moteur, en mémoire, alimenté par des sondes, il mesure *comment* le serveur s'exécute — requêtes, attentes, étapes, mémoire. Désactivé par défaut, il s'active au démarrage puis s'affine via les tables `setup_*`. Sa table phare, `events_statements_summary_by_digest`, identifie les requêtes les plus coûteuses, tandis que le schéma `sys` en offre une lecture confortable. Ses pièges — picosecondes, volatilité, surcoût, implémentation plus restreinte que MySQL — se maîtrisent aisément une fois connus.

Restent les tables que le serveur utilise réellement pour fonctionner — comptes, privilèges, configuration : la base `mysql`, dernier espace système de ce chapitre, examinée au **§9.7.3 — Tables système `mysql`**.
