# 17.3.3 Sequelize (Node.js)

**Sequelize** est un ORM Node.js basé sur les promesses, compatible avec MariaDB, MySQL, PostgreSQL, SQLite, SQL Server et d'autres. Il offre la gestion des transactions, les relations, le chargement anticipé et paresseux, et la réplication en lecture. Il relève du patron **Active Record** (cf. §17.3) et s'inscrit naturellement dans le modèle asynchrone de Node.js (`async`/`await`, §17.1.4). Sous le capot, il s'appuie sur les connecteurs vus au §17.1.4 et intègre son propre **pool** (§17.2.1).

---

## Version : v6 stable, v7 en alpha

Repère important sur les versions : **Sequelize v6 demeure la version stable**, la documentation de **v7 restant à l'état d'alpha** (non publiée). La v7 introduit une refonte **TypeScript-first** avec décorateurs, le découpage des dialectes en paquets distincts et un nouveau paquet `@sequelize/core`. Tant qu'elle n'est pas stabilisée, **v6 reste le choix de production** : les exemples ci-dessous suivent son API (paquet `sequelize`).

---

## Initialiser Sequelize

On installe Sequelize **et** un pilote (`npm install sequelize` puis `npm install mariadb` ou `mysql2`). L'instance `Sequelize` configure la connexion, le dialecte et le pool :

```javascript
import { Sequelize } from 'sequelize';

const sequelize = new Sequelize('boutique', 'app_user', process.env.DB_PASSWORD, {
  host: 'localhost',
  port: 3306,
  dialect: 'mariadb',     // utilise le connecteur "mariadb" (§17.1.4) ; 'mysql' → mysql2
  pool: { max: 10, min: 0, acquire: 30000, idle: 10000 },  // §17.2.1
  logging: console.log,   // affiche le SQL généré (utile pour traquer le N+1)
});

await sequelize.authenticate();   // vérifie réellement la connexion
```

Le choix `dialect: 'mariadb'` s'appuie sur le connecteur officiel ; `'mysql'` utilise mysql2. Les deux conviennent à MariaDB (§17.1.4).

## Définir un modèle

Un modèle se déclare soit avec `sequelize.define(...)`, soit par classe avec `Model.init(...)` (illustré ici) :

```javascript
import { DataTypes, Model } from 'sequelize';

class Produit extends Model {}

Produit.init({
  id:   { type: DataTypes.INTEGER, primaryKey: true, autoIncrement: true },
  nom:  { type: DataTypes.STRING(200), allowNull: false },
  prix: { type: DataTypes.DECIMAL(10, 2), allowNull: false },
}, {
  sequelize,
  modelName: 'produit',
  tableName: 'produits',
});
```

Les `DataTypes` (`STRING`, `INTEGER`, `DECIMAL`, `BOOLEAN`, `DATE`, `TEXT`, `JSON`…) déterminent les types de colonnes ; `allowNull` contrôle la nullabilité.

---

## Active Record en pratique

Conformément au patron Active Record, les **instances savent se sauvegarder** et se supprimer, et le modèle expose des méthodes statiques :

```javascript
// Créer (insertion)
const clavier = await Produit.create({ nom: 'Clavier', prix: 49.90 });

// Lire par clé primaire
const p = await Produit.findByPk(42);

// Modifier une instance, puis la persister
p.prix = 39.90;
await p.save();

// Supprimer
await p.destroy();
```

## Interroger

`findAll` accepte des clauses `where`, `order`, `limit`, `attributes`… Les opérateurs de comparaison passent par le symbole `Op` :

```javascript
import { Op } from 'sequelize';

const produits = await Produit.findAll({
  where: { prix: { [Op.lt]: prixMax } },   // prix < prixMax
  order: [['nom', 'ASC']],
  limit: 10,
});
```

## Associations et chargement

Les relations se déclarent par `belongsTo`, `hasMany`, `hasOne` et `belongsToMany` (avec `through` pour le plusieurs-à-plusieurs) :

```javascript
Produit.belongsTo(Categorie, { foreignKey: 'categorie_id' });
Categorie.hasMany(Produit, { foreignKey: 'categorie_id' });
```

Par défaut, les associations sont chargées **paresseusement** (via des méthodes `getCategorie()`, `getProduits()`…). Le **chargement anticipé** se demande avec `include`, qui produit une jointure.

---

## Le problème N+1 dans Sequelize

Le piège N+1 (§17.3) survient lorsqu'on parcourt une liste puis qu'on accède à une association en boucle (par exemple `produit.getCategorie()` pour chaque produit), déclenchant une requête par élément. La parade est d'utiliser **`include`** pour rapatrier les données liées en une seule requête :

```javascript
// Avec include : jointure, en une requête, au lieu de N+1
const produits = await Produit.findAll({
  include: [{ model: Categorie }],
});
```

Comme partout, activer la journalisation du SQL (`logging`) permet de repérer ces situations — le réflexe recommandé au §17.4.

## Transactions

Sequelize propose deux modes. Le mode **géré** (le plus sûr) valide ou annule automatiquement selon l'issue du *callback* :

```javascript
await sequelize.transaction(async (t) => {
  await Compte.increment({ solde: -montant }, { where: { id: source },      transaction: t });
  await Compte.increment({ solde:  montant }, { where: { id: destination }, transaction: t });
});
// commit automatique si succès, rollback automatique en cas d'exception
```

Le mode **non géré** fournit un objet transaction sur lequel on appelle explicitement `t.commit()` ou `t.rollback()`.

---

## Pool de connexions

Le pool est intégré et se configure dans l'option `pool` de l'instance (`max`, `min`, `acquire` — délai d'acquisition, `idle` — durée d'inactivité). Ces leviers correspondent à ceux du §17.2.1, et le pool repose sur le connecteur sous-jacent (§17.1.4).

---

## Migrations et synchronisation

Deux approches coexistent pour le schéma :

- **`sequelize.sync()`** crée les tables manquantes à partir des modèles. À manier avec prudence :

  ```javascript
  await sequelize.sync();                  // crée les tables absentes
  // await sequelize.sync({ force: true }); // ⚠️ SUPPRIME puis recrée — jamais en production
  // await sequelize.sync({ alter: true }); // ⚠️ tente d'aligner le schéma — risqué en production
  ```

  > ⚠️ Comme pour `ddl-auto` côté Hibernate (§17.3.1), `force` et `alter` ne doivent **pas** servir en production (perte de données, évolutions non maîtrisées).

- **Migrations versionnées** via **`sequelize-cli`** (`db:migrate`). À la différence d'Alembic (§17.3.2) ou d'EF Core, Sequelize **ne dérive pas automatiquement** les migrations des modèles : on les écrit explicitement. C'est l'approche à privilégier en production (§17.5, chap. 16.8).

---

## Spécificités MariaDB

- **Dialecte** — `'mariadb'` (connecteur officiel) ou `'mysql'` (mysql2), au choix (§17.1.4) ; options fines via `dialectOptions`.
- **Grands entiers** — le piège des `BIGINT` au-delà de 2⁵³−1 (§17.1.4) reste à surveiller, le comportement dépendant du connecteur sous-jacent.
- **SQL brut** — la porte de sortie (cf. §17.3) est `sequelize.query('SELECT …', { type: QueryTypes.SELECT })`, pour l'analytique ou les fonctionnalités MariaDB non couvertes par l'ORM (chap. 18).

---

## Ce qu'il faut retenir

- **Sequelize** est un ORM Node.js à promesses, de type **Active Record** (§17.3), reposant sur mysql2/mariadb (§17.1.4) avec **pool intégré** (§17.2.1).
- En 2026, **v6 est stable** et reste le choix de production ; **v7** (TS-first, décorateurs) demeure en alpha.
- Modèles via `Model.init`/`define` et `DataTypes` ; instances **Active Record** (`create`, `findByPk`, `save`, `destroy`).
- Associations `belongsTo`/`hasMany`/`belongsToMany` ; corriger le **N+1** avec **`include`** (jointure) et surveiller le SQL (`logging`, §17.4).
- Transactions **gérées** (commit/rollback automatiques) ou non gérées.
- **`sync({ force/alter })` jamais en production** : préférer les **migrations** `sequelize-cli` (écrites à la main), §17.5.
