# 17.3.4 Prisma

**Prisma** est un ORM « nouvelle génération » pour Node.js et TypeScript. Il se distingue nettement des ORM précédents : ce n'est pas un ORM à classes modèles (ni Active Record, ni Data Mapper au sens classique), mais une approche **schema-first** où l'on déclare un schéma et où Prisma **génère un client entièrement typé** (cf. §17.3). Il s'organise autour de trois éléments :

- **Prisma Schema** — un fichier déclaratif (`schema.prisma`) décrivant la source de données et le modèle ;
- **Prisma Client** — le client typé généré à partir du schéma (`prisma.produit.findMany()`) ;
- **Prisma Migrate** — l'outil de migration de schéma.

---

## Une architecture singulière (Prisma 7)

Prisma a longtemps reposé sur un **moteur de requête écrit en Rust** (un binaire externe). Cela a changé : Prisma 7, publié le 19 novembre 2025, abandonne ce moteur de requête Rust au profit d'un *Query Compiler* livré sous forme de **module WebAssembly (WASM)**, piloté par un *runtime* **TypeScript**. Prisma 7.0.0 en a fait le mode par défaut, supprimant le binaire Rust externe ; le client produit des bundles environ 90 % plus petits, des requêtes jusqu'à 3 fois plus rapides et une vérification de types plus rapide. Cette architecture « sans Rust » est devenue généralement disponible à partir de la v6.16.0 pour PostgreSQL, CockroachDB, Neon, MySQL, PlanetScale, SQLite, D1 et MS SQL Server.

Concrètement, cela apporte un déploiement plus simple (pas de binaire natif) et de meilleures performances en environnements *serverless*/edge (Cloudflare Workers, Vercel Edge, Bun, Deno). Le nouveau générateur **`prisma-client`** est *ESM-first* et produit du **code sous le contrôle du développeur** (dans un dossier de sortie choisi, et non plus dans `node_modules`). Avec ce moteur, on configure un **adaptateur de pilote** (*driver adapter*) qui s'appuie sur un pilote JavaScript.

## Le schéma Prisma

Tout part de `schema.prisma`, qui réunit le générateur, la source de données et les modèles :

```prisma
generator client {
  provider = "prisma-client"        // générateur ESM-first (Prisma 7)
  output   = "./generated/prisma"   // code généré sous contrôle du développeur
}

datasource db {
  provider = "mysql"                // ⚠️ "mysql" couvre MySQL ET MariaDB
  // Prisma 7 : plus de `url` ici — la connexion est fournie par le driver adapter
  // (passé à PrismaClient) ; l'URL des migrations se déclare dans prisma.config.ts
}

model Produit {
  id          Int       @id @default(autoincrement())
  nom         String    @db.VarChar(200)
  prix        Decimal   @db.Decimal(10, 2)
  description String?                       // le ? rend la colonne NULL-able
  categorieId Int       @map("categorie_id")
  categorie   Categorie @relation(fields: [categorieId], references: [id])

  @@map("produits")
}

model Categorie {
  id       Int       @id @default(autoincrement())
  nom      String
  produits Produit[]
}
```

> ⚠️ **Point MariaDB essentiel** : il n'existe **pas de provider `mariadb`**. On utilise **`provider = "mysql"`**, qui couvre MySQL comme MariaDB. Sous **Prisma 7**, la chaîne de connexion (de la forme `mysql://app_user:motdepasse@localhost:3306/boutique`) **ne se déclare plus dans le bloc `datasource`** : elle est fournie au **driver adapter** passé à `PrismaClient` (et, pour les migrations, via `prisma.config.ts`).

Les annotations `@id`, `@default(autoincrement())`, `@relation`, `@map`/`@@map` (correspondance attribut↔colonne et modèle↔table) et `@db.…` (type natif précis) déclarent le mapping.

## Générer et migrer

Le flux de travail Prisma comporte deux commandes clés :

```bash
npx prisma generate        # (re)génère le client typé à partir du schéma
npx prisma migrate dev     # crée et applique une migration (développement)
npx prisma migrate deploy  # applique les migrations en attente (production)
```

`migrate dev` compare le schéma à l'état de la base et génère le SQL de migration. Particularité à connaître : **Prisma ne génère pas de migrations « descendantes » (rollback) automatiques** — pour revenir en arrière, on écrit une nouvelle migration qui annule le changement. Prisma Migrate s'inscrit dans la gestion des migrations (§17.5, chap. 16.8).

## Le client typé

Le client généré expose une API entièrement typée, par modèle. Les opérations CRUD : `findUnique`, `findMany`, `create`, `update`, `delete`, `upsert`, `createMany`.

```typescript
import { PrismaClient } from './generated/prisma/client';
import { PrismaMariaDb } from '@prisma/adapter-mariadb';   // driver adapter (Prisma 7)

// Prisma 7 : la connexion passe par le driver adapter, pas par le schéma
const adapter = new PrismaMariaDb({
  host: 'localhost', user: 'app_user',
  password: process.env.DB_PASSWORD, database: 'boutique',
});
const prisma = new PrismaClient({ adapter });

// Lecture par clé primaire (résultat typé)
const p = await prisma.produit.findUnique({ where: { id: 42 } });

// Liste : filtre, tri, pagination, sélection de colonnes
const produits = await prisma.produit.findMany({
  where:   { prix: { lt: prixMax } },     // prix < prixMax
  orderBy: { nom: 'asc' },
  take:    10,
  select:  { id: true, nom: true, prix: true },
});

// Création (avec rattachement d'une relation)
const clavier = await prisma.produit.create({
  data: { nom: 'Clavier', prix: 49.90, categorie: { connect: { id: 1 } } },
});

// Mise à jour / suppression
await prisma.produit.update({ where: { id: 42 }, data: { prix: 39.90 } });
await prisma.produit.delete({ where: { id: 42 } });
```

Le **typage de bout en bout** est l'atout majeur de Prisma : les modèles, filtres et résultats sont vérifiés à la compilation, et l'autocomplétion guide l'écriture.

## Relations et chargement

Les relations, déclarées dans le schéma via `@relation`, se chargent avec **`include`** (lecture imbriquée) ou se restreignent avec `select` :

```typescript
const produits = await prisma.produit.findMany({
  include: { categorie: true },   // charge la catégorie de chaque produit
});
```

---

## Le problème N+1 dans Prisma

Prisma résiste assez bien au N+1 (§17.3) **par construction** : lorsqu'on utilise `include`, il ne génère pas une requête par ligne, mais un **nombre constant de requêtes** quelle que soit la taille du résultat (jointure réalisée au niveau de l'application). Il offre en outre le contrôle de la **stratégie de chargement** des relations.

L'option `relationLoadStrategy` (en *Preview*, à activer via le drapeau `relationJoins`) permet de choisir entre deux stratégies : `query` (plusieurs requêtes au niveau applicatif, le **défaut**) et `join` (une seule requête avec jointure au niveau de la base) :

```typescript
const produits = await prisma.produit.findMany({
  relationLoadStrategy: 'join',   // 'query' (défaut) = requêtes séparées
  include: { categorie: true },
});
```

> ⚠️ **Point MariaDB essentiel.** Cette option n'est disponible que sur **PostgreSQL, CockroachDB et MySQL — pas (encore) sur MariaDB**. Bien que MariaDB utilise le provider `mysql`, la stratégie `join` s'appuie sur des constructions SQL propres à chaque moteur (sous-requêtes corrélées pour MySQL, *lateral joins* pour PostgreSQL) que MariaDB n'implémente pas de la même manière ; une prise en charge dédiée fait l'objet d'une demande ouverte côté Prisma. Sur MariaDB, Prisma s'en tient donc à la stratégie **`query`** — qui, via `include`, émet déjà un **nombre constant de requêtes** (et non une par ligne) et résiste ainsi au N+1. Là où elle est disponible, la stratégie `join` est souvent la plus efficace. Comme toujours, observer les requêtes émises reste recommandé (§17.4).

## SQL brut

Pour l'analytique ou les fonctionnalités MariaDB non couvertes par le schéma, la porte de sortie (cf. §17.3) est **`$queryRaw`** (et `$executeRaw`), qui paramètre automatiquement les valeurs interpolées :

```typescript
const min = 10;
const rows = await prisma.$queryRaw`
  SELECT categorie_id, COUNT(*) AS total
  FROM produits WHERE prix > ${min}
  GROUP BY categorie_id
`;
```

Les valeurs `${…}` sont liées en tant que paramètres (protection contre les injections, §17.8). La variante `$queryRawUnsafe` permet du SQL dynamique mais perd cette garantie : à réserver aux cas maîtrisés.

---

## Spécificités MariaDB

- **Provider** — **`"mysql"`** (il n'y a pas de provider `mariadb`) ; chaîne `mysql://…` dans `DATABASE_URL`.
- **Adaptateur de pilote** — avec le moteur sans Rust (Prisma 7), on configure un *driver adapter* MariaDB (par exemple `@prisma/adapter-mariadb`), qui s'appuie sur un pilote JavaScript (§17.1.4).
- **Types natifs** — précisés par les annotations `@db.…` (`@db.VarChar`, `@db.Decimal`, etc.).
- **Fonctionnalités non modélisées** — séquences, tables temporelles, VECTOR (chap. 18) s'atteignent via `$queryRaw`.

---

## Ce qu'il faut retenir

- Prisma est un ORM **schema-first** à **client généré typé** (§17.3), pour Node.js/TypeScript : Schema + Client + Migrate.
- Depuis **Prisma 7 (nov. 2025)**, le moteur Rust est supprimé au profit d'un *Query Compiler* **TypeScript/WASM** (bundles plus légers, requêtes plus rapides, meilleur support *serverless*/edge) ; générateur `prisma-client` *ESM-first* + adaptateur de pilote.
- **Point MariaDB clé** : provider **`"mysql"`** (pas de `mariadb`), URL `mysql://…` dans `DATABASE_URL`.
- Client **entièrement typé** (`findMany`, `create`, `update`…) ; relations via **`include`**.
- Prisma **limite le N+1 par construction** ; `relationLoadStrategy` (`join`/`query`) règle la stratégie (MySQL pris en charge) ; surveiller le SQL (§17.4).
- **`$queryRaw`** (paramétré) pour le SQL brut ; migrations **forward-only** via Prisma Migrate (§17.5, chap. 16.8).
