# 17.1.6 .NET : MySqlConnector, MySql.Data, ADO.NET

Dans l'écosystème .NET, l'accès aux bases de données relationnelles repose sur **ADO.NET**, la couche d'accès standard de la plateforme. ADO.NET définit un ensemble d'interfaces (`DbConnection`, `DbCommand`, `DbDataReader`, `DbParameter`, `DbTransaction`) que chaque fournisseur implémente. Pour MariaDB, le fournisseur de référence est **MySqlConnector**, écrit entièrement en C# (famille des implémentations natives au langage, cf. §17.1).

Cette section traite l'accès via ADO.NET. L'usage d'**Entity Framework Core**, l'ORM de .NET, est présenté au §17.3.5.

---

## Quel connecteur choisir ?

Le paysage .NET comporte plusieurs fournisseurs, ce qui prête parfois à confusion :

- **MySqlConnector** — fournisseur ADO.NET open source pour MySQL, MariaDB, Amazon Aurora, Azure Database for MySQL et autres bases compatibles. C'est un fournisseur ADO.NET « véritablement asynchrone » écrit entièrement en C#, prenant en charge MySQL Server, MariaDB et les services compatibles. C'est le choix recommandé. Fait important : la documentation de MariaDB désigne explicitement MySqlConnector comme « MariaDB Connector/NET », un fournisseur de données ADO.NET permettant aux applications .NET de se connecter à MariaDB et MySQL. Son espace de noms est `MySqlConnector`.
- **MySql.Data** — le connecteur officiel d'Oracle (*MySQL Connector/NET*), espace de noms `MySql.Data.MySqlClient`. Il fonctionne avec MariaDB, mais ses méthodes asynchrones ont longtemps été implémentées de façon bloquante, là où MySqlConnector offre un véritable asynchronisme.

> 💡 **À propos de « MariaDB.Data »** : il n'existe pas aujourd'hui de paquet `MariaDB.Data` distinct et largement maintenu. Le connecteur .NET officiel de MariaDB est **MySqlConnector** ; c'est donc lui qu'on utilise, avec l'espace de noms `MySqlConnector`. Noter au passage que MySqlConnector et MySql.Data exposent tous deux des classes nommées `MySqlConnection`, `MySqlCommand`, etc. : il ne faut référencer **qu'un seul** des deux dans un projet pour éviter les collisions.

## Installer MySqlConnector

```bash
dotnet add package MySqlConnector
```

Ou via une référence de paquet dans le fichier `.csproj` :

```xml
<PackageReference Include="MySqlConnector" Version="2.x.x" /> <!-- dernière version stable -->
```

## La chaîne de connexion

Comme toute chaîne ADO.NET, elle prend la forme d'une suite de paires clé–valeur :

```
Server=localhost;Port=3306;Database=boutique;Uid=app_user;Pwd=...;SslMode=Required;CharacterSet=utf8mb4;
```

La chaîne contient des secrets : elle ne doit jamais être codée en dur, mais lue depuis la configuration ou un gestionnaire de secrets.

## Se connecter et lire (asynchrone)

.NET privilégie fortement le modèle `async`/`await`, et MySqlConnector y répond pleinement. Les objets de connexion et de commande sont jetables : l'instruction `using` est recommandée car elle garantit la libération correcte de la connexion, même en cas d'erreur.

```csharp
using MySqlConnector;

await using var conn = new MySqlConnection(connectionString);
await conn.OpenAsync();

await using var cmd = new MySqlCommand(
    "SELECT id, nom, prix FROM produits WHERE categorie = @cat AND prix < @max",
    conn);
cmd.Parameters.AddWithValue("@cat", categorie);
cmd.Parameters.AddWithValue("@max", prixMax);

await using var reader = await cmd.ExecuteReaderAsync();
while (await reader.ReadAsync())
{
    var nom  = reader.GetString(reader.GetOrdinal("nom"));
    var prix = reader.GetDecimal(reader.GetOrdinal("prix"));
    Console.WriteLine($"{nom} : {prix:F2} €");
}
```

Le marqueur de paramètre est **nommé, préfixé par `@`** (`@cat`, `@max`). `AddWithValue` est commode, mais pour un contrôle précis du type SQL on peut construire des `MySqlParameter` typés. Il faut toujours utiliser des requêtes paramétrées pour prévenir les injections SQL (§17.8, §17.9).

## Écrire

Pour les opérations sans jeu de résultats (`INSERT`, `UPDATE`, `DELETE`), on utilise `ExecuteNonQueryAsync`. L'identifiant auto-incrémenté généré se lit via `LastInsertedId` :

```csharp
await using var cmd = new MySqlCommand(
    "INSERT INTO produits (nom, prix, categorie) VALUES (@nom, @prix, @cat)",
    conn);
cmd.Parameters.AddWithValue("@nom", "Clavier");
cmd.Parameters.AddWithValue("@prix", 49.90m);
cmd.Parameters.AddWithValue("@cat", "peripheriques");

int lignes      = await cmd.ExecuteNonQueryAsync();
long nouvelId   = cmd.LastInsertedId;
```

Pour récupérer une valeur unique (un agrégat, par exemple), `ExecuteScalarAsync` est plus direct :

```csharp
await using var cmd = new MySqlCommand("SELECT COUNT(*) FROM produits", conn);
long total = Convert.ToInt64(await cmd.ExecuteScalarAsync());
```

## Pooling intégré

Point essentiel en ADO.NET : le **pooling de connexions est intégré et activé par défaut**. On ne gère pas le pool manuellement — on ouvre une connexion, on l'utilise, puis on la libère (via `using`/`await using`), et le fournisseur recycle la connexion physique en arrière-plan. Le comportement se règle dans la chaîne de connexion :

```
...;Pooling=true;MinimumPoolSize=0;MaximumPoolSize=100;ConnectionIdleTimeout=180;
```

Ce modèle explique pourquoi il est non seulement permis mais **recommandé** d'ouvrir et de fermer fréquemment les connexions : la fermeture les rend au pool plutôt que de les détruire. Les principes du *pooling* sont développés au §17.2.

## Transactions

```csharp
await using var conn = new MySqlConnection(connectionString);
await conn.OpenAsync();
await using var tx = await conn.BeginTransactionAsync();
try
{
    await using (var debit = new MySqlCommand(
        "UPDATE comptes SET solde = solde - @m WHERE id = @id", conn, tx))
    {
        debit.Parameters.AddWithValue("@m", montant);
        debit.Parameters.AddWithValue("@id", source);
        await debit.ExecuteNonQueryAsync();
    }
    await using (var credit = new MySqlCommand(
        "UPDATE comptes SET solde = solde + @m WHERE id = @id", conn, tx))
    {
        credit.Parameters.AddWithValue("@m", montant);
        credit.Parameters.AddWithValue("@id", destination);
        await credit.ExecuteNonQueryAsync();
    }
    await tx.CommitAsync();
}
catch
{
    await tx.RollbackAsync();
    throw;
}
```

La commande est rattachée à la transaction via le troisième argument du constructeur (`new MySqlCommand(sql, conn, tx)`).

## Authentification et MariaDB 12.3

- **`caching_sha2_password`** (§10.5.5) est pris en charge, ce qui simplifie la connexion à un MariaDB 12.3 utilisant ce plugin hérité de MySQL.
- **`ed25519`** (§10.5.2), propre à MariaDB, est disponible via un paquet compagnon : un paquet dédié implémente le plugin d'authentification `client_ed25519` pour MariaDB. Il s'ajoute séparément puis s'enregistre au démarrage de l'application :

  ```bash
  dotnet add package MySqlConnector.Authentication.Ed25519
  ```

  C'est un avantage notable du connecteur .NET : la prise en charge d'`ed25519` y est explicitement outillée, là où plusieurs pilotes orientés MySQL d'autres écosystèmes ne la fournissent pas nativement.

Le chiffrement TLS se règle via l'option **`SslMode`** de la chaîne de connexion (`None`, `Preferred`, `Required`, `VerifyCA`, `VerifyFull`) ; voir §10.7.

---

## Et Entity Framework Core ?

En pratique, beaucoup d'applications .NET n'écrivent pas d'ADO.NET « à la main » mais passent par **Entity Framework Core**, l'ORM de référence. Le fournisseur EF Core le plus courant pour MariaDB, **Pomelo.EntityFrameworkCore.MySql**, repose précisément sur MySqlConnector. Tout ce qui précède (chaîne de connexion, pooling, authentification) reste donc pertinent même avec un ORM. La modélisation et les requêtes via EF Core sont traitées au §17.3.5.

---

## Ce qu'il faut retenir

- **ADO.NET** est la couche d'accès standard ; **MySqlConnector** en est le fournisseur recommandé pour MariaDB — c'est aussi ce que MariaDB désigne par « MariaDB Connector/NET » (espace de noms `MySqlConnector`).
- Préférer MySqlConnector à MySql.Data, notamment pour son **asynchronisme réel** ; ne référencer qu'un seul des deux dans un projet.
- Code **async-first** : `await using`, `OpenAsync`, `ExecuteReaderAsync`, `ExecuteNonQueryAsync`, `ExecuteScalarAsync`.
- Marqueurs **nommés `@nom`** et requêtes paramétrées systématiques (§17.8, §17.9) ; identifiant via `LastInsertedId`.
- Le **pooling est intégré et actif par défaut** : ouvrir/libérer librement les connexions, régler le pool dans la chaîne de connexion (§17.2).
- `caching_sha2_password` géré ; **`ed25519` via le paquet `MySqlConnector.Authentication.Ed25519`** ; TLS via `SslMode`.
- Avec **Entity Framework Core**, le fournisseur Pomelo s'appuie sur MySqlConnector (§17.3.5).
