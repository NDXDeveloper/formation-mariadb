# 17.3.5 Entity Framework Core (.NET)

**Entity Framework Core** (EF Core) est l'ORM moderne, multiplateforme et open source de Microsoft pour .NET. Il relève de l'approche *Data Mapper* organisée autour d'un **`DbContext`** — l'unité de travail, qui assure le suivi des modifications et une *identity map* (cf. §17.3) — et s'interroge en **LINQ**. Comme tout l'écosystème .NET, il est résolument **asynchrone** (§17.1.6).

EF Core est neutre vis-à-vis de la base : il faut lui adjoindre un **provider**. Pour MariaDB, c'est Pomelo, bâti sur MySqlConnector (§17.1.6).

---

## Le provider MariaDB : Pomelo

**Pomelo.EntityFrameworkCore.MySql** est le provider EF Core le plus répandu pour MySQL, MariaDB et bases compatibles, construit au-dessus de MySqlConnector. Il prend en charge EF Core jusqu'à sa dernière version et tire parti de MySqlConnector pour la communication avec le serveur.

Un point de version mérite attention : Pomelo suit le rythme d'EF Core. Au moment de la rédaction, sa version stable est compatible avec .NET 8.0+, **EF Core 9.0.x** et MySqlConnector 2.4.0+, et testée notamment contre MariaDB 11.x et 10.x. Le **support d'EF Core 10** (aligné sur .NET 10) est en cours côté projet. En pratique : aligner la version de Pomelo sur celle d'EF Core de l'application, et consulter les notes de version. Une alternative existe (le provider d'Oracle, `MySql.EntityFrameworkCore`), mais Pomelo est le choix communautaire dominant.

---

## Configurer le DbContext

L'enregistrement se fait par injection de dépendances avec `UseMySql`, en précisant la **version du serveur**. Point MariaDB essentiel : on utilise **`MariaDbServerVersion`** (et non `MySqlServerVersion`), ou la détection automatique :

```csharp
var connectionString = builder.Configuration.GetConnectionString("Boutique")!;

builder.Services.AddDbContext<BoutiqueContext>(options =>
    options.UseMySql(
        connectionString,
        new MariaDbServerVersion(new Version(12, 3, 0))   // version MariaDB
        // ou : ServerVersion.AutoDetect(connectionString)
    ));
```

> ⚠️ Même pour MariaDB, la méthode reste **`UseMySql`** (le provider Pomelo est le provider « MySql », qui couvre les deux). Seule la **version** est de type `MariaDbServerVersion`. Par ailleurs, il est recommandé d'utiliser **`utf8mb4`** comme jeu de caractères par défaut de la base.

## Entités et DbContext

Les entités sont de simples classes (POCO) ; le `DbContext` expose un `DbSet<T>` par entité :

```csharp
using Microsoft.EntityFrameworkCore;

public class Produit
{
    public int Id { get; set; }
    public string Nom { get; set; } = "";
    public decimal Prix { get; set; }
    public int CategorieId { get; set; }
    public Categorie? Categorie { get; set; }   // propriété de navigation
}

public class BoutiqueContext(DbContextOptions<BoutiqueContext> options) : DbContext(options)
{
    public DbSet<Produit> Produits => Set<Produit>();
    public DbSet<Categorie> Categories => Set<Categorie>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Configuration "Fluent API" (alternative aux annotations)
        modelBuilder.Entity<Produit>(e =>
        {
            e.ToTable("produits");
            e.Property(p => p.Nom).HasMaxLength(200).IsRequired();
            e.Property(p => p.Prix).HasPrecision(10, 2);
        });
    }
}
```

Le mapping se déclare par **conventions**, par **annotations** (`[Key]`, `[Required]`, `[Table]`, `[Column]`, `[MaxLength]`) ou par la **Fluent API** dans `OnModelCreating` (illustrée ci-dessus).

## Code First ou Database First ?

Deux démarches sont possibles :

- **Code First** — on écrit les classes, puis on génère et fait évoluer la base par des **migrations** (le cas le plus courant) ;
- **Database First** — on génère les classes à partir d'une base existante (*scaffolding*) :

  ```bash
  dotnet ef dbcontext scaffold "votre-chaîne-de-connexion" "Pomelo.EntityFrameworkCore.MySql"
  ```

---

## Interroger avec LINQ

LINQ est la signature d'EF Core : les requêtes s'écrivent en code typé et sont **traduites en SQL**. Les variantes asynchrones (`ToListAsync`, `FindAsync`, `FirstOrDefaultAsync`) sont à privilégier.

```csharp
// Lecture par clé primaire
var p = await context.Produits.FindAsync(42);

// Requête LINQ → SQL
var produits = await context.Produits
    .Where(p => p.Prix < prixMax)
    .OrderBy(p => p.Nom)
    .Take(10)
    .ToListAsync();
```

---

## Écrire : l'unité de travail

On ajoute/modifie/supprime des entités, puis `SaveChangesAsync` matérialise les changements suivis par le `DbContext`, dans une transaction :

```csharp
var clavier = new Produit { Nom = "Clavier", Prix = 49.90m, CategorieId = 1 };
context.Produits.Add(clavier);
await context.SaveChangesAsync();
```

Pour les requêtes en **lecture seule**, `AsNoTracking()` désactive le suivi des modifications et accélère l'exécution :

```csharp
var produits = await context.Produits
    .AsNoTracking()
    .Where(p => p.Prix < prixMax)
    .ToListAsync();
```

---

## Relations et chargement

Les relations s'expriment par des **propriétés de navigation** (comme `Produit.Categorie`). Le **chargement anticipé** se demande avec `Include` (et `ThenInclude` pour les niveaux suivants). Le **chargement paresseux** est *optionnel* (il nécessite le paquet de *proxies* et des propriétés de navigation `virtual`) ; le **chargement explicite** est aussi disponible.

---

## Le problème N+1 dans EF Core

Si le chargement paresseux est activé, accéder à une navigation dans une boucle déclenche le N+1 (§17.3). La parade est **`Include`** :

```csharp
var produits = await context.Produits
    .Include(p => p.Categorie)
    .ToListAsync();
```

Particularité utile d'EF Core : avec **plusieurs `Include`** sur des collections, une seule requête jointe peut provoquer une **explosion cartésienne** (lignes dupliquées). **`AsSplitQuery()`** émet alors une requête par collection plutôt qu'une jointure unique :

```csharp
var commandes = await context.Commandes
    .Include(c => c.Lignes)
    .Include(c => c.Paiements)
    .AsSplitQuery()
    .ToListAsync();
```

Comme partout, observer le SQL généré (journalisation EF Core) reste le réflexe décisif (§17.4).

## Migrations

EF Core fournit un système de **migrations** qui, à la différence de Sequelize (§17.3.3) et à l'image d'Alembic (§17.3.2) ou de Prisma, **dérive automatiquement** le code de migration des changements du modèle :

```bash
dotnet ef migrations add InitialCreate
dotnet ef database update
```

> ⚠️ La méthode `EnsureCreated()` crée la base sans historique de migration : à réserver aux tests, **jamais en production** (même esprit que les avertissements `ddl-auto`/`sync`, §17.3.1, §17.3.3). En production, on applique les migrations (`dotnet ef database update`, ou `Database.Migrate()` au démarrage). Les migrations s'inscrivent dans la gestion globale du schéma (§17.5, chap. 16.8).

## Pooling

Deux niveaux de mutualisation coexistent en .NET :

- le **pooling de connexions**, intégré à ADO.NET / MySqlConnector et actif par défaut (§17.1.6, §17.2.1) ;
- le **pooling de `DbContext`**, propre à EF Core, qui réutilise les instances de contexte via `AddDbContextPool` — utile pour les applications à fort débit :

  ```csharp
  builder.Services.AddDbContextPool<BoutiqueContext>(options =>
      options.UseMySql(connectionString, new MariaDbServerVersion(new Version(12, 3, 0))));
  ```

---

## Spécificités MariaDB et SQL brut

- **Version serveur** — `MariaDbServerVersion` (ou `AutoDetect`) ; jeu de caractères `utf8mb4`.
- **TINYINT(1) et booléens** — l'option de connexion `TreatTinyAsBoolean` règle la correspondance `TINYINT(1)`↔`bool`.
- **JSON** — pris en charge ; les colonnes JSON de MariaDB se mappent en conséquence.
- **Fonctionnalités non modélisées** — séquences, tables temporelles, VECTOR (chap. 18) s'atteignent par le SQL brut.

La porte de sortie (cf. §17.3) est **`FromSql`** (interpolation paramétrée, donc protégée des injections, §17.8) et `ExecuteSqlAsync` pour les commandes :

```csharp
var produits = await context.Produits
    .FromSql($"SELECT * FROM produits WHERE prix > {min}")
    .ToListAsync();
```

---

## Ce qu'il faut retenir

- **EF Core** est l'ORM .NET de Microsoft, *Data Mapper* autour du **`DbContext`** (unité de travail + suivi des modifications), interrogé en **LINQ** et **asynchrone** (§17.1.6).
- Provider MariaDB : **Pomelo**, bâti sur **MySqlConnector** (§17.1.6) ; aligner sa version sur celle d'EF Core (stable : EF Core 9 ; EF Core 10 en cours).
- Configuration via **`UseMySql` + `MariaDbServerVersion`** (ou `AutoDetect`) ; jeu de caractères **`utf8mb4`**.
- Entités POCO + `DbSet<T>` ; mapping par conventions/annotations/**Fluent API** ; **Code First** (migrations) ou **Database First** (*scaffolding*).
- LINQ → SQL ; **`SaveChangesAsync`** pour l'unité de travail ; **`AsNoTracking()`** en lecture seule.
- **N+1** : corriger par **`Include`** ; **`AsSplitQuery()`** contre l'explosion cartésienne ; surveiller le SQL (§17.4).
- Migrations **autogénérées** (`dotnet ef migrations add`) — `EnsureCreated()` **jamais en production** (§17.5) ; **pooling de `DbContext`** via `AddDbContextPool` (§17.2.1).
