# 17.2.1 Pool côté application

Le *pooling* **côté application** désigne un pool qui vit **dans le processus de l'application**, fourni par le pilote ou une bibliothèque dédiée. C'est l'approche la plus répandue : aucune infrastructure supplémentaire à déployer, une latence minimale (le pool est local), et une configuration souvent réduite à quelques lignes.

Sa limite tient à sa localité : **chaque instance applicative possède son propre pool**. Le nombre total de connexions au serveur croît donc avec le nombre d'instances — un point qui motive le recours à un *pooler* externe (§17.2.2) lorsque les instances se multiplient.

Cette sous-section décline la configuration concrète pour chaque écosystème vu au §17.1, en s'appuyant sur les paramètres universels définis au §17.2.

---

## Java — HikariCP

HikariCP est le pool de référence (utilisé par défaut par Spring Boot). On l'alimente avec l'URL JDBC et l'on règle les paramètres essentiels :

```java
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:mariadb://localhost:3306/boutique");
config.setUsername("app_user");
config.setPassword(motDePasse);
config.setMaximumPoolSize(10);          // taille max (défaut : 10)
config.setMinimumIdle(10);              // pool de taille fixe (recommandé)
config.setConnectionTimeout(30_000);    // délai d'acquisition, en ms
config.setMaxLifetime(1_800_000);       // recyclage (< wait_timeout serveur)
config.setKeepaliveTime(300_000);       // maintien des connexions inactives
HikariDataSource ds = new HikariDataSource(config);
```

Deux points propres à HikariCP : il **recommande un pool de taille fixe** (`minimumIdle` égal à `maximumPoolSize`) pour des performances prévisibles, et il **valide les connexions via `Connection.isValid()`** automatiquement — inutile, en général, de définir une requête de test. Sous Spring Boot, ces réglages passent par les propriétés `spring.datasource.hikari.*`.

## .NET — pooling intégré d'ADO.NET

En .NET, le pool est **intégré et activé par défaut** ; il se configure dans la **chaîne de connexion**, et les connexions partageant une chaîne identique partagent le même pool (cf. §17.1.6) :

```
Server=localhost;Port=3306;Database=boutique;Uid=app_user;Pwd=...;Pooling=true;MinimumPoolSize=5;MaximumPoolSize=100;ConnectionIdleTimeout=180;ConnectionLifeTime=600;ConnectionTimeout=30;
```

Repères : `MaximumPoolSize` (défaut 100), `MinimumPoolSize` (défaut 0), `ConnectionIdleTimeout` (durée d'inactivité, en secondes), `ConnectionLifeTime` (durée de vie max ; 0 = illimitée) et `ConnectionTimeout` (délai d'acquisition/ouverture). On ouvre et libère les connexions librement : la fermeture les rend au pool.

## Go — `database/sql`

En Go, le pool **est** l'objet `*sql.DB` (cf. §17.1.5) ; on l'ajuste après ouverture :

```go
db.SetMaxOpenConns(25)                  // connexions ouvertes simultanées
db.SetMaxIdleConns(25)                  // connexions inactives conservées
db.SetConnMaxLifetime(5 * time.Minute)  // recyclage (< wait_timeout serveur)
db.SetConnMaxIdleTime(2 * time.Minute)  // durée d'inactivité max
```

> ⚠️ **Piège classique** : `SetMaxIdleConns` vaut **2 par défaut**. Si `MaxOpenConns` est élevé mais `MaxIdleConns` reste à 2, les connexions au-delà de 2 sont ouvertes puis refermées en boucle (*churn*), annulant le bénéfice du pool. On fixe généralement `MaxIdleConns` proche de `MaxOpenConns`.

L'absence de connexion libre fait **patienter** `Query`/`Exec` jusqu'à libération ou expiration du `context` : c'est ce dernier qui joue le rôle de délai d'acquisition.

## Node.js — mysql2 et connecteur mariadb

Les deux connecteurs fournissent un pool via `createPool` (cf. §17.1.4).

**mysql2 :**

```javascript
const pool = mysql.createPool({
  host: 'localhost', user: 'app_user',
  password: process.env.DB_PASSWORD, database: 'boutique',
  waitForConnections: true,   // mettre en file plutôt qu'échouer
  connectionLimit: 10,        // taille max
  queueLimit: 0,              // file illimitée
  idleTimeout: 60000,
  maxIdle: 10,
  enableKeepAlive: true,
});
```

**Connecteur `mariadb` :** il expose en plus un paramètre précieux pour traquer les fuites :

```javascript
const pool = mariadb.createPool({
  host: 'localhost', user: 'app_user',
  password: process.env.DB_PASSWORD, database: 'boutique',
  connectionLimit: 10,
  acquireTimeout: 30000,        // délai d'acquisition
  idleTimeout: 600,             // en secondes
  minimumIdle: 5,
  leakDetectionTimeout: 10000,  // signale une connexion non restituée à temps
});
```

## Python — SQLAlchemy (et pilotes bruts)

À noter d'abord : les pilotes DB-API bruts (**PyMySQL**, **mysql-connector-python**) **ne mettent pas en pool par défaut** — chaque `connect()` ouvre une nouvelle connexion. mysql-connector-python propose son propre pool (`MySQLConnectionPool`), mais l'approche la plus robuste et la plus répandue consiste à passer par le **pool intégré de SQLAlchemy**, même quand on n'utilise que SQLAlchemy Core.

SQLAlchemy emploie un `QueuePool` par défaut, configuré dans `create_engine` :

```python
engine = create_engine(
    url,
    pool_size=5,          # connexions persistantes (défaut 5)
    max_overflow=10,      # connexions temporaires supplémentaires (défaut 10)
    pool_timeout=30,      # délai d'acquisition (s)
    pool_recycle=1800,    # recyclage (< wait_timeout serveur)
    pool_pre_ping=True,   # validation avant usage
)
```

> 💡 Le nombre **maximal** de connexions ouvertes par SQLAlchemy est `pool_size + max_overflow` (ici 15) : c'est ce total qu'il faut prendre en compte dans le calcul vis-à-vis de `max_connections`.

## PHP — un modèle à part

PHP fait exception. Dans le modèle classique (mod_php, PHP-FPM), chaque requête HTTP est traitée par un *worker* à durée de vie courte : il n'existe pas, nativement, de pool partagé à long terme comme dans les écosystèmes précédents.

Le seul levier intégré est la **connexion persistante** — préfixe `p:` avec mysqli, `PDO::ATTR_PERSISTENT => true` avec PDO (cf. §17.1.1) — qui réutilise une connexion entre requêtes **au sein d'un même worker**. Il s'agit d'un cache de connexion par *worker*, non d'un véritable pool, et il faut s'en méfier : un état résiduel (transaction non terminée, table temporaire, variable de session) peut « fuir » d'une requête à l'autre. Par ailleurs, le nombre de connexions tend vers *nombre de workers FPM × nombre de serveurs*, ce qui grimpe vite.

C'est pourquoi, en PHP, le **vrai** *pooling* se fait le plus souvent **en dehors** de l'application, via un *pooler* externe comme ProxySQL (§17.2.2) — sauf à adopter un modèle PHP à processus long (Swoole, RoadRunner) qui change la donne.

## Correspondance des paramètres

Les leviers universels du §17.2 portent des noms différents selon l'outil :

| Paramètre (§17.2) | HikariCP (Java) | ADO.NET (.NET) | `database/sql` (Go) | SQLAlchemy (Python) | mysql2 / mariadb (Node) |
|-------------------|-----------------|----------------|---------------------|---------------------|-------------------------|
| Taille max | `maximumPoolSize` | `MaximumPoolSize` | `SetMaxOpenConns` | `pool_size` (+`max_overflow`) | `connectionLimit` |
| Connexions inactives | `minimumIdle` | `MinimumPoolSize` | `SetMaxIdleConns` | (= `pool_size`) | `maxIdle` / `minimumIdle` |
| Délai d'acquisition | `connectionTimeout` | `ConnectionTimeout` | via `context` | `pool_timeout` | `acquireTimeout` / file |
| Durée d'inactivité | `idleTimeout` | `ConnectionIdleTimeout` | `SetConnMaxIdleTime` | (via `pool_recycle`) | `idleTimeout` |
| Durée de vie max | `maxLifetime` | `ConnectionLifeTime` | `SetConnMaxLifetime` | `pool_recycle` | (selon connecteur) |
| Validation | `isValid()` (auto) | (interne) | manuelle / `Ping` | `pool_pre_ping` | *pre-ping* / validation |

## Bonnes pratiques transversales

- **Caler la durée de vie maximale sous `wait_timeout`** du serveur (et sous tout délai d'inactivité d'un pare-feu), afin que le pool recycle ses connexions avant qu'elles ne soient coupées à son insu (sinon : connexions périmées, §17.2).
- **Activer la validation / le *pre-ping*** sur les réseaux peu fiables ou en présence de coupures d'inactivité.
- **Éviter le *churn*** : connexions inactives suffisantes (le défaut de 2 en Go est un piège).
- **Superviser les métriques du pool** — connexions actives, inactives, en attente — exposées par la plupart des outils (métriques HikariCP, méthodes du pool `mariadb`, etc.). Voir le monitoring applicatif (§17.6) et l'observabilité (chap. 16).
- **Un pool par instance** : garder à l'esprit que les connexions se cumulent à mesure que les instances se multiplient.

---

## Quand le pool applicatif atteint ses limites

Le pool applicatif suffit dans l'immense majorité des cas. Il montre ses limites quand **les instances se multiplient** : microservices nombreux, *autoscaling*, ou fonctions *serverless* à durée de vie très courte (qui ne profitent pas d'un pool persistant). Chaque instance ouvrant son propre pool, on peut alors dépasser `max_connections` malgré des pools individuellement modestes. C'est précisément le terrain d'un *pooler* centralisé, objet du §17.2.2.

---

## Ce qu'il faut retenir

- Le pool **côté application** vit dans le processus : simple, local, sans infrastructure — mais **un pool par instance**.
- Chaque écosystème expose les mêmes leviers sous des noms différents (voir le tableau de correspondance) : HikariCP (Java), pooling intégré d'ADO.NET (.NET), `*sql.DB` (Go), `QueuePool` de SQLAlchemy (Python), `createPool` (Node.js).
- Pièges spécifiques : le **`MaxIdleConns=2`** de Go, l'**absence de pool natif** des pilotes Python bruts, et le **modèle par requête** de PHP (connexions persistantes ≠ pool).
- Régler la **durée de vie max sous `wait_timeout`**, activer la **validation**, éviter le ***churn***, **superviser** le pool.
- Quand les instances se multiplient au point de saturer `max_connections`, passer à un *pooler* externe (§17.2.2).
