# 17.3.1 Hibernate (Java)

**Hibernate** est l'ORM le plus répandu de l'écosystème Java. Il relève du patron *Data Mapper* (cf. §17.3) et se construit au-dessus de JDBC et de MariaDB Connector/J (§17.1.3), en s'appuyant typiquement sur HikariCP pour le *pooling* (§17.2.1).

---

## JPA et Hibernate

Une distinction s'impose d'emblée : **JPA** (*Jakarta Persistence API*, anciennement *Java Persistence API*) est une **spécification** — un ensemble standard d'interfaces et d'annotations. **Hibernate** en est l'**implémentation** la plus utilisée (il en existe d'autres, comme EclipseLink). En pratique, on écrit la majeure partie du code contre l'API standard JPA, ce qui le rend portable, Hibernate fournissant le moteur et quelques extensions propres.

À noter pour les versions récentes : depuis **Hibernate 6** (et Spring Boot 3), l'espace de noms est **`jakarta.persistence`** et non plus `javax.persistence`. Les exemples ci-dessous suivent cette convention.

---

## Le dialecte MariaDB

Hibernate traduit ses opérations en SQL adapté à la base cible via un **dialecte**. Pour MariaDB, c'est `org.hibernate.dialect.MariaDBDialect`. Depuis Hibernate 6, ce dialecte **détecte automatiquement la version** du serveur (les dialectes versionnés antérieurs sont dépréciés), et il est souvent même inutile de le déclarer explicitement, Hibernate le déduisant de la connexion. Le pilote reste MariaDB Connector/J avec une URL `jdbc:mariadb://…` (§17.1.3).

## Mapper une entité

Une **entité** est une classe annotée associée à une table. Le mapping se déclare par annotations JPA :

```java
import jakarta.persistence.*;
import java.math.BigDecimal;

@Entity
@Table(name = "produits")
public class Produit {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) // colonne AUTO_INCREMENT
    private Long id;

    @Column(nullable = false, length = 200)
    private String nom;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal prix;

    @ManyToOne(fetch = FetchType.LAZY)   // LAZY explicite (voir plus bas)
    @JoinColumn(name = "categorie_id")
    private Categorie categorie;

    // getters / setters
}
```

La stratégie de génération de la clé mérite attention : **`IDENTITY`** s'appuie sur une colonne `AUTO_INCREMENT`, le plus courant avec MariaDB. **`SEQUENCE`** utilise une véritable séquence — possible car MariaDB prend en charge `CREATE SEQUENCE` (chap. 18.1) :

```java
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "produit_seq")
@SequenceGenerator(name = "produit_seq", sequenceName = "produit_sequence", allocationSize = 50)
private Long id;
```

## L'EntityManager : l'unité de travail

L'`EntityManager` (ou la `Session` Hibernate) est l'**unité de travail** : il suit les entités, leurs modifications, et les écrit lors d'un *flush*/commit. Les opérations de base sont `persist` (insérer), `find` (lire par clé), `merge` (réattacher/mettre à jour) et `remove` (supprimer), encadrées par une transaction.

```java
// Lecture par clé primaire
Produit p = em.find(Produit.class, 42L);

// Écriture, dans une transaction
em.getTransaction().begin();
Produit nouveau = new Produit();
nouveau.setNom("Clavier");
nouveau.setPrix(new BigDecimal("49.90"));
em.persist(nouveau);
em.getTransaction().commit();
```

## Associations et chargement

Les relations se déclarent par `@OneToMany`, `@ManyToOne`, `@ManyToMany` et `@JoinColumn`/`@JoinTable`, traduites en clés étrangères (§2.5). Un point de vigilance majeur concerne les **stratégies de chargement par défaut** :

- `@ManyToOne` et `@OneToOne` sont **`EAGER` par défaut** — les entités liées sont chargées systématiquement, même quand on n'en a pas besoin ;
- `@OneToMany` et `@ManyToMany` sont **`LAZY` par défaut**.

La bonne pratique consensuelle est de **forcer `LAZY` partout** (comme dans l'exemple d'entité ci-dessus) et de demander explicitement le chargement anticipé là où c'est utile. L'`EAGER` par défaut sur les relations *-à-un* est une source fréquente de requêtes superflues.

## Interroger : JPQL, Criteria et SQL natif

Hibernate offre plusieurs niveaux d'interrogation :

- **JPQL/HQL** — un langage orienté objet qui interroge les **entités** (et non les tables) :

  ```java
  List<Produit> resultats = em.createQuery(
          "SELECT p FROM Produit p WHERE p.categorie.nom = :cat AND p.prix < :max",
          Produit.class)
      .setParameter("cat", categorie)
      .setParameter("max", prixMax)
      .getResultList();
  ```

- **Criteria API** — une construction programmatique et typée des requêtes, pratique pour les filtres dynamiques ;
- **SQL natif** — la porte de sortie vers le SQL brut (cf. §17.3), pour les requêtes analytiques ou les fonctionnalités propres à MariaDB :

  ```java
  List<Object[]> stats = em.createNativeQuery(
          "SELECT categorie_id, COUNT(*) FROM produits GROUP BY categorie_id")
      .getResultList();
  ```

## Le problème N+1 dans Hibernate

Le piège N+1 (§17.3) se manifeste typiquement ainsi : on charge une liste de commandes, puis on accède à `commande.getClient()` (relation paresseuse) dans une boucle — Hibernate émet alors une requête par commande. Les parades principales :

- **`JOIN FETCH`** en JPQL, pour rapatrier la relation en une seule requête :

  ```java
  List<Commande> commandes = em.createQuery(
          "SELECT c FROM Commande c JOIN FETCH c.client", Commande.class)
      .getResultList();
  ```

- **`@EntityGraph`** (JPA), pour déclarer le graphe de chargement sans modifier la requête ;
- **`@BatchSize`** (extension Hibernate), pour charger les relations par lots plutôt qu'une par une.

Pour détecter ces situations, activer la journalisation du SQL généré (voir ci-dessous) est indispensable — c'est le réflexe recommandé au §17.4.

## Configuration et génération de schéma

En *standalone* (Jakarta EE), la configuration passe par `persistence.xml` ; avec **Spring Boot**, par `application.properties` (le cas le plus courant) :

```properties
spring.datasource.url=jdbc:mariadb://localhost:3306/boutique
spring.datasource.username=app_user
spring.datasource.password=${DB_PASSWORD}
spring.datasource.driver-class-name=org.mariadb.jdbc.Driver

spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MariaDBDialect
spring.jpa.hibernate.ddl-auto=validate
spring.jpa.show-sql=true

spring.datasource.hikari.maximum-pool-size=10
spring.datasource.hikari.max-lifetime=1800000
```

> ⚠️ **`ddl-auto` (génération de schéma)** : les valeurs `update` et `create` laissent Hibernate modifier le schéma automatiquement. **À proscrire en production** : perte de données possible et évolutions non maîtrisées. En production, utiliser **`validate`** (vérifie la cohérence schéma/entités) ou **`none`**, et gérer l'évolution du schéma par des **migrations** (§17.5) avec Flyway ou Liquibase (chap. 16.8).

L'option `show-sql` (et `hibernate.format_sql`) affiche le SQL réellement exécuté : précieux pour repérer un N+1 ou une requête inattendue.

---

## Pool de connexions et Spring Data JPA

Hibernate délègue le *pooling* : avec la dépendance adéquate, il utilise **HikariCP** (§17.2.1), que Spring Boot configure par défaut via les propriétés `spring.datasource.hikari.*`. Tout ce qui a été dit sur le dimensionnement et la durée de vie des connexions s'applique ici.

Par ailleurs, beaucoup d'applications n'utilisent pas l'`EntityManager` directement mais **Spring Data JPA**, qui ajoute une couche de **dépôts** (*repositories*) au-dessus de JPA/Hibernate : on déclare une interface étendant `JpaRepository`, et les méthodes de requête sont dérivées du nom ou annotées avec `@Query`. Le moteur sous-jacent reste Hibernate, et les mêmes précautions (N+1, SQL généré, schéma) demeurent.

---

## Spécificités MariaDB

- **Séquences** — prises en charge (chap. 18.1) via `GenerationType.SEQUENCE`.
- **JSON** — Hibernate 6 a amélioré le mapping des colonnes JSON (mapping d'un attribut vers un type JSON ou une représentation textuelle).
- **Fonctionnalités non mappées** — les **tables temporelles** (*system-versioned*, chap. 18.2) et d'autres extensions MariaDB ne sont pas exposées directement : on y accède par **SQL natif**.

---

## Ce qu'il faut retenir

- **Hibernate** implémente la spécification **JPA** (espace de noms `jakarta.persistence` depuis Hibernate 6) ; il s'appuie sur JDBC/Connector/J (§17.1.3) et HikariCP (§17.2.1).
- Le **dialecte `MariaDBDialect`** est souvent auto-détecté ; le mapping se fait par annotations (`@Entity`, `@Id`, `@GeneratedValue`, `@Column`, associations).
- L'**`EntityManager`** est l'unité de travail (`persist`/`find`/`merge`/`remove`) ; **forcer `LAZY`** sur les relations et charger en anticipé au cas par cas.
- Interroger via **JPQL**, **Criteria** ou **SQL natif** ; surveiller le **SQL généré** pour traquer le **N+1** (`JOIN FETCH`, `@EntityGraph`, `@BatchSize`).
- **Ne jamais utiliser `ddl-auto=update/create` en production** : préférer `validate`/`none` et des migrations (§17.5, chap. 16.8).
- Souvent employé via **Spring Data JPA** (dépôts), Hibernate restant le moteur sous-jacent.
