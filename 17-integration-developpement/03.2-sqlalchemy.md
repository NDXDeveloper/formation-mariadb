# 17.3.2 SQLAlchemy (Python)

SQLAlchemy a déjà été présenté au §17.1.2 sous l'angle de la **connexion** : l'`Engine`, l'URL, le pool intégré, l'exécution de SQL brut via `text()` et le langage d'expression **Core**. Cette section traite sa couche **ORM**, qui se construit au-dessus de Core et relève du patron *Data Mapper* (cf. §17.3) : modèles déclaratifs, `Session` (unité de travail), relations et requêtes objet.

L'ORM réutilise tout l'appareillage de connexion vu précédemment : il s'appuie sur le même `Engine` et donc sur le même **pool** (§17.2.1), avec le **dialecte MariaDB** choisi dans l'URL (§17.1.2).

---

## Le style SQLAlchemy 2.0

La version **2.0** (parue début 2023) a unifié l'API. Le style moderne s'appuie sur :

- des **mappings typés** avec `DeclarativeBase`, l'annotation `Mapped[...]` et `mapped_column()` ;
- une interrogation par **`select()` + `Session.execute()`/`Session.scalars()`**, qui remplace l'ancienne API `session.query()` (toujours disponible, mais déconseillée).

Les exemples ci-dessous suivent ce style 2.0.

---

## Déclarer un modèle

Un modèle est une classe héritant d'une base déclarative, dont les attributs typés sont associés aux colonnes :

```python
from decimal import Decimal
from typing import Optional
from sqlalchemy import String, Numeric, ForeignKey
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship

class Base(DeclarativeBase):
    pass

class Produit(Base):
    __tablename__ = "produits"

    id: Mapped[int] = mapped_column(primary_key=True)               # clé primaire
    nom: Mapped[str] = mapped_column(String(200))
    prix: Mapped[Decimal] = mapped_column(Numeric(10, 2))
    description: Mapped[Optional[str]] = mapped_column(String(500)) # colonne NULL-able

    categorie_id: Mapped[int] = mapped_column(ForeignKey("categories.id"))
    categorie: Mapped["Categorie"] = relationship(back_populates="produits")
```

Le typage `Mapped[...]` détermine en partie le mapping : un `Mapped[Optional[str]]` (ou `Mapped[str | None]`) produit une colonne acceptant `NULL`. Les clés étrangères se déclarent avec `ForeignKey` (§2.5), et les relations avec `relationship`.

## La Session : l'unité de travail

La `Session` est l'**unité de travail** de l'ORM : elle suit les objets (*identity map* : un seul objet par ligne), enregistre les modifications et les écrit lors du `commit`. On la crée généralement via `sessionmaker`, et on l'utilise comme gestionnaire de contexte.

```python
from sqlalchemy.orm import sessionmaker

# engine construit comme au §17.1.2
Session = sessionmaker(engine)

with Session() as session:
    nouveau = Produit(nom="Clavier", prix=Decimal("49.90"))
    session.add(nouveau)
    session.commit()
```

Les opérations courantes : `session.add()` (persister), `session.get()` (lire par clé), `session.execute()`/`session.scalars()` (interroger), `session.delete()` (supprimer), `session.commit()` / `session.rollback()`.

---

## Interroger en style 2.0

On construit une requête avec `select()`, on la filtre avec `where()`, puis on l'exécute. `scalars()` renvoie directement les entités (et non des lignes de tuples) :

```python
from sqlalchemy import select

with Session() as session:
    # Lecture par clé primaire
    p = session.get(Produit, 42)

    # Requête : select() + scalars()
    stmt = (
        select(Produit)
        .where(Produit.prix < prix_max)
        .order_by(Produit.nom)
    )
    for produit in session.scalars(stmt):
        print(f"{produit.nom} : {produit.prix} €")
```

## Relations et chargement

Les relations se déclarent par `relationship` avec `back_populates` (qui maintient la cohérence des deux côtés de l'association). Comme tout ORM, SQLAlchemy propose plusieurs **stratégies de chargement** pour les relations :

- `lazy="select"` — **paresseux**, le défaut : la relation est lue au premier accès (c'est là que naît le N+1) ;
- `lazy="joined"` — chargement anticipé par jointure externe ;
- `lazy="selectin"` — chargement anticipé par une requête `IN` séparée, souvent le meilleur compromis pour les collections ;
- `lazy="subquery"` — chargement par sous-requête ;
- `lazy="raise"` — lève une erreur si la relation est chargée paresseusement, utile pour **traquer** les N+1 en développement.

---

## Le problème N+1 dans SQLAlchemy

Avec le chargement paresseux par défaut, parcourir une liste de commandes puis accéder à `commande.lignes` dans une boucle déclenche une requête par commande (§17.3). On corrige soit **ponctuellement** par une *option de chargement* sur la requête, soit **structurellement** en réglant la stratégie sur la relation.

```python
from sqlalchemy.orm import selectinload

# Option ponctuelle : charge les lignes en une requête IN supplémentaire
stmt = select(Commande).options(selectinload(Commande.lignes))
commandes = session.scalars(stmt).all()
```

Les options principales sont **`selectinload()`** (idéale pour les collections), **`joinedload()`** (pour les relations *-à-un*) et `subqueryload()`. En développement, `lazy="raise"` sur une relation force à expliciter chaque chargement et révèle les N+1 latents. Comme toujours, surveiller le SQL émis (en activant `echo=True` sur l'`Engine`, ou via la journalisation) reste le réflexe décisif (§17.4).

---

## Core et SQL brut en secours

Pour les requêtes analytiques, les agrégations complexes ou les fonctionnalités propres à MariaDB, on conserve la porte de sortie vue au §17.1.2 : le langage **Core** (`select()` sur des tables) et le **SQL brut** via `text()`. L'ORM et le Core partagent la même session/connexion : on mélange librement les deux selon les besoins, conformément à la posture du §17.3.

---

## Migrations : Alembic

L'évolution du schéma s'appuie sur **Alembic**, l'outil de migration issu du même auteur que SQLAlchemy. Il génère des scripts de migration versionnés et peut les **dériver automatiquement** des modèles :

```bash
alembic revision --autogenerate -m "ajoute la table produits"
alembic upgrade head
```

L'autogénération compare les modèles à l'état de la base et propose les changements ; ils doivent toujours être **relus** avant application. Alembic s'inscrit dans la gestion des migrations de schéma (§17.5) au même titre que Flyway ou Liquibase (chap. 16.8).

---

## Spécificités MariaDB

- **Dialecte** — choisi dans l'URL (`mariadb+pymysql`, `mariadb+mariadbconnector`…, §17.1.2) ; le dialecte `mariadb` est distinct de `mysql`.
- **JSON** — SQLAlchemy expose un type `JSON` mappant les colonnes JSON de MariaDB.
- **Fonctionnalités non mappées** — séquences, tables temporelles (chap. 18) et autres extensions s'atteignent par **`text()`** ou des constructions Core.

---

## Ce qu'il faut retenir

- L'**ORM** SQLAlchemy se construit sur le **Core** et l'`Engine` vus au §17.1.2 ; il réutilise le même **pool** (§17.2.1) et le **dialecte MariaDB**.
- Style **2.0** : mappings typés (`DeclarativeBase`, `Mapped[...]`, `mapped_column()`), requêtes par **`select()` + `scalars()`** (l'ancien `session.query()` est déconseillé).
- La **`Session`** est l'unité de travail (`add`/`get`/`execute`/`commit`), avec *identity map* et suivi des modifications.
- Relations via **`relationship(back_populates=…)`** ; stratégies de chargement (`select`, `joined`, **`selectin`**, `subquery`, `raise`).
- **N+1** : corriger par **`selectinload()`/`joinedload()`** ou une stratégie sur la relation ; `lazy="raise"` pour les détecter ; surveiller le SQL (§17.4).
- **SQL brut/Core** en secours (§17.1.2) ; migrations par **Alembic** (§17.5, chap. 16.8).
