# 17.1.5 Go : go-sql-driver/mysql

En Go, l'accès aux bases relationnelles passe par le paquet de la **bibliothèque standard `database/sql`**, qui définit une interface générique et neutre vis-à-vis du fournisseur. On y branche un **pilote**, et pour MariaDB le pilote de référence est **`go-sql-driver/mysql`** — écrit entièrement en Go (sans *cgo*), donc trivial à compiler de façon croisée (famille des implémentations natives au langage, cf. §17.1).

Particularité culturelle : Go favorise l'usage direct de `database/sql`, éventuellement assorti d'aides légères (`sqlx`) ou d'un ORM (`GORM`). Cette section se concentre sur l'API standard, qui suffit à la grande majorité des cas.

---

## `database/sql` et le pilote

`database/sql` ne sait pas parler à MariaDB par lui-même : il délègue à un pilote enregistré. Ce pilote s'importe pour son **effet de bord** (l'enregistrement), au moyen d'un *blank import* :

```go
import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql" // enregistre le pilote sous le nom "mysql"
)
```

> 💡 Le pilote s'enregistre sous le nom **`"mysql"`**, y compris pour MariaDB : il n'existe pas de nom de pilote « mariadb » avec cette bibliothèque. On ouvre donc toujours avec `sql.Open("mysql", …)`.

### Point clé : `sql.DB` est un **pool**, pas une connexion

C'est l'idiome le plus important à intégrer en Go : un objet `*sql.DB` ne représente **pas** une connexion unique, mais un **pool de connexions géré automatiquement**. On l'ouvre **une seule fois** pour la durée de vie de l'application et on le partage : il est conçu pour un usage concurrent par plusieurs *goroutines*. Inutile, donc, d'ajouter une bibliothèque de *pooling* externe comme dans d'autres écosystèmes.

## Installer et ouvrir

```bash
go get github.com/go-sql-driver/mysql
```

Le **DSN** du pilote suit la forme `utilisateur:motdepasse@tcp(hôte:port)/base?option=valeur`. Pour une construction sûre (échappement du mot de passe, options typées), on peut utiliser la structure `mysql.Config` et sa méthode `FormatDSN()` :

```go
import (
    "database/sql"
    "os"
    "time"

    "github.com/go-sql-driver/mysql" // import normal pour accéder à mysql.Config
)

cfg := mysql.Config{
    User:                 "app_user",
    Passwd:               os.Getenv("DB_PASSWORD"),  // jamais en dur
    Net:                  "tcp",
    Addr:                 "localhost:3306",
    DBName:               "boutique",
    Collation:            "utf8mb4_general_ci",
    ParseTime:            true,        // voir la section sur les types
    Loc:                  time.Local,
    AllowNativePasswords: true,
}

db, err := sql.Open("mysql", cfg.FormatDSN()) // n'ouvre AUCUNE connexion ici
if err != nil {
    return err
}
defer db.Close()

// Vérifie réellement la connectivité
if err := db.Ping(); err != nil {
    return err
}
```

À retenir : **`sql.Open` n'établit pas de connexion** ; il prépare le pool de manière paresseuse. C'est `db.Ping()` (ou la première requête) qui ouvre effectivement une connexion — d'où l'usage systématique de `Ping()` au démarrage pour détecter tôt une mauvaise configuration.

### Configurer le pool

Quelques réglages encadrent le comportement du pool (détails au §17.2) :

```go
db.SetMaxOpenConns(25)                  // connexions ouvertes simultanément
db.SetMaxIdleConns(25)                  // connexions inactives conservées
db.SetConnMaxLifetime(5 * time.Minute)  // durée de vie max d'une connexion
db.SetConnMaxIdleTime(2 * time.Minute)  // durée d'inactivité max
```

## Interroger la base

Le marqueur de paramètre est `?`. On distingue les requêtes renvoyant plusieurs lignes (`Query`), une seule (`QueryRow`) ou aucune (`Exec`). Les variantes suffixées `Context` acceptent un `context.Context` pour la gestion des délais et de l'annulation, et sont à privilégier.

### Plusieurs lignes

```go
rows, err := db.QueryContext(ctx,
    "SELECT id, nom, prix FROM produits WHERE categorie = ? AND prix < ?",
    categorie, prixMax,
)
if err != nil {
    return err
}
defer rows.Close() // INDISPENSABLE : libère la connexion vers le pool

for rows.Next() {
    var (
        id   int64
        nom  string
        prix float64
    )
    if err := rows.Scan(&id, &nom, &prix); err != nil { // scanner dans des pointeurs
        return err
    }
    fmt.Printf("%s : %.2f €\n", nom, prix)
}
if err := rows.Err(); err != nil { // erreur survenue pendant l'itération
    return err
}
```

Trois réflexes obligatoires : **`defer rows.Close()`** (sans quoi la connexion n'est pas rendue au pool — fuite garantie), scanner dans des **pointeurs**, et vérifier **`rows.Err()`** après la boucle.

### Une seule ligne

`QueryRow` renvoie un résultat unique ; l'absence de ligne se détecte via la sentinelle `sql.ErrNoRows` :

```go
var nom string
err := db.QueryRowContext(ctx,
    "SELECT nom FROM produits WHERE id = ?", id,
).Scan(&nom)

switch {
case err == sql.ErrNoRows:
    // aucun produit avec cet identifiant
case err != nil:
    return err
default:
    // succès
}
```

## Écrire et récupérer l'identifiant

`Exec` couvre `INSERT`/`UPDATE`/`DELETE` et renvoie un `sql.Result` :

```go
res, err := db.ExecContext(ctx,
    "INSERT INTO produits (nom, prix, categorie) VALUES (?, ?, ?)",
    "Clavier", 49.90, "peripheriques",
)
if err != nil {
    return err
}

nouvelId, _ := res.LastInsertId()   // identifiant auto-incrémenté
nbLignes, _ := res.RowsAffected()   // lignes modifiées
```

## Types, dates et valeurs NULL

Deux pièges spécifiques à Go méritent une attention particulière.

**Les dates et `parseTime`.** Sans l'option `parseTime=true` (ou `ParseTime: true` dans `mysql.Config`), les colonnes `DATE`, `DATETIME` et `TIMESTAMP` sont renvoyées sous forme de `[]byte`/chaîne. Avec cette option, le pilote les convertit en `time.Time`, et le paramètre `loc` fixe le fuseau d'interprétation. C'est presque toujours le comportement souhaité.

**Les valeurs NULL.** Scanner une valeur `NULL` dans un type Go non *nullable* (comme `string` ou `int64`) provoque une erreur. Pour les colonnes pouvant être nulles, on utilise les types dédiés (`sql.NullString`, `sql.NullInt64`, `sql.NullTime`, etc.) ou des pointeurs :

```go
var description sql.NullString
err := db.QueryRowContext(ctx,
    "SELECT description FROM produits WHERE id = ?", id,
).Scan(&description)

if description.Valid {
    fmt.Println(description.String) // valeur présente
}
```

## Transactions

`Begin`/`BeginTx` ouvre une transaction (`*sql.Tx`) sur laquelle on exécute les requêtes, avant `Commit` ou `Rollback` :

```go
tx, err := db.BeginTx(ctx, nil)
if err != nil {
    return err
}

if _, err := tx.ExecContext(ctx,
    "UPDATE comptes SET solde = solde - ? WHERE id = ?", montant, source); err != nil {
    tx.Rollback()
    return err
}
if _, err := tx.ExecContext(ctx,
    "UPDATE comptes SET solde = solde + ? WHERE id = ?", montant, destination); err != nil {
    tx.Rollback()
    return err
}

return tx.Commit()
```

## Prepared statements et performance

Lorsqu'on passe des paramètres à `Query`/`Exec`, le pilote effectue par défaut un cycle de préparation côté serveur, ce qui est sûr. L'option DSN **`interpolateParams=true`** demande au pilote d'interpoler les valeurs côté client (en les échappant correctement), économisant un aller-retour réseau au prix de l'abandon de la préparation serveur. Pour réutiliser une même requête de nombreuses fois, `db.Prepare()` renvoie un `*sql.Stmt` réutilisable.

Dans tous les cas, la sécurité repose sur l'usage des **marqueurs `?`** : ne jamais concaténer de valeurs dans la chaîne SQL (§17.8, §17.9).

## Authentification et MariaDB 12.3

- **`caching_sha2_password`** (§10.5.5) est pris en charge par le pilote, ce qui simplifie la connexion à un MariaDB 12.3 utilisant ce plugin hérité de MySQL.
- **`ed25519`** (§10.5.2) et **PARSEC** (§10.6), propres à MariaDB, ne sont pas reconnus nativement par ce pilote orienté MySQL : leur usage nécessite un *plugin* d'authentification tiers enregistré auprès du pilote. Si ces plugins sont imposés, ce point est à valider en amont.

Le **chiffrement TLS** se configure en enregistrant une configuration nommée, puis en la référençant dans le DSN :

```go
import "crypto/tls"

mysql.RegisterTLSConfig("custom", &tls.Config{ /* RootCAs, etc. */ })
// puis dans le DSN : ...?tls=custom
// valeurs intégrées également disponibles : tls=true, tls=skip-verify, tls=preferred
```

Voir §10.7 pour la mise en place TLS côté serveur.

---

## Ce qu'il faut retenir

- L'accès se fait via la bibliothèque standard **`database/sql`** + le pilote **`go-sql-driver/mysql`** (pur Go), enregistré sous le nom **`"mysql"`**.
- **`*sql.DB` est un pool**, pas une connexion : l'ouvrir une fois, le partager entre *goroutines*, configurer ses limites (§17.2).
- **`sql.Open` ne connecte pas** ; vérifier avec `db.Ping()`.
- Toujours **`defer rows.Close()`**, scanner dans des pointeurs, contrôler `rows.Err()`, et tester `sql.ErrNoRows` pour `QueryRow`.
- Activer **`parseTime=true`** pour obtenir des `time.Time`, et utiliser **`sql.NullXxx`** pour les colonnes *nullable*.
- Marqueur `?` exclusivement (§17.8, §17.9) ; les variantes `…Context` pour les délais et l'annulation.
- `caching_sha2_password` fonctionne ; `ed25519`/PARSEC requièrent un plugin tiers. TLS via `RegisterTLSConfig` + `?tls=`.
