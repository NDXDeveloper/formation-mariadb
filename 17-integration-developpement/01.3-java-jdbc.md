# 17.1.3 Java : JDBC, MariaDB Connector/J

En Java, l'accès aux bases de données relationnelles repose sur **JDBC** (*Java Database Connectivity*), l'API standard livrée avec le JDK. JDBC est **neutre vis-à-vis du fournisseur** : le code s'écrit contre l'API (`java.sql`, `javax.sql`), et l'on branche un **pilote** spécifique sous la forme d'une bibliothèque (un JAR). Pour MariaDB, ce pilote est **MariaDB Connector/J**, écrit entièrement en Java — il appartient donc à la famille des implémentations natives au langage (cf. §17.1).

Cette section couvre JDBC au niveau brut. L'usage d'un ORM tel que Hibernate/JPA, qui s'appuie sur JDBC, est traité au §17.3.1.

---

## JDBC : l'API standard

JDBC fait partie du JDK : aucune dépendance à ajouter pour l'API elle-même. Seul le **pilote** doit être fourni. Depuis JDBC 4.0 (Java 6), les pilotes s'enregistrent automatiquement via le mécanisme de *service provider* (`META-INF/services`) : l'ancien appel `Class.forName("org.mariadb.jdbc.Driver")` n'est **plus nécessaire**, la simple présence du JAR sur le *classpath* suffit.

---

## Installer Connector/J

Le pilote s'ajoute comme dépendance du projet.

**Maven :**

```xml
<dependency>
    <groupId>org.mariadb.jdbc</groupId>
    <artifactId>mariadb-java-client</artifactId>
    <version>3.x.x</version> <!-- utiliser la dernière version stable -->
</dependency>
```

**Gradle :**

```gradle
implementation 'org.mariadb.jdbc:mariadb-java-client:3.x.x'
```

---

## L'URL JDBC

La connexion se décrit par une URL de la forme :

```
jdbc:mariadb://hôte:port/base?option1=valeur1&option2=valeur2
```

Par exemple : `jdbc:mariadb://localhost:3306/boutique?useServerPrepStmts=true`.

> 💡 **Compatibilité** : MariaDB Connector/J est conçu pour MariaDB tout en restant compatible MySQL. Pour migrer une application existante, il suffit le plus souvent de remplacer le pilote et de changer le préfixe d'URL de `jdbc:mysql://` en `jdbc:mariadb://`.

---

## Obtenir une connexion

Deux approches coexistent :

- **`DriverManager`** — la plus directe, pratique pour un script ou un apprentissage ;
- **`DataSource`** (`javax.sql.DataSource`) — l'approche **recommandée en production** : elle permet le *pooling* de connexions et l'intégration à un annuaire JNDI.

```java
import java.sql.Connection;
import java.sql.DriverManager;

String url = "jdbc:mariadb://localhost:3306/boutique";
Connection conn = DriverManager.getConnection(url, "app_user", motDePasse);
```

Le pooling via `DataSource` est abordé plus bas et détaillé au §17.2.

## PreparedStatement et ResultSet

Le cœur de JDBC repose sur trois objets : `Connection`, `PreparedStatement` (requête paramétrée) et `ResultSet` (jeu de résultats). On évite `Statement` avec des données utilisateur, car la concaténation expose aux injections. Tous ces objets implémentent `AutoCloseable` : le **try-with-resources** garantit leur fermeture, y compris en cas d'exception.

```java
import java.math.BigDecimal;
import java.sql.*;

String url = "jdbc:mariadb://localhost:3306/boutique";
String sql = "SELECT id, nom, prix FROM produits WHERE categorie = ? AND prix < ?";

try (Connection conn = DriverManager.getConnection(url, "app_user", motDePasse);
     PreparedStatement stmt = conn.prepareStatement(sql)) {

    stmt.setString(1, categorie);        // les index commencent à 1
    stmt.setBigDecimal(2, prixMax);

    try (ResultSet rs = stmt.executeQuery()) {
        while (rs.next()) {
            String nom = rs.getString("nom");
            BigDecimal prix = rs.getBigDecimal("prix");
            System.out.printf("%s : %.2f €%n", nom, prix);
        }
    }
}
```

Quelques points de méthode : les marqueurs sont positionnels (`?`) et les index débutent à **1** ; le type `DECIMAL` se lit en `BigDecimal` (à préférer à `double` pour les montants) ; les colonnes se lisent par nom ou par position.

## Écritures et clés générées

Pour récupérer l'identifiant auto-incrémenté produit par un `INSERT`, on déclare l'intention à la préparation, puis on lit `getGeneratedKeys()` :

```java
String sql = "INSERT INTO produits (nom, prix, categorie) VALUES (?, ?, ?)";

try (Connection conn = DriverManager.getConnection(url, "app_user", motDePasse);
     PreparedStatement stmt =
         conn.prepareStatement(sql, Statement.RETURN_GENERATED_KEYS)) {

    stmt.setString(1, "Clavier");
    stmt.setBigDecimal(2, new BigDecimal("49.90"));
    stmt.setString(3, "peripheriques");
    stmt.executeUpdate();

    try (ResultSet keys = stmt.getGeneratedKeys()) {
        if (keys.next()) {
            long nouvelId = keys.getLong(1);
        }
    }
}
```

## Transactions

Par défaut, une connexion JDBC est en **autocommit** : chaque instruction est validée immédiatement. Pour grouper plusieurs opérations, on désactive l'autocommit et on encadre par `commit()` / `rollback()` :

```java
try (Connection conn = DriverManager.getConnection(url, "app_user", motDePasse)) {
    conn.setAutoCommit(false);
    try (PreparedStatement debit = conn.prepareStatement(
             "UPDATE comptes SET solde = solde - ? WHERE id = ?");
         PreparedStatement credit = conn.prepareStatement(
             "UPDATE comptes SET solde = solde + ? WHERE id = ?")) {

        debit.setBigDecimal(1, montant);
        debit.setLong(2, source);
        debit.executeUpdate();

        credit.setBigDecimal(1, montant);
        credit.setLong(2, destination);
        credit.executeUpdate();

        conn.commit();
    } catch (SQLException e) {
        conn.rollback();
        throw e;
    }
}
```

À noter que ce comportement (autocommit *activé* par défaut) est l'inverse de celui des pilotes Python vus au §17.1.2.

## Pooling et DataSource

En production, on n'ouvre jamais une connexion par requête : on passe par un **pool**. La solution de référence dans l'écosystème Java est **HikariCP** (utilisé par défaut par Spring Boot). On l'alimente avec la même URL JDBC :

```java
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;

HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:mariadb://localhost:3306/boutique");
config.setUsername("app_user");
config.setPassword(motDePasse);
config.setMaximumPoolSize(10);

HikariDataSource dataSource = new HikariDataSource(config);

try (Connection conn = dataSource.getConnection()) {
    // ... ici, conn.close() rend la connexion au pool (ne la ferme pas physiquement)
}
```

MariaDB Connector/J fournit aussi son propre pool (`MariaDbPoolDataSource`), mais HikariCP reste le choix le plus répandu. Les principes du *pooling* (dimensionnement, *timeouts*, connexions périmées) sont développés au §17.2.

## Options et fonctionnalités de Connector/J

Plusieurs options d'URL ajustent le comportement du pilote :

- **`useServerPrepStmts=true`** — utilise de vrais *prepared statements* côté serveur (par défaut, le pilote les gère côté client) ; souvent associé à **`cachePrepStmts=true`** pour les mettre en cache ;
- **`connectTimeout`** et **`socketTimeout`** — délais d'établissement et de lecture ;
- **`sessionVariables`** — fixe des variables de session à la connexion.

Connector/J prend également en charge des **modes haute disponibilité** directement dans l'URL, en insérant un mot-clé après `mariadb` :

- `jdbc:mariadb:replication://primaire:3306,replica1:3306,replica2:3306/boutique` — écritures vers le primaire, lectures réparties sur les réplicas (voir chap. 13) ;
- `jdbc:mariadb:loadbalance://nœud1:3306,nœud2:3306/boutique` — répartition de charge entre plusieurs nœuds (utile avec Galera, chap. 14) ;
- `jdbc:mariadb:sequential://...` — bascule (*failover*) dans l'ordre des hôtes déclarés.

## Authentification et MariaDB 12.3

Atout du pilote officiel : MariaDB Connector/J embarque les plugins d'authentification propres à MariaDB.

- **`ed25519`** (§10.5.2) est pris en charge **nativement** — un avantage net sur les pilotes génériques.
- **`caching_sha2_password`** (§10.5.5) est également géré, ce qui simplifie la connexion à un MariaDB 12.3 configuré avec ce plugin issu de MySQL.
- **PARSEC** (§10.6) étant récent, vérifier qu'il est couvert par la version du pilote employée.

Le chiffrement TLS se règle via l'option **`sslMode`** (`DISABLE`, `TRUST`, `VERIFY_CA`, `VERIFY_FULL`), complétée par `serverSslCert` pour fournir le certificat de l'autorité (voir §10.7).

---

## Connector/J ou MySQL Connector/J ?

Les deux pilotes fonctionnent avec MariaDB, mais **MariaDB Connector/J** est optimisé pour le serveur et expose ses fonctionnalités spécifiques (authentification `ed25519`, modes HA dans l'URL, comportements MariaDB). C'est le choix recommandé dès lors que la cible est MariaDB.

---

## Ce qu'il faut retenir

- **JDBC** est l'API standard du JDK ; **MariaDB Connector/J** est le pilote pur Java qui l'implémente pour MariaDB (à ajouter via Maven/Gradle).
- L'enregistrement du pilote est automatique depuis JDBC 4.0 : `Class.forName(...)` est inutile.
- URL : `jdbc:mariadb://hôte:port/base?options` ; migration depuis MySQL souvent limitée au changement de préfixe.
- Utiliser **`PreparedStatement`** (jamais `Statement` avec des entrées utilisateur) et le **try-with-resources** pour fermer connexions, *statements* et *result sets*.
- L'**autocommit est activé par défaut** (l'inverse des pilotes Python) : le désactiver pour grouper des opérations.
- En production, passer par un **pool** (HikariCP) et un `DataSource` (§17.2).
- Connector/J apporte `ed25519` natif, des **modes HA** dans l'URL (chap. 13, 14) et l'option TLS `sslMode` (§10.7).
