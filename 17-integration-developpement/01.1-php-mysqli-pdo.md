# 17.1.1 PHP : mysqli et PDO

PHP dispose de deux extensions modernes pour dialoguer avec MariaDB : **mysqli** et **PDO** (via son pilote `PDO_mysql`). Toutes deux exposent les mêmes capacités fondamentales — connexion, requêtes, *prepared statements*, transactions — mais avec des philosophies différentes. L'ancienne extension `mysql_*`, supprimée depuis PHP 7, ne doit plus jamais être utilisée : elle ne supportait pas les *prepared statements* et exposait directement aux injections SQL.

> Cette section illustre les API au moyen d'exemples de code. Les bonnes pratiques transversales (injections, *prepared statements*) sont approfondies en §17.8 et §17.9 ; le *pooling* en §17.2.

---

## Le pilote sous-jacent : mysqlnd

Avant de choisir entre mysqli et PDO, il faut savoir qu'elles s'appuient toutes deux sur un pilote bas niveau. Depuis PHP 5.4, le pilote par défaut est **mysqlnd** (*MySQL Native Driver*) : écrit en C, intégré à PHP, il ne nécessite aucune bibliothèque externe. C'est lui qui parle le protocole compatible MySQL/MariaDB.

Une alternative existe (compilation contre `libmysqlclient` / `libmariadb`), mais elle est rare. Dans la quasi-totalité des installations modernes, on utilise mysqlnd. Ce détail a son importance pour quelques fonctionnalités (la méthode `get_result()` de mysqli, par exemple, requiert mysqlnd) et pour la prise en charge des plugins d'authentification.

---

## L'extension mysqli

`mysqli` (*MySQL improved*) est spécifique à MySQL/MariaDB. Elle offre deux styles : **orienté objet** (recommandé) et **procédural**. Les exemples ci-dessous utilisent le style objet.

### Connexion

```php
<?php
// Depuis PHP 8.1, mysqli signale les erreurs par des exceptions.
// On l'active explicitement pour les versions antérieures.
mysqli_report(MYSQLI_REPORT_ERROR | MYSQLI_REPORT_STRICT);

$conn = new mysqli(
    'localhost',                 // hôte
    'app_user',                  // utilisateur
    getenv('DB_PASSWORD'),       // mot de passe (jamais en dur)
    'boutique',                  // base par défaut
    3306                         // port
);

// Toujours fixer le jeu de caractères de la session
$conn->set_charset('utf8mb4');
```

Avant PHP 8.1, l'échec de connexion ne levait pas d'exception : il fallait tester `$conn->connect_error`. Avec le rapport d'erreurs activé, `new mysqli(...)` lève désormais une `mysqli_sql_exception`, ce qui simplifie la gestion d'erreurs et évite les oublis de contrôle.

### Prepared statements

C'est le point central pour la sécurité et la performance. On prépare la requête avec des marqueurs positionnels `?`, on lie les valeurs, puis on exécute :

```php
$stmt = $conn->prepare(
    'SELECT id, nom, prix FROM produits WHERE categorie = ? AND prix < ?'
);

// Types : 's' = string, 'i' = integer, 'd' = double, 'b' = blob
$stmt->bind_param('sd', $categorie, $prixMax);
$stmt->execute();

$result = $stmt->get_result();              // nécessite mysqlnd
while ($row = $result->fetch_assoc()) {
    printf("%s : %.2f €\n", $row['nom'], $row['prix']);
}

$stmt->close();
$conn->close();
```

Pour une insertion, l'identifiant auto-incrémenté généré est accessible via `$conn->insert_id` :

```php
$stmt = $conn->prepare('INSERT INTO produits (nom, prix, categorie) VALUES (?, ?, ?)');
$stmt->bind_param('sds', $nom, $prix, $categorie);
$stmt->execute();
$nouvelId = $conn->insert_id;
```

## L'extension PDO

`PDO` (*PHP Data Objects*) est une couche d'abstraction **portable** : la même API pilote une douzaine de SGBD différents, le pilote `PDO_mysql` se chargeant des spécificités de MariaDB. PDO n'expose que le style orienté objet.

### DSN et connexion

La connexion se décrit par une **chaîne DSN**, suivie de l'utilisateur, du mot de passe et d'un tableau d'options :

```php
<?php
$dsn = 'mysql:host=localhost;port=3306;dbname=boutique;charset=utf8mb4';

$options = [
    // Depuis PHP 8.0, ERRMODE_EXCEPTION est déjà le mode par défaut.
    PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
    // Récupérer les lignes sous forme de tableaux associatifs
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
    // Utiliser de vrais prepared statements côté serveur (voir plus bas)
    PDO::ATTR_EMULATE_PREPARES   => false,
];

$pdo = new PDO($dsn, 'app_user', getenv('DB_PASSWORD'), $options);
```

Noter que le jeu de caractères se déclare directement dans le DSN (`charset=utf8mb4`).

### Prepared statements

PDO accepte deux formes de marqueurs : **positionnels** (`?`) et **nommés** (`:nom`). Les marqueurs nommés améliorent la lisibilité des requêtes complexes :

```php
$stmt = $pdo->prepare(
    'SELECT id, nom, prix FROM produits WHERE categorie = :cat AND prix < :max'
);
$stmt->execute(['cat' => $categorie, 'max' => $prixMax]);

// L'objet statement est itérable directement
foreach ($stmt as $row) {
    printf("%s : %.2f €\n", $row['nom'], $row['prix']);
}
```

Pour une insertion et la récupération de l'identifiant généré :

```php
$stmt = $pdo->prepare('INSERT INTO produits (nom, prix, categorie) VALUES (?, ?, ?)');
$stmt->execute(['Clavier', 49.90, 'peripheriques']);
$nouvelId = $pdo->lastInsertId();
```

### Prepared statements émulés ou natifs

Par défaut, `PDO_mysql` **émule** les *prepared statements* : il construit la requête finale côté client en échappant les valeurs, sans solliciter le mécanisme de préparation du serveur. En passant `PDO::ATTR_EMULATE_PREPARES` à `false`, on utilise de **vrais** *prepared statements* préparés par le serveur.

Les deux approches protègent contre les injections dès lors que l'on passe par des paramètres liés. La désactivation de l'émulation est néanmoins souvent recommandée : elle confie au serveur le typage des valeurs et évite certaines surprises subtiles. À noter qu'avec l'émulation activée, une même requête peut accepter le même marqueur nommé plusieurs fois, ce que la préparation native n'autorise pas.

### Transactions

PDO offre une API claire pour les transactions, à encadrer systématiquement par une gestion d'erreurs :

```php
$pdo->beginTransaction();
try {
    $pdo->prepare('UPDATE comptes SET solde = solde - ? WHERE id = ?')
        ->execute([100, $source]);
    $pdo->prepare('UPDATE comptes SET solde = solde + ? WHERE id = ?')
        ->execute([100, $destination]);
    $pdo->commit();
} catch (PDOException $e) {
    $pdo->rollBack();
    throw $e;
}
```

## mysqli ou PDO ?

| Critère | mysqli | PDO |
|---------|--------|-----|
| Bases supportées | MySQL/MariaDB uniquement | ~12 SGBD (code portable) |
| Style d'API | objet **et** procédural | objet uniquement |
| Marqueurs | positionnels (`?`) | positionnels **et** nommés (`:nom`) |
| Fonctions spécifiques au serveur | oui | non (dénominateur commun) |
| Prepared statements natifs | oui | oui (émulés par défaut) |
| Requêtes multiples | oui | limité |

En pratique :

- **PDO** est le choix par défaut pour un nouveau projet : portabilité, marqueurs nommés, API homogène, intégration naturelle avec la plupart des frameworks PHP modernes.
- **mysqli** se justifie lorsqu'on a besoin de fonctionnalités propres à MySQL/MariaDB qu'elle expose directement, ou lorsqu'on intervient sur une base de code existante déjà bâtie autour d'elle.

Dans les deux cas, le critère décisif n'est pas l'extension mais l'usage systématique des *prepared statements*.

## Authentification et MariaDB 12.3

Le plugin d'authentification du compte (chap. 10) doit être pris en charge par PHP :

- **`mysql_native_password`** et **`caching_sha2_password`** (§10.5.5) sont gérés par mysqlnd. Comme MariaDB 12.3 prend désormais en charge `caching_sha2_password` côté serveur, un compte configuré ainsi se connecte sans difficulté depuis PHP — ce qui est commode lors d'une migration depuis MySQL. Une connexion TLS simplifie l'échange initial requis par ce plugin.
- **`ed25519`** (§10.5.2) et **PARSEC** (§10.6) ne sont pas systématiquement reconnus par mysqlnd : leur prise en charge peut nécessiter un *plugin* client ou une compilation sur `libmariadb`. À vérifier selon l'environnement avant de choisir ces plugins pour des comptes utilisés par une application PHP.

Pour le chiffrement de la connexion, mysqli (`mysqli_ssl_set()` / options) comme PDO (options `PDO::MYSQL_ATTR_SSL_*`) permettent de fournir certificats et CA ; se reporter au §10.7.

## Connexions persistantes et pooling

Le modèle d'exécution classique de PHP (mod_php, PHP-FPM) ouvre et ferme généralement une connexion **par requête HTTP**. PHP ne maintient pas, nativement, un pool de connexions à long terme comme le font Java ou Node.js.

Deux leviers atténuent ce coût :

- les **connexions persistantes** — préfixe `p:` sur l'hôte avec mysqli, ou `PDO::ATTR_PERSISTENT => true` avec PDO — réutilisent une connexion existante entre requêtes au sein d'un même *worker*. À manier avec prudence : un état de session résiduel (variables, transaction non terminée) peut « fuir » d'une requête à l'autre ;
- un **pooler externe** tel que ProxySQL, placé entre l'application et le serveur, mutualise les connexions à grande échelle.

Ces stratégies sont détaillées en §17.2.

---

## Ce qu'il faut retenir

- mysqli et PDO sont les deux seules extensions à employer ; `mysql_*` est supprimée.
- Toutes deux reposent en général sur **mysqlnd**, sans dépendance externe.
- **PDO** est recommandé par défaut (portable, marqueurs nommés) ; **mysqli** convient pour les besoins spécifiques MySQL/MariaDB ou le code existant.
- Activez les exceptions (par défaut depuis PHP 8.0 pour PDO, PHP 8.1 pour mysqli) et fixez toujours `utf8mb4`.
- Avec PDO, désactiver l'émulation (`ATTR_EMULATE_PREPARES => false`) pour de vrais *prepared statements* côté serveur.
- `caching_sha2_password` fonctionne depuis PHP ; pour `ed25519`/PARSEC, vérifier la prise en charge du build.
- La sécurité repose sur l'usage systématique des *prepared statements* (§17.8, §17.9), pas sur le choix de l'extension.
