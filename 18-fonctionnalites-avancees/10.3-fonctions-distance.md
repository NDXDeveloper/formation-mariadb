# 18.10.3 Fonctions de distance (VEC_DISTANCE_EUCLIDEAN, VEC_DISTANCE_COSINE)

> **Version** : disponible depuis MariaDB 11.7, GA en 11.8 LTS, standard en 12.3 LTS (optimisations Vector en 12.3, §18.10.7).

Toute recherche vectorielle repose sur une mesure de **distance** entre vecteurs : plus deux vecteurs sont proches, plus les contenus qu'ils représentent se ressemblent (§18.10). MariaDB expose trois fonctions pour cela — deux explicites, `VEC_DISTANCE_EUCLIDEAN` et `VEC_DISTANCE_COSINE`, et une générique, `VEC_DISTANCE`, qui s'adapte à l'index. Elles font le pont entre le type `VECTOR` (§18.10.1) et l'index HNSW (§18.10.2).

## Les deux métriques

| Fonction | Calcule | Plage de valeurs | Métrique idéale pour |
|----------|---------|------------------|----------------------|
| `VEC_DISTANCE_EUCLIDEAN` | la distance euclidienne (L2), « à vol d'oiseau » | [0, +∞) | données où la magnitude porte du sens |
| `VEC_DISTANCE_COSINE` | la distance cosinus | [0, 2] | embeddings de texte, recherche sémantique |

Formellement, pour deux vecteurs `a` et `b` de même longueur :

- Distance euclidienne : `d = √( Σᵢ (aᵢ − bᵢ)² )` — elle vaut 0 pour deux vecteurs identiques et croît avec leur écart.
- Distance cosinus : `d = 1 − (a · b) / (‖a‖ · ‖b‖)`, où `a · b` est le produit scalaire. Elle vaut 0 quand les vecteurs pointent dans la même direction, 1 quand ils sont orthogonaux, 2 quand ils sont opposés — indépendamment de leur magnitude.

Dans tous les cas, **une valeur plus petite signifie « plus proche », donc plus similaire**. Attention à ne pas confondre la *distance cosinus* renvoyée par MariaDB avec la *similarité cosinus* employée ailleurs : les deux sont liées par `distance = 1 − similarité`.

Le choix entre les deux relève surtout de la nature des embeddings (voir §18.10.2) : `cosine` pour le texte, `euclidean` quand la magnitude compte. Pour des vecteurs **normalisés**, les deux classent les résultats à l'identique et sont aussi rapides l'une que l'autre dans l'implémentation MariaDB. (Il n'existe volontairement pas de fonction de produit scalaire : ce n'est pas une vraie distance et elle n'apporterait aucun gain ici.)

## Signature et valeur de retour

Chaque fonction prend **deux arguments `VECTOR` de même longueur** et renvoie la distance sous forme de nombre à virgule flottante. Les arguments peuvent être une colonne, une valeur binaire ou le résultat de `VEC_FromText()` (§18.10.4).

```sql
-- Distance entre deux vecteurs littéraux
SELECT VEC_DISTANCE_COSINE(VEC_FromText('[1,2,3]'), VEC_FromText('[3,5,7]'));
-- => 0.00258509695694209   (vecteurs presque colinéaires : distance quasi nulle)
```

Deux règles de comportement sont à connaître :

- **Longueurs différentes → `NULL`.** La distance entre des vecteurs de dimensions différentes n'est pas définie ; la fonction renvoie `NULL` plutôt qu'une erreur.
- **Métrique ≠ index → balayage complet.** Si la fonction appelée ne correspond pas à la métrique de l'index existant (par exemple `VEC_DISTANCE_EUCLIDEAN` sur un index `cosine`), l'index n'est pas employé : MariaDB effectue un balayage complet et calcule la distance ligne à ligne. Le résultat reste **exact**, mais la requête est plus lente.

## La fonction générique VEC_DISTANCE

`VEC_DISTANCE(a, b)` ne fixe pas la métrique : elle se comporte comme `VEC_DISTANCE_EUCLIDEAN` ou `VEC_DISTANCE_COSINE` **selon le type de l'index** sous-jacent. Elle évite ainsi de coder la métrique en dur dans les requêtes — pratique si l'on veut pouvoir changer la métrique de l'index sans réécrire le SQL.

Sa contrepartie : elle a **besoin d'un index pour déterminer la métrique**. Si celui-ci est introuvable, MariaDB renvoie l'erreur :

```
ERROR 4206 (HY000): Cannot determine distance type for VEC_DISTANCE, index is not found
```

C'est la différence majeure avec les fonctions explicites : `VEC_DISTANCE_EUCLIDEAN` et `VEC_DISTANCE_COSINE` fonctionnent même sans index (par balayage complet), alors que `VEC_DISTANCE` exige un index.

## Deux usages : recherche indexée et calcul direct

Ces fonctions servent dans deux contextes distincts.

**Recherche des plus proches voisins (avec index).** C'est l'usage central : on trie sur la distance à un vecteur de requête et on limite le nombre de résultats. L'optimiseur déclenche alors l'index HNSW (§18.10.2) — à condition que l'appel soit littéral, le tri **croissant** et un `LIMIT` présent :

```sql
SELECT doc_id, titre
FROM documents
ORDER BY VEC_DISTANCE_COSINE(embedding, VEC_FromText('[...]'))
LIMIT 5;
```

La même mécanique se combine avec des filtres relationnels pour une **recherche hybride** :

```sql
SELECT doc_id, titre
FROM documents
WHERE proprietaire_id = 1234
ORDER BY VEC_DISTANCE_COSINE(embedding, VEC_FromText('[...]'))
LIMIT 10;
```

(Rappel du §18.10.2 : en présence d'un `WHERE`, le `LIMIT` plafonne le nombre de lignes remontées de l'index *avant* filtrage — à dimensionner généreusement.)

**Calcul direct (sans index).** On peut aussi évaluer une distance ponctuelle entre deux vecteurs, ou trier sans `LIMIT` : MariaDB calcule alors la distance sur chaque ligne, ce qui donne un résultat **exact** (un véritable *k*-NN, et non sa version approximative) mais impose un parcours complet. C'est utile sur de petits volumes ou pour mesurer la qualité d'une recherche approximative.

```sql
-- Classement exact, sans recours à l'index (balayage complet)
SELECT doc_id,
       VEC_DISTANCE_EUCLIDEAN(embedding, VEC_FromText('[...]')) AS distance
FROM documents
ORDER BY distance;
```

## Performance

Le calcul de distance est l'opération la plus sollicitée d'une recherche vectorielle ; il est accéléré par les jeux d'instructions SIMD du processeur (AVX2, AVX-512, ARM, IBM Power10), traités en §18.10.5. La version 12.3 y ajoute des optimisations spécifiques — distance par extrapolation et rapprochement du calcul au niveau du moteur de stockage — détaillées en §18.10.7.

## À retenir

- Trois fonctions : `VEC_DISTANCE_EUCLIDEAN` (distance L2), `VEC_DISTANCE_COSINE` (1 − similarité cosinus) et la générique `VEC_DISTANCE`.
- Elles prennent deux vecteurs **de même longueur** (sinon `NULL`) et renvoient une distance ; **plus petit = plus proche**.
- `VEC_DISTANCE` s'adapte à la métrique de l'index, mais **échoue (erreur 4206) sans index** ; les fonctions explicites, elles, fonctionnent par balayage complet à défaut.
- La fonction appelée doit **correspondre à la métrique de l'index** pour que celui-ci soit utilisé ; sinon le résultat reste exact, mais via un balayage complet.
- Usage indexé = `ORDER BY VEC_DISTANCE_*(col, vec)` croissant + `LIMIT` ; usage direct = calcul exact sans index.
