# 18.10.1 Type de données VECTOR

> **Version** : disponible depuis MariaDB 11.7, GA en 11.8 LTS, standard en 12.3 LTS.

Le type `VECTOR` est la brique de base de MariaDB Vector : c'est lui qui permet de **stocker un embedding** — la liste ordonnée de nombres produite par un modèle d'IA (§18.10) — dans une colonne d'une table InnoDB ordinaire.

## Déclaration

La syntaxe est `VECTOR(N)`, où **`N` est le nombre de dimensions** du vecteur :

```sql
CREATE TABLE documents (
    id        BIGINT UNSIGNED PRIMARY KEY,
    embedding VECTOR(1536) NOT NULL
);
```

Trois règles structurent ce type :

- **La dimension `N` est obligatoire.** Contrairement à un `VARCHAR`, on ne peut pas déclarer un `VECTOR` sans préciser sa taille : MariaDB doit connaître le nombre de composantes pour valider les données et dimensionner l'index.
- **La dimension est fixe pour toute la colonne.** Toutes les valeurs d'une colonne `VECTOR(N)` ont exactement `N` composantes. Une validation intégrée rejette toute tentative d'insérer un vecteur d'une autre longueur.
- **Pas de valeur `DEFAULT`.** Le type `VECTOR` n'accepte pas de clause `DEFAULT` : chaque ligne doit fournir explicitement son vecteur (ou `NULL`, si la colonne l'autorise — voir plus bas).

### Choisir N : l'aligner sur le modèle d'embedding

La valeur de `N` n'est pas un choix libre : elle doit **correspondre exactement à la dimension de sortie du modèle d'embedding** utilisé. Un modèle qui produit des vecteurs de 768 composantes impose une colonne `VECTOR(768)`. À titre indicatif :

| Modèle d'embedding (exemple) | Dimensions (N) |
|------------------------------|----------------|
| all-MiniLM-L6-v2 (Sentence-Transformers) | 384 |
| Cohere embed v3 | 1024 |
| OpenAI text-embedding-3-small | 1536 |
| OpenAI text-embedding-3-large | 3072 |

La dimension maximale prise en charge est de **16 383** — très au-delà des besoins des modèles actuels, dont les sorties dépassent rarement quelques milliers de composantes. (Le vecteur étant stocké en flottants 32 bits, soit 4 octets par dimension, ce plafond correspond à une représentation binaire d'environ 64 Ko.)

## Format de stockage

En interne, MariaDB range une valeur `VECTOR` comme une **chaîne binaire de nombres à virgule flottante IEEE 754 simple précision (`float32`)**, dans l'ordre petit-boutiste (*little-endian*). Chaque composante occupe donc **4 octets**, et une valeur `VECTOR(N)` pèse `4 × N` octets dans la table — par exemple environ 6 Ko pour un `VECTOR(1536)`.

L'index vectoriel, lui, utilise une représentation **plus compacte** : les composantes y sont quantifiées en `int16` (entiers 16 bits, soit ~2 octets par dimension), ce qui offre 15 bits utiles de précision — davantage que les 10 bits d'un `float16` — tout en divisant par deux l'empreinte par rapport au stockage `float32`. Concrètement, un index portant un million de vecteurs de 1 000 dimensions occupe de l'ordre de 2 Go. La structure et le paramétrage de cet index relèvent de l'index HNSW (§18.10.2).

| Élément | Représentation | Taille approximative |
|---------|----------------|----------------------|
| Valeur stockée en table | `float32` (IEEE 754) | 4 octets × N |
| Entrée dans l'index HNSW | `int16` quantifié + graphe | ~2 octets × N (+ surcoût du graphe) |

Le type `VECTOR` est classé parmi les types **binaires** : ce n'est pas du texte lisible. Pour le manipuler sous forme humaine (`'[0.12, 0.98, …]'`), on passe par les fonctions de conversion `VEC_FromText()` et `VEC_ToText()` (§18.10.4).

## Renseigner et lire une colonne VECTOR

Deux écritures équivalentes permettent d'insérer un vecteur.

La forme **textuelle**, la plus lisible, s'appuie sur `VEC_FromText()`, qui attend un tableau JSON de nombres :

```sql
INSERT INTO documents (id, embedding) VALUES
    (1, VEC_FromText('[0.0123, 0.9876, 0.4410, ..., 0.2071]'));
-- valeurs tronquées pour la lisibilité : un vecteur réel comporte ici 1536 nombres
```

La forme **binaire** insère directement la chaîne d'octets `float32`, le plus souvent produite par le pilote ou la bibliothèque côté application (par exemple `array('f', v).tobytes()` en Python) :

```sql
INSERT INTO documents (id, embedding) VALUES
    (2, x'e360d63ebe554f3f...');   -- chaîne binaire tronquée
```

À la lecture, une colonne `VECTOR` renvoie par défaut sa forme binaire ; on la rend lisible avec `VEC_ToText()` :

```sql
SELECT id, VEC_ToText(embedding) FROM documents WHERE id = 1;
```

Le détail de ces fonctions de conversion figure en §18.10.4. La recherche de similarité proprement dite (avec `VEC_DISTANCE_*`) est traitée en §18.10.3.

## VECTOR et l'index : la contrainte NOT NULL

Dans la quasi-totalité des cas, une colonne `VECTOR` est accompagnée d'un **index vectoriel** (`VECTOR INDEX`) — c'est tout l'intérêt de la fonctionnalité. Or **une colonne portant un index vectoriel doit être déclarée `NOT NULL`** :

```sql
CREATE TABLE documents (
    id        BIGINT UNSIGNED PRIMARY KEY,
    embedding VECTOR(1536) NOT NULL,
    VECTOR INDEX (embedding)        -- définition et options détaillées en §18.10.2
);
```

La définition de l'index, ses options (`M`, `DISTANCE = euclidean | cosine`) et son fonctionnement HNSW sont traités séparément en §18.10.2. On retient ici simplement que le type `VECTOR` et son index forment un couple, et que la présence de l'index impose `NOT NULL`.

## Intégration avec le reste du schéma

Le type `VECTOR` est une colonne InnoDB comme une autre : il cohabite naturellement avec des colonnes relationnelles classiques au sein de la même table, participe aux transactions ACID et se prête aux accès concurrents (lectures et écritures simultanées, tous les niveaux d'isolation étant pris en charge). C'est précisément ce qui rend possible la **recherche hybride** évoquée en §18.10 — croiser, dans une même requête, la proximité vectorielle et des filtres métier (`WHERE`, jointures, agrégats) — sans recourir à une base vectorielle séparée.

## À retenir

- `VECTOR(N)` stocke un embedding de **`N` dimensions** ; `N` est **obligatoire**, **fixe** pour la colonne, et doit **coïncider avec la sortie du modèle** d'embedding.
- Les valeurs sont des **flottants 32 bits** (4 octets/dimension) ; l'index les quantifie en **`int16`** pour une empreinte environ deux fois plus faible.
- Le type est **binaire** : on passe par `VEC_FromText()` / `VEC_ToText()` (§18.10.4) pour la forme texte.
- Une colonne indexée doit être **`NOT NULL`** ; le type n'accepte **pas de `DEFAULT`**.
- Dimension maximale : **16 383**, bien au-delà des modèles courants.
