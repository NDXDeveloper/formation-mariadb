# 18.10.6 Intégration avec les LLMs (OpenAI, Claude, LLaMA)

> **Version** : intégrations valables pour MariaDB Vector, GA en 11.8 LTS, standard en 12.3 LTS.

MariaDB Vector **stocke et recherche** des vecteurs (§18.10.1 à §18.10.3) ; il ne **génère** rien. Les modèles d'IA — qui produisent les vecteurs et, le cas échéant, les réponses — vivent **à l'extérieur** de la base. MariaDB est volontairement **agnostique** vis-à-vis de ces modèles : on peut indifféremment l'utiliser avec OpenAI, Anthropic Claude, Meta LLaMA ou tout autre fournisseur. Cette section précise comment ces modèles s'articulent avec la base.

## Deux rôles distincts : modèle d'embedding vs modèle génératif

L'intégration mobilise **deux types de modèles**, qu'il ne faut pas confondre :

| | Modèle d'embedding | Modèle génératif (LLM) |
|---|--------------------|------------------------|
| **Rôle** | transformer un contenu en vecteur | produire une réponse en langage naturel |
| **Quand** | à l'ingestion *et* à la requête | en fin de chaîne (RAG) |
| **Sortie** | un vecteur (`N` flottants) | du texte |
| **Exemples** | OpenAI `text-embedding-3`, Voyage AI, Cohere, Google, ou modèles ouverts (Sentence-Transformers, BGE…) | GPT (OpenAI), Claude (Anthropic), LLaMA (Meta) |
| **Lien avec MariaDB** | son vecteur est stocké et recherché | il consomme les résultats de la recherche |

C'est le modèle d'embedding qui produit la donnée rangée dans la colonne `VECTOR` ; le modèle génératif n'intervient qu'en aval, pour rédiger une réponse à partir du contexte retrouvé.

Les trois fournisseurs cités dans le titre — OpenAI, Claude, LLaMA — sont avant tout des **modèles génératifs**. Point important : **Anthropic ne propose pas de modèle d'embedding** et recommande un fournisseur dédié (Voyage AI). Une chaîne « Claude + MariaDB » utilise donc Claude pour la *génération* et un modèle d'embedding séparé pour produire les vecteurs. OpenAI, à l'inverse, fournit les deux familles (embeddings *et* génération) ; l'écosystème LLaMA s'accompagne quant à lui de nombreux modèles d'embedding ouverts.

## Le partage des responsabilités

Dans une chaîne de type RAG (§18.10), chacun fait ce qu'il sait faire le mieux :

- **Les modèles externes** génèrent les embeddings (à l'ingestion comme à la requête) et, en bout de chaîne, rédigent la réponse.
- **MariaDB** stocke les vecteurs, les indexe (HNSW, §18.10.2), calcule les distances (§18.10.3) et renvoie les plus proches voisins — accompagnés, dans la même requête, des **données relationnelles** associées (texte source, métadonnées, filtres métier).

C'est tout l'intérêt de garder les vecteurs *dans* la base : la recherche de similarité et le filtrage relationnel se font d'un seul tenant, sans système vectoriel séparé (§18.10).

## Contraintes d'intégration

Trois règles, déjà esquissées au §18.10.1, conditionnent une intégration correcte :

- **Un seul et même modèle d'embedding** doit servir à l'indexation et à l'interrogation : des vecteurs issus de modèles différents ne sont pas comparables.
- **La dimension de la colonne `VECTOR(N)` doit égaler la dimension de sortie du modèle** (par exemple `VECTOR(1536)` pour `text-embedding-3-small`).
- **Changer de modèle d'embedding impose de tout ré-encoder** : les anciens vecteurs deviennent inexploitables avec le nouveau modèle.

Le modèle génératif, lui, est librement interchangeable : il ne touche pas aux vecteurs, on peut passer d'un LLM à un autre sans rien réindexer.

## Le flux applicatif

Concrètement, l'application orchestre les appels aux modèles externes et les requêtes SQL. En pseudo-code Python (les détails de connexion sont omis) :

```python
# 1) Ingestion : encoder puis stocker
vec = embed_model.embed(texte)                 # modèle d'embedding externe
cur.execute(
    "INSERT INTO documents (contenu, embedding) VALUES (%s, VEC_FromText(%s))",
    (texte, json.dumps(vec)),
)

# 2) Interrogation : rechercher puis générer
qvec = embed_model.embed(question)             # LE MÊME modèle d'embedding
cur.execute(
    "SELECT contenu FROM documents "
    "ORDER BY VEC_DISTANCE_COSINE(embedding, VEC_FromText(%s)) LIMIT 5",
    (json.dumps(qvec),),
)
contexte = [row[0] for row in cur]
reponse = llm.generate(question, contexte)     # LLM : OpenAI, Claude, LLaMA…
```

On y retrouve les briques des sections précédentes : `VEC_FromText` pour transmettre le vecteur (§18.10.4), `VEC_DISTANCE_COSINE` avec `ORDER BY … LIMIT` pour la recherche indexée (§18.10.2, §18.10.3).

## Intégration via les frameworks d'IA

En pratique, on écrit rarement ce flux à la main : des frameworks l'orchestrent (découpage, embeddings, recherche, synthèse par le LLM) et exposent MariaDB comme magasin de vecteurs. Les intégrations officielles couvrent les principaux écosystèmes :

| Framework | Langage | Intégration MariaDB |
|-----------|---------|---------------------|
| **LangChain** | Python / JS | `langchain-mariadb` (`MariaDBStore`) |
| **LangChain4j** | Java | MariaDB Embedding Store |
| **LlamaIndex** | Python | MariaDB Vector Store |
| **Spring AI** | Java | MariaDB Vector Store |
| **MCP** (*Model Context Protocol*) | Python | MariaDB MCP Server (§20.10) |

Avec ces frameworks, on manipule une API de haut niveau plutôt que le SQL vectoriel : il suffit de leur indiquer la **dimension** des embeddings, qui doit coïncider avec le `VECTOR(N)` de la table (par exemple `embedding_length=1536`). Ces intégrations sont développées au §20.11.

## MariaDB MCP Server

Le **MCP Server** de MariaDB mérite une mention à part : via le *Model Context Protocol*, il expose la base comme une **source de données et d'outils** directement consommable par un agent ou un assistant IA, qui peut alors interroger les données — vectorielles ou relationnelles — sans intégration sur mesure. Ce composant est traité au §20.10.

## À retenir

- MariaDB **stocke et recherche** les vecteurs ; il est **agnostique** quant aux modèles d'IA, qui restent externes et interchangeables.
- Deux rôles à distinguer : le **modèle d'embedding** (contenu → vecteur, à l'index comme à la requête) et le **modèle génératif** (la réponse en RAG).
- Parmi OpenAI, Claude et LLaMA — tous **génératifs** — seul OpenAI fournit aussi des embeddings ; **Anthropic n'a pas de modèle d'embedding** et oriente vers Voyage AI.
- Mêmes contraintes qu'au §18.10.1 : un seul modèle d'embedding, `VECTOR(N)` = dimension du modèle, ré-encodage complet en cas de changement.
- Les **frameworks** (LangChain, LangChain4j, LlamaIndex, Spring AI) et le **MCP Server** simplifient l'intégration (§20.10, §20.11).
