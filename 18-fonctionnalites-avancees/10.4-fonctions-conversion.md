# 18.10.4 Fonctions de conversion (VEC_FromText, VEC_ToText)

> **Version** : disponible depuis MariaDB 11.7, GA en 11.8 LTS, standard en 12.3 LTS.

Une valeur `VECTOR` est stockée sous forme **binaire** — une séquence d'octets `float32` illisible (§18.10.1). Deux fonctions font le pont avec une représentation **textuelle** lisible, le tableau JSON de nombres : `VEC_FromText` (texte → binaire) pour écrire un vecteur, et `VEC_ToText` (binaire → texte) pour le relire. Ce sont des opérations inverses l'une de l'autre.

| Fonction | Sens | Entrée | Sortie |
|----------|------|--------|--------|
| `VEC_FromText` | texte → binaire | tableau JSON de nombres, ex. `'[0.3, 0.5, 0.2, 0.1]'` | `VECTOR` binaire (`float32` *little-endian*, 4 octets/dimension) |
| `VEC_ToText` | binaire → texte | `VECTOR` binaire | tableau JSON de nombres, ex. `[0.3,0.5,0.2,0.1]` |

(Comme toutes les fonctions SQL de MariaDB, leurs noms sont insensibles à la casse : `VEC_FromText`, `vec_fromtext`, `VEC_FROMTEXT` sont équivalents.)

## VEC_FromText : du texte vers le vecteur

`VEC_FromText()` analyse un **tableau JSON de nombres** et le convertit en valeur `VECTOR` binaire (suite d'octets IEEE 754 simple précision, petit-boutiste, 4 octets par composante). C'est la voie la plus lisible pour insérer un vecteur ou construire un vecteur de requête :

```sql
INSERT INTO documents (doc_id, embedding) VALUES
    (1, VEC_FromText('[0.0123, 0.9876, 0.4410, 0.2071]'));
```

Le nombre d'éléments du tableau doit correspondre à la dimension `N` déclarée pour la colonne (`VECTOR(N)`) ; sinon l'insertion échoue (§18.10.1). Le texte doit par ailleurs être un tableau JSON de nombres valide.

La même fonction sert à fabriquer le vecteur de recherche dans une requête de similarité (§18.10.3) :

```sql
SELECT doc_id
FROM documents
ORDER BY VEC_DISTANCE_COSINE(embedding, VEC_FromText('[0.02, 0.97, 0.44, 0.21]'))
LIMIT 5;
```

## VEC_ToText : du vecteur vers le texte

`VEC_ToText()` réalise l'opération inverse : elle convertit un vecteur binaire en **tableau JSON de nombres** lisible. Elle est précieuse pour l'inspection, le débogage ou l'affichage, car une colonne `VECTOR` renvoie sinon ses octets bruts :

```sql
SELECT VEC_ToText(x'e360d63ebe554f3fcdbc523f4522193f5236083d');
-- => [0.418708,0.809902,0.823193,0.598179,0.0332549]
```

Si l'argument n'est **pas** un vecteur binaire valide (par exemple une chaîne d'octets dont la longueur n'est pas un multiple de 4), `VEC_ToText` renvoie `NULL` et lève l'avertissement 4203 :

```sql
SELECT VEC_ToText(x'aabbcc');
-- => NULL
-- Warning (Code 4203): Invalid binary vector format.
--   Must use IEEE standard float representation in little-endian format.
--   Use VEC_FromText() to generate it.
```

## Aller-retour et précision

`VEC_FromText` et `VEC_ToText` sont inverses l'une de l'autre, **à la précision du `float32` près**. Les composantes sont stockées sur 32 bits (~7 chiffres significatifs, §18.10.1) et `VEC_ToText` les restitue avec environ **6 chiffres significatifs** : un aller-retour arrondit donc les valeurs trop précises. Par exemple, insérer la valeur `0.123456789` puis relire la colonne via `VEC_ToText` renvoie `0.123457`. Il ne faut donc pas attendre une restitution exacte au-delà de cette précision — ce qui est sans incidence pour la recherche de similarité, qui raisonne sur des proximités et non sur des égalités exactes.

## Texte ou binaire : quand utiliser quoi

L'écriture et la lecture peuvent se passer de ces fonctions lorsque l'application manipule directement les octets `float32`.

- **À l'écriture**, deux voies équivalentes : `VEC_FromText('[...]')` quand on dispose du vecteur sous forme de texte ou de JSON (lisible, pratique) ; ou l'envoi direct de la chaîne d'octets `float32` par le pilote (par exemple `array('f', v).tobytes()` en Python), qui évite l'analyse du texte et s'avère plus efficace pour les insertions massives.
- **À la lecture**, `VEC_ToText` produit une sortie lisible ; pour la performance, une application décode plutôt les octets bruts renvoyés et les reconstitue côté client.

En pratique : `VEC_FromText` / `VEC_ToText` pour le confort, la lisibilité et le débogage ; la forme binaire directe sur les chemins critiques en performance.

## Compatibilité MySQL

MySQL nomme différemment ces conversions — `STRING_TO_VECTOR()` et `VECTOR_TO_STRING()` — là où MariaDB utilise `VEC_FromText` et `VEC_ToText`. C'est un point d'attention lors d'une migration ou de l'écriture de code portable (voir §19.1).

## À retenir

- Une valeur `VECTOR` est binaire ; `VEC_FromText` (texte → binaire) et `VEC_ToText` (binaire → texte) font le pont avec le tableau JSON de nombres.
- `VEC_FromText` attend un tableau JSON valide dont la taille correspond à `N` ; il sert aussi bien pour l'`INSERT` que pour le vecteur de requête.
- `VEC_ToText` restitue un tableau JSON lisible (~6 chiffres significatifs) ; sur une entrée invalide, il renvoie `NULL` + avertissement 4203.
- L'aller-retour est fidèle **à la précision `float32` près** ; les valeurs trop précises sont arrondies (≈ 6 chiffres significatifs à l'affichage).
- Pour la performance, on peut court-circuiter ces fonctions en échangeant directement les octets `float32` avec l'application.
