# 18.2.3 Partitionnement des données historiques

## Pourquoi partitionner l'historique

Le versionnement système conserve **toutes** les versions d'une ligne (§18.2) : sur une table soumise à de nombreuses écritures, l'historique finit par éclipser largement les données courantes en volume. Deux problèmes en découlent. D'abord la **performance** : sans précaution, une simple lecture des données courantes doit cohabiter avec une masse croissante de lignes périmées dans la même table. Ensuite la **rétention** : purger l'historique ancien par un `DELETE` classique est lent et fragmente le stockage.

Le partitionnement par temps système répond aux deux à la fois. Il **sépare physiquement les lignes courantes des lignes historiques**, et range ces dernières par fenêtres de temps que l'on peut consulter — ou supprimer — indépendamment.

## `PARTITION BY SYSTEM_TIME` : le principe

MariaDB propose un mode de partitionnement dédié aux tables versionnées : `PARTITION BY SYSTEM_TIME`. Il repose sur deux catégories de partitions :

- une **unique partition `CURRENT`**, qui contient toutes les lignes courantes ;
- une ou plusieurs **partitions `HISTORY`**, qui accueillent les lignes devenues historiques.

La partition `CURRENT` doit toujours être déclarée en **dernier**. Lorsqu'une ligne cesse d'être courante (à la suite d'un `UPDATE` ou d'un `DELETE`), elle quitte `CURRENT` pour rejoindre une partition d'historique, choisie selon la règle de répartition retenue.

## Répartir l'historique

### Par intervalle de temps (`INTERVAL`)

C'est la forme la plus utile pour une politique de rétention. Chaque partition d'historique couvre une fenêtre de temps de largeur fixe ; une ligne historique est rangée dans la partition dont la fenêtre contient sa borne de fin. La clause `STARTS` ancre le début de la première fenêtre :

```sql
CREATE TABLE employe (
  id      INT PRIMARY KEY,
  nom     VARCHAR(100),
  salaire DECIMAL(10,2)
) WITH SYSTEM VERSIONING
PARTITION BY SYSTEM_TIME INTERVAL 1 MONTH STARTS '2026-01-01 00:00:00' (
  PARTITION p_2026_01 HISTORY,
  PARTITION p_2026_02 HISTORY,
  PARTITION p_2026_03 HISTORY,
  PARTITION p_courant  CURRENT
);
```

Ici, l'historique de janvier, février et mars est isolé dans trois partitions distinctes, prêtes à être détachées une à une.

### Par nombre de partitions

Plutôt que de nommer chaque partition, on indique un **nombre** de partitions avec `PARTITIONS n` et MariaDB les crée et les nomme automatiquement (`p0`, `p1`, …, la dernière — `pn` — tenant lieu de partition courante). Cette forme se combine à une condition de rotation — **`INTERVAL`** (ou `LIMIT`) —, **obligatoire dès qu'il y a plusieurs partitions d'historique** :

```sql
CREATE TABLE journal (
  id BIGINT PRIMARY KEY,
  message TEXT
) WITH SYSTEM VERSIONING
PARTITION BY SYSTEM_TIME INTERVAL 1 MONTH PARTITIONS 6;
```

Pratique pour démarrer rapidement, elle laisse MariaDB gérer le nommage, mais offre moins de contrôle que la forme explicite. (Omettre la condition de rotation avec plusieurs partitions d'historique déclenche l'erreur *« no rotation condition for multiple HISTORY partitions »*.)

### Par nombre de lignes (`LIMIT`)

`LIMIT` répartit l'historique non par le temps mais par **quantité de lignes** : chaque partition se remplit jusqu'au plafond indiqué avant que la suivante ne prenne le relais.

```sql
CREATE TABLE mesure (
  id BIGINT PRIMARY KEY,
  valeur DOUBLE
) WITH SYSTEM VERSIONING
PARTITION BY SYSTEM_TIME LIMIT 1000000 (
  PARTITION p0 HISTORY,
  PARTITION p1 HISTORY,
  PARTITION p_courant CURRENT
);
```

Cette approche convient quand le rythme d'écriture est irrégulier et qu'un découpage par volume est plus prévisible qu'un découpage par date.

## Le bénéfice à la lecture : l'élagage de partitions

L'intérêt majeur côté performance est l'**élagage de partitions** (*partition pruning*) : l'optimiseur ne lit que les partitions susceptibles de contenir les lignes recherchées.

Une requête sur les données courantes ne touche que la partition `CURRENT`, qui reste compacte quelle que soit la taille de l'historique :

```sql
SELECT * FROM employe WHERE id = 42;   -- partition CURRENT uniquement
```

À l'inverse, une requête temporelle (§18.2.2) cible les partitions dont la fenêtre recouvre la période demandée :

```sql
SELECT * FROM employe
FOR SYSTEM_TIME AS OF '2026-02-15';    -- partition de février
```

`EXPLAIN PARTITIONS` (ou `EXPLAIN`) permet de vérifier quelles partitions sont effectivement parcourues.

## Gérer la rétention : ajouter et supprimer des partitions

C'est ici que le partitionnement transforme la gestion de la rétention.

**Purger l'historique ancien** se fait par simple suppression de partition — une opération de métadonnées quasi instantanée, sans parcours ligne à ligne ni fragmentation :

```sql
ALTER TABLE employe DROP PARTITION p_2026_01;   -- l'historique de janvier disparaît d'un coup
```

À comparer avec la purge applicable en l'absence de partitionnement, `DELETE HISTORY`, qui supprime physiquement les lignes périmées et reste coûteuse à grande échelle :

```sql
DELETE HISTORY FROM employe BEFORE SYSTEM_TIME '2025-01-01';
```

**Étendre la couverture** consiste à provisionner de nouvelles partitions d'historique avant que les fenêtres existantes ne soient dépassées :

```sql
ALTER TABLE employe ADD PARTITION (PARTITION p_2026_04 HISTORY);
```

## Éviter le débordement

Le point de vigilance principal est le **débordement**. Si toutes les partitions d'historique prévues sont saturées — fenêtres temporelles dépassées ou plafond de lignes atteint —, les nouvelles lignes historiques s'accumulent dans la **dernière** partition d'historique, qui croît alors sans limite et perd le bénéfice de l'élagage. MariaDB émet un **avertissement** signalant ce débordement et invitant à ajouter des partitions.

La parade est de provisionner les partitions **à l'avance**, soit manuellement, soit de façon automatisée au moyen d'un Event planifié (§8.4) qui ajoute périodiquement la partition de la fenêtre suivante. Un découpage temporel régulier rend cette automatisation simple à mettre en place.

## Inspecter le partitionnement

`SHOW CREATE TABLE` restitue la définition complète des partitions, et `INFORMATION_SCHEMA.PARTITIONS` fournit le nombre de lignes par partition — utile pour repérer un déséquilibre ou une partition proche de la saturation :

```sql
SELECT partition_name, table_rows
FROM information_schema.PARTITIONS
WHERE table_schema = DATABASE()
  AND table_name = 'employe';
```

## Points clés à retenir

- `PARTITION BY SYSTEM_TIME` sépare une partition **`CURRENT`** (lignes courantes, toujours déclarée en dernier) de partitions **`HISTORY`**.
- La répartition de l'historique se fait par **`INTERVAL`** de temps (idéal pour la rétention), par nombre de partitions (`PARTITIONS n`), ou par nombre de lignes (`LIMIT`).
- L'**élagage de partitions** maintient les lectures de données courantes rapides et cible les fenêtres pertinentes pour les requêtes temporelles.
- Purger revient à un **`DROP PARTITION`** quasi instantané, bien plus efficace qu'un `DELETE HISTORY` ligne à ligne.
- Attention au **débordement** dans la dernière partition : provisionner les partitions à l'avance, au besoin via un Event planifié (§8.4).
- `SHOW CREATE TABLE` et `INFORMATION_SCHEMA.PARTITIONS` permettent de surveiller la configuration et le remplissage.
