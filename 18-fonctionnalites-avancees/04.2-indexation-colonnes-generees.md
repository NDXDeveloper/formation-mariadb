# 18.4.2 Indexation de colonnes générées

## Indexer une valeur calculée

Une colonne générée (§18.4) prend toute sa valeur lorsqu'elle est **indexée** : on indexe alors le *résultat d'un calcul*, et non plus une donnée brute. C'est ainsi que MariaDB obtient l'équivalent d'un **index fonctionnel** — là où d'autres systèmes l'offrent par une syntaxe dédiée, MariaDB s'appuie sur le couple « colonne générée + index ».

Les deux modes vus en §18.4.1 sont indexables. Pour une colonne `STORED`, l'index porte sur une valeur déjà matérialisée et fonctionne comme tout index ordinaire. Pour une colonne `VIRTUAL`, le cas est plus subtil et fait l'objet d'une section à part ci-dessous.

## Comment indexer

L'index se déclare comme pour n'importe quelle colonne — à la création, ou après coup :

```sql
-- à la création
CREATE TABLE evenement (
  id      INT PRIMARY KEY,
  donnees JSON,
  ville   VARCHAR(100) AS (JSON_VALUE(donnees, '$.ville')),
  INDEX idx_ville (ville)
);

-- ou ultérieurement
ALTER TABLE evenement ADD INDEX idx_ville (ville);
CREATE INDEX idx_ville ON evenement (ville);
```

Une colonne générée peut aussi entrer dans un **index composite**, aux côtés de colonnes ordinaires, pour couvrir des requêtes plus complexes.

## Ce que l'indexation apporte

### Accélérer le filtrage sur une expression

Une condition portant sur une *expression* ne peut généralement pas exploiter l'index de la colonne sous-jacente. Filtrer par `YEAR(date_commande)`, par exemple, oblige le serveur à évaluer la fonction ligne par ligne, ce qui conduit le plus souvent à un parcours complet :

```sql
SELECT * FROM commande WHERE YEAR(date_commande) = 2026;   -- parcours complet probable
```

En matérialisant l'expression dans une colonne générée indexée, la recherche redevient indexée :

```sql
ALTER TABLE commande
  ADD COLUMN annee SMALLINT AS (YEAR(date_commande)),
  ADD INDEX idx_annee (annee);

SELECT * FROM commande WHERE annee = 2026;   -- exploite idx_annee
```

### Indexer des champs JSON

L'usage le plus répandu consiste à extraire un attribut d'une colonne JSON dans une colonne générée, puis à l'indexer — comme la colonne `ville` de l'exemple plus haut. La recherche `WHERE ville = 'Rouen'` s'appuie alors sur `idx_ville`, alors qu'un filtrage direct sur le document JSON resterait coûteux. Ce scénario, central pour exploiter efficacement des données semi-structurées, est traité en détail en §4.10.

## Cas des colonnes `VIRTUAL` : matérialisation dans l'index

Une colonne `VIRTUAL` n'est pas stockée dans la ligne — mais **lorsqu'elle est indexée, sa valeur est calculée et conservée dans l'index**. L'expression est évaluée à l'écriture afin de maintenir l'index à jour ; la ligne, elle, reste allégée de cette donnée.

C'est ce qui justifiait la remarque de la §18.4.1 : indexer une colonne `VIRTUAL` procure des lectures rapides *par cet index*, comparables à celles d'une colonne `STORED`, sans en payer le coût de stockage dans la table principale. Le surcoût se déplace simplement vers l'écriture, qui doit évaluer l'expression et entretenir l'index.

## Utiliser l'index pour `GROUP BY` et `ORDER BY`

Au-delà du filtrage, MariaDB sait exploiter un index posé sur une colonne générée pour satisfaire directement un **`GROUP BY`** ou un **`ORDER BY`** portant sur l'expression correspondante : l'index fournit l'ordre voulu, ce qui évite un tri en mémoire (*filesort*) ou une table temporaire. Comme pour tout index, l'optimiseur ne l'emploie toutefois que s'il y trouve un intérêt — typiquement lorsque la requête est **couverte** par l'index (elle ne lit que des colonnes qu'il contient). Une requête qui réclame d'autres colonnes — un `SUM(montant)`, un `SELECT *` — peut, elle, retomber sur un balayage complet suivi d'un tri.

```sql
CREATE TABLE vente (
  id         BIGINT PRIMARY KEY,
  montant    DECIMAL(10,2),
  date_vente DATE,
  annee      SMALLINT AS (YEAR(date_vente)),
  INDEX idx_annee (annee)
);

-- couvert par idx_annee : l'index fournit directement l'ordre des groupes
SELECT annee, COUNT(*) FROM vente GROUP BY annee;

-- et l'ordre du tri
SELECT annee FROM vente ORDER BY annee;
```

L'optimiseur peut même reconnaître qu'une expression écrite explicitement dans la requête **correspond** à la définition d'une colonne générée, et lui substituer cette colonne pour utiliser son index — particulièrement efficace au **filtrage** :

```sql
-- YEAR(date_vente) est rapproché de la colonne « annee » : idx_annee est utilisé
SELECT * FROM vente WHERE YEAR(date_vente) = 2026;
```

Référencer directement la colonne générée reste néanmoins la voie la plus sûre et la plus lisible. Pour confirmer que l'index est bien mis à profit, on inspecte le plan d'exécution :

```sql
EXPLAIN SELECT annee, COUNT(*) FROM vente GROUP BY annee;
-- la colonne « key » doit indiquer idx_annee, et « Extra » afficher « Using index »
-- (ni « Using temporary » ni « Using filesort »)
```

## Points de vigilance

Quelques précautions accompagnent cette technique. L'indexation **ajoute un coût à l'écriture** : l'expression est évaluée et l'index maintenu à chaque modification des colonnes sources — un compromis à mettre en regard du gain en lecture. Pour une colonne issue d'une chaîne de caractères, la **collation** doit être cohérente avec celle des recherches (une collation insensible à la casse peut d'ailleurs rendre inutile un `LOWER()` explicite). Sur une expression produisant un texte long, il peut être nécessaire de **borner la longueur** de la colonne ou de recourir à un index préfixé. Enfin, l'expression doit rester **déterministe**, exigence déjà imposée à toute colonne générée (§18.4).

## Points clés à retenir

- Indexer une colonne générée revient à créer un **index fonctionnel** : c'est la façon idiomatique de MariaDB d'indexer le résultat d'une expression.
- L'index se déclare comme un index ordinaire (`INDEX`, `CREATE INDEX`, `ALTER … ADD INDEX`) et peut être **composite**.
- Il rend **sargable** un filtrage sur expression (`WHERE annee = 2026`) et permet d'**indexer des champs JSON** extraits (§4.10).
- Une colonne `VIRTUAL` indexée voit sa valeur **matérialisée dans l'index** : lectures rapides sans stockage dans la ligne, surcoût reporté à l'écriture.
- Un index sur colonne générée peut servir à **`GROUP BY` et `ORDER BY`** (surtout quand la requête est **couverte** par l'index), évitant *filesort* et table temporaire ; l'optimiseur sait rapprocher une expression de la colonne correspondante (utile au filtrage). Vérifier avec `EXPLAIN`.
- Surveiller le coût en écriture, la collation, la longueur des expressions textuelles et le déterminisme.
