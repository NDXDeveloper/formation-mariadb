# 18.2.1 Création et configuration

Cette sous-section détaille la mise en place concrète d'une table à versionnement système : déclaration des colonnes de période, activation sur une table existante, exclusion de colonnes, et les quelques variables de configuration qui encadrent l'évolution du schéma et le chargement d'historique. Les colonnes de période s'appuient sur le moteur transactionnel **InnoDB**, utilisé par défaut.

## Créer une table versionnée

### Colonnes de période implicites

La forme la plus concise consiste à ajouter simplement la clause `WITH SYSTEM VERSIONING` à la définition de la table :

```sql
CREATE TABLE employe (
  id      INT PRIMARY KEY,
  nom     VARCHAR(100),
  salaire DECIMAL(10,2)
) WITH SYSTEM VERSIONING;
```

MariaDB crée alors automatiquement les deux colonnes de période, de type `TIMESTAMP(6)`, et les rend **invisibles** : elles n'apparaissent ni dans un `SELECT *`, ni dans un `INSERT` sans liste de colonnes. Les requêtes et le code applicatif existants fonctionnent donc sans la moindre adaptation. Pour consulter ces colonnes, il faut les nommer explicitement :

```sql
SELECT id, nom, salaire, row_start, row_end FROM employe;
```

### Colonnes de période explicites

On peut au contraire déclarer soi-même les colonnes de période, en choisir le nom et les rendre visibles. Trois éléments sont nécessaires : deux colonnes marquées `GENERATED ALWAYS AS ROW START` et `GENERATED ALWAYS AS ROW END`, la déclaration `PERIOD FOR SYSTEM_TIME` qui les associe, et la clause `WITH SYSTEM VERSIONING` :

```sql
CREATE TABLE employe (
  id        INT PRIMARY KEY,
  nom       VARCHAR(100),
  salaire   DECIMAL(10,2),
  debut     TIMESTAMP(6) GENERATED ALWAYS AS ROW START,
  fin       TIMESTAMP(6) GENERATED ALWAYS AS ROW END,
  PERIOD FOR SYSTEM_TIME (debut, fin)
) WITH SYSTEM VERSIONING;
```

Ces colonnes sont alimentées exclusivement par le serveur : on ne les renseigne jamais dans un `INSERT` ou un `UPDATE` ordinaire. La colonne `ROW START` reçoit l'horodatage de la transaction qui crée la version ; la colonne `ROW END` vaut « l'infini » pour une ligne courante et l'horodatage de transaction au moment où la ligne devient historique.

La précision `TIMESTAMP(6)` (microseconde) est recommandée : elle permet de distinguer des transactions très rapprochées.

### Visibilité des colonnes de période

Lorsqu'elles sont déclarées explicitement, les colonnes de période sont **visibles** par défaut. Pour conserver la transparence d'un `SELECT *` tout en gardant des noms maîtrisés, on peut les marquer `INVISIBLE` :

```sql
  debut TIMESTAMP(6) GENERATED ALWAYS AS ROW START INVISIBLE,
  fin   TIMESTAMP(6) GENERATED ALWAYS AS ROW END   INVISIBLE,
```

Le choix entre implicite et explicite relève surtout du confort : les colonnes explicites et nommées facilitent les requêtes d'audit lisibles et le chargement d'historique (voir plus bas), tandis que l'implicite minimise le bruit dans le schéma.

## Activer ou désactiver le versionnement sur une table existante

Le versionnement n'est pas réservé aux nouvelles tables. On l'active sur une table déjà en place avec :

```sql
ALTER TABLE commande ADD SYSTEM VERSIONING;
```

Les lignes existantes deviennent les premières versions courantes ; leur borne de début est fixée au moment de l'opération. À partir de là, toute modification commence à être historisée.

À l'inverse, on retire le versionnement avec :

```sql
ALTER TABLE commande DROP SYSTEM VERSIONING;
```

> ⚠️ **Attention** : retirer le versionnement supprime les colonnes de période **et l'ensemble des lignes historiques**. Seules les données courantes subsistent. Cette opération est donc destructrice pour l'historique : à n'effectuer qu'en connaissance de cause, après sauvegarde si l'historique doit être conservé.

## Exclure des colonnes du versionnement

Par défaut, **toutes** les colonnes sont versionnées : la moindre modification crée une nouvelle version historique. Ce comportement n'est pas toujours souhaitable pour des champs très volatils dont l'historique n'a aucune valeur (un compteur de consultations, une date de dernière connexion…), qui feraient grossir inutilement l'historique.

On peut alors exclure une colonne du versionnement avec l'option `WITHOUT SYSTEM VERSIONING` :

```sql
CREATE TABLE produit (
  id      INT PRIMARY KEY,
  libelle VARCHAR(100),
  prix    DECIMAL(10,2),
  vues    BIGINT WITHOUT SYSTEM VERSIONING
) WITH SYSTEM VERSIONING;
```

Une modification ne portant **que** sur la colonne `vues` ne génère pas de nouvelle version : la valeur est mise à jour sur place. Seuls les changements touchant des colonnes versionnées (ici `libelle` ou `prix`) déclenchent l'historisation.

## Comportement des colonnes de période

Deux propriétés méritent l'attention au moment de la conception.

**Horodatage à l'échelle de la transaction.** Toutes les lignes modifiées au sein d'une même transaction reçoivent le **même** horodatage de période. Conséquence pratique : mettre à jour deux fois la même ligne dans une seule transaction ne conserve que l'état final ; les états intermédiaires, qui auraient une durée nulle, ne sont pas historisés.

**La borne « infinie ».** Une ligne courante porte une borne de fin égale à la valeur maximale du type `TIMESTAMP`. Avant l'extension introduite en 11.8 (§11.12), cette valeur correspondait au 19 janvier 2038 ; elle atteint désormais une date bien plus lointaine (jusqu'en 2106), ce qui sécurise durablement cette borne ouverte. Le format d'horodatage est également un point de vigilance lors des migrations (§19.9).

## Faire évoluer le schéma d'une table versionnée

Modifier la structure d'une table versionnée (ajouter, supprimer ou changer le type d'une colonne) soulève une question de sens : faut-il appliquer la modification à l'historique, au risque de réécrire des données passées ? Pour éviter toute altération involontaire, MariaDB encadre ces opérations via la variable `system_versioning_alter_history`.

Sa valeur par défaut, `ERROR`, **refuse** un `ALTER TABLE` qui modifierait les colonnes d'une table versionnée. Pour autoriser explicitement l'opération — en comprenant qu'elle s'appliquera aussi aux lignes historiques —, on positionne la variable à `KEEP` le temps de la session :

```sql
SET @@system_versioning_alter_history = KEEP;
ALTER TABLE employe ADD COLUMN email VARCHAR(255);
```

Les opérations qui ne touchent pas la structure des lignes (création d'index, par exemple) ne sont pas concernées par ce garde-fou.

## Charger des données historiques

Lors d'une migration ou d'une reprise de données, on souhaite parfois insérer directement des lignes *passées*, en renseignant soi-même les bornes de période — ce qu'un `INSERT` ordinaire interdit. La variable `system_versioning_insert_history` lève cette restriction pour la session :

```sql
SET @@system_versioning_insert_history = ON;

INSERT INTO employe (id, nom, salaire, debut, fin)
VALUES (1, 'Alice', 3000, '2026-01-10 09:00:00', '2026-03-01 14:00:00');
```

Ce mécanisme suppose des colonnes de période accessibles (d'où l'intérêt de colonnes explicites et nommées) et reste réservé aux chargements de confiance. Il est précieux pour reconstituer un historique existant lors d'une bascule vers des tables versionnées.

## Vérifier la configuration

`SHOW CREATE TABLE` restitue fidèlement la configuration de versionnement — colonnes `GENERATED ALWAYS AS ROW START/END`, déclaration `PERIOD FOR SYSTEM_TIME` et clause `WITH SYSTEM VERSIONING` :

```sql
SHOW CREATE TABLE employe\G
```

C'est le réflexe à adopter pour confirmer qu'une table est bien versionnée, vérifier le nom et la visibilité des colonnes de période, et repérer d'éventuelles colonnes exclues.

## Points clés à retenir

- `WITH SYSTEM VERSIONING` suffit à activer le versionnement ; les colonnes de période (`TIMESTAMP(6)`) sont alors créées **implicitement et invisibles**.
- On peut les déclarer explicitement (`GENERATED ALWAYS AS ROW START/END` + `PERIOD FOR SYSTEM_TIME`), les nommer et les rendre `INVISIBLE` au besoin.
- `ALTER TABLE … ADD SYSTEM VERSIONING` active le suivi sur une table existante ; `DROP SYSTEM VERSIONING` le retire **et détruit l'historique**.
- `WITHOUT SYSTEM VERSIONING` exclut une colonne volatile pour éviter une historisation inutile.
- Toutes les modifications d'une transaction partagent le même horodatage ; les lignes courantes portent une borne de fin « infinie » (jusqu'en 2106 depuis 11.8).
- `system_versioning_alter_history = KEEP` autorise les évolutions de schéma (par défaut `ERROR`) ; `system_versioning_insert_history = ON` autorise le chargement d'historique.
