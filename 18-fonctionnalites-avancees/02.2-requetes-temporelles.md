# 18.2.2 Requêtes temporelles (`AS OF`, `BETWEEN`, `FROM…TO`)

Une fois l'historique conservé automatiquement (§18.2.1), encore faut-il pouvoir le consulter. C'est le rôle des **clauses temporelles** : elles permettent d'interroger une table versionnée non plus telle qu'elle est *maintenant*, mais telle qu'elle était à un instant ou sur une période donnés.

Pour illustrer cette sous-section, reprenons la table d'employés du §18.2 dans laquelle le salaire d'Alice a été porté de 3000 à 3300 € le 1ᵉʳ mars. La table contient donc physiquement deux versions de sa ligne :

| nom | salaire | row_start | row_end | état |
|---|---|---|---|---|
| Alice | 3000 | 2026-01-10 09:00:00 | 2026-03-01 14:00:00 | historique |
| Alice | 3300 | 2026-03-01 14:00:00 | *(infini)* | courante |

## Le comportement par défaut : seulement le présent

Sans clause particulière, une requête ne voit que les lignes **courantes**, exactement comme sur une table classique :

```sql
SELECT nom, salaire FROM employe;
-- Alice | 3300
```

L'historique reste invisible tant qu'on ne le demande pas explicitement. C'est ce qui garantit la transparence pour les applications existantes.

## La clause `FOR SYSTEM_TIME`

Pour interroger l'historique, on ajoute `FOR SYSTEM_TIME` juste après la référence de table, suivie de l'une des quatre formes prévues par le standard SQL:2011 :

```sql
SELECT ... FROM table FOR SYSTEM_TIME AS OF <instant> ...
SELECT ... FROM table FOR SYSTEM_TIME BETWEEN <t1> AND <t2> ...
SELECT ... FROM table FOR SYSTEM_TIME FROM <t1> TO <t2> ...
SELECT ... FROM table FOR SYSTEM_TIME ALL ...
```

Les bornes acceptent aussi bien un littéral horodaté (`TIMESTAMP '2026-02-01 00:00:00'`, ou une simple chaîne) qu'une expression (`NOW() - INTERVAL 6 MONTH`). Ces clauses sont propres à la **lecture** : un `UPDATE` ou un `DELETE` opèrent toujours sur les lignes courantes.

## `AS OF` : l'état à un instant donné

`AS OF` renvoie les lignes qui étaient **courantes** au moment indiqué — un véritable instantané du passé.

```sql
SELECT nom, salaire FROM employe
FOR SYSTEM_TIME AS OF TIMESTAMP '2026-02-01 00:00:00';
-- Alice | 3000   (la version en vigueur au 1er février)
```

L'instant peut être calculé dynamiquement :

```sql
SELECT nom, salaire FROM employe
FOR SYSTEM_TIME AS OF NOW() - INTERVAL 3 MONTH;
```

Une ligne est considérée valide à l'instant `t` lorsque `row_start ≤ t < row_end` : la période est traitée comme un intervalle **fermé à gauche, ouvert à droite**. À l'instant exact d'un changement, c'est donc la nouvelle version qui prévaut.

## `ALL` : l'historique complet

`ALL` renvoie **toutes** les versions, courantes et historiques. Associée à un tri sur la borne de début, elle reconstitue la chronologie complète d'une donnée :

```sql
SELECT nom, salaire, row_start, row_end FROM employe
FOR SYSTEM_TIME ALL
ORDER BY row_start;
-- Alice | 3000 | 2026-01-10 09:00:00 | 2026-03-01 14:00:00
-- Alice | 3300 | 2026-03-01 14:00:00 | (infini)
```

C'est la forme privilégiée pour les analyses d'historique : compter le nombre de modifications, mesurer la durée de chaque état, ou suivre l'évolution d'une valeur dans le temps.

## `FROM … TO …` et `BETWEEN … AND …` : les versions actives sur une période

Ces deux formes renvoient les versions qui ont été actives **à un moment quelconque** de l'intervalle. Elles ne diffèrent que par le traitement de la borne supérieure.

`FROM … TO …` considère un intervalle **fermé à gauche, ouvert à droite** `[t1, t2)` :

```sql
SELECT nom, salaire FROM employe
FOR SYSTEM_TIME FROM '2026-01-01' TO '2026-04-01';
-- Alice | 3000
-- Alice | 3300   (les deux versions ont chevauché l'intervalle)
```

`BETWEEN … AND …` considère un intervalle **entièrement fermé** `[t1, t2]`, borne supérieure incluse :

```sql
SELECT nom, salaire FROM employe
FOR SYSTEM_TIME BETWEEN '2026-01-01' AND '2026-02-15';
-- Alice | 3000   (seule version active avant le 15 février)
```

### Comparaison des formes

| Forme | Intervalle considéré | Borne basse | Borne haute |
|---|---|---|---|
| `AS OF t` | instant unique `t` | — | — |
| `FROM t1 TO t2` | `[t1, t2)` | incluse | **exclue** |
| `BETWEEN t1 AND t2` | `[t1, t2]` | incluse | **incluse** |
| `ALL` | tout l'historique | — | — |

La distinction entre `FROM…TO` et `BETWEEN` n'a d'incidence que sur une version dont la borne de début coïncide *exactement* avec `t2` : `BETWEEN` la retient, `FROM…TO` l'écarte. Dans la pratique, on choisit `FROM…TO` lorsqu'on raisonne en intervalles juxtaposés (pour éviter de compter deux fois une bascule), et `BETWEEN` lorsqu'on veut une inclusion symétrique des deux bornes.

## Voyage dans le temps à l'échelle de la session

Réécrire chaque requête avec `FOR SYSTEM_TIME` devient vite fastidieux lorsqu'on veut analyser tout un jeu de tables à une date passée. La variable de session `system_versioning_asof` répond à ce besoin : une fois positionnée, **toutes** les requêtes suivantes (sur les tables versionnées, sans clause explicite) renvoient les données telles qu'elles étaient à cet instant.

```sql
SET @@system_versioning_asof = TIMESTAMP '2026-02-01 00:00:00';

SELECT nom, salaire FROM employe;   -- Alice | 3000, sans clause explicite

SET @@system_versioning_asof = DEFAULT;   -- retour aux données courantes
```

Cette variable n'affecte que les tables versionnées ; une clause `FOR SYSTEM_TIME` explicite dans une requête reste prioritaire sur elle. C'est l'outil idéal pour rejouer un état historique complet — par exemple produire un rapport « tel qu'il aurait été » à la clôture d'un trimestre — sans toucher au code des requêtes.

## Requêtes temporelles et jointures

`FOR SYSTEM_TIME` s'applique **table par table**, ce qui permet de combiner plusieurs tables versionnées à une même date et d'obtenir un instantané cohérent de l'ensemble :

```sql
SELECT e.nom, e.salaire, d.libelle
FROM employe     FOR SYSTEM_TIME AS OF '2026-02-01' AS e
JOIN departement FOR SYSTEM_TIME AS OF '2026-02-01' AS d
  ON e.dept_id = d.id;
```

Chaque table est ainsi figée à l'instant voulu, ce qui reconstitue fidèlement les relations telles qu'elles existaient à cette date.

## Clauses temporelles ou filtrage manuel ?

Lorsque les colonnes de période sont visibles, on pourrait être tenté de les filtrer directement dans un `WHERE` (`WHERE row_start <= … AND row_end > …`). C'est techniquement possible, mais déconseillé : la clause `FOR SYSTEM_TIME` est plus lisible, conforme au standard, fonctionne même lorsque les colonnes sont invisibles, et exprime sans ambiguïté l'intention temporelle. Le filtrage manuel est à réserver aux cas particuliers non couverts par les quatre formes standard.

## Points clés à retenir

- Sans clause, une requête ne voit que les lignes **courantes** ; l'historique se consulte avec `FOR SYSTEM_TIME`, placée après la table.
- `AS OF t` donne l'instantané valide à l'instant `t` ; `ALL` retourne toutes les versions (idéal pour la chronologie, avec `ORDER BY row_start`).
- `FROM t1 TO t2` (borne haute **exclue**) et `BETWEEN t1 AND t2` (borne haute **incluse**) renvoient les versions actives sur un intervalle ; la nuance ne joue que sur la borne supérieure.
- `system_versioning_asof` fige toute une session à une date passée ; une clause explicite reste prioritaire.
- `FOR SYSTEM_TIME` s'applique **par table** dans une jointure, garantissant un instantané cohérent multi-tables.
- Préférer ces clauses standard au filtrage manuel des colonnes de période.
