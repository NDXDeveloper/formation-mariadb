# 18.7.1 Data at Rest Encryption

Cette sous-section détaille **ce que l'on chiffre et comment l'activer**. La gestion des clés — d'où elles viennent, comment les protéger et les faire tourner — fait l'objet de la sous-section suivante (§18.7.2).

## Prérequis : un plugin de clés

Le chiffrement au repos ne fonctionne pas seul : il lui faut une source de clés, fournie par un **plugin de gestion de clés** (par exemple `file_key_management`). Tant qu'aucun plugin de clés n'est chargé, les options de chiffrement restent sans effet. On suppose donc, dans toute cette sous-section, qu'un tel plugin est en place ; sa configuration est traitée en §18.7.2. Le chiffrement lui-même s'appuie sur l'algorithme symétrique **AES**.

## Chiffrer les tables InnoDB

### Globalement : `innodb_encrypt_tables`

La variable `innodb_encrypt_tables` définit la politique de chiffrement des tablespaces InnoDB, avec trois valeurs :

- **`OFF`** (défaut) : aucune table n'est chiffrée automatiquement ; on peut toujours en chiffrer une explicitement.
- **`ON`** : les tables sont chiffrées par défaut, mais on peut en exclure une au cas par cas.
- **`FORCE`** : toutes les tables doivent être chiffrées ; toute tentative de créer une table en clair est refusée.

### Table par table : `ENCRYPTED` et `ENCRYPTION_KEY_ID`

Indépendamment de la politique globale, chaque table peut être pilotée par des options de table. `ENCRYPTED=YES` ou `ENCRYPTED=NO` force le comportement, et `ENCRYPTION_KEY_ID` désigne **quelle clé** (par son identifiant) chiffre la table :

```sql
CREATE TABLE dossier_medical (
  id      INT PRIMARY KEY,
  patient VARCHAR(100),
  notes   TEXT
) ENCRYPTED=YES ENCRYPTION_KEY_ID=2;
```

On peut aussi chiffrer une table existante, ce qui la reconstruit :

```sql
ALTER TABLE client ENCRYPTED=YES;
```

### Chiffrer l'existant : `innodb_encryption_threads`

Activer `innodb_encrypt_tables=ON` chiffre les tables nouvellement créées, mais pas, à lui seul, les tables déjà présentes. Pour **chiffrer l'existant en arrière-plan** — et, plus tard, pour faire tourner les clés (§18.7.2) —, il faut allouer des threads dédiés :

```sql
SET GLOBAL innodb_encryption_threads = 4;
```

Ces threads parcourent progressivement les tablespaces pour les chiffrer (ou les rechiffrer) sans interrompre l'activité.

## Ne pas oublier le reste : journaux et fichiers temporaires

Chiffrer les tables tout en laissant les journaux et les fichiers temporaires en clair laisserait fuir, sur le disque, exactement ce que l'on cherche à protéger : le redo log et le binlog contiennent les données modifiées, et les fichiers temporaires peuvent matérialiser des résultats de tris ou de jointures. Un chiffrement au repos **cohérent** couvre donc l'ensemble de la surface :

- `innodb_encrypt_log` : chiffre le **redo log** d'InnoDB ;
- `encrypt_binlog` : chiffre les **journaux binaires** (et de relais) ;
- `encrypt_tmp_files` : chiffre les **fichiers temporaires** écrits par le serveur ;
- `innodb_encrypt_temporary_tables` : chiffre les **tables temporaires InnoDB** ;
- `aria_encrypt_tables` : chiffre les tables **Aria** (qui sous-tendent les tables temporaires sur disque).

## Une configuration complète

En pratique, ces réglages se posent dans le fichier de configuration. Les options globales de tables et le nombre de threads sont **ajustables à chaud** ; le chiffrement des journaux et du binlog se définit en revanche **au démarrage** :

```ini
[mariadb]
# tables InnoDB chiffrées par défaut
innodb_encrypt_tables           = ON
# threads de (re)chiffrement de l'existant et de rotation des clés
innodb_encryption_threads       = 4
# couvrir aussi journaux et fichiers temporaires
innodb_encrypt_log              = ON
encrypt_binlog                  = ON
encrypt_tmp_files               = ON
innodb_encrypt_temporary_tables = ON
aria_encrypt_tables             = ON
```

Pour imposer le chiffrement à l'échelle de l'instance, on remplacera la première ligne par :

```ini
innodb_encrypt_tables = FORCE
```

## Vérifier l'état du chiffrement

La vue `INFORMATION_SCHEMA.INNODB_TABLESPACES_ENCRYPTION` renseigne, tablespace par tablespace, sur l'état du chiffrement et la version de clé en cours — utile pour confirmer que toutes les tables sont bien chiffrées et pour suivre l'avancement d'un chiffrement de fond ou d'une rotation :

```sql
SELECT name, encryption_scheme, current_key_version
FROM information_schema.INNODB_TABLESPACES_ENCRYPTION;
```

Le champ `encryption_scheme` distingue les tablespaces chiffrés des autres, tandis que `current_key_version` indique la version de clé utilisée.

## Chiffrement et sauvegardes

Les sauvegardes d'une base chiffrée contiennent des **données chiffrées** : leur restauration et leur exploitation supposent de disposer des clés correspondantes. C'est un point d'attention pour le plan de reprise — perdre les clés, c'est perdre l'accès aux sauvegardes autant qu'à la base. Les mécanismes de sauvegarde, dont Mariabackup, sont traités en §12.

## Points clés à retenir

- Le chiffrement au repos exige au préalable un **plugin de clés** (§18.7.2) et s'appuie sur **AES**.
- `innodb_encrypt_tables` fixe la politique InnoDB (`OFF` / `ON` / `FORCE`) ; `ENCRYPTED=YES|NO` et `ENCRYPTION_KEY_ID` pilotent chaque table.
- `innodb_encryption_threads > 0` chiffre **l'existant** en arrière-plan (et permet la rotation des clés).
- Pour un chiffrement **cohérent**, couvrir aussi le redo log (`innodb_encrypt_log`), le binlog (`encrypt_binlog`), les fichiers temporaires (`encrypt_tmp_files`, `innodb_encrypt_temporary_tables`) et Aria (`aria_encrypt_tables`).
- Contrôler l'état via `INFORMATION_SCHEMA.INNODB_TABLESPACES_ENCRYPTION`.
- Les **sauvegardes** d'une base chiffrée nécessitent les clés pour être restaurées (§12).
