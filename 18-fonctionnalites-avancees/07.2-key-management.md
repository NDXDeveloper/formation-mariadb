# 18.7.2 Key management

## La clé du chiffrement, c'est la clé

Comme le rappelait l'introduction de la section (§18.7), un chiffrement au repos ne vaut que par la **protection de ses clés**. Une clé laissée en clair sur le même serveur que les données n'apporte rien ; une clé perdue rend les données définitivement illisibles. La gestion des clés est donc le véritable enjeu, et MariaDB la confie à un **plugin** que l'on choisit selon le niveau d'exigence : un simple fichier sur le serveur, ou un gestionnaire de clés externe.

## Le plugin `file_key_management` : des clés dans un fichier

L'option la plus simple et autonome conserve les clés dans un **fichier de clés** local. Chaque ligne associe un identifiant de clé à une clé AES en hexadécimal (128, 192 ou 256 bits) :

```
1;F5502320F8429037B8DAEF761B189D12C1C2305DA8FF...
2;770A8A65DA156D24EE2A093277530142...
```

L'identifiant est celui que l'on désigne dans `ENCRYPTION_KEY_ID` (§18.7.1) ; on peut ainsi chiffrer différentes tables avec différentes clés.

### Chiffrer le fichier de clés — et la dérivation SHA-2 🆕

Laisser ce fichier en clair reviendrait à entreposer les clés à côté des serrures. Le fichier de clés doit donc lui-même être **chiffré**, typiquement avec OpenSSL, puis déchiffré au démarrage à l'aide d'une clé secondaire (un mot de passe ou un second fichier).

C'est ici qu'intervient le point relatif à **SHA-2**. Lorsqu'on chiffre le fichier de clés par mot de passe, OpenSSL *dérive* la clé de chiffrement à partir de ce mot de passe au moyen d'une fonction de hachage ; le plugin doit appliquer **exactement la même** dérivation pour déchiffrer le fichier au démarrage. Historiquement, `file_key_management` imposait **SHA-1** (le fichier devait être produit avec `openssl enc … -md sha1`). MariaDB 12.3 lui ajoute la prise en charge de la famille **SHA-2** et de **PBKDF2** (MDEV-34712), réglée par deux variables :

```ini
file_key_management_digest     = sha256   # sha1 | sha224 | sha256 | sha384 | sha512
file_key_management_use_pbkdf2 = 20000    # nombre d'itérations PBKDF2 (0 = désactivé)
```

Le fichier de clés se chiffre alors avec un condensé SHA-256 et PBKDF2 :

```bash
openssl enc -aes-256-cbc -md sha256 -pbkdf2 -iter 20000 -pass pass:VOTRE_SECRET \
  -in keyfile.txt -out keyfile.enc
```

Le digest — et, le cas échéant, le nombre d'itérations PBKDF2 — déclarés côté serveur doivent rester **cohérents** avec ceux employés pour chiffrer le fichier, sans quoi le plugin échoue à le déchiffrer au démarrage. Cette montée vers SHA-2/PBKDF2 apporte un gain de robustesse par rapport au SHA-1 hérité.

### Configuration

Le plugin et ses paramètres se déclarent dans le fichier de configuration :

```ini
[mariadb]
plugin_load_add = file_key_management
file_key_management_filename             = /etc/mysql/encryption/keyfile.enc
file_key_management_filekey              = FILE:/etc/mysql/encryption/keyfile.key
file_key_management_encryption_algorithm = AES_CTR
```

`file_key_management_filename` pointe vers le fichier de clés chiffré ; `file_key_management_filekey` fournit la clé qui le déchiffre — de préférence sous la forme `FILE:` renvoyant à un fichier protégé, plutôt qu'en clair dans la configuration ; `file_key_management_encryption_algorithm` fixe le mode AES utilisé pour les tablespaces (`AES_CTR` ou `AES_CBC`).

## Les gestionnaires de clés externes (KMS)

Pour une sécurité supérieure, on confie les clés à un **gestionnaire externe** : elles ne résident alors plus du tout sur le serveur de base de données, ce qui élimine le maillon faible du fichier local.

### HashiCorp Vault — et le vidage du cache

Le plugin `hashicorp_key_management` récupère les clés depuis **HashiCorp Vault** :

```ini
[mariadb]
plugin_load_add = hashicorp_key_management
hashicorp_key_management_vault_url     = https://vault.example.com:8200/v1/secret/data/mariadb
hashicorp_key_management_token         = s.xxxxxxxxxxxxxxxx
hashicorp_key_management_cache_timeout = 60000   # durée de vie du cache en ms (défaut 60000 ≈ 1 min)
```

Pour ne pas solliciter Vault à chaque accès à une page, le plugin **met les clés en cache** localement, avec des délais d'expiration paramétrables (durée de vie d'une clé en cache, durée de validité de la résolution de la dernière version). Le revers de ce cache : après une **rotation** de clé effectuée dans Vault, le serveur peut continuer à servir l'ancienne version jusqu'à l'expiration du cache.

C'est précisément ce que résout le **vidage du cache à la demande** (*cache flush*) introduit en **MariaDB 12.3** 🆕 (MDEV-30847) : la commande `FLUSH HASHICORP_KEY_MANAGEMENT_CACHE` (qui requiert le privilège `RELOAD`) purge le cache du plugin sans attendre l'expiration ni redémarrer le serveur, de sorte que la nouvelle version de clé soit prise en compte immédiatement après une rotation. La commande `SHOW HASHICORP_KEY_MANAGEMENT_CACHE` permet par ailleurs d'inspecter les clés actuellement en cache.

### AWS KMS et autres

D'autres plugins existent dans le même esprit, notamment l'intégration avec **AWS KMS** (`aws_key_management`), où la clé maître ne quitte jamais le service de gestion de clés d'AWS. Le principe commun de ces solutions est de **déporter les clés hors du serveur**, dans un système conçu pour leur stockage, leur audit et leur rotation.

## Versions de clés et rotation

Une clé est identifiée par un **identifiant** et une **version**. La rotation consiste à introduire une nouvelle version d'une clé et à rechiffrer progressivement les données avec celle-ci. Deux variables la gouvernent côté InnoDB :

```ini
innodb_encryption_threads        = 4   # threads de rechiffrement en arrière-plan
innodb_encryption_rotate_key_age = 1   # rechiffrer dès qu'une version est trop ancienne
```

Les threads de chiffrement (§18.7.1) parcourent alors les tablespaces pour les rechiffrer avec la dernière version. À noter : `file_key_management` gère plusieurs identifiants de clé mais **ne propose pas de versionnement dynamique** des clés ; la rotation automatique par version est, elle, l'apanage des plugins adossés à un **KMS**.

## Bonnes pratiques de gestion des clés

La sécurité d'ensemble tient à quelques principes constants. Le fichier de clés de `file_key_management` doit **toujours être chiffré** et protégé par des permissions de fichier strictes — ou, mieux, remplacé par un **KMS** qui sort les clés du serveur. Les clés doivent être **sauvegardées séparément** des données et de façon sécurisée : les perdre revient à perdre la base et toutes ses sauvegardes (§18.7.1, §12). Enfin, une **rotation périodique** des clés, naturelle avec un KMS, limite l'exposition en cas de compromission.

## Points clés à retenir

- La gestion des clés est le point critique du chiffrement au repos : une clé mal protégée ou perdue en annule tout le bénéfice.
- **`file_key_management`** stocke les clés dans un fichier (`id;clé hex`) qui doit être **chiffré** ; depuis **MariaDB 12.3**, le plugin gère la dérivation **SHA-2** et **PBKDF2** (`file_key_management_digest`, `file_key_management_use_pbkdf2`), au-delà du **SHA-1** historique.
- Les **KMS** (HashiCorp Vault, AWS KMS) déportent les clés hors du serveur ; le plugin **Vault** met les clés en cache, et `FLUSH HASHICORP_KEY_MANAGEMENT_CACHE` (🆕 12.3, privilège `RELOAD`) applique une rotation sans redémarrage.
- La **rotation** par version s'appuie sur `innodb_encryption_threads` et `innodb_encryption_rotate_key_age` ; `file_key_management` ne versionne pas les clés, contrairement aux plugins KMS.
- Protéger les clés (permissions, chiffrement, KMS), les **sauvegarder à part**, et les faire **tourner** régulièrement.
