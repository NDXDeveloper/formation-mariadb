# 10.5.5 — caching_sha2_password : compatibilité d'authentification MySQL 8 🆕

`caching_sha2_password` est le **plugin d'authentification par défaut de MySQL 8**. MariaDB l'a ajouté à partir de la **12.1** dans un but précis : **faciliter la migration depuis MySQL** et l'interopérabilité avec les clients MySQL. Sa vocation est claire — c'est une **aide à la migration**, pas un plugin recommandé pour de nouveaux comptes, pour lesquels PARSEC (10.6) ou `ed25519` (10.5.2) restent préférables. Cette section en explique le contexte, le fonctionnement (cache et RSA), les exigences de connexion et l'usage en migration.

---

## Pourquoi ce plugin ? Le contexte MySQL 8

Depuis MySQL 8.0.4, `caching_sha2_password` est le plugin d'authentification **par défaut** de MySQL. Fondé sur l'algorithme **SHA-256** (famille SHA-2) et doté d'un **cache** côté serveur, il est nettement plus robuste que l'ancien `mysql_native_password` (SHA-1), que MySQL a déprécié. Conséquence : les instances MySQL récentes, et les clients qui s'y connectent, utilisent massivement ce plugin.

Pour qu'une migration de MySQL vers MariaDB n'impose pas de réinitialiser tous les mots de passe, MariaDB a donc ajouté en 12.1 un plugin `caching_sha2_password` **compatible**. À noter : MariaDB ne prend en charge **que** ce plugin, et **pas** l'ancien `sha256_password` (variante sans cache).

---

## Installation

La bibliothèque du plugin est livrée avec MariaDB sous le nom **`auth_mysql_sha2`** (`auth_mysql_sha2.so` / `.dll`) — un nom de fichier qui ne reprend pas littéralement celui du plugin. Comme les autres plugins chargeables, **il n'est pas installé par défaut** :

```sql
INSTALL SONAME 'auth_mysql_sha2';
```

---

## Comment ça fonctionne : cache et RSA

Le plugin combine deux chemins d'authentification :

- **Authentification rapide (cache).** Le serveur maintient un **cache en mémoire** des authentifications réussies. Lorsqu'un utilisateur déjà connecté se reconnecte, un *cache hit* valide la connexion en quelques étapes seulement — c'est l'intérêt du « caching ».
- **Vérification complète (RSA).** En cas de *cache miss* (premier accès du compte, redémarrage du serveur, changement de mot de passe…), une vérification complète est nécessaire. Sur une connexion **non chiffrée**, elle s'appuie sur un **échange RSA** : le client demande la clé publique du serveur, chiffre le mot de passe avec, et l'envoie. Sur une connexion **sécurisée** (TLS ou socket Unix), le mot de passe peut être transmis directement de façon protégée.

---

## L'exigence d'une connexion sécurisée

C'est la principale contrainte opérationnelle. Pour s'authentifier avec `caching_sha2_password`, il faut **soit une connexion sécurisée** (TLS, socket Unix), **soit une connexion non chiffrée acceptant l'échange RSA**.

Le chemin RSA présente toutefois des inconvénients soulignés par la documentation : il oblige à **distribuer la clé publique du serveur** à tous les utilisateurs — ce qui est lourd et peu pratique —, et le serveur **reçoit le mot de passe en clair** (après déchiffrement), ce qui pose problème si le client se connecte à un serveur malveillant. En pratique, on privilégie donc nettement le **chiffrement TLS** (section 10.7), plus simple et plus sûr que la gestion de paires de clés RSA.

---

## Variables de configuration

Le chemin RSA se règle via plusieurs variables système :

- `caching_sha2_password_private_key_path` — chemin de la clé RSA privée ;
- `caching_sha2_password_public_key_path` — chemin de la clé RSA publique ;
- `caching_sha2_password_auto_generate_rsa_keys` — génère automatiquement les clés au démarrage si elles ne sont pas définies ni présentes ;
- `caching_sha2_password_digest_rounds` — nombre de tours SHA-2 appliqués au calcul de l'empreinte.

```ini
[mariadb]
caching_sha2_password_auto_generate_rsa_keys=ON
caching_sha2_password_private_key_path=/etc/mariadb/private_key.pem
caching_sha2_password_public_key_path=/etc/mariadb/public_key.pem
```

---

## Créer un compte

On déclare le plugin comme les autres, avec `IDENTIFIED VIA` :

```sql
CREATE USER 'compte_mysql'@'%'
  IDENTIFIED VIA caching_sha2_password USING PASSWORD('mot_de_passe');
```

Pour une **migration**, on peut aussi fournir directement l'**empreinte** existante issue de MySQL :

```sql
-- L'empreinte provient de la colonne authentication_string de mysql.user (côté MySQL)
CREATE USER 'compte_mysql'@'%'
  IDENTIFIED VIA caching_sha2_password USING '<empreinte_mysql>';
```

---

## Migrer des comptes depuis MySQL

L'intérêt majeur du plugin est de **transférer des comptes MySQL sans changer leurs mots de passe** : on reprend l'empreinte stockée dans la colonne `authentication_string` du `mysql.user` de l'instance MySQL, et on crée le compte correspondant côté MariaDB avec cette même empreinte.

Un **piège** mérite l'attention : l'empreinte `caching_sha2_password` de MySQL peut contenir des **caractères non imprimables**. La copier-coller depuis une fenêtre de terminal ne fonctionnera donc probablement pas ; il faut recourir à un transfert préservant fidèlement les octets. Par ailleurs, si l'instance source utilise l'ancien `sha256_password` (non pris en charge par MariaDB), il faut d'abord convertir ces comptes vers une autre méthode.

---

## Compatibilité client

Le support côté client n'est pas universel :

- **MariaDB Connector/C** prend en charge `caching_sha2_password` depuis la **version 3.0.2** (le plugin est fourni comme bibliothèque séparée) ; tous les clients qui s'appuient dessus en bénéficient, y compris les clients en ligne de commande `mysql` et `mariadb` ;
- en revanche, la bibliothèque **`libmysqlclient` de MariaDB ne le prend pas en charge** : un client qui y est lié ne pourra pas s'authentifier ainsi ;
- pour les applications Python, PHP, Go, Java ou ODBC, il faut utiliser des **versions récentes** des connecteurs et pilotes.

---

## Quand l'utiliser

`caching_sha2_password` est à réserver à son rôle : **la migration depuis MySQL** et l'interopérabilité avec un parc de clients MySQL. Il évite de réinitialiser les mots de passe lors d'un passage MySQL → MariaDB. Mais pour de **nouveaux comptes** sur une instance MariaDB, la documentation recommande explicitement **PARSEC** (10.6), plus sûr et plus pratique (sans la complexité de distribution de clés RSA), ou `ed25519` (10.5.2). On considérera donc ce plugin comme une **étape transitoire**, non comme une cible d'authentification définitive.

---

## À retenir

`caching_sha2_password` est le plugin par défaut de **MySQL 8** (SHA-256 + cache), ajouté à MariaDB en **12.1** comme **aide à la migration** — il permet de reprendre des comptes MySQL **sans changer les mots de passe**. On l'installe via la bibliothèque **`auth_mysql_sha2`** (non installée par défaut). Son authentification combine un **cache** (accès rapide) et, en vérification complète, un **échange RSA** sur connexion non chiffrée ; il exige donc une **connexion sécurisée** (TLS recommandé) ou la gestion de clés RSA. Côté client, **MariaDB Connector/C ≥ 3.0.2** le supporte, mais pas `libmysqlclient`. Attention à l'empreinte MySQL qui peut contenir des **caractères non imprimables**. À usage de migration uniquement : pour de nouveaux comptes, préférer **PARSEC** ou `ed25519`.

---

> 🔔 **Note de version.** Le plugin `caching_sha2_password` a été ajouté dans **MariaDB Community Server 12.1** (et Enterprise Server 11.8), via MDEV-9804 ; il fait partie de la **12.3 LTS** mais reste **non installé par défaut** (bibliothèque `auth_mysql_sha2`). MariaDB ne prend pas en charge l'ancien `sha256_password`. Ce plugin est destiné à la migration depuis MySQL ; PARSEC (10.6) est recommandé pour les nouveaux comptes.
