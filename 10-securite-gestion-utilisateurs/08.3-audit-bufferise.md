# 10.8.3 — Logging bufferisé (`server_audit_file_buffer_size`), HOST:PORT et `tls_version` 🆕

Les sections précédentes ont décrit le plugin d'audit (10.8.1) puis l'audit des connexions et des requêtes (10.8.2). La série **12.x**, consolidée dans la **12.3 LTS**, apporte trois améliorations directement utiles à l'exploitation de cet audit : une **journalisation bufferisée** qui réduit le coût en écriture (la limite évoquée en fin de section 10.8.2), un enrichissement des enregistrements de connexion avec le port source (**HOST:PORT**), et un nouveau champ **`tls_version`** qui donne une visibilité sur le chiffrement de chaque session. Ces trois nouveautés concernent uniquement le plugin d'audit (`server_audit`) ; elles n'altèrent pas la sémantique des événements vue précédemment.

## La journalisation bufferisée (`server_audit_file_buffer_size`)

Historiquement, le plugin écrit de façon **synchrone** : chaque événement audité provoque une écriture individuelle dans le fichier journal. Sur un serveur très sollicité, ce comportement génère un volume d'I/O important et constitue un goulet d'étranglement, d'autant plus marqué que l'audit des requêtes est large.

Depuis **MariaDB 12.1**, le plugin peut accumuler les enregistrements dans un tampon en mémoire avant de les écrire par lots, ce qui rend la journalisation asynchrone et réduit nettement la surcharge. Deux variables pilotent ce comportement.

| Variable | Rôle |
|----------|------|
| `server_audit_file_buffer_size` | Taille (en octets) du tampon mémoire. `0` (valeur par défaut) = pas de bufferisation, comportement synchrone d'origine. Toute valeur non nulle active le tampon. |
| `server_audit_sync_log_file` | Force le vidage immédiat du tampon vers le fichier (`SET GLOBAL server_audit_sync_log_file = 1`). |

La taille fournie est **tronquée au multiple de 8192 inférieur** (arrondi vers le bas) : une valeur de `100` retombe donc à `0` (pas de bufferisation), `10000` à `8192`, `65535` à `57344`. Il faut ainsi fournir **au moins 8192** pour activer réellement un tampon. La valeur **maximale est 65536** (soit 8 × 8192) ; toute valeur supérieure y est plafonnée. Lorsque le tampon dépasse le seuil configuré, son contenu est écrit sur disque. *(Comportement vérifié sur 12.3.2.)*

```sql
-- Activer un tampon d'audit de 32 Ko (réduit le nombre d'écritures disque)
SET GLOBAL server_audit_file_buffer_size = 32768;

-- Vérifier la valeur effective (arrondie au multiple de 8192)
SHOW GLOBAL VARIABLES LIKE 'server_audit_file_buffer_size';
```

De façon persistante :

```ini
[mariadb]
server_audit_output_type      = file
server_audit_file_path        = /var/log/mysql/server_audit.log
server_audit_file_buffer_size = 32768
```

La bufferisation introduit un compromis qu'il faut comprendre. Tant que des enregistrements sont dans le tampon, ils **ne sont pas visibles dans le fichier**. Sur un serveur peu actif, le délai avant qu'un événement n'apparaisse dans le journal peut donc devenir significatif. C'est précisément le rôle de `server_audit_sync_log_file` : avant de lire le journal ou pendant une investigation, on force l'écriture du tampon pour s'assurer de disposer des enregistrements les plus récents.

```sql
-- Avant de consulter le journal, vider le tampon pour ne rien manquer
SET GLOBAL server_audit_sync_log_file = 1;
```

Ce compromis a aussi une implication sur la durabilité : en cas d'arrêt brutal du serveur, les enregistrements encore présents en mémoire et non encore écrits seraient perdus. Pour un audit à forte exigence d'intégrité, on dimensionnera donc le tampon avec prudence (voire on conservera la journalisation synchrone), tandis que les environnements à très haut débit privilégieront un tampon plus large pour préserver les performances. La bufferisation s'applique à la sortie fichier (`server_audit_output_type = file`).

## HOST:PORT : identification enrichie des connexions

Jusqu'à la série 11.x, le quatrième champ de l'enregistrement (`host`) ne contenait que l'hôte client. **Depuis MariaDB 12.0**, le plugin journalise désormais le couple **hôte et port source** des connexions entrantes, lorsqu'un port s'applique. Une connexion TCP fait ainsi apparaître l'adresse IP suivie du port source ; une connexion locale via socket Unix, qui n'utilise pas de port, n'en affiche pas.

```text
# Jusqu'à la série 11.x — seul l'hôte client
20260605 14:23:01,dbprod01,appuser,192.168.1.50,4521,0,CONNECT,appdb,,0

# Depuis MariaDB 12.0 — hôte ET port source dans le champ host
20260605 14:23:01,dbprod01,appuser,192.168.1.50:54344,4521,0,CONNECT,appdb,,0
```

L'intérêt est avant tout corrélationnel. Le port source permet de rapprocher un enregistrement d'audit des journaux réseau (pare-feu, répartiteur de charge, table de suivi de connexions) et de distinguer des connexions concurrentes issues d'une même adresse — par exemple plusieurs sessions provenant d'un même pool applicatif ou d'un hôte derrière du NAT. Pour une analyse *post mortem*, c'est un élément discriminant supplémentaire.

Cette évolution a une conséquence pratique à anticiper lors d'une montée de version : tout outil d'analyse ou pipeline d'ingestion (SIEM, scripts de parsing) qui découpe le champ `host` doit être adapté pour tolérer la présence facultative du suffixe `:port`, sous peine d'interpréter incorrectement les enregistrements produits par un serveur 12.x.

## `tls_version` : visibilité sur le chiffrement des connexions

La série **12.x** ajoute par ailleurs un champ **`tls_version`** aux enregistrements de connexion du plugin d'audit. Ce champ consigne la **version du protocole TLS négociée** pour la session (par exemple `TLSv1.2` ou `TLSv1.3`) ; il reste vide pour une connexion non chiffrée.

C'est un apport notable pour la sécurité et la conformité. L'audit ne se contente plus d'indiquer *qui* s'est connecté et *depuis où* (10.8.2) : il documente aussi la **posture de sécurité** de chaque session. On peut ainsi vérifier qu'une connexion attendue en TLS l'était effectivement, repérer les connexions en clair qui auraient dû être chiffrées, et démontrer le respect d'une politique interdisant les versions TLS obsolètes. Ce champ complète naturellement la configuration du chiffrement des connexions abordée en 10.7.

```text
# Enregistrement de connexion incluant la version TLS négociée (vérifié sur 12.3.2)
20260605 14:23:01,dbprod01,appuser,192.168.1.50:54344,4521,0,CONNECT,appdb,TLSv1.3,0
```

> **Note de position.** Vérifié sur 12.3.2 : dans un enregistrement de connexion, `tls_version` occupe la place du champ **`object`** du format général (10.8.2) — c'est-à-dire **juste avant le code retour** (`…,CONNECT,<base>,TLSv1.3,<retcode>`), et non en fin de ligne. Le format reste donc à dix champs : pour les événements de connexion, c'est le champ `object` (vide auparavant) qui porte la version TLS. Ce champ est **vide lorsque la session ne négocie pas TLS** (client connecté en `--skip-ssl`, ou serveur sans TLS). Attention en revanche à une idée reçue : avec le **TLS zéro-configuration** actif par défaut depuis la 11.4 (10.7.3), même une connexion par **socket Unix local** négocie TLS et renseigne donc ce champ (`TLSv1.3`) — un socket n'est pas, en soi, une connexion en clair (vérifié sur 12.3.2). Les outils de parsing du journal doivent en tenir compte.

## Synthèse

Ces trois évolutions répondent à des besoins d'exploitation complémentaires. La **bufferisation** (`server_audit_file_buffer_size`, depuis 12.1) lève le principal frein de performance de l'audit sur les serveurs à fort trafic, au prix d'un délai d'apparition des enregistrements que `server_audit_sync_log_file` permet de maîtriser. L'ajout du **port source** (depuis 12.0) et du champ **`tls_version`** enrichit la traçabilité des connexions, en améliorant la corrélation avec les journaux réseau d'une part, et la vérifiabilité du chiffrement d'autre part. En production, on retiendra : activer un tampon dimensionné au débit réel et synchroniser avant lecture ; mettre à jour les outils de parsing pour le format `host:port` ; et exploiter `tls_version` dans les contrôles de conformité au chiffrement. Ces mécanismes s'appuient sur l'installation décrite en 10.8.1 et sur les catégories d'événements détaillées en 10.8.2.
