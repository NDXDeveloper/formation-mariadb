# 10.8.1 — Server Audit Plugin

Le *Server Audit Plugin* (`server_audit`) est le mécanisme d'audit présenté en 10.8. Cette section détaille son **installation** et sa **configuration** : comment l'activer, choisir les événements à tracer, diriger la sortie vers un fichier ou syslog, gérer la rotation, filtrer par utilisateur, et — point crucial pour la sécurité — empêcher sa désinstallation. La répartition des événements par type est approfondie en 10.8.2, et les améliorations récentes de la journalisation (buffer, `HOST:PORT`, version TLS) en 10.8.3.

---

## Ce que le plugin enregistre

Pour chaque session cliente, `server_audit` consigne **qui** s'est connecté (nom d'utilisateur et hôte), **quelles requêtes** ont été exécutées, **quelles tables** ont été accédées, et **quelles variables serveur** ont été modifiées. Ces informations sont écrites dans un **fichier journal rotatif** ou envoyées au **syslog** local.

---

## Installation

La bibliothèque `server_audit.so` (ou `.dll`) est fournie avec MariaDB, mais le plugin **n'est pas actif par défaut**. On l'installe dynamiquement, ou — préférable pour l'audit — on le charge dès le démarrage via le fichier de configuration :

```sql
INSTALL SONAME 'server_audit';
```

```ini
[mariadb]
plugin_load_add = server_audit
```

La bibliothèque doit se trouver dans le répertoire des plugins (variable `plugin_dir`).

---

## Activer la journalisation

Par défaut, la journalisation est **désactivée** (`server_audit_logging = OFF`). On l'active en passant cette variable à `ON`. Si la sortie est un fichier, ce basculement **crée et ouvre** le fichier journal ; il faut donc avoir défini `server_audit_file_path` au préalable.

```sql
SET GLOBAL server_audit_logging = ON;
```

Attention : certaines variables d'audit peuvent être **réinitialisées au redémarrage** du serveur. Pour une configuration durable, on les inscrit dans le **fichier de configuration** plutôt que de se contenter de `SET GLOBAL`.

---

## Choisir les événements

La variable `server_audit_events` détermine les types d'événements journalisés. **Si elle est vide, tous les types le sont** ; sinon, elle restreint à une liste séparée par des virgules. Les valeurs possibles :

- **CONNECT** — connexions, déconnexions, tentatives échouées ;
- **QUERY** — requêtes exécutées ;
- **TABLE** — accès aux tables ;
- et des sous-catégories de requêtes : **QUERY_DDL**, **QUERY_DML**, **QUERY_DCL**, **QUERY_DML_NO_SELECT**.

```sql
SET GLOBAL server_audit_events = 'CONNECT,QUERY,TABLE';
```

Le choix est un compromis (10.8) : auditer tout alourdit le journal, auditer trop étroit laisse passer des actions. Les sous-catégories (par exemple ne tracer que `QUERY_DDL,QUERY_DCL` pour les changements de structure et de droits) permettent un ciblage fin.

---

## Choisir la sortie : fichier ou syslog

`server_audit_output_type` vaut **`file`** ou **`syslog`**.

En mode **fichier**, `server_audit_file_path` fixe le chemin et le nom du journal (par défaut `server_audit.log` dans le répertoire de données) :

```ini
server_audit_output_type = file
server_audit_file_path   = /var/log/mysql/server_audit.log
```

En mode **syslog**, les événements sont remis au syslog du système, paramétrable via `server_audit_syslog_facility`, `server_audit_syslog_ident`, `server_audit_syslog_priority` et `server_audit_syslog_info`. C'est l'option à privilégier pour **centraliser** les journaux vers un collecteur ou un SIEM, conformément aux bonnes pratiques de 10.8.

---

## Rotation des fichiers

En mode fichier, le journal est **rotatif** :

- `server_audit_file_rotate_size` — taille (en octets) déclenchant une rotation automatique ;
- `server_audit_file_rotations` — nombre de fichiers conservés (par défaut **9**) ;
- `server_audit_file_rotate_now` — passé à `1`, force une rotation **immédiate**.

```sql
SET GLOBAL server_audit_file_rotate_size = 1000000;  -- 1 Mo
SET GLOBAL server_audit_file_rotations   = 30;
SET GLOBAL server_audit_file_rotate_now  = 1;        -- rotation manuelle
```

---

## Filtrer par utilisateur

On affine la journalisation par compte avec deux listes :

- `server_audit_incl_users` — comptes **à journaliser** explicitement ;
- `server_audit_excl_users` — comptes dont l'activité **n'est pas** journalisée.

```sql
SET GLOBAL server_audit_excl_users = 'monitoring, backup';
SET GLOBAL server_audit_incl_users = 'app_rw';
```

Deux subtilités importantes : les **enregistrements CONNECT ne sont pas affectés** par `excl_users` — les connexions sont **toujours** journalisées, même pour un utilisateur exclu (un choix logique, car les connexions sont des événements de sécurité essentiels). Et `incl_users` **l'emporte** : un compte qui y figure est journalisé même s'il est par ailleurs dans `excl_users`.

---

## Limiter la taille des requêtes journalisées

`server_audit_query_log_limit` (par défaut **1024**) fixe la longueur maximale du texte de requête consigné. Cela évite que des requêtes très longues ne saturent le journal, au prix d'une éventuelle troncature.

---

## Le piège du query cache

Si le **query cache** est activé et qu'une requête est servie **depuis le cache**, **aucun enregistrement TABLE** n'apparaît : le serveur n'a ni ouvert ni accédé aux tables, il a renvoyé un résultat mis en cache. Pour un audit **complet** des accès aux tables, il peut donc être nécessaire de **désactiver le query cache** — d'autant qu'il est de toute façon déprécié (chapitre 15.3).

---

## Durcir : empêcher la désinstallation du plugin

C'est un point souvent négligé. Un compte disposant des privilèges d'administration pourrait **désinstaller** le plugin d'audit (`UNINSTALL SONAME 'server_audit'`) pour **cesser de laisser des traces** — anéantissant l'intérêt de l'audit. Pour s'en prémunir, on charge le plugin au démarrage avec un niveau d'activation **permanent**, qui interdit sa désinstallation à chaud :

```ini
[mariadb]
plugin_load_add = server_audit
server_audit    = FORCE_PLUS_PERMANENT
```

Avec `FORCE_PLUS_PERMANENT`, le plugin est chargé et **ne peut plus être désinstallé** tant que le serveur tourne — une garantie d'inviolabilité du dispositif d'audit, à combiner avec la protection des fichiers journaux (10.8).

---

## Exemple de configuration

Une configuration d'audit typique, persistante et durcie, dans le fichier de configuration :

```ini
[mariadb]
plugin_load_add               = server_audit
server_audit                  = FORCE_PLUS_PERMANENT
server_audit_logging          = ON
server_audit_events           = CONNECT,QUERY,TABLE
server_audit_output_type      = file
server_audit_file_path        = /var/log/mysql/server_audit.log
server_audit_file_rotate_size = 1000000
server_audit_file_rotations   = 30
server_audit_excl_users       = monitoring
```

On vérifie l'ensemble des paramètres effectifs avec :

```sql
SHOW GLOBAL VARIABLES LIKE 'server_audit%';
```

---

## À retenir

Le *Server Audit Plugin* (`server_audit`) s'installe avec `INSTALL SONAME` ou, mieux, via `plugin_load_add` dans la configuration. La journalisation, **désactivée par défaut**, s'active avec `server_audit_logging = ON`. On choisit les événements via `server_audit_events` (**vide = tout** ; sinon `CONNECT`, `QUERY`, `TABLE`, et sous-catégories `QUERY_DDL/DML/DCL/DML_NO_SELECT`), la sortie via `server_audit_output_type` (**file** avec `server_audit_file_path`, ou **syslog** pour la centralisation), et la rotation via `server_audit_file_rotate_size`/`_rotations`. Le filtrage par compte (`incl_users`/`excl_users`) **ne désactive jamais les CONNECT**. Attention au **query cache** (pas d'enregistrement TABLE sur résultat mis en cache). Pour l'**inviolabilité**, charger le plugin en `FORCE_PLUS_PERMANENT` afin d'empêcher sa désinstallation. La configuration se met dans le fichier d'options, certaines variables étant réinitialisées au redémarrage.

---

> 🔔 **Note de version.** Le *Server Audit Plugin* et la famille `server_audit_*` sont stables dans **MariaDB 12.3 LTS** ; le plugin reste fourni mais non actif par défaut. Les améliorations 12.x — buffer d'écriture (`server_audit_file_buffer_size`), journalisation `HOST:PORT` et champ de version TLS — sont traitées en 10.8.3. Le détail des types d'événements est en 10.8.2. Sources : *MariaDB Server Documentation — MariaDB Audit Plugin* (Installation, Configuration, Log Settings, Options and System Variables).
