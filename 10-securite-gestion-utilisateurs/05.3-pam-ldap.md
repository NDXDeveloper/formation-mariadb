# 10.5.3 — PAM et LDAP

Les plugins vus jusqu'ici (`mysql_native_password`, `ed25519`) gèrent eux-mêmes les mots de passe. Le plugin **PAM** appartient à l'autre famille évoquée en 10.5 : celle de l'**identité externe**. Plutôt que de vérifier le mot de passe lui-même, il **délègue** l'authentification au cadre PAM (*Pluggable Authentication Modules*) du système d'exploitation. PAM devient alors une passerelle vers une multitude de mécanismes : comptes système, **LDAP**, Active Directory, Kerberos, mots de passe à usage unique, clés SSH… Cette section présente le plugin PAM, ses deux versions, son point de vigilance sécurité (le mot de passe en clair) et son usage le plus courant : l'authentification contre un annuaire LDAP.

---

## Le principe : déléguer à PAM

PAM est un cadre d'authentification utilisé par Linux, FreeBSD, Solaris et les autres systèmes de type Unix. Le plugin `pam` de MariaDB confie à ce cadre la vérification de l'identité du client. **Conséquence importante : Windows ne prend pas en charge PAM**, donc le plugin ne fonctionne pas sur un serveur Windows (un client Windows peut néanmoins se connecter à un serveur Unix configuré pour PAM).

La manière dont l'authentification est réellement effectuée dépend entièrement de la configuration de PAM. PAM est organisé en **services**, décrits par des fichiers de configuration situés dans `/etc/pam.d/` (par exemple `/etc/pam.d/mariadb` pour un service nommé `mariadb`). Selon les modules activés dans ce service, PAM permet une grande variété de scénarios :

- mots de passe `/etc/shadow` (module `pam_unix`) — la configuration par défaut la plus courante ;
- **LDAP** (`pam_ldap`) et **Active Directory** (`pam_winbind`, `pam_lsass`, `pam_centrifydc`) ;
- mots de passe à usage unique / double authentification (`pam_google_authenticator`, `pam_securid`) ;
- clés SSH (`pam_ssh`) ;
- mapping d'utilisateurs et de groupes (`pam_user_map`) ;
- restrictions horaires (`pam_time`), journalisation de chaque tentative, combinaison de plusieurs modules…

C'est cette richesse qui fait l'intérêt du plugin : il ouvre MariaDB à toute la chaîne d'authentification du système.

---

## Créer un compte PAM

Comme tout plugin, PAM se déclare avec `IDENTIFIED VIA`. La clause `USING` permet de préciser le **service PAM** à employer :

```sql
CREATE USER 'chris'@'%' IDENTIFIED VIA pam;                 -- service PAM par défaut
CREATE USER 'chris'@'%' IDENTIFIED VIA pam USING 'mariadb'; -- service PAM 'mariadb'
```

Le nom de service renvoie au fichier `/etc/pam.d/<service>`. Un service minimal authentifiant via les mots de passe Unix ressemble à :

```text
# /etc/pam.d/mariadb
auth     required   pam_unix.so
account  required   pam_unix.so
```

---

## Les deux versions du plugin : v1 et v2

Le plugin PAM existe en deux versions, et la distinction est **importante pour la sécurité**.

Historiquement (version 1), le plugin exécutait la pile PAM **dans le processus du serveur**. Pour lire `/etc/shadow` avec `pam_unix`, il fallait donc autoriser l'utilisateur `mysql` à accéder à ce fichier — typiquement en l'ajoutant au groupe `shadow` :

```bash
sudo groupadd shadow
sudo usermod -a -G shadow mysql
sudo chown root:shadow /etc/shadow
sudo chmod g+r /etc/shadow
```

Or ouvrir l'accès à `/etc/shadow` constitue un **risque de sécurité**. **Depuis MariaDB 10.4.0**, le plugin utilise un **processus auxiliaire setuid** (`auth_pam_tool`) pour effectuer ses vérifications PAM : le serveur n'a alors plus besoin de ce contournement pour les opérations privilégiées comme la lecture de `/etc/shadow`.

Concrètement, en 10.4.0 et au-delà (donc en 12.3) :

- la bibliothèque **`auth_pam`** correspond à la **version 2.0** (avec le *wrapper* setuid) — c'est celle qu'on installe par défaut ;
- la **version 1.0** reste disponible sous le nom **`auth_pam_v1`**, à n'installer que si un besoin spécifique l'impose.

```sql
INSTALL SONAME 'auth_pam';      -- version 2.0 (recommandée)
INSTALL SONAME 'auth_pam_v1';   -- version 1.0 (compatibilité)
```

En pratique, on privilégie la **version 2.0** pour son meilleur cloisonnement.

## Le mot de passe transite en clair : TLS obligatoire

Point de vigilance majeur. Pour transmettre le mot de passe à PAM, le plugin s'appuie sur le plugin client `dialog`, qui **envoie le mot de passe au serveur en clair**. Il est donc **impératif de chiffrer les connexions par TLS** (section 10.7) afin que ces mots de passe ne soient pas exposés sur le réseau.

Une alternative existe : le plugin client `mysql_clear_password`, activable via la variable serveur `pam_use_cleartext_plugin`. Il est toutefois plus limité — il ne convient qu'aux services PAM **à mot de passe simple** (une seule question), là où `dialog` gère des échanges plus riches (par exemple une double authentification). Côté clients, la prise en charge varie selon les connecteurs : certains (comme MariaDB Connector/J ou Connector/Node.js) gèrent PAM indépendamment de cette variable, tandis que d'autres (comme MySqlConnector pour ADO.NET) exigent que `pam_use_cleartext_plugin` soit activé.

---

## LDAP via PAM

MariaDB ne dispose pas d'un plugin LDAP « natif » dans l'édition Community : l'authentification LDAP s'obtient **à travers PAM**, en configurant le service PAM pour qu'il interroge l'annuaire. Le module historique est `pam_ldap` ; sur les distributions modernes (RHEL 8+ et équivalentes) qui s'appuient sur **SSSD**, c'est le module `pam_sss.so` qui en tient lieu.

Un service PAM dédié à l'authentification LDAP se résume souvent à remplacer `pam_unix.so` par `pam_ldap.so` :

```text
# /etc/pam.d/mariadb (authentification LDAP)
auth     required   pam_ldap.so
account  required   pam_ldap.so
```

L'authentification contre **Active Directory** suit la même logique avec les modules adaptés (`pam_winbind`, `pam_lsass`, `pam_centrifydc`). MariaDB fournit un guide dédié (« Configuring PAM Authentication and User Mapping with LDAP Authentication ») détaillant la mise en place complète, y compris la configuration de l'annuaire ; on s'y reportera pour le déroulé pas à pas.

## Mapping d'utilisateurs et comptes proxy

Authentifier un utilisateur LDAP, c'est bien ; mais à quel **compte MariaDB** correspond-il ? Le module `pam_user_map` répond à cette question : il **associe** des utilisateurs ou des groupes (LDAP, système…) à des comptes MariaDB. Par exemple, l'utilisateur LDAP `foo` peut être mappé vers le compte `bar`, et tout membre du groupe LDAP `dba` vers le compte `dba`.

Ce mécanisme s'appuie sur les **comptes proxy** (le privilège `PROXY` vu en 10.3) : un compte d'authentification unique délègue son identité au compte cible, qui porte les privilèges réels.

```sql
-- Le compte anonyme s'authentifie via PAM…
CREATE USER ''@'%' IDENTIFIED VIA pam USING 'mariadb';
-- … et peut « devenir » bar ou dba selon le mapping pam_user_map
GRANT PROXY ON 'bar'@'%' TO ''@'%';
GRANT PROXY ON 'dba'@'%' TO ''@'%';
```

Lorsque `foo` se connecte, PAM l'authentifie via LDAP, `pam_user_map` le mappe vers `bar`, et la session s'exécute avec les privilèges de `bar`. On gère ainsi un grand nombre d'utilisateurs d'annuaire avec une poignée de comptes MariaDB porteurs de droits.

---

## Quand l'utiliser

Le plugin PAM est le choix de l'**authentification centralisée** et de l'**intégration au système d'information** : annuaire d'entreprise (LDAP/AD), comptes système, double authentification, SSO. Il évite de dupliquer les mots de passe dans MariaDB et permet d'appliquer les politiques de l'annuaire (expiration, verrouillage, etc.) de façon uniforme. Ses contreparties sont la dépendance à une configuration système (hors SQL), le transit du mot de passe en clair (d'où TLS), et l'absence de support sur serveur Windows.

---

## À retenir

Le plugin **PAM** délègue l'authentification au cadre PAM du système (Linux/Unix uniquement), ouvrant l'accès à LDAP, Active Directory, Kerberos, 2FA et bien d'autres via les modules PAM appropriés. On crée un compte avec `IDENTIFIED VIA pam [USING 'service']`, le service renvoyant à `/etc/pam.d/<service>`. **Depuis MariaDB 10.4, la v2 du plugin (`auth_pam`) utilise un processus setuid (`auth_pam_tool`)** qui évite d'ouvrir `/etc/shadow` au serveur (la v1 reste disponible sous `auth_pam_v1`). Le mot de passe transitant **en clair** (`dialog`, ou `mysql_clear_password`), le **chiffrement TLS est impératif**. L'authentification **LDAP** passe par `pam_ldap` (ou `pam_sss`/SSSD), et `pam_user_map` mappe utilisateurs/groupes vers des comptes MariaDB via des **comptes proxy** (`GRANT PROXY`). C'est l'outil de l'authentification d'entreprise centralisée.

---

> 🔔 **Note de version.** Dans **MariaDB 12.3 LTS**, la version par défaut du plugin PAM est la **2.0** (processus setuid `auth_pam_tool`), introduite en 10.4.0 (MDEV-7032) ; la v1.0 demeure disponible sous `auth_pam_v1`. Le plugin client `mysql_clear_password` et la variable `pam_use_cleartext_plugin` existent depuis 5.5.32 / 10.1.1. Le plugin est fourni dans les paquets binaires Linux et ne fonctionne pas sur serveur Windows. Sources : *MariaDB Server Documentation — Authentication Plugin PAM* et *Configuring PAM Authentication and User Mapping with LDAP*.
