# 10.2.1 — DROP USER : avertissement si sessions actives (échec en mode Oracle) 🆕

Comme vu en 10.2, `DROP USER` supprime un compte et retire l'ensemble de ses privilèges des tables de droits. Mais que se passe-t-il si le compte possède encore des **sessions ouvertes** au moment de sa suppression ? Historiquement, ce cas était traité de façon discrète, au point de surprendre. La série 12.x le rend explicite. Cette section détaille ce comportement, le rôle de `KILL CONNECTION`, et le statut — encore à venir — de la clause `FORCE`.

---

## Le comportement historique (avant la 12.1)

Jusqu'à la version 12.1, supprimer un compte connecté ne mettait pas fin à ses sessions. Les connexions existantes continuaient de fonctionner — avec l'intégralité des privilèges du compte — jusqu'à ce que le client se déconnecte de lui-même. Seules les **nouvelles** connexions étaient refusées. L'opération se faisait sans aucun avertissement.

Le risque était réel : un administrateur pouvait croire avoir révoqué un accès, alors qu'une session active poursuivait son travail, parfois pendant des heures. Cette absence de signal rendait le comportement difficile à diagnostiquer.

---

## Le nouveau comportement (12.1 → 12.3)

La série 12.x rend ce cas explicite. Le comportement dépend désormais du `sql_mode`.

### Mode standard : un avertissement

En mode standard, `DROP USER` réussit toujours — le compte est retiré des tables de droits et ne peut plus établir de nouvelle connexion —, mais le serveur émet maintenant un **avertissement** (de niveau *Note*) signalant que les comptes supprimés conservent des connexions actives.

```sql
DROP USER 'foo'@'localhost';
-- Query OK, 0 rows affected, 1 warning
SHOW WARNINGS;
```

L'avertissement (code 4227) liste les comptes concernés et rappelle qu'il faut recourir à `KILL CONNECTION` pour mettre fin à leurs sessions si elles ne doivent plus être utilisées.

Un point demeure inchangé sous le capot : tant qu'une session existante n'est pas effectivement fermée, elle conserve ses privilèges. L'avertissement ne modifie pas ce fait, il le rend simplement **visible**.

### Mode Oracle : un échec

Lorsque le `sql_mode` inclut `ORACLE`, `DROP USER` ne se contente plus d'avertir : l'instruction **échoue** dès qu'un des comptes nommés possède une session active, et **aucun** compte n'est supprimé. Ce comportement s'aligne sur celui d'Oracle, où l'on rencontre l'erreur *ORA-01940 : cannot drop a user that is currently connected*.

```sql
SET sql_mode = 'ORACLE';
DROP USER 'foo'@'localhost';
-- ERROR 1396 (HY000): Operation DROP USER failed for 'foo'@'localhost'
```

L'erreur renvoyée est `ER_CANNOT_USER` (code 1396), la même famille d'erreur que `DROP USER` utilise déjà pour ses autres échecs. Pour supprimer le compte, il faut d'abord **fermer ses sessions** avec `KILL CONNECTION` (voir plus bas).

---

## La clause `FORCE` : documentée, mais absente de la 12.3

La documentation « Current » de MariaDB décrit une clause **`FORCE`** censée fermer d'office les connexions des comptes nommés et les supprimer **immédiatement** (ce qui éviterait aussi l'erreur du mode Oracle). **Mais cette clause n'est pas implémentée dans la 12.3 LTS** : sur 12.3.2, l'employer provoque une simple erreur de syntaxe.

```sql
DROP USER 'foo'@'localhost' FORCE;
-- ERROR 1064 (42000): You have an error in your SQL syntax ... near 'FORCE'
```

En 12.3, la syntaxe de `DROP USER` reste donc, sans `FORCE` :

```sql
DROP USER [IF EXISTS] compte [, compte] ...
```

Pour forcer la suppression d'un compte connecté en 12.3, on **ferme explicitement ses sessions avec `KILL CONNECTION`** (ci-dessous), puis on exécute `DROP USER`. La clause `FORCE` pourra être employée lorsqu'elle sera réellement disponible dans une version ultérieure.

---

## Terminer les sessions manuellement : `KILL CONNECTION`

En 12.3, c'est **la** façon de forcer la suppression d'un compte connecté : on identifie ses sessions dans la liste des processus, on les termine, puis on exécute `DROP USER` :

```sql
SELECT Id, User, Host, Command, Time
FROM information_schema.PROCESSLIST
WHERE User = 'foo';

KILL CONNECTION 42;  -- met fin à la session d'identifiant 42
```

Mettre fin à la session d'un autre compte requiert le privilège `CONNECTION_ADMIN` (ou `SUPER` sur les serveurs plus anciens). Une fois toutes les sessions fermées, un `DROP USER` ordinaire réussit sans avertissement (mode standard) et sans erreur (mode Oracle).

---

## Récapitulatif

| Situation | Mode standard | Mode Oracle |
|---|---|---|
| Aucune session active | Suppression silencieuse | Suppression silencieuse |
| Sessions actives | Suppression **+ avertissement** (Note 4227) | **Échec** (ERROR 1396), aucune suppression |
| Après `KILL CONNECTION` des sessions | Suppression silencieuse | Suppression silencieuse |

> En mode standard, « suppression » signifie que le compte est retiré des tables de droits, mais que ses sessions existantes continuent de fonctionner jusqu'à leur terminaison. (La clause `FORCE` qui fermerait ces sessions automatiquement n'est pas encore disponible en 12.3 — voir plus haut.)

---

## Impact sur la migration

Il s'agit d'un **changement de comportement** à anticiper lors d'une montée de version de la 11.8 vers la 12.3 (voir chapitre 19.10). Les scripts qui suppriment des comptes potentiellement connectés ne réagissent plus de la même façon : ils émettent désormais des avertissements et, surtout, ils **échouent en mode Oracle**. Les automatisations concernées (démantèlement d'environnements de *provisioning*, jeux de test, scripts de nettoyage) devraient ajouter une étape `KILL CONNECTION` explicite là où c'est pertinent (la clause `FORCE` n'étant pas disponible en 12.3), et les environnements configurés en mode Oracle doivent revoir leur logique de suppression — sous peine de voir le `DROP USER` échouer.

---

## À retenir

- **Mode standard** : `DROP USER` sur un compte connecté réussit mais émet un avertissement (Note 4227) ; les sessions existantes subsistent jusqu'à leur terminaison ; les nouvelles connexions sont refusées.
- **Mode Oracle** : `DROP USER` échoue (ERROR 1396) si le compte est connecté ; aucun compte n'est supprimé.
- **`KILL CONNECTION`** est, en 12.3, le moyen de fermer les sessions avant un `DROP USER` ; il exige `CONNECTION_ADMIN` pour les sessions d'autres comptes.
- **`FORCE`** (fermeture des sessions + suppression immédiate) est décrite dans la documentation MariaDB mais **n'est pas implémentée en 12.3** (erreur de syntaxe sur 12.3.2) ; elle relève d'une version ultérieure.

---

> 🔔 **Note de version.** L'avertissement en mode standard (Note **4227**) et l'échec en mode Oracle (ERROR **1396**) ont été introduits en **MariaDB 12.1** (MDEV-35617) et font partie de la **12.3 LTS** (comportement et codes **vérifiés sur 12.3.2**). Avant la 12.1, la suppression d'un compte connecté se faisait silencieusement. En revanche, la clause **`FORCE`** — bien que présente dans la documentation « Current » de MariaDB — **n'est pas disponible en 12.3** : sur 12.3.2, `DROP USER … FORCE` renvoie une erreur de syntaxe (ERROR 1064). Elle relève d'une version postérieure à la 12.3. Sources : *MariaDB Server Documentation — DROP USER* et *MariaDB Community Server 12.1/12.3 Changes & Improvements*.
