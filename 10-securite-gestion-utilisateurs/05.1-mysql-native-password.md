# 10.5.1 — mysql_native_password

`mysql_native_password` est le plugin d'authentification **par défaut** de MariaDB : c'est lui qui est employé lorsqu'on crée un compte avec `IDENTIFIED BY` sans préciser de méthode (10.5). Il fournit une authentification par mot de passe conventionnelle, héritée de MySQL, fondée sur l'algorithme de hachage **SHA-1**. Sa grande force est sa compatibilité universelle ; sa faiblesse, une sécurité aujourd'hui datée. Cette section en détaille le fonctionnement, l'usage et les limites.

---

## Fonctionnement

Le serveur ne stocke jamais le mot de passe en clair, mais une **empreinte** calculée par un double hachage SHA-1 du mot de passe. Cette empreinte se présente comme une chaîne de 41 caractères préfixée d'un astérisque (par exemple `*6BB4837EBA4...`) et correspond à ce que renvoie la fonction `PASSWORD()` :

```sql
SELECT PASSWORD('mot_de_passe');   -- empreinte native : 41 caractères, préfixe '*'
```

L'authentification proprement dite repose sur un échange **défi-réponse** : le mot de passe en clair ne traverse pas le réseau. À la connexion, le serveur envoie un nombre aléatoire (un *scramble*) ; le client en dérive une réponse à partir de son mot de passe ; le serveur vérifie cette réponse contre l'empreinte stockée, sans jamais reconstituer le mot de passe d'origine. Ce mécanisme protège le mot de passe lui-même, mais pas l'empreinte — un point décisif pour la sécurité, abordé plus bas.

---

## Créer et modifier un compte avec ce plugin

Comme c'est le plugin par défaut, `IDENTIFIED BY` suffit à l'employer ; on peut aussi le désigner explicitement, ou fournir une empreinte déjà calculée :

```sql
-- Plugin par défaut (mysql_native_password)
CREATE USER 'alice'@'localhost' IDENTIFIED BY 'mot_de_passe';

-- Choix explicite du plugin
CREATE USER 'bob'@'%' IDENTIFIED VIA mysql_native_password USING PASSWORD('mot_de_passe');

-- À partir d'une empreinte pré-calculée
CREATE USER 'carol'@'%' IDENTIFIED BY PASSWORD '*6BB4837EBA4...';
```

Particularité utile de ce plugin : la fonction `PASSWORD()` et l'instruction `SET PASSWORD` fonctionnent directement avec lui (ce qui n'est pas le cas d'`ed25519`, par exemple) :

```sql
SET PASSWORD FOR 'alice'@'localhost' = PASSWORD('nouveau_mot_de_passe');
ALTER USER 'alice'@'localhost' IDENTIFIED BY 'nouveau_mot_de_passe';
```

---

## Avantages : compatibilité et simplicité

Deux atouts expliquent la longévité de ce plugin :

- Il est **intégré statiquement** au serveur : aucune installation n'est nécessaire, il est toujours disponible.
- Étant l'une des méthodes d'authentification historiques, il est **pris en charge par tous les clients et pilotes**. C'est la raison principale de l'employer encore : garantir qu'un connecteur ancien, ou un outil tiers, puisse se connecter sans difficulté.

C'est donc le choix de la **compatibilité maximale** — au prix, on va le voir, d'une sécurité en retrait.

---

## Limites de sécurité

Le talon d'Achille de `mysql_native_password` est son algorithme. SHA-1 était considéré comme sûr en 2001 ; il ne l'est plus face à la puissance de calcul actuelle. Deux faiblesses concrètes en découlent :

- **Empreinte non salée.** Le hachage ne comporte pas de sel : deux comptes ayant le même mot de passe ont la même empreinte, et celle-ci est exposée aux attaques par dictionnaire, par force brute hors ligne ou par tables arc-en-ciel (*rainbow tables*) — particulièrement efficaces sur les mots de passe faibles.
- **Empreinte sensible.** Si un attaquant parvient **à la fois** à écouter le protocole de connexion (capturer un échange défi-réponse) **et** à obtenir une copie de la table `mysql.user`, il dispose alors de tout ce qu'il faut pour s'authentifier. L'empreinte stockée n'est donc pas un simple « hash » inoffensif : sa fuite est lourde de conséquences.

Deux mesures atténuent ces risques sans changer de plugin : chiffrer les connexions par **TLS** (section 10.7), qui empêche la capture des échanges sur le réseau ; et **protéger rigoureusement** la base `mysql` et ses sauvegardes. Mais pour de nouveaux comptes à forte exigence de sécurité, la vraie réponse est de **changer de plugin** (voir plus bas).

---

## Détail MariaDB : les colonnes `Password` et `authentication_string`

Pour des raisons de compatibilité, `mysql_native_password` lit l'empreinte dans **deux** colonnes de `mysql.user` : l'ancienne colonne `Password` et la colonne `authentication_string`. Par le passé, une divergence entre ces deux colonnes pouvait provoquer des comportements incohérents. Depuis **MariaDB 10.2.19 et 10.3.11**, les instructions `CREATE USER`, `ALTER USER`, `GRANT` et `SET PASSWORD` renseignent **systématiquement les deux colonnes** lors de tout changement de mot de passe (MDEV-16774), ce qui élimine cette source d'incohérence.

---

## Inspecter

On vérifie le plugin et l'empreinte associés à chaque compte directement dans `mysql.user` :

```sql
SELECT User, Host, plugin, authentication_string FROM mysql.user;
```

Une valeur `mysql_native_password` dans la colonne `plugin` indique un compte authentifié par cette méthode.

---

## Quand l'utiliser — et vers quoi migrer

`mysql_native_password` se justifie surtout pour la **compatibilité avec des clients ou pilotes anciens** qui ne savent pas dialoguer avec des plugins plus récents. En dehors de ce cas, il vaut mieux l'éviter pour toute nouvelle installation soucieuse de sécurité.

Le contexte de migration mérite une note : **MySQL a déprécié** `mysql_native_password` (depuis la 8.0.34) au profit de `caching_sha2_password`, tandis que **MariaDB le conserve** comme plugin par défaut, pour la compatibilité — tout en déconseillant son usage en haute sécurité. Pour faire évoluer un compte vers une méthode plus robuste, il suffit de changer son plugin :

```sql
-- Migrer un compte vers ed25519
ALTER USER 'alice'@'localhost' IDENTIFIED VIA ed25519 USING PASSWORD('mot_de_passe');
```

Les alternatives recommandées — `ed25519` et PARSEC — sont détaillées respectivement en 10.5.2 et 10.6.

---

## À retenir

`mysql_native_password` est le plugin **par défaut** de MariaDB : authentification par mot de passe conventionnelle, fondée sur un double hachage **SHA-1** (empreinte de 41 caractères, préfixe `*`, calculée par `PASSWORD()`). Le mot de passe ne circule jamais en clair grâce à un échange défi-réponse, et `PASSWORD()`/`SET PASSWORD` fonctionnent avec ce plugin. Son atout est une **compatibilité universelle** (intégré statiquement, supporté par tous les clients) ; sa faiblesse, un algorithme **daté et non salé**, dont l'empreinte est sensible (combinée à une écoute du réseau, elle suffit à s'authentifier). À réserver à la **compatibilité ascendante** ; pour de nouveaux comptes sécurisés, lui préférer `ed25519` (10.5.2) ou PARSEC (10.6), en protégeant les connexions par TLS (10.7).

---

> 🔔 **Note de version.** Dans **MariaDB 12.3 LTS**, `mysql_native_password` reste le plugin par défaut et demeure pleinement supporté, contrairement à MySQL qui l'a déprécié depuis la 8.0.34. Le renseignement conjoint des colonnes `Password` et `authentication_string` est acquis depuis la 10.2.19 / 10.3.11.
