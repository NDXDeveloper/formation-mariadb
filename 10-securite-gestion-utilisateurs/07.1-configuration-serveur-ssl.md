# 10.7.1 — Configuration serveur SSL

Cette section décrit comment **activer TLS côté serveur** par configuration explicite. Comme indiqué en 10.7, le serveur n'offre une connexion chiffrée qu'à condition de disposer d'un **certificat** et d'une **clé privée**. On verra ici quelles variables système renseigner, comment restreindre les versions du protocole, comment protéger les fichiers et comment vérifier que TLS est bien actif. La création des certificats et de l'autorité de certification fait l'objet de la section 10.7.2 ; le TLS automatique (qui dispense de cette configuration dans certains cas) est traité en 10.7.3.

---

## Les variables système de TLS serveur

Pour compatibilité historique, les variables liées à TLS conservent le préfixe `ssl_` (bien que le protocole soit TLS). Les notations avec tiret (`ssl-cert`) et avec souligné (`ssl_cert`) sont toutes deux acceptées. Les principales sont :

- **`ssl_cert`** — chemin du **certificat X.509** du serveur (format PEM).
- **`ssl_key`** — chemin de la **clé privée** du serveur (PEM).
- **`ssl_ca`** — chemin de la **chaîne de l'autorité de certification** qui valide le certificat (et les certificats clients). À renseigner sauf si le certificat est signé par une CA déjà reconnue.
- **`ssl_capath`** — répertoire contenant plusieurs certificats de CA, alternative à `ssl_ca`.
- **`ssl_cipher`** — liste des **chiffrements** autorisés.
- **`tls_version`** — **versions du protocole TLS** autorisées.

---

## Activer TLS : la marche à suivre

La configuration s'effectue dans un fichier d'options, sous la section `[mariadbd]` (ou `[mysqld]` sur les versions antérieures à la 10.5). L'emplacement du fichier dépend de la distribution : `/etc/mysql/mariadb.conf.d/50-server.cnf` sous Debian/Ubuntu, `/etc/my.cnf.d/server.cnf` sous RHEL/CentOS, par exemple. On évite de modifier les fichiers de configuration fournis par les paquets ; on ajoute plutôt un fichier dédié.

La démarche se résume ainsi : obtenir (ou générer, voir 10.7.2) un certificat X.509 et une clé privée pour le serveur ; renseigner les variables dans le fichier de configuration ; **redémarrer le serveur** ; puis vérifier que TLS est actif.

```ini
[mariadbd]
ssl_cert = /etc/mysql/ssl/server-cert.pem
ssl_key  = /etc/mysql/ssl/server-key.pem
ssl_ca   = /etc/mysql/ssl/ca-cert.pem
tls_version = TLSv1.2,TLSv1.3
```

Un point de contexte important : **avant l'introduction du TLS automatique** (série 11.x, voir 10.7.3), le serveur n'offrait TLS qu'une fois ces variables configurées. Depuis, MariaDB active TLS automatiquement pour les connexions non locales ; la configuration manuelle décrite ici reste néanmoins la voie à suivre lorsqu'on fournit ses propres certificats (par exemple signés par une CA d'entreprise). Dans tous les cas, une fois TLS disponible, il n'est **pas obligatoire par défaut** : c'est `require_secure_transport` ou la clause `REQUIRE` (10.7) qui l'imposent.

---

## Restreindre les versions du protocole

La variable `tls_version` limite les versions TLS que le serveur accepte. On recommande de n'autoriser que **TLS 1.2 et 1.3**, les versions 1.0 et 1.1 étant dépréciées et à proscrire :

```ini
[mariadbd]
tls_version = TLSv1.2,TLSv1.3
```

À noter : la variable `tls_version` est disponible à partir de MariaDB 10.4.6 ; sur les versions plus anciennes, le choix des versions dépendait de la configuration d'OpenSSL. En 12.3, elle est pleinement utilisable.

---

## Restreindre les chiffrements

`ssl_cipher` permet de n'autoriser que des suites de chiffrement modernes, en écartant les algorithmes faibles :

```ini
[mariadbd]
ssl_cipher = ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384
```

En l'absence de contrainte particulière, les valeurs par défaut du serveur conviennent généralement ; on durcit `ssl_cipher` surtout pour répondre à des exigences de conformité.

---

## Permissions des fichiers

La distinction est essentielle pour la sécurité :

- le **certificat du serveur** et le **certificat de la CA** ne sont pas secrets : ils peuvent être lisibles par tous ;
- la **clé privée du serveur** (et la clé de la CA) sont **confidentielles** : elles ne doivent être lisibles que par l'utilisateur `mysql`.

```bash
chown mysql:mysql /etc/mysql/ssl/server-key.pem
chmod 400 /etc/mysql/ssl/server-key.pem
```

Une clé privée laissée lisible par tous ruinerait l'intérêt du chiffrement.

---

## Vérifier que TLS est activé

La variable d'état globale `have_ssl` indique l'état du chiffrement côté serveur :

```sql
SHOW GLOBAL VARIABLES LIKE 'have_ssl';
-- YES      => TLS activé et opérationnel
-- DISABLED => certificats non chargés (problème de chemin ou de permission)
-- NO       => binaire compilé sans support TLS
```

La valeur **`DISABLED`** est le diagnostic le plus courant : elle signifie que MariaDB n'a **pas pu charger** les certificats. Il faut alors consulter le **journal d'erreurs** du serveur, où figure presque toujours la cause (chemin incorrect, fichier illisible par l'utilisateur `mysql`). On peut aussi écrire `SELECT @@have_ssl;`.

Pour inspecter les paramètres effectifs et l'état de la session :

```sql
SHOW VARIABLES LIKE '%ssl%';            -- chemins et options TLS configurés
SHOW VARIABLES LIKE 'tls_version';      -- versions de protocole autorisées
SHOW SESSION STATUS LIKE 'Ssl_cipher';  -- chiffrement de la connexion en cours
SHOW SESSION STATUS LIKE 'Ssl_version'; -- version TLS de la connexion en cours
```

Une connexion réellement chiffrée affiche un chiffrement (par exemple `TLS_AES_256_GCM_SHA384`) et une version (`TLSv1.3`).

---

## TLS disponible n'est pas TLS obligatoire

Activer TLS sur le serveur le rend **disponible**, mais les clients restent libres de s'en passer tant qu'on ne l'exige pas. Le rappel de 10.7 s'applique : on impose le chiffrement globalement avec `require_secure_transport = ON`, ou par compte avec `REQUIRE SSL` / `REQUIRE X509`. La colonne `ssl_type` de `mysql.user` permet de contrôler les exigences par compte :

```sql
SELECT user, host, ssl_type FROM mysql.user;
-- ssl_type = ANY  => REQUIRE SSL
-- ssl_type = X509 => REQUIRE X509
```

---

## La bibliothèque TLS

MariaDB s'appuie sur une bibliothèque cryptographique choisie à la compilation — le plus souvent **OpenSSL** (recommandée), parfois WolfSSL ou GnuTLS. Le choix influe sur certaines capacités, notamment la prise en charge de **TLS 1.3** (qui requiert une version récente d'OpenSSL). En pratique, les paquets officiels sont construits avec une bibliothèque adaptée ; il n'y a rien à faire de particulier sur une installation standard.

---

## À retenir

Activer TLS côté serveur consiste à renseigner, dans un fichier d'options sous `[mariadbd]`, le **certificat** (`ssl_cert`), la **clé privée** (`ssl_key`) et la **chaîne de CA** (`ssl_ca`/`ssl_capath`), puis à **redémarrer** le serveur. On restreint les versions avec `tls_version = TLSv1.2,TLSv1.3` (variable disponible depuis la 10.4.6) et, au besoin, les chiffrements avec `ssl_cipher`. La **clé privée** doit être réservée à l'utilisateur `mysql`. La vérification se fait avec `have_ssl` : `YES` (actif), `DISABLED` (certificats non chargés — voir le journal d'erreurs), `NO` (pas de support). TLS rendu disponible n'est **pas obligatoire** tant qu'on ne l'impose pas (`require_secure_transport`, `REQUIRE`). La génération des certificats est traitée en 10.7.2.

---

> 🔔 **Note de version.** Les variables `ssl_cert`, `ssl_key`, `ssl_ca` et la vérification par `have_ssl` sont stables dans **MariaDB 12.3 LTS** ; la variable `tls_version` est disponible depuis la 10.4.6. Le **TLS automatique** introduit dans la série 11.x (voir 10.7.3) rend la configuration manuelle facultative dans les cas simples, mais celle-ci reste nécessaire pour des certificats fournis par l'administrateur. La protection d'une clé privée par passphrase est traitée en 10.7.4. Sources : *MariaDB Server Documentation — Enabling TLS on MariaDB Server* et *Securing Connections for Client and Server*.
