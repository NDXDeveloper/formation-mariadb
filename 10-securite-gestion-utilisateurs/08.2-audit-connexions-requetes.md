# 10.8.2 — Audit de connexions et requêtes

Une fois le **Server Audit Plugin** chargé et la journalisation activée (`server_audit_logging = ON`, voir la section 10.8.1), encore faut-il décider *quoi* auditer. Dans la pratique, deux familles d'événements concentrent l'essentiel de la valeur pour la sécurité et la conformité : les **connexions** (qui se connecte, depuis où, avec quel résultat) et les **requêtes** (quelles instructions sont exécutées, sur quelles données). Cette section détaille la configuration de ces deux axes, le format des enregistrements produits et la manière de les restreindre pour rester exploitables.

## Choisir les événements à journaliser

Le périmètre de l'audit est piloté par la variable système `server_audit_events`. Elle accepte une liste de catégories séparées par des virgules. Lorsqu'elle est laissée **vide** (sa valeur par défaut), le plugin journalise l'ensemble des événements : connexions, requêtes et accès aux objets. En production, on précise au contraire explicitement les catégories pour ne conserver que celles qui sont utiles.

| Valeur | Événements capturés |
|--------|---------------------|
| `CONNECT` | Connexions réussies, déconnexions, tentatives échouées et opérations `CHANGE USER` |
| `QUERY` | Texte intégral de toutes les requêtes exécutées, avec leur code retour |
| `TABLE` | Objets (tables) effectivement lus ou modifiés lors de l'exécution d'une requête |
| `QUERY_DDL` | Sous-ensemble de `QUERY` : uniquement les instructions DDL (`CREATE`, `ALTER`, `DROP`, `RENAME`…) |
| `QUERY_DML` | Sous-ensemble de `QUERY` : uniquement le DML (`INSERT`, `UPDATE`, `DELETE`, `SELECT`) |
| `QUERY_DML_NO_SELECT` | DML *sans* les `SELECT` — capture les écritures sans le bruit des lectures |
| `QUERY_DCL` | Sous-ensemble de `QUERY` : uniquement le DCL (`GRANT`, `REVOKE`, `CREATE USER`, `SET PASSWORD`…) |

Les sous-catégories `QUERY_*` sont précieuses pour réduire le volume tout en gardant l'essentiel. Une configuration de surveillance courante combine ainsi `CONNECT`, `QUERY_DDL` (suivi des changements de schéma) et `QUERY_DCL` (suivi des changements de privilèges), sans journaliser le flux des `SELECT` applicatifs.

## Le format d'un enregistrement d'audit

Chaque événement génère une ligne au format texte délimité par des virgules, identique pour les connexions et les requêtes. Comprendre ce format est indispensable pour relire le journal ou l'ingérer dans un outil d'analyse.

```
[timestamp],[serverhost],[username],[host],[connectionid],[queryid],[operation],[database],[object],[retcode]
```

| Champ | Description |
|-------|-------------|
| `timestamp` | Date et heure de l'événement, au format `YYYYMMDD HH:MM:SS` |
| `serverhost` | Nom d'hôte du serveur MariaDB ayant produit l'enregistrement |
| `username` | Utilisateur authentifié |
| `host` | Hôte client à l'origine de la connexion |
| `connectionid` | Identifiant de la connexion (thread) |
| `queryid` | Identifiant incrémental de la requête au sein de la connexion |
| `operation` | Type d'opération : `CONNECT`, `DISCONNECT`, `FAILED_CONNECT`, `QUERY`, ou — pour les événements `TABLE` — `READ`, `WRITE`, `CREATE`, `ALTER`, `DROP`, `RENAME` |
| `database` | Base de données active par défaut au moment de l'événement |
| `object` | Le texte de la requête (événements `QUERY`) ou le nom de l'objet concerné (événements `TABLE`) |
| `retcode` | Code retour : `0` en cas de succès, sinon le code d'erreur MariaDB |

## Auditer les connexions

L'activation de la catégorie `CONNECT` trace le cycle de vie complet de chaque session. Trois opérations principales apparaissent dans le journal : `CONNECT` à l'ouverture d'une session, `DISCONNECT` à sa fermeture, et `FAILED_CONNECT` lorsqu'une authentification échoue.

```
20260605 14:23:01,dbprod01,appuser,192.168.1.50,4521,0,CONNECT,appdb,,0
20260605 14:25:17,dbprod01,appuser,192.168.1.50,4521,0,DISCONNECT,,,0
20260605 14:26:03,dbprod01,root,10.0.0.99,0,0,FAILED_CONNECT,,,1045
```

Sur la dernière ligne, le code retour `1045` correspond à l'erreur *Access denied* : la tentative de connexion sous le compte `root` depuis `10.0.0.99` a été rejetée. C'est précisément ce type d'enregistrement qui rend l'audit des connexions central pour la sécurité : une rafale de `FAILED_CONNECT` sur un même compte ou depuis une même adresse signale une attaque par force brute, tandis que le couple `username` + `host` permet de retracer précisément l'origine de chaque session.

Un point important pour la traçabilité : les événements de connexion sont enregistrés **pour tous les utilisateurs**, indépendamment des filtres présentés plus bas. Un compte exclu de l'audit des requêtes verra malgré tout ses connexions et déconnexions journalisées, ce qui garantit qu'aucune session ne passe totalement sous le radar.

## Auditer les requêtes

La catégorie `QUERY` (et ses sous-catégories) enregistre le texte intégral des instructions exécutées dans le champ `object`, accompagné de leur code retour.

```
20260605 14:23:05,dbprod01,appuser,192.168.1.50,4521,88,QUERY,appdb,'SELECT * FROM commandes WHERE id=42',0
20260605 14:24:10,dbprod01,admin,localhost,4522,91,QUERY,appdb,'CREATE TABLE audit_test (id INT)',0
20260605 14:24:55,dbprod01,appuser,192.168.1.50,4521,93,QUERY,appdb,'SELECT * FROM table_inexistante',1146
```

Le code retour différencie ici les succès (`0`) des échecs : la troisième ligne (`1146`, *Table doesn't exist*) révèle une requête en erreur. Comme les requêtes échouées sont journalisées au même titre que les autres, l'audit des requêtes devient un outil de détection : une succession d'erreurs de syntaxe ou d'accès à des objets inexistants peut trahir une tentative d'injection SQL ou une exploration non autorisée du schéma.

Le plugin masque automatiquement les mots de passe contenus dans les instructions sensibles (`CREATE USER … IDENTIFIED BY`, `SET PASSWORD`, `GRANT … IDENTIFIED BY`, `ALTER USER`). Le secret est remplacé par des astérisques avant écriture dans le journal :

```
20260605 14:30:00,dbprod01,root,localhost,4523,95,QUERY,mysql,'CREATE USER \'bob\'@\'%\' IDENTIFIED BY *****',0
```

Ce masquage évite que des identifiants en clair ne se retrouvent dans des journaux susceptibles d'être consultés par des tiers. Il ne couvre cependant que les mots de passe : des données personnelles présentes dans une clause `WHERE` ou un `INSERT`, par exemple, apparaîtront telles quelles, ce qui a des implications sur la protection du fichier (voir plus bas).

Enfin, la catégorie `TABLE` complète l'audit des requêtes en exposant les objets réellement touchés. Là où `QUERY` enregistre l'instruction, `TABLE` produit un enregistrement par table avec l'opération correspondante (`READ` pour une lecture, `WRITE` pour une modification). C'est utile lorsqu'on souhaite savoir *quelles données* ont été consultées plutôt que *quelle requête* a été soumise, mais cette catégorie multiplie les enregistrements et reste à réserver aux contextes l'exigeant.

## Restreindre la portée : filtrage par utilisateur

Auditer toutes les requêtes de tous les utilisateurs sature rapidement le journal. Deux variables permettent de cibler le périmètre :

`server_audit_incl_users` définit une liste blanche : seuls les utilisateurs mentionnés sont audités pour leurs requêtes. `server_audit_excl_users` définit à l'inverse une liste noire, typiquement employée pour écarter les comptes de monitoring ou de sauvegarde, très verbeux mais sans intérêt pour la sécurité.

Lorsqu'un même utilisateur figure dans les deux listes, `server_audit_incl_users` l'emporte : l'utilisateur reste audité. Et comme indiqué précédemment, ces filtres n'affectent pas les événements de connexion, qui continuent d'être journalisés pour l'ensemble des comptes.

## Appliquer la configuration

Les variables d'événements et de filtrage sont **dynamiques** : elles s'ajustent à chaud, sans redémarrage, ce qui permet d'élargir temporairement le périmètre d'audit lors d'une investigation.

```sql
-- Auditer les connexions ainsi que les changements de schéma et de privilèges
SET GLOBAL server_audit_events    = 'CONNECT,QUERY_DDL,QUERY_DCL';

-- Exclure les comptes techniques du journal des requêtes
SET GLOBAL server_audit_excl_users = 'monitoring,backup';

-- Vérifier la configuration courante
SHOW GLOBAL VARIABLES LIKE 'server_audit%';
```

Pour que la configuration survive à un redémarrage, on la fixe dans le fichier de configuration (`my.cnf` / `my.ini`), dans une section lue par le serveur :

```ini
[mariadb]
server_audit_logging          = ON
server_audit_events           = CONNECT,QUERY_DDL,QUERY_DCL
server_audit_output_type      = file
server_audit_file_path        = /var/log/mysql/server_audit.log
server_audit_file_rotate_size = 1000000
server_audit_file_rotations   = 9
server_audit_excl_users       = monitoring,backup
```

## Volume, performance et protection du journal

L'audit des requêtes est de loin le plus coûteux : sur un serveur traitant des milliers de requêtes par seconde, journaliser l'intégralité du trafic `QUERY` génère des volumes considérables et ajoute une latence à chaque instruction. La première parade est le **ciblage** : privilégier les sous-catégories (`QUERY_DDL`, `QUERY_DCL`, `QUERY_DML_NO_SELECT`) et les filtres utilisateurs plutôt que `QUERY` seul. La réduction de l'impact en écriture, via la **bufferisation** du journal, fait l'objet de la section 10.8.3.

Le fichier d'audit doit par ailleurs être considéré comme une donnée sensible. Même avec le masquage des mots de passe, il contient le texte des requêtes — donc potentiellement des données personnelles ou métier. Ses permissions doivent être strictement limitées (lecture réservée au compte système de MariaDB et aux administrateurs habilités), et sa conservation alignée sur les politiques de rétention applicables.

## Audit et conformité

La traçabilité des connexions et des requêtes répond directement à plusieurs exigences réglementaires et normatives. L'audit des `FAILED_CONNECT` et des `QUERY_DCL` documente le contrôle d'accès attendu par des cadres comme PCI-DSS ou ISO 27001 ; la journalisation des modifications (`QUERY_DDL`, `QUERY_DML_NO_SELECT`) alimente la piste d'audit demandée pour la conformité financière ou le RGPD. Disposer d'un enregistrement horodaté, attribué à un utilisateur et à un hôte précis, constitue également la base de toute analyse *post mortem* après un incident de sécurité.
