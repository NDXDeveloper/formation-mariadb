# 10.5.2 — ed25519 : Authentification moderne

`ed25519` est la réponse de MariaDB au vieillissement de SHA-1 : il fournit une authentification par mot de passe **à sécurité renforcée**, fondée sur la signature numérique sur courbe elliptique **Ed25519** — le même algorithme que celui employé par OpenSSH, conçu par Daniel J. Bernstein. Du point de vue de l'utilisateur, rien ne change : il saisit toujours un mot de passe. C'est en interne que la robustesse cryptographique s'améliore nettement par rapport à `mysql_native_password` (10.5.1). Cette section en détaille l'installation, l'usage, les propriétés de sécurité et le principal point d'attention : la compatibilité des clients.

---

## Pourquoi ed25519 ?

L'authentification de MySQL/MariaDB reposait sur SHA-1 depuis la version 4.1 (le plugin `mysql_native_password`). À mesure que la puissance de calcul a augmenté, SHA-1 a cessé d'être considéré comme sûr. Le plugin `ed25519` a été créé pour offrir une alternative moderne : il s'appuie sur une **signature sur courbe elliptique** pour stocker les mots de passe et authentifier les comptes, ce qui le rend bien plus résistant tout en conservant une expérience d'authentification par mot de passe classique.

---

## Installation

La bibliothèque du plugin est livrée avec MariaDB (`auth_ed25519.so` sous Unix, `auth_ed25519.dll` sous Windows), mais **le plugin n'est pas installé par défaut**. Il faut donc l'activer, soit dynamiquement, soit au démarrage via `--plugin-load` / `--plugin-load-add` :

```sql
INSTALL SONAME 'auth_ed25519';     -- active le plugin
UNINSTALL SONAME 'auth_ed25519';   -- le désactive
```

Une fois chargé, le plugin apparaît à l'état `ACTIVE` dans `SHOW PLUGINS`.

---

## Créer et modifier un compte avec ed25519

La méthode moderne et la plus simple consiste à laisser le serveur calculer l'empreinte, avec `USING PASSWORD('...')` :

```sql
CREATE USER 'carol'@'%' IDENTIFIED VIA ed25519 USING PASSWORD('mot_de_passe');
ALTER USER  'bob'@'%'   IDENTIFIED VIA ed25519 USING PASSWORD('mot_de_passe');
```

On peut aussi fournir une **empreinte pré-calculée** avec `USING '<empreinte>'`. L'empreinte ed25519 se présente comme une chaîne en base64 (par exemple `v6UHXdhA+jsUh6UjX2jAEF559fn//CcBjBA0ylFfxWs`), d'un format différent de celui de `mysql_native_password`. Pour la calculer hors connexion, on installe la fonction utilisateur (UDF) fournie avec le plugin :

```sql
CREATE FUNCTION ed25519_password RETURNS STRING SONAME 'auth_ed25519.so';
SELECT ed25519_password('mot_de_passe');
-- -> v6UHXdhA+jsUh6UjX2jAEF559fn//CcBjBA0ylFfxWs

CREATE USER 'dan'@'%' IDENTIFIED VIA ed25519 USING 'v6UHXdhA+jsUh6UjX2jAEF559fn//CcBjBA0ylFfxWs';
```

Un compte ed25519 **doit avoir un mot de passe** pour pouvoir se connecter : il est possible d'en créer un sans, mais il sera alors incapable d'ouvrir une session.

Concernant le changement de mot de passe, le comportement a évolué selon les versions : **depuis MariaDB 10.4** (et donc en 12.3), `SET PASSWORD` suivi de `PASSWORD()` fonctionne avec un compte ed25519 ; **en 10.3 et avant**, ce n'était pas le cas et l'on devait obligatoirement passer par l'UDF `ed25519_password()`.

```sql
-- Valable en MariaDB 10.4+ (donc 12.3)
SET PASSWORD FOR 'carol'@'%' = PASSWORD('nouveau_mot_de_passe');
```

---

## Sécurité : ce que ed25519 améliore

L'apport principal d'ed25519 tient au mode d'authentification : le client **signe** un défi à l'aide d'une clé dérivée de son mot de passe, et le serveur vérifie cette signature à l'aide de la valeur stockée. Contrairement à l'empreinte de `mysql_native_password` — qui, combinée à une écoute du réseau, suffit à se faire passer pour l'utilisateur —, la valeur ed25519 stockée n'est pas directement « rejouable » de la même manière.

Cela ne dispense pas pour autant d'un **mot de passe robuste** : si un attaquant obtient le contenu de `mysql.user`, il peut toujours tenter de retrouver un mot de passe **faible** par force brute hors ligne ou tables arc-en-ciel. La règle reste donc : un mot de passe complexe, difficile à deviner. Comme pour toute méthode, le chiffrement **TLS** des connexions (10.7) demeure recommandé pour protéger les échanges.

---

## Compatibilité client : le point d'attention

C'est la principale contrepartie d'ed25519. L'authentification exige un **plugin client compatible** : le plugin `client_ed25519` hache et signe le mot de passe côté client avant de l'envoyer. MariaDB Connector/C (et donc les connecteurs qui s'appuient dessus) prend en charge ed25519 ; mais **beaucoup de pilotes anciens** utilisent par défaut `mysql_native_password` et **échouent** face à un compte ed25519.

Quelques conséquences pratiques :

- vérifier que le **connecteur applicatif** supporte ed25519 (par exemple, côté Python, la bibliothèque `mariadb` plutôt qu'un ancien `mysql-connector-python`) ;
- certains intermédiaires (proxies, poolers) ont historiquement eu des difficultés à relayer l'authentification ed25519 — à valider avant déploiement ;
- il peut être nécessaire d'indiquer au client où trouver le plugin client, via l'option `--plugin-dir`.

Pour une **transition en douceur**, on peut déclarer deux méthodes acceptées sur un même compte (voir 10.5), le temps que tous les clients migrent :

```sql
CREATE USER 'svc'@'%'
  IDENTIFIED VIA ed25519 USING PASSWORD('mot_de_passe')
  OR mysql_native_password USING PASSWORD('mot_de_passe');
```

---

## Inspecter

Le plugin associé à un compte et l'état du plugin serveur se vérifient ainsi :

```sql
SELECT User, Host, plugin FROM mysql.user WHERE plugin = 'ed25519';
SHOW PLUGINS;   -- le plugin ed25519 doit être ACTIVE
```

---

## Quand l'utiliser

`ed25519` est le choix recommandé pour de **nouveaux comptes** lorsqu'on souhaite une sécurité supérieure à celle de `mysql_native_password` **et** que les clients concernés le prennent en charge. Il convient particulièrement aux comptes applicatifs et d'administration internes, où l'on maîtrise les connecteurs. À l'inverse, lorsque la compatibilité avec d'anciens clients prime, `mysql_native_password` reste plus universel (10.5.1). Pour les déploiements visant le standard d'authentification le plus moderne, le plugin **PARSEC** (10.6) est appelé à devenir le futur défaut de MariaDB.

---

## À retenir

`ed25519` fournit une authentification par mot de passe **à sécurité renforcée**, fondée sur la signature sur courbe elliptique Ed25519 (la même qu'OpenSSH), en remplacement du SHA-1 vieillissant. Le plugin est livré avec MariaDB mais **non installé par défaut** (`INSTALL SONAME 'auth_ed25519'`). On crée un compte avec `IDENTIFIED VIA ed25519 USING PASSWORD('...')`, ou avec une empreinte pré-calculée via l'UDF `ed25519_password()`. Son atout est un mode d'authentification par **signature**, dont la valeur stockée n'est pas directement rejouable ; sa contrepartie, la nécessité d'un **client compatible** (`client_ed25519`), beaucoup de pilotes anciens ne sachant dialoguer qu'en `mysql_native_password`. À privilégier pour les nouveaux comptes maîtrisés, avec un mot de passe robuste et des connexions TLS.

---

> 🔔 **Note de version.** `ed25519` est disponible depuis **MariaDB 10.1.22 / 10.2.5** et son comportement est stable dans **MariaDB 12.3 LTS** ; il reste **non installé par défaut**. Depuis la 10.4 (donc en 12.3), `SET PASSWORD`/`PASSWORD()` fonctionnent avec ce plugin.
