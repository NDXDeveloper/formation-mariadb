# 10.5.4 — GSSAPI/Kerberos

Le plugin **GSSAPI** appartient, comme PAM (10.5.3), à la famille de l'**identité externe** — mais avec une différence décisive : il authentifie **sans transmettre de mot de passe**. Il s'appuie sur GSSAPI (sur Unix, c'est généralement **Kerberos**) ou sur son équivalent Windows **SSPI**, pour réaliser une authentification unique (*single sign-on*) : un utilisateur déjà authentifié auprès de son domaine Kerberos ou Active Directory se connecte à MariaDB grâce à son **ticket** existant, sans ressaisir d'identifiant. C'est le plugin de prédilection pour intégrer MariaDB à un annuaire d'entreprise. Cette section en présente le principe, la préparation serveur et la création de comptes.

---

## GSSAPI, Kerberos, SSPI : de quoi parle-t-on ?

GSSAPI (*Generic Security Services API*) est une interface standardisée (RFC 2743/2744) ; le client et le serveur négocient selon un protocole standard (RFC 7546). Selon la plateforme :

- sur **Unix**, le service GSSAPI dominant est **Kerberos** : le plugin réalise donc une authentification Kerberos ;
- sur **Windows**, l'API équivalente est **SSPI**, qui prend en charge **Kerberos *et* NTLM**, que l'environnement utilise un domaine ou non.

En pratique, le plugin `gssapi` est **le plus souvent employé pour s'authentifier auprès de Microsoft Active Directory**, en environnement mixte ou homogène.

---

## La différence clé avec PAM : aucun mot de passe transmis

PAM (10.5.3) délègue lui aussi à des mécanismes externes, y compris Kerberos (via `pam_krb5`), mais il **transmet le mot de passe au serveur en clair** — d'où la nécessité absolue de TLS. GSSAPI fonctionne autrement : l'authentification repose sur des **tickets** (jetons GSSAPI/Kerberos), et **le mot de passe n'est jamais envoyé** au serveur. C'est à la fois plus sûr et plus pratique : l'utilisateur n'a pas de mot de passe propre à MariaDB, et l'expérience est celle d'un véritable SSO.

Le chiffrement TLS (10.7) reste recommandé pour la **confidentialité des données** échangées, mais le risque d'exposition d'un mot de passe — central avec PAM — ne se pose plus ici.

---

## Installation

Le plugin s'active comme les autres plugins chargeables :

```sql
INSTALL SONAME 'auth_gssapi';
```

---

## Préparation côté serveur (Unix)

Sur Unix, le serveur MariaDB a besoin d'un **fichier keytab** en lecture, contenant le **nom de principal de service** (SPN) qui identifie le serveur auprès du KDC (centre de distribution de clés Kerberos). Le principal de service par défaut est de la forme `mariadb/hote.domaine.com@REALM`.

La génération du keytab dépend du KDC :

- avec un **KDC MIT Kerberos**, on crée le principal et on l'exporte avec l'utilitaire `kadmin` ;
- avec un **KDC Active Directory**, on génère le keytab côté Windows avec `ktpass.exe`, par exemple :

```text
ktpass.exe /princ mariadb/db1.example.com@EXAMPLE.COM /mapuser svc_mariadb
           /pass ******** /out mariadb.keytab /crypto all
           /ptype KRB5_NT_PRINCIPAL /mapop set
```

Le keytab est ensuite transféré sur le serveur et son chemin renseigné via `gssapi_keytab_path`. Si le principal diffère du défaut, on précise `gssapi_principal_name` :

```ini
[mariadb]
gssapi_keytab_path=/etc/mariadb/mariadb.keytab
gssapi_principal_name=mariadb/db1.example.com@EXAMPLE.COM
```

La mise en place complète (création du principal, jonction au realm) est décrite dans le guide officiel ; on s'y reportera pour le déroulé détaillé.

---

## Préparation côté serveur (Windows)

Sur **Windows**, l'intégration est plus directe : le serveur utilise **SSPI** et n'a pas besoin de keytab. On configure le principal de service via `gssapi_principal_name`, et la variable `gssapi_mech_name` (propre à Windows) permet de choisir le paquet SSPI employé — `Kerberos` ou `Negotiate`.

---

## Créer un compte GSSAPI

On déclare le plugin avec `IDENTIFIED VIA gssapi` (la forme MySQL `IDENTIFIED WITH gssapi AS '...'` est également acceptée). La clause `USING` précise, le cas échéant, le **principal** attendu :

```sql
-- Forme longue : principal complet (avec realm)
CREATE USER 'alice'@'%' IDENTIFIED VIA gssapi USING 'alice@EXAMPLE.COM';

-- Forme courte : le realm n'est pas comparé
CREATE USER 'alice'@'%' IDENTIFIED VIA gssapi;
```

La partie indiquée dans `USING` est **spécifique au mécanisme**. Pour un utilisateur Windows authentifié par NTLM, elle prend la forme `machine\utilisateur` :

```sql
CREATE USER 'alice'@'%' IDENTIFIED VIA gssapi USING 'WORKSTATION\alice';
```

Point important sur la **correspondance** : avec la forme courte, la partie realm est ignorée. Ainsi `alice@EXAMPLE.COM`, `alice@EXAMPLE.CO.UK` et `mymachine\alice` seront tous reconnus comme `alice`. La forme longue, en spécifiant le principal complet, restreint la correspondance à ce realm précis.

---

## Côté client

L'utilisateur se connecte **sans mot de passe**, en s'appuyant sur le ticket Kerberos qu'il détient déjà (obtenu via `kinit` sur Unix, ou via sa session de domaine sur Windows) :

```bash
mariadb -u alice    # utilise le ticket Kerberos existant, aucun mot de passe
```

Le client doit disposer du plugin client GSSAPI/SSPI (au besoin, on lui indique son emplacement avec `--plugin-dir`). Quelques subtilités existent en environnement hétérogène : par exemple, depuis un client Unix vers un serveur Windows en ADO.NET, .NET Core ne gère pas les noms de principal au format UPN (`utilisateur@domaine`) ; on contourne en forçant un SPN « basé sur l'hôte » (`HOST/machine`) côté serveur (`gssapi_principal_name`) ou côté client (paramètre `ServerSPN`).

---

## Quand l'utiliser

GSSAPI/Kerberos est le choix de l'**authentification unique d'entreprise** : intégration native à Active Directory ou à un realm Kerberos, connexion sans mot de passe, gestion centralisée des identités. Comparé à PAM — qui peut lui aussi faire du Kerberos, mais en transmettant le mot de passe en clair —, GSSAPI offre un vrai SSO sans exposition d'identifiant. Ses contreparties sont une mise en place plus exigeante (keytab, SPN, jonction au realm) et la dépendance à une infrastructure Kerberos/AD opérationnelle.

---

## À retenir

Le plugin **`gssapi`** réalise une authentification **sans mot de passe** (SSO) via GSSAPI — **Kerberos** sur Unix, **SSPI** (Kerberos ou NTLM) sur Windows —, principalement pour s'intégrer à **Active Directory**. On l'installe avec `INSTALL SONAME 'auth_gssapi'`. Sur Unix, le serveur a besoin d'un **keytab** (variable `gssapi_keytab_path`) et d'un **principal de service** (`gssapi_principal_name`, défaut `mariadb/hote@REALM`) ; sur Windows, SSPI évite le keytab. On crée un compte avec `IDENTIFIED VIA gssapi [USING 'principal']` : la forme courte ignore le realm, la forme longue le compare. Le client se connecte avec son **ticket** existant, sans mot de passe. Sa différence majeure avec PAM est précisément cette **absence de mot de passe transmis**.

---

> 🔔 **Note de version.** Le plugin `gssapi` et ses variables (`gssapi_keytab_path`, `gssapi_principal_name`, `gssapi_mech_name` sous Windows) sont stables dans **MariaDB 12.3 LTS**. Sur Unix, le serveur a besoin d'un accès en lecture au keytab Kerberos ; sur Windows, l'authentification s'appuie sur SSPI (Kerberos ou NTLM), avec ou sans domaine.
