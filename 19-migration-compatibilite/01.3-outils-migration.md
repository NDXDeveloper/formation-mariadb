# 19.1.3 Outils de migration

> **Chapitre 19 — Migration et Compatibilité** · Section 19.1 · Sous-section 19.1.3  
> Niveau : DBA / Architecte

---

## Quels outils pour quelle stratégie

Comme établi en 19.1.2, une migration depuis **MySQL 8** ne peut pas se faire par copie de fichiers : elle doit être **logique** (export puis import) ou s'appuyer sur une **réplication** compatible. L'outillage de migration s'organise donc autour de ces deux approches, complétées par des **utilitaires de conversion** (pour le JSON et l'authentification, vus précédemment) et des **outils de validation**.

Avant de choisir un outil, un réflexe s'impose : consulter le **guide officiel de migration de MariaDB**, *« MySQL to MariaDB Migration: The Master Guide »*. C'est la source de référence du sujet. Elle couvre les migrations depuis **MySQL 5.7, 8.0 et 8.4 LTS vers MariaDB 11.8 / 12.3**, fournit une **matrice de compatibilité** et décrit des procédures de *dump*/restauration éprouvées. Le guide recommande, depuis MySQL 8.0/8.4, de cibler **MariaDB 11.8 LTS ou 12.3 LTS**, qui offrent la parité de fonctionnalités la plus adaptée aux applications de l'ère 8.x.

---

## 1. La migration logique : l'approche principale

C'est la voie privilégiée pour une migration depuis MySQL 8, et la seule possible sans réplication.

### `mysqldump` / `mariadb-dump`

L'outil historique d'export logique produit un fichier SQL contenant à la fois les définitions d'objets et les données. MariaDB a renommé `mysqldump` en `mariadb-dump` (l'ancien nom restant disponible comme lien symbolique).

Un point **contre-intuitif mais important**, recommandé par le guide officiel : pour exporter **depuis MySQL**, il faut utiliser le **`mysqldump` de MySQL**, et non le `mariadb-dump` de MariaDB — ce dernier n'ayant pas été testé en profondeur contre MySQL. On exporte donc avec les outils du moteur source, et on importe avec ceux de la cible.

Quelques options essentielles lors d'un export en vue d'une migration :

- `--single-transaction` : produit un instantané cohérent sans verrouiller les tables (sur InnoDB) ;
- `--routines` : inclut les procédures et fonctions stockées ;
- `--triggers` : inclut les déclencheurs ;
- `--events` : inclut les événements planifiés ;
- `--hex-blob` : sécurise l'export des données binaires ;
- `--set-gtid-purged=OFF` : évite d'embarquer des informations de GTID, incompatibles entre les deux moteurs (voir 19.1.2).

Un point de vigilance majeur : il **ne faut pas utiliser `--all-databases`**. Cela inclurait les **bases système de MySQL** (`mysql`, etc.), qui diffèrent de celles de MariaDB ; leur import échouerait. On sélectionne donc explicitement les bases applicatives avec `--databases`.

```bash
# Export sélectif depuis MySQL (à lancer avec les outils MySQL)
mysqldump --user=root --password \
  --databases app_db1 app_db2 \
  --single-transaction --routines --events --triggers --hex-blob \
  > migration_dump.sql
```

Côté MariaDB, on crée au préalable les bases cibles **avec le bon jeu de caractères et la bonne collation** (la collation par défaut de MySQL 8 n'existant pas dans MariaDB — voir 19.1.2), puis on importe avec le client `mariadb` :

```bash
# Création de la base cible avec une collation MariaDB valide
mariadb -u root -p -e \
  "CREATE DATABASE app_db1 CHARACTER SET utf8mb4 COLLATE utf8mb4_uca1400_ai_ci;"

# Import du dump
mariadb -u root -p app_db1 < migration_dump.sql
```

### `mydumper` / `myloader`

Pour les bases volumineuses, le couple **`mydumper`/`myloader`** réalise un export et un import **parallélisés** (multi-threads), bien plus rapides qu'un `mysqldump` séquentiel. Il prend en charge la compression et le découpage par tables, et se combine aisément avec des outils de transfert (`ssh`, `rsync`, `ncat`) pour une migration entre hôtes distincts.

`mydumper` n'est généralement **pas installé par défaut** et doit être ajouté séparément. Les options usuelles incluent le nombre de threads et la compression :

```bash
mydumper --database app_db1 --outputdir /backup/dump --compress --threads 4
```

Cet outil est également présenté dans le contexte des sauvegardes au chapitre 12 (voir §12.2.3).

### Chargement en masse

Pour des tables très volumineuses ou des cas ciblés, `SELECT ... INTO OUTFILE` côté source et `LOAD DATA INFILE` côté cible (ou les utilitaires `mariadb-import`/`mysqlimport`) offrent un chargement de données brutes performant, complémentaire des outils de *dump*.

## 2. Les utilitaires de conversion

Certains écarts vus dans les sous-sections précédentes nécessitent une conversion explicite :

- **Plugin `mysql_json`** : il convertit automatiquement le format **binaire** des colonnes JSON de MySQL vers le format **texte** de MariaDB lors de l'import. Il est indispensable lorsque le schéma comporte des colonnes JSON (voir 19.1.2). À défaut, on convertit les colonnes en `TEXT` côté MySQL avant l'export.
- **Plugin `caching_sha2_password`** : il permet de **réutiliser les empreintes de mot de passe** de MySQL 8 côté MariaDB, évitant de réinitialiser les comptes (voir 19.1.1).
- **Recréation des utilisateurs et privilèges** : les *dumps* standards **capturent mal les permissions complexes** issues de MySQL 8.0+. Plutôt que de copier les tables système (qui diffèrent entre les deux moteurs), il est recommandé de **recréer explicitement les comptes et les droits** avec `CREATE USER` et `GRANT` sur la cible MariaDB.

## 3. La migration par réplication (faible interruption)

Lorsque l'interruption de service doit être minimale, on peut configurer **MariaDB comme réplica de MySQL** : MariaDB rattrape les données existantes, se maintient à jour, puis l'on bascule l'application au moment voulu.

Cette approche est **possible mais contrainte**, pour les raisons détaillées en 19.1.2 : la GTID de MariaDB étant incompatible avec celle de MySQL, on s'appuie sur les coordonnées de binlog plutôt que sur la GTID ; la réplication par lignes des colonnes JSON ne fonctionne pas en l'état, etc. Elle demande donc une validation soigneuse. Les stratégies sans interruption sont développées en 19.8.

---

## 4. Le cas du « binary swap » : sources anciennes uniquement

Pour les versions **anciennes de MySQL (5.7 et antérieures)**, une migration **en place** (« *binary swap* ») était envisageable : arrêter MySQL, installer MariaDB sur le même répertoire de données, puis exécuter `mariadb-upgrade`. Cela tenait à la compatibilité des formats sur disque.

Cette approche **n'est plus possible depuis MySQL 8.0** en raison du dictionnaire de données transactionnel : seule la migration **logique** (ou par réplication) s'applique. Par ailleurs, l'outil `mariadb-upgrade` lui-même concerne avant tout les **montées de version internes à MariaDB** (voir §19.4.1), et non un passage depuis MySQL 8.

Quelle que soit l'approche, le guide officiel recommande de conserver une **sauvegarde physique du répertoire de données source** (serveur arrêté) comme **filet de sécurité de retour arrière** avant toute bascule. Le mécanisme de rollback est traité plus en détail en 19.7.

## 5. Outils de validation

La migration ne s'achève pas à l'import : il faut **vérifier** son résultat.

- **Comptages et sommes de contrôle** : comparer le nombre de lignes et des sommes de contrôle entre source et cible pour garantir l'intégrité des données.
- **Percona Toolkit** : `pt-table-checksum` et `pt-table-sync` permettent de vérifier (et de corriger) la cohérence des données entre deux instances. Attention aux prérequis de connexion liés à `caching_sha2_password` (versions récentes des connecteurs nécessaires).
- **Tests fonctionnels ciblés** : valider en priorité les colonnes **JSON**, les **plugins d'authentification** et les **routines stockées** avant de basculer le trafic de production.

Ces vérifications s'inscrivent dans la démarche plus large de **tests de compatibilité**, approfondie en 19.6.

---

## 6. Outils managés et tiers

Au-delà des outils natifs, d'autres options existent selon le contexte :

- les **fournisseurs cloud** proposent des services de migration et des bases MariaDB managées, alimentées le plus souvent par les mêmes flux de *dump*/restauration logiques ;
- divers **outils tiers** d'ETL et de migration peuvent orchestrer l'extraction, la transformation et le chargement.

Ces solutions ne dispensent pas des vérifications de compatibilité décrites précédemment : elles s'appuient sur les mêmes mécanismes sous-jacents et héritent des mêmes points d'attention (JSON, collations, authentification).

---

## En résumé

Pour une migration depuis MySQL 8 vers MariaDB 12.3, la base de travail recommandée est la suivante :

1. partir du **guide officiel de migration** de MariaDB et de sa matrice de compatibilité, pour les **versions exactes** source et cible ;
2. réaliser un **export logique** avec le `mysqldump` de MySQL (ou `mydumper` pour les gros volumes), en sélectionnant les bases applicatives (`--databases`, jamais `--all-databases`) ;
3. **transposer les collations** et **créer les bases cibles** avec un jeu de caractères valide côté MariaDB ;
4. **convertir le JSON** (plugin `mysql_json`) et **recréer les comptes** explicitement (`CREATE USER`/`GRANT`) ;
5. **valider** par comptages, sommes de contrôle et tests fonctionnels avant la bascule.

La réplication offre une variante à faible interruption (voir 19.8), tandis que les tests de compatibilité (19.6) et les procédures de retour arrière (19.7) complètent le dispositif.
