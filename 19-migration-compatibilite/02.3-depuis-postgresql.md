# 19.2.3 Depuis PostgreSQL

> **Chapitre 19 — Migration et Compatibilité** · Section 19.2 · Sous-section 19.2.3  
> Niveau : DBA / Architecte

---

## Un cas intermédiaire, avec des leviers spécifiques

La migration depuis PostgreSQL se situe entre les deux cas précédents : moins outillée qu'Oracle, mais disposant de leviers que SQL Server n'offre pas. La documentation officielle de MariaDB (« Migrating to MariaDB from PostgreSQL ») met en avant deux mécanismes particuliers : le **moteur CONNECT** pour le transfert des données, et — de façon plus inattendue — le **mode Oracle** pour la conversion du code procédural, en raison de la proximité du PL/pgSQL avec le PL/SQL.

---

## Modes SQL : `POSTGRESQL` et, surtout, `ORACLE`

MariaDB propose un mode `POSTGRESQL`, mais il s'agit, comme le mode `MSSQL`, d'un **mode de combinaison limité**. Il active les drapeaux `PIPES_AS_CONCAT`, `ANSI_QUOTES`, `IGNORE_SPACE`, `NO_KEY_OPTIONS`, `NO_TABLE_OPTIONS` et `NO_FIELD_OPTIONS`. Son intérêt est qu'il couvre les **deux conventions syntaxiques clés** de PostgreSQL : la concaténation par `||` (`PIPES_AS_CONCAT`) et la délimitation des identifiants par guillemets doubles `"` (`ANSI_QUOTES`). En revanche, il **n'émule pas le PL/pgSQL**.

C'est ici qu'intervient une recommandation officielle utile : le **PL/pgSQL ressemble davantage au PL/SQL d'Oracle** qu'au SQL/PSM de MariaDB. Pour la conversion des fonctions et procédures, il peut donc être avantageux de tenter la migration avec **`SQL_MODE=ORACLE`** activé. Le choix du mode dépend ainsi du contexte : le mode `POSTGRESQL` pour la surface syntaxique, le mode `ORACLE` comme appui pour le code procédural.

---

## Le moteur CONNECT pour le transfert de données

Pour le transfert des données, MariaDB recommande le **moteur de stockage CONNECT**. PostgreSQL fournit un pilote **ODBC** libre, `psqlODBC` : une fois ce pilote installé sur le serveur MariaDB et le cadre ODBC du système (par exemple `unixODBC`) configuré, MariaDB peut se connecter au serveur PostgreSQL distant. On crée alors des tables CONNECT qui référencent les tables PostgreSQL, puis on rapatrie les données par `INSERT … SELECT` vers des tables natives InnoDB. Une connexion **JDBC** ou des *foreign data wrappers* constituent des alternatives.

```sql
-- Table externe pointant vers PostgreSQL via JDBC
CREATE TABLE pg_clients (id INT, nom VARCHAR(50))
  ENGINE = CONNECT table_type = JDBC tabname = 'clients'
  CONNECTION = 'jdbc:postgresql://serveur_pg/base1';

-- Table cible native, alimentée depuis la source
CREATE TABLE clients (id INT, nom VARCHAR(50)) ENGINE = InnoDB;
INSERT INTO clients SELECT * FROM pg_clients;
```

## Correspondance des types de données

PostgreSQL possède un système de types riche. Les correspondances usuelles :

| Type PostgreSQL | Type MariaDB | Remarques |
|---|---|---|
| `SERIAL`, `BIGSERIAL` | `INT`/`BIGINT` `AUTO_INCREMENT` | `SERIAL` s'appuie sur une séquence côté PostgreSQL |
| `BOOLEAN` | `BOOLEAN` / `TINYINT(1)` | |
| `TEXT` | `TEXT` / `LONGTEXT` | un `VARCHAR` sans longueur devient `TEXT` |
| `BYTEA` | `BLOB` / `LONGBLOB` | |
| `JSON`, `JSONB` | `JSON` | alias `LONGTEXT` ; le `JSONB` binaire devient du texte ; opérateurs JSONB à réécrire |
| `UUID` | `UUID` (depuis 10.7) | atout pour la migration |
| `INET`, `CIDR` | `INET4` / `INET6` | MariaDB dispose de types IP natifs |
| `ARRAY` (`int[]`, …) | normalisation ou `JSON` | **pas de type tableau natif** |
| `TIMESTAMP`, `TIMESTAMPTZ` | `DATETIME` / `TIMESTAMP` | gérer le fuseau (`TIMESTAMPTZ` stocke en UTC) |
| `NUMERIC`/`DECIMAL`, `MONEY` | `DECIMAL` | |
| `ENUM` (type créé) | `ENUM` (de colonne) | l'`ENUM` PostgreSQL est un type réutilisable ; celui de MariaDB est attaché à une colonne |
| `hstore` | `JSON` / colonnes dynamiques | |
| `tsvector` (plein texte) | index `FULLTEXT` | approche différente |
| types `range` | décomposition en deux colonnes | pas d'équivalent direct |
| types géométriques | types GIS de MariaDB | conversion nécessaire |
| `XML` | type `XML` basique (§2.2.6) / `LONGTEXT` | |

Les types **`UUID`, `INET4`/`INET6`** sont de réels atouts ; à l'inverse, les **tableaux**, les **opérateurs JSONB**, les **types range** et les **types composites** constituent les principaux points de friction.

## Différences de dialecte SQL

Plusieurs écarts de dialecte doivent être anticipés :

- **Identifiants et casse** : PostgreSQL replie les identifiants non délimités en **minuscules** et utilise `"` pour les délimiter ; MariaDB ne replie pas la casse. Le drapeau `ANSI_QUOTES` (inclus dans le mode `POSTGRESQL`) active les guillemets doubles.
- **Concaténation** : `||` nécessite `PIPES_AS_CONCAT` (inclus dans le mode `POSTGRESQL`) ou la fonction `CONCAT()`.
- **Transtypage** : la syntaxe PostgreSQL `expr::type` se réécrit en `CAST(expr AS type)`.
- **Casse des comparaisons** : PostgreSQL compare les chaînes de façon **sensible à la casse** par défaut, alors que les collations par défaut de MariaDB sont **insensibles à la casse**. C'est une différence de comportement importante : il faut choisir les collations avec soin (par exemple `_cs` ou `_bin` pour rétablir la sensibilité). De même, `ILIKE` se transpose en `LIKE` avec une collation insensible à la casse.
- **Séquences** : `nextval('seq')`/`currval('seq')`/`lastval()` deviennent `NEXTVAL(seq)`/`NEXT VALUE FOR seq` et `LASTVAL(seq)`/`PREVIOUS VALUE FOR seq` ; les deux moteurs disposent de `CREATE SEQUENCE`.
- **`RETURNING`** : MariaDB prend en charge `RETURNING` sur `INSERT`, `DELETE` et `REPLACE` ; la compatibilité avec le `RETURNING` de PostgreSQL est donc partielle.
- **`generate_series()`** : sans équivalent direct ; à remplacer par une CTE récursive ou le moteur `SEQUENCE`.
- **`DISTINCT ON`** : à réécrire avec des fonctions de fenêtrage ou un `GROUP BY`.
- **Opérateurs de tableaux et de JSONB** (`@>`, `?`, etc.) : à réécrire à l'aide des fonctions JSON de MariaDB.

Fonctions de fenêtrage et CTE sont, en revanche, largement compatibles entre les deux moteurs.

## Du PL/pgSQL vers le SQL/PSM

Les fonctions, procédures et déclencheurs écrits en **PL/pgSQL** doivent être **réécrits**. Comme indiqué plus haut, la proximité du PL/pgSQL avec le PL/SQL fait du mode `ORACLE` un appui utile. Plusieurs points structurants :

- **Déclencheurs** : PostgreSQL sépare la **fonction de déclenchement** (qui renvoie le type `TRIGGER` et manipule `NEW`/`OLD`) de l'instruction `CREATE TRIGGER` qui l'appelle. MariaDB intègre le corps **directement** dans le déclencheur : ce schéma doit donc être restructuré.
- **Volatilité** : les qualificatifs `VOLATILE`/`STABLE`/`IMMUTABLE` de PostgreSQL se transposent vers les caractéristiques `DETERMINISTIC`/`NOT DETERMINISTIC` et les clauses d'accès aux données de MariaDB.
- **Erreurs et résultats** : `RAISE EXCEPTION` devient `SIGNAL` ; les constructions `RETURN QUERY`, `SETOF` et `RETURNS TABLE` doivent être repensées selon les mécanismes de MariaDB.

---

## Schémas

PostgreSQL organise les objets en `base → schéma → objet`, avec un `search_path` déterminant l'ordre de résolution. MariaDB, où la **base joue le rôle de schéma**, ne possède pas cette hiérarchie intermédiaire. Les schémas PostgreSQL (comme `public`) sont donc généralement **repliés dans des bases MariaDB**, et le `search_path` se traduit en partie par la sélection de base. L'instruction `SET PATH` (voir §19.2.1) facilite la résolution des routines et paquets, mais ne reproduit pas le chemin de recherche des **tables** de PostgreSQL.

## Outils

Pour mener la migration :

- le **moteur CONNECT** (voir §7.10.4) assure le transfert des données via ODBC/JDBC, comme décrit plus haut ;
- des **outils de conversion** — la famille SQLines (PostgreSQL vers MariaDB), des kits comme ceux d'Ispirer, l'**assistant de migration de MySQL Workbench** avec `psqlODBC`, ou des kits commerciaux — automatisent une partie de la conversion du schéma et du code.

**Point d'attention important** : l'outil très répandu **`pgloader` ne convient pas ici**. Il est conçu pour migrer divers systèmes **vers** PostgreSQL, et non l'inverse ; il ne faut pas le confondre avec un outil de migration *depuis* PostgreSQL vers MariaDB. Comme toujours, les outils accélèrent le travail sans le garantir : une **relecture et des tests** restent indispensables, en particulier pour le code procédural.

---

## En résumé

La migration depuis PostgreSQL est un cas intermédiaire, qui combine plusieurs leviers :

- le **moteur CONNECT** (via `psqlODBC` ou JDBC) pour le transfert des données — l'approche recommandée ;
- le mode **`ORACLE`** comme appui pour convertir le **PL/pgSQL**, plus proche du PL/SQL que du SQL/PSM ;
- le mode **`POSTGRESQL`** pour la surface syntaxique (`||`, guillemets `"`).

Les types **`UUID`** et **`INET4`/`INET6`** facilitent la transposition ; les **tableaux**, les **opérateurs JSONB**, les **types range** et la **restructuration des déclencheurs** PL/pgSQL constituent les principaux efforts. La sensibilité à la casse, différente par défaut, mérite une attention particulière. Le tout se valide par des tests rigoureux (voir §19.6 et §19.7), en évitant le piège de `pgloader`, qui migre dans l'autre sens.
