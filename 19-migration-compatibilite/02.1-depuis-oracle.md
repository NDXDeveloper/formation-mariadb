# 19.2.1 Depuis Oracle

> **Chapitre 19 — Migration et Compatibilité** · Section 19.2 · Sous-section 19.2.1  
> Niveau : DBA / Architecte

---

## Le cas le mieux pris en charge

Parmi les migrations hétérogènes, **Oracle est de loin le système le mieux accompagné par MariaDB**. Depuis la version 10.3, le mode `SQL_MODE=ORACLE` fournit une compatibilité étendue avec le dialecte SQL et le PL/SQL d'Oracle, et MariaDB enrichit cette compatibilité à chaque version. La série **12.x** a marqué une étape importante, en ajoutant plusieurs constructions que les applications Oracle utilisent quotidiennement.

Cette sous-section présente ces mécanismes **du point de vue de la migration** : ce qu'ils permettent et ce qu'ils dispensent de réécrire. Pour la syntaxe détaillée, chaque fonctionnalité renvoie à la section technique dédiée de la formation.

---

## Le socle : `SQL_MODE=ORACLE`

L'activation du mode Oracle est le point de départ. Optionnel, il s'active pour la session, globalement ou en configuration :

```sql
SET SESSION sql_mode = 'ORACLE';
```

Une fois actif, MariaDB reconnaît une large part de la syntaxe Oracle : identifiants entre guillemets doubles, types `VARCHAR2` et `NUMBER`, séquences à syntaxe Oracle, et constructions procédurales de type PL/SQL. La quantité de code à réécrire manuellement s'en trouve fortement réduite.

Deux limites doivent rester présentes à l'esprit : le mode Oracle **ne couvre pas l'intégralité** des fonctionnalités d'Oracle, et certaines constructions devront tout de même être converties vers le SQL natif de MariaDB. Les fonctionnalités décrites ci-dessous étendent ce socle.

---

## Les fonctions de conversion Oracle

MariaDB a progressivement implémenté les fonctions de conversion emblématiques d'Oracle (voir §3.7.1 pour le détail). Leurs versions d'apparition diffèrent :

- **`TO_CHAR`** (depuis MariaDB 10.6) convertit une valeur de date/heure/horodatage en chaîne, selon un masque de format. Elle accepte le modificateur **`FM`**, qui désactive le remplissage (*padding*) des composants ; un nombre impair de `FM` désactive le remplissage, un nombre pair le réactive. **Point d'attention** : contrairement à Oracle, `TO_CHAR` de MariaDB ne convertit **pas** les nombres en chaînes — un tel usage produit une erreur.
- **`TO_DATE`** (depuis MariaDB 12.3) convertit une chaîne en valeur de type datetime selon un format. Elle prend en charge des constructions Oracle comme `DEFAULT … ON CONVERSION ERROR` et le paramètre `NLS_FORMAT_STRING` (par exemple `NLS_DATE_LANGUAGE`).
- **`TO_NUMBER`** (depuis MariaDB 12.2) convertit une chaîne en valeur numérique selon un format ; elle renvoie le type `DOUBLE`. Certaines options de format restent en cours d'implémentation.
- **`TRUNC`** (depuis MariaDB 12.2, **en mode Oracle uniquement**) tronque une valeur de date à une unité donnée (jour, mois, année selon le format passé). À ne pas confondre avec la troncature numérique standard.

```sql
-- En mode Oracle
SELECT TO_DATE('2026-06-08', 'YYYY-MM-DD');
SELECT TO_NUMBER('1210.73', '9999.99');   -- masque simple (formats avec séparateurs G/D pas encore gérés)
SELECT TRUNC(NOW(), 'MM');                -- premier jour du mois
SELECT TO_CHAR(NOW(), 'FMDD/MM/YYYY');
```

## La syntaxe de jointure externe Oracle `( + )`

Oracle propose une **syntaxe historique de jointure externe** plaçant l'opérateur `(+)` dans la clause `WHERE`. De nombreuses applications anciennes l'emploient encore. **Depuis MariaDB 12.1**, et lorsque le mode Oracle est actif, cette syntaxe est reconnue (voir §3.3.5).

La règle de lecture est la suivante : la table dont une colonne est **marquée** par `(+)` est la partie « interne » (le côté **optionnel**, dont les colonnes deviennent `NULL` en l'absence de correspondance) ; la table **non marquée** est préservée. La requête suivante conserve donc tous les clients, même sans commande :

```sql
-- Mode Oracle : syntaxe (+)
SELECT c.nom, cmd.id_commande
FROM clients c, commandes cmd
WHERE c.id = cmd.id_client(+);

-- Équivalent en jointure ANSI
SELECT c.nom, cmd.id_commande
FROM clients c
LEFT JOIN commandes cmd ON cmd.id_client = c.id;
```

Comme dans Oracle, des **restrictions** s'appliquent : l'opérateur ne peut être utilisé dans une expression `OR`, `IN` ou de type `ROW`, ne permet pas de joindre une table à une sous-requête, et s'applique à une colonne et non à une expression arbitraire. En cas d'usage interdit, MariaDB renvoie une erreur explicite.

## Les tableaux associatifs (index-by tables)

Le **tableau associatif** est l'une des structures les plus utilisées du PL/SQL : une collection en mémoire indexée par une clé entière ou textuelle, idéale pour les recherches temporaires, le traitement par lots ou la génération de rapports. **Depuis MariaDB 12.1** (en mode Oracle), il est possible de déclarer de telles structures dans les routines stockées et les blocs anonymes, via `DECLARE TYPE … IS TABLE OF … INDEX BY …` (voir §8.1.4).

```sql
-- Mode Oracle
DECLARE TYPE salaires_t IS TABLE OF NUMBER INDEX BY VARCHAR2(20);
  salaires salaires_t;
BEGIN
  salaires('Dupont') := 62000;
  salaires('Martin') := 78000;
END;
```

La clause `INDEX BY` accepte des types entiers et chaînes, et le type d'élément peut être scalaire ou un type `RECORD`. Les méthodes usuelles de collection (comptage, premier/dernier élément, etc.) sont prises en charge. Quelques **différences avec Oracle** subsistent : dans les constructeurs, les clés doivent être des littéraux (Oracle accepte des expressions) ; le contrôle de collation passe par la clause standard `COLLATE` plutôt que par `NLS_SORT`/`NLS_COMP` ; enfin, les tableaux associatifs **imbriqués** (tableaux de tableaux) ne sont pas pris en charge.

## `SYS_REFCURSOR` et les curseurs de référence

Les **curseurs de référence** (*REF CURSOR*) permettent de renvoyer des jeux de résultats depuis une procédure et de transmettre des requêtes entre routines. Oracle fournit pour cela le type prédéfini **`SYS_REFCURSOR`**, un curseur de référence « faible » (sans type de retour fixe) qui évite d'avoir à déclarer un type de curseur.

MariaDB a ajouté, **dès la 12.0**, la prise en charge de `TYPE … IS REF CURSOR` et du type prédéfini `SYS_REFCURSOR` (voir §8.5.2). On peut ainsi porter des procédures Oracle qui retournent un curseur via un paramètre `OUT` :

```sql
-- Mode Oracle
CREATE PROCEDURE get_employes(p_curseur OUT SYS_REFCURSOR)
AS
BEGIN
  OPEN p_curseur FOR SELECT id, nom FROM employes;
END;
```

Cette fonctionnalité s'accompagne de la variable **`max_open_cursors`**, qui plafonne le nombre de curseurs ouverts — un garde-fou analogue au paramètre `open_cursors` d'Oracle, utile lorsque l'usage dynamique des curseurs se multiplie. Elle se combine par ailleurs avec les **curseurs sur instructions préparées** (`prepared statements`), traités en §8.5.1, qui rapprochent encore le comportement de l'`OPEN … FOR` dynamique d'Oracle.

## `SET PATH` : résolution des routines et des paquets

En Oracle, le code s'appuie largement sur des **paquets** (`packages`) et sur une résolution des noms non qualifiés dépendante du schéma courant. Pour faciliter ce portage, MariaDB a implémenté l'instruction standard **`SET PATH`** (depuis **MariaDB 12.3**), accompagnée d'une variable `@@path`. Particularité utile : cette instruction est **disponible quel que soit le `sql_mode`** (pas uniquement en mode Oracle).

`SET PATH` définit la **liste de recherche** (schémas et paquets) utilisée pour résoudre les appels de routines dont le nom n'est pas qualifié. En MariaDB, ce chemin sert également à localiser les **paquets** lors des appels de routines de paquet :

```sql
SET PATH 'test, sys';      -- ajoute « test » et « sys » au chemin de résolution
SELECT @@path;             -- → `test`,`sys`
-- une fois un paquet SYS.* disponible, un appel non qualifié comme
-- UTL_ENCODE.BASE64_DECODE('…') se résoudrait via SYS.UTL_ENCODE

SET PATH DEFAULT;          -- retour au schéma courant (CURRENT_SCHEMA)
```

> ⚠️ Syntaxe : l'instruction standard s'écrit **`SET PATH 'schéma1, schéma2'`** (sans `=`), ou de façon équivalente `SET @@path = 'schéma1, schéma2'` ; `SET PATH DEFAULT` rétablit le schéma courant.

Cette capacité accompagne l'effort de MariaDB pour fournir des **paquets système compatibles Oracle** (de la famille `SYS.*`), dont l'implémentation se poursuit. Elle simplifie le portage de code organisé en paquets, sans réécriture systématique des appels.

## Ce qu'il reste à convertir

Ces fonctionnalités réduisent considérablement l'effort, mais ne suppriment pas tout le travail de portage. Restent typiquement à traiter :

- les **constructions PL/SQL** non couvertes par le mode Oracle, à réécrire en SQL/PSM natif ;
- les **paquets système** Oracle dont l'équivalent MariaDB n'est pas encore disponible (certains `SYS.*`/`DBMS_*` sont en cours d'implémentation) ;
- les **cas limites de types et de fonctions**, ainsi que les comportements implicites propres à Oracle ;
- le **transfert des données** lui-même, via le moteur CONNECT ou un outil d'ETL (voir §19.2).

Les **outils de conversion** (par exemple la famille SQLines) automatisent une partie de la conversion du schéma et du code, y compris vers le mode Oracle, mais une **relecture et des tests** restent indispensables — aucun outil ni mode ne couvre 100 % des fonctionnalités d'Oracle.

---

## En résumé

La migration depuis Oracle bénéficie d'un socle solide — `SQL_MODE=ORACLE` depuis la 10.3 — significativement renforcé par la série 12.x. Les apports clés, tous disponibles dans **MariaDB 12.3 LTS**, sont :

- les **fonctions de conversion** `TO_DATE` (12.3), `TO_NUMBER` (12.2) et `TRUNC` (12.2), aux côtés de `TO_CHAR` (10.6) ;
- la **syntaxe de jointure externe `( + )`** (12.1, mode Oracle) ;
- les **tableaux associatifs** `DECLARE TYPE … TABLE OF … INDEX BY` (12.1, mode Oracle) ;
- `SYS_REFCURSOR` et les **curseurs de référence** (12.0), avec `max_open_cursors` ;
- l'instruction standard **`SET PATH`** (12.3) pour la résolution des routines et paquets.

Ensemble, ces fonctionnalités permettent de porter une part importante du code Oracle avec peu, voire pas, de réécriture. Le reste — constructions non couvertes, paquets système, transfert de données — relève de la conversion manuelle assistée par outils et d'une validation rigoureuse (voir §19.6 et §19.7).
