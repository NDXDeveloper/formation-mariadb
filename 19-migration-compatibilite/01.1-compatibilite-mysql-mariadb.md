# 19.1.1 Compatibilité MySQL/MariaDB

> **Chapitre 19 — Migration et Compatibilité** · Section 19.1 · Sous-section 19.1.1  
> Niveau : DBA / Architecte

---

## Une compatibilité large, mais à vérifier sur quelques points sensibles

MariaDB a été conçue dès l'origine pour entretenir une **forte compatibilité avec MySQL** : protocole réseau commun, tronc SQL partagé, formats de données très proches. En pratique, une part importante d'une application MySQL — schéma, requêtes courantes, types de données usuels — fonctionne sans modification sur MariaDB. Cette compatibilité de surface est précisément ce qui fait de la migration depuis MySQL le scénario le plus accessible.

L'arrivée de **MySQL 8** a toutefois introduit plusieurs changements structurants qui ont obligé MariaDB à ajouter des mécanismes de compatibilité dédiés. Cette sous-section se concentre sur deux d'entre eux, particulièrement importants lors d'une migration depuis MySQL 8 : l'**authentification** (`caching_sha2_password`) et le **support GIS** (nouvelles fonctions spatiales). Les autres écarts de comportement — fonctions, types, options implicites — sont traités dans la sous-section suivante (19.1.2).

La règle générale à retenir est la suivante : la compatibilité doit se **vérifier**, et non se présumer. Les deux points abordés ci-dessous illustrent bien cette nécessité, car ils touchent à des comportements qui ne sont pas visibles dans le schéma mais qui peuvent bloquer la connexion des clients ou fausser les résultats de calculs.

---

## L'authentification : le plugin `caching_sha2_password`

### Le problème de migration

À partir de MySQL 8.0.4, le plugin d'authentification par défaut est passé de `mysql_native_password` à **`caching_sha2_password`**. Concrètement, dans une installation MySQL 8 récente, les comptes utilisateurs sont, par défaut, créés avec ce plugin, et leurs empreintes de mot de passe sont stockées dans ce format.

Or, MariaDB n'a **jamais pris en charge** le plugin `sha256_password` de MySQL, et n'a longtemps pas pris en charge `caching_sha2_password` non plus. Les raisons de ce choix tenaient notamment à la nécessité de distribuer la clé publique du serveur à tous les clients (peu pratique) et au risque lié à la transmission du mot de passe en clair lors de certains échanges. MariaDB a privilégié d'autres mécanismes : `mysql_native_password`, `ed25519`, puis le plugin moderne PARSEC.

Cette divergence créait un point de friction réel : lors d'une migration depuis MySQL 8, les comptes authentifiés par `caching_sha2_password` ne pouvaient pas être repris tels quels. Il fallait soit réinitialiser les mots de passe, soit changer le plugin d'authentification — une opération coûteuse et sensible sur un parc important.

### La réponse de MariaDB

Pour lever cet obstacle, MariaDB a ajouté un plugin **`caching_sha2_password` compatible MySQL**, disponible côté serveur depuis la **série 12.x** (MariaDB Community Server 12.1 ; côté MariaDB Enterprise Server dès la 11.8). Il est donc présent dans **MariaDB 12.3 LTS**, version cible de cette formation.

Ce plugin utilise les **mêmes empreintes de mot de passe** que MySQL. Son objectif est explicite : permettre de **déplacer des utilisateurs depuis MySQL vers MariaDB sans leur faire changer de mot de passe**. Sur le plan technique, il fonctionne comme côté MySQL — un cache en mémoire permet une authentification rapide en cas de succès antérieur, et un échange chiffré par RSA assure la vérification complète sinon.

Il convient de distinguer deux faces de cette compatibilité :

- **Côté client**, les connecteurs MariaDB savent depuis longtemps se connecter à un serveur MySQL 8 utilisant `caching_sha2_password` (MariaDB Connector/C, Connector/ODBC, etc., dans des versions récentes). Ce n'est donc pas la difficulté.
- **Côté serveur**, la nouveauté apportée par la série 12.x est que **le serveur MariaDB lui-même** peut désormais authentifier des comptes stockés au format `caching_sha2_password`. C'est ce qui rend possible la reprise des empreintes MySQL.

### Mise en œuvre pratique

Le plugin n'est **pas installé par défaut** : sa bibliothèque partagée (`auth_mysql_sha2.so` sous Linux, `auth_mysql_sha2.dll` sous Windows) est livrée avec MariaDB mais doit être activée explicitement. Elle peut l'être sans redémarrage du serveur :

```sql
-- Installation du plugin (à chaud)
INSTALL SONAME 'auth_mysql_sha2';
```

Pour reprendre un compte existant de MySQL en conservant son mot de passe, on crée l'utilisateur en réutilisant la chaîne d'authentification (`authentication_string`) lue dans la table `mysql.user` de l'instance MySQL d'origine :

```sql
CREATE USER 'app'@'%'
  IDENTIFIED WITH caching_sha2_password
  AS '<authentication_string issue de mysql.user>';
```

Deux précautions méritent l'attention :

1. La chaîne d'authentification de `caching_sha2_password` peut contenir des **caractères non imprimables** ; un simple copier-coller depuis un terminal échoue souvent. Il faut extraire la valeur de manière fiable (export programmatique, requête dédiée).
2. Le plugin s'appuie sur une paire de clés RSA, configurable via les variables `caching_sha2_password_private_key_path`, `caching_sha2_password_public_key_path` et `caching_sha2_password_auto_generate_rsa_keys`.

### Recommandation

`caching_sha2_password` doit être considéré comme une **aide à la migration**, et non comme un choix d'authentification définitif. Une fois la migration stabilisée, il est recommandé de basculer les comptes vers un mécanisme plus sûr et plus pratique du point de vue de MariaDB — **PARSEC** (par défaut visé pour l'avenir) ou `ed25519`. Le plugin `caching_sha2_password` permet la transition ; il n'a pas vocation à rester le mode d'authentification de production sur le long terme.

## Le support spatial (GIS) et les fonctions de MySQL 8

### Pourquoi le GIS est un point d'attention particulier

Le domaine spatial (GIS, *Geographic Information System*) est l'un de ceux où MySQL et MariaDB ont le plus divergé. Les deux moteurs s'appuient sur le standard OpenGIS et exposent des fonctions préfixées par `ST_`, mais MySQL 8 a opéré un changement de fond qui dépasse le simple ajout de fonctions.

Dans MySQL 5.7 et les versions antérieures, l'identifiant de système de référence spatiale (**SRID**) était stocké avec la géométrie mais **ignoré dans les calculs** : toutes les opérations se faisaient sur un plan cartésien (SRID 0). À partir de **MySQL 8.0**, le SRID **a un sens et influence le calcul** :

- **SRID 0** désigne un plan cartésien infini, sans unité ;
- un **SRS géographique** comme **SRID 4326** (WGS 84, le système des coordonnées GPS) déclenche un calcul **géodésique sur un modèle d'ellipsoïde**, et non plus sur un plan ;
- un **SRS projeté** comme SRID 3857 (Web Mercator, utilisé par les fonds de carte en ligne) correspond à une projection plane.

MySQL 8 est livré avec plus de 5000 définitions de SRS (issues du catalogue EPSG) et a introduit des fonctions associées telles que `ST_Transform` (conversion entre SRS), `ST_Distance_Sphere`, `ST_GeomFromGeoJSON` / `ST_AsGeoJSON`, ou encore `ST_Latitude` / `ST_Longitude`. Dans le même mouvement, certaines fonctions historiques sans préfixe `ST_` (comme `GeomFromText` ou `AsText`) ont été dépréciées puis retirées.

Conséquence pratique : avec un SRS géographique, deux mêmes coordonnées peuvent produire un résultat de distance différent selon le SRID employé. C'est exactement ce type de comportement qu'une migration doit valider.

```sql
-- Illustration du principe (comportement MySQL 8) :
-- calcul cartésien (plan) avec SRID 0
SELECT ST_Distance(
  ST_GeomFromText('POINT(0 0)', 0),
  ST_GeomFromText('POINT(1 1)', 0)
);

-- calcul géodésique (ellipsoïde) avec SRID 4326
SELECT ST_Distance(
  ST_GeomFromText('POINT(0 0)', 4326),
  ST_GeomFromText('POINT(1 1)', 4326)
);
```

### Ce qu'apporte MariaDB 12.3

MariaDB prend en charge depuis longtemps les extensions spatiales OpenGIS et une large gamme de fonctions `ST_` (dont `ST_Distance_Sphere` ou `ST_GeomFromGeoJSON`), utilisables avec les moteurs Aria, MyISAM et InnoDB notamment.

**MariaDB 12.3 ajoute plusieurs nouvelles fonctions GIS spécifiquement destinées à améliorer la compatibilité avec MySQL 8** : `ST_IsValid`, `ST_Validate`, `ST_Simplify`, `ST_GeoHash`, `ST_LatFromGeoHash`, `ST_LongFromGeoHash`, `ST_PointFromGeoHash`, `MBRCoveredBy`, ainsi que la fonction d'agrégation `ST_Collect`. Cet apport réduit le risque qu'une application utilisant les fonctions spatiales de MySQL 8 échoue à la migration faute de fonction équivalente côté MariaDB.

Certaines fonctions de MySQL 8 demeurent toutefois sans équivalent direct en 12.3 — notamment `ST_Transform` (conversion entre SRS), `ST_Latitude` et `ST_Longitude` : un code qui les emploie devra être adapté. L'apport de la 12.3 est donc une avancée directe pour les applications géospatiales migrant depuis MySQL 8, sous réserve de ces fonctions encore absentes et de la sémantique de calcul examinée ci-dessous.

### Le point de vigilance qui demeure

La compatibilité **au niveau des fonctions** s'est donc nettement améliorée avec la 12.3. Mais il faut distinguer la disponibilité d'une fonction de la **sémantique du calcul** :

- l'élément le plus délicat de MySQL 8 est le **calcul géodésique dépendant du SRS** (résultats sur ellipsoïde pour les SRS géographiques) ;
- historiquement, le modèle de calcul de MariaDB reposait sur une approche planaire, comparable à celle de MySQL 5.7 ;
- il est donc **indispensable de tester les résultats** des requêtes spatiales après migration, en particulier les distances et les relations géométriques portant sur des données en SRS géographique (4326). Une requête peut « passer » syntaxiquement tout en renvoyant une valeur calculée selon un modèle différent.

En complément, il faut vérifier le comportement des **index spatiaux** et le moteur de stockage utilisé pour les colonnes géométriques, deux éléments qui conditionnent à la fois la validité fonctionnelle et la performance des requêtes géospatiales après bascule.

---

## En résumé

L'authentification et le GIS constituent deux exemples concrets de la façon dont MariaDB **suit les évolutions de MySQL 8** tout en conservant sa propre identité technique :

- le plugin **`caching_sha2_password`** (série 12.x, présent en 12.3) permet de reprendre les comptes MySQL 8 sans réinitialiser les mots de passe, comme **aide à la migration** — PARSEC ou `ed25519` restant préférables ensuite ;
- les **nouvelles fonctions GIS de la 12.3** rapprochent MariaDB des capacités spatiales de MySQL 8, tout en laissant la **sémantique du calcul géodésique (SRS)** comme principal point à valider par des tests.

Ces deux cas confirment le principe directeur de la migration depuis MySQL : une compatibilité large, mais qui se confirme par la vérification. Les différences de comportement plus générales — fonctions, types et options divergentes — font l'objet de la sous-section **19.1.2 (Points d'attention et différences)**.
