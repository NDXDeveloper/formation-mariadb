# 19.1.2 Points d'attention et différences

> **Chapitre 19 — Migration et Compatibilité** · Section 19.1 · Sous-section 19.1.2  
> Niveau : DBA / Architecte

---

## De la compatibilité aux différences concrètes

La sous-section précédente (19.1.1) a traité deux points de compatibilité précis et structurants : l'authentification (`caching_sha2_password`) et le support spatial (GIS). Cette sous-section élargit la perspective : elle recense les **différences de comportement et les pièges** à anticiper lors d'une migration depuis MySQL 8, au-delà de ces deux cas.

Un repère méthodologique important avant d'entrer dans le détail : MariaDB publie des **pages de référence officielles** comparant ses versions à MySQL 8, et c'est la source à consulter en priorité. On y trouve notamment la liste des incompatibilités générales, les différences de fonctions, les différences de variables système et la compatibilité de réplication. Ces pages sont **rattachées à des versions précises** (par exemple une comparaison établie entre MySQL 8.0.36 et MariaDB 12.1.x) ; comme MySQL 8 est une version « evergreen » qui évolue en continu, et que MariaDB évolue elle aussi d'une version à l'autre, il faut toujours **vérifier ces différences contre les versions exactes** que l'on migre — ici, une cible MariaDB 12.3 LTS.

Les différences se regroupent en quelques familles : la méthode de migration imposée par l'architecture, la réplication, les types de données (et surtout le JSON), les jeux de caractères et collations, la syntaxe et le comportement SQL, la configuration et l'exploitation, et enfin les fonctionnalités absentes.

---

## 1. La méthode de migration est contrainte par l'architecture

MySQL 8 stocke ses métadonnées dans un **dictionnaire de données transactionnel** et a abandonné les fichiers `.frm`. MariaDB repose sur une organisation interne différente. La conséquence est directe et fondamentale : **on ne peut pas transférer une base entre MySQL 8 et MariaDB par simple copie des fichiers de données**, et les sauvegardes physiques ne sont pas interchangeables (Mariabackup ne lit pas un instantané physique produit par les outils de MySQL, et inversement).

La migration depuis MySQL doit donc être **logique** : export puis import des données et des définitions d'objets (via `mysqldump`/`mariadb-dump` ou des outils dédiés), ou s'appuyer sur des mécanismes de réplication compatibles. L'outillage correspondant est détaillé dans la sous-section 19.1.3.

---

## 2. Réplication

La réplication est l'autre grand point structurant, car elle conditionne les stratégies de migration à faible interruption.

- La **GTID de MariaDB n'est pas compatible avec celle de MySQL**, et les deux moteurs utilisent des variables système de GTID différentes, qu'il faut ajuster lors d'une migration.
- Il en découle qu'une migration **par réplication** depuis MySQL 8 vers MariaDB est possible mais **contrainte** : elle doit être validée soigneusement et ne se configure pas comme une simple réplication homogène. Les stratégies sans interruption sont traitées en 19.8.
- D'autres nuances existent côté journal binaire : pour fiabiliser la réplication, `CREATE TABLE ... SELECT` est par défaut exécuté comme `CREATE OR REPLACE TABLE` sur le réplica (réglable via `slave-ddl-exec-mode`) ; par ailleurs, le contenu du binlog diffère historiquement entre les deux moteurs.

La page officielle « Replication Compatibility Between MariaDB and MySQL » fait référence sur ce sujet et doit être consultée avant tout projet de migration par réplication.

---

## 3. Le type JSON : le point de données le plus sensible

C'est la différence de type la plus susceptible de poser problème lors d'une migration depuis MySQL 8.

**Stockage.** MariaDB stocke le JSON comme du **texte** : son type `JSON` est un alias pour `LONGTEXT COLLATE utf8mb4_bin`, assorti automatiquement d'une contrainte `CHECK (JSON_VALID(...))`. MySQL, lui, stocke le JSON sous forme **binaire compacte**. Les deux formats internes ne sont pas interchangeables.

**Conséquences pour la migration.**

- En migration **logique**, MariaDB sait relire un `mysqldump` issu de MySQL. Pour convertir automatiquement le format binaire de MySQL en texte, MariaDB fournit le plugin **`mysql_json`** ; à défaut, on convertit les colonnes en `TEXT` côté MySQL avant l'export.
- En **réplication**, la réplication par lignes (*row-based*) des colonnes JSON depuis MySQL vers MariaDB **ne fonctionne pas**. Les contournements consistent à utiliser la réplication par instructions (*statement-based*), à stocker les colonnes en `TEXT`/`LONGTEXT` côté MySQL, ou à employer le plugin `mysql_json`.

**Sémantique de comparaison.** Dans MySQL, le JSON est comparé selon ses **valeurs JSON** ; dans MariaDB, les chaînes JSON sont des chaînes ordinaires, **comparées comme du texte**. Toute requête qui s'appuie sur le tri ou la comparaison de valeurs JSON peut donc se comporter différemment après migration et doit être testée.

**Comportement sur données invalides.** Sur une entrée non valide, `JSON_SEARCH` renvoie côté MariaDB une valeur nulle accompagnée d'un avertissement (comportement conforme au standard), là où MySQL produit une erreur.

---

## 4. Jeux de caractères et collations

C'est un piège fréquent et facile à sous-estimer, car il bloque l'import d'un dump avant même toute exécution applicative.

- La **collation par défaut de MySQL 8, `utf8mb4_0900_ai_ci`** (fondée sur Unicode 9.0.0), **n'existe pas dans MariaDB**. Un dump produit par MySQL 8 référence donc des collations que MariaDB ne reconnaît pas. Il faut les **remplacer** par une collation MariaDB équivalente — par exemple une collation `utf8mb4_uca1400_*` (collations UCA fondées sur Unicode 14.0.0, disponibles dans les versions récentes de MariaDB) ou, plus simplement, `utf8mb4_general_ci`.
- Les **catalogues diffèrent** : MariaDB prend en charge davantage de collations que MySQL, tandis que MySQL propose des jeux de caractères que MariaDB n'a pas (par exemple `gb18030`).
- La **convention de nommage** des collations diffère : MariaDB indique le statut de remplissage (*pad*) directement dans le nom (par exemple `utf8mb3_unicode_nopad_ci`), alors que MySQL l'expose via une colonne supplémentaire de `SHOW COLLATION`.

Avant tout import, il est donc indispensable d'**auditer les clauses `CHARACTER SET` et `COLLATE`** présentes dans le schéma et le dump, et de prévoir leur transposition.

---

## 5. Différences de syntaxe et de comportement SQL

Certaines différences, plus ponctuelles, peuvent néanmoins casser des requêtes applicatives ou des scripts. Les plus utiles à connaître :

- **Fonction `VALUE()` à côté de `VALUES()`.** MariaDB a ajouté la fonction **`VALUE()`**, qui renvoie la valeur à insérer dans un `INSERT ... ON DUPLICATE KEY UPDATE`, parce que le mot-clé `VALUES` sert désormais aussi à construire un ensemble de lignes (*Table Value Constructors*). Point rassurant pour la migration : dans un `INSERT ... ON DUPLICATE KEY UPDATE`, **`VALUES(colonne)` continue de fonctionner** (compatibilité MySQL) — le code existant n'est donc généralement pas cassé. `VALUE()` est simplement la forme native et non ambiguë de MariaDB ; `VALUES(...)` employé isolément (hors `ON DUPLICATE KEY UPDATE`) y est, lui, interprété comme un constructeur de lignes.
- **Verrous de lecture.** Le mot-clé `FOR SHARE` introduit par MySQL 8 correspond, côté MariaDB, à `LOCK IN SHARE MODE` ; la syntaxe exacte des clauses de verrouillage doit être vérifiée lors de la migration.
- **Chiffrement applicatif AES.** Historiquement, MariaDB ne gérait que le mode ECB, sans vecteur d'initialisation. **Depuis MariaDB 11.2** (donc en 12.3), `AES_ENCRYPT`/`AES_DECRYPT` acceptent un argument `init_vector` et la variable `block_encryption_mode` (`aes-{128,192,256}-{ecb,cbc,ctr}`), à l'instar de MySQL : l'écart historique est comblé. Les deux moteurs ayant par défaut le mode `aes-128-ecb`, le point à valider lors d'une migration est surtout la **concordance du mode et de la clé** (et de l'IV en CBC/CTR) entre l'écriture et la lecture des données chiffrées.
- **Tables dérivées latérales.** MariaDB ne prend pas en charge les *Lateral Derived Tables* ; une requête MySQL utilisant `LATERAL` doit être réécrite.
- **Littéraux binaires `X'HHHH'`.** MariaDB les traite toujours comme des chaînes, alors que MySQL peut les interpréter comme un nombre ou une chaîne selon le contexte.
- **`SHOW CREATE TABLE`.** MariaDB ne met pas entre guillemets la valeur `DEFAULT` d'un entier et ajoute `DEFAULT NULL` aux colonnes `BLOB`/`TEXT` nullables (MariaDB autorise des valeurs par défaut pour `BLOB`/`TEXT`, contrairement à MySQL). Ces écarts affectent surtout les outils de comparaison de schéma et de migration.
- **Spatial (complément à 19.1.1).** La syntaxe `REF_SYSTEM_ID` de MariaDB pour les colonnes géométriques n'existe pas dans MySQL ; à l'inverse, l'instruction `CREATE SPATIAL REFERENCE SYSTEM` de MySQL 8 n'est pas prise en charge par MariaDB.
- **Codes d'erreur de verrouillage.** En cas d'échec d'acquisition d'un verrou, MariaDB renvoie `ERROR 1205`, tandis que MySQL renvoie `ERROR 3572` dans le contexte `NOWAIT` ; le code applicatif de gestion d'erreur peut devoir être ajusté.

## 6. Configuration et exploitation

Au-delà du SQL, plusieurs différences touchent l'administration et doivent être intégrées aux scripts et aux procédures d'exploitation.

- **Persistance de la configuration.** MariaDB ne prend pas en charge `SET PERSIST` (ni l'instruction `RESTART`). La configuration persistante passe donc par les fichiers `my.cnf`/`my.ini`, et non par des commandes en ligne persistées.
- **Performance Schema.** Activé par défaut dans MySQL, il est **désactivé par défaut dans MariaDB** pour des raisons de performance ; on l'active en démarrant le serveur avec l'option `--performance-schema`. La chaîne de supervision doit en tenir compte.
- **Query Cache.** Retiré dans MySQL 8, il reste présent dans MariaDB (mais y est déprécié — voir §15.3). Une configuration importée ne doit pas s'appuyer dessus.
- **InnoDB.** MariaDB a supprimé le *Change Buffer* d'InnoDB et implémente le **chiffrement au repos différemment** de MySQL 8 ; une base chiffrée demande donc une attention particulière lors de la migration.
- **Outils et binaires.** Tous les binaires `mysql*` sont renommés `mariadb*` (des liens symboliques `mysql*` sont conservés pour compatibilité) ; l'initialisation se fait via `mariadb-install-db` et non l'option `--initialize`. Les scripts d'automatisation référençant les anciens noms fonctionnent généralement grâce aux liens, mais méritent d'être revus.
- **Préfixes d'options.** MariaDB accepte (avec avertissement) un préfixe unique d'option, là où MySQL exige le nom complet — un point à relier aux variables retirées en 12.x (voir §11.2.3).

---

## 7. Fonctionnalités de MySQL absentes de MariaDB

Si l'application repose sur l'une des fonctionnalités suivantes, il faut prévoir une solution de remplacement avant la migration :

- le **X Plugin / X DevAPI** de MySQL (mode *document store*) n'est pas pris en charge ;
- les analyseurs *full-text* **ngram** et **MeCab** (recherche plein texte sur le chinois, le japonais, le coréen) ne sont pas pris en charge ;
- l'instruction **`CREATE TABLESPACE`** pour InnoDB n'est pas prise en charge ;
- le **plugin memcached** n'est pas pris en charge (les données restent toutefois accessibles puisqu'elles sont stockées dans des tables InnoDB).

À l'inverse, MariaDB offre de nombreuses fonctionnalités absentes de MySQL (séquences, tables temporelles à versionnement système et à période applicative, recherche vectorielle, moteurs de stockage supplémentaires, compatibilité Oracle, etc.). Pour une migration MySQL → MariaDB, c'est généralement un atout ; cela ne devient un point d'attention que si l'on doit préserver une compatibilité croisée avec MySQL.

---

## 8. Optimiseur et performances

MariaDB et MySQL ont des **optimiseurs et des modèles de coût distincts**. Une requête identique peut donc obtenir un **plan d'exécution différent** sur les deux moteurs, et donc des performances différentes — meilleures comme moins bonnes. Il ne faut pas présumer une équivalence des plans : après migration, il convient de **revalider les performances**, de réexaminer les index et de relire les plans avec `EXPLAIN` sur les requêtes critiques.

---

## En résumé

La grande majorité de ces différences sont gérables, mais elles doivent être **inventoriées avant la migration**, et non découvertes en production. La démarche recommandée combine trois réflexes :

1. **Consulter les pages de référence officielles** de MariaDB (incompatibilités, différences de fonctions et de variables, compatibilité de réplication) pour les **versions exactes** concernées — sources qui priment sur toute synthèse, car le périmètre évolue d'une version à l'autre.
2. **Construire une campagne de tests de compatibilité** représentative (voir 19.6).
3. **Concentrer la vigilance** sur les points à plus fort impact : le **JSON**, les **collations**, les stratégies de migration **par réplication**, et tout usage applicatif particulier (chiffrement AES avec vecteur d'initialisation, comparaison de valeurs JSON, X Plugin).

L'outillage permettant de mener concrètement l'export, la conversion et l'import fait l'objet de la sous-section suivante, **19.1.3 (Outils de migration)** ; les tests de compatibilité sont approfondis en **19.6**, et les mécanismes de retour arrière en **19.7**.
