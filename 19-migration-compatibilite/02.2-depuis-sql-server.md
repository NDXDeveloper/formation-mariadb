# 19.2.2 Depuis SQL Server

> **Chapitre 19 — Migration et Compatibilité** · Section 19.2 · Sous-section 19.2.2  
> Niveau : DBA / Architecte

---

## Une migration plus manuelle que depuis Oracle

Contrairement à Oracle, qui bénéficie du mode `SQL_MODE=ORACLE` couvrant une large part du dialecte et du PL/SQL (voir 19.2.1), **SQL Server ne dispose que d'un mode de compatibilité limité** dans MariaDB. La conséquence directe est qu'une migration depuis SQL Server demande **davantage de réécriture manuelle**, en particulier celle du code procédural T-SQL.

MariaDB propose néanmoins des ressources dédiées : une section officielle « Migrating to MariaDB from SQL Server », accompagnée notamment d'une page « Syntax Differences between MariaDB and SQL Server » et d'un guide de transfert de données. Ce sont les points de départ recommandés.

---

## Le mode `MSSQL` : une aide limitée

Disponible depuis **MariaDB 10.4.5**, le mode s'active comme les autres :

```sql
SET SQL_MODE = 'MSSQL';
```

Il implique un ensemble de drapeaux : `PIPES_AS_CONCAT`, `ANSI_QUOTES`, `IGNORE_SPACE`, `NO_KEY_OPTIONS`, `NO_TABLE_OPTIONS` et `NO_FIELD_OPTIONS`. Sa fonctionnalité la plus visible est de permettre l'usage des **crochets `[ ]`** pour délimiter les identifiants, comme dans SQL Server :

```sql
SET SQL_MODE = 'MSSQL';
CREATE TABLE [t 1] ([a b] INT);
```

Il faut bien mesurer sa portée : le mode `MSSQL` ne permet de comprendre qu'un **petit sous-ensemble** du langage de SQL Server, et sa fonctionnalité reste **limitée**. À la différence du mode Oracle, **il n'émule pas le T-SQL**. Il facilite donc la surface syntaxique (quoting, concaténation par `||`), mais l'essentiel de la conversion — types, code procédural, fonctions — reste à la charge de l'équipe de migration.

---

## Quoting des identifiants et des chaînes

Les conventions de délimitation diffèrent et constituent un premier point d'attention :

- SQL Server délimite les **identifiants** par des crochets `[ ]` (et, avec `QUOTED_IDENTIFIER ON`, par des guillemets doubles `"`). MariaDB utilise par défaut les **accents graves** `` ` `` ; le drapeau `ANSI_QUOTES` active les guillemets doubles (équivalent de `QUOTED_IDENTIFIER ON`), et le mode `MSSQL` active les crochets.
- Pour les **chaînes**, SQL Server utilise par défaut les guillemets simples `'` (un guillemet simple interne se double : `''`), comportement identique dans MariaDB. SQL Server autorise aussi les guillemets doubles `"` pour les chaînes : cela fonctionne par défaut dans MariaDB, **mais cesse de fonctionner dès que `ANSI_QUOTES` (donc le mode `MSSQL`) est actif**, puisque `"` devient alors un délimiteur d'identifiant.

Ce dernier point est un piège classique : activer le mode `MSSQL` pour gérer les crochets peut casser des requêtes qui utilisaient `"` autour de chaînes.

## Correspondance des types de données

De nombreux types SQL Server n'ont pas d'équivalent strict et doivent être transposés. Les correspondances usuelles :

| Type SQL Server | Type MariaDB | Remarques |
|---|---|---|
| `NVARCHAR(n)`, `NCHAR(n)` | `VARCHAR(n)`, `CHAR(n)` | en `utf8mb4` |
| `VARCHAR(MAX)`, `NVARCHAR(MAX)` | `LONGTEXT` (ou `TEXT`) | |
| `VARBINARY(MAX)`, `IMAGE` | `LONGBLOB` | |
| `DATETIME2`, `SMALLDATETIME` | `DATETIME` | |
| `DATETIMEOFFSET` | `DATETIME` / `TIMESTAMP` | gérer le fuseau horaire séparément |
| `MONEY`, `SMALLMONEY` | `DECIMAL` | choisir précision et échelle |
| `BIT` | `TINYINT(1)` / `BOOLEAN` | |
| `TINYINT` | `TINYINT UNSIGNED` | **signedness** : `TINYINT` de SQL Server va de 0 à 255, celui de MariaDB est signé |
| `UNIQUEIDENTIFIER` | `UUID` (depuis 10.7) ou `CHAR(36)`/`BINARY(16)` | |
| `IDENTITY` | `AUTO_INCREMENT` | |
| `XML` | type `XML` basique (§2.2.6) ou `LONGTEXT` | le type XML de MariaDB est basique |
| `ROWVERSION` / `TIMESTAMP` | traitement particulier | **piège** : le `TIMESTAMP` de SQL Server n'est pas une date, mais un compteur binaire de version de ligne |

Deux pièges méritent d'être soulignés : la **différence de signe** de `TINYINT`, et le `TIMESTAMP`/`ROWVERSION` de SQL Server qui, contrairement à son homonyme MariaDB, **n'est pas une donnée temporelle** mais un marqueur de version de ligne — il faut le remplacer par une logique applicative ou un déclencheur.

## Du T-SQL vers le SQL/PSM (le chantier le plus lourd)

Les langages procéduraux sont fondamentalement différents : procédures stockées, fonctions et déclencheurs écrits en **T-SQL** doivent être **réécrits** en SQL/PSM, le langage procédural de MariaDB. Les principaux points de conversion :

- **Variables** : en T-SQL `DECLARE @v INT; SET @v = 1;` ; en MariaDB, les variables locales d'une routine n'ont pas de préfixe `@` (réservé aux variables de session) : `DECLARE v INT; SET v = 1;`.
- **Structures de contrôle** : `IF … BEGIN … END` et `WHILE … BEGIN … END` deviennent `IF … THEN … END IF;` et `WHILE … DO … END WHILE;`.
- **Gestion des erreurs** : `TRY … CATCH`, `RAISERROR`/`THROW` et `@@ERROR` deviennent `DECLARE … HANDLER` et `SIGNAL`/`RESIGNAL`.
- **Récupération de l'identité** : `SCOPE_IDENTITY()`, `@@IDENTITY` deviennent `LAST_INSERT_ID()`.
- **Tables temporaires** : `#temp` (session) et `##temp` (globale) deviennent des `CREATE TEMPORARY TABLE`.
- **`MERGE`** : à transposer en `INSERT … ON DUPLICATE KEY UPDATE` ou `REPLACE`.
- **Séparateur de lots `GO`** : inexistant dans MariaDB ; on utilise `;` et, pour la création de routines, `DELIMITER`.
- **`SELECT … INTO nouvelle_table`** : en T-SQL, cette forme crée une table ; en MariaDB, on utilise `CREATE TABLE … SELECT` (la forme `SELECT … INTO` de MariaDB cible des variables).

## Différences de fonctions et de requêtes

Au niveau des requêtes, plusieurs constructions et fonctions diffèrent :

| Construction SQL Server | Équivalent MariaDB | Remarques |
|---|---|---|
| `TOP n` | `LIMIT n` | `OFFSET/FETCH` → `LIMIT … OFFSET …` |
| `'a' + 'b'` (concat.) | `CONCAT('a','b')` | en MariaDB `+` est une addition ; `||` possible avec `PIPES_AS_CONCAT` |
| `GETDATE()`, `SYSDATETIME()` | `NOW()`, `SYSDATE()` | |
| `ISNULL(a,b)` | `IFNULL(a,b)` / `COALESCE` | |
| `LEN(s)` | `CHAR_LENGTH(s)` | `LEN` ignore les espaces de fin |
| `DATALENGTH(s)` | `LENGTH(s)` | en octets |
| `CHARINDEX(sub,str)` | `LOCATE(sub,str)` | `INSTR(str,sub)` inverse l'ordre des arguments |
| `DATEADD(part,n,date)` | `DATE_ADD(date, INTERVAL n part)` | |
| `DATEDIFF(part,d1,d2)` | `TIMESTAMPDIFF(part,d1,d2)` | **sémantique différente** : le `DATEDIFF` de MariaDB ne prend que deux dates et renvoie un nombre de jours |
| `DATEPART(part,date)` | `EXTRACT(part FROM date)` | |
| `IIF(c,a,b)` | `IF(c,a,b)` / `CASE` | |
| `CONVERT(type,expr,style)` | `CAST`/`CONVERT` | les codes de style de date diffèrent |

Le passage de la concaténation par `+` à `CONCAT()` et la différence de sémantique de `DATEDIFF` figurent parmi les sources d'erreurs les plus fréquentes.

---

## Schémas et nommage des objets

L'organisation des espaces de noms diffère : SQL Server utilise une hiérarchie à trois niveaux `base.schéma.objet` (avec le schéma `dbo` par défaut), tandis que MariaDB utilise `base.objet`, la **base jouant le rôle de schéma**. Lors de la migration, les schémas SQL Server doivent être transposés : le plus souvent, un schéma (comme `dbo`) est replié dans une base MariaDB, et les références à trois niveaux sont réécrites en deux niveaux.

À noter également : dans SQL Server, les procédures et fonctions sont insensibles à la casse, ce qui simplifie ce point particulier de la migration.

---

## Transfert des données et outils

Pour le transfert des données proprement dit :

- le **moteur CONNECT** (voir §7.10.4) permet, via un pilote **ODBC** SQL Server, de créer des tables externes puis de rapatrier les données par `INSERT … SELECT` vers des tables natives InnoDB ;
- des **outils d'ETL et de migration** automatisent l'extraction et le chargement, ainsi qu'une partie de la conversion.

Côté outillage de conversion :

- la famille **SQLines** convertit le schéma (DDL), les vues, les procédures, fonctions, déclencheurs et le code T-SQL depuis Microsoft SQL Server (ainsi qu'Azure SQL Database et Azure Synapse) vers MariaDB ;
- l'**assistant de migration de MySQL Workbench** (avec un pilote ODBC) et divers **kits commerciaux** offrent une migration assistée.

Comme toujours, ces outils accélèrent le travail sans le garantir : la conversion du T-SQL en particulier exige une **relecture et des tests** approfondis.

---

## En résumé

La migration depuis SQL Server est plus exigeante que depuis Oracle, car le mode `MSSQL` n'apporte qu'une aide **limitée** :

- il facilite la **surface syntaxique** (crochets `[ ]`, concaténation `||`, guillemets) mais **n'émule pas le T-SQL** ;
- la **correspondance des types** demande des choix explicites, avec des pièges notables (`TINYINT` signé, `ROWVERSION`/`TIMESTAMP` non temporel) ;
- la **conversion du T-SQL** vers le SQL/PSM constitue le poste le plus lourd ;
- les **fonctions et requêtes** (concaténation, `DATEDIFF`, `TOP`, conversions de date) doivent être adaptées une à une ;
- les **schémas** à trois niveaux sont repliés en deux niveaux.

La démarche s'appuie sur les guides officiels de migration SQL Server de MariaDB, le moteur CONNECT pour les données, et les outils de conversion — le tout validé par des tests rigoureux (voir §19.6 et §19.7).
