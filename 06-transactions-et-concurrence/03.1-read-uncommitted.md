# 6.3.1 READ UNCOMMITTED : lectures sales possibles

`READ UNCOMMITTED` est le **niveau d'isolation le plus permissif**. Une transaction qui s'exécute à ce niveau peut **lire les modifications d'autres transactions avant même qu'elles soient validées**. C'est le seul niveau qui autorise la **lecture sale** (*dirty read*) — et il laisse également passer les lectures non répétables et les lectures fantômes.

> Les trois anomalies de lecture sont définies dans la [page parent §6.3](03-niveaux-isolation.md). Cette sous-section se concentre sur le comportement propre à `READ UNCOMMITTED`.

## Principe

À ce niveau, le SGBD **n'isole pratiquement pas** la transaction : ses lectures voient l'état le plus récent des données, **y compris les écritures non encore validées** par d'autres transactions. Aucune garantie n'est offerte quant à la stabilité ou à la validité de ce qui est lu.

| Anomalie | Autorisée à `READ UNCOMMITTED` ? |
|----------|:--------------------------------:|
| Lecture sale (*dirty read*) | ✅ Oui |
| Lecture non répétable | ✅ Oui |
| Lecture fantôme | ✅ Oui |

## Démonstration d'une lecture sale

Soit un compte `id = 1` dont le solde initial est `1000`. Deux sessions s'exécutent en parallèle ; la session A lit au niveau `READ UNCOMMITTED`.

```sql
-- État initial
SELECT solde FROM comptes WHERE id = 1;   -- 1000
```

| Étape | Session A — `READ UNCOMMITTED` | Session B |
|:---:|---|---|
| ① | | `START TRANSACTION;` |
| ② | | `UPDATE comptes SET solde = 5000 WHERE id = 1;` *(non validé)* |
| ③ | `SELECT solde FROM comptes WHERE id = 1;` → **5000** | |
| ④ | | `ROLLBACK;` *(le solde redevient 1000)* |
| ⑤ | A a lu **5000**, une valeur qui n'a **jamais existé** durablement | |

À l'étape ③, la session A lit `5000` : une valeur **fantôme**, issue d'une transaction que la session B finit par **annuler**. Si la session A avait pris une décision sur la base de ce `5000` (autoriser un retrait, déclencher une alerte…), cette décision reposerait sur une donnée erronée.

## Comportement dans InnoDB

À `READ UNCOMMITTED`, InnoDB effectue ses lectures **sans constituer d'instantané cohérent** : il lit la version courante des lignes, telle qu'elle se présente à l'instant de la lecture, **sans tenir compte** du fait que les modifications observées soient validées ou non.

> 💡 **Nuance importante** : dans un moteur fondé sur le verrouillage, `READ UNCOMMITTED` servait historiquement à **éviter de poser des verrous de lecture**, et donc à ne pas être bloqué par les écrivains. Dans InnoDB, cet avantage est en grande partie **déjà acquis aux niveaux supérieurs** : grâce au **MVCC** ([§6.6](06-mvcc.md)), les lectures simples (`SELECT` non verrouillants) ne posent pas de verrou et ne bloquent pas les écrivains, même à `READ COMMITTED` ou `REPEATABLE READ`. Le gain de performance de `READ UNCOMMITTED` est donc **marginal** sous InnoDB, alors que le risque d'incohérence, lui, est bien réel.

## Comment l'activer

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
START TRANSACTION;
  SELECT solde FROM comptes WHERE id = 1;
COMMIT;
```

> Les trois portées (`SET TRANSACTION`, `SET SESSION`, `SET GLOBAL`) sont décrites en [§6.3](03-niveaux-isolation.md).

## Cas d'usage (rares) et risques

`READ UNCOMMITTED` ne se justifie que dans des situations **où l'exactitude n'est pas critique** :

- **estimations approximatives** : comptages d'ordre de grandeur, tableaux de bord tolérant une légère imprécision ;
- **diagnostic / supervision** : observer la **progression** d'une opération longue en cours (par exemple un chargement massif), précisément parce qu'on souhaite voir le travail non encore validé.

> ⚠️ En contrepartie, ce niveau est **à proscrire** dès que la justesse compte :  
> - ne **jamais** fonder une **écriture** ni une **décision métier** sur une lecture sale ;  
> - les agrégats (`SUM`, `COUNT`…) calculés à ce niveau peuvent être **incohérents** ;  
> - une valeur lue peut être **annulée** l'instant d'après et n'avoir jamais réellement existé.

Dans la quasi-totalité des applications, on lui préférera `READ COMMITTED` ou `REPEATABLE READ`.

## À retenir

- `READ UNCOMMITTED` = niveau **minimal** ; **toutes** les anomalies de lecture sont possibles.
- Sa signature est la **lecture sale** : voir des données non validées, susceptibles d'être annulées.
- Sous InnoDB, son intérêt en performance est **faible** (le MVCC rend déjà les lectures non bloquantes aux niveaux supérieurs).
- Réservé à des usages **non critiques** (estimations, supervision) ; déconseillé partout ailleurs.
