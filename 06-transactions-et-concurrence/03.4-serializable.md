# 6.3.4 SERIALIZABLE : isolation maximale

`SERIALIZABLE` est le niveau d'isolation **le plus strict**. Il garantit que le résultat de transactions concurrentes est **identique** à celui qu'on obtiendrait en les exécutant **les unes après les autres**, en série. À ce niveau, **aucune** des trois anomalies de lecture n'est possible — y compris la lecture fantôme, formellement écartée.

> Les trois anomalies de lecture sont définies dans la [page parent §6.3](03-niveaux-isolation.md).

## Principe

Le SGBD se comporte comme si les transactions n'avaient **aucun recouvrement** dans le temps. Cette garantie a un coût : pour l'obtenir, InnoDB **verrouille** beaucoup plus largement, ce qui **sérialise** de fait les accès concurrents et **réduit fortement la concurrence**.

| Anomalie | Autorisée à `SERIALIZABLE` ? |
|----------|:----------------------------:|
| Lecture sale (*dirty read*) | ❌ Non |
| Lecture non répétable | ❌ Non |
| Lecture fantôme | ❌ Non |

## Comportement dans InnoDB

Sous InnoDB, `SERIALIZABLE` fonctionne **comme `REPEATABLE READ`**, à une différence majeure près : lorsque l'autocommit est désactivé (c'est-à-dire à l'intérieur d'une transaction explicite), InnoDB **convertit implicitement chaque `SELECT` simple en `SELECT … LOCK IN SHARE MODE`**. Autrement dit, **toutes les lectures posent un verrou partagé**.

Les conséquences :

- une lecture **bloque** désormais les écrivains qui voudraient modifier les lignes lues (et réciproquement) — fini les lectures non bloquantes du MVCC ([§6.6](06-mvcc.md)) ;
- les verrous d'intervalle (*next-key locks*) posés sur les plages lues **empêchent toute insertion** de lignes correspondantes : c'est ainsi que les **lectures fantômes** sont éliminées ([§6.4](04-verrous.md)) ;
- en contrepartie, les **attentes de verrou**, les **dépassements de délai** (`innodb_lock_wait_timeout`) et les **interblocages** (*deadlocks*) deviennent **nettement plus fréquents** ([§6.5](05-deadlocks-resolution.md)).

> Cas particulier : si l'autocommit est **activé**, un `SELECT` simple constitue sa propre transaction, en lecture seule ; il peut être sérialisé sans poser de verrou ni bloquer personne.

## Démonstration : une lecture bloque une écriture

Compte `id = 1`, solde `1000`. La session A lit au niveau `SERIALIZABLE`, dans une transaction.

| Étape | Session A — `SERIALIZABLE` | Session B |
|:---:|---|---|
| ① | `START TRANSACTION;` | |
| ② | `SELECT solde … id = 1;` → **1000** *(pose un verrou partagé)* | |
| ③ | | `UPDATE comptes SET solde = 2000 WHERE id = 1;` → **bloquée (en attente)** |
| ④ | `COMMIT;` *(libère le verrou)* | |
| ⑤ | | l'`UPDATE` se débloque et s'exécute → `2000` |

À l'étape ③, la session B doit **attendre** : le simple `SELECT` de A a verrouillé la ligne. Au niveau `REPEATABLE READ`, ce même `SELECT` n'aurait posé **aucun verrou** (lecture MVCC), et l'`UPDATE` de B serait passé immédiatement. Si l'attente de B excédait `innodb_lock_wait_timeout` (50 s par défaut), elle échouerait avec une erreur de dépassement de délai.

## Journalisation binaire

Comme `REPEATABLE READ`, `SERIALIZABLE` est **compatible avec tous les formats de binlog** (`STATEMENT`, `ROW`, `MIXED`).

## Cas d'usage

`SERIALIZABLE` est **rarement** nécessaire et doit rester l'**exception**. Il ne se justifie que lorsque la **cohérence absolue** prime sur la performance :

- traitements critiques exigeant une stricte sérialisation (certains traitements comptables ou financiers) ;
- maintien d'**invariants complexes** portant sur plusieurs lignes, que les niveaux inférieurs ne peuvent garantir.

> **À noter** : l'isolation par instantané activée par défaut (depuis la 11.6.2) renforce `REPEATABLE READ` contre les **mises à jour perdues** ([§6.9](09-snapshot-isolation.md)), ce qui réduit le besoin de recourir à `SERIALIZABLE` dans certains cas. Mais attention : l'isolation par instantané **ne protège pas contre l'écriture biaisée** (*write skew*) — deux transactions qui lisent un même état, écrivent chacune sur des lignes distinctes et, ensemble, violent une règle qu'aucune ne violait seule (exemple classique : deux médecins de garde se retirant chacun du planning parce que « l'autre est de garde », aboutissant à zéro médecin). Seul `SERIALIZABLE` garantit une **sérialisabilité complète** et écarte ce type d'anomalie.

En contrepartie systématique, à ce niveau l'application **doit** être conçue pour **détecter et rejouer** les transactions échouées sur interblocage ou dépassement de délai ([§6.5](05-deadlocks-resolution.md)).

## Comment l'activer

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
START TRANSACTION;
  SELECT solde FROM comptes WHERE id = 1;   -- pris en charge comme SELECT … LOCK IN SHARE MODE
  -- … traitement nécessitant une isolation totale …
COMMIT;
```

> Les trois portées (`SET TRANSACTION`, `SET SESSION`, `SET GLOBAL`) sont décrites en [§6.3](03-niveaux-isolation.md).

## À retenir

- `SERIALIZABLE` = isolation **maximale** ; les transactions se comportent comme si elles s'exécutaient **en série**.
- **Aucune** anomalie de lecture (y compris les fantômes).
- Sous InnoDB, il **transforme les `SELECT` simples en lectures verrouillantes** (`LOCK IN SHARE MODE`) : les lectures bloquent les écritures → **concurrence réduite**, **deadlocks fréquents**.
- À réserver aux exigences de cohérence **absolue** ; prévoir le **rejeu** des transactions échouées.
- Seul niveau à écarter l'**écriture biaisée** (*write skew*), que l'isolation par instantané (activée par défaut) ne couvre pas.
