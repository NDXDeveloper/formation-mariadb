# 6.3.2 READ COMMITTED : lectures cohérentes

À `READ COMMITTED`, une transaction ne lit **que des données validées** : elle ne voit jamais les modifications non encore confirmées des autres transactions. Ce niveau **élimine la lecture sale**, mais autorise encore les **lectures non répétables** et les **lectures fantômes**.

> Les trois anomalies de lecture sont définies dans la [page parent §6.3](03-niveaux-isolation.md).

## Principe

Chaque requête voit la base **telle qu'elle a été validée à l'instant précis où la requête s'exécute**. La conséquence directe : deux exécutions successives d'une même requête, au sein d'une même transaction, peuvent renvoyer des résultats **différents** si une autre transaction a validé un changement entre-temps.

| Anomalie | Autorisée à `READ COMMITTED` ? |
|----------|:------------------------------:|
| Lecture sale (*dirty read*) | ❌ Non (éliminée) |
| Lecture non répétable | ✅ Oui |
| Lecture fantôme | ✅ Oui |

## Démonstration

Compte `id = 1`, solde initial `1000`. La session A lit au niveau `READ COMMITTED` ; la session B modifie puis valide.

| Étape | Session A — `READ COMMITTED` | Session B |
|:---:|---|---|
| ① | `START TRANSACTION;` | |
| ② | `SELECT solde … id = 1;` → **1000** | |
| ③ | | `START TRANSACTION;` |
| ④ | | `UPDATE comptes SET solde = 1500 WHERE id = 1;` *(non validé)* |
| ⑤ | `SELECT solde … id = 1;` → **1000** | |
| ⑥ | | `COMMIT;` |
| ⑦ | `SELECT solde … id = 1;` → **1500** | |
| ⑧ | `COMMIT;` | |

Cette séquence illustre les **deux faces** de `READ COMMITTED` :

- à l'étape ⑤, la session A lit toujours `1000` bien que B ait déjà exécuté son `UPDATE` : **pas de lecture sale**, car la modification de B n'est pas encore validée ;
- à l'étape ⑦, après le `COMMIT` de B, la même requête renvoie `1500` : c'est une **lecture non répétable**, puisque la valeur a changé à l'intérieur de la transaction A.

## Comportement dans InnoDB

Sous InnoDB, `READ COMMITTED` repose sur le **MVCC** ([§6.6](06-mvcc.md)), avec une particularité : **chaque lecture cohérente (chaque `SELECT`) établit son propre instantané**, recalculé à neuf à chaque instruction. C'est précisément ce rafraîchissement à chaque requête qui rend les lectures **non répétables** — par opposition à `REPEATABLE READ`, qui fige un instantané unique pour toute la durée de la transaction ([§6.3.3](03.3-repeatable-read.md)).

En matière de **verrouillage**, `READ COMMITTED` est plus « léger » :

- InnoDB **ne pose pas de verrous d'intervalle** (*gap locks*), sauf pour la vérification des contraintes de clé étrangère et d'unicité. Les insertions concurrentes dans les « trous » d'un intervalle restent donc possibles — ce qui **autorise les lectures fantômes** mais **améliore la concurrence** ;
- lors d'un `UPDATE`, InnoDB pratique une **lecture semi-cohérente** : les lignes qui ne correspondent pas (selon leur dernière version validée) à la clause `WHERE` ne sont **pas conservées verrouillées**, ce qui réduit encore la contention (voir [§6.4](04-verrous.md)).

> ⚠️ **Réplication / journalisation binaire** : `READ COMMITTED` est **incompatible avec le format de binlog `STATEMENT`** — toute écriture échouerait avec une erreur. Ce niveau impose donc un `binlog_format` à **`ROW`** (ou `MIXED`). Voir [§11.5.2 — Formats du binlog](../11-administration-configuration/05.2-formats-binlog.md).

## Comment l'activer

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
  -- chaque requête verra le dernier état validé
COMMIT;
```

> Les trois portées (`SET TRANSACTION`, `SET SESSION`, `SET GLOBAL`) sont décrites en [§6.3](03-niveaux-isolation.md).

## Cas d'usage

`READ COMMITTED` est un choix pertinent quand :

- on veut que **chaque requête voie les données les plus fraîches** déjà validées (par exemple des traitements qui doivent réagir rapidement aux mises à jour d'autres sessions) ;
- on cherche à **réduire la contention** en environnement très concurrent (absence de *gap locks*).

> 💡 **Repère migration** : `READ COMMITTED` est le niveau d'isolation **par défaut** d'**Oracle**, **PostgreSQL** et **SQL Server**. Une application portée depuis l'un de ces SGBD pourra y retrouver un comportement familier — alors que le défaut d'InnoDB est `REPEATABLE READ`. Voir le [Chapitre 19 — Migration et Compatibilité](../19-migration-compatibilite/README.md).

## À retenir

- `READ COMMITTED` **supprime la lecture sale** : on ne lit que du **validé**.
- Il **laisse passer** les lectures non répétables et les lectures fantômes.
- Sous InnoDB, **chaque requête** prend un instantané neuf → lectures non répétables ; **pas de gap locks** → meilleure concurrence mais fantômes possibles.
- Il **exige** un binlog au format `ROW` ou `MIXED`.
- C'est le **niveau par défaut** de plusieurs autres SGBD (Oracle, PostgreSQL, SQL Server).
