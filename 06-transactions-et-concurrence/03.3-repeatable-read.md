# 6.3.3 REPEATABLE READ : le niveau par défaut d'InnoDB

`REPEATABLE READ` est le **niveau d'isolation par défaut** d'InnoDB, et donc de MariaDB. Il garantit que, **tout au long d'une même transaction**, les lectures répétées d'une donnée renvoient **toujours la même valeur** : la transaction travaille sur un **instantané cohérent** et figé de la base. Il élimine ainsi la **lecture sale** *et* la **lecture non répétable**.

> Les trois anomalies de lecture sont définies dans la [page parent §6.3](03-niveaux-isolation.md).

## Principe

Au moment de sa première lecture, la transaction fige une **vue cohérente** des données. Toutes ses lectures simples ultérieures voient cet état initial, **indépendamment** des modifications validées par d'autres transactions entre-temps. Les lectures sont donc **répétables**.

| Anomalie | Autorisée à `REPEATABLE READ` ? |
|----------|:-------------------------------:|
| Lecture sale (*dirty read*) | ❌ Non (éliminée) |
| Lecture non répétable | ❌ Non (éliminée) |
| Lecture fantôme | ⚠️ Possible selon le standard — **évitée en pratique par InnoDB** |

## Démonstration

Reprenons exactement le scénario de la [§6.3.2](03.2-read-committed.md), mais cette fois la session A lit au niveau `REPEATABLE READ`. Compte `id = 1`, solde initial `1000`.

| Étape | Session A — `REPEATABLE READ` | Session B |
|:---:|---|---|
| ① | `START TRANSACTION;` | |
| ② | `SELECT solde … id = 1;` → **1000** *(fige l'instantané)* | |
| ③ | | `START TRANSACTION;` |
| ④ | | `UPDATE comptes SET solde = 1500 WHERE id = 1;` |
| ⑤ | `SELECT solde … id = 1;` → **1000** | |
| ⑥ | | `COMMIT;` |
| ⑦ | `SELECT solde … id = 1;` → **1000** | |
| ⑧ | `COMMIT;` | |
| ⑨ | `SELECT solde … id = 1;` → **1500** *(nouvelle transaction)* | |

Le contraste avec `READ COMMITTED` est net : à l'étape ⑦, **après** le `COMMIT` de B, la session A lit toujours `1000` (et non `1500`). La lecture est **répétable** dans toute la transaction. Ce n'est qu'à l'étape ⑨, dans une **nouvelle** transaction, que A voit la valeur validée par B.

## Comportement dans InnoDB

- **Un seul instantané par transaction** : la vue cohérente est établie à la **première lecture** de la transaction (et non au `START TRANSACTION`), puis conservée jusqu'au `COMMIT`/`ROLLBACK`. Pour la figer dès l'ouverture, on utilise `START TRANSACTION WITH CONSISTENT SNAPSHOT` ([§6.2](02-gestion-transactions.md)). Le mécanisme sous-jacent (vue de lecture, versions de lignes) est détaillé au [§6.6 — MVCC](06-mvcc.md).
- **Prévention des fantômes** : c'est la grande force d'InnoDB à ce niveau. Les lectures simples ne voient jamais de lignes validées après l'instantané (donc pas de fantôme), et les lectures **verrouillantes** (`SELECT … FOR UPDATE`) s'appuient sur des **verrous d'intervalle** (*next-key locks*) qui empêchent l'insertion de lignes dans la plage parcourue ([§6.4](04-verrous.md)). En pratique, le `REPEATABLE READ` d'InnoDB **écarte les lectures fantômes** dans la quasi-totalité des cas, ce qui le rend **plus strict** que le minimum exigé par le standard.

## Lectures d'instantané vs lectures courantes

Un point **subtil mais essentiel** distingue deux types de lectures à `REPEATABLE READ` :

- les **lectures simples** (`SELECT` non verrouillant) lisent **l'instantané** figé de la transaction ;
- les **lectures courantes** — c'est-à-dire les lectures **verrouillantes** (`SELECT … FOR UPDATE` / `LOCK IN SHARE MODE`) et les écritures (`UPDATE`, `DELETE`) — lisent la **dernière version validée** des lignes, **pas** l'instantané.

Ce comportement peut surprendre : un `SELECT` simple peut afficher une ancienne valeur, alors qu'un `UPDATE` exécuté juste après agit sur la valeur la plus récente.

| Étape | Session A — `REPEATABLE READ` | Session B |
|:---:|---|---|
| ① | `START TRANSACTION;` `SELECT solde … id = 1;` → **1000** | |
| ② | | `UPDATE comptes SET solde = solde + 500 WHERE id = 1;` `COMMIT;` *(validé : 1500)* |
| ③ | `SELECT solde … id = 1;` → **1000** *(instantané)* | |
| ④ | `UPDATE comptes SET solde = solde + 100 WHERE id = 1;` *(lecture courante : part de 1500 → 1600)* | |
| ⑤ | `SELECT solde … id = 1;` → **1600** *(A voit désormais sa propre écriture)* | |

À l'étape ④, l'`UPDATE` ne part pas de `1000` (l'instantané) mais de `1500` (la dernière valeur validée), produisant `1600`. C'est par ce mécanisme que peut survenir une **mise à jour perdue** (*lost update*) si l'application n'y prend pas garde.

> **Isolation par instantané (activée par défaut).** Comme `innodb_snapshot_isolation` est **activée par défaut** (depuis MariaDB 11.6.2, donc en 11.8 LTS comme en 12.3), ce scénario change. Si une transaction tente de verrouiller ou de modifier une ligne **modifiée et validée par une autre transaction depuis la prise de l'instantané**, InnoDB **lève une erreur** au lieu d'effectuer silencieusement la lecture courante. À l'étape ④, l'`UPDATE` échouerait donc (erreur *« record has changed since last read »*) plutôt que de produire `1600`, protégeant ainsi contre la mise à jour perdue. Le détail — et le cas d'une migration depuis une version où la variable était encore à `OFF` (≤ 11.4 LTS / 10.x) — est traité au [§6.9](09-snapshot-isolation.md).

## Journalisation binaire

Contrairement à `READ COMMITTED`, `REPEATABLE READ` est **compatible avec tous les formats de binlog** (`STATEMENT`, `ROW`, `MIXED`). C'est l'une des raisons pour lesquelles il constitue un défaut sûr et polyvalent (voir [§11.5.2](../11-administration-configuration/05.2-formats-binlog.md)).

## Comment l'activer et le vérifier

`REPEATABLE READ` étant le défaut, aucune action n'est nécessaire pour l'utiliser. Pour le définir explicitement ou vérifier le niveau courant :

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;   -- explicite (portées : cf. §6.3)

SELECT @@transaction_isolation;                    -- 'REPEATABLE-READ' par défaut
```

## Pourquoi est-ce le bon défaut ?

- Il offre des **lectures stables** au sein d'une transaction, comportement intuitif et attendu par la plupart des applications.
- Il **élimine** lecture sale et lecture non répétable, et **écarte les fantômes** en pratique (InnoDB) — un excellent compromis cohérence/concurrence.
- Il fonctionne avec **tous les formats de réplication**.
- Il gagne en robustesse grâce à l'isolation par instantané activée par défaut (depuis la 11.6.2).

## À retenir

- `REPEATABLE READ` = **défaut d'InnoDB** ; instantané **unique** pour toute la transaction → lectures **répétables**.
- Élimine lecture sale **et** lecture non répétable ; InnoDB **évite aussi les fantômes** (MVCC + *next-key locks*), ce qui le rend plus strict que le standard.
- Attention à la distinction **lecture d'instantané** (`SELECT` simple) vs **lecture courante** (verrouillante / écriture), source possible de **mises à jour perdues**.
- Par défaut (depuis la 11.6.2, donc en 11.8 LTS et 12.3), `innodb_snapshot_isolation = ON` fait de ce niveau une véritable **isolation par instantané** (erreur sur conflit) — voir [§6.9](09-snapshot-isolation.md).
- Compatible avec **tous les formats de binlog**.
