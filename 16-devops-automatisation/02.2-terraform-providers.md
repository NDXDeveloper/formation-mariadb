# 16.2.2 — Terraform : Providers cloud

> Chapitre 16 — DevOps et Automatisation · Mise en pratique de Terraform pour le provisionnement de l'infrastructure MariaDB.

---

## Introduction

Après Ansible, qui configure les machines (§16.2.1), cette sous-section traite de **Terraform**, l'outil qui **fait exister l'infrastructure** sur laquelle MariaDB s'exécutera. Le titre — « Providers cloud » — désigne l'élément central de Terraform pour s'adresser aux fournisseurs : le *provider*. On verra ce qu'est un provider, comment le déclarer, et comment provisionner les ressources d'une plateforme MariaDB selon deux modèles bien distincts.

Comme dans la sous-section précédente, le code HCL présenté ici sert d'**illustration de référence** et reste volontairement minimal.

## La notion de provider

Terraform ne connaît, en lui-même, aucune ressource cloud. Toute sa capacité d'action passe par des **providers** : des greffons qui traduisent les ressources décrites en HCL en appels vers l'API d'une plateforme cible. Le provider `aws` sait créer une instance AWS, le provider `azurerm` une machine Azure, le provider `google` une instance GCP, et il existe des providers pour bien d'autres cibles — OpenStack, VMware vSphere pour les environnements sur site, ou des fournisseurs comme DigitalOcean, Scaleway ou OVH.

Un provider se déclare et se **configure** : on précise quelle version utiliser, et on lui fournit le contexte nécessaire (région, identifiants). Les identifiants ne doivent jamais être inscrits en clair dans le code ; ils proviennent de l'environnement, d'un profil, ou d'un rôle d'identité attaché à la machine qui exécute Terraform.

## HCL : les briques de base

Le langage HCL repose sur quelques constructions. La **ressource** (`resource`) décrit un objet à créer et à maintenir. La **source de données** (`data`) interroge des objets existants sans les gérer. La **variable** (`variable`) paramètre le code, et la **sortie** (`output`) en expose des valeurs vers l'extérieur. Le **provider** (`provider`) configure l'accès à la plateforme, et le **module** (`module`) regroupe un ensemble de ressources réutilisable. À partir de ces briques, on décrit l'infrastructure désirée, que Terraform réalise au travers du flux `plan` puis `apply` déjà présenté en §16.1.

## Déclarer un provider et fixer ses versions

La première étape consiste à déclarer le provider et à **figer sa version**, afin d'éviter qu'une montée de version automatique n'introduise un changement de comportement inattendu.

```hcl
terraform {
  required_version = ">= 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = var.region
}
```

Le contrainte `~> 5.0` autorise les versions correctives de la série 5 mais interdit le passage à une version majeure suivante. Ce code reste valable, à peu de chose près, avec **OpenTofu**, la bifurcation libre de Terraform évoquée en §16.2.

## Deux modèles de déploiement

C'est ici que se présente un choix structurant pour MariaDB sur le cloud : **gérer soi-même** la base sur des instances de calcul, ou s'appuyer sur un **service de base de données managé**. Terraform sait provisionner les deux, mais leurs implications diffèrent profondément.

### Le modèle auto-géré (instances de calcul)

Dans ce modèle, Terraform provisionne les ressources brutes — instances, volumes, réseau, sécurité — et la suite (installation, configuration, mise en cluster) revient à Ansible (§16.2.1). C'est l'approche qui offre le **contrôle le plus complet**, et notamment la liberté d'exécuter exactement la version voulue, ici **MariaDB 12.3 LTS**.

L'illustration suivante provisionne un nombre paramétrable de nœuds, chacun doté d'un **volume de données persistant**. Le détail le plus important n'est pas la création des instances, mais la **protection des volumes** : un volume portant des données ne doit jamais être détruit par mégarde.

```hcl
resource "aws_instance" "mariadb" {
  count                  = var.node_count
  ami                    = var.ami_id
  instance_type          = var.instance_type
  subnet_id              = var.subnet_id
  vpc_security_group_ids = [aws_security_group.mariadb.id]
  tags                   = { Name = "mariadb-${count.index}" }
}

resource "aws_ebs_volume" "data" {
  count             = var.node_count
  availability_zone = var.availability_zone
  size              = var.data_volume_size
  type              = "gp3"

  lifecycle {
    prevent_destroy = true   # garde-fou : empêche la destruction du volume de données
  }
}

resource "aws_volume_attachment" "data" {
  count       = var.node_count
  device_name = "/dev/xvdf"
  volume_id   = aws_ebs_volume.data[count.index].id
  instance_id = aws_instance.mariadb[count.index].id
}
```

Les adresses des instances ainsi créées sont ensuite **exposées en sortie**, où elles serviront de pont vers Ansible.

```hcl
output "mariadb_private_ips" {
  value = aws_instance.mariadb[*].private_ip
}
```

### Le modèle managé (services de base de données)

Dans ce modèle, le fournisseur exploite la base à votre place : il prend en charge la haute disponibilité, les sauvegardes automatiques et les correctifs, et vous n'administrez plus de système d'exploitation. Terraform se contente alors de déclarer le service.

```hcl
resource "aws_db_instance" "mariadb" {
  identifier              = "app-mariadb"
  engine                  = "mariadb"
  engine_version          = var.mariadb_version
  instance_class          = "db.t3.medium"
  allocated_storage       = 50
  storage_encrypted       = true
  multi_az                = true   # haute disponibilité prise en charge par le service
  backup_retention_period = 7
  username                = var.admin_user
  password               = var.admin_password   # depuis un coffre, jamais en clair
  skip_final_snapshot     = false
}
```

Le paysage des services managés est, pour MariaDB **spécifiquement**, plus contrasté qu'on ne l'imagine, et mérite d'être connu avant de s'engager. **AWS** propose une offre managée MariaDB (via son service RDS). En revanche, l'offre **Azure Database for MariaDB a été retirée** (fin de vie en septembre 2025) ; sur Azure, on se tourne donc en pratique vers l'auto-gestion sur machines virtuelles. Quant à **GCP**, son service managé ne prend pas en charge MariaDB nativement (il couvre MySQL, PostgreSQL et SQL Server). À côté des trois grands, l'offre managée spécialisée de référence est **SkySQL**, le cloud de MariaDB Corporation (certains hébergeurs proposent aussi MariaDB en service managé). Le catalogue des fournisseurs évoluant régulièrement, il est prudent de **vérifier l'offre courante** avant de concevoir une architecture.

Une limite est par ailleurs structurelle : les services managés **suivent les versions amont avec un décalage**. La toute dernière LTS — ici la 12.3 — n'y est pas nécessairement disponible immédiatement. Si la contrainte de version est forte, le modèle auto-géré redevient la voie naturelle.

## Le relais vers Ansible

Le pont entre les deux outils, décrit en §16.2 du point de vue de l'enchaînement, se concrétise côté Terraform par les **sorties**. Les adresses exposées par `output` (comme `mariadb_private_ips` ci-dessus) alimentent l'**inventaire dynamique** d'Ansible, qui sait ainsi quelles machines configurer. Terraform fait exister les hôtes ; Ansible les prend en charge à partir de leurs adresses. Cette articulation n'a de sens que pour le modèle auto-géré : un service managé n'expose pas de système d'exploitation à configurer.

## L'état distant et son verrouillage

L'§16.1 a introduit la notion de fichier d'état ; le cloud en fournit les supports adéquats. On stocke l'état dans un **backend distant** — un compartiment d'objets chiffré — plutôt que sur un poste local.

```hcl
terraform {
  backend "s3" {
    bucket         = "monorg-terraform-state"
    key            = "mariadb/prod/terraform.tfstate"
    region         = "eu-west-3"
    dynamodb_table = "terraform-locks"   # verrouillage concurrent
    encrypt        = true
  }
}
```

Le **verrouillage** empêche deux exécutions simultanées de corrompre l'état. Historiquement assuré, pour le backend S3, par une table DynamoDB dédiée, il peut désormais s'appuyer sur un mécanisme de **verrou natif** dans le compartiment lui-même sur les versions récentes de Terraform, ce qui réduit le besoin d'une ressource séparée. On veillera enfin à **séparer l'état par environnement** (un état distinct pour la production, la préproduction, etc.) et à **ne jamais versionner cet état** dans Git : il peut contenir des informations sensibles, à commencer par les identifiants d'un service managé.

## Modules réutilisables

Pour éviter de réécrire le même socle d'une plateforme à l'autre, on encapsule un ensemble de ressources — instances, volumes, réseau, sécurité — dans un **module** paramétrable. Un module « cluster MariaDB » peut ainsi être instancié plusieurs fois avec des valeurs différentes, ou partagé entre équipes. Des modules publics existent dans le registre Terraform, mais une plateforme de base de données justifie souvent un module interne, taillé sur les conventions de l'organisation.

## Auto-géré ou managé : le compromis

Le choix entre les deux modèles est moins une affaire d'outillage que d'arbitrage opérationnel. Le modèle **auto-géré** maximise le contrôle : on choisit la version exacte (la 12.3), on règle finement la configuration, on déploie l'architecture de haute disponibilité que l'on veut (Galera, réplication) — au prix d'une **charge d'exploitation** que l'on assume entièrement. Le modèle **managé** allège cette charge en déléguant haute disponibilité, sauvegardes et correctifs, mais **contraint la version et la configuration**, et n'offre pas toujours la dernière LTS. Le facteur décisif, pour MariaDB, est fréquemment celui de la **disponibilité de la version** : exiger la 12.3 oriente vers l'auto-gestion.

## Spécificités MariaDB 12.3 LTS

Pour le provisionnement, le point d'attention propre à MariaDB tient à la nature stateful de la base : le **volume de données** est la ressource critique, et sa protection (`prevent_destroy`) est une précaution de premier ordre.

Pour un **cluster Galera** en mode auto-géré, le partage des responsabilités entre les deux outils s'illustre parfaitement : c'est Terraform qui doit **ouvrir les ports de communication** entre nœuds dans le groupe de sécurité, tandis que l'installation du paquet `mariadb-server-galera` (§16.2.1) revient à Ansible. Galera utilise, outre le port client 3306, les ports 4567 (trafic de réplication), 4568 (transfert incrémental) et 4444 (transfert d'état) :

```hcl
# Ports à ouvrir entre les nœuds du cluster Galera
ingress {
  from_port = 4567
  to_port   = 4568
  protocol  = "tcp"
  self      = true
}
ingress {
  from_port = 4567
  to_port   = 4567
  protocol  = "udp"
  self      = true
}
ingress {
  from_port = 4444
  to_port   = 4444
  protocol  = "tcp"
  self      = true
}
```

L'attribut `self = true` restreint ces ports aux seules machines du groupe, conformément au principe de moindre exposition. La configuration applicative du cluster relève ensuite du chapitre 14 (§14.2).

## Bonnes pratiques

Quelques règles structurent un usage sain de Terraform. On **fige les versions** de Terraform et des providers pour garantir la reproductibilité. On stocke l'**état dans un backend distant chiffré et verrouillé**, séparé par environnement, et on ne le versionne jamais. On **ne place aucun secret en clair** dans le code ni dans l'état, on s'appuie sur des coffres et sur des identifiants fournis par l'environnement, et l'on marque les variables et sorties sensibles avec `sensitive = true` pour éviter qu'elles n'apparaissent dans les journaux. On **protège les volumes de données** par `prevent_destroy`, et l'on examine toujours le **`plan`** avant d'appliquer, à plus forte raison sur une infrastructure portant des données. Enfin, on **encapsule** les socles répétés dans des modules.
