# 16.3.3 — Volumes et persistance

> Chapitre 16 — DevOps et Automatisation · La résolution de la tension stateless / stateful : où et comment résident les données.

---

## Introduction

L'introduction de la section (§16.3) a posé une tension : le conteneur est **éphémère**, mais une base de données est **gardienne d'un état durable**. Les deux sous-sections précédentes y ont sans cesse renvoyé. C'est ici qu'on la résout. Cette sous-section explique comment Docker stocke les données, quel mécanisme retenir pour MariaDB, et les préoccupations opérationnelles qui accompagnent ce choix — au premier rang desquelles une distinction trop souvent négligée : **persistance n'est pas sauvegarde**.

Les commandes présentées sont des **illustrations de référence**.

## Le problème : la couche modifiable est éphémère

Le système de fichiers d'un conteneur se compose des **couches en lecture seule** de l'image, surmontées d'une fine **couche modifiable** propre au conteneur. Tout ce qui y est écrit — y compris le répertoire de données `/var/lib/mysql` s'il n'est pas externalisé — **disparaît à la suppression du conteneur**.

À ce risque s'ajoute un coût : l'écriture dans cette couche modifiable, fondée sur un mécanisme de copie à l'écriture (*copy-on-write*), est **plus lente** que l'écriture dans un volume. Pour une base de données, dont l'activité est dominée par les écritures sur disque, ces deux raisons — durabilité et performance — convergent vers une même conclusion : **le répertoire de données de MariaDB doit résider hors de la couche modifiable**.

## Les mécanismes de stockage de Docker

Docker propose trois mécanismes, aux usages distincts.

| Mécanisme | Géré par | Persistant | Usage recommandé pour MariaDB |
|---|---|---|---|
| **Volume nommé** | Docker | Oui | Répertoire de données (`/var/lib/mysql`) |
| **Montage lié** (*bind mount*) | L'hôte | Oui | Configuration et scripts d'initialisation (en lecture seule) |
| **tmpfs** | Mémoire | Non | Données réellement jetables (tests, CI) |

Le **volume nommé** est un espace de stockage géré par Docker, indépendant des chemins de l'hôte. Le **montage lié** expose un répertoire de la machine hôte à l'intérieur du conteneur. Le **tmpfs** réside en mémoire et n'est, par nature, jamais persistant.

## Le choix pour MariaDB : un volume nommé pour les données

Pour le répertoire de données, on retient le **volume nommé**. Géré par Docker, il est **portable**, découplé de l'arborescence de l'hôte, offre de **meilleures performances** que la couche modifiable, et évite les écueils d'appartenance évoqués plus bas. On le monte sur le répertoire de données, `/var/lib/mysql`.

```bash
docker volume create mariadb_data

docker run -d --name mariadb \
  -v mariadb_data:/var/lib/mysql \
  -e MARIADB_ROOT_PASSWORD_FILE=/run/secrets/db_root \
  mariadb:12.3
```

La syntaxe `--mount`, plus explicite, est équivalente et souvent préférée dans les scripts :

```bash
docker run -d --name mariadb \
  --mount type=volume,source=mariadb_data,target=/var/lib/mysql \
  mariadb:12.3
```

Ce volume **survit à la destruction et à la recréation** du conteneur : on peut mettre à jour l'étiquette d'image (§16.3.1) et relancer un conteneur neuf sur le même volume, les données demeurant intactes.

## Montages liés : configuration et initialisation

Le montage lié garde toute son utilité, mais pour d'**autres usages** que les données : exposer, depuis le dépôt, les fichiers de configuration (`/etc/mysql/conf.d/`) et les scripts d'initialisation (`/docker-entrypoint-initdb.d/`), généralement en lecture seule.

```bash
docker run -d --name mariadb \
  -v "$(pwd)/conf.d:/etc/mysql/conf.d:ro" \
  -v mariadb_data:/var/lib/mysql \
  mariadb:12.3
```

Employer un montage lié pour les **données** reste possible, mais s'accompagne de deux réserves : des **performances dégradées** sur certaines plateformes (voir plus bas) et un **risque d'appartenance** de fichiers, traité à la section suivante. En règle générale, on préfère donc le volume nommé pour les données et le montage lié pour ce qui provient du dépôt.

## tmpfs : données réellement jetables

Le montage `tmpfs` stocke les données en mémoire : rapides mais **effacées à l'arrêt** du conteneur. Son seul usage légitime concerne les données dont la perte est sans conséquence — typiquement une base de **test en intégration continue**, où la vitesse prime et où la persistance n'est pas recherchée. On ne l'emploie **jamais** pour des données que l'on tient à conserver.

## Gérer le cycle de vie des volumes

Quelques commandes suffisent à administrer les volumes.

```bash
docker volume ls                  # lister les volumes  
docker volume inspect mariadb_data  # localiser et inspecter un volume  
docker volume rm mariadb_data     # SUPPRESSION DÉFINITIVE des données  
docker volume prune               # supprimer les volumes inutilisés (avec prudence)  
```

Sous Compose (§16.3.2), les volumes nommés se déclarent au niveau supérieur, et l'on retrouve la distinction décisive : `docker compose down` **conserve** le volume, tandis que `docker compose down -v` le **supprime** — et réinitialise donc la base. Les commandes `docker volume rm` et `prune` étant **irréversibles**, on les manie avec la même prudence que toute suppression de données.

## Permissions et appartenance

Une difficulté classique mérite d'être anticipée. Le serveur s'exécute, dans l'image officielle, sous l'utilisateur `mysql` (d'identifiant `999`). Avec un **volume nommé**, Docker établit correctement l'appartenance lors de la première utilisation, et le problème ne se pose pas. Avec un **montage lié** en revanche, le répertoire de l'hôte doit être **accessible en écriture par cet utilisateur** : à défaut, l'initialisation échoue sur des erreurs de permission. C'est l'une des principales raisons pour lesquelles on **préfère le volume nommé** pour les données ; lorsqu'un montage lié est imposé, il faut veiller à en régler l'appartenance en conséquence.

## Persistance n'est pas sauvegarde

C'est le point capital de cette sous-section. Un volume protège les données contre la **suppression du conteneur**, mais **nullement** contre une corruption, un `DROP` accidentel, une panne de disque ou la suppression du volume lui-même. **Un volume n'est pas une sauvegarde.**

Une base conteneurisée requiert donc une véritable stratégie de sauvegarde, dont les modalités relèvent du chapitre 12, mais dont voici les formes appliquées au conteneur.

La **sauvegarde logique** exécute `mariadb-dump` contre le conteneur :

```bash
docker exec mariadb \
  mariadb-dump -uroot -p"$ROOT_PW" --single-transaction app > app.sql
```

La **sauvegarde physique** s'appuie sur Mariabackup, exécuté dans un conteneur éphémère **partageant le volume de données** grâce à `--volumes-from` (§12.3). Un point pratique mérite attention : Mariabackup doit non seulement **lire les fichiers de données** (fournis par `--volumes-from`), mais aussi **se connecter au serveur en cours d'exécution** pour coordonner la sauvegarde. Or l'image n'expose en volume que `/var/lib/mysql`, et non le socket : on fournit donc la connexion en **partageant le réseau** du conteneur (`--network "container:mariadb"`, puis `--host=127.0.0.1`).

```bash
docker run --rm \
  --volumes-from mariadb \
  --network "container:mariadb" \
  -v "$(pwd)/backup:/backup" \
  mariadb:12.3 \
  mariadb-backup --backup --target-dir=/backup \
    --host=127.0.0.1 --user=root --password="$ROOT_PW"
```

Plus simplement, on peut aussi lancer la sauvegarde **à l'intérieur** du conteneur via `docker exec` (le socket local est alors directement disponible), puis récupérer le résultat depuis un volume monté.

Enfin, un **instantané** (*snapshot*) du volume ou de son stockage sous-jacent est envisageable, mais sur une base en activité il exige une **cohérence** des données — assurée par un verrouillage approprié ou par la reprise sur incident d'InnoDB à partir d'un instantané du système de fichiers (§12.3, §12.5). Dans tous les cas, on retiendra que la persistance et la sauvegarde répondent à des risques différents et **ne se substituent pas l'une à l'autre**.

## Performance du stockage

Le choix du stockage a des conséquences directes sur les performances. La couche modifiable étant lente, le recours à un **volume** est déjà un gain. Une particularité mérite attention : sur **Docker Desktop** (macOS, Windows), les montages liés traversent une frontière de machine virtuelle, ce qui les rend **lents pour les entrées-sorties intensives** d'une base — les volumes nommés y sont nettement préférables. Plus largement, le **stockage qui sous-tend** le volume (disque local NVMe contre stockage réseau) conditionne les performances réelles ; ces considérations rejoignent le chapitre 15, consacré au réglage des entrées-sorties.

## Intégrité et durabilité des écritures

La durabilité d'InnoDB — journal de reprise, double écriture — protège des arrêts brutaux, **à condition que le stockage honore réellement les écritures**. Deux règles en découlent. On ne place **jamais** sur tmpfs des données que l'on souhaite conserver. Et l'on s'assure que le volume **persiste fidèlement** les écritures, en évitant un stockage qui les mettrait en tampon au risque de les perdre. Les arbitrages de durabilité (par exemple `innodb_flush_log_at_trx_commit`) relèvent des chapitres 6 et 15 ; au niveau de la persistance, l'essentiel est que le support sous-jacent ne trahisse pas les garanties du moteur.

## Du volume Docker au stockage Kubernetes

Ce qui a été dit ici se transpose à l'orchestration, en changeant d'abstraction mais non de principe. Sur **Kubernetes**, la persistance des données est assurée par les **PersistentVolumes** et les **StorageClasses** (§16.4.2), tandis que les **StatefulSets** garantissent à chaque réplique un volume **stable et propre** (§16.4.1). La préoccupation reste identique — externaliser l'état hors d'un conteneur éphémère — et la maîtrise des volumes Docker acquise ici constitue le socle de la section suivante.

## Bonnes pratiques

En synthèse : on monte un **volume nommé** sur `/var/lib/mysql` pour les données, et l'on réserve les **montages liés en lecture seule** à la configuration et à l'initialisation. On ne stocke **jamais** les données dans la couche modifiable, ni sur tmpfs hors contexte jetable. On **distingue persistance et sauvegarde**, en mettant en place une stratégie de sauvegarde réelle (chapitre 12). On **privilégie le volume nommé** pour éviter les écueils d'appartenance, on tient compte du **stockage sous-jacent** pour les performances (chapitre 15), et l'on s'assure que ce stockage **garantit la durabilité** des écritures. Enfin, `down -v`, `volume rm` et `prune` ne s'emploient que de façon **délibérée**, ces opérations étant irréversibles.

## Spécificités MariaDB 12.3 LTS

La persistance est, pour l'essentiel, **indépendante de la version** : le répertoire de données reste `/var/lib/mysql`, et c'est le seul emplacement à persister. Une évolution de la 12.3 simplifie même le tableau : le **journal binaire désormais intégré à InnoDB** (§11.5.4) est consolidé avec les données du moteur plutôt que géré comme un ensemble de fichiers distincts — de sorte que persister le répertoire de données couvre l'ensemble, sans emplacement supplémentaire à monter séparément.
