# 16.3.1 — Images officielles MariaDB (impact du packaging Galera 12.3)

> Chapitre 16 — DevOps et Automatisation · L'image officielle, sa configuration, et l'incidence du changement de packaging de Galera en 12.3.

---

## Introduction

Conteneuriser MariaDB commence par le choix d'une **image**. Dans l'immense majorité des cas, on s'appuie sur l'**image officielle** plutôt que sur une image construite de zéro. Cette sous-section présente cette image, la façon de la configurer et de l'initialiser, puis traite son sujet central — annoncé dans le titre — : l'**impact du changement de packaging de Galera** intervenu dans la série 12.x.

Les commandes et fichiers présentés sont des **illustrations de référence**, réduites à l'essentiel.

## L'image officielle MariaDB

L'image officielle est publiée sous le nom `mariadb` sur le registre public Docker Hub. Elle fait partie du programme des **images officielles Docker** et est maintenue en lien avec la **Fondation MariaDB**, ce qui en fait la référence pour exécuter MariaDB en conteneur. Elle est distincte de l'image `mysql` : bien que les deux partagent des conventions héritées (variables d'environnement, scripts d'initialisation), elles correspondent à des serveurs différents, et l'on choisit `mariadb` pour exécuter MariaDB.

## Étiquettes et verrouillage de version

Une image se référence par une **étiquette** (*tag*), et le choix de cette étiquette est une décision de fond pour une base de données. Plusieurs formes coexistent : une version précise (`mariadb:12.3.2`), une version mineure (`mariadb:12.3`), un alias pointant vers la dernière LTS (`mariadb:lts`), ou l'étiquette mouvante `mariadb:latest`.

Pour un composant stateful, on **proscrit `latest`** : une montée de version involontaire d'une base de données peut être lourde de conséquences. La bonne pratique consiste à **figer la version** — au minimum sur la mineure (`12.3`), idéalement sur la version précise (`12.3.2`) en production — afin de garantir la reproductibilité et de maîtriser le moment d'une mise à niveau. C'est précisément ce verrouillage qui fait de l'étiquette d'image l'un des grands atouts évoqués en §16.3 : exécuter **exactement** la 12.3 voulue.

## Exécuter l'image

Au premier démarrage, l'image a besoin de savoir comment initialiser le compte `root`. La forme la plus simple fournit un mot de passe par variable d'environnement, expose le port client et laisse l'image écouter sur le port 3306 :

```bash
docker run --name mariadb-dev \
  -e MARIADB_ROOT_PASSWORD="$ROOT_PW" \   # illustratif — voir la gestion des secrets plus bas
  -e MARIADB_DATABASE=app \
  -p 3306:3306 \
  mariadb:12.3
```

Les données résident dans le répertoire `/var/lib/mysql` du conteneur. **Sans volume monté sur ce chemin, elles disparaîtront avec le conteneur** : c'est le point décisif traité en détail dans la sous-section consacrée aux volumes (§16.3.3). On le mentionne ici pour mémoire, car il conditionne tout usage sérieux de l'image.

## Configurer l'image

### Variables d'environnement

L'image se paramètre principalement par variables d'environnement, dont les plus courantes sont les suivantes.

- `MARIADB_ROOT_PASSWORD` — mot de passe du compte `root` (initialisation).
- `MARIADB_RANDOM_ROOT_PASSWORD` — génère un mot de passe aléatoire, utile pour une instance éphémère.
- `MARIADB_ALLOW_EMPTY_ROOT_PASSWORD` — autorise un `root` sans mot de passe ; à éviter hors contexte jetable.
- `MARIADB_DATABASE` — crée une base au premier démarrage.
- `MARIADB_USER` / `MARIADB_PASSWORD` — crée un compte applicatif et son mot de passe.
- `MARIADB_AUTO_UPGRADE` — déclenche `mariadb-upgrade` automatiquement lors d'un changement de version.

Chacune de ces variables sensibles dispose d'une variante suffixée `_FILE` (voir la gestion des secrets). Les anciens noms préfixés `MYSQL_*` restent par ailleurs reconnus, par compatibilité.

### Fichiers de configuration

Au-delà des variables, on injecte une configuration fine en **montant des fichiers `.cnf`** dans le répertoire d'inclusion de l'image, `/etc/mysql/conf.d/`. Cette approche prolonge naturellement la configuration modulaire de MariaDB (§11.1.3) : chaque préoccupation fait l'objet d'un fichier, déposé sans réécrire la configuration de base.

```bash
docker run ... \
  -v "$(pwd)/conf.d:/etc/mysql/conf.d:ro" \
  mariadb:12.3
```

## Initialisation au premier démarrage

L'image offre un mécanisme d'amorçage : tout fichier `.sql`, `.sql.gz`, `.sql.xz`, `.sql.zst` ou `.sh` placé dans le répertoire `/docker-entrypoint-initdb.d/` est **exécuté au premier démarrage**, c'est-à-dire lorsque le répertoire de données est vide.

```bash
docker run ... \
  -v "$(pwd)/init:/docker-entrypoint-initdb.d:ro" \
  mariadb:12.3
```

Le comportement essentiel à retenir est que ces scripts ne s'exécutent **qu'à l'initialisation**, jamais lors des démarrages ultérieurs sur des données existantes. Ce mécanisme convient donc à un **amorçage** (schéma initial, données de référence, comptes additionnels), mais **pas à la gestion continue du schéma** : faire évoluer une base déjà en service relève des outils de migration versionnée (§16.8), et non de ce répertoire.

## Gestion des secrets

Transmettre un mot de passe par variable d'environnement en clair est commode mais peu sûr : la valeur apparaît dans l'inspection du conteneur et dans l'historique des commandes. L'image propose donc, pour chaque variable sensible, une **variante `_FILE`** qui lit la valeur depuis un fichier — typiquement un secret monté par l'orchestrateur.

```bash
docker run ... \
  -e MARIADB_ROOT_PASSWORD_FILE=/run/secrets/db_root \
  mariadb:12.3
```

Deux règles en découlent : **ne jamais inscrire d'identifiant en clair dans une image** (il serait visible par quiconque accède à l'image), et **privilégier la variante `_FILE`** dès qu'un gestionnaire de secrets est disponible. Pour une instance réellement jetable, `MARIADB_RANDOM_ROOT_PASSWORD` évite même d'avoir à gérer un mot de passe.

## Construire une image personnalisée

Lorsqu'on souhaite figer une configuration ou des éléments propres à un projet, on **étend l'image officielle** par un Dockerfile :

```dockerfile
FROM mariadb:12.3  
COPY conf.d/ /etc/mysql/conf.d/  
COPY init/  /docker-entrypoint-initdb.d/  
```

Ce procédé reste un arbitrage. **Embarquer** la configuration dans une image la rend autonome, mais impose une reconstruction à chaque changement ; **monter** la configuration au démarrage garde l'image générique et évite ces reconstructions. En règle générale, on réserve la construction d'image personnalisée aux éléments réellement stables, et l'on monte au runtime ce qui varie d'un environnement à l'autre.

## L'impact du packaging de Galera en 12.3

C'est ici le cœur de la sous-section. La série 12.x a modifié le packaging de **Galera** : le fournisseur de réplication (la bibliothèque *wsrep*) n'est plus une **dépendance intégrée** au paquet serveur, mais relève désormais d'un **paquet distinct**, `mariadb-server-galera` (§14.2.5). Ce changement, transparent pour une installation classique sur machine, a une incidence directe sur les images de conteneurs, qui sont précisément **construites à partir de paquets**.

La conséquence pratique se formule simplement, selon le type de déploiement.

Pour une **instance autonome** (un nœud unique sans cluster), **rien ne change** : Galera n'est pas nécessaire, et l'image s'utilise comme décrit plus haut.

Pour un **déploiement en cluster Galera**, en revanche, on ne peut plus **présumer** que le fournisseur Galera est implicitement présent du seul fait que le serveur l'est. Le support Galera étant désormais un paquet à part, un déploiement en cluster doit **s'assurer de sa présence**. Si l'on construit une image personnalisée à cette fin, cela se traduit par l'ajout explicite du paquet :

```dockerfile
FROM mariadb:12.3
# Déploiement en cluster Galera (MariaDB 12.3) : ajout explicite du fournisseur
RUN apt-get update \
 && apt-get install -y mariadb-server-galera \
 && rm -rf /var/lib/apt/lists/*
```

Un point de méthode s'impose ici. Le fait que l'image officielle a **longtemps permis** d'exécuter des clusters Galera ne garantit pas, à lui seul, le contenu d'une étiquette donnée en 12.3 : un changement de packaging est précisément ce qui peut modifier ce que contient une image de base. Il est donc prudent de **vérifier, dans la documentation de l'image et pour l'étiquette retenue**, si le paquet Galera y est inclus par défaut ou s'il doit être ajouté. Quelle que soit la réponse, la règle de conception reste la même : un déploiement clusterisé doit explicitement **garantir la présence de `mariadb-server-galera`**.

> **État vérifié pour la 12.3.** À la suite du revirement communautaire évoqué au chapitre 14 (le découplage du paquet ayant suscité de vives réactions), l'image officielle `mariadb:12.3` (12.3.2) **embarque toujours** `mariadb-server-galera` ainsi que `galera-4` : la variable `wsrep_provider` y pointe d'emblée vers `/usr/lib/galera/libgalera_smm.so`. En **étendant l'image officielle**, le fournisseur Galera est donc déjà disponible et l'ajout `apt-get install mariadb-server-galera` ci-dessus est redondant ; il n'est réellement nécessaire que si l'on construit une image **minimale depuis zéro**. La règle de prudence (vérifier par étiquette) demeure néanmoins valable, le packaging pouvant évoluer d'une version à l'autre.

Cette incidence ne se limite pas aux images construites à la main. Elle se répercute sur les **clusters de développement définis avec Compose** (§16.3.2) et sur la façon dont un **opérateur** provisionne Galera sur Kubernetes (§16.5) : dans tous les cas, le fournisseur Galera doit être présent dans l'image employée.

## Nœud unique ou cluster Galera depuis l'image

En pratique, l'usage de l'image se répartit selon deux situations. Pour un **nœud unique**, l'image officielle suffit telle quelle, en développement comme pour des besoins simples. Pour un **cluster Galera**, deux conditions s'ajoutent : disposer d'une image contenant le fournisseur Galera (point ci-dessus) et fournir la **configuration *wsrep*** appropriée. Il faut toutefois rappeler la mise en garde de §16.3 : Docker seul n'est pas l'outil d'un cluster Galera de **production** — qui suppose orchestration, amorçage ordonné et auto-réparation (chapitre 14, §16.5). Le scénario réaliste d'un cluster Galera « sur Docker » se limite donc le plus souvent à un **cluster de développement** monté avec Compose (§16.3.2).

## Bonnes pratiques

Pour l'usage de l'image officielle, quelques règles s'imposent. On **fige l'étiquette de version** et l'on bannit `latest`, à plus forte raison en production. On **gère les secrets par les variantes `_FILE`** plutôt qu'en clair, et l'on n'inscrit jamais d'identifiant dans une image. On **monte la configuration** dans `/etc/mysql/conf.d/` quand elle varie, en ne l'embarquant dans une image que lorsqu'elle est stable. On réserve le répertoire d'**initialisation** au seul amorçage, en confiant l'évolution du schéma aux migrations (§16.8). On veille à la **persistance** des données via un volume (§16.3.3). Enfin, pour tout **déploiement en cluster**, on s'assure de la présence du paquet `mariadb-server-galera` et l'on confie l'orchestration à un outil adapté plutôt qu'à un `docker run` isolé.
