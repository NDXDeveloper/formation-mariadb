# 16.2.1 — Ansible : Playbooks MariaDB

> Chapitre 16 — DevOps et Automatisation · Mise en pratique d'Ansible pour la configuration de MariaDB.

---

## Introduction

La section précédente a situé Ansible dans le paysage de l'IaC : il occupe la **couche système et configuration**, là où Terraform occupe la couche infrastructure. Cette sous-section descend au niveau des **playbooks** : comment Ansible installe MariaDB, dépose sa configuration, gère son service et administre ses comptes, de manière idempotente et reproductible.

Les exemples de code qui suivent sont des **illustrations de référence**, conçues pour montrer la forme idiomatique d'un playbook MariaDB. Ils sont volontairement réduits à l'essentiel pour rester lisibles.

## Rappel des notions Ansible utiles

Ansible décrit l'état voulu d'un parc de machines au moyen de quelques objets. L'**inventaire** liste les hôtes et les organise en groupes ; pour une plateforme MariaDB, on distingue typiquement un primaire, des réplicas et, le cas échéant, les nœuds d'un cluster :

```ini
[mariadb_primary]
db01.example.internal

[mariadb_replicas]
db02.example.internal  
db03.example.internal  

[galera_cluster]
gdb01.example.internal  
gdb02.example.internal  
gdb03.example.internal  
```

Un **playbook** associe des groupes d'hôtes à une suite de **tâches**, chaque tâche faisant appel à un **module**. Les **variables** (rangées dans `group_vars/` et `host_vars/`) paramètrent ces tâches sans modifier leur logique. Les **handlers** sont des tâches déclenchées à la demande, typiquement pour redémarrer un service après un changement de configuration. Enfin, **Ansible Vault** chiffre les valeurs sensibles, comme les mots de passe. Pour structurer un ensemble cohérent, on regroupe tout cela dans un **rôle** réutilisable.

## La collection `community.mysql`

Ansible ne gère pas MariaDB au travers de modules intégrés, mais d'une **collection** dédiée, `community.mysql`, qui sert indifféremment MySQL et MariaDB. Ses principaux modules sont les suivants.

| Module | Rôle |
|---|---|
| `community.mysql.mysql_db` | Créer / supprimer des bases, importer / exporter des données |
| `community.mysql.mysql_user` | Gérer les comptes, mots de passe et privilèges |
| `community.mysql.mysql_role` | Gérer les rôles (chapitre 10) |
| `community.mysql.mysql_variables` | Lire et ajuster les variables système à chaud |
| `community.mysql.mysql_replication` | Configurer et piloter la réplication |
| `community.mysql.mysql_query` | Exécuter des requêtes SQL arbitraires |
| `community.mysql.mysql_info` | Collecter des informations sur l'instance |

Ces modules s'exécutent sur la machine cible et requièrent qu'un **connecteur Python** y soit présent — `PyMySQL` (recommandé) ou `mysqlclient`. C'est une dépendance pratique à ne pas oublier : sans elle, les tâches échouent avant même de se connecter.

## Structurer un rôle MariaDB

Un rôle Ansible suit une arborescence conventionnelle. Pour MariaDB, on l'organise par préoccupation :

```text
roles/mariadb/
├── defaults/main.yml      # valeurs par défaut (surchargées par l'inventaire)
├── vars/main.yml          # variables internes au rôle
├── tasks/
│   ├── main.yml           # point d'entrée, inclut les autres fichiers
│   ├── install.yml        # dépôt + paquets
│   ├── configure.yml      # fichiers de configuration
│   ├── service.yml        # gestion du service
│   ├── secure.yml         # durcissement des accès
│   ├── databases.yml      # bases
│   └── users.yml          # comptes et privilèges
├── templates/
│   └── 99-server.cnf.j2   # configuration générée
└── handlers/main.yml      # redémarrage du service
```

Ce découpage sépare clairement la **logique** (les tâches) des **données** (les variables et templates), ce qui facilite la relecture et la réutilisation entre environnements.

## Installer MariaDB

La 12.3 n'étant en général pas la version fournie par les dépôts des distributions, on ajoute d'ordinaire au préalable le **dépôt officiel MariaDB** (via le script `mariadb_repo_setup` ou une déclaration de dépôt explicite), puis on installe les paquets. Le point d'attention propre à la 12.3 apparaît ici : le support **Galera** relève désormais d'un paquet distinct, à n'installer que sur les nœuds de cluster.

```yaml
# tasks/install.yml
- name: Installer le connecteur Python requis par les modules MySQL
  ansible.builtin.package:
    name: python3-pymysql
    state: present

- name: Installer les paquets MariaDB
  ansible.builtin.package:
    name:
      - mariadb-server
      - mariadb-client
    state: present

- name: Installer le support Galera (nœuds de cluster uniquement — MariaDB 12.3)
  ansible.builtin.package:
    name: mariadb-server-galera
    state: present
  when: "'galera_cluster' in group_names"
```

Le module `package` est volontairement générique ; sur une distribution donnée, on pourra lui préférer `apt` ou `dnf` selon les besoins. Le `when` conditionne l'installation du paquet Galera à l'appartenance de l'hôte au groupe `galera_cluster`.

## Gérer la configuration

La configuration modulaire de MariaDB (§11.1.3) se marie idéalement avec Ansible : plutôt que de réécrire un `my.cnf` monolithique, on dépose un **fichier dédié** dans le répertoire d'inclusion, généré à partir d'un template **Jinja2** alimenté par les variables.

```jinja
{# templates/99-server.cnf.j2 #}
[mariadbd]
bind-address            = {{ mariadb_bind_address }}  
max_connections         = {{ mariadb_max_connections }}  
innodb_buffer_pool_size = {{ mariadb_buffer_pool_size }}  
character-set-server    = utf8mb4  
```

```yaml
# tasks/configure.yml
- name: Déployer la configuration serveur
  ansible.builtin.template:
    src: 99-server.cnf.j2
    dest: /etc/mysql/mariadb.conf.d/99-server.cnf
    owner: root
    group: root
    mode: "0644"
  notify: Redémarrer MariaDB
```

On notera que le groupe `[mariadbd]` est celui propre à MariaDB, et que le chemin du répertoire d'inclusion varie selon la distribution (`/etc/mysql/mariadb.conf.d/` sur Debian, `/etc/my.cnf.d/` sur les distributions de la famille RHEL). Le `notify` déclenche le handler de redémarrage **uniquement si** le fichier change réellement — illustration concrète de l'idempotence.

## Gérer le service

Le service est géré de façon déclarative, et le handler associé n'agit qu'à la demande des tâches qui le notifient.

```yaml
# tasks/service.yml
- name: Activer et démarrer MariaDB
  ansible.builtin.systemd:
    name: mariadb
    enabled: true
    state: started
```

```yaml
# handlers/main.yml
- name: Redémarrer MariaDB
  ansible.builtin.systemd:
    name: mariadb
    state: restarted
```

## Durcissement des accès et problème d'amorçage

Le durcissement reproduit, sous forme idempotente, ce que ferait l'assistant interactif de sécurisation : suppression des comptes anonymes, suppression de la base `test`, et maîtrise du compte `root`. Mais il se heurte à un **problème d'amorçage** (*chicken-and-egg*) : pour administrer la base, Ansible doit s'y connecter, or les identifiants ne sont pas encore établis lors de la toute première exécution.

Deux stratégies répondent à ce problème. La plus simple et la plus recommandée consiste à **conserver l'authentification par socket Unix** pour `root` : sur les paquets récents, `root` s'authentifie via le plugin `unix_socket`, et Ansible se connecte alors en passant par la socket, sans aucun mot de passe à gérer. C'est ce que reflète le paramètre `login_unix_socket` dans les tâches qui suivent.

```yaml
# tasks/secure.yml
- name: Supprimer les comptes anonymes
  community.mysql.mysql_user:
    name: ""
    host_all: true
    state: absent
    login_unix_socket: /run/mysqld/mysqld.sock

- name: Supprimer la base de test
  community.mysql.mysql_db:
    name: test
    state: absent
    login_unix_socket: /run/mysqld/mysqld.sock
```

La seconde stratégie **attribue un mot de passe à `root`** : on le définit lors du premier passage (toujours via la socket), on le range dans Ansible Vault, puis on le fournit aux exécutions suivantes — par exemple au moyen d'un fichier `~/.my.cnf` déposé sur la machine, ou via les paramètres `login_user`/`login_password`. Cette approche est plus lourde et n'apporte d'avantage que dans des contextes spécifiques ; à défaut de besoin particulier, l'authentification par socket reste préférable.

> Pour éviter de répéter `login_unix_socket` dans chaque tâche, on peut le factoriser à l'échelle du play grâce à `module_defaults` :
> ```yaml
> module_defaults:
>   community.mysql.mysql_db:
>     login_unix_socket: /run/mysqld/mysqld.sock
>   community.mysql.mysql_user:
>     login_unix_socket: /run/mysqld/mysqld.sock
> ```

## Bases de données, comptes et privilèges

La création des bases et des comptes s'appuie sur `mysql_db` et `mysql_user`. Le mot de passe applicatif provient de Vault, et le privilège est exprimé au plus juste, conformément au principe de moindre privilège (chapitre 10).

```yaml
# tasks/databases.yml
- name: Créer la base applicative
  community.mysql.mysql_db:
    name: "{{ app_db_name }}"
    state: present
    login_unix_socket: /run/mysqld/mysqld.sock
```

```yaml
# tasks/users.yml
- name: Créer l'utilisateur applicatif et ses privilèges
  community.mysql.mysql_user:
    name: "{{ app_db_user }}"
    password: "{{ app_db_password }}"   # chiffré dans Ansible Vault
    host: "%"
    priv: "{{ app_db_name }}.*:SELECT,INSERT,UPDATE,DELETE"
    plugin: ed25519                     # authentification moderne (§10.5.2)
    state: present
    login_unix_socket: /run/mysqld/mysqld.sock
```

Le paramètre `plugin` permet de choisir le mécanisme d'authentification — par exemple `ed25519` pour une authentification moderne, ou `caching_sha2_password` (§10.5.5) dans un contexte de compatibilité MySQL 8. Pour la gestion des rôles, le module `mysql_role` joue un rôle analogue.

## Ajustement des variables d'exécution

Le module `mysql_variables` ajuste une variable système à chaud :

```yaml
- name: Augmenter max_connections à chaud
  community.mysql.mysql_variables:
    variable: max_connections
    value: "500"
    login_unix_socket: /run/mysqld/mysqld.sock
```

Une précision importante, propre à MariaDB : cet ajustement modifie la variable **globale en mémoire**, mais ne la **persiste pas**. Contrairement à MySQL 8, **MariaDB ne dispose pas de `SET PERSIST`** ; un changement de variable effectué à chaud est donc perdu au redémarrage. Pour une modification durable, il faut **passer par le fichier de configuration** (section précédente) ; `mysql_variables` ne convient que pour un ajustement temporaire ou un réglage que l'on reportera par ailleurs dans la configuration.

## Orchestration multi-nœuds

La force d'Ansible pour MariaDB ne se limite pas à la configuration d'un nœud isolé : sa capacité à **ordonner les opérations sur un groupe** le rend précieux pour les déploiements en cluster, point soulevé en §16.2.

Pour un cluster **Galera**, l'amorçage demande de démarrer un premier nœud en mode initialisation, puis d'y rattacher les autres dans un second temps. Ansible exprime cet ordonnancement avec des mots-clés tels que `serial` (traiter les hôtes par vagues), `run_once` (n'exécuter une tâche que sur un seul hôte du groupe) et `delegate_to` (exécuter une tâche sur un hôte précis). Les détails du démarrage d'un cluster relèvent du chapitre 14 (§14.2).

Pour la **réplication**, on configure le primaire, on en relève la position ou le GTID, puis on configure chaque réplica — le module `mysql_replication` et la délégation de tâches vers le primaire permettant de coordonner l'ensemble. Ce sujet renvoie au chapitre 13.

## Idempotence, mode simulation et tests

L'idempotence est la propriété qui rend un playbook rejouable sans risque ; encore faut-il la vérifier. Les options `--check` (simulation sans modification) et `--diff` (affichage des changements prévus) permettent d'auditer un playbook avant de l'appliquer.

Tous les modules ne sont cependant pas idempotents par nature. En particulier, `mysql_query` exécute la requête qu'on lui fournit **à chaque passage** : il revient à l'auteur du playbook de rendre la requête idempotente (par exemple via `CREATE ... IF NOT EXISTS`) ou de la conditionner. À l'inverse, `mysql_db` et `mysql_user` gèrent eux-mêmes la comparaison entre état désiré et état réel.

Pour valider un rôle de façon systématique, l'outil **Molecule** permet de le tester dans des environnements jetables (souvent des conteneurs), en vérifiant notamment qu'une seconde exécution n'entraîne aucun changement — la preuve concrète de l'idempotence.

## Bonnes pratiques

L'expérience se résume en quelques règles. On structure le code en **rôles réutilisables** et on **sépare les variables de la logique**. On **chiffre systématiquement les secrets** avec Vault, sans jamais les inscrire en clair. On **privilégie les modules dédiés** (`mysql_db`, `mysql_user`…) plutôt que des commandes `shell` ou `command`, afin de bénéficier de l'idempotence. On **fige les versions** de la collection `community.mysql` et des paquets pour garantir la reproductibilité. Enfin, on emploie les **handlers** pour ne redémarrer le service qu'en cas de changement effectif, et l'on conserve à l'esprit que toute modification durable de configuration doit passer par les fichiers, MariaDB ne persistant pas les changements à chaud.

## Spécificités MariaDB 12.3 LTS

Deux points méritent d'être retenus pour la 12.3. D'une part, l'installation d'un cluster Galera implique d'**ajouter explicitement** le paquet `mariadb-server-galera`, comme le montre la tâche d'installation, là où une version antérieure l'aurait embarqué d'office. D'autre part, le choix du **plugin d'authentification** des comptes (`ed25519`, `unix_socket` pour `root`, ou `caching_sha2_password` en contexte de compatibilité MySQL) se déclare directement via le paramètre `plugin` du module `mysql_user`, ce qui permet d'aligner la politique d'authentification du chapitre 10 sur l'automatisation.
