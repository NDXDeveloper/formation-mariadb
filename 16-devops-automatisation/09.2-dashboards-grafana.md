# 16.9.2 — Dashboards Grafana

> **Prérequis de cette section** : la section [16.9.1 (mysqld_exporter)](09.1-mysqld-exporter.md) est opérationnelle, les métriques MariaDB sont visibles dans Prometheus (`mysql_up` renvoie `1`), et un serveur Grafana est installé — via le paquet officiel, le conteneur `grafana/grafana`, ou Grafana Cloud.

Une fois les métriques de MariaDB collectées par Prometheus, elles ne sont encore que des séries temporelles brutes, peu lisibles en l'état. **Grafana** est la couche qui transforme ces séries en tableaux de bord exploitables : graphes de tendance, jauges de saturation, indicateurs d'état et alertes visuelles. C'est l'outil qui répond, en un coup d'œil, à la question opérationnelle de base : *« Est-ce que mon instance va bien, et si non, depuis quand ? »*

## Positionnement dans la chaîne d'observabilité

Il est essentiel de comprendre que **Grafana ne stocke aucune donnée**. Il s'agit d'une couche de présentation qui interroge Prometheus à la demande, en **PromQL**, à chaque rafraîchissement d'un panneau ou d'une fenêtre temporelle. La répartition des responsabilités est donc claire :

| Composant | Rôle | Détient les données ? |
|-----------|------|------------------------|
| `mysqld_exporter` | Collecte et exposition des métriques MariaDB | Non (instantané) |
| Prometheus | Scraping périodique et stockage time-series | Oui (TSDB) |
| Grafana | Requêtage, visualisation, alerting | Non (interroge Prometheus) |

Cette séparation a une conséquence pratique : la profondeur d'historique consultable dans Grafana dépend entièrement de la **rétention configurée dans Prometheus**, pas de Grafana lui-même. Si Prometheus conserve 15 jours de données, aucun dashboard ne pourra remonter au-delà.

## Connecter Grafana à Prometheus (data source)

Avant tout dashboard, Grafana doit savoir où trouver les données. On déclare Prometheus comme **source de données** (*data source*).

### Configuration via l'interface

Dans `Connections → Data sources → Add data source → Prometheus`, le seul champ réellement obligatoire est l'URL du serveur Prometheus, par exemple `http://prometheus:9090` (nom de service en environnement conteneurisé) ou `http://localhost:9090`. Le mode d'accès recommandé est `Server (proxy)` : c'est le backend Grafana qui contacte Prometheus, ce qui évite d'exposer Prometheus au navigateur de l'utilisateur.

### Configuration as Code (recommandée en production)

En contexte DevOps, on évite la configuration manuelle au profit du *provisioning* déclaratif. Grafana lit automatiquement les fichiers YAML déposés dans `/etc/grafana/provisioning/datasources/` au démarrage :

```yaml
# /etc/grafana/provisioning/datasources/prometheus.yaml
apiVersion: 1  
datasources:  
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    jsonData:
      # Doit correspondre au scrape_interval de Prometheus.
      # Évite les "trous" dans les graphes calculés avec rate().
      timeInterval: "15s"
```

Le paramètre `timeInterval` est souvent négligé alors qu'il est important : il indique à Grafana la résolution réelle des données, ce qui garantit la cohérence des fonctions comme `rate()` et évite les graphes en dents de scie.

## Importer un dashboard existant

Le moyen le plus rapide d'obtenir un tableau de bord complet est d'importer un dashboard communautaire déjà conçu pour `mysqld_exporter`. La procédure est identique pour tous : `Dashboards → New → Import`, saisir l'identifiant, puis sélectionner la source de données Prometheus créée précédemment.

Les références incontournables pour MariaDB/MySQL sont :

| ID Grafana | Nom | Source |
|------------|-----|--------|
| **7362** | MySQL Overview | Percona (projet PMM) |
| 11323 | MySQL | Communauté `mysqld_exporter` |

Le dashboard **7362** (Percona « MySQL Overview ») est de loin le plus utilisé : il fournit d'emblée connexions, débit de requêtes, activité InnoDB et réseau. L'ensemble des dashboards Percona est par ailleurs disponible dans le dépôt `github.com/percona/grafana-dashboards`, et peut être cloné puis copié dans `/var/lib/grafana/dashboards` pour un import en masse.

**Point de vigilance — compatibilité des versions.** Un dashboard importé suppose une version donnée de l'exporter et un nommage précis des métriques. Un dashboard ancien peut afficher des panneaux vides si certaines métriques ont été renommées ou si un collecteur n'est pas activé côté exporter. La règle est simple : si un panneau reste vide, vérifier que la métrique attendue existe réellement dans Prometheus (via `mysql_up` puis une requête manuelle), avant de soupçonner Grafana.

> 💡 **Spécificité MariaDB 12.x.** Les dashboards communautaires sont pensés pour les métriques génériques MySQL/MariaDB et accusent généralement un retard sur les nouveautés. Les indicateurs propres aux fonctionnalités récentes de MariaDB — par exemple le comportement du nouveau binlog intégré à InnoDB ([§11.5.4](../11-administration-configuration/05.4-binlog-innodb-performance.md)) ou les métriques Galera (`wsrep`) — nécessitent souvent des **panneaux personnalisés**, abordés plus bas.

## Construire un dashboard MariaDB de zéro

Importer un dashboard tout fait est pédagogiquement limité : on ne comprend pas ce qu'on observe. Construire ses propres panneaux à partir de PromQL est la meilleure façon de maîtriser réellement la supervision. Un panneau Grafana se résume à trois éléments : une **requête** (PromQL), un **type de visualisation** (Time series, Gauge, Stat…) et des **options d'affichage** (unités, seuils, légende).

### Les indicateurs essentiels et leurs requêtes

Le tableau suivant constitue un socle de supervision pour une instance MariaDB. Les métriques sont issues du collecteur de statut global de `mysqld_exporter`.

| Indicateur | Requête PromQL | Visualisation |
|------------|----------------|---------------|
| Disponibilité du serveur | `mysql_up` | Stat (rouge si `0`) |
| Uptime | `mysql_global_status_uptime` | Stat (unité : durée) |
| Connexions actives | `mysql_global_status_threads_connected` | Time series |
| Threads en exécution | `mysql_global_status_threads_running` | Time series |
| Débit de requêtes (QPS) | `rate(mysql_global_status_queries[5m])` | Time series |
| Ventilation par commande | `rate(mysql_global_status_commands_total{command=~"select\|insert\|update\|delete"}[5m])` | Time series empilé |
| Requêtes lentes / s | `rate(mysql_global_status_slow_queries[5m])` | Time series |
| Connexions avortées / s | `rate(mysql_global_status_aborted_connects[5m])` | Time series |
| Lignes lues / s | `rate(mysql_global_status_rows_read[5m])` | Time series |
| Tables temp. sur disque / s | `rate(mysql_global_status_created_tmp_disk_tables[5m])` | Time series |
| Trafic réseau reçu | `rate(mysql_global_status_bytes_received[5m])` | Time series (octets/s) |
| Lag de réplication | `mysql_slave_status_seconds_behind_master` | Time series (secondes) |

> ⚠️ La plupart des métriques MariaDB sont des **compteurs cumulatifs** (ils ne font que croître). Les afficher bruts n'a aucun sens : il faut systématiquement les encapsuler dans `rate(...[fenêtre])`, qui en donne le **taux par seconde**. C'est l'erreur la plus fréquente des débutants en PromQL. Seules les valeurs « instantanées » (connexions actives, lag, état du serveur) s'affichent directement.

### Deux formules de ratio à connaître

Certains indicateurs clés ne sont pas des métriques directes mais des **ratios calculés** à partir de plusieurs métriques.

**Taux de hits du Buffer Pool InnoDB** — sans doute l'indicateur de santé le plus important d'une instance OLTP. Il mesure la proportion de lectures servies depuis la RAM plutôt que depuis le disque. On le veut le plus proche possible de 1 (soit 100 %) :

```promql
1 - (
  rate(mysql_global_status_innodb_buffer_pool_reads[5m])
  /
  rate(mysql_global_status_innodb_buffer_pool_read_requests[5m])
)
```

Un ratio qui décroche durablement sous ~0,99 sur une charge OLTP est un signal fort de sous-dimensionnement du Buffer Pool (voir [§15.2.1](../15-performance-tuning/02.1-innodb-buffer-pool.md)).

**Saturation des connexions** — exprime en pourcentage la proximité de la limite `max_connections`. Au-delà de 80 %, le risque d'erreurs `Too many connections` devient réel :

```promql
mysql_global_status_threads_connected
  / mysql_global_variables_max_connections * 100
```

Cet indicateur se présente idéalement en panneau **Gauge** avec des seuils colorés (vert / orange / rouge).

### Panneaux spécifiques à MariaDB

C'est ici que l'on dépasse les dashboards génériques MySQL pour tirer parti des spécificités de MariaDB.

**Supervision d'un cluster Galera** (voir [chapitre 14](../14-haute-disponibilite/02-galera-cluster.md)). Les variables de statut `wsrep_*` sont exposées par l'exporter et sont indispensables au suivi d'un cluster synchrone :

| Indicateur Galera | Requête PromQL |
|-------------------|----------------|
| Taille du cluster | `mysql_global_status_wsrep_cluster_size` |
| Nœud opérationnel | `mysql_global_status_wsrep_ready` |
| État local (4 = `Synced`) | `mysql_global_status_wsrep_local_state` |
| Pause par flow control | `rate(mysql_global_status_wsrep_flow_control_paused[5m])` |
| File de réception locale | `mysql_global_status_wsrep_local_recv_queue` |

La taille du cluster mérite une **alerte** : si `wsrep_cluster_size` tombe sous le nombre de nœuds attendus, un nœud a quitté le cluster. Une file de réception (`local_recv_queue`) qui gonfle indique un nœud qui n'arrive pas à suivre le rythme d'application des write sets.

**Thread Pool** (voir [§11.10](../11-administration-configuration/10-thread-pool.md)). Sur les instances configurées en *thread pool*, le suivi de `mysql_global_status_threadpool_threads` et `threadpool_idle_threads` permet de détecter une saturation des threads de travail.

## Variables et templating : un dashboard, plusieurs instances

Coder en dur le nom d'une instance dans chaque panneau ne passe pas à l'échelle. Les **variables de template** rendent un dashboard générique et réutilisable sur tout le parc.

On crée par exemple une variable `$instance` de type *Query* basée sur la requête `label_values(mysql_up, instance)`, qui peuple automatiquement un menu déroulant avec toutes les instances connues de Prometheus. Chaque requête de panneau filtre alors sur cette variable :

```promql
mysql_global_status_threads_connected{instance="$instance"}
```

Résultat : un **seul dashboard** sert tout le parc MariaDB, et l'opérateur bascule d'une instance à l'autre via un simple menu, sans dupliquer quoi que ce soit. C'est aussi cette approche qui garantit la **portabilité** des dashboards : en exposant la source de données elle-même comme variable (`$datasource`), un même JSON fonctionne dans plusieurs environnements (dev, préprod, prod).

## Annotations : corréler métriques et événements

Une montée du lag de réplication coïncide-t-elle avec un déploiement ? Les **annotations** superposent des marqueurs verticaux sur les graphes pour répondre à ce type de question. Elles peuvent être saisies manuellement, mais surtout être injectées automatiquement par la CI/CD (voir [§16.7](07-cicd-bases-donnees.md)) via l'API HTTP de Grafana à chaque release. Visuellement, on relie alors instantanément une dégradation de performance à un changement applicatif ou à une migration de schéma.

## Alerting natif Grafana

Grafana intègre un moteur d'alerting qui évalue périodiquement des règles fondées sur les mêmes requêtes PromQL que les panneaux. On peut par exemple déclencher une alerte quand `mysql_up == 0` pendant plus d'une minute, quand le lag de réplication dépasse un seuil, ou quand la saturation des connexions franchit 90 %.

La stratégie générale — où placer la logique d'alerte (Grafana vs Prometheus Alertmanager), comment router les notifications et éviter la fatigue d'alertes — relève de la section dédiée [16.11 (Alerting et incident response)](11-alerting-incident-response.md). On retiendra ici simplement que Grafana **peut** porter les alertes, mais qu'en architecture mature, l'alerting est souvent délégué à Alertmanager pour découpler alerte et visualisation.

## Dashboards as Code (intégration GitOps)

Un dashboard construit à la souris est fragile : il vit dans la base SQLite de Grafana, n'est pas versionné, et disparaît si l'instance est recréée. L'approche professionnelle consiste à **versionner les dashboards sous forme de JSON dans Git**, dans l'esprit GitOps développé en [§16.12](12-gitops-bases-donnees.md).

Le *provisioning* de dashboards fonctionne sur le même principe que celui des sources de données :

```yaml
# /etc/grafana/provisioning/dashboards/mariadb.yaml
apiVersion: 1  
providers:  
  - name: 'mariadb'
    orgId: 1
    folder: 'MariaDB'
    type: file
    disableDeletion: false
    updateIntervalSeconds: 30
    options:
      # Répertoire contenant les fichiers .json versionnés
      path: /var/lib/grafana/dashboards
```

Grafana charge alors automatiquement tous les fichiers `.json` du répertoire indiqué. Le cycle de vie complet devient reproductible : on modifie un dashboard dans Grafana, on **exporte son JSON** (`Dashboard settings → JSON Model`), on le commite dans Git, et le pipeline le redéploie. Le tableau de bord devient ainsi un artefact d'infrastructure au même titre que la configuration du serveur.

## Bonnes pratiques de conception

Quelques principes distinguent un dashboard réellement utile d'un mur de graphes illisible :

- **Hiérarchiser de haut en bas.** Placer en première ligne les indicateurs de santé synthétiques (état, uptime, saturation connexions, lag), puis descendre vers le détail (InnoDB, réseau, requêtes). L'œil doit pouvoir diagnostiquer un problème en trois secondes avant d'investiguer.
- **Toujours fixer les unités.** Octets, secondes, pourcentages : une unité explicite (et l'usage des préfixes binaires pour la mémoire) évite des erreurs d'interprétation grossières.
- **Utiliser des seuils colorés** sur les panneaux critiques (Gauge, Stat) plutôt que de demander à l'opérateur de lire des valeurs : le rouge doit signifier « agir maintenant ».
- **Aligner les fenêtres de `rate()` sur le contexte.** Une fenêtre courte (`[1m]`) est réactive mais bruitée ; une fenêtre plus longue (`[5m]`) lisse les tendances. Choisir selon qu'on supervise en temps réel ou qu'on analyse une tendance.
- **Un dashboard, un objectif.** Mieux vaut plusieurs dashboards ciblés (vue d'ensemble, InnoDB, réplication, Galera) qu'un unique tableau fourre-tout. Les variables de template permettent de factoriser sans tout entasser.
- **Versionner, ne pas cliquer.** Tout dashboard de production doit exister sous forme de JSON dans Git.

## Points clés à retenir

Grafana est la couche de visualisation et d'alerting qui exploite, en PromQL, les métriques que `mysqld_exporter` a publiées et que Prometheus stocke ; il ne détient lui-même aucune donnée. On démarre vite en important un dashboard communautaire (notamment l'ID **7362** de Percona), mais la maîtrise réelle vient de la construction de panneaux personnalisés, en gardant à l'esprit que la quasi-totalité des métriques sont des compteurs à encapsuler dans `rate()`. Les indicateurs prioritaires pour MariaDB sont la disponibilité, la saturation des connexions, le taux de hits du Buffer Pool, le débit de requêtes et — en cluster — les métriques Galera `wsrep_*`. Enfin, en contexte DevOps, sources de données et dashboards se déclarent **as Code** (provisioning YAML + JSON versionné dans Git), pour une supervision reproductible et intégrée à la chaîne GitOps.
