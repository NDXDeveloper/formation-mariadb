# 5.5.1 — Index sur colonnes fréquemment filtrées

> **Chapitre 5 — Index et Performance** · Section 5.5.1  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

La clause **`WHERE`** est le premier endroit où regarder. C'est là que s'effectue le **filtrage** : sélectionner quelques lignes parmi des milliers ou des millions. Sans index sur les colonnes filtrées, MariaDB doit examiner chaque ligne pour tester la condition ; avec un index adapté, il se rend directement aux lignes pertinentes. Indexer les colonnes fréquemment filtrées est donc, le plus souvent, l'action d'indexation la plus rentable.

Encore faut-il choisir les **bonnes** colonnes — et écrire les conditions de façon à ce que l'index soit effectivement utilisé. Ces deux points font l'objet de cette section.

---

## Quelles colonnes indexer ? Fréquence × sélectivité

Toutes les colonnes d'une clause `WHERE` ne méritent pas un index. Deux critères se combinent pour identifier les meilleurs candidats :

- **La fréquence** : la colonne est-elle filtrée par des requêtes **souvent exécutées** ? Indexer une colonne consultée une fois par mois n'a guère d'intérêt.
- **La sélectivité** : la colonne a-t-elle une **forte cardinalité** (beaucoup de valeurs distinctes) ? Comme vu en [section 5.2.1](02.1-btree.md), un index n'écarte efficacement des lignes que s'il est discriminant.

Le croisement de ces deux critères hiérarchise les priorités :

| | **Sélectif** | **Peu sélectif** |
|---|---|---|
| **Filtré souvent** | ✅ **Priorité absolue** (ex. `email`, `reference`) | ⚠️ Bénéfice limité (ex. `statut` à 3 valeurs) |
| **Filtré rarement** | Index optionnel, à évaluer | ❌ Inutile |

Une colonne comme `email` (presque unique) filtrée à chaque connexion est le candidat idéal. À l'inverse, une colonne `actif` à deux valeurs, même souvent filtrée, profitera rarement d'un index seul — l'optimiseur préférera un balayage.

```sql
-- email : très sélectif et souvent filtré → excellent candidat
CREATE INDEX idx_email ON clients (email);
SELECT * FROM clients WHERE email = 'alice@example.fr';
```

> **Repère pratique.** L'optimiseur étant fondé sur les coûts, il n'existe pas de seuil strict ; mais en règle générale, dès qu'une condition sélectionne une **forte proportion** de la table (de l'ordre de 20 à 30 % ou plus), un balayage complet devient plus économique qu'un index assorti de ses retours à la table.

---

## Les conditions qui exploitent bien un index

Sur une colonne indexée par un B-Tree, les conditions de filtrage suivantes tirent pleinement parti de l'index (voir [section 5.2.1](02.1-btree.md)) :

- l'**égalité** : `WHERE statut = 'expediee'` ;
- les **plages** : `WHERE montant BETWEEN 50 AND 200`, `WHERE date_commande >= '2026-01-01'` ;
- l'**appartenance** : `WHERE id IN (101, 102, 103)`.

Mais la simple présence d'un index ne suffit pas : la **manière d'écrire la condition** est déterminante.

---

## ⚠️ Les pièges qui désactivent l'index

C'est la partie la plus utile de cette section. De nombreuses requêtes possèdent un index parfaitement adapté… qui n'est jamais utilisé, à cause de la **formulation** de la condition. Voici les cas les plus fréquents.

### Une fonction ou une expression sur la colonne

Appliquer une fonction à la colonne indexée **neutralise l'index** : MariaDB devrait calculer la fonction sur chaque ligne, ce que l'index — qui contient les valeurs *brutes* — ne permet pas.

```sql
-- ❌ La fonction sur la colonne neutralise l'index sur date_commande
SELECT * FROM commandes WHERE YEAR(date_commande) = 2026;

-- ✅ Réécriture en plage : l'index est utilisable
SELECT * FROM commandes
WHERE date_commande >= '2026-01-01' AND date_commande < '2027-01-01';
```

Lorsque la transformation est inévitable (par exemple une recherche insensible à la casse), on peut indexer une **colonne générée** portant le résultat de la fonction ([section 18.4](../18-fonctionnalites-avancees/04-virtual-generated-columns.md)).

### Une conversion de type implicite

Comparer une colonne à une valeur d'un **type différent** force MariaDB à convertir — et si c'est la **colonne** qui doit être convertie, l'index devient inutilisable (avec, en prime, un risque de résultat erroné).

```sql
-- code_postal est de type VARCHAR
-- ❌ Comparé à un nombre : la colonne est convertie → index inutilisable
SELECT * FROM clients WHERE code_postal = 75001;

-- ✅ Littéral du bon type (chaîne) : l'index est utilisé
SELECT * FROM clients WHERE code_postal = '75001';
```

Le sens de la conversion compte : comparer une colonne **numérique** à une chaîne (`WHERE id = '42'`) ne pose pas de problème, car c'est la constante qui est convertie. Le cas dangereux est celui d'une colonne **textuelle** comparée à un nombre. Veiller à ce que **les types des deux côtés de la comparaison concordent** — y compris les **collations**, dont la divergence peut produire le même effet dans les jointures.

### Un joker en tête

Rappel de la [section 5.2.1](02.1-btree.md) : une recherche par préfixe exploite l'index, mais un **joker en début** de motif l'en empêche.

```sql
SELECT * FROM clients WHERE nom LIKE 'Dup%';     -- ✅ préfixe : index utilisé
SELECT * FROM clients WHERE nom LIKE '%dupont%'; -- ❌ joker en tête : index inutilisable
```

### La négation

Les conditions de **négation** (`!=`, `<>`, `NOT IN`) sélectionnent généralement une grande partie de la table : leur faible sélectivité conduit l'optimiseur à ignorer l'index au profit d'un balayage.

```sql
SELECT * FROM commandes WHERE statut <> 'annulee';  -- index rarement retenu
```

### Un `OR` entre colonnes différentes

Un index **composite** sur `(a, b)` accélère un filtre `a = ? AND b = ?`, mais **n'aide pas** un `a = ? OR b = ?`. Pour une condition `OR` portant sur des colonnes distinctes, l'optimiseur peut combiner **deux index séparés** (technique de *index merge*), ou l'on peut réécrire la requête en `UNION`.

```sql
SELECT * FROM commandes WHERE client_id = 42 OR statut = 'urgent';
-- → deux index distincts (index merge) ou réécriture en UNION ; un composite n'aiderait pas
```

---

## Filtrer sur plusieurs colonnes : préférer un index composite

Lorsqu'une requête fréquente filtre sur **plusieurs colonnes combinées par `AND`**, un seul **index composite** est presque toujours préférable à plusieurs index mono-colonne :

```sql
-- Filtre récurrent sur deux colonnes combinées
SELECT * FROM commandes WHERE client_id = 42 AND statut = 'expediee';

-- ✅ Un index composite est plus efficace que deux index séparés
CREATE INDEX idx_client_statut ON commandes (client_id, statut);
```

L'**ordre des colonnes** dans un index composite est crucial (principe du *left-most prefix*) et fait l'objet de la [section 5.6](06-index-composites.md). À défaut de composite, l'optimiseur peut recourir au *index merge* pour intersecter deux index mono-colonne, mais cette solution reste généralement moins efficace qu'un composite bien conçu.

---

## Toujours vérifier

Tous ces principes se valident sur le terrain avec **`EXPLAIN`** ([section 5.7](07-analyse-plans-execution.md)) : il révèle si la requête utilise réellement l'index attendu, ou si elle retombe sur un balayage complet — symptôme typique de l'un des pièges ci-dessus.

---

> ### 📝 À retenir  
>  
> - Les colonnes de la clause `WHERE` sont la **cible la plus rentable** ; on les hiérarchise par **fréquence × sélectivité**.  
> - Un index ne sert que si la condition est **bien écrite**. Les principaux pièges qui le désactivent :  
>   - **fonction sur la colonne** (`YEAR(col) = …`) → réécrire en plage ou indexer une colonne générée ;  
>   - **conversion de type implicite** (colonne textuelle comparée à un nombre) → faire concorder les types ;  
>   - **joker en tête** (`LIKE '%x'`), **négation** (`<>`, `NOT IN`), et **`OR`** entre colonnes (un composite n'aide pas).  
> - Pour un filtre **multi-colonnes en `AND`**, préférer un **index composite** (ordre des colonnes : 5.6) à plusieurs index séparés.  
> - **Vérifier** systématiquement l'usage de l'index avec `EXPLAIN` (5.7).
