# 5.2.4 — Spatial : les données géographiques

> **Chapitre 5 — Index et Performance** · Section 5.2.4  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

Trouver les magasins les plus proches d'une position, déterminer quelle zone de livraison contient une adresse, identifier les routes qui traversent un secteur : ces questions portent sur des données **géométriques**, et elles partagent une caractéristique commune — elles sont **bidimensionnelles**. C'est précisément ce qui les met hors de portée des structures vues jusqu'ici. L'index **spatial**, fondé sur une structure de **R-Tree**, est conçu pour répondre efficacement à ce type d'interrogation.

---

## Pourquoi un B-Tree échoue sur les données géographiques

Un B-Tree ordonne les valeurs selon **une seule dimension**. C'est parfait pour des nombres ou des dates, mais une position géographique a **deux coordonnées** (longitude et latitude, ou X et Y), et aucune ne suffit à elle seule.

On pourrait imaginer un index B-Tree sur la latitude et un autre sur la longitude. Mais deux points peuvent avoir des latitudes très proches tout en étant séparés par des milliers de kilomètres en longitude. Trier sur une dimension **ne préserve donc pas la proximité spatiale** : des points voisins sur la carte se retrouvent éparpillés dans l'index. La notion même de « ce qui est proche » exige de raisonner sur les **deux dimensions à la fois** — ce qu'un B-Tree ne sait pas faire.

---

## Les types géométriques de MariaDB

MariaDB implémente la hiérarchie de types géométriques de la norme **OGC** (*Open Geospatial Consortium*) :

- `GEOMETRY` : le type de base, qui peut contenir n'importe quelle géométrie ;
- `POINT` : un point (une position) ;
- `LINESTRING` : une ligne brisée (un trajet, une route) ;
- `POLYGON` : une surface fermée (une zone, un territoire) ;
- ainsi que les types composés `MULTIPOINT`, `MULTILINESTRING`, `MULTIPOLYGON` et `GEOMETRYCOLLECTION`.

En interne, ces valeurs sont stockées au format binaire **WKB** (*Well-Known Binary*). On les construit le plus souvent à partir de leur représentation textuelle **WKT** (*Well-Known Text*), via des fonctions comme `ST_GeomFromText()` ou des constructeurs dédiés (`POINT()`, `ST_PointFromText()`…).

```sql
-- Un point au format WKT : POINT(longitude latitude)
ST_GeomFromText('POINT(1.0993 49.4431)', 4326)   -- Rouen, en WGS84
```

---

## La structure : R-Tree et boîte englobante (MBR)

L'index spatial repose sur un **R-Tree** (*Region Tree*). Son principe : plutôt que d'indexer la forme exacte de chaque géométrie — qui peut être un polygone complexe à des centaines de sommets —, il indexe sa **boîte englobante**, ou **MBR** (*Minimum Bounding Rectangle*) : le plus petit rectangle aligné sur les axes qui contient entièrement la forme.

```
   MBR (boîte englobante)
   ┌───────────────────────┐
   │        _____          │
   │       /     \         │
   │      /  forme \       │   ← le polygone réel
   │      \  réelle /      │
   │       \_______/       │
   │                       │
   └───────────────────────┘
   L'index ne manipule que le rectangle :
   c'est rapide, mais approximatif.
```

Le R-Tree regroupe ensuite hiérarchiquement les MBR voisines dans des MBR plus grandes, formant un arbre où chaque nœud englobe ses enfants. Une recherche peut ainsi écarter d'emblée des régions entières de l'espace.

Cette approche par boîtes englobantes implique une logique en **deux temps**, dite « **filtrer puis raffiner** » :

1. **Filtrage** (rapide, via l'index R-Tree) : on sélectionne les géométries dont la **boîte englobante** satisfait grossièrement la condition. C'est approximatif — une MBR peut se recouper sans que les formes réelles ne se touchent.
2. **Raffinage** (précis) : sur ce petit ensemble de candidats, MariaDB calcule le prédicat **géométrique exact**.

C'est ce mécanisme qui rend les requêtes spatiales à la fois rapides *et* exactes.

---

## Créer un index spatial

Comme le full-text, l'index spatial se déclare **explicitement**, avec le mot-clé `SPATIAL`. Deux contraintes s'appliquent : la colonne doit être d'un **type géométrique**, et elle doit être déclarée **`NOT NULL`**. Il est pris en charge par les moteurs **InnoDB**, **MyISAM** et **Aria**.

```sql
CREATE TABLE magasins (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  nom         VARCHAR(150) NOT NULL,
  emplacement POINT        NOT NULL,        -- géométrie, NOT NULL obligatoire
  SPATIAL INDEX sp_emplacement (emplacement)
) ENGINE = InnoDB;

-- Ou a posteriori
ALTER TABLE magasins ADD SPATIAL INDEX sp_emplacement (emplacement);
```

## Interroger les données spatiales

Les recherches spatiales s'expriment via des **fonctions de relation géométrique**, réparties en deux familles qu'il importe de distinguer.

| Famille | Exemples | Calcul | Index R-Tree |
|---------|----------|--------|:------------:|
| Fonctions **MBR** | `MBRContains`, `MBRWithin`, `MBRIntersects`, `MBROverlaps`, `MBRTouches` | Sur les **boîtes englobantes** (approximatif) | ✅ directement |
| Fonctions **exactes** (`ST_`) | `ST_Contains`, `ST_Within`, `ST_Intersects`, `ST_Crosses`, `ST_Overlaps` | Sur la **géométrie réelle** (précis) | ✅ pour l'étape de pré-filtrage |

Le point essentiel : l'optimiseur peut exploiter le R-Tree pour la phase de **pré-filtrage MBR** même lorsqu'on utilise les fonctions exactes `ST_` (comme `ST_Contains` ou `ST_Intersects`), puis il calcule le prédicat précis sur les seuls candidats retenus. On bénéficie donc de l'index tout en obtenant un résultat exact.

Font exception les fonctions de **mesure** comme `ST_Distance` et `ST_Distance_Sphere` : ne se ramenant pas à une relation entre boîtes englobantes, elles **n'exploitent pas** le R-Tree (l'index n'apparaît même pas dans `possible_keys`). Une recherche par distance impose donc le motif « **boîte englobante + distance** » détaillé plus bas.

### Exemple : quel territoire contient ce point ?

```sql
SELECT z.nom
FROM zones_livraison z
WHERE ST_Contains(
        z.perimetre,
        ST_GeomFromText('POINT(1.0993 49.4431)', 4326)
      );
```

### Exemple : quelles routes traversent une zone ?

```sql
SELECT r.nom
FROM routes r
WHERE ST_Intersects(r.trace, @zone);
```

## SRID et distances réelles

Une coordonnée n'a de sens que rapportée à un **système de référence spatial**, identifié par un **SRID** (*Spatial Reference System Identifier*). Deux valeurs sont à connaître :

- **SRID 0** : un plan cartésien abstrait, **sans unité** géographique. C'est la valeur par défaut.
- **SRID 4326** : le système **WGS84**, celui des coordonnées GPS (longitude / latitude). C'est le SRID à employer pour des données terrestres réelles.

Le SRID est déterminant pour le **calcul des distances**. La fonction `ST_Distance()` traite les coordonnées comme des points d'un plan : sur des longitudes/latitudes, elle renvoie une distance « en degrés », dépourvue de signification physique. Pour obtenir une **distance réelle en mètres**, on utilise `ST_Distance_Sphere()`, qui calcule la distance orthodromique (à la surface d'une sphère).

### Le motif « boîte englobante + distance » pour une recherche dans un rayon

Une recherche du type « tous les magasins à moins de 5 km » illustre parfaitement le « filtrer puis raffiner ». Une condition de distance ne peut pas exploiter directement le R-Tree ; on procède donc en deux étapes :

```sql
-- ① Pré-filtrage par une boîte englobante autour du centre  → utilise l'index R-Tree
-- ② Raffinage par la distance réelle (en mètres)            → calcul exact
SELECT nom,
       ST_Distance_Sphere(emplacement, @centre) AS distance_m
FROM magasins
WHERE MBRContains(@boite, emplacement)                  -- ① rectangle de recherche
  AND ST_Distance_Sphere(emplacement, @centre) <= 5000  -- ② rayon de 5 km
ORDER BY distance_m;
```

Ici `@boite` est un rectangle couvrant la zone de recherche : il permet à l'index d'éliminer rapidement la grande majorité des points, avant que `ST_Distance_Sphere` ne calcule la distance exacte sur les rares candidats restants.

---

## Quand utiliser un index spatial ?

Dès qu'une application manipule des **positions ou des formes géographiques** sur lesquelles on effectue des recherches de proximité, de contenance ou d'intersection : cartographie, géolocalisation de points d'intérêt, zones de chalandise ou de livraison, suivi de trajets, applications SIG. Pour de simples coordonnées que l'on ne fait que filtrer par plages (par exemple « latitude entre X et Y »), un B-Tree classique peut suffire ; l'index spatial devient indispensable dès que la **proximité bidimensionnelle** entre en jeu.

> **Note de compatibilité.** L'écosystème géospatial évolue ; MariaDB 12.3 enrichit sa prise en charge de fonctions GIS pour se rapprocher de MySQL 8 (voir [chapitre 19](../19-migration-compatibilite/01.1-compatibilite-mysql-mariadb.md)). En cas de migration ou de portabilité, vérifiez la disponibilité et le comportement exact de chaque fonction sur la version cible.

---

> ### 📝 À retenir  
>  
> - Les données géographiques sont **bidimensionnelles** : un B-Tree, ordonné sur une seule dimension, ne préserve pas la **proximité spatiale**.  
> - L'index spatial repose sur un **R-Tree** indexant les **boîtes englobantes** (**MBR**), ce qui impose une logique « **filtrer puis raffiner** » : pré-filtrage MBR rapide via l'index, puis calcul géométrique exact sur les candidats.  
> - Il se déclare avec `SPATIAL`, sur une colonne **géométrique** et **`NOT NULL`** (moteurs InnoDB, MyISAM, Aria).  
> - On interroge via les fonctions **MBR** (`MBRContains`…) ou **exactes** (`ST_Contains`, `ST_Intersects`…) ; l'index sert au pré-filtrage dans les deux cas.  
> - Le **SRID** définit le système de coordonnées (0 = plan abstrait, 4326 = WGS84/GPS) ; pour une distance **en mètres**, utiliser `ST_Distance_Sphere`, pas `ST_Distance`.  
> - Recherche dans un rayon = **boîte englobante** (indexée) **+** distance exacte (raffinage).
