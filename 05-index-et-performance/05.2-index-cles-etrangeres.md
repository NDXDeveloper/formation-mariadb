# 5.5.2 — Index sur clés étrangères

> **Chapitre 5 — Index et Performance** · Section 5.5.2  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

Les **clés étrangères** constituent une cible d'indexation à part. D'abord parce qu'elles sont les **colonnes de jointure** par excellence : c'est par elles que les tables se relient. Ensuite — et c'est la particularité de cette section — parce qu'**InnoDB en gère l'indexation automatiquement**. La stratégie ne consiste donc pas tant à *décider* d'indexer qu'à **comprendre ce comportement**, en éviter les pièges, et savoir quand faire mieux que l'index par défaut.

---

## Pourquoi une clé étrangère doit être indexée

Deux raisons distinctes l'imposent — l'une bien connue, l'autre souvent négligée.

### Raison n° 1 : la performance des jointures

Une clé étrangère est la colonne sur laquelle on **joint** la table enfant à la table parent. Sans index sur la colonne portant la clé étrangère, chaque jointure dégénère en balayage de la table enfant.

```sql
SELECT c.id, cl.nom
FROM commandes c
JOIN clients cl ON c.client_id = cl.id   -- client_id : la clé étrangère
WHERE cl.nom = 'Durand';
```

Ici, un index sur `commandes.client_id` permet de retrouver rapidement les commandes d'un client donné, au lieu de parcourir toute la table des commandes.

### Raison n° 2 : la performance des vérifications d'intégrité

C'est la raison la plus souvent oubliée. Lorsqu'on **modifie ou supprime une ligne parent**, InnoDB doit vérifier les lignes enfants qui la référencent, afin d'appliquer la règle de la contrainte (`RESTRICT`, `CASCADE`, `SET NULL`…).

```sql
DELETE FROM clients WHERE id = 42;
-- InnoDB doit retrouver toutes les commandes référençant le client 42.
```

Sans index sur `commandes.client_id`, cette vérification exige un **balayage complet de la table enfant à chaque modification du parent** — un coût rédhibitoire sur des tables volumineuses. C'est précisément pour éviter cela qu'InnoDB rend cet index **obligatoire**.

---

## L'indexation automatique d'InnoDB

Conséquence directe de ces exigences : lorsqu'on déclare une contrainte de clé étrangère, **InnoDB crée automatiquement un index sur la ou les colonnes portant la clé étrangère**, si aucun index approprié n'existe déjà.

```sql
CREATE TABLE commandes (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  client_id     INT NOT NULL,
  date_commande DATETIME,
  CONSTRAINT fk_commande_client
    FOREIGN KEY (client_id) REFERENCES clients (id)
) ENGINE = InnoDB;
```

Ici, même sans index explicite sur `client_id`, InnoDB en crée un. On le constate avec :

```sql
SHOW INDEX FROM commandes;   -- révèle l'index auto-créé sur client_id
```

Du **côté parent**, la colonne référencée (`clients.id`) doit elle aussi être indexée — c'est généralement le cas, puisqu'il s'agit le plus souvent de la **clé primaire**. (Les fondements clé primaire / index secondaire sont rappelés en [section 5.1](01-fonctionnement-index.md).)

---

## ⚠️ Le piège des clés étrangères « logiques »

L'indexation automatique ne s'applique qu'aux **contraintes de clé étrangère réellement déclarées**. Or de nombreux schémas comportent des relations purement **logiques** : la colonne `commandes.client_id` *correspond* à `clients.id`, mais **aucune contrainte `FOREIGN KEY` n'est définie** (par choix d'architecture, héritage d'un schéma ancien, ou contrainte applicative).

Dans ce cas, **aucun index n'est créé automatiquement**. La colonne de jointure reste silencieusement non indexée, et toutes les jointures qui l'utilisent sont lentes — un problème de performance fréquent et facile à manquer.

```sql
-- Relation logique, SANS contrainte FK déclarée → aucun index auto-créé
-- commandes.client_id n'est pas indexé : jointures lentes.

-- ✅ Il faut alors l'indexer manuellement, comme une colonne filtrée ordinaire
CREATE INDEX idx_client ON commandes (client_id);
```

> **À retenir absolument :** que la relation soit matérialisée par une contrainte ou seulement logique, **toute colonne de jointure mérite un index**. Avec une contrainte déclarée, InnoDB s'en charge ; sans contrainte, c'est à vous de le faire.

---

## Optimiser : un index composite avec la clé étrangère en tête

L'index auto-créé par InnoDB est **mono-colonne** : il porte sur la seule clé étrangère. C'est suffisant pour les deux raisons ci-dessus, mais souvent perfectible. Beaucoup de requêtes filtrent sur la clé étrangère **et** trient ou filtrent sur une autre colonne :

```sql
-- Requête fréquente : les commandes d'un client, triées par date
SELECT * FROM commandes WHERE client_id = 42 ORDER BY date_commande;
```

Un **index composite** plaçant la clé étrangère **en première position** sert alors la requête complète *et* satisfait l'exigence d'InnoDB (la clé étrangère étant le préfixe gauche de l'index) :

```sql
CREATE INDEX idx_client_date ON commandes (client_id, date_commande);
```

Deux conséquences pratiques :

- Si l'on définit ce composite **dès la création** de la table (clé étrangère en tête), InnoDB **ne crée pas** d'index mono-colonne redondant : la contrainte est déjà satisfaite.
- Si l'index mono-colonne a déjà été créé automatiquement, il devient **redondant** une fois le composite en place (son contenu est couvert par le préfixe gauche du composite, cf. [section 5.6](06-index-composites.md)). InnoDB autorise alors sa **suppression**, dès lors qu'un autre index satisfait l'exigence de la clé étrangère.

C'est l'illustration parfaite du principe de **mutualisation** vu en [section 5.5](05-strategies-indexation.md) : un seul index bien conçu remplit plusieurs rôles.

---

## Note sur les moteurs de stockage

Tout ce qui précède concerne **InnoDB**, qui prend en charge et **fait respecter** les contraintes de clé étrangère. Le moteur **MyISAM**, lui, **analyse mais ignore** ces contraintes : il n'y a donc ni vérification d'intégrité, ni index auto-créé. Sur une table MyISAM, la raison « intégrité » disparaît, mais la raison « performance des jointures » demeure entière — les colonnes de jointure doivent y être indexées **manuellement**.

---

> ### 📝 À retenir  
>  
> - Une clé étrangère doit être indexée pour **deux** raisons : la **performance des jointures** et celle des **vérifications d'intégrité référentielle** lors des modifications du parent.  
> - Sur **InnoDB**, déclarer une contrainte `FOREIGN KEY` **crée automatiquement** l'index nécessaire sur la colonne enfant ; la colonne parent référencée doit elle aussi être indexée (souvent la clé primaire).  
> - **Piège majeur** : les clés étrangères **purement logiques** (sans contrainte déclarée) ne sont **pas** auto-indexées — il faut les indexer soi-même.  
> - Pour aller plus loin, un **index composite avec la clé étrangère en première position** sert à la fois la requête et l'exigence d'InnoDB, et rend l'index mono-colonne redondant (mutualisation).  
> - **MyISAM** ignore les contraintes FK : aucun index automatique, mais l'indexation des colonnes de jointure reste nécessaire pour les performances.
