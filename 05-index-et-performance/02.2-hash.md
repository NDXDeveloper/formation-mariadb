# 5.2.2 — Hash : l'égalité stricte

> **Chapitre 5 — Index et Performance** · Section 5.2.2  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

Là où le B-Tree est un généraliste, l'index **Hash** est un **spécialiste** : il fait une seule chose, mais il la fait remarquablement vite. Cette unique spécialité, c'est la **recherche d'égalité stricte** — répondre à la question « où se trouve la ligne dont la clé vaut exactement *X* ? ». En contrepartie de cette rapidité, il est totalement inopérant pour tout le reste : plages, tris, préfixes. C'est ce compromis radical qu'il faut comprendre pour savoir quand — rarement — y recourir.

---

## Principe de fonctionnement

Un index Hash repose sur une **fonction de hachage** : on lui fournit une clé, elle calcule en retour un nombre (le *hash*) qui désigne directement un emplacement de stockage, appelé *bucket* (alvéole). Pour retrouver une ligne, il suffit donc de :

1. calculer le *hash* de la valeur recherchée ;
2. se rendre directement dans le *bucket* correspondant ;
3. y lire le ou les pointeurs vers la ou les lignes.

Aucun parcours d'arbre, aucune comparaison successive : l'accès est **direct**. C'est pourquoi la recherche d'égalité s'effectue en temps **constant en moyenne — O(1)** —, indépendamment du nombre de lignes de la table. Lorsque deux clés différentes produisent le même *hash* (une **collision**), elles partagent le même *bucket* ; MariaDB compare alors les clés réelles pour départager. Tant que la fonction répartit bien les valeurs, les collisions restent rares et la performance reste optimale.

Le point capital est une **conséquence directe** de ce mécanisme : le *hash* d'une valeur n'a **aucun rapport avec son ordre**. Deux valeurs proches (`100` et `101`) aboutissent à des *buckets* sans lien entre eux. L'index Hash ne conserve donc **aucune notion d'ordre** — et c'est de là que découlent toutes ses limites.

---

## Ce que le Hash fait (très) bien… et ce qu'il ne sait pas faire

Puisqu'il ne sait que localiser une clé exacte, l'index Hash excelle uniquement sur :

- l'**égalité** : `WHERE col = ?` ;
- l'**appartenance à un ensemble** : `WHERE col IN (?, ?, ?)`, traitée comme une suite d'égalités.

En revanche, faute d'ordre, il est **incapable** de servir :

- les **plages** (`>`, `<`, `BETWEEN`) ;
- les **tris** (`ORDER BY`) ;
- les recherches par **préfixe** (`LIKE 'abc%'`) ;
- les recherches de **valeurs extrêmes** (`MIN`, `MAX`).

Il existe une dernière limite, propre aux index Hash **composites** : le *hash* étant calculé sur la **combinaison complète** des colonnes, on ne peut interroger l'index qu'en fournissant **toutes** ses colonnes à la fois. Contrairement au B-Tree, il n'offre **aucune capacité de préfixe de clé** (*left-most prefix*, voir [section 5.6](06-index-composites.md)).

### Comparatif synthétique

| Opération | B-Tree | Hash |
|-----------|:------:|:----:|
| Égalité (`=`) | ✅ O(log *n*) | ✅ **O(1) en moyenne** |
| Appartenance (`IN`) | ✅ | ✅ |
| Plage (`<`, `>`, `BETWEEN`) | ✅ | ❌ |
| Tri (`ORDER BY`) | ✅ | ❌ |
| Préfixe (`LIKE 'x%'`) | ✅ | ❌ |
| `MIN` / `MAX` | ✅ | ❌ |
| Préfixe de clé composite | ✅ | ❌ *(clé entière requise)* |

## Où trouve-t-on des index Hash dans MariaDB ?

C'est ici que se niche une confusion fréquente. Il faut distinguer **deux choses très différentes** qui portent le mot « hash ».

### 1. L'index Hash utilisateur : le moteur MEMORY

Le seul endroit où l'on crée explicitement un index Hash est le **moteur de stockage MEMORY** (tables entièrement en RAM), pour lequel le Hash est même le **type par défaut** :

```sql
CREATE TABLE cache_sessions (
  session_id CHAR(32)  NOT NULL,
  user_id    INT       NOT NULL,
  expire_at  DATETIME,
  PRIMARY KEY (session_id)          -- index HASH par défaut (table MEMORY)
) ENGINE = MEMORY;
```

On peut, sur ce même moteur, choisir explicitement l'une ou l'autre structure :

```sql
CREATE TABLE correspondance (
  code  CHAR(8) NOT NULL,
  label VARCHAR(100),
  INDEX idx_code (code) USING HASH   -- Hash explicite
) ENGINE = MEMORY;
-- USING BTREE forcerait au contraire un B-Tree, utile si l'on a besoin de plages ou de tris.
```

Le moteur MEMORY est détaillé au [chapitre 7](../07-moteurs-de-stockage/10.1-memory.md).

### 2. InnoDB : pas de Hash utilisateur

Sur une table **InnoDB**, il est **impossible** de créer un index Hash défini par l'utilisateur. La clause `USING HASH` y est **acceptée mais ignorée** : MariaDB crée en réalité un index **B-Tree**.

```sql
-- Sur une table InnoDB, ceci NE crée PAS un index Hash :
-- la clause USING HASH est ignorée, un B-Tree est créé à la place.
CREATE INDEX idx_statut ON commandes (statut) USING HASH;
```

> **À noter :** `SHOW INDEX` et `SHOW CREATE TABLE` afficheront tout de même `USING HASH` (la clause est conservée dans le catalogue), mais le comportement est bien celui d'un **B-Tree** : une recherche de **plage** (`BETWEEN`, `<`, `>`) ou un **`ORDER BY`** l'exploitent sans difficulté — ce qu'un véritable index Hash ne permettrait jamais.

### 3. L'Adaptive Hash Index : le hachage automatique d'InnoDB

InnoDB n'expose pas d'index Hash, mais il en construit **automatiquement** en interne. L'**Adaptive Hash Index** (AHI) est un index Hash que le moteur bâtit **de lui-même**, en mémoire, par-dessus ses index B-Tree, pour les pages les plus fréquemment consultées. L'objectif : transformer à la volée certains accès B-Tree répétés en accès Hash quasi instantanés.

Ce mécanisme est **entièrement transparent** : vous n'avez rien à déclarer, et vous bénéficiez de l'accélération sur les recherches d'égalité répétées sans le moindre effort. Il se pilote par la variable `innodb_adaptive_hash_index`, et peut être activé ou désactivé selon la charge — sur certains profils très concurrents, il peut en effet devenir un point de contention. Son fonctionnement, sa configuration et l'arbitrage de son activation sont traités au [chapitre 15](../15-performance-tuning/13-adaptive-hash-index.md).

> **En résumé :** sur InnoDB, on ne *crée* pas d'index Hash — le moteur s'en charge tout seul, quand c'est pertinent.

---

## Quand utiliser un index Hash ?

Le cas d'usage est étroit et bien identifié : des **tables de consultation purement par égalité**, hébergées en mémoire avec le moteur MEMORY — caches applicatifs, tables de correspondance, stockage de sessions, données de travail temporaires. Dès qu'un besoin de **plage**, de **tri** ou de **préfixe** apparaît, le Hash devient inadapté et le **B-Tree reprend l'avantage**.

Dans l'immense majorité des schémas reposant sur InnoDB, la question ne se pose d'ailleurs même pas : on profite déjà de l'accélération d'égalité **gratuitement et automatiquement**, grâce à l'Adaptive Hash Index.

---

> ### 📝 À retenir  
>  
> - L'index Hash applique une **fonction de hachage** pour un accès **direct** : il sert la recherche d'**égalité stricte** (`=`, `IN`) en temps **O(1) en moyenne**.  
> - Faute d'**ordre**, il est **incapable** de gérer plages, tris, préfixes, `MIN`/`MAX`, ainsi que les préfixes de clé composite (la clé entière est requise).  
> - Le seul index Hash **utilisateur** se crée sur le moteur **MEMORY**, dont c'est le type par défaut.  
> - Sur **InnoDB**, `USING HASH` est **ignoré** (un B-Tree est créé) ; en revanche l'**Adaptive Hash Index** construit automatiquement des index Hash internes, de façon transparente (voir chapitre 15).  
> - Usage recommandé : tables **en mémoire** consultées **uniquement par égalité**. Partout ailleurs, préférer le **B-Tree**.
