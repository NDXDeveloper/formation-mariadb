# 5.8.1 — Optimisations sur scans inversés 🆕

> *Rowid Filtering, Index Condition Pushdown, Loose Index Scan sur clés DESC*  
>  
> **Chapitre 5 — Index et Performance** · Section 5.8.1  
> Version de référence : **MariaDB 12.3 LTS** · 🆕 Nouveauté de la série 12.x

---

## Introduction

Cette section traite l'une des améliorations d'optimiseur de la série **12.x**. Pour l'apprécier, il faut connaître trois optimisations internes de MariaDB — l'**Index Condition Pushdown**, le **Rowid Filtering** et le **Loose Index Scan** — et comprendre une limitation qu'elles partageaient jusqu'ici : elles étaient **désactivées lors d'un scan d'index en ordre inversé**. La série 12.x lève cette restriction. Autrement dit, les requêtes qui parcourent un index « à l'envers » ne renoncent plus à ces accélérations.

---

## Le contexte : le scan inversé

Un **scan d'index en ordre inversé** (*backward-ordered index scan*) consiste à parcourir un index de la fin vers le début, plutôt que dans son sens de tri naturel. MariaDB y recourt typiquement pour satisfaire un `ORDER BY … DESC` à partir d'un index ascendant ([section 5.2.1](02.1-btree.md)), ou lorsqu'on exploite un index dont des parties de clé sont déclarées `DESC` ([section 5.6](06-index-composites.md)).

Le problème, jusqu'à la série 12.x : choisir ce parcours inversé obligeait l'optimiseur à **renoncer** à plusieurs optimisations majeures. Une requête `ORDER BY … DESC` pouvait ainsi se retrouver paradoxalement **moins** optimisée que son équivalent en ordre croissant — un compromis fâcheux que la 12.x élimine.

---

## Les trois optimisations concernées

### Index Condition Pushdown (ICP)

L'ICP réduit le nombre de **lectures de lignes complètes** en vérifiant, **dès l'accès à l'index**, la part de la clause `WHERE` qui porte sur des colonnes indexées (la *condition poussée*). Si cette condition n'est pas satisfaite, MariaDB **évite de lire la ligne** dans la table. C'est particulièrement utile avec un index composite : les premières colonnes servent à l'accès, les suivantes à vérifier des conditions avant le coûteux retour à la table ([section 5.1](01-fonctionnement-index.md)).

```sql
-- Index idx (statut, reference)
EXPLAIN SELECT * FROM commandes
WHERE statut = 'expediee' AND reference LIKE '%2026%';
-- Accès par statut ; la condition sur reference est « poussée » dans l'index
-- et vérifiée AVANT de lire la ligne complète.
-- Extra : Using index condition
```

`EXPLAIN` signale l'ICP par la mention **`Using index condition`** ([section 5.7](07-analyse-plans-execution.md)). L'optimisation s'applique aux méthodes d'accès `range`, `ref`, `eq_ref`, `ref_or_null` et *Batched Key Access*. Elle est active par défaut (`optimizer_switch` : `index_condition_pushdown=on`).

> **🆕 En 12.x** : l'ICP s'applique désormais aussi aux **scans inversés** (MDEV-34413). Auparavant, un parcours en ordre inversé le désactivait.

### Rowid Filtering

Disponible depuis MariaDB 10.4, le Rowid Filtering accélère les requêtes où **deux index** pourraient servir : l'un pour l'**accès**, l'autre — plus **sélectif** — pour **filtrer**. Le principe :

1. effectuer un *index-only scan* sur l'index sélectif et **collecter les identifiants de ligne** (rowids) dans une structure de filtrage ;
2. lors de l'accès via l'autre index, **vérifier ce filtre avant de lire la ligne complète**.

On évite ainsi de nombreux retours à la table pour des lignes qui seraient finalement écartées.

```sql
SELECT * FROM commandes
WHERE date_commande BETWEEN '2026-01-01' AND '2026-01-31'  -- index idx_date (accès)
  AND montant BETWEEN 200 AND 250;                         -- index idx_montant (sélectif → filtre)
-- EXPLAIN : type « range|filter », key « idx_date|idx_montant »,
--           Extra « Using where; Using rowid filter »
```

`EXPLAIN` le révèle par un type suffixé **`|filter`**, une `key` de la forme **`index_accès|index_filtre`**, et la mention **`Using rowid filter`** ; `ANALYZE FORMAT=JSON` détaille même la sélectivité réelle du filtre (`r_selectivity_pct`). Actif par défaut (`optimizer_switch` : `rowid_filter=on`).

> **🆕 En 12.x** : le Rowid Filtering s'applique désormais aux **scans inversés** (MDEV-36094). Auparavant, un parcours en ordre inversé le désactivait.

### Loose Index Scan sur clés DESC

Le *Loose Index Scan*, déjà présenté en [section 5.5.3](05.3-index-order-group.md), accélère les `GROUP BY` sur un **préfixe** d'index avec des agrégats `MIN`/`MAX`, en **sautant** d'une valeur distincte à la suivante (mention `Using index for group-by`).

```sql
-- Index (client_id, date_commande DESC)
SELECT client_id, MAX(date_commande) FROM commandes GROUP BY client_id;
-- Extra : Using index for group-by
```

> **🆕 En 12.x** : le Loose Index Scan peut désormais exploiter des index comportant des **parties de clé `DESC`** (MDEV-32732). Cette optimisation reste donc disponible même sur des index descendants.

---

## Ce que cela change concrètement

L'impact pratique de ces trois améliorations se résume simplement : **le tri descendant cesse d'être pénalisant** du point de vue de l'optimiseur.

- Les requêtes **`ORDER BY … DESC`**, très courantes (afficher les éléments les plus récents, paginer du plus grand au plus petit), conservent désormais l'**ICP** et le **Rowid Filtering** même lorsqu'elles sont servies par un scan inversé — d'où moins de retours à la table et moins d'I/O.
- Les **index descendants** et les composites à parties de clé `DESC` ([sections 5.2.1](02.1-btree.md) et [5.6](06-index-composites.md)) deviennent des citoyens de **première classe** pour ces optimisations. Cela renforce notamment l'intérêt d'un index aux **directions concordantes** pour un tri multi-colonnes mixte, vu en [section 5.5.3](05.3-index-order-group.md).

```sql
-- Cas typique désormais pleinement optimisé en 12.x :
SELECT * FROM commandes
WHERE statut = 'expediee' AND reference LIKE '%2026%'
ORDER BY id DESC
LIMIT 50;
-- Scan inversé pour l'ORDER BY DESC, MAIS l'ICP (sur reference) reste actif.
```

---

## Observer et contrôler

Comme toujours, ces mécanismes se constatent avec **`EXPLAIN`** et **`ANALYZE`** ([section 5.7](07-analyse-plans-execution.md)) via les mentions `Using index condition`, `Using rowid filter` et `Using index for group-by`. Ils se pilotent individuellement par la variable **`optimizer_switch`** (modifiable par session), utile pour diagnostiquer leur effet en les désactivant temporairement :

```sql
SELECT @@optimizer_switch;                    -- état des bascules d'optimisation
SET optimizer_switch = 'rowid_filter=off';    -- désactivation ciblée (diagnostic)
```

En usage normal, on les laisse **activés** : leur bénéfice est quasi systématique, et désormais indépendant du sens du parcours.

---

> ### 📝 À retenir  
>  
> - Trois optimisations internes accélèrent l'accès aux index : l'**ICP** (vérifier les conditions sur l'index avant de lire la ligne), le **Rowid Filtering** (filtrer par rowids issus d'un index sélectif), et le **Loose Index Scan** (`GROUP BY` clairsemé).  
> - Jusqu'ici, un **scan d'index en ordre inversé** (typique d'un `ORDER BY … DESC`) **désactivait** l'ICP et le Rowid Filtering, et le Loose Index Scan ne gérait pas les clés `DESC`.  
> - **🆕 La série 12.x lève ces restrictions** : ICP (MDEV-34413) et Rowid Filtering (MDEV-36094) s'appliquent désormais aux scans inversés, et le Loose Index Scan exploite les parties de clé `DESC` (MDEV-32732).  
> - **Conséquence** : les requêtes `ORDER BY … DESC` et les **index descendants** ne perdent plus ces accélérations — moins de retours à la table, moins d'I/O.  
> - On observe ces optimisations dans `EXPLAIN` (`Using index condition`, `Using rowid filter`, `Using index for group-by`) et on les pilote via `optimizer_switch`.
