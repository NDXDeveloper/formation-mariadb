# 5.5.3 — Index pour ORDER BY et GROUP BY

> **Chapitre 5 — Index et Performance** · Section 5.5.3  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

On pense d'abord aux index pour **filtrer** ; or ils servent tout autant à **trier** et à **regrouper**. La raison tient à une propriété déjà connue : un index B-Tree maintient ses valeurs **dans l'ordre** ([section 5.2.1](02.1-btree.md)). MariaDB peut donc réutiliser cet ordre déjà établi pour satisfaire un `ORDER BY` ou un `GROUP BY` **sans opération supplémentaire** — en évitant un tri coûteux (le *filesort*) ou une table temporaire. C'est un usage moins évident des index, mais à fort impact, en particulier sur les requêtes paginées et les agrégations.

---

## ORDER BY : éviter le *filesort*

### Le principe

Quand les colonnes d'un `ORDER BY` correspondent à un index (ou à son **préfixe gauche**), MariaDB lit les lignes **dans l'ordre de l'index** : le résultat est déjà trié, aucune étape de tri n'est nécessaire.

```sql
-- Avec un index sur date_commande, le tri est « gratuit »
SELECT * FROM commandes ORDER BY date_commande LIMIT 20;
```

À l'inverse, en l'absence d'index utilisable, MariaDB doit **trier explicitement** les lignes : `EXPLAIN` ([section 5.7](07-analyse-plans-execution.md)) le signale par la mention **`Using filesort`** dans la colonne `Extra`. L'objectif de l'indexation pour le tri est précisément de **faire disparaître ce `Using filesort`**.

### Le motif clé : filtre par égalité + tri

Le cas le plus courant combine un filtre et un tri : *« les commandes de tel client, des plus récentes aux plus anciennes »*.

```sql
SELECT * FROM commandes WHERE client_id = 42 ORDER BY date_commande;
```

Un **index composite** sur `(client_id, date_commande)` sert alors les **deux** opérations à la fois : l'**égalité** sur `client_id` fixe une position dans l'index, et `date_commande` y est déjà trié — pas de *filesort*.

```sql
CREATE INDEX idx_client_date ON commandes (client_id, date_commande);
```

L'**ordre des colonnes** d'un tel index est déterminant (colonne d'égalité d'abord, colonne de tri ensuite) ; c'est le sujet central de la [section 5.6](06-index-composites.md).

### Le gain décisif : ORDER BY + LIMIT

C'est ici que l'index fait toute la différence. Pour un « Top N » ou une pagination :

```sql
SELECT * FROM commandes ORDER BY date_commande DESC LIMIT 10;
```

- **Sans index** : MariaDB doit trier **toutes** les lignes correspondantes, puis n'en garder que 10.
- **Avec index** : il lit l'index dans l'ordre voulu et **s'arrête après 10 lignes**. Le coût devient quasi indépendant de la taille de la table.

Sur des requêtes paginées fréquentes, ce mécanisme est souvent le gain de performance le plus spectaculaire.

### Le sens du tri (ASC / DESC)

Un index peut être parcouru **en avant ou à l'envers** : un index sur `(date_commande)` sert donc aussi bien `ORDER BY date_commande` que `ORDER BY date_commande DESC`. Le seul cas problématique est le tri **multi-colonnes de directions mixtes** :

```sql
SELECT * FROM commandes ORDER BY client_id ASC, date_commande DESC;
```

Pour l'éviter, il faut un index aux **directions concordantes**, possible grâce aux véritables index descendants de MariaDB :

```sql
CREATE INDEX idx_mixte ON commandes (client_id ASC, date_commande DESC);
```

Sans cet alignement, MariaDB retombe sur un *filesort*.

---

## GROUP BY : éviter le tri et la table temporaire

### Le principe

Le regroupement profite de la même propriété : si les lignes sont déjà ordonnées par la colonne de regroupement, les valeurs identiques sont **adjacentes**, et MariaDB peut les agréger au fil de la lecture, sans table temporaire ni tri.

```sql
-- GROUP BY sur une colonne indexée : regroupement via l'ordre de l'index
SELECT statut, COUNT(*) FROM commandes GROUP BY statut;
```

Sans index exploitable, un `GROUP BY` recourt typiquement à une **table temporaire** et/ou à un tri — `EXPLAIN` affiche alors **`Using temporary; Using filesort`**, le symptôme à combattre.

### Tight scan et loose index scan

MariaDB dispose de deux manières d'exploiter un index pour un `GROUP BY` :

- le **balayage d'index complet** (*tight index scan*) : il lit l'index dans l'ordre et regroupe les valeurs adjacentes, économisant le tri ;
- le **balayage clairsemé** (*loose index scan*), bien plus puissant : lorsqu'on regroupe sur un **préfixe** d'un index avec des agrégats limités (typiquement `MIN`/`MAX`), MariaDB **saute** directement d'une valeur distincte à la suivante, sans lire la plupart des lignes.

```sql
-- Loose index scan : la dernière commande de chaque client
SELECT client_id, MAX(date_commande)
FROM commandes
GROUP BY client_id;
-- Avec un index (client_id, date_commande), EXPLAIN affiche « Using index for group-by »
```

La mention **`Using index for group-by`** dans `EXPLAIN` confirme un *loose index scan*. Cette optimisation, et notamment son extension aux clés **descendantes**, est détaillée dans la nouveauté de la [section 5.8.1](08.1-optimisations-scans-inverses.md).

---

## Le rôle central de l'index composite

Un même index composite peut couvrir **filtrage, jointure, tri et regroupement** à la fois. Bien conçu, il évite à la fois le balayage, le *filesort* et la table temporaire pour une requête complète. Tout repose sur l'**ordre des colonnes**, qui obéit à des règles précises développées en [section 5.6](06-index-composites.md). Un principe directeur s'en dégage déjà : placer les colonnes d'**égalité** en tête, puis la colonne de **tri** ou de **regroupement**.

---

## ⚠️ Pièges courants

- **Trier ou regrouper sur une expression.** Comme pour le filtrage ([section 5.5.1](05.1-index-colonnes-filtrees.md)), `ORDER BY YEAR(date_commande)` ou `GROUP BY UPPER(nom)` empêche l'usage de l'index. Solution : indexer une [colonne générée](../18-fonctionnalites-avancees/04-virtual-generated-columns.md).

- **Une plage qui « casse » l'ordre.** Avec un index `(client_id, date_commande)`, une **égalité** sur `client_id` préserve l'ordre de `date_commande` — mais une **plage** le rompt :

  ```sql
  SELECT * FROM commandes WHERE client_id = 100 ORDER BY date_commande;  -- ✅ pas de filesort
  SELECT * FROM commandes WHERE client_id > 100 ORDER BY date_commande;  -- ⚠️ filesort
  ```

  Dès qu'une colonne fait l'objet d'une **plage**, les colonnes situées **après elle** dans l'index ne sont plus globalement triées (cf. [section 5.6](06-index-composites.md)).

- **Tri sur des colonnes de tables différentes.** Dans une jointure, un `ORDER BY` mêlant des colonnes de plusieurs tables est difficile à servir par un seul index et entraîne souvent un *filesort*.

---

## Toujours vérifier

Le juge de paix reste **`EXPLAIN`** ([section 5.7](07-analyse-plans-execution.md)). On y traque les mentions **`Using filesort`** et **`Using temporary`** (à éliminer) et l'on confirme les bons cas par **`Using index for group-by`** (loose index scan) ou l'absence de tri superflu.

---

> ### 📝 À retenir  
>  
> - Un index B-Tree étant **déjà trié**, MariaDB le réutilise pour servir `ORDER BY` et `GROUP BY` **sans tri ni table temporaire**.  
> - **`ORDER BY`** : viser la disparition de `Using filesort`. Le motif gagnant est l'index **composite** « **égalité + colonne de tri** » ; combiné à **`LIMIT`**, l'index permet de **s'arrêter tôt** — gain majeur pour le Top N et la pagination.  
> - **Directions** : un index se lit en avant ou à l'envers ; seul un tri **multi-colonnes mixte** exige un index aux directions concordantes.  
> - **`GROUP BY`** : l'index évite la table temporaire ; le **loose index scan** (`Using index for group-by`) saute la plupart des lignes pour les agrégats `MIN`/`MAX` sur un préfixe (voir 5.8.1).  
> - **Pièges** : trier/regrouper sur une **expression**, et une **plage** sur une colonne qui **casse l'ordre** des colonnes suivantes.  
> - **Vérifier** avec `EXPLAIN` : éliminer `Using filesort` et `Using temporary`.
