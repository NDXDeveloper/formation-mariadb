# 5.2.1 — B-Tree : le standard

> **Chapitre 5 — Index et Performance** · Section 5.2.1  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

La [section 5.1](01-fonctionnement-index.md) a détaillé *comment* fonctionne un B-Tree. Cette section adopte l'angle complémentaire : le B-Tree **en tant qu'outil**, celui que l'on emploie au quotidien. Si on l'appelle « le standard », ce n'est pas un hasard — c'est le type d'index **par défaut**, le plus polyvalent, et de très loin le plus utilisé. La plupart du temps, lorsqu'on crée un index sans réfléchir au type, on crée un B-Tree, et c'est le bon choix.

L'objectif ici est de savoir **reconnaître les requêtes qu'un B-Tree accélère**, et de comprendre les facteurs qui font qu'il sera réellement exploité ou non.

---

## Pourquoi le B-Tree est « le standard »

Trois raisons font du B-Tree le choix par défaut :

1. **C'est l'option implicite.** Une commande aussi simple que `CREATE INDEX idx_montant ON commandes (montant)` crée un index B-Tree. Il est inutile de préciser quoi que ce soit ; le mot-clé `USING BTREE` est facultatif.

   ```sql
   -- Ces deux commandes sont strictement équivalentes
   CREATE INDEX idx_montant ON commandes (montant);
   CREATE INDEX idx_montant ON commandes (montant) USING BTREE;
   ```

2. **Vous l'utilisez déjà partout.** Toute **clé primaire**, toute contrainte **`UNIQUE`** et tout index créé pour soutenir une **clé étrangère** reposent sur un B-Tree. Avant même d'ajouter le moindre index « manuel », votre schéma en est déjà constellé.

3. **Il est universellement polyvalent.** Le B-Tree fonctionne avec la quasi-totalité des **types de données scalaires** — entiers, décimaux, chaînes, dates et heures — et il prend en charge la **plus grande variété d'opérations** parmi toutes les structures d'index. C'est cette polyvalence qui en fait le type par défaut : les autres structures (Hash, Full-Text, Spatial, Vector) ne sont préférables que pour des besoins spécialisés bien précis.

---

## Les opérations accélérées par un B-Tree

Parce qu'il maintient les valeurs **triées**, le B-Tree sert efficacement toutes les opérations qui tirent parti de cet ordre. Voici les principales, illustrées sur une table `commandes`.

### Recherche par égalité

Le cas le plus fréquent. L'index mène directement à la valeur recherchée.

```sql
SELECT * FROM commandes WHERE statut = 'expediee';
```

### Appartenance à un ensemble (`IN`)

Traité comme une série de recherches d'égalité, chacune bénéficiant de l'index.

```sql
SELECT * FROM commandes WHERE id IN (101, 102, 103);
```

### Recherche de plage

L'un des points forts du B-Tree : on descend jusqu'à la borne inférieure, puis on **suit le chaînage des feuilles** jusqu'à la borne supérieure.

```sql
SELECT * FROM commandes WHERE montant BETWEEN 50 AND 200;
SELECT * FROM commandes WHERE date_commande >= '2026-01-01';
```

### Recherche par préfixe de chaîne

Tant que le **début** de la valeur est connu, l'ordre alphabétique permet d'exploiter l'index.

```sql
SELECT * FROM clients WHERE nom LIKE 'Dur%';   -- ✅ utilise l'index
SELECT * FROM clients WHERE nom LIKE '%dur%';  -- ❌ joker en tête : index inutile
```

### Tri sans opération supplémentaire (`ORDER BY`)

Comme l'index restitue les valeurs déjà ordonnées, MariaDB peut éviter une étape de tri explicite (le *filesort*). C'est particulièrement précieux combiné à `LIMIT`.

```sql
-- Avec un index sur date_commande, le tri est « gratuit »
SELECT * FROM commandes ORDER BY date_commande LIMIT 20;
```

### Regroupement (`GROUP BY`)

Pour les mêmes raisons, un `GROUP BY` sur une colonne indexée peut s'appuyer sur l'ordre de l'index plutôt que de trier ou de hacher les données.

### Valeurs extrêmes (`MIN` / `MAX`)

Le minimum se trouve dans la première feuille, le maximum dans la dernière : leur lecture est quasi instantanée.

```sql
SELECT MIN(date_commande), MAX(date_commande) FROM commandes;
```

### Recherche de valeurs `NULL`

Point souvent mal connu : dans MariaDB, **les index B-Tree d'InnoDB incluent les valeurs `NULL`**. Une condition `IS NULL` peut donc, elle aussi, exploiter l'index.

```sql
SELECT * FROM commandes WHERE date_livraison IS NULL;
```

---

## Sélectivité et cardinalité : la clé de l'efficacité

Un index B-Tree n'est utile que s'il permet d'**écarter une grande partie des lignes**. Cette capacité se mesure par deux notions liées :

- La **cardinalité** : le nombre de **valeurs distinctes** dans la colonne.
- La **sélectivité** : le rapport entre cette cardinalité et le nombre total de lignes (de 0 à 1).

Plus la sélectivité est **élevée**, plus l'index est efficace. Un index sur une colonne `email` (presque toutes les valeurs sont uniques) est excellent. À l'inverse, un index sur une colonne `actif` ne contenant que deux valeurs (`0` / `1`) est de **faible sélectivité** : chaque valeur correspond à la moitié de la table, et l'index n'apporte presque rien.

C'est pourquoi **l'optimiseur peut délibérément ignorer un index** lorsqu'une condition correspond à une trop grande proportion de lignes : effectuer des milliers d'accès indexés (avec, pour les index secondaires, un retour à la table à chaque fois) coûterait plus cher qu'un simple balayage séquentiel. Le B-Tree donne donc le meilleur de lui-même sur des **colonnes à forte cardinalité**.

L'optimiseur fonde ces décisions sur des **statistiques** qu'il convient de tenir à jour. La commande `ANALYZE TABLE` recalcule ces statistiques de cardinalité ; on l'aborde au [chapitre 11](../11-administration-configuration/06.2-analyze-table.md). La manière dont l'optimiseur utilise concrètement ces estimations s'observe avec `EXPLAIN`, présenté en [section 5.7](07-analyse-plans-execution.md).

## Variantes utiles du B-Tree

### Index préfixe sur les longues chaînes

Sur une colonne textuelle volumineuse, il est possible d'indexer uniquement les **premiers caractères**, ce qui réduit la taille de l'index tout en restant efficace pour l'égalité et le préfixe :

```sql
CREATE INDEX idx_nom ON clients (nom(20));   -- indexe les 20 premiers caractères
```

Pour les colonnes `BLOB` et `TEXT`, cette longueur de préfixe est d'ailleurs **obligatoire**. Le compromis est clair : un préfixe trop court réduit la sélectivité (davantage de valeurs partagent le même début).

### Ordre ascendant ou descendant

Par défaut, un index est trié en ordre **ascendant**. MariaDB sait néanmoins le parcourir à l'envers pour servir un `ORDER BY ... DESC`. On peut aussi définir explicitement un index **descendant**, utile pour certains tris ou index composites mêlant les deux sens :

```sql
CREATE INDEX idx_date_desc ON commandes (date_commande DESC);
```

### Index composite (multi-colonnes)

Un B-Tree peut porter sur **plusieurs colonnes** à la fois. L'**ordre des colonnes** y est déterminant (principe du *left-most prefix*), ce qui fait l'objet d'une section dédiée :

```sql
CREATE INDEX idx_client_date ON commandes (client_id, date_commande);
```

Le sujet est approfondi en [section 5.6](06-index-composites.md).

## Ce que le B-Tree ne sait pas faire

Pour mémoire (ces limites sont détaillées en [section 5.1](01-fonctionnement-index.md)), un index B-Tree reste **inopérant** dans trois cas :

- **Joker en tête** : `LIKE '%terme'` ne peut pas l'exploiter.
- **Fonction appliquée à la colonne** : `WHERE YEAR(date_commande) = 2026` neutralise l'index sur `date_commande`. On contourne le problème en réécrivant la condition sous forme de plage, ou en indexant une [colonne générée](../18-fonctionnalites-avancees/04-virtual-generated-columns.md).
- **Faible sélectivité** : sur une colonne à très peu de valeurs distinctes, l'index est rarement rentable.

Dans le doute, la règle est simple : **vérifier** que l'index est réellement utilisé, à l'aide de `EXPLAIN` ([section 5.7](07-analyse-plans-execution.md)).

---

> ### 📝 À retenir  
>  
> - Le B-Tree est le **type d'index par défaut** : `CREATE INDEX … ON table (colonne)` en crée un, sans mot-clé particulier.  
> - Vous l'utilisez **déjà partout** : clés primaires, contraintes `UNIQUE` et index de clés étrangères sont tous des B-Tree.  
> - Il accélère un **large éventail d'opérations** : égalité, `IN`, plages, préfixe de chaîne, `ORDER BY`, `GROUP BY`, `MIN`/`MAX`, et même `IS NULL` (les `NULL` sont indexés dans InnoDB).  
> - Son efficacité dépend de la **sélectivité** : il excelle sur les colonnes à **forte cardinalité** ; sur une colonne à peu de valeurs distinctes, l'optimiseur préférera souvent un balayage complet.  
> - Variantes utiles : **index préfixe** sur les longues chaînes, ordre **descendant**, et index **composite** (traité en 5.6).
