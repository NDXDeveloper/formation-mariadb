# 5.2.3 — Full-Text : la recherche textuelle

> **Chapitre 5 — Index et Performance** · Section 5.2.3  
> Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

Rechercher un **mot à l'intérieur d'un texte** — un terme dans un article, une référence dans une description de produit, une expression dans des commentaires — est un problème que ni le B-Tree ni le Hash ne savent traiter. Ces structures indexent une **valeur de colonne entière** ; elles ignorent ce qui se passe *à l'intérieur* de cette valeur. L'index **full-text** répond précisément à ce besoin : il indexe les **mots** d'un texte, et permet d'y effectuer des recherches rapides, pertinentes et classées par score.

---

## Pourquoi `LIKE '%mot%'` ne suffit pas

La tentation, pour chercher un mot, est d'écrire :

```sql
SELECT * FROM articles WHERE contenu LIKE '%galera%';
```

Cette approche fonctionne sur de petits volumes, mais elle est mauvaise à plusieurs titres :

- **Aucun index exploitable.** Le joker en tête (`%`) interdit l'usage d'un index B-Tree ([section 5.2.1](02.1-btree.md)) : MariaDB doit balayer toute la table.
- **Recherche de sous-chaîne, pas de mot.** `LIKE '%galera%'` trouverait aussi « galeras » ou un fragment au milieu d'un autre mot, sans distinguer les frontières de mots.
- **Aucun classement.** Tous les résultats sont équivalents ; impossible de savoir lesquels sont les plus pertinents.
- **Pas de gestion linguistique.** Ni mots vides, ni recherche multi-termes, ni pondération.

L'index full-text apporte une réponse propre à chacun de ces points.

---

## La structure : l'index inversé

Un index full-text repose sur une structure d'**index inversé**. Plutôt que d'associer chaque ligne à son contenu, il fait l'inverse : il associe **chaque mot à la liste des lignes qui le contiennent**.

```
"galera"      → documents 3, 17, 42, 108 …
"réplication" → documents 3, 9, 42, 77 …
"cluster"     → documents 17, 42, 91 …
```

C'est exactement le principe de l'index alphabétique à la fin d'un livre : un mot, puis les pages où il apparaît. Pour répondre à une recherche, MariaDB n'a plus qu'à consulter les listes des mots recherchés — sans jamais relire les textes eux-mêmes. La construction de cet index implique une phase de **tokenisation** : le texte est découpé en mots selon les espaces et la ponctuation.

---

## Créer un index full-text

Contrairement au B-Tree, un index full-text doit être déclaré **explicitement** avec le mot-clé `FULLTEXT`. Il s'applique aux colonnes **textuelles** (`CHAR`, `VARCHAR`, `TEXT`) et peut porter sur **plusieurs colonnes** à la fois. Il est pris en charge par les moteurs **InnoDB**, **MyISAM** et **Aria**.

```sql
-- À la création de la table
CREATE TABLE articles (
  id      INT AUTO_INCREMENT PRIMARY KEY,
  titre   VARCHAR(200) NOT NULL,
  contenu TEXT         NOT NULL,
  FULLTEXT INDEX ft_article (titre, contenu)
) ENGINE = InnoDB;

-- Ou a posteriori
ALTER TABLE articles ADD FULLTEXT INDEX ft_article (titre, contenu);
CREATE FULLTEXT INDEX ft_article ON articles (titre, contenu);
```

---

## Interroger avec `MATCH … AGAINST`

Un index full-text **ne s'utilise pas** avec les opérateurs SQL habituels. Le seul moyen de l'exploiter est la construction `MATCH(colonnes) AGAINST(termes)`. La liste de colonnes du `MATCH` doit **correspondre exactement** à celle d'un index full-text défini sur la table.

```sql
SELECT id, titre
FROM articles
WHERE MATCH(titre, contenu) AGAINST('réplication galera');
```

L'expression `MATCH … AGAINST` retourne un **score de pertinence** : un nombre flottant positif, d'autant plus élevé que la ligne correspond bien aux termes (zéro signifiant aucune correspondance). On peut afficher ce score et trier dessus :

```sql
SELECT id, titre,
       MATCH(titre, contenu) AGAINST('réplication galera') AS pertinence
FROM articles
WHERE MATCH(titre, contenu) AGAINST('réplication galera')
ORDER BY pertinence DESC;
```

---

## Les modes de recherche

### Mode langage naturel (par défaut)

Le mode par défaut interprète la chaîne comme un **ensemble de mots** et calcule un score de pertinence pour chaque ligne. Une requête simple retourne d'ailleurs les résultats triés par **pertinence décroissante** ; pour un contrôle explicite — et fiable en présence d'autres clauses (jointures, `GROUP BY`…) —, on trie soi-même avec `ORDER BY` sur l'expression `MATCH`. On peut le rendre explicite :

```sql
... AGAINST('réplication galera' IN NATURAL LANGUAGE MODE)
```

### Mode booléen

Le mode booléen (`IN BOOLEAN MODE`) introduit des **opérateurs** offrant un contrôle fin sur la présence et le poids des termes :

| Opérateur | Effet |
|-----------|-------|
| `+mot` | Le mot **doit** être présent |
| `-mot` | Le mot doit être **absent** |
| `mot*` | **Troncature** : recherche par préfixe |
| `"expression exacte"` | Séquence de mots **exacte** |
| `>mot` / `<mot` | **Augmente** / **diminue** le poids du mot |
| `(...)` | **Groupement** de termes |
| `~mot` | Contribue **négativement** au score, sans exclure |

```sql
-- Articles contenant « galera », sans « postgres », avec préfixe « réplicat… »
SELECT id, titre
FROM articles
WHERE MATCH(titre, contenu)
      AGAINST('+galera -postgres réplicat*' IN BOOLEAN MODE);
```

À la différence du mode langage naturel, le mode booléen **ne trie pas** automatiquement par pertinence : il faut ajouter un `ORDER BY` explicite si l'on souhaite un classement.

### Avec extension de requête

Le mode `WITH QUERY EXPANSION` effectue une **double recherche** : une première passe identifie les lignes les plus pertinentes, puis les mots qui y figurent enrichissent une seconde recherche, afin de remonter des documents thématiquement proches même s'ils ne contiennent pas les termes initiaux. Utile pour élargir le rappel, au prix d'un peu de précision.

```sql
... AGAINST('cluster' WITH QUERY EXPANSION)
```

## Comportements et réglages à connaître

Plusieurs paramètres conditionnent ce qui est indexé et retrouvé :

- **Longueur minimale des mots.** Les mots trop courts sont ignorés à l'indexation. Le seuil dépend du moteur : `innodb_ft_min_token_size` (par défaut **3**) pour InnoDB, `ft_min_word_len` (par défaut **4**) pour MyISAM et Aria. Modifier ces variables impose de **reconstruire** l'index.
- **Mots vides (*stopwords*).** Les mots très courants (articles, conjonctions…) sont écartés par défaut, via une liste de mots vides intégrée. On peut la personnaliser ou la désactiver.
- **Le seuil des 50 % (MyISAM uniquement).** En mode langage naturel **sur MyISAM**, un mot présent dans plus de la moitié des lignes est considéré comme trop commun et reçoit une pertinence nulle. Cette règle **ne s'applique pas** à InnoDB, ni au mode booléen — un point d'attention classique lors d'une migration de moteur.
- **Langues asiatiques.** Le découpage par espaces ne convient pas aux langues sans séparateur de mots (chinois, japonais, coréen). ⚠️ **Point d'attention important** : contrairement à **MySQL** — qui fournit pour cela un parseur **`ngram`** (`WITH PARSER ngram`) —, **MariaDB ne dispose d'aucun parseur `ngram` ou `MeCab` intégré**. La clause `WITH PARSER ngram` y échoue d'ailleurs avec une erreur (`ERROR 1128 : Function 'ngram' is not defined`), ces parseurs restant à ce jour des demandes d'évolution **non implémentées** (MDEV-10267, MDEV-10268). Pour de la recherche full-text sur du **texte CJK** avec MariaDB, on se tourne vers le moteur de stockage **Mroonga** (paquet séparé, fondé sur Groonga et conçu pour le CJK) ou vers une **tokenisation côté application**.

---

## Limites et alternatives

L'index full-text couvre très bien la **recherche par mots-clés au sein de la base**, mais il a ses bornes. Pour un simple test de sous-chaîne ponctuel, un `LIKE` ou une [expression régulière](../04-concepts-avances-sql/11-expressions-regulieres.md) reste plus direct. Pour des besoins très avancés (analyse linguistique poussée, facettes, tolérance aux fautes à grande échelle), un moteur de recherche dédié peut s'avérer plus adapté.

Surtout, il faut distinguer la recherche full-text de la **recherche sémantique**. Le full-text repose sur la **correspondance de mots** : il trouve les textes contenant les termes demandés, mais ignore le *sens*. Pour retrouver des contenus **conceptuellement proches** (synonymes, reformulations), c'est l'index vectoriel **HNSW** ([section 5.3](03-index-vector-hnsw.md)) qui entre en jeu. Les deux approches sont complémentaires et peuvent même se combiner (recherche hybride, abordée au [chapitre 20](../20-cas-usage-architectures/09.4-hybrid-search.md)).

---

> ### 📝 À retenir  
>  
> - L'index full-text indexe les **mots** d'un texte via un **index inversé** (mot → lignes), là où le B-Tree indexe la valeur entière.  
> - Il remplace avantageusement `LIKE '%mot%'` : il exploite un index, raisonne par **mots** (et non par sous-chaînes) et fournit un **score de pertinence**.  
> - Il se déclare avec `FULLTEXT` (sur `CHAR`/`VARCHAR`/`TEXT`, moteurs InnoDB, MyISAM, Aria) et s'interroge **uniquement** via `MATCH(colonnes) AGAINST(termes)`.  
> - Trois modes : **langage naturel** (par défaut, classé par pertinence), **booléen** (`+ - * " ()`…), et **extension de requête**.  
> - Réglages clés : **longueur minimale des mots** (`innodb_ft_min_token_size` = 3 ; `ft_min_word_len` = 4), **mots vides**, **seuil des 50 %** propre à MyISAM. Pour le **CJK**, MariaDB n'a **pas** le parseur `ngram` de MySQL (→ moteur **Mroonga** ou tokenisation applicative).  
> - Full-Text = correspondance de **mots** ; pour la proximité de **sens**, voir l'index vectoriel HNSW (5.3).
