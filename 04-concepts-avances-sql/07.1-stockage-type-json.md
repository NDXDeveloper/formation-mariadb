# 4.7.1 Stockage et type de données JSON

> **Chapitre 4 — Concepts Avancés SQL** · Niveau : Avancé  
> Alias `JSON` depuis 10.2.7 · `CHECK` automatique depuis 10.4.3 · Référence : **MariaDB 12.3 LTS**

Cette section détaille **comment MariaDB stocke réellement le JSON** — un point dont découle tout le reste : la validation, les performances et la compatibilité. Nous avons annoncé l'essentiel au § 4.7 ; entrons maintenant dans le concret.

## 🎯 Objectif de la section

Comprendre que le type `JSON` est un alias de `LONGTEXT`, savoir déclarer et valider une colonne JSON, saisir les conséquences du stockage verbatim, et connaître les limites ainsi que les enjeux de compatibilité avec MySQL.

## `JSON`, un alias de `LONGTEXT`

En MariaDB, le type `JSON` n'est pas un type de stockage binaire dédié : c'est un **alias de `LONGTEXT`**, doté de la collation `utf8mb4_bin`. Ce choix répond à deux objectifs : assurer la compatibilité avec le type `JSON` de MySQL (réplication basée sur les instructions, lecture de *dumps*), et éviter un type qui s'écarterait du standard SQL — les mesures de MariaDB indiquant par ailleurs une performance au moins équivalente.

La collation **`utf8mb4_bin`** n'est pas anodine : le JSON est sensible à la casse et exact à l'octet près (les clés et les valeurs textuelles distinguent majuscules et minuscules), ce qu'une comparaison binaire garantit.

Conséquence à connaître : à l'introspection, la colonne se présente comme un `LONGTEXT`, **pas** comme un type `json`. Certains outils ou ORM peuvent donc ne pas la reconnaître spontanément comme du JSON, et s'appuient pour cela sur la présence de la contrainte `JSON_VALID`.

## Déclarer une colonne JSON

La déclaration est directe :

```sql
CREATE TABLE produits (
    id        INT PRIMARY KEY,
    attributs JSON
);
```

Cette écriture est équivalente à un `LONGTEXT` assorti de la collation et d'une contrainte de validité explicites :

```sql
attributs LONGTEXT CHARACTER SET utf8mb4 COLLATE utf8mb4_bin
          CHECK (JSON_VALID(attributs))
```

Un `SHOW CREATE TABLE` révèle d'ailleurs cette réalité sous-jacente :

```sql
SHOW CREATE TABLE produits;
-- `attributs` longtext CHARACTER SET utf8mb4 COLLATE utf8mb4_bin DEFAULT NULL
--   CHECK (json_valid(`attributs`))
```

## La validation automatique (`JSON_VALID`)

Depuis MariaDB 10.4.3, déclarer une colonne avec l'alias `JSON` **ajoute automatiquement** une contrainte `CHECK (JSON_VALID(colonne))`. Tout document mal formé est alors rejeté à l'insertion ou à la mise à jour :

```sql
INSERT INTO produits VALUES (1, 'ceci n''est pas du JSON');
-- ERROR 4025 (23000): CONSTRAINT `attributs` failed
```

La fonction `JSON_VALID(x)` renvoie `1` si `x` est un document valide, `0` sinon, et `NULL` si `x` est `NULL`.

> 💡 **Le cas `NULL`.** Une colonne JSON accepte la valeur `NULL` (sauf si elle est déclarée `NOT NULL`). En effet, `JSON_VALID(NULL)` vaut `NULL`, et une contrainte `CHECK` ne **échoue que sur `FALSE`** : une condition `UNKNOWN` la laisse passer. C'est une application directe de la logique ternaire (§ 4.6).

Si l'on souhaite stocker du texte qui n'est pas nécessairement du JSON, ou éviter le léger coût de validation à l'écriture, on utilise un `LONGTEXT` simple, sans l'alias ni la contrainte.

## Le stockage verbatim

Point capital : MariaDB conserve le texte JSON **tel quel**. Espaces, indentation, ordre des clés et même **clés en double** sont préservés à l'identique :

```sql
INSERT INTO produits VALUES (2, '{ "a":1, "b":2, "a":3 }');
SELECT attributs FROM produits WHERE id = 2;
-- Renvoie exactement : { "a":1, "b":2, "a":3 }
```

Ce comportement diffère de celui de MySQL, qui parse le document en une représentation binaire **normalisée** (suppression des espaces, dédoublonnage des clés, réordonnancement éventuel). Il en résulte deux conséquences :

- La **taille** stockée inclut le formatage : deux documents sémantiquement identiques mais formatés différemment occupent un espace différent.
- L'**égalité par défaut est textuelle** : comparer deux colonnes JSON avec `=` compare les chaînes brutes. Ainsi `'{"a":1}'` et `'{ "a": 1 }'` sont considérés comme **différents**. Pour une comparaison *sémantique*, il faut d'abord ramener les documents à une forme canonique (fonction de normalisation, § 4.7.2).

## Taille et limites

Un `LONGTEXT` peut contenir jusqu'à environ **4 Go**, ce qui borne la taille d'un document JSON. En pratique, le transfert est aussi limité par la variable `max_allowed_packet`, et des documents volumineux restent peu commodes. Le JSON convient à des structures de taille raisonnable, non à de très gros objets.

## Indexation : pas directement sur le JSON

Parce que la donnée est du texte, le moteur doit **analyser la chaîne entière** à chaque accès à un élément, et il n'existe pas d'index portant directement sur les chemins JSON. Pour interroger efficacement un champ enfoui, on l'**extrait dans une colonne virtuelle (générée)** que l'on indexe ensuite — technique détaillée au § 4.10 (et au § 18.4 pour les colonnes générées).

## Compatibilité et migration MySQL

Les formats de stockage divergeant (binaire normalisé côté MySQL, texte côté MariaDB), les colonnes JSON des deux SGBD **ne sont pas directement interchangeables**. MariaDB sait néanmoins lire les champs JSON issus de MySQL (à partir de la 10.5.7), et l'alias `JSON` permet à la réplication basée sur les instructions et au chargement de *dumps* MySQL de fonctionner. Ces aspects sont approfondis au chapitre 19 (migration).

## Points clés à retenir

En MariaDB, le type `JSON` est un **alias de `LONGTEXT`** en collation `utf8mb4_bin` — la colonne apparaît d'ailleurs comme `longtext` à l'introspection. Une contrainte **`CHECK (JSON_VALID(...))`** est ajoutée automatiquement (depuis 10.4.3) et rejette les documents invalides, tout en laissant passer `NULL` (logique ternaire, § 4.6). Le texte est stocké **verbatim** : espaces, ordre et clés en double sont conservés, d'où une **égalité textuelle** par défaut (la comparaison sémantique exige une normalisation, § 4.7.2). La taille est bornée par `LONGTEXT` (≈ 4 Go), l'indexation passe par des **colonnes virtuelles** (§ 4.10), et la divergence de format avec MySQL relève de la migration (chapitre 19).
