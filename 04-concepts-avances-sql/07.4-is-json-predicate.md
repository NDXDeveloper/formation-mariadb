# 4.7.4 Prédicat `IS JSON` (standard SQL) et suppression de la limite de profondeur 🆕

> **Chapitre 4 — Concepts Avancés SQL** · Niveau : Avancé  
> 🆕 `IS JSON` : nouveau en **MariaDB 12.3** (12.3.1, MDEV-37072) · Limite de profondeur retirée en **12.x** (MDEV-32854)  
> Référence : **MariaDB 12.3 LTS**

La série 12.x apporte au JSON deux améliorations notables : un **prédicat `IS JSON`** conforme au standard SQL, et la **suppression de la limite de profondeur** historique des documents. Ces deux nouveautés sont bien présentes dans la 12.3 LTS.

## 🎯 Objectif de la section

Maîtriser le prédicat `IS JSON` (validité, contrainte de type, unicité des clés), savoir le situer par rapport à `JSON_VALID`, et comprendre l'incidence de la levée de la limite de profondeur.

## Le prédicat `IS JSON` (nouveau en 12.3)

Issu du standard SQL:2016, le prédicat `IS JSON` vérifie qu'une expression est un document JSON valide. Sa syntaxe complète permet en outre de **contraindre le type** et de **contrôler l'unicité des clés** :

```text
expr IS [ NOT ] JSON [ { VALUE | ARRAY | OBJECT | SCALAR } ] [ { WITH | WITHOUT } UNIQUE [ KEYS ] ]
```

Dans sa forme la plus simple, il renvoie vrai ou faux selon la validité :

```sql
SELECT '{"a":1}'     IS JSON;       -- 1 (TRUE)
SELECT 'pas du json' IS JSON;       -- 0 (FALSE)
SELECT 'abc'         IS NOT JSON;   -- 1 (TRUE)
```

**Contrainte de type.** Le mot-clé optionnel restreint le type attendu — `VALUE` (n'importe quel JSON, valeur par défaut), `ARRAY`, `OBJECT` ou `SCALAR` :

```sql
SELECT '[1,2,3]'  IS JSON ARRAY;    -- 1
SELECT '{"a":1}'  IS JSON ARRAY;    -- 0
SELECT '{"a":1}'  IS JSON OBJECT;   -- 1
SELECT '42'       IS JSON SCALAR;   -- 1
```

**Unicité des clés.** La clause `WITH UNIQUE KEYS` exige que les clés d'un objet soient toutes distinctes ; `WITHOUT UNIQUE KEYS` (valeur par défaut) tolère les doublons. Ce contrôle prend tout son sens en MariaDB, où le stockage **verbatim** conserve les clés en double (§ 4.7.1) :

```sql
SELECT '{"a":1,"a":2}' IS JSON WITH UNIQUE KEYS;     -- 0 (clé "a" en double)
SELECT '{"a":1,"b":2}' IS JSON WITH UNIQUE KEYS;     -- 1
SELECT '{"a":1,"a":2}' IS JSON;                       -- 1 (doublons tolérés par défaut)
```

Étant un prédicat booléen, `IS JSON` s'emploie là où une condition est attendue — dans un `WHERE` pour filtrer, ou dans une contrainte `CHECK` comme alternative lisible à `JSON_VALID` :

```sql
SELECT * FROM journaux WHERE charge_utile IS JSON;
```

Notamment, une contrainte `... CHECK (donnees IS JSON WITH UNIQUE KEYS)` permet de **rejeter les clés en double**, ce que la contrainte `JSON_VALID` ajoutée automatiquement (§ 4.7.1) ne fait pas.

## `IS JSON` ou `JSON_VALID` ?

Pour la simple validité, `expr IS JSON` et `JSON_VALID(expr) = 1` sont équivalents. Leurs différences :

- **`IS JSON`** est la forme **prédicat**, conforme au standard SQL et donc plus portable ; elle apporte en une seule expression les contraintes de **type** et d'**unicité des clés**.
- **`JSON_VALID`** est la forme **fonction**, propre à MariaDB/MySQL et disponible de plus longue date ; c'est elle qui sous-tend la validation automatique des colonnes `JSON`.

Avant la 12.3, `JSON_VALID` était le seul moyen de tester la validité ; l'arrivée de `IS JSON` renforce la **compatibilité avec Oracle** (qui possède ce prédicat) et avec le standard SQL — un atout en migration (chapitre 19).

## La suppression de la limite de profondeur (nouveau en 12.x)

Jusqu'à la série 12.x, les fonctions JSON imposaient une **profondeur d'imbrication maximale de 32 niveaux** ; au-delà, une erreur (code 4040) était levée. Cette limite, non documentée, gênait les documents profondément imbriqués.

Elle a été **entièrement supprimée** (MDEV-32854), une évolution apparue dans la série 12.x (previews 12.1/12.2) et présente dans la 12.3 LTS. MariaDB accepte désormais des documents et des chemins JSON à imbrication **quasi illimitée**. Le changement est **transparent** : aucune modification de syntaxe ni de logique applicative n'est requise, tout continue de fonctionner à l'identique, sans le plafond.

Les bénéfices sont directs : prise en charge native des charges utiles d'API très imbriquées, modèles de données plus expressifs, et évolution de schéma sans devoir aplatir les structures pour contourner une limite artificielle. La fonction `JSON_DEPTH` continue, elle, de renvoyer la profondeur effective d'un document.

## Points clés à retenir

La série 12.x dote MariaDB du prédicat standard **`IS JSON`** (12.3.1) : `expr IS [NOT] JSON [VALUE|ARRAY|OBJECT|SCALAR] [WITH|WITHOUT UNIQUE KEYS]` teste la validité, contraint éventuellement le type, et — fait notable au vu du stockage verbatim — peut **exiger l'unicité des clés**. Il complète `JSON_VALID` (la forme fonction) en offrant une syntaxe conforme au standard, utile en filtre comme en contrainte `CHECK`. Par ailleurs, la **limite de 32 niveaux d'imbrication a été supprimée** (MDEV-32854) de façon transparente, autorisant des documents quasi illimités en profondeur.
