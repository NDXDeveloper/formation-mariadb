# 7.4.1 Segmented key cache (`aria_pagecache_segments`) 🆕

> **Chapitre 7 — Moteurs de Stockage** · MariaDB 12.3 LTS  
> **Nouveauté de la série 12.x** — introduite dans MariaDB Community Server **12.1**, donc présente dans la 12.3 LTS.

## Une réponse à la concurrence sur le page cache d'Aria

Comme vu en §7.4, Aria met en cache ses pages de données et d'index dans un **page cache**, dimensionné par `aria_pagecache_buffer_size`. Jusqu'à la série 12.x, ce cache était une structure **unique**. La nouveauté apportée par MariaDB 12.1 permet désormais de le **segmenter** — de le découper en plusieurs caches indépendants — grâce à la variable `aria_pagecache_segments`, afin de **réduire la contention** sous forte charge concurrente. C'est l'équivalent, pour le page cache d'Aria, du *segmented key cache* que MyISAM connaissait de longue date.

## Le problème : la contention sur le verrou du cache

Un cache unique est protégé par un **unique verrou** (mutex). Le problème apparaît en forte concurrence : **chaque** appel aux fonctions du cache doit acquérir ce verrou, si bien que les threads se le disputent **même lorsqu'ils ne font que lire des pages déjà présentes** dans le cache et qu'ils détiennent déjà des verrous partagés sur le fichier. Ce point de sérialisation limite la montée en charge sur les serveurs à nombreux cœurs : on a beau avoir des données en mémoire et des lectures qui pourraient se faire en parallèle, l'accès au cache lui-même devient le goulet d'étranglement.

## La solution : segmenter le cache

Un **cache segmenté** est une collection de N sous-caches indépendants — les **segments** —, chacun doté de **son propre verrou**. L'idée est simple et efficace : une opération qui n'a besoin que d'une seule page n'acquiert que le verrou **du segment auquel cette page est affectée**, et non un verrou global. Les threads qui travaillent sur des segments différents ne se gênent donc plus, ce qui améliore nettement les performances en accès parallèle.

L'affectation d'une page à un segment est **déterministe** : elle est calculée à partir du numéro de fichier (et de la position de la page dans le fichier) et reste **toujours la même** pour une page donnée. Concrètement, les pages d'un même fichier de table sont rattachées à un segment, et les différents fichiers se répartissent entre les segments. Ce mécanisme n'est pas une invention nouvelle — MyISAM proposait déjà un *segmented key cache* — mais la série 12.x l'apporte au page cache d'Aria.

## La variable `aria_pagecache_segments`

`aria_pagecache_segments` définit le **nombre de segments** du page cache :

- sa **valeur par défaut est `1`** : un cache unique, soit le comportement antérieur (la nouveauté est donc sans effet tant qu'on ne l'active pas) ;
- chaque segment a une taille de `aria_pagecache_buffer_size / aria_pagecache_segments` ;
- les segments sont **créés au démarrage** d'Aria : la variable se règle dans le fichier de configuration et sa modification **nécessite un redémarrage** ;
- elle a été **introduite dans MariaDB Community Server 12.1** (et MariaDB Enterprise Server 11.4.8-5 / 11.8).

```ini
[mariadb]
aria_pagecache_buffer_size = 1G    # taille TOTALE du page cache (données + index)
aria_pagecache_segments    = 8     # 8 segments ⇒ ~128 Mo chacun ; défaut = 1 (cache unique)
```

## Qui en bénéficie

L'intérêt dépasse les seules tables créées explicitement avec `ENGINE = Aria`. Comme rappelé en §7.4, **toutes les instances MariaDB utilisent Aria** pour leurs **tables temporaires internes** et la plupart de leurs **tables système**. La réduction de contention profite donc à ces usages internes aussi : sur un serveur fortement sollicité en parallèle, segmenter le page cache d'Aria peut améliorer des opérations qui passent par des tables temporaires (tris, regroupements volumineux) ou par les tables système, **même si l'on ne crée jamais de table Aria soi-même**.

## Bien le dimensionner

Un point essentiel à comprendre : la segmentation améliore le **parallélisme**, pas la **capacité**. La taille totale du cache (`aria_pagecache_buffer_size`) **ne change pas** ; elle est simplement **répartie** entre les segments. Il en découle un arbitrage :

- avec **trop de segments** rapportés à un cache total trop petit, **chaque segment devient minuscule**, ce qui dégrade le taux de présence en cache et peut annuler le bénéfice (voire nuire) ;
- il faut donc **augmenter `aria_pagecache_buffer_size` en proportion** lorsqu'on augmente le nombre de segments, afin que chaque segment reste d'une taille utile.

En règle générale : on **dimensionne le nombre de segments selon la concurrence** attendue (cœurs/threads actifs sur Aria) tout en gardant un cache total généreux, puis on **mesure** l'effet via les compteurs `Aria_pagecache_reads` (lectures physiques, à garder faibles) rapportés à `Aria_pagecache_read_requests` (lectures logiques). La valeur par défaut `1` convient à une faible concurrence ; on n'envisage la segmentation que sur des serveurs à nombreux cœurs soumis à une activité Aria (ou temporaire/système) intensément parallèle.

## Vérification

```sql
SHOW VARIABLES LIKE 'aria_pagecache%';      -- voir buffer_size et segments
SHOW GLOBAL STATUS LIKE 'Aria_pagecache_read%';  -- reads vs read_requests
```

## Ce qu'il faut retenir

- `aria_pagecache_segments` (nouveauté **12.1**, présente en 12.3) découpe le **page cache d'Aria** en segments indépendants pour **réduire la contention** du verrou de cache en forte concurrence.
- Chaque page est affectée de façon **déterministe** à un segment (selon le fichier), et chaque segment fait `aria_pagecache_buffer_size / segments`.
- La **valeur par défaut est `1`** (cache unique, comportement antérieur) ; la variable se règle au **démarrage** (redémarrage requis).
- Le bénéfice profite aussi aux **tables temporaires et système** (qui utilisent Aria), donc à tous les utilisateurs en charge parallèle.
- La segmentation accroît le **parallélisme, pas la capacité** : augmentez `aria_pagecache_buffer_size` en proportion pour que chaque segment reste assez grand, et **mesurez** le résultat.
