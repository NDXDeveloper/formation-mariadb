# 7.2.1 Caractéristiques : ACID, FK, Row-level locking

> **Chapitre 7 — Moteurs de Stockage** · MariaDB 12.3 LTS

Trois caractéristiques résument ce qui fait d'InnoDB un moteur **transactionnel et robuste**, et ce qui le distingue des moteurs non transactionnels comme MyISAM ou Aria : la conformité **ACID**, la prise en charge des **clés étrangères** et le **verrouillage au niveau ligne**. Cette section présente ces trois caractéristiques du point de vue d'InnoDB — *ce qu'il garantit et comment*. La théorie générale des transactions et de la concurrence (niveaux d'isolation, MVCC, deadlocks) est, elle, développée au chapitre 6 ; les journaux qui sous-tendent ces garanties sont détaillés en §7.2.3.

## ACID : un moteur pleinement transactionnel

Une transaction est une suite d'opérations traitée comme une unité indivisible. InnoDB garantit les quatre propriétés désignées par l'acronyme **ACID** :

- **Atomicité** (*Atomicity*) — une transaction est « tout ou rien ». Si elle est validée (`COMMIT`), toutes ses modifications sont appliquées ; si elle est annulée (`ROLLBACK`) ou interrompue, aucune ne l'est. InnoDB s'appuie sur les **undo logs** pour défaire les modifications non validées (voir §7.2.3).
- **Cohérence** (*Consistency*) — une transaction fait passer la base d'un état valide à un autre état valide. InnoDB y contribue en faisant respecter les contraintes définies : clé primaire, `UNIQUE`, `NOT NULL`, `CHECK` et clés étrangères. Une transaction qui violerait une contrainte échoue plutôt que de laisser des données incohérentes.
- **Isolation** (*Isolation*) — les transactions concurrentes ne se perturbent pas indûment. InnoDB propose plusieurs **niveaux d'isolation** (`REPEATABLE READ` par défaut) et met en œuvre le **MVCC** pour permettre des lectures cohérentes sans blocage. Le détail des niveaux figure en §6.3 et le MVCC en §6.6.
- **Durabilité** (*Durability*) — une fois une transaction validée, ses modifications survivent à un arrêt brutal du serveur. InnoDB l'assure via le **redo log** (journalisation par écriture anticipée) et le mécanisme de *flush* sur disque (voir §7.2.3).

À titre d'exemple, le transfert d'un montant entre deux comptes doit être atomique : soit les deux écritures sont validées, soit aucune.

```sql
START TRANSACTION;
  UPDATE comptes SET solde = solde - 100 WHERE id = 1;
  UPDATE comptes SET solde = solde + 100 WHERE id = 2;
COMMIT;   -- ROLLBACK annulerait l'intégralité de la transaction
```

C'est précisément ce qu'un moteur **non transactionnel** ne sait pas faire : avec MyISAM ou Aria, il n'existe ni `COMMIT` ni `ROLLBACK` au sens d'InnoDB, et une interruption en cours peut laisser les données dans un état partiel. Le degré de durabilité d'InnoDB est par ailleurs réglable via le paramètre `innodb_flush_log_at_trx_commit`, qui arbitre entre sécurité maximale et débit — un point traité en §7.2.4.

> **Isolation par instantané (par défaut).** Le niveau `REPEATABLE READ` applique par défaut une **isolation par instantané** (`innodb_snapshot_isolation = ON`) — un comportement en place **depuis MariaDB 11.6** (donc déjà en 11.8 LTS comme en 12.3), et **non** une nouveauté de la 12.3. Voir §6.9.

## Les clés étrangères (Foreign Keys)

InnoDB **fait respecter l'intégrité référentielle** déclarée par les clés étrangères. C'est une différence majeure avec MyISAM, qui accepte la syntaxe `FOREIGN KEY` mais l'ignore silencieusement : aucune vérification n'y est faite. Avec InnoDB, une clé étrangère garantit qu'une ligne « enfant » ne peut référencer qu'une ligne « parent » existante, ce qui empêche l'apparition de lignes orphelines.

```sql
CREATE TABLE clients (
  id   BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  nom  VARCHAR(120) NOT NULL
) ENGINE = InnoDB;

CREATE TABLE commandes (
  id         BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  client_id  BIGINT UNSIGNED NOT NULL,
  total      DECIMAL(12,2)   NOT NULL,
  CONSTRAINT fk_commande_client
    FOREIGN KEY (client_id) REFERENCES clients (id)
    ON DELETE CASCADE
    ON UPDATE RESTRICT
) ENGINE = InnoDB;
```

Les clauses `ON DELETE` et `ON UPDATE` définissent l'**action référentielle** appliquée lorsqu'une ligne parent est supprimée ou que sa clé change :

| Action | Effet sur les lignes enfants |
|--------|------------------------------|
| `CASCADE` | propage la suppression / la mise à jour aux lignes enfants |
| `SET NULL` | met les colonnes de la clé étrangère à `NULL` (qui doivent être nullable) |
| `RESTRICT` | rejette l'opération s'il existe des lignes enfants correspondantes (vérification immédiate) |
| `NO ACTION` | dans InnoDB, équivaut à `RESTRICT` (pas de vérification différée) |
| `SET DEFAULT` | reconnu par l'analyseur mais **rejeté** par InnoDB |

Quelques règles pratiques accompagnent les clés étrangères :

- La colonne référencée du côté parent doit être **indexée** (généralement la clé primaire ou un index `UNIQUE`). Du côté enfant, si aucun index n'existe sur les colonnes de la clé étrangère, InnoDB en crée un automatiquement — un index utile aux performances, les vérifications référentielles s'appuyant dessus.
- La variable de session `foreign_key_checks` permet de **désactiver temporairement** les vérifications, ce qui est précieux lors d'un chargement en masse ou de la restauration d'un dump où l'ordre des tables n'est pas garanti :

  ```sql
  SET foreign_key_checks = 0;
  -- ... import / chargement en masse ...
  SET foreign_key_checks = 1;
  ```

- Une clé étrangère ne peut relier que des **tables InnoDB** : on ne crée pas de contrainte référentielle entre deux moteurs différents (voir aussi §7.1 sur le mélange de moteurs).

> 🆕 **Note de version (12.1).** Depuis MariaDB **12.1** (MDEV-28933), les **noms de contraintes de clés étrangères** ne doivent être uniques **qu'au sein d'une table** (et non plus à l'échelle de toute la base) : deux tables d'une même base peuvent donc réutiliser le même nom de contrainte, ce qui facilite l'usage d'ORM et de scripts DDL générés. Voir §18.12.

## Le verrouillage au niveau ligne (Row-level locking)

InnoDB pose ses verrous **au niveau de la ligne**, et non de la table entière. C'est ce qui lui permet de soutenir une **forte concurrence** : deux transactions qui modifient des lignes différentes d'une même table peuvent progresser en parallèle. À l'inverse, MyISAM et Aria pratiquent un **verrouillage au niveau table** : toute écriture verrouille la table entière, sérialisant les écrivains et limitant fortement le débit en écriture concurrente.

### Lectures verrouillantes et lectures non verrouillantes

Un point essentiel : grâce au **MVCC**, un `SELECT` ordinaire est une **lecture cohérente non verrouillante**. Il lit un instantané des données (reconstitué au besoin à partir des undo logs) sans poser de verrou, si bien que **les lecteurs ne bloquent pas les écrivains** et réciproquement, pour ces lectures simples (voir §6.6).

Lorsqu'une transaction a besoin de verrouiller explicitement les lignes qu'elle lit — par exemple pour les modifier ensuite en toute sécurité — elle utilise une **lecture verrouillante** :

```sql
START TRANSACTION;
  -- Pose un verrou exclusif sur la ligne lue, conservé jusqu'au COMMIT
  SELECT solde FROM comptes WHERE id = 1 FOR UPDATE;
  UPDATE comptes SET solde = solde - 50 WHERE id = 1;
COMMIT;
```

`FOR UPDATE` pose un verrou exclusif ; **`LOCK IN SHARE MODE`** pose un verrou partagé (la syntaxe `FOR SHARE` de MySQL 8.0 n'est **pas** reconnue par MariaDB). Ces instructions sont approfondies en §6.4.

### Les types de verrous

InnoDB combine plusieurs types de verrous. À ce stade, il suffit d'en connaître les grandes catégories ; leur mécanique fine relève du chapitre 6 :

- **Verrou partagé (S) et verrou exclusif (X)** : plusieurs verrous S peuvent coexister sur une même ressource (lectures concurrentes), mais un verrou X est exclusif. Leur compatibilité se résume ainsi :

  | Demandé \ Détenu | S | X |
  |------------------|---|---|
  | **S** | compatible | incompatible |
  | **X** | incompatible | incompatible |

- **Record lock** : verrou sur un enregistrement d'index.
- **Verrou d'intervalle (gap lock)** : verrou sur l'espace entre deux enregistrements d'index, qui empêche l'insertion dans cet intervalle — un mécanisme destiné à éviter les lignes fantômes sous `REPEATABLE READ`.
- **Next-key lock** : combinaison d'un *record lock* et d'un *gap lock*, mode de verrouillage par défaut des parcours d'intervalle sous `REPEATABLE READ`.
- **Verrous d'intention (IS, IX)** : verrous au niveau table signalant l'intention de poser des verrous de ligne, afin de coordonner verrous de table et verrous de ligne.

Le recours aux *gap locks* et *next-key locks* dépend du niveau d'isolation : `READ COMMITTED` les réduit fortement, là où `REPEATABLE READ` les emploie pour garantir l'absence de fantômes (voir §6.3).

### Un point déterminant : InnoDB verrouille des enregistrements d'index

InnoDB ne verrouille pas « la ligne » dans l'absolu : il verrouille des **enregistrements d'index**. La conséquence est concrète et importante : si une requête de modification ne peut s'appuyer sur aucun index adapté, InnoDB doit parcourir l'index clustered et risque de **verrouiller bien plus d'enregistrements que les seules lignes ciblées**. Un bon index ne sert donc pas seulement la performance de lecture : il **réduit l'étendue des verrous** et, par ricochet, les conflits de concurrence. C'est l'une des raisons pour lesquelles l'indexation (chapitre 5) et le verrouillage sont indissociables.

### Deadlocks

Le verrouillage au niveau ligne rend possibles les **interblocages** (deadlocks) : deux transactions s'attendent mutuellement. InnoDB **détecte automatiquement** ces situations et annule l'une des transactions (la « victime ») pour débloquer les autres, en renvoyant une erreur que l'application doit gérer en réessayant. La détection, le réglage (`innodb_lock_wait_timeout`, etc.) et les bonnes pratiques pour limiter les deadlocks sont traités en §6.5.

## Comment ces caractéristiques se combinent

Prises ensemble, ces trois caractéristiques expliquent pourquoi InnoDB est le moteur par défaut. La **conformité ACID** apporte la fiabilité (rien n'est validé à moitié, rien n'est perdu après validation) ; les **clés étrangères** garantissent la cohérence du modèle de données au niveau du moteur ; le **verrouillage au niveau ligne**, conjugué au **MVCC**, permet à de nombreuses sessions de travailler simultanément avec un minimum de contention. C'est cet équilibre entre robustesse et concurrence qui fait d'InnoDB un choix sûr pour les charges OLTP et mixtes.

## Ce qu'il faut retenir

- InnoDB est **pleinement ACID** : atomicité (undo), cohérence (contraintes), isolation (niveaux + MVCC), durabilité (redo).
- À la différence de MyISAM, InnoDB **fait respecter les clés étrangères** ; les actions `CASCADE`, `SET NULL`, `RESTRICT` et `NO ACTION` sont utilisables, `SET DEFAULT` est rejetée, et `foreign_key_checks` permet de suspendre les vérifications le temps d'un import.
- Le **verrouillage au niveau ligne** (vs niveau table chez MyISAM/Aria) autorise une forte concurrence ; les `SELECT` simples sont non verrouillants grâce au MVCC.
- InnoDB **verrouille des enregistrements d'index** : une requête sans index adapté peut verrouiller bien plus que prévu — l'indexation conditionne donc aussi la concurrence.
- Les **deadlocks** sont détectés automatiquement et résolus par annulation d'une transaction victime (détail en §6.5).
