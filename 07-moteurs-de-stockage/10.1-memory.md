# 7.10.1 Memory : Tables en RAM

> **Chapitre 7 — Moteurs de Stockage** · MariaDB 12.3 LTS

## Des données en mémoire vive : rapide, mais volatil

Le moteur **Memory** (anciennement **HEAP**) stocke le contenu des tables **en mémoire vive** plutôt que sur disque. Il en résulte des accès **très rapides**, au prix d'une contrepartie majeure : les données sont **volatiles**. Il est donc « idéal pour des caches de données issues d'autres tables, ou pour des zones de travail temporaires », mais **inadapté au stockage permanent**.

## Volatilité : structure persistante, données éphémères

Il faut bien distinguer deux choses. La **définition** de la table (`.frm`) est, elle, **sur disque** : la table survit donc à un redémarrage. Mais **son contenu, lui, disparaît** : après un redémarrage du serveur, les tables Memory sont recréées **vides**. De même, une coupure de courant ou une panne matérielle entraîne la perte des données.

Si l'on a besoin de retrouver des données dans une table Memory au démarrage, on peut la **repeupler** à l'aide de l'option serveur `--init-file` (un script SQL exécuté au démarrage). À noter qu'en réplication, ce vidage au redémarrage peut demander une attention particulière, la table redevenant vide côté serveur redémarré.

Précisons aussi qu'une table Memory **non temporaire** est une table **ordinaire, partagée par toutes les connexions** — à ne pas confondre avec une `CREATE TEMPORARY TABLE`, qui est propre à une session. Le moteur Memory décrit ici concerne des tables persistantes (en structure) mais stockées en RAM.

## Index : Hash par défaut, B-Tree en option

Le moteur Memory accepte deux types d'index, et c'est un choix qui compte :

- **Hash** (le **type par défaut**) : excellent pour les **recherches par égalité** (`=`, `IN`), mais inadapté aux plages et au tri ;
- **B-Tree** : pour les requêtes par **plage**, les `ORDER BY`, et pour éviter une dégradation des performances que les index Hash subissent sur des colonnes à **faible cardinalité** (beaucoup de doublons).

On précise l'algorithme avec `USING HASH` ou `USING BTREE`. Une table Memory peut comporter jusqu'à **64 index**, **16 colonnes par index** et une longueur de clé maximale de **3072 octets**.

## Limites de types et de comportement

La simplicité du moteur impose plusieurs restrictions :

- **pas de colonnes `BLOB` ni `TEXT`** : leur présence empêche la création (ou la conversion) d'une table Memory ;
- les types à longueur variable comme `VARCHAR` sont stockés en **longueur fixe**, ce qui peut **gaspiller de la mémoire** pour de grandes colonnes ;
- **verrouillage au niveau table** : la concurrence en écriture est limitée (les écrivains se sérialisent) ;
- **ni transactions, ni clés étrangères, ni résistance aux pannes** (données volatiles).

En revanche, `AUTO_INCREMENT` est pris en charge, et — **contrairement à MySQL 8.0**, qui a retiré le partitionnement générique — une table Memory **peut être partitionnée** dans MariaDB, qui conserve ce mécanisme pour les moteurs sans partitionnement natif (MyISAM, Aria, Memory, Archive).

## Dimensionner et gérer la mémoire

La taille totale d'une table Memory ne peut **dépasser `max_heap_table_size`**. Cette valeur s'applique à une table **à sa création**, et de nouveau aux tables existantes **au redémarrage**. L'option de table `MAX_ROWS` fournit une **indication** sur le nombre de lignes attendu : ce n'est pas une limite stricte et elle ne permet pas de dépasser `max_heap_table_size`, mais le moteur l'utilise, avec cette dernière, pour estimer la mémoire à allouer.

Un point pratique important : lorsqu'on **supprime des lignes, l'espace n'est pas libéré automatiquement**. Pour le récupérer sans détruire la table, on la **reconstruit** ou on la **vide** :

```sql
SET max_heap_table_size = 64 * 1024 * 1024;   -- 64 Mo

CREATE TABLE cache_sessions (
  session_id CHAR(32)     NOT NULL,
  donnees    VARCHAR(255) NOT NULL,
  maj_le     INT UNSIGNED NOT NULL,
  PRIMARY KEY USING HASH (session_id),   -- recherche par égalité (Hash, le défaut)
  KEY        USING BTREE (maj_le)         -- requêtes par plage / tri
) ENGINE = MEMORY;
```

```sql
-- L'espace n'est pas libéré par DELETE ; pour le récupérer :
TRUNCATE TABLE cache_sessions;        -- vide la table et libère la mémoire
-- ou reconstruire la table
ALTER TABLE cache_sessions ENGINE = MEMORY;
```

## Le rôle de Memory dans les tables temporaires internes

Au-delà des tables que l'on crée explicitement, le moteur Memory joue un rôle **interne** déterminant : dans **MariaDB**, c'est lui qui héberge les **tables temporaires internes en mémoire** — celles que le serveur matérialise pour traiter certaines requêtes (tris, regroupements, résultats intermédiaires). C'est une différence avec **MySQL 8.0**, qui utilise par défaut le moteur **TempTable** à cette fin.

Cette table temporaire interne reste en mémoire **tant qu'elle ne dépasse pas la plus petite des deux valeurs `tmp_table_size` et `max_heap_table_size`** ; au-delà — ou si elle requiert des colonnes `BLOB`/`TEXT` que Memory ne gère pas —, elle est **convertie en table sur disque** (Aria, §7.4), ce qui est plus lent. Le réglage de `tmp_table_size` et `max_heap_table_size` influence donc les performances de requêtes courantes, et pas seulement les tables Memory explicites.

## Quand utiliser Memory

Le moteur Memory convient lorsque la **perte des données au redémarrage est acceptable** et que la **rapidité** prime : caches alimentés depuis d'autres tables, tables de correspondance chargées au démarrage, données de travail éphémères. Il ne convient **pas** :

- au **stockage permanent** ou à toute donnée que l'on ne peut pas se permettre de perdre ;
- aux données comportant des colonnes **`BLOB`/`TEXT`** ;
- à la **forte concurrence en écriture** (verrouillage de table) ;
- à tout ce qui exige **transactions, clés étrangères ou durabilité** → **InnoDB** (§7.2).

À garder en tête : pour des données **durables** mais que l'on veut servir depuis la RAM, un **buffer pool InnoDB** correctement dimensionné (§7.2.2) garde déjà les données chaudes en mémoire, souvent sans qu'il soit nécessaire de recourir au moteur Memory. La grille de décision complète figure en §7.8.

## Ce qu'il faut retenir

- Le moteur **Memory** stocke les données **en RAM** : très rapide, mais **volatil** (les tables sont **vidées au redémarrage**, leur structure persistant).
- Index **Hash par défaut** (égalité) ou **B-Tree** (plages, tri) ; **pas de `BLOB`/`TEXT`**, `VARCHAR` en longueur fixe, **verrouillage de table**, ni transactions ni FK.
- La taille est plafonnée par **`max_heap_table_size`** ; l'espace **n'est pas libéré par `DELETE`** (utiliser `TRUNCATE`, `DROP` ou un `ALTER … ENGINE=MEMORY`).
- Dans MariaDB, Memory héberge aussi les **tables temporaires internes en mémoire** (contrairement au TempTable de MySQL 8.0), qui **débordent sur Aria** au-delà de `tmp_table_size`/`max_heap_table_size`.
- À réserver aux **caches et données éphémères** ; pour du durable, c'est **InnoDB** (un buffer pool bien dimensionné garde déjà les données chaudes en RAM).
