# 7.10.4 CONNECT : Accès données externes

> **Chapitre 7 — Moteurs de Stockage** · MariaDB 12.3 LTS

## Le couteau suisse des données externes

Le moteur **CONNECT** permet à MariaDB d'**accéder à des données externes de toutes sortes** — fichiers, autres bases de données, sources NoSQL, requêtes REST — **comme s'il s'agissait de tables MariaDB ordinaires**. Point capital : **les données ne sont pas chargées dans MariaDB**, elles **restent dans leur source** ; CONNECT s'y connecte au moment où l'on interroge la table.

C'est une forme de **gestion de données externes (MED)**, mais réalisée « de façon bien plus simple » que la norme — CONNECT **ne prétend pas implémenter la norme ISO SQL/MED** (contrairement à Spider, §7.10.3) et n'introduit aucune extension du langage SQL. C'est un **plugin** (`ha_connect`), **introduit en MariaDB 10.0**, présent sous Windows comme sous Linux mais **non chargé par défaut**. Son terrain de prédilection : l'**intégration de données** hétérogènes, les tâches de type ETL, et l'interrogation en SQL de sources variées sans avoir à les importer.

## Le principe : des tables qui pointent vers l'extérieur

Une table CONNECT ne contient pas de données : elle **décrit où et comment** trouver une source externe. Chaque table porte un **type** (`TABLE_TYPE`) qui indique la nature de la source — un fichier CSV, un fichier JSON, une table d'une autre base via ODBC, etc. Les requêtes lisent (et, pour beaucoup de types, écrivent) directement la source.

## Installation

CONNECT s'active comme plugin :

```sql
INSTALL SONAME 'ha_connect';
```

## Une grande variété de types de tables

La force de CONNECT tient à l'éventail de ses `TABLE_TYPE`. On peut les regrouper en trois familles.

**Fichiers** — accès en **lecture/écriture** à des fichiers de la plupart des formats courants :

- **CSV** / **FMT** (texte délimité ou formaté), **DOS** / **FIX** (longueur fixe), **BIN**, **DBF** (dBASE) ;
- **XML**, **JSON**, **INI** — avec des facilités de requêtage « NoSQL » sur ces formats.

**Bases de données et NoSQL** — les **quatre principaux types externes** :

- **ODBC** : accès à tout SGBD doté d'un pilote ODBC (PostgreSQL, SQL Server, Oracle…) ; sur Linux, via `unixODBC` ;
- **JDBC** : accès à tout SGBD doté d'un connecteur JDBC (au moyen d'une classe Java servant d'interface) ;
- **MYSQL** : la **manière privilégiée** d'accéder à une table d'un autre serveur MariaDB/MySQL ;
- **MONGO** : accès aux collections MongoDB via le pilote C — type **non disponible dans les distributions binaires** (il faut compiler MariaDB depuis les sources).

**Tables spéciales et virtuelles** — **TBL** (liste de tables, pour fédérer plusieurs tables en une), **PROXY**, **XCOL**, **OCCUR**, **PIVOT**, **ZIP**, **VIR**, **DIR**, etc.

La liste complète des types est : `DOS, FIX, BIN, CSV, FMT, XML, JSON, INI, DBF, VEC, ODBC, JDBC, MYSQL, TBL, PROXY, XCOL, OCCUR, PIVOT, ZIP, VIR, DIR, WMI, MAC, OEM`.

## Créer une table CONNECT

On précise `ENGINE = CONNECT`, le `TABLE_TYPE`, et les options propres au type. Pour un **fichier**, l'option clé est `FILE_NAME` :

```sql
CREATE TABLE clients_csv (
  id    INT,
  nom   VARCHAR(80),
  ville VARCHAR(80)
) ENGINE = CONNECT
TABLE_TYPE = CSV
FILE_NAME  = '/data/clients.csv'
HEADER     = 1
SEP_CHAR   = ',';
```

Pour une **base externe**, on indique la source (`CONNECTION`) et la table distante (`TABNAME`). Détail appréciable : on peut **omettre la définition des colonnes**, CONNECT les **découvrant automatiquement** (fonction *discovery*) :

```sql
-- Accès à une table PostgreSQL via ODBC (colonnes auto-découvertes)
CREATE TABLE produits_externes
ENGINE     = CONNECT
TABLE_TYPE = ODBC
CONNECTION = 'DSN=ma_source_postgres'
TABNAME    = 'public.produits';
```

```sql
-- Accès à une table d'un autre serveur MariaDB/MySQL (type MYSQL)
CREATE TABLE commandes_distantes
ENGINE     = CONNECT
TABLE_TYPE = MYSQL
CONNECTION = 'mysql://user:motdepasse@192.168.0.50:3306/crm/commandes';
```

## Accès aux SGBD externes : pushdown et serveurs fédérés

Pour les types ODBC, JDBC, MYSQL et MONGO, CONNECT établit une connexion à la source et lui **envoie une requête** à exécuter. On peut aller plus loin que l'accès table par table grâce à l'option **`SRCDEF`**, qui fournit une **requête personnalisée poussée vers la source distante** — par exemple une agrégation calculée à distance plutôt que rapatriée :

```sql
... SRCDEF='SELECT pays, COUNT(*) AS clients FROM custnum WHERE %s GROUP BY pays';
```

Par ailleurs, une table ODBC/JDBC peut puiser ses informations de connexion dans un **serveur fédéré** (`CREATE SERVER`), ce qui centralise la configuration : l'emplacement de la source peut changer à un seul endroit sans avoir à recréer toutes les tables qui l'utilisent. Enfin, lorsqu'une source contient des types que CONNECT ne gère pas nativement, il en effectue une **conversion automatique** vers un type compatible (variable `connect_type_conv`).

## Lecture et écriture

À la différence du moteur S3 (en lecture seule, §7.6), CONNECT offre un **accès en lecture/écriture** aux fichiers externes des formats courants : on peut créer un fichier CSV, y insérer des lignes, etc. Selon les capacités de la source, des opérations de modification sont également possibles sur les tables de bases de données externes.

## Quand utiliser CONNECT — et la distinction avec Spider

CONNECT s'impose dès qu'il faut **intégrer en SQL des données qui vivent ailleurs** : interroger un fichier CSV/JSON/XML produit par une autre application, lire une table PostgreSQL ou SQL Server via ODBC/JDBC, accéder à des collections MongoDB, consommer une API REST, ou bâtir des traitements d'intégration sans importer les données.

La distinction avec **Spider** (§7.10.3) est nette : Spider est dédié au **sharding et à la fédération de tables MariaDB/MySQL** (et s'appuie sur SQL/MED) ; **CONNECT** vise un périmètre bien plus large — **toute source externe**, y compris fichiers et SGBD tiers via ODBC/JDBC, avec écriture dans les fichiers. (Pour accéder spécifiquement à une autre table MariaDB/MySQL, le type `MYSQL` de CONNECT ou le moteur FederatedX sont des options.) À noter aussi que les tables JSON/XML de CONNECT diffèrent des **fonctions JSON natives** de MariaDB (§4.7) : CONNECT décrit un fichier externe comme une table, là où les fonctions JSON manipulent une colonne `JSON` stockée localement.

Les principaux points d'attention sont la **performance** (qui dépend de la source et du réseau) et la **richesse des options**, qui rend CONNECT puissant mais demande de la rigueur. La grille de choix des moteurs figure en §7.8.

## Ce qu'il faut retenir

- **CONNECT** rend interrogeables comme des tables MariaDB des **données externes** très variées (fichiers CSV/JSON/XML…, bases via **ODBC/JDBC/MYSQL**, **MongoDB**, REST) — **sans les charger** dans MariaDB.
- C'est un **plugin** (`ha_connect`, depuis 10.0, **non chargé par défaut**) ; on choisit la source via **`TABLE_TYPE`** et les options (`FILE_NAME` pour les fichiers, `CONNECTION`/`TABNAME` pour les bases).
- Les **colonnes peuvent être auto-découvertes** ; l'option **`SRCDEF`** pousse une requête vers la source distante, et un **serveur fédéré** centralise la connexion.
- CONNECT offre un accès en **lecture/écriture** aux fichiers (contrairement à S3, en lecture seule).
- À distinguer de **Spider** (sharding de tables MariaDB/MySQL) : **CONNECT** vise **toute source externe** hétérogène (fichiers comme SGBD tiers), avec accès en lecture/écriture aux fichiers.
