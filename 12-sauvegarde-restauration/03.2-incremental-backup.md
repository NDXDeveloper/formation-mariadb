# 12.3.2 Incremental backup

> **Chapitre 12 — Sauvegarde et Restauration** · MariaDB 12.3 LTS

---

## Introduction

Une **sauvegarde incrémentale** avec Mariabackup ne copie que les **pages de données modifiées** depuis une sauvegarde de référence, au lieu de recopier l'intégralité de l'instance. Elle suppose donc qu'une **sauvegarde complète existe déjà** ([12.3.1](03.1-full-backup.md)) : c'est sur elle que viendront s'appuyer les incréments. L'intérêt est direct, et rejoint la théorie de la [section 12.1](01-strategies-sauvegarde.md) : des sauvegardes plus rapides, plus légères, donc réalisables plus fréquemment — ce qui améliore le RPO.

Cette section détaille le mécanisme (fondé sur le LSN d'InnoDB), la réalisation des incréments, puis l'étape délicate de leur **préparation** — qui diffère sensiblement de celle d'une sauvegarde complète.

---

## Le mécanisme : le LSN

Chaque page de données InnoDB porte un **LSN** (*Log Sequence Number*), un numéro de séquence que le moteur **incrémente à chaque modification** d'une ligne. Pour réaliser un incrément, Mariabackup compare le LSN le plus récent de la sauvegarde de référence aux LSN actuels de la base, et ne copie que les pages **ayant pris du retard** — c'est-à-dire celles qui ont changé. Ces différences sont stockées sous forme de **fichiers delta**.

Le chaînage est matérialisé par le fichier `mariadb_backup_checkpoints`, présent dans chaque répertoire de sauvegarde. La sauvegarde complète y inscrit par exemple :

```
backup_type = full-backuped
from_lsn = 0
to_lsn = 1635102
last_lsn = 1635102
```

Le premier incrément, lui, démarre exactement là où la sauvegarde complète s'est arrêtée :

```
backup_type = incremental
from_lsn = 1635102      ← égal au to_lsn de la sauvegarde complète
to_lsn = 1635114
last_lsn = 1635114
```

C'est ce `from_lsn` / `to_lsn` qui définit la **base de référence** d'un incrément, et c'est exactement le levier qui distingue, en pratique, une stratégie incrémentale d'une stratégie différentielle.

---

## Phase 1 — Réaliser les sauvegardes incrémentales

Une fois la **sauvegarde complète** réalisée (voir [12.3.1](03.1-full-backup.md)), on produit un incrément avec les options `--backup` et `--target-dir` habituelles, auxquelles s'ajoute **`--incremental-basedir`**, qui indique le répertoire de la sauvegarde servant de point de référence. Comme toujours, le répertoire cible doit être **vide**.

Premier incrément, prenant la sauvegarde complète comme base :

```bash
mariabackup --backup \
  --target-dir=/sauvegardes/inc1 \
  --incremental-basedir=/sauvegardes/base \
  --user=mariabackup --password=mot_de_passe
```

Pour un incrément suivant, on prend comme base de référence le **répertoire de l'incrément précédent** :

```bash
mariabackup --backup \
  --target-dir=/sauvegardes/inc2 \
  --incremental-basedir=/sauvegardes/inc1 \
  --user=mariabackup --password=mot_de_passe
```

On obtient ainsi une **chaîne** : `base → inc1 → inc2 → …`, chaque incrément ne contenant que les changements survenus depuis le précédent.

### Incrémental ou différentiel : tout se joue sur le `--incremental-basedir`

C'est ici que la théorie de la [section 12.1](01-strategies-sauvegarde.md) prend tout son sens. Mariabackup ne propose pas de mode « différentiel » distinct : la différence tient **uniquement** au répertoire passé à `--incremental-basedir`.

| Stratégie | `--incremental-basedir` pointe vers… | Contenu de chaque sauvegarde |
|-----------|--------------------------------------|------------------------------|
| **Incrémentale** (chaîne) | l'**incrément précédent** | changements depuis le précédent |
| **Différentielle** (cumulative) | **toujours la sauvegarde complète** | changements depuis la complète |

Autrement dit, pour mettre en œuvre une stratégie **différentielle**, il suffit de toujours indiquer la sauvegarde complète comme base :

```bash
# Différentiel : la base reste systématiquement la sauvegarde complète
mariabackup --backup \
  --target-dir=/sauvegardes/diff-mercredi \
  --incremental-basedir=/sauvegardes/base \
  --user=mariabackup --password=mot_de_passe
```

Ce choix a une conséquence directe sur la restauration, conforme à ce que nous avions vu : la chaîne incrémentale exige d'appliquer **tous** les incréments dans l'ordre, là où l'approche différentielle ne nécessite que la sauvegarde complète et la **dernière** différentielle.

### Cas du streaming : l'option `--extra-lsndir`

Lorsque l'on utilise `--stream` (par exemple pour compresser ou chiffrer la sauvegarde à la volée), le fichier `mariadb_backup_checkpoints` — pourtant indispensable pour servir de base au prochain incrément — se retrouve **inclus dans le flux** compressé ou chiffré, et n'est donc plus directement accessible. La solution consiste à demander, via **`--extra-lsndir`**, une copie supplémentaire de ce fichier dans un répertoire accessible, que l'on passera ensuite en `--incremental-basedir` :

```bash
# Sauvegarde complète diffusée et compressée
mariabackup --backup --stream=mbstream \
  --user=mariabackup --password=mot_de_passe \
  --extra-lsndir=/sauvegardes/base_lsn | gzip > base.gz

# Incrément, prenant le LSN externalisé comme base
mariabackup --backup --stream=mbstream \
  --incremental-basedir=/sauvegardes/base_lsn \
  --user=mariabackup --password=mot_de_passe \
  --extra-lsndir=/sauvegardes/inc1_lsn | gzip > inc1.gz
```

## Phase 2 — La préparation (l'enchaînement)

C'est l'étape la plus subtile, et celle où l'on rencontre le plus d'informations obsolètes. Sur **MariaDB 10.2 et au-delà** (donc sur la 12.3), la procédure est la suivante :

**1. Préparer d'abord la sauvegarde de base** (avec un simple `--prepare`) :

```bash
mariabackup --prepare --target-dir=/sauvegardes/base
```

Cette commande met la sauvegarde complète en cohérence avec le redo log capturé pendant sa propre copie.

**2. Appliquer ensuite chaque incrément**, **dans l'ordre**, via `--incremental-dir` :

```bash
# Appliquer inc1 à la base
mariabackup --prepare \
  --target-dir=/sauvegardes/base \
  --incremental-dir=/sauvegardes/inc1

# Puis inc2
mariabackup --prepare \
  --target-dir=/sauvegardes/base \
  --incremental-dir=/sauvegardes/inc2
```

Chaque commande **fusionne** les changements de l'incrément dans le répertoire de la **base**. On répète cette étape pour chaque incrément restant.

> ⚠️ **Point de version essentiel.** Sur les anciennes versions (jusqu'à MariaDB 10.1), il fallait ajouter l'option **`--apply-log-only`** à chacune de ces commandes de préparation. Depuis la **10.2**, non seulement ce n'est plus nécessaire, mais l'option a été **retirée** : sur la 12.3, `mariabackup --prepare --apply-log-only` échoue avec « *unknown option `--apply-log-only`* » (vérifié sur 12.3.2). De nombreux tutoriels en ligne montrent encore l'ancienne procédure : sur une installation moderne, contentez-vous de `--prepare`.

Pour une stratégie **différentielle**, le principe est le même mais plus court : on prépare la base, puis on applique **uniquement la dernière différentielle** (qui contient déjà tous les changements cumulés depuis la complète).

---

## Phase 3 — La restauration

Une fois tous les incréments fusionnés dans la sauvegarde de base, **c'est ce répertoire de base** — désormais à jour — que l'on restaure, exactement comme une sauvegarde complète ([12.3.1](03.1-full-backup.md)) :

```bash
# 1. Arrêter le serveur
systemctl stop mariadb

# 2. S'assurer que le datadir est vide

# 3. Restaurer la BASE (qui contient tous les incréments appliqués)
mariabackup --copy-back --target-dir=/sauvegardes/base

# 4. Corriger les permissions
chown -R mysql:mysql /var/lib/mysql/

# 5. Redémarrer
systemctl start mariadb
```

On ne restaure jamais directement un répertoire d'incrément : les fichiers delta n'ont de sens qu'une fois appliqués à la base lors de la préparation.

---

## Workflow complet (récapitulatif)

| Étape | Commande |
|-------|----------|
| Sauvegarde complète | `mariabackup --backup --target-dir=/sauvegardes/base --user=… --password=…` |
| Incrément 1 | `mariabackup --backup --target-dir=/sauvegardes/inc1 --incremental-basedir=/sauvegardes/base …` |
| Incrément 2 | `mariabackup --backup --target-dir=/sauvegardes/inc2 --incremental-basedir=/sauvegardes/inc1 …` |
| Préparer la base | `mariabackup --prepare --target-dir=/sauvegardes/base` |
| Appliquer inc1 | `mariabackup --prepare --target-dir=/sauvegardes/base --incremental-dir=/sauvegardes/inc1` |
| Appliquer inc2 | `mariabackup --prepare --target-dir=/sauvegardes/base --incremental-dir=/sauvegardes/inc2` |
| Restaurer | `mariabackup --copy-back --target-dir=/sauvegardes/base` (serveur arrêté, `datadir` vide) |

---

## Bonnes pratiques et points d'attention

- **Un répertoire vide par sauvegarde** (complète comme incrémentale) ; ne réutilisez jamais un répertoire.
- **Respectez l'ordre** d'application des incréments. Comme rappelé en [12.1](01-strategies-sauvegarde.md), une chaîne incrémentale est **fragile** : un incrément manquant, corrompu ou appliqué dans le désordre rend inexploitables tous les suivants.
- **Préparez toujours la base avant** d'appliquer le moindre incrément.
- **Pas de `--apply-log-only`** sur la 12.3 (option **retirée** après la 10.1) : un simple `--prepare` suffit pour la base comme pour les incréments.
- **Différentiel = restauration simplifiée** : si la robustesse et la rapidité de restauration priment, faites toujours pointer `--incremental-basedir` vers la sauvegarde complète ; il ne restera qu'une différentielle à appliquer.
- **En streaming**, n'oubliez pas `--extra-lsndir` pour conserver un `mariadb_backup_checkpoints` accessible au prochain incrément.
- **Restaurez la base**, jamais un répertoire d'incrément isolé.

---

## À retenir

- La sauvegarde incrémentale ne copie que les **pages modifiées** depuis une base de référence, identifiées grâce au **LSN** d'InnoDB ; elle exige une sauvegarde complète préalable.
- **`--incremental-basedir`** désigne la base de référence : l'**incrément précédent** (stratégie incrémentale en chaîne) ou **toujours la sauvegarde complète** (stratégie différentielle).
- La préparation se fait en deux temps : `--prepare` sur la **base**, puis `--prepare --incremental-dir` pour **chaque incrément, dans l'ordre**, ce qui les fusionne dans la base.
- Sur **MariaDB 10.2+** (dont 12.3), **`--apply-log-only` n'est plus ni nécessaire ni supporté** (l'option a été retirée ; elle n'existait qu'en 10.1) — beaucoup de documentations en ligne sont périmées sur ce point.
- On **restaure la base** (une fois tous les incréments appliqués), jamais un répertoire delta isolé.
- L'option **`--extra-lsndir`** permet de poursuivre une chaîne d'incréments même lorsque la sauvegarde est diffusée via `--stream`.
