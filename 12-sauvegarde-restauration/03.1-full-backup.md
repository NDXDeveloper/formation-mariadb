# 12.3.1 Full backup

> **Chapitre 12 — Sauvegarde et Restauration** · MariaDB 12.3 LTS

---

## Introduction

Une **sauvegarde complète** (*full backup*) avec Mariabackup produit une copie intégrale de l'instance dans un répertoire vide. C'est le point de départ obligatoire de toute stratégie physique : les sauvegardes incrémentales ([12.3.2](03.2-incremental-backup.md)) viennent ensuite s'y greffer. Cette section met en œuvre, de façon concrète, le workflow en trois phases présenté en [section 12.3](03-sauvegarde-physique-mariabackup.md) : **sauvegarde** (`--backup`), **préparation** (`--prepare`), puis **restauration** (`--copy-back` ou `--move-back`).

Deux prérequis sont nécessaires : disposer de l'outil (paquet `mariadb-backup`, fournissant le binaire `mariabackup`) et d'un **utilisateur de sauvegarde** correctement habilité. Mariabackup s'exécutant sur la **même machine** que le serveur, il accède directement aux fichiers de données.

---

## Préparation : l'utilisateur de sauvegarde

Pendant la phase de sauvegarde, Mariabackup se connecte au serveur en cours d'exécution pour coordonner l'opération (poser un verrou bref, relever la position du journal binaire…). Il lui faut donc un compte disposant des privilèges adéquats. Sur MariaDB 12.3 (comme sur toute version ≥ 10.5), ces privilèges sont **RELOAD**, **PROCESS**, **LOCK TABLES** et **BINLOG MONITOR** :

```sql
CREATE USER 'mariabackup'@'localhost' IDENTIFIED BY 'mot_de_passe';
GRANT RELOAD, PROCESS, LOCK TABLES, BINLOG MONITOR
  ON *.* TO 'mariabackup'@'localhost';
FLUSH PRIVILEGES;
```

> 📌 **Note de version.** Le privilège **`BINLOG MONITOR`** a remplacé `REPLICATION CLIENT` à partir de **MariaDB 10.5** (l'ancien nom reste accepté pour compatibilité). Contrairement à MySQL 8, MariaDB **ne possède pas** de privilège dédié `BACKUP_ADMIN` (`GRANT BACKUP_ADMIN` y est une erreur de syntaxe, vérifié sur 12.3.2) : les quatre privilèges ci-dessus constituent la combinaison de référence.

Point important : **aucun utilisateur n'est nécessaire pour la restauration**. Celle-ci s'effectuant serveur arrêté, Mariabackup écrit directement les fichiers sur le disque sans passer par une connexion SQL.

---

## Phase 1 — La sauvegarde (`--backup`)

On lance la sauvegarde avec l'option `--backup`, en indiquant le répertoire cible via `--target-dir`. **Ce répertoire doit être vide ou ne pas exister** (Mariabackup le crée au besoin) ; il ne faut jamais réutiliser un répertoire ayant déjà servi.

```bash
mariabackup --backup \
  --target-dir=/sauvegardes/full-$(date +%F) \
  --user=mariabackup --password=mot_de_passe
```

Pendant cette phase, le serveur **continue de fonctionner** : la sauvegarde ne modifie pas la base et peut être interrompue sans risque. Mariabackup copie les fichiers de données tout en **enregistrant en parallèle le redo log** d'InnoDB. À la toute fin de la copie, il acquiert un **verrou global très bref** (généralement quelques secondes, même sur une grosse base) pour figer les tables non transactionnelles et relever la position du journal binaire.

Le répertoire obtenu ressemble à ceci :

```
/sauvegardes/full-2026-06-06/
├── aria_log.00000001
├── aria_log_control
├── backup-my.cnf
├── ibdata1
├── mysql/
├── performance_schema/
├── boutique/                ← vos bases de données
├── mariadb_backup_binlog_info   ← fichier + position du binlog au moment du backup
├── mariadb_backup_checkpoints   ← LSN (numéros de séquence) de la sauvegarde
├── mariadb_backup_info          ← métadonnées de la sauvegarde
└── ib_logfile0       ← redo log capturé pendant la copie
```

Trois fichiers méritent l'attention. **`mariadb_backup_checkpoints`** contient les LSN de la sauvegarde, qui serviront de point de référence pour une éventuelle sauvegarde incrémentale. **`ib_logfile0`** est le redo log capturé, qui sera appliqué à la phase suivante. **`mariadb_backup_binlog_info`** enregistre les coordonnées du journal binaire (fichier, position et, le cas échéant, GTID) : c'est l'information clé pour amorcer une réplique ou réaliser un *Point-in-Time Recovery* (voir [12.4](04-sauvegarde-incrementale-binlog.md) et [12.5](05-restauration.md)).

> 📎 **Noms de fichiers.** Ces fichiers de métadonnées portaient le préfixe `xtrabackup_` (`xtrabackup_checkpoints`, `xtrabackup_info`, `xtrabackup_binlog_info`) dans les versions plus anciennes de MariaDB, héritage de Percona XtraBackup ; ils ont été renommés avec le préfixe **`mariadb_backup_`** (MDEV-18931). Les anciens noms restent reconnus **en lecture** pour préparer/restaurer une sauvegarde héritée, mais une sauvegarde produite par la 12.3 emploie les nouveaux noms.

---

## Phase 2 — La préparation (`--prepare`)

Les fichiers copiés ne sont **pas cohérents** : ayant été lus à des instants différents pendant que le serveur écrivait, ils forment une image *floue*. Tenter de les restaurer tels quels provoquerait un *crash* d'InnoDB, qui détecterait les incohérences et refuserait de démarrer pour éviter toute corruption. La phase de **préparation** applique le redo log capturé pour rendre les fichiers **cohérents à un instant précis** :

```bash
mariabackup --prepare \
  --target-dir=/sauvegardes/full-2026-06-06
```

Pour une sauvegarde complète **autonome**, un simple `--prepare` suffit — tout comme pour l'enchaînement de sauvegardes incrémentales sur la 12.3 (voir [12.3.2](03.2-incremental-backup.md) ; l'ancienne option `--apply-log-only` n'a plus cours, elle est traitée là-bas). Une fois préparée, la sauvegarde est un **répertoire de données MariaDB pleinement fonctionnel**.

---

## Phase 3 — La restauration (`--copy-back` / `--move-back`)

La restauration replace les fichiers préparés dans le répertoire de données (`datadir`) du serveur. Elle se déroule en plusieurs étapes :

```bash
# 1. Arrêter le serveur MariaDB
systemctl stop mariadb

# 2. S'assurer que le datadir est VIDE
#    (sauvegardez d'abord son contenu actuel si nécessaire)

# 3. Restaurer les fichiers
mariabackup --copy-back \
  --target-dir=/sauvegardes/full-2026-06-06

# 4. Corriger le propriétaire des fichiers
chown -R mysql:mysql /var/lib/mysql/

# 5. Redémarrer le serveur
systemctl start mariadb
```

Deux options de restauration existent. **`--copy-back`** copie les fichiers vers le `datadir` en **conservant** la sauvegarde d'origine (à privilégier pour garder le backup réutilisable). **`--move-back`** *déplace* les fichiers, ce qui est plus rapide mais **détruit** la sauvegarde source ; à réserver aux cas où l'espace disque est contraint.

Deux points de vigilance. D'une part, le `datadir` doit être **vide** ; pour écraser un répertoire non vide, il faut explicitement ajouter `--force-non-empty-directories`. D'autre part, la **correction des permissions** est indispensable : Mariabackup écrit les fichiers sous l'identité de l'utilisateur qui restaure, alors que le serveur attend qu'ils appartiennent à l'utilisateur `mysql`. D'où le `chown -R mysql:mysql` avant le redémarrage.

> 💡 **Restauration avec d'autres outils.** Une sauvegarde **préparée** étant un `datadir` fonctionnel, on peut techniquement la restaurer avec n'importe quel outil de copie (`cp`, `rsync`) tant que le serveur est arrêté — `--copy-back` n'est qu'une commodité.

## Récapitulatif des commandes

| Phase | Commande |
|-------|----------|
| Sauvegarde | `mariabackup --backup --target-dir=/sauvegardes/full-AAAA-MM-JJ --user=… --password=…` |
| Préparation | `mariabackup --prepare --target-dir=/sauvegardes/full-AAAA-MM-JJ` |
| Restauration | `mariabackup --copy-back --target-dir=/sauvegardes/full-AAAA-MM-JJ` (serveur arrêté, `datadir` vide) |
| Permissions | `chown -R mysql:mysql /var/lib/mysql/` |

---

## Bonnes pratiques et points d'attention

- **Un répertoire vide par sauvegarde.** Le `--target-dir` doit être vide ou inexistant ; ne réutilisez jamais un répertoire de sauvegarde précédent. Datez-le pour vous y retrouver.
- **N'oubliez pas la préparation.** Une sauvegarde non préparée n'est pas restaurable — c'est l'erreur la plus fréquente.
- **Vérifiez la réussite** de chaque phase : code de retour de la commande et présence de `mariadb_backup_checkpoints` / `mariadb_backup_info`.
- **Conservez `mariadb_backup_binlog_info`** : il contient les coordonnées du binlog nécessaires à la réplication et au PITR.
- **Décharger le primaire** : lorsque c'est possible, exécutez la sauvegarde sur une réplique (la phase de copie reste toutefois légère pour InnoDB).
- **Permissions après restauration** : un `chown` vers `mysql:mysql` est presque toujours requis avant de redémarrer.
- **Streaming et compression** : Mariabackup peut diffuser le flux (vers la sortie standard ou le réseau) et le compresser à la volée, ce qui est utile pour archiver à distance — sujet abordé dans les sections d'automatisation et de sauvegarde cloud.

---

## À retenir

- Une sauvegarde complète Mariabackup copie l'intégralité de l'instance dans un **répertoire vide**, puis suit le cycle **`--backup` → `--prepare` → `--copy-back`/`--move-back`**.
- L'utilisateur de sauvegarde requiert **RELOAD, PROCESS, LOCK TABLES, BINLOG MONITOR** (≥ 10.5) ; **aucun utilisateur** n'est nécessaire pour la restauration (serveur arrêté).
- La phase `--backup` copie les fichiers et capture le **redo log**, avec un verrou global de quelques secondes en fin de copie.
- La phase `--prepare` applique ce redo log pour rendre la sauvegarde **cohérente** ; sans elle, InnoDB refuse de démarrer.
- La restauration exige un **serveur arrêté**, un **`datadir` vide** et une **correction des permissions** (`chown -R mysql:mysql`).
- `--copy-back` conserve la sauvegarde ; `--move-back` la déplace (plus rapide mais destructeur).
