# 12.2.1 `mysqldump` / `mariadb-dump` (support des wildcards `-L`)

> **Chapitre 12 — Sauvegarde et Restauration** · MariaDB 12.3 LTS

---

## Présentation

`mariadb-dump` est l'outil de référence pour réaliser des **sauvegardes logiques** dans MariaDB. Livré en standard avec le serveur, c'est un client en ligne de commande qui se connecte à une instance, lit les objets demandés (bases, tables, déclencheurs…) et produit un script SQL capable de les recréer. C'est l'outil le plus universel de l'écosystème : son résultat n'étant que du SQL, il peut être restauré sur une autre version de MariaDB, transféré vers un autre serveur, voire utilisé comme support de migration.

Par défaut, `mariadb-dump` fonctionne avec **un seul thread**. Il consomme donc peu de processeur, ce qui en fait un outil adapté à un serveur déjà chargé. En revanche, il procède **table par table de façon séquentielle**, en effectuant un parcours complet de chaque table : sur des données qui ne tiennent pas entièrement en cache, cela génère un volume non négligeable d'opérations disque. C'est pourquoi il est recommandé, lorsque c'est possible, d'écrire la sauvegarde sur un emplacement distant plutôt que sur le même disque que la base — en veillant à ne pas saturer le réseau de production. Pour dépasser la limite du mono-thread sur les gros volumes, on se tournera vers `mydumper`/`myloader`, présenté en [section 12.2.3](02.3-mydumper-myloader.md).

---

## Deux noms pour un même outil : `mysqldump` et `mariadb-dump`

Historiquement, ce client s'appelait `mysqldump` (héritage du code commun avec MySQL). MariaDB a depuis renommé ses utilitaires selon la convention `mariadb-*` : le nom officiel est désormais **`mariadb-dump`**. L'ancien nom `mysqldump` reste accessible (via un lien symbolique sous Linux, un binaire équivalent sous Windows), mais **depuis MariaDB 11.0 ce lien `mysqldump` est déprécié** et a notamment été retiré de l'image Docker officielle de MariaDB.

La conséquence pratique est simple : dans tout nouveau script ou toute nouvelle procédure, privilégiez `mariadb-dump`. Les deux noms acceptent exactement les mêmes options. Dans les fichiers de configuration, les sections `[mariadb-dump]` et `[mysqldump]` sont d'ailleurs toutes deux lues, ce qui facilite la transition des configurations existantes.

> ⚠️ **Note de compatibilité.** Les dumps récents commencent par une instruction activant le *sandbox mode* du client, qui n'est pas comprise par les très anciennes versions du client `mariadb` ni par le client `mysql`. Un dump produit par MariaDB 12.3 se restaure donc sans difficulté avec un client récent, mais peut provoquer une erreur s'il est rejoué avec un client trop ancien.

## Syntaxe et modes d'invocation

`mariadb-dump` s'utilise selon quatre formes principales, qui se distinguent par la portée de la sauvegarde :

```bash
# 1. Une base, ou certaines tables d'une base
mariadb-dump [options] nom_base [table ...]

# 2. Plusieurs bases nommées explicitement
mariadb-dump [options] --databases base1 base2 ...

# 3. Toutes les bases de l'instance
mariadb-dump [options] --all-databases

# 4. Les tables système (utilisateurs, privilèges, etc.)
mariadb-dump [options] --system=liste_options
```

Sans nom de table à la suite du nom de base, ou avec `--databases`/`--all-databases`, ce sont les bases entières qui sont sauvegardées. À noter que les bases `INFORMATION_SCHEMA` et `PERFORMANCE_SCHEMA` **ne sont pas exportées par défaut** : il faut les nommer explicitement (et ajouter `--skip-lock-tables`) si on souhaite les inclure.

Par défaut, le résultat est écrit sur la **sortie standard** : on le redirige donc vers un fichier. L'exemple le plus courant est la sauvegarde d'une base unique :

```bash
mariadb-dump -u sauvegarde -p boutique > boutique.sql
```

L'option `--result-file` (`-r`) permet d'écrire directement dans un fichier, ce qui est notamment recommandé sous Windows pour préserver les fins de ligne. Le jeu de caractères utilisé par défaut pour l'export est `utf8mb4`.

---

## Ce que contient (et ne contient pas) un dump

Le fichier produit est un script SQL contenant typiquement les instructions `CREATE DATABASE`, `CREATE TABLE` et `INSERT` reconstituant la structure et les données. Tous les objets ne sont cependant pas inclus automatiquement, et c'est une source d'erreurs fréquente :

| Objet | Inclus par défaut ? | Option pour l'inclure |
|-------|---------------------|------------------------|
| Tables et données | ✅ Oui | — |
| Déclencheurs (*triggers*) | ✅ Oui (font partie de la définition de la table) | (désactiver avec `--skip-triggers`) |
| Procédures et fonctions stockées | ❌ Non | `--routines` (`-R`) |
| Événements (Event Scheduler) | ❌ Non | `--events` (`-E`) |
| Tables système (`mysql`) | ❌ Non | `--system=…` |

Autrement dit, une sauvegarde « complète » d'une base applicative qui s'appuie sur des procédures ou des événements **doit explicitement** ajouter `--routines` et `--events`, sous peine de restaurer une base privée de sa logique serveur. Le détail de ces options, ainsi que celles garantissant la cohérence (`--single-transaction`) et l'optimisation du rechargement, est traité dans la sous-section suivante, [12.2.2 — Options essentielles](02.2-options-essentielles.md).

`mariadb-dump` sait par ailleurs produire d'autres formats que le SQL : du **XML** (`--xml`, `-X`) et des fichiers **tabulés** exploitables par `LOAD DATA INFILE` (`--tab`, `-T`).

## Restaurer un dump

Une sauvegarde logique n'étant qu'un script SQL, sa restauration consiste à l'exécuter sur le serveur cible à l'aide du client `mariadb` :

```bash
# Par redirection
mariadb -u admin -p boutique < boutique.sql

# Ou depuis le client, avec SOURCE
mariadb -u admin -p -e "SOURCE /sauvegardes/boutique.sql" boutique
```

Les mécanismes de restauration complète et de *Point-in-Time Recovery* sont approfondis dans la [section 12.5](05-restauration.md).

## Le support des wildcards : l'option `-L` / `--wildcards` 🆕

Introduite dans **MariaDB 12.1** (et donc disponible dans la 12.3 LTS), l'option `-L` (forme longue `--wildcards`) autorise l'emploi de **motifs avec jokers** dans les noms passés à `mariadb-dump`, au lieu de devoir énumérer chaque table ou chaque base. Désactivée par défaut, elle modifie l'interprétation des arguments de la manière suivante :

- **sans** l'option `--databases`, les jokers s'appliquent aux **noms de tables** ;
- **avec** l'option `--databases`, ils s'appliquent aux **noms de bases**.

Les motifs suivent la convention de correspondance de SQL `LIKE` : le caractère `%` représente une suite quelconque de caractères, et `_` représente un caractère unique.

**Exemple 1 — toutes les tables d'une base correspondant à un motif.** Pour sauvegarder, dans la base `boutique`, toutes les tables dont le nom commence par `commande_` :

```bash
mariadb-dump -L -u sauvegarde -p boutique 'commande_%' > commandes.sql
```

**Exemple 2 — toutes les bases correspondant à un motif.** Pour sauvegarder toutes les bases dont le nom commence par `client_` (utile dans une architecture multi-tenant à une base par client) :

```bash
mariadb-dump -L --databases -u sauvegarde -p 'client_%' > clients.sql
```

> 💡 **Bonne pratique shell.** Entourez toujours le motif de **guillemets simples**. Cela évite que l'interpréteur de commandes ne tente d'interpréter lui-même certains caractères (en particulier si vous prenez l'habitude d'utiliser `*`, qui serait alors développé par le shell avant même d'atteindre `mariadb-dump`).

Cette fonctionnalité est particulièrement utile lorsque les objets suivent une **convention de nommage** : tables d'historique horodatées (`log_2026_01`, `log_2026_02`…), tables d'un même module applicatif partageant un préfixe, ou bases d'un découpage multi-tenant. Elle permet de cibler dynamiquement un groupe d'objets dans un script de sauvegarde, sans avoir à maintenir manuellement la liste lorsque de nouveaux objets apparaissent.

## Au-delà de l'outil standard

`mariadb-dump` couvre l'immense majorité des besoins de sauvegarde logique, mais sa nature **mono-thread** en fait un goulot d'étranglement sur les très grosses bases. Pour ces cas, MariaDB et son écosystème proposent des alternatives parallélisées — notamment `mydumper`/`myloader` ([12.2.3](02.3-mydumper-myloader.md)) — capables de répartir la sauvegarde et la restauration sur plusieurs threads. La maîtrise des **options essentielles** ([12.2.2](02.2-options-essentielles.md)) reste toutefois le préalable indispensable à toute sauvegarde fiable, quel que soit l'outil retenu.

---

## À retenir

- `mariadb-dump` est le client standard de sauvegarde **logique** de MariaDB ; il produit un script SQL portable et se restaure avec le client `mariadb`.
- Le nom historique `mysqldump` reste disponible mais est **déprécié depuis MariaDB 11.0** ; privilégiez `mariadb-dump` dans les nouveaux scripts.
- Il s'invoque sur une base, plusieurs bases, toutes les bases ou les tables système ; la sortie va par défaut sur la **sortie standard** (à rediriger vers un fichier).
- Les **déclencheurs** sont inclus par défaut, mais les **procédures/fonctions** et **événements** nécessitent `--routines` et `--events`.
- L'option **`-L` / `--wildcards`** (nouveauté **MariaDB 12.1**) permet d'utiliser des motifs `LIKE` (`%`, `_`) sur les noms de tables (ou de bases avec `--databases`), idéale pour cibler des objets par convention de nommage.
- Pour les très grands volumes, l'outil mono-thread montre ses limites : voir `mydumper`/`myloader` ([12.2.3](02.3-mydumper-myloader.md)).
