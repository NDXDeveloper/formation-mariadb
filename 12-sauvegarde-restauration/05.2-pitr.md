# 12.5.2 Point-in-time recovery (PITR)

> **Chapitre 12 — Sauvegarde et Restauration** · MariaDB 12.3 LTS

---

## Introduction

La **restauration à un instant précis** (*Point-in-Time Recovery*, PITR) consiste à ramener la base non pas au dernier instantané, mais à un **moment exact** dans le temps. Elle repose sur le mécanisme posé en [section 12.4](04-sauvegarde-incrementale-binlog.md) : une **sauvegarde de base** restaure les données jusqu'à l'instant où elle a été prise, puis le **rejeu des journaux binaires** rejoue les modifications survenues ensuite, jusqu'au point visé.

C'est la capacité qui distingue radicalement une stratégie complète : elle permet de remonter à l'instant qui précède immédiatement une **fausse manœuvre** — un `DROP TABLE` ou un `DELETE` exécuté par erreur — pour récupérer **tout sauf l'erreur**. Cette section détaille la procédure complète, du repérage du point de départ jusqu'au rejeu borné des journaux.

---

## Le principe : base + rejeu des journaux

Une sauvegarde restaure les données telles qu'elles étaient **au moment où elle a été prise** ; elle ne contient pas le journal binaire. Le PITR se déroule donc en deux temps :

1. **restaurer la sauvegarde de base** (complète ou incrémentale) — ce qui ramène la base à l'instant de la sauvegarde ;
2. **rejouer les journaux binaires** à partir de la **coordonnée** enregistrée par cette sauvegarde, **jusqu'au point d'arrêt** souhaité (une date, une heure ou une position).

La base fournit donc le **point de départ** du rejeu (où commencer), et le **point d'arrêt** que l'on choisit définit jusqu'où aller.

---

## Prérequis

- Le **journal binaire devait être activé** au moment de l'incident (prérequis absolu du PITR, voir [12.4](04-sauvegarde-incrementale-binlog.md)), et les journaux couvrant la période à rejouer doivent être **disponibles**.
- Une **sauvegarde de base** (Mariabackup ou logique) avec sa **coordonnée** enregistrée.
- Le format **ROW** est recommandé pour un rejeu fiable.

---

## La procédure complète (base physique Mariabackup)

La procédure officielle restaure vers un **nouveau répertoire de données**, ce qui présente un double avantage : préserver l'instance d'origine (et ses journaux binaires) en cas de besoin, et offrir une cible propre. Elle se déroule en six étapes.

**1. Repérer la coordonnée de départ** dans le fichier `mariadb_backup_binlog_info` de la sauvegarde :

```bash
cat /sauvegardes/full-2026-06-06/mariadb_backup_binlog_info
# mariadb-bin.000002    208286
```

Ici, le rejeu devra commencer au journal `mariadb-bin.000002`, à la position `208286`.

**2. Configurer un nouveau répertoire de données** dans le fichier de configuration :

```ini
[mysqld]
datadir=/var/lib/mysql_pitr
```

**3. Restaurer la sauvegarde** (préalablement **préparée**) vers ce nouveau répertoire, selon la procédure de la [section 12.3.1](03.1-full-backup.md) :

```bash
mariabackup --copy-back --target-dir=/sauvegardes/full-2026-06-06
chown -R mysql:mysql /var/lib/mysql_pitr/
```

**4. Démarrer le serveur** :

```bash
systemctl start mariadb
```

**5. Générer le script de rejeu** avec `mariadb-binlog`, en utilisant les journaux binaires de l'**instance d'origine**, la **position de départ** relevée à l'étape 1, et le **point d'arrêt** voulu :

```bash
mariadb-binlog \
  --start-position=208286 \
  --stop-datetime="2026-06-06 11:59:59" \
  /chemin/vers/binlogs_origine/mariadb-bin.000002 \
  > /tmp/rejeu.sql
```

**6. Exécuter le script** sur la base restaurée :

```bash
mariadb < /tmp/rejeu.sql
```

La base est désormais ramenée à l'état qu'elle avait à l'instant choisi. Si la période à couvrir s'étend sur **plusieurs journaux**, on les passe tous, **dans l'ordre**, à `mariadb-binlog` :

```bash
mariadb-binlog --start-position=208286 --stop-position=512340 \
  mariadb-bin.000002 mariadb-bin.000003 mariadb-bin.000004 \
  > /tmp/rejeu.sql
```

---

## Cas d'une base logique

Lorsque la sauvegarde de base est un **dump logique** pris avec `--master-data=2` ([12.2.2](02.2-options-essentielles.md)), la coordonnée de départ figure **en commentaire** dans le dump, sous la forme d'une instruction `CHANGE MASTER` indiquant le fichier et la position. La démarche est identique : on restaure le dump ([12.5.1](05.1-restauration-complete.md)), on relève la coordonnée dans le commentaire, puis on rejoue les journaux à partir de ce point avec `mariadb-binlog`.

---

## Choisir le point d'arrêt

Le point d'arrêt se définit de deux manières :

- **par date et heure** (`--stop-datetime`) : le rejeu s'arrête à la **première transaction dont l'horodatage est égal ou postérieur** à la valeur indiquée. Pour **inclure** une transaction, sa valeur doit donc être au moins une seconde avant la borne ;
- **par position** (`--stop-position`) : le rejeu s'arrête à une position précise du journal ; c'est la méthode la plus **exacte**.

> 💡 **Bonne pratique.** Bornez par **date** pour *localiser* la zone d'intérêt, puis effectuez le rejeu réel **par position** : c'est plus fiable, car borner par horodatage comporte un risque de manquer ou d'inclure un événement de trop. Rappelons aussi qu'une **position n'a de sens qu'associée à son nom de fichier**.

---

## Scénario : récupérer avant une fausse manœuvre

C'est le cas d'usage emblématique. Supposons qu'un `DROP TABLE` ait été exécuté par erreur vers 14 h 32. La démarche est la suivante.

D'abord, **localiser la position** de l'instruction fautive en inspectant le journal autour de l'heure approximative :

```bash
mariadb-binlog \
  --start-datetime="2026-06-06 14:30:00" \
  --stop-datetime="2026-06-06 14:35:00" \
  mariadb-bin.000005 | less
# repérer le DROP TABLE et la marque "# at NNN" qui le précède
```

La sortie de `mariadb-binlog` est une suite d'événements, chacun précédé d'une marque `# at NNN` qui donne sa **position** dans le journal. Autour de l'erreur, on lit par exemple :

```
# at 991234                       ← position juste AVANT le DROP
#260606 14:32:07 server id 1  end_log_pos 991456   Query   thread_id=42
SET TIMESTAMP=1780000327/*!*/;
DROP TABLE `boutique`.`articles` /* generated by server */
/*!*/;
# at 991456                       ← position juste APRÈS le DROP
#260606 14:32:09 server id 1  end_log_pos 991574   Gtid 0-1-9003 trans
```

Les marques `# at NNN` indiquent les positions ; on note celle qui **précède immédiatement** le `DROP` (ici `991234`). On rejoue ensuite depuis la coordonnée de la base **jusqu'à cette position**, sur la base restaurée :

```bash
mariadb-binlog --start-position=208286 --stop-position=991234 \
  mariadb-bin.000002 mariadb-bin.000003 mariadb-bin.000004 mariadb-bin.000005 \
  | mariadb
```

On a ainsi récupéré toutes les données **jusqu'à l'instant qui précède l'erreur**. Mieux : il est possible de **sauter uniquement la transaction fautive** et de reprendre ensuite, en relançant le rejeu à partir de la position qui **suit** le `DROP` :

```bash
# reprendre après l'instruction fautive
mariadb-binlog --start-position=991456 mariadb-bin.000005 | mariadb
```

Cette récupération chirurgicale — conserver l'avant et l'après en n'écartant que l'erreur — est impossible avec les seules sauvegardes par instantané.

---

## Points d'attention

- **Restaurez vers un nouveau `datadir`** : vous préservez ainsi l'instance d'origine et, surtout, ses journaux binaires nécessaires au rejeu.
- Les **journaux doivent couvrir toute la période** allant de la coordonnée de la base au point d'arrêt — d'où l'importance de la discipline d'archivage et d'expiration vue en [12.4](04-sauvegarde-incrementale-binlog.md).
- **Passez les journaux dans l'ordre** et n'oubliez pas que la **position dépend du fichier**.
- **Localisez par date, rejouez par position** pour un résultat précis.
- **Vérifiez** systématiquement le résultat après le rejeu (cohérence, volumes, contrôles applicatifs).
- **Testez la procédure PITR** régulièrement, comme toute restauration — voir [section 12.7](07-tests-restauration-pra.md).

> 📌 **Note prospective.** La série **13.x** introduit une approche PITR alternative fondée sur l'archivage du *write-ahead log* d'InnoDB (`innodb_log_archive`), qui rejoue les journaux InnoDB plutôt que les journaux binaires. Sur la 12.3, le PITR repose sur les binlogs décrits ici.

---

## À retenir

- Le PITR restaure à un **instant précis** en combinant la **restauration d'une sauvegarde de base** et le **rejeu des journaux binaires** jusqu'au point voulu.
- La **coordonnée de départ** provient de la sauvegarde : `mariadb_backup_binlog_info` (physique) ou le commentaire `--master-data` (logique).
- Procédure type : repérer la coordonnée → restaurer vers un **nouveau `datadir`** → démarrer → générer le rejeu avec **`mariadb-binlog`** (`--start-position` + `--stop-datetime`/`--stop-position`) → exécuter via `mariadb`.
- Le **point d'arrêt** se choisit par date (`--stop-datetime`) ou par position (`--stop-position`) ; on **localise par date et on rejoue par position**.
- Cas chirurgical : rejouer **jusqu'à juste avant** une fausse manœuvre, et éventuellement **sauter** la seule transaction fautive avant de reprendre.
- Le PITR n'est possible que si les **journaux binaires** étaient activés et **couvrent toute la période** — restaurez vers un nouveau répertoire pour les préserver.
