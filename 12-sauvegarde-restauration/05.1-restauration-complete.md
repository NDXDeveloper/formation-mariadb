# 12.5.1 Restauration complète

> **Chapitre 12 — Sauvegarde et Restauration** · MariaDB 12.3 LTS

---

## Introduction

Une **restauration complète** ramène la base dans l'état exact d'une sauvegarde donnée. C'est le scénario de la reprise après sinistre (perte d'un serveur), de la reconstruction d'une instance, ou du clonage d'un environnement (production vers recette). La marche à suivre dépend du **type de sauvegarde** : rejeu d'un script SQL pour une sauvegarde **logique**, remise en place de fichiers pour une sauvegarde **physique**. Cette section détaille les deux voies, ainsi que les opérations transverses (clonage, restauration des comptes, vérification).

---

## Restaurer une sauvegarde logique

### Le principe

Une sauvegarde logique n'étant qu'un script SQL, sa restauration consiste à **l'exécuter** sur un serveur démarré, à l'aide du client `mariadb`. Deux cas se présentent selon la façon dont le dump a été produit.

### Cas 1 — Dump réalisé avec `--databases`

Si la sauvegarde a été prise avec l'option `--databases` (ou `--all-databases`), elle contient déjà les instructions `CREATE DATABASE` et `USE`. Il suffit alors de rejouer le fichier, **sans préciser de base** :

```bash
mariadb -u admin -p < boutique_2026-06-06.sql
```

### Cas 2 — Dump d'une base unique (sans `--databases`)

Si la sauvegarde ne porte que sur une base, sans `CREATE DATABASE`, il faut **créer la base cible au préalable** et l'indiquer en argument :

```bash
mariadb -u admin -p -e "CREATE DATABASE IF NOT EXISTS boutique"
mariadb -u admin -p boutique < boutique_2026-06-06.sql
```

Ce second cas a un avantage : il permet de restaurer vers une base de **nom différent**, ce qui est précieux pour cloner un environnement sans écraser l'existant.

Pour un dump **compressé**, on décompresse à la volée :

```bash
gunzip < boutique_2026-06-06.sql.gz | mariadb -u admin -p boutique
```

### Accélérer et fiabiliser le rejeu

Un dump produit avec les réglages par défaut (`--opt`) est déjà optimisé pour le rechargement : insertions multi-lignes, désactivation temporaire des index non uniques (MyISAM), et, depuis MariaDB 11.8, encadrement des insertions par `SET autocommit=0` … `COMMIT`. Pour une **restauration volumineuse**, on peut toutefois réduire encore la charge en désactivant temporairement certaines vérifications :

```bash
mariadb -u admin -p boutique <<'SQL'
SET autocommit=0;
SET unique_checks=0;
SET foreign_key_checks=0;
SOURCE /sauvegardes/boutique_2026-06-06.sql
SET foreign_key_checks=1;
SET unique_checks=1;
COMMIT;
SQL
```

Désactiver `foreign_key_checks` accélère le chargement **et** évite les erreurs d'ordre : sans cela, une table enfant insérée avant sa table parente provoquerait une violation de clé étrangère. Pour poursuivre malgré d'éventuelles erreurs (par exemple une vue invalide), on peut aussi rejouer avec l'option `--force` du client.

### Restaurer une sauvegarde `mydumper` (avec `myloader`)

Une sauvegarde produite par `mydumper` ([12.2.3](02.3-mydumper-myloader.md)) se restaure avec son outil compagnon **`myloader`**, qui exploite le parallélisme :

```bash
myloader \
  --directory=/sauvegardes/boutique \
  --database=boutique \
  --threads=8 \
  --overwrite-tables \
  --user=admin --password=***
```

`--directory` pointe vers le répertoire produit par `mydumper`, `--threads` fixe le parallélisme, et `--overwrite-tables` remplace les tables existantes. Comme pour un dump classique, `myloader` peut restaurer vers une base de nom différent (`--database`). C'est cette restauration parallèle qui constitue le principal atout de `mydumper`/`myloader` sur les gros volumes.

## Restaurer une sauvegarde physique (Mariabackup)

La restauration d'une sauvegarde Mariabackup a été détaillée en [12.3.1](03.1-full-backup.md) (et [12.3.2](03.2-incremental-backup.md) pour les incréments) ; rappelons-en la séquence dans le contexte d'une restauration complète. Le préalable indispensable est que la sauvegarde ait été **préparée** (`--prepare`) — et, dans le cas d'une chaîne incrémentale, que **tous les incréments aient été appliqués** à la base.

```bash
# 1. La sauvegarde doit être PRÉPARÉE au préalable (--prepare)

# 2. Arrêter le serveur
systemctl stop mariadb

# 3. S'assurer que le datadir est VIDE
#    (sauvegardez son contenu actuel si nécessaire)

# 4. Restaurer les fichiers
mariabackup --copy-back --target-dir=/sauvegardes/full-2026-06-06

# 5. Corriger les permissions
chown -R mysql:mysql /var/lib/mysql/

# 6. Redémarrer
systemctl start mariadb
```

Rappelons les trois exigences propres à cette voie : un **serveur arrêté**, un **`datadir` vide** (sauf `--force-non-empty-directories`), et la **correction des permissions** vers l'utilisateur `mysql`. Une sauvegarde préparée étant un répertoire de données fonctionnel, elle peut aussi être remise en place avec `cp` ou `rsync`.

## Restaurer sur un autre serveur ou environnement (clonage)

Une restauration complète sert fréquemment à **rafraîchir un environnement** (alimenter la recette ou le développement à partir de la production) ou à **migrer** vers un nouveau serveur. Les deux approches ne se valent pas pour cet usage :

- la sauvegarde **logique** est idéale : portable entre versions, architectures et noms de bases, elle se rejoue sur n'importe quelle cible ;
- la sauvegarde **physique** impose une cible de **même version et même architecture** ; on copie alors le répertoire préparé vers le `datadir` du serveur cible, serveur arrêté.

---

## Restaurer les utilisateurs et privilèges

Une restauration complète d'instance doit aussi prendre en compte les **comptes et privilèges**, stockés dans la base système `mysql`. Selon le type de sauvegarde :

- en **logique**, ces objets ne sont inclus que si la sauvegarde les a explicitement capturés — via `--system` (qui produit des `CREATE USER` portables) ou en sauvegardant la base `mysql`. **Après** leur restauration, exécutez `FLUSH PRIVILEGES` (ou redémarrez le serveur) pour que les changements soient pris en compte ;
- en **physique**, la base `mysql` fait partie de la sauvegarde et est restaurée automatiquement avec le reste de l'instance.

> ⚠️ Soyez prudent en restaurant les tables système **entre versions majeures** : leur structure peut différer. Pour un changement de version, privilégiez une restauration logique des comptes (`--system`) suivie de `mariadb-upgrade`.

---

## Après la restauration : vérifier

Une restauration n'est **terminée qu'une fois validée**. Quelques contrôles systématiques :

- vérifier la **cohérence des données** et les **volumes attendus** (nombre de lignes des tables clés) ;
- exécuter `FLUSH PRIVILEGES` si les tables système ont été restaurées ;
- consulter le **journal d'erreurs** au démarrage, en particulier après une restauration physique ;
- effectuer des **contrôles applicatifs** (l'application se connecte-t-elle, les requêtes critiques répondent-elles correctement ?).

---

## Points d'attention

- **Validez sur une cible sûre** avant d'écraser une production : restaurez d'abord sur une instance temporaire en cas de doute.
- **Logique** : désactivez `foreign_key_checks` pour accélérer et éviter les erreurs d'ordre ; pensez à le réactiver.
- **Physique** : sauvegarde **préparée** obligatoire, serveur arrêté, `datadir` vide, permissions à corriger.
- **Compatibilité** : le physique exige une même version/architecture ; le logique est portable.
- Pour une restauration **partielle** d'objets précis, la sauvegarde logique offre une granularité que le physique n'a pas.
- Une restauration à un **instant précis** (et non au dernier instantané) relève du PITR, traité en [section 12.5.2](05.2-pitr.md).

---

## À retenir

- La restauration complète ramène la base à l'état d'une sauvegarde ; la méthode dépend du type de sauvegarde.
- **Logique** : rejeu du SQL avec le client `mariadb` (sans base si le dump utilise `--databases`, sinon en créant la base cible) ; `myloader` pour une sauvegarde `mydumper` (restauration parallèle).
- Pour les gros volumes logiques, désactivez temporairement `foreign_key_checks` / `unique_checks` et regroupez le tout dans une transaction.
- **Physique** : sauvegarde **préparée**, serveur arrêté, `datadir` vide, `--copy-back`, puis `chown -R mysql:mysql` et redémarrage.
- Pour **cloner un environnement**, la sauvegarde logique (portable) est préférable ; le physique exige une cible identique.
- N'oubliez ni les **comptes** (`--system` + `FLUSH PRIVILEGES`, ou base `mysql`) ni la **vérification** finale du résultat.
