# 13.4.1 — Configuration GTID

> **Chapitre 13 — Réplication › 13.4 GTID** · Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

Configurer le GTID sous MariaDB est étonnamment **léger** : contrairement à MySQL, **il n'existe pas de `gtid_mode` à activer**. Dès que le binary log est en service, **chaque transaction porte déjà son GTID**. La « configuration » consiste donc surtout à : (1) s'assurer que le binlog est actif, (2) régler quelques variables de discipline (`gtid_domain_id`, `gtid_strict_mode`), puis (3) **basculer le réplica en mode GTID** via `MASTER_USE_GTID`. Le tout **sans redémarrage ni interruption**, et de façon réversible.

---

## 1. Prérequis : le binary log

Le GTID s'appuie sur le binary log : il faut donc que `log_bin` soit activé sur la source (cf. 13.2.1). En MariaDB, **tout événement du binlog inclut nativement un GTID** — il n'y a rien d'autre à « allumer ».

> 🆚 **Différence avec MySQL :** MySQL exige `gtid_mode=ON` **et** `enforce_gtid_consistency`. MariaDB n'a aucun équivalent : le GTID est **inhérent** au binlog. C'est pourquoi on peut l'adopter (ou y renoncer) **sans downtime** (point de migration, cf. 19.1).

---

## 2. Variables de configuration côté serveur

Quelques variables, à poser dans le fichier d'options, encadrent le comportement GTID.

```ini
[mariadb]
server_id          = 1
log_bin            = /var/log/mysql/mariadb-bin
gtid_domain_id     = 0        # domaine de réplication (0 par défaut)
gtid_strict_mode   = ON       # discipline GTID (OFF par défaut) — recommandé
```

- **`server_id`** : déjà indispensable (13.2.1), il constitue le **deuxième composant** de chaque GTID généré par ce serveur.
- **`gtid_domain_id`** : le **domaine** attribué aux transactions générées localement. **`0` par défaut**, à conserver pour une topologie à source unique ; à différencier par source en **multi-source** (13.5).
- **`gtid_strict_mode`** : voir la section suivante.

---

## 3. `gtid_strict_mode`

Ce mode optionnel impose une **discipline stricte** : binlogs **identiques** entre serveurs et numéros de séquence **strictement croissants** au sein de chaque domaine.

- **Désactivé par défaut** (compatibilité ascendante) ; le GTID fonctionne sans lui.
- **Activé**, la sémantique est plus simple à raisonner (l'ordre du binlog est identique partout), et le réplica **s'arrête avec une erreur** dès qu'une anomalie est rencontrée — ce qui permet au DBA de la traiter immédiatement plutôt que de laisser une divergence s'installer.
- Il **facilite l'automatisation** des grandes topologies et est **recommandé** pour les nouveaux déploiements.

---

## 4. Activer la réplication GTID sur le réplica

C'est l'étape réellement déterminante. Sur le réplica, on précise le mode GTID via `MASTER_USE_GTID` (cf. 13.2.3) :

```sql
STOP REPLICA;
CHANGE MASTER TO MASTER_USE_GTID = slave_pos;   -- slave_pos | current_pos | no
START REPLICA;
```

- **`slave_pos`** (recommandé) : le réplica se positionne sur `gtid_slave_pos` — la position de la **dernière transaction répliquée**. Idéal pour un réplica en **lecture seule**.
- **`current_pos`** : positionnement sur `gtid_current_pos` (union de `gtid_slave_pos` et `gtid_binlog_pos`). À **éviter** si des écritures locales sont possibles sur le réplica (de nouveaux GTID locaux fausseraient la position).
- **`no`** : retour à la réplication par coordonnées.

Après démarrage, `SHOW REPLICA STATUS` confirme le mode avec **`Using_Gtid: Slave_Pos`** et expose la position courante via **`Gtid_IO_Pos`**.

---

## 5. Convertir une réplication existante (coordonnées → GTID)

Comme chaque transaction porte déjà son GTID, la bascule depuis une réplication par coordonnées est **simple et sans downtime**.

### Cas A — réplica déjà actif en mode coordonnées

MariaDB a **suivi `gtid_slave_pos` en continu**, même en mode coordonnées. Il suffit donc de changer de mode :

```sql
STOP REPLICA;
CHANGE MASTER TO MASTER_USE_GTID = slave_pos;
START REPLICA;
SHOW REPLICA STATUS\G          -- Using_Gtid doit indiquer Slave_Pos
```

### Cas B — nouveau réplica amorcé depuis une sauvegarde (coordonnées connues)

Si l'on ne dispose que de coordonnées binlog (issues d'un dump), on les **convertit en position GTID** avec la fonction `BINLOG_GTID_POS()` exécutée **sur la source** :

```sql
-- Sur la SOURCE : coordonnée → GTID
SELECT BINLOG_GTID_POS('mariadb-bin.000003', 1245);
-- → par exemple : 0-1-42
```

```sql
-- Sur le RÉPLICA : on impose la position de départ, puis on bascule en GTID
STOP REPLICA;
SET GLOBAL gtid_slave_pos = '0-1-42';
CHANGE MASTER TO
    MASTER_HOST     = '192.168.1.10',
    MASTER_USER     = 'repl',
    MASTER_PASSWORD = 'mot_de_passe_robuste',
    MASTER_USE_GTID = slave_pos;
START REPLICA;
```

> 💡 Avec **Mariabackup**, la position est consignée dans les métadonnées de la sauvegarde ; et avec le **binlog InnoDB** (12.3), `gtid_slave_pos` est même **renseigné automatiquement** à partir de l'état InnoDB lors de la restauration (voir §8 et chapitre 12).

---

## 6. Définir et réinitialiser `gtid_slave_pos`

`gtid_slave_pos` peut être **fixée manuellement** (réplica arrêté) pour imposer un point de départ :

```sql
STOP REPLICA;
SET GLOBAL gtid_slave_pos = '0-1-42';
```

Pour **repartir d'un état propre** (réamorçage complet), on enchaîne typiquement :

```sql
STOP REPLICA;
RESET REPLICA ALL;            -- supprime la configuration de réplication
RESET MASTER;                 -- réinitialise le binlog local (et gtid_binlog_pos)
SET GLOBAL gtid_slave_pos = '<GTID issu de la sauvegarde>';
```

> ⚠️ `RESET MASTER` **ne réinitialise pas** `gtid_slave_pos` ; comme `gtid_current_pos` est l'union de `gtid_slave_pos` et `gtid_binlog_pos`, il faut **ajuster explicitement `gtid_slave_pos`** si l'on veut remettre à zéro la position d'un domaine.

---

## 7. Topologies multi-domaines et chemins redondants

- **Multi-source** (13.5) : on attribue à chaque source un **`gtid_domain_id` distinct**, pour que les flux restent indépendants et applicables en parallèle.
- **Chemins de réplication redondants** (anneaux multi-maîtres) : l'option **`gtid_ignore_duplicates`** évite d'appliquer deux fois une même transaction arrivant par plusieurs chemins (comparaison par numéro de séquence dans chaque domaine). Son bon fonctionnement suppose des séquences strictement croissantes, d'où l'intérêt d'activer **`gtid_strict_mode`**.
- **Nettoyage de domaines obsolètes** : `FLUSH BINARY LOGS DELETE_DOMAIN_ID = (liste)` retire des domaines de l'état du binlog (après purge des événements concernés) ; il faut aussi ajuster `gtid_slave_pos` sur les réplicas.

---

## 8. Spécificités du binlog InnoDB (12.3) 🆕

Avec le binlog intégré à InnoDB (13.2.1), qui **impose le mode GTID** :

- l'**état GTID** est inscrit périodiquement sous forme d'enregistrements **à l'intérieur du binlog**, à un intervalle réglé par **`innodb_binlog_state_interval`** (en octets) ; la récupération de position repart du dernier enregistrement d'état ;
- lors d'une **restauration Mariabackup**, `gtid_slave_pos` est **renseigné automatiquement** depuis l'état InnoDB, ce qui simplifie l'amorçage d'un réplica ;
- l'option `CHANGE MASTER TO … MASTER_DEMOTE_TO_SLAVE = 1` aide à **rétrograder un ancien serveur source en réplica** en intégrant ses écritures locales dans la position GTID ;
- la variable `gtid_binlog_state` (consultable/positionnable lorsque le binlog est vide) intervient dans les procédures de migration depuis le binlog traditionnel (cf. 19.10).

---

## 9. Revenir à la réplication par coordonnées

Le GTID est réversible **avec le binlog traditionnel** : on revient aux coordonnées via `RESET REPLICA` puis en repositionnant le lien à l'ancienne (`MASTER_LOG_FILE` / `MASTER_LOG_POS`, `MASTER_USE_GTID = no`). En revanche, ce retour **n'est pas possible avec le binlog InnoDB**, qui exige le GTID.

---

## Exemple complet

### Configuration serveur (source et réplica)

```ini
[mariadb]
server_id          = 1            # 2, 3… sur les réplicas
log_bin            = /var/log/mysql/mariadb-bin
gtid_domain_id     = 0
gtid_strict_mode   = ON
```

### Mise en place du lien GTID sur le réplica

```sql
STOP REPLICA;
CHANGE MASTER TO
    MASTER_HOST     = '192.168.1.10',
    MASTER_PORT     = 3306,
    MASTER_USER     = 'repl',
    MASTER_PASSWORD = 'mot_de_passe_robuste',
    MASTER_USE_GTID = slave_pos;
START REPLICA;
```

## Vérifications

```sql
-- Variables GTID en vigueur
SHOW VARIABLES LIKE 'gtid%';

-- Positions courantes
SELECT @@gtid_binlog_pos, @@gtid_slave_pos, @@gtid_current_pos;

-- Mode et position côté réplica
SHOW REPLICA STATUS\G          -- Using_Gtid: Slave_Pos ; Gtid_IO_Pos: …
```

---

## Idées clés à retenir

- En MariaDB, **pas de `gtid_mode`** : le GTID est inhérent au binlog ; on l'active/désactive **sans downtime**.
- Régler **`gtid_domain_id`** (0 par défaut) et, de préférence, **`gtid_strict_mode = ON`**.
- L'activation se fait sur le réplica via **`CHANGE MASTER TO MASTER_USE_GTID = slave_pos`**.
- **Conversion** depuis les coordonnées : bascule directe pour un réplica actif, ou `BINLOG_GTID_POS()` + `SET GLOBAL gtid_slave_pos` pour un réplica amorcé d'une sauvegarde.
- `RESET MASTER` **ne réinitialise pas** `gtid_slave_pos` ; gérer ce dernier explicitement.
- **Binlog InnoDB (12.3)** : GTID obligatoire, état GTID dans le binlog (`innodb_binlog_state_interval`), `gtid_slave_pos` auto-renseigné à la restauration, option `MASTER_DEMOTE_TO_SLAVE`.
