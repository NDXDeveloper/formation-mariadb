# 13.7.1 — SHOW SLAVE STATUS / SHOW REPLICA STATUS

> **Chapitre 13 — Réplication › 13.7 Monitoring et troubleshooting** · Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

`SHOW REPLICA STATUS` est l'**outil de diagnostic de première intention** de la réplication : exécutée **sur le réplica**, elle livre en une commande l'état des threads, la position, le retard et les dernières erreurs. Savoir la lire — et repérer d'un coup d'œil les **quelques champs qui comptent** — est une compétence DBA essentielle.

> 🧩 **Terminologie :** `SHOW REPLICA STATUS` est l'**alias** de `SHOW SLAVE STATUS`. En MariaDB, **seul le nom de la commande change** : les **colonnes du résultat conservent les préfixes historiques** `Slave_*` et `Master_*` (par ex. `Slave_IO_Running`, `Master_Host`, `Seconds_Behind_Master`) — à la différence de MySQL, qui les a renommées en `Replica_*` / `Source_*`. Ce support emploie donc les noms de colonnes **MariaDB**.

---

## 1. La commande

```sql
SHOW REPLICA STATUS\G        -- alias : SHOW SLAVE STATUS
```

Le suffixe **`\G`** affiche le résultat **en colonnes verticales** (une ligne par champ), bien plus lisible que la sortie tabulaire pour cette commande très large.

En **multi-source** (13.5), on obtient l'état de **toutes les connexions** (une ligne par source, débutant par `Connection_name`) avec :

```sql
SHOW ALL REPLICAS STATUS\G   -- alias : SHOW ALL SLAVES STATUS
```

La commande requiert le privilège **`REPLICA MONITOR`** (ou `SUPER`).

---

## 2. À lire en priorité

Avant tout, quatre champs donnent l'**état de santé immédiat** :

| Champ | Sain si… |
|-------|----------|
| `Slave_IO_Running` | `Yes` |
| `Slave_SQL_Running` | `Yes` |
| `Seconds_Behind_Master` | proche de `0` |
| `Last_Error` (et `Last_IO_Error` / `Last_SQL_Error`) | vide |

Si les deux threads sont `Yes`, le retard maîtrisé et aucune erreur affichée, la réplication est saine. Les sections suivantes détaillent ces champs et les autres, regroupés par thème.

---

## 3. État des threads (santé)

| Champ | Signification |
|-------|---------------|
| `Slave_IO_Running` | Le **thread IO** se connecte à la source et télécharge le flux. Doit être `Yes`. |
| `Slave_IO_State` | Description de l'activité du thread IO (ex. *Waiting for master to send event*). |
| `Slave_SQL_Running` | Le **thread SQL** applique les événements du relay log. Doit être `Yes`. |
| `Slave_SQL_Running_State` | Description de l'activité du thread SQL (ex. *Slave has read all relay log; waiting for more updates*). |
| `Master_Server_Id` | `server_id` de la source à laquelle ce réplica est connecté. |
| `Connection_name` | Nom de la connexion (multi-source, `SHOW ALL REPLICAS STATUS` uniquement). |

Un **`No`** sur l'un des deux threads signale une réplication **arrêtée** — on lit alors immédiatement les champs d'erreur (§7). Le cas `IO = Yes` / `SQL = No` (ou l'inverse) oriente le diagnostic vers le thread concerné.

---

## 4. Position et progression

| Champ | Signification |
|-------|---------------|
| `Master_Log_File` / `Read_Master_Log_Pos` | Jusqu'où le **thread IO a lu** dans le binlog de la source. |
| `Relay_Log_File` / `Relay_Log_Pos` | Position courante dans le **relay log local**. |
| `Relay_Master_Log_File` / `Exec_Master_Log_Pos` | Coordonnée du binlog **source** correspondant à ce que le **thread SQL a exécuté**. |
| `Relay_Log_Space` | Taille totale (octets) des relay logs sur disque. |

L'**écart** entre `Read_Master_Log_Pos` (lu par l'IO) et `Exec_Master_Log_Pos` (exécuté par le SQL) reflète le **travail en attente** d'application : un écart qui se creuse est le signe d'un thread SQL qui n'arrive pas à suivre (voir 13.7.2). Un `Relay_Log_Space` qui gonfle indique un backlog de relay logs non encore appliqués.

---

## 5. Positionnement GTID

| Champ | Signification |
|-------|---------------|
| `Using_Gtid` | Mode de positionnement : `No`, `Slave_Pos` ou `Current_Pos`. |
| `Gtid_IO_Pos` | Position GTID **reçue** par le thread IO. |

En mode GTID (13.4), `Using_Gtid` confirme `Slave_Pos` (recommandé) et `Gtid_IO_Pos` affiche la position courante (ex. `0-1-10472`, ou plusieurs GTID en multi-domaines).

---

## 6. Retard (aperçu)

`Seconds_Behind_Master` estime le **retard** du réplica, en secondes. Quelques pièges à connaître dès maintenant :

- la valeur est **`NULL`** lorsque le **thread SQL est arrêté** ;
- au **redémarrage** d'un réplica, elle peut afficher transitoirement `0` tant qu'aucune transaction répliquée n'a commencé à s'exécuter (comportement connu), même si le réplica était en retard à l'arrêt.

L'interprétation fine de ce champ et la **mesure fiable** du lag (notamment avec `pt-heartbeat`) font l'objet de **13.7.2**.

---

## 7. Erreurs

| Champ | Signification |
|-------|---------------|
| `Last_IO_Errno` / `Last_IO_Error` | Dernière erreur ayant arrêté le **thread IO** (problème de connexion, binlog absent…). |
| `Last_SQL_Errno` / `Last_SQL_Error` | Dernière erreur ayant arrêté le **thread SQL** (conflit d'application, doublon…). |
| `Last_Errno` / `Last_Error` | Dernière erreur (forme combinée, historique). |
| `Skip_Counter` | Valeur courante de `sql_slave_skip_counter` (événements à sauter). |

Ce sont les **premiers champs à consulter** lorsqu'un thread est `No`. Le message d'erreur est aussi écrit dans le **journal d'erreurs** (préfixé en multi-source par `Master 'nom':`). Le diagnostic et la résolution des erreurs fréquentes sont traités en **13.7.3**.

---

## 8. Filtres de réplication

Ces champs **reflètent les filtres actifs** (cf. 13.2.2). S'ils ne sont pas vides alors qu'on n'en attendait pas, ils peuvent expliquer une **divergence** de données.

- `Replicate_Do_DB`, `Replicate_Ignore_DB`
- `Replicate_Do_Table`, `Replicate_Ignore_Table`
- `Replicate_Wild_Do_Table`, `Replicate_Wild_Ignore_Table`
- `Replicate_Do_Domain_Ids`, `Replicate_Ignore_Domain_Ids` (domaines GTID)
- `Replicate_Ignore_Server_Ids`

---

## 9. Réplication parallèle, délai et bornes

| Champ | Signification |
|-------|---------------|
| `Parallel_Mode` | Mode de réplication parallèle en vigueur (`slave_parallel_mode`, cf. 13.2.2). |
| `Slave_DDL_Groups`, `Slave_Transactional_Groups`, `Slave_Non_Transactional_Groups` | Compteurs de groupes d'événements appliqués, par catégorie. |
| `SQL_Delay` / `SQL_Remaining_Delay` | Délai configuré (`MASTER_DELAY`) pour une **réplication différée**, et délai restant. |
| `Until_Condition` / `Until_Log_File` / `Until_Log_Pos` | Borne d'arrêt fixée par `START REPLICA UNTIL` (utile en maintenance). |

---

## 10. Connexion sécurisée (TLS)

Ces champs **confirment la configuration TLS** du lien (cf. 13.2.3) :

- `Master_SSL_Allowed`, `Master_SSL_Cipher`
- `Master_SSL_CA_File`, `Master_SSL_CA_Path`
- `Master_SSL_Cert`, `Master_SSL_Key`
- `Master_SSL_Verify_Server_Cert`
- `Master_SSL_Crl`, `Master_SSL_Crlpath`

`Master_SSL_Allowed: Yes` indique que la connexion de réplication est bien chiffrée.

---

## Exemple de sortie commentée (réplica sain)

```
SHOW REPLICA STATUS\G
*************************** 1. row ***************************
              Slave_IO_State: Waiting for master to send event
                 Master_Host: 192.168.1.10
                 Master_User: repl
                 Master_Port: 3306
               Master_Log_File: mariadb-bin.000045
           Read_Master_Log_Pos: 894532        -- lu par l'IO
              Slave_IO_Running: Yes            -- ✅ IO actif
             Slave_SQL_Running: Yes            -- ✅ SQL actif
           Exec_Master_Log_Pos: 894532        -- exécuté par le SQL (= lu → pas de backlog)
            Relay_Log_Space:   895210
        Seconds_Behind_Master: 0              -- ✅ pas de retard
               Last_IO_Error:                 -- ✅ aucune erreur
              Last_SQL_Error:                 -- ✅ aucune erreur
                  Using_Gtid: Slave_Pos       -- positionnement GTID recommandé
                 Gtid_IO_Pos: 0-1-10472
               Parallel_Mode: optimistic
       Slave_SQL_Running_State: Slave has read all relay log; waiting for more updates
```

Ici, tout est au vert : les deux threads tournent, `Exec_Master_Log_Pos` a rejoint `Read_Master_Log_Pos` (aucun backlog), le retard est nul et aucune erreur n'est signalée.

---

## Lecture rapide : la checklist

1. `Slave_IO_Running` **et** `Slave_SQL_Running` à **`Yes`** ?
2. `Seconds_Behind_Master` **proche de 0** (ou `NULL` = SQL arrêté → alerte) ?
3. `Last_IO_Error` / `Last_SQL_Error` **vides** ?
4. `Read_Master_Log_Pos` vs `Exec_Master_Log_Pos` : **écart maîtrisé** ?
5. `Using_Gtid` conforme à l'attendu (`Slave_Pos`) ?
6. Filtres (`Replicate_*`) conformes à l'attendu ?

---

## Idées clés à retenir

- `SHOW REPLICA STATUS\G` (alias `SHOW SLAVE STATUS`) est le diagnostic central ; **`SHOW ALL REPLICAS STATUS`** en multi-source. Privilège : **`REPLICA MONITOR`**.
- En MariaDB, les **noms de colonnes** restent en `Slave_*` / `Master_*` (≠ MySQL).
- Priorité de lecture : **`Slave_IO_Running` + `Slave_SQL_Running` = Yes**, **`Seconds_Behind_Master` ≈ 0**, **erreurs vides**.
- L'écart **`Read_Master_Log_Pos` − `Exec_Master_Log_Pos`** révèle un backlog d'application.
- `Using_Gtid` / `Gtid_IO_Pos` renseignent le positionnement ; les champs `Replicate_*` exposent les filtres actifs (source possible de divergence).
