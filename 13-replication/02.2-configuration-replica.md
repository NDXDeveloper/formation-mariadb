# 13.2.2 — Configuration du Replica

> **Chapitre 13 — Réplication › 13.2 Réplication Master-Slave** · Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

Le **réplica** (*replica*, historiquement *slave*) se connecte à la source, télécharge son flux d'événements dans un **relay log** local, puis l'**applique** à ses propres données. Sa préparation comporte deux volets :

1. **configurer le serveur réplica** lui-même (`server_id`, relay log, lecture seule, robustesse au crash) ;
2. **charger un instantané initial** des données de la source, à partir duquel la réplication prendra le relais.

L'**établissement effectif du lien** vers la source — via `CHANGE MASTER TO` puis `START REPLICA` — fait l'objet de la section **13.2.3**. La présente section prépare le réplica jusqu'à ce point.

---

## 1. Configuration de base du serveur réplica

### `server_id` unique

Comme tout serveur de la topologie, le réplica doit porter un `server_id` **distinct** de la source et des autres réplicas :

```ini
[mariadb]
server_id = 2
```

### Lecture seule : protéger le réplica

Un réplica ne doit recevoir **aucune écriture directe** d'une application ou d'un utilisateur : toute écriture locale risque d'entrer en conflit avec le flux répliqué et de **casser la réplication** (doublons, lignes introuvables…). Avec le GTID, des écritures locales corrompent en outre le suivi de position (voir §3). On met donc systématiquement le réplica en lecture seule :

```ini
[mariadb]
read_only = ON          # bloque les écritures des utilisateurs ordinaires
```

`read_only = ON` n'empêche **ni** la réplication de s'appliquer, **ni** les comptes disposant du privilège **`READ ONLY ADMIN`** (anciennement `SUPER`, dont il a été séparé depuis MariaDB 10.5.2) d'écrire.

> 🆚 **Différence avec MySQL :** MariaDB **ne possède pas** la variable `super_read_only` (propre à MySQL ; la déclarer dans le fichier d'options empêche le serveur de démarrer). Pour **étendre la lecture seule aux comptes à privilèges élevés** — afin de garantir un réplica strictement identique au primaire — on **révoque le privilège `READ ONLY ADMIN`** des comptes concernés :
>
> ```sql
> REVOKE READ ONLY ADMIN ON *.* FROM 'admin_applicatif'@'%';
> ```
>
> Une fois ce privilège retiré à **tous** les comptes, plus personne (hormis les threads de réplication) ne peut écrire tant que `read_only = ON`.

## 2. Le relay log

Le **relay log** est la copie locale, sur le réplica, des événements reçus de la source. Le thread IO l'alimente, le thread SQL le consomme (voir 13.1).

```ini
[mariadb]
relay_log = /var/log/mysql/relay-bin
```

Quelques points importants :

- **Indépendance vis-à-vis du binlog de la source** : même si la source utilise le **binlog intégré à InnoDB** (nouveauté 12.3, cf. 13.2.1), le réplica **continue d'utiliser le relay log traditionnel** basé sur fichiers. Le choix d'implémentation côté source n'affecte donc pas la configuration du relay log.
- **Purge automatique** : `relay_log_purge = ON` (valeur par défaut) supprime les relay logs une fois appliqués.
- **Robustesse au crash** : `relay_log_recovery = ON` est recommandé. Au redémarrage après un arrêt brutal, le réplica **écarte la fin potentiellement corrompue du relay log** et **re-télécharge** les événements manquants depuis la source, à partir de sa dernière position appliquée.

```ini
[mariadb]
relay_log_recovery = ON
```

---

## 3. Crash-safety et suivi de position

La manière dont le réplica mémorise « où il en est » conditionne sa résistance aux pannes.

### Réplication par coordonnées (non-GTID) : non *crash-safe*

En mode coordonnées binlog, la position du thread IO est stockée dans le fichier `master.info`, et celle du thread SQL dans `relay-log.info`. Ces fichiers sont mis à jour **indépendamment** des données : après un crash, ils peuvent **se désynchroniser** des changements réellement appliqués, entraînant doublons ou corruption silencieuse.

### Réplication par GTID : *crash-safe*

En mode GTID, le réplica enregistre sa position (le GTID de la dernière transaction appliquée) dans la table système **`mysql.gtid_slave_pos`**. Comme cette table utilise un **moteur transactionnel** (InnoDB par défaut), la mise à jour de la position est effectuée **dans la même transaction** que les données. La position et les données restent donc cohérentes après un crash. *(Cette garantie vaut pour le DML sur des tables transactionnelles ; le DDL et les tables non transactionnelles n'en bénéficient pas.)*

> ✅ **Recommandation :** privilégier le **GTID** (13.4) **avec** `relay_log_recovery = ON` pour une réplication robuste. Côté lien, préférer `MASTER_USE_GTID = slave_pos` à `current_pos` : `current_pos` peut générer des erreurs si des transactions locales sont écrites sur le réplica — raison de plus pour le maintenir en **lecture seule**.

---

## 4. Réplication parallèle (aperçu)

Par défaut, le thread SQL applique les événements **séquentiellement**, ce qui peut devenir un goulot d'étranglement et creuser le retard (*lag*). La **réplication parallèle** répartit l'application sur plusieurs *workers*.

```ini
[mariadb]
slave_parallel_threads = 4          # 0 = désactivé (défaut), > 1 = parallèle
slave_parallel_mode = optimistic    # conservative, optimistic, aggressive, minimal, none
```

- `slave_parallel_threads > 1` améliore le débit d'application et **réduit le lag** sous forte charge.
- Le `slave_parallel_mode` arbitre entre parallélisme et prudence. En modes « hors ordre » (*out-of-order*), l'ordre transitoire d'apparition des données sur le réplica peut **différer momentanément** de celui de la source.
- Pour la réplication parallèle **entre clusters Galera**, voir 13.11 🆕 ; pour le réglage fin, le chapitre 15 (Performance et Tuning).

---

## 5. Filtres de réplication (optionnel)

On peut restreindre, **côté réplica**, les bases ou tables réellement appliquées :

```ini
[mariadb]
# Exemples (à utiliser avec discernement)
replicate_do_db        = appdb
replicate_wild_ignore_table = appdb.tmp_%
```

Les options disponibles incluent `replicate_do_db`, `replicate_ignore_db`, `replicate_do_table`, `replicate_ignore_table`, `replicate_wild_do_table` et `replicate_wild_ignore_table`.

> ⚠️ À distinguer des filtres **côté source** (`binlog_do_db` / `binlog_ignore_db`, cf. 13.2.1). Les filtres de réplication sont délicats avec les **transactions multi-bases** et peuvent introduire des incohérences ; à réserver aux cas maîtrisés.

---

## 6. Charger l'instantané initial

La réplication **ne crée pas** les données préexistantes : avant de démarrer, le réplica doit recevoir une **copie cohérente** des données de la source, prise à une **position binlog ou un GTID connus**. Cette position servira de point de départ lors de l'établissement du lien (13.2.3).

Deux approches principales (détaillées au **chapitre 12**) :

- **Sauvegarde logique** avec `mariadb-dump` : l'option `--single-transaction` garantit un instantané cohérent d'InnoDB sans verrouillage ; `--master-data` inscrit dans le dump la position de départ (les coordonnées binlog via `CHANGE MASTER TO … MASTER_LOG_FILE/POS`, accompagnées — sous MariaDB — d'un `SET GLOBAL gtid_slave_pos` en commentaire) ; `--gtid`, **à combiner avec `--master-data`** (dont elle modifie la sortie — seule, elle n'inscrit aucune position), exprime cette position sous forme **GTID** (`CHANGE MASTER TO … MASTER_USE_GTID = slave_pos`) plutôt qu'en coordonnées. On y ajoute généralement `--routines`, `--triggers` et `--events`.
- **Sauvegarde physique** avec **Mariabackup** : nettement plus rapide pour les **gros volumes** ; la position binlog/GTID est consignée dans les métadonnées de la sauvegarde.

Le dump (ou la restauration Mariabackup) est ensuite appliqué sur le réplica. La **position** ainsi capturée est celle que l'on fournira à `CHANGE MASTER TO` (13.2.3), sauf si l'on adopte une réplication GTID en `slave_pos`, auquel cas le réplica repart de sa position GTID restaurée.

> 💡 Un réplica **n'a pas besoin** de son propre binary log pour fonctionner. On l'active toutefois (`log_bin`, cf. 13.2.1) si le réplica doit lui-même **réémettre** les changements (cascade, 13.6) — via `log_slave_updates` — ou servir de base aux sauvegardes/PITR.

---

## Exemple de configuration complète d'un réplica

```ini
[mariadb]
# Identité unique
server_id = 2

# Relay log + robustesse au crash
relay_log = /var/log/mysql/relay-bin
relay_log_recovery = ON

# Lecture seule (protection contre la divergence)
read_only = ON
# Pour verrouiller aussi les comptes privilégiés : révoquer le privilège
# READ ONLY ADMIN (MariaDB n'a pas de super_read_only, propre à MySQL)

# (Optionnel) Réplication parallèle
slave_parallel_threads = 4
slave_parallel_mode = optimistic

# (Optionnel) Réémission vers d'autres réplicas (cascade)
# log_bin            = /var/log/mysql/mariadb-bin
# log_slave_updates  = ON
```

Un **redémarrage** du serveur applique ces paramètres. Le réplica est alors prêt à recevoir l'instantané initial, puis à être relié à la source (13.2.3).

## Vérifications

```sql
-- Identité et lecture seule
SHOW VARIABLES LIKE 'server_id';
SHOW VARIABLES LIKE 'read_only';
-- Comptes pouvant outrepasser read_only (privilège READ ONLY ADMIN) :
SELECT GRANTEE FROM information_schema.USER_PRIVILEGES
WHERE PRIVILEGE_TYPE = 'READ_ONLY ADMIN';

-- Relay log et robustesse
SHOW VARIABLES LIKE 'relay_log%';

-- Réplication parallèle
SHOW VARIABLES LIKE 'slave_parallel_threads';
SHOW VARIABLES LIKE 'slave_parallel_mode';
```

> L'état complet de la réplication (`SHOW REPLICA STATUS` / `SHOW SLAVE STATUS`) ne devient pertinent **qu'après** l'établissement du lien et le démarrage des threads — voir 13.2.3 et 13.7.

---

## Idées clés à retenir

- Le réplica doit avoir un **`server_id` unique** et être placé en **lecture seule** (`read_only` ; pour verrouiller aussi les comptes privilégiés, **révoquer `READ ONLY ADMIN`** — MariaDB n'a pas de `super_read_only`).
- Le **relay log** est local et **toujours traditionnel**, même si la source utilise le binlog InnoDB (12.3).
- La **réplication GTID** est *crash-safe* (position dans `mysql.gtid_slave_pos`, table InnoDB) ; la réplication par coordonnées ne l'est pas. Compléter avec `relay_log_recovery = ON`.
- La **réplication parallèle** (`slave_parallel_threads`, `slave_parallel_mode`) réduit le lag, au prix d'un ordre transitoire pouvant différer de la source.
- Le réplica doit d'abord recevoir un **instantané initial cohérent** (dump ou Mariabackup), pris à une position connue (chapitre 12).
- Un réplica n'a **pas besoin** de son propre binlog, sauf pour la cascade ou les sauvegardes.
