# 13.2.3 — CHANGE MASTER TO / CHANGE REPLICATION SOURCE

> **Chapitre 13 — Réplication › 13.2 Réplication Master-Slave** · Version de référence : **MariaDB 12.3 LTS**

---

## Introduction

C'est l'étape qui **relie effectivement le réplica à la source**, puis **démarre** la réplication. Elle se déroule **sur le réplica**, une fois la source configurée (13.2.1), le réplica préparé (13.2.2) et l'**instantané initial** chargé. On y indique au réplica : *à quel serveur se connecter*, *avec quels identifiants*, *à partir de quelle position* et *selon quels paramètres de sécurité*. Le réplica enregistre alors ces métadonnées et peut lancer ses threads IO et SQL.

---

## 1. La commande et ses deux vocabulaires

Sous MariaDB, la commande est **`CHANGE MASTER TO`**, avec des options préfixées par `MASTER_*`. Sa forme générale :

```sql
CHANGE MASTER ['nom_de_connexion'] TO option = valeur [, option = valeur] ... ;
```

Quelques principes :

- elle s'exécute **sur le réplica** (jamais sur la source) ;
- si la réplication tourne déjà, on **arrête d'abord** les threads (`STOP REPLICA`) ;
- les options **non spécifiées conservent leur valeur** précédente ; il est donc inutile de tout réindiquer à chaque modification ;
- les paramètres sont **persistés** dans le fichier `master.info` (ou le fichier défini par `master_info_file`) ;
- la réplication s'effectue **en TCP/IP** (les sockets Unix ne sont pas utilisables).

> 🧩 **Terminologie *master* → *source* :** les termes historiques *master*/*slave* restent la base de MariaDB, qui ajoute progressivement les synonymes *primary*/*replica* (suivi MDEV-18777). Au niveau des **commandes**, MariaDB propose déjà les alias `START REPLICA`, `STOP REPLICA`, `SHOW REPLICA STATUS`, `RESET REPLICA`, etc. Pour la configuration du lien, la forme **documentée et canonique reste `CHANGE MASTER TO`** avec les options `MASTER_*` ; c'est celle employée ci-dessous. (L'écosystème MySQL emploie de son côté `CHANGE REPLICATION SOURCE TO` avec des options `SOURCE_*`.)

---

## 2. Options de connexion essentielles

| Option | Rôle |
|--------|------|
| `MASTER_HOST` | Adresse (IP ou nom DNS) de la source. |
| `MASTER_PORT` | Port de la source (3306 par défaut). |
| `MASTER_USER` | Compte de réplication créé en 13.2.1 (`REPLICATION SLAVE`). |
| `MASTER_PASSWORD` | Mot de passe de ce compte. |
| `MASTER_CONNECT_RETRY` | Délai (s) entre deux tentatives de reconnexion. |

---

## 3. Définir le point de départ : coordonnées ou GTID

Le réplica doit savoir **à partir d'où** lire le flux de la source. Deux méthodes, détaillées en 13.3 et 13.4.

### Par coordonnées binlog

On fournit le fichier et l'offset relevés sur la source (`SHOW BINLOG STATUS`, alias `SHOW MASTER STATUS`) ou inscrits dans les métadonnées de l'instantané (cf. 13.2.1 et 13.2.2) :

```sql
MASTER_LOG_FILE = 'mariadb-bin.000001',
MASTER_LOG_POS  = 4897
```

### Par GTID — recommandé

```sql
MASTER_USE_GTID = slave_pos      -- slave_pos (recommandé) | current_pos | no
```

- **`slave_pos`** : le réplica repart de sa propre position GTID (`gtid_slave_pos`), restaurée avec l'instantané. C'est l'option **conseillée**, robuste aux bascules.
- **`current_pos`** : à éviter si des transactions locales peuvent être écrites sur le réplica (risque d'erreurs) — d'où l'intérêt de la **lecture seule** (13.2.2).
- **`no`** : réplication par coordonnées classiques.

Le GTID rend la **reconfiguration de topologie** et le **failover** nettement plus simples (voir 13.4 et 13.8).

---

## 4. Chiffrer le lien : les options `MASTER_SSL_*`

Par défaut, MariaDB **ne chiffre pas** le trafic de réplication. Si la source et le réplica ne sont pas isolés dans un réseau de confiance, il faut activer **TLS** :

| Option | Rôle |
|--------|------|
| `MASTER_SSL` | Active TLS (`1`). **Doit être actif** pour que les autres options `MASTER_SSL_*` aient un effet. |
| `MASTER_SSL_CA` | Autorité de certification (CA) pour valider le certificat de la source. |
| `MASTER_SSL_CERT` / `MASTER_SSL_KEY` | Certificat et clé du réplica (authentification mutuelle). |
| `MASTER_SSL_CIPHER` | Restriction des suites cryptographiques. |
| `MASTER_SSL_VERIFY_SERVER_CERT` | Vérifie l'identité de la source contre la CA (recommandé pour une protection complète). |
| `MASTER_SSL_CRL` / `MASTER_SSL_CRLPATH` | Listes de révocation de certificats. |

Le détail de la mise en place TLS (génération des certificats, CA, TLS zéro-configuration hérité de 11.8) figure au **chapitre 10.7**.

---

## 5. Défauts configurables pour `MASTER_SSL_*` (nouveauté 12.3) 🆕

Configurer les options TLS dans **chaque** `CHANGE MASTER TO` devient vite répétitif, surtout en multi-source où un réplica gère plusieurs connexions. Depuis **MariaDB 12.3** (MDEV-28302), on peut définir une **configuration TLS centrale** au niveau du serveur et y faire référence.

Le mécanisme repose sur deux éléments :

1. des **options serveur** (à définir dans le fichier de configuration — ce sont des options de démarrage, et non des variables interrogeables via `@@`) qui portent les valeurs par défaut : `master_ssl`, `master_ssl_ca`, `master_ssl_cert`, `master_ssl_key`, `master_ssl_cipher`, `master_ssl_verify_server_cert`, `master_ssl_crl`, `master_ssl_crlpath` ;
2. le mot-clé **`DEFAULT`**, désormais accepté par les options TLS de `CHANGE MASTER TO` : la valeur est alors **dérivée de l'option serveur correspondante** (par ex. `MASTER_SSL_CA = DEFAULT` reprend `master_ssl_ca`). L'option `MASTER_SSL` accepte de son côté les valeurs `0`, `1` ou `DEFAULT`.

```ini
# Fichier d'options du serveur — configuration TLS centrale, définie une fois
[mariadb]
master_ssl                    = ON
master_ssl_ca                 = /etc/mysql/ssl/ca.pem
master_ssl_cert               = /etc/mysql/ssl/client-cert.pem
master_ssl_key                = /etc/mysql/ssl/client-key.pem
master_ssl_verify_server_cert = ON
```

```sql
-- Le lien hérite de la configuration TLS du serveur, sans la répéter
CHANGE MASTER TO
    MASTER_HOST     = '192.168.1.10',
    MASTER_USER     = 'repl',
    MASTER_PASSWORD = 'mot_de_passe_robuste',
    MASTER_USE_GTID = slave_pos,
    MASTER_SSL      = DEFAULT,
    MASTER_SSL_CA   = DEFAULT,
    MASTER_SSL_CERT = DEFAULT,
    MASTER_SSL_KEY  = DEFAULT;
```

**Bénéfice** : une TLS cohérente, mutualisée entre tous les canaux de réplication, et des commandes `CHANGE MASTER TO` plus courtes et moins sujettes aux erreurs — particulièrement précieux en **réplication multi-source** (13.5).

---

## 6. Démarrer et arrêter la réplication

Une fois le lien défini, on démarre les threads :

```sql
START REPLICA;        -- alias historique : START SLAVE
```

Et pour les arrêter :

```sql
STOP REPLICA;         -- alias historique : STOP SLAVE
```

On peut aussi piloter chaque thread séparément (utile pour le dépannage, cf. 13.7) :

```sql
START REPLICA IO_THREAD;
START REPLICA SQL_THREAD;
```

**Enchaînement type** sur le réplica :

```mermaid
flowchart LR
    A["STOP REPLICA
(si déjà actif)"] --> B["CHANGE MASTER TO …"]
    B --> C[START REPLICA]
    C --> D["SHOW REPLICA STATUS\G"]
```

---

## 7. Vérifier le lien

L'état complet se consulte avec :

```sql
SHOW REPLICA STATUS\G        -- alias historique : SHOW SLAVE STATUS
```

Quelques champs à contrôler immédiatement après le démarrage :

- **`Slave_IO_Running`** : doit être `Yes` (le réplica se connecte et télécharge le flux). *(MariaDB conserve les noms de colonnes `Slave_*` même via `SHOW REPLICA STATUS`.)*
- **`Slave_SQL_Running`** : doit être `Yes` (le réplica applique les événements).
- **`Seconds_Behind_Master`** : le retard d'application, idéalement proche de 0.
- **`Using_Gtid`** : `No`, `Slave_Pos` ou `Current_Pos`, selon `MASTER_USE_GTID`.
- **`Last_IO_Error` / `Last_SQL_Error`** : doivent être vides.

> 🔎 Si les deux threads sont `Yes` et qu'aucune erreur n'apparaît, la réplication est saine. L'interprétation détaillée de ces champs, le suivi du lag et la résolution des erreurs sont traités en **13.7**.

## Exemples complets

### Exemple A — démarrage par coordonnées binlog

```sql
STOP REPLICA;

CHANGE MASTER TO
    MASTER_HOST          = '192.168.1.10',
    MASTER_PORT          = 3306,
    MASTER_USER          = 'repl',
    MASTER_PASSWORD      = 'mot_de_passe_robuste',
    MASTER_LOG_FILE      = 'mariadb-bin.000001',
    MASTER_LOG_POS       = 4897,
    MASTER_CONNECT_RETRY = 10;

START REPLICA;
SHOW REPLICA STATUS\G
```

### Exemple B — démarrage par GTID avec TLS explicite

```sql
STOP REPLICA;

CHANGE MASTER TO
    MASTER_HOST     = '192.168.1.10',
    MASTER_PORT     = 3306,
    MASTER_USER     = 'repl',
    MASTER_PASSWORD = 'mot_de_passe_robuste',
    MASTER_USE_GTID = slave_pos,
    MASTER_SSL      = 1,
    MASTER_SSL_CA   = '/etc/mysql/ssl/ca.pem',
    MASTER_SSL_CERT = '/etc/mysql/ssl/client-cert.pem',
    MASTER_SSL_KEY  = '/etc/mysql/ssl/client-key.pem';

START REPLICA;
SHOW REPLICA STATUS\G
```

### Exemple C — démarrage par GTID avec TLS par défaut (12.3) 🆕

```sql
-- (prérequis : variables master_ssl_* définies dans le fichier d'options du serveur)
STOP REPLICA;

    MASTER_HOST     = '192.168.1.10',
    MASTER_USER     = 'repl',
    MASTER_PASSWORD = 'mot_de_passe_robuste',

    MASTER_SSL_KEY  = DEFAULT;

START REPLICA;
SHOW REPLICA STATUS\G
```

## Connexions nommées et persistance

Le **nom de connexion** facultatif (`CHANGE MASTER 'nom' TO …`) permet à un réplica de gérer **plusieurs sources** simultanément : c'est le socle de la **réplication multi-source** (13.5). La variable `default_master_connection` désigne la connexion utilisée lorsqu'aucun nom n'est précisé (par défaut, la connexion `''`).

```sql
-- Deux sources distinctes sur un même réplica
CHANGE MASTER 'paris'  TO MASTER_HOST = '10.0.1.10', MASTER_USER = 'repl', MASTER_USE_GTID = slave_pos;
CHANGE MASTER 'lyon'   TO MASTER_HOST = '10.0.2.10', MASTER_USER = 'repl', MASTER_USE_GTID = slave_pos;
START REPLICA 'paris';
START REPLICA 'lyon';
```

Tous ces paramètres sont **persistés** : ils survivent à un redémarrage du réplica.

---

## Idées clés à retenir

- `CHANGE MASTER TO` (options `MASTER_*`) est la commande **canonique MariaDB** ; elle s'exécute **sur le réplica**, après `STOP REPLICA` si nécessaire.
- Les **synonymes de commandes** `START REPLICA` / `STOP REPLICA` / `SHOW REPLICA STATUS` sont disponibles ; la terminologie *primary*/*replica* progresse (MDEV-18777).
- Le **point de départ** se définit par coordonnées (`MASTER_LOG_FILE` + `MASTER_LOG_POS`) ou, de préférence, par **GTID** (`MASTER_USE_GTID = slave_pos`).
- Le **chiffrement TLS** s'active via `MASTER_SSL = 1` et les options `MASTER_SSL_*` ; `MASTER_SSL` doit être actif pour que les autres prennent effet.
- **Nouveauté 12.3** : le mot-clé **`DEFAULT`** et les variables serveur `master_ssl_*` permettent une **configuration TLS centrale** partagée entre canaux — idéal en multi-source.
- Le lien se vérifie avec `SHOW REPLICA STATUS\G` : `IO_Running` et `SQL_Running` à `Yes`, sans erreur.
