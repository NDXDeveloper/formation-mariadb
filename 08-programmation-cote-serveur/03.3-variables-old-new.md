# 8.3.3 Variables `OLD` et `NEW`

Un trigger agit sur la ligne en cours de modification grâce à deux pseudo-enregistrements, `OLD` et `NEW`, qui exposent les valeurs de ses colonnes. `OLD` donne accès à la ligne *avant* l'opération, `NEW` à la ligne *après*. On lit ou affecte leurs colonnes par la notation pointée — `OLD.colonne`, `NEW.colonne` — sans préfixe `@` : ce ne sont pas des variables utilisateur, mais des références fournies par le contexte du trigger.

## Disponibilité selon l'événement

Comme vu en 8.3.2, l'événement détermine lequel des deux a un sens. Plus précisément, `OLD` désigne une colonne d'une ligne existante avant sa mise à jour ou sa suppression, tandis que `NEW` désigne la colonne d'une ligne à insérer, ou d'une ligne existante après sa mise à jour.

| Événement | `OLD` | `NEW` |
|---|---|---|
| `INSERT` | — | la ligne insérée |
| `UPDATE` | la ligne avant | la ligne après |
| `DELETE` | la ligne supprimée | — |

Référencer `NEW` dans un trigger `DELETE`, ou `OLD` dans un trigger `INSERT`, n'a pas de sens et provoque une erreur.

## Lecture et écriture

Les règles d'accès dépendent à la fois de la nature du pseudo-enregistrement et du timing du trigger.

`OLD` est toujours en **lecture seule** : on peut consulter ses colonnes (avec le privilège `SELECT`), jamais les modifier. `NEW` est lisible dans tous les cas, mais on ne peut le **modifier** que dans un trigger `BEFORE`, par `SET NEW.colonne = valeur` — ce qui requiert le privilège `UPDATE` — ; c'est ainsi qu'un `BEFORE` ajuste ce qui sera écrit. Dans un trigger `AFTER`, en revanche, une affectation à `NEW` **n'a aucun effet** : la ligne étant déjà écrite, le `SET` est simplement ignoré.

| | Lecture | Écriture |
|---|---|---|
| `OLD.col` | oui | non (jamais) |
| `NEW.col` en `BEFORE` | oui | oui (`SET NEW.col = …`) |
| `NEW.col` en `AFTER` | oui | sans effet |

## Modifier la ligne en `BEFORE`

Le cas le plus courant d'écriture est le renseignement ou le calcul de valeurs avant enregistrement, dans un trigger `BEFORE` :

```sql
DELIMITER $$

CREATE OR REPLACE TRIGGER trg_commandes_before_insert
    BEFORE INSERT ON commandes
    FOR EACH ROW
BEGIN
    SET NEW.cree_le = NOW();                              -- horodatage automatique
    SET NEW.total   = NEW.quantite * NEW.prix_unitaire;  -- valeur dérivée calculée
END $$

DELIMITER ;
```

## Comparer l'avant et l'après

Dans un trigger `UPDATE`, la coexistence de `OLD` et `NEW` permet de détecter ce qui a changé. Une précaution s'impose : comparer avec `=` ou `<>` n'est pas fiable en présence de `NULL`, car toute comparaison impliquant `NULL` renvoie `NULL` (et non `TRUE`). Ainsi, un `IF OLD.prix <> NEW.prix` ne se déclencherait pas lorsque l'une des deux valeurs est `NULL`. On utilise donc l'opérateur d'égalité **sûr vis-à-vis de `NULL`**, `<=>`, qui renvoie toujours `TRUE` ou `FALSE`.

```sql
DELIMITER $$

CREATE OR REPLACE TRIGGER trg_produits_before_update
    BEFORE UPDATE ON produits
    FOR EACH ROW
BEGIN
    -- on n'agit que si le prix a réellement changé (comparaison sûre vis-à-vis de NULL)
    IF NOT (OLD.prix <=> NEW.prix) THEN
        SET NEW.prix_modifie_le = NOW();
    END IF;
END $$

DELIMITER ;
```

## En mode Oracle

En mode Oracle (`SET sql_mode = ORACLE`), ces pseudo-enregistrements se référencent avec un **deux-points** : `:OLD` et `:NEW`, suivant la convention de PL/SQL. L'affectation utilise alors l'opérateur `:=`.

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE TRIGGER trg_commandes_before_insert
    BEFORE INSERT ON commandes
    FOR EACH ROW
BEGIN
    :NEW.cree_le := NOW();
    :NEW.total   := :NEW.quantite * :NEW.prix_unitaire;
END;
$$

DELIMITER ;
```
