# 8.5.2 `SYS_REFCURSOR` (REF CURSOR faible) et `max_open_cursors` 🆕

> 🆕 Disponible depuis MariaDB 12.0 (MDEV-20034, corrigé en 12.0.1), donc présent dans la 12.3 LTS. Fonctionnalité d'inspiration Oracle, mais **utilisable aussi bien en `sql_mode=ORACLE` qu'en mode par défaut** (seul l'attribut `%NOTFOUND` employé dans l'exemple ci-dessous est propre au mode Oracle).

Le curseur classique vu en 8.5 est *figé* (lié à une requête écrite en dur) et *local* (il ne sort pas du bloc qui le déclare). Un **curseur de référence** (*REF CURSOR*) lève ces deux contraintes : c'est une *variable* qui référence un jeu de résultats, à laquelle on associe une requête à l'exécution, et que l'on peut passer entre blocs de programme. `SYS_REFCURSOR` en est la forme *faible* et *prédéfinie*.

## Du curseur figé au curseur variable

Avec un curseur classique, la requête est fixée dans le `DECLARE` et le curseur reste cantonné à son bloc. Un REF CURSOR, lui, est une variable : on lui associe une requête au moment de l'`OPEN` (et non à la déclaration), on peut donc le réutiliser pour des requêtes différentes, et — c'est son intérêt premier en compatibilité Oracle — le transmettre d'une routine à une autre.

En 12.3, le **seul** curseur de référence disponible est le type prédéfini `SYS_REFCURSOR` : il ne demande aucune déclaration de type et équivaut à un REF CURSOR **faible**. Définir son propre type avec `DECLARE TYPE … IS REF CURSOR` (faible si aucun `RETURN`, fort sinon) n'est possible qu'à partir de MariaDB **13.0.1** (MDEV-10152) ; en 12.3, cette syntaxe est rejetée (erreur de syntaxe).

## `SYS_REFCURSOR` : le curseur de référence faible prédéfini

*Faible* signifie qu'il n'est lié à aucune structure de résultat particulière : on peut l'ouvrir sur n'importe quelle requête. *Prédéfini* signifie qu'on déclare directement une variable de ce type, sans définir de type au préalable. L'utilisation suit le cycle ouvrir / parcourir / fermer, mais l'`OPEN` prend ici une clause `FOR` qui porte la requête. `SYS_REFCURSOR` fonctionne **en mode Oracle comme en mode par défaut** ; seule la **détection de fin** diffère : en mode Oracle, on emploie l'attribut de curseur `%NOTFOUND` (style PL/SQL, illustré ci-dessous) ; en mode par défaut, on déclare la variable avec `DECLARE cur SYS_REFCURSOR;` et l'on s'appuie sur un **gestionnaire `NOT FOUND`** classique (8.5), `%NOTFOUND` n'y étant pas reconnu.

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE PROCEDURE lister_par_statut(p_statut VARCHAR(20))
AS
  cur  SYS_REFCURSOR;     -- curseur de référence faible, prédéfini
  v_id INT;
BEGIN
  OPEN cur FOR SELECT id FROM commandes WHERE statut = p_statut;
  LOOP
    FETCH cur INTO v_id;
    EXIT WHEN cur%NOTFOUND;          -- attribut de curseur, style Oracle
    -- traitement de v_id
  END LOOP;
  CLOSE cur;
END;
$$

DELIMITER ;
```

La requête est associée au curseur par `OPEN cur FOR …` : on pourrait, après un `CLOSE`, rouvrir la même variable sur une requête différente, ce qu'un curseur classique ne permet pas.

## Passer un curseur entre routines

L'usage emblématique d'un REF CURSOR, hérité d'Oracle, est de **confier un jeu de résultats à l'appelant** : une routine ouvre le curseur et le transmet (en paramètre de sortie, ou comme valeur de retour d'une fonction), à charge pour l'appelant de le parcourir. C'est la raison d'être des curseurs de référence.

Une nuance de version mérite d'être levée. Le passage de `SYS_REFCURSOR` entre routines — en paramètre `IN` ou `OUT`, ou comme **type de retour de fonction** — **fonctionne dès la 12.3** (vérifié sur 12.3.2 : une routine ouvre le curseur, une autre le reçoit et le parcourt) ; c'est précisément l'objet de MDEV-20034. La restriction documentée pour MariaDB **13.0.1** ne vise **pas** le type prédéfini `SYS_REFCURSOR`, mais les types **définis par `DECLARE TYPE`** — un `REF CURSOR` nommé ou un `RECORD` (voir 8.1.4) —, qui, eux, ne peuvent servir de paramètre de routine ou de type de retour qu'à partir de cette version.

## `max_open_cursors` : limiter les curseurs ouverts

La variable système `max_open_cursors` plafonne le nombre de curseurs **ouverts simultanément**. Ce garde-fou prend tout son sens avec les curseurs de référence : comme ce sont des variables que l'on peut ouvrir de façon répétée ou dynamique, un `CLOSE` oublié laisse des curseurs ouverts qui s'accumulent. Lorsque la limite est franchie, l'ouverture d'un nouveau curseur échoue, ce qui révèle souvent une fuite de curseurs à corriger.

On consulte et ajuste cette limite comme les autres variables système :

```sql
SHOW VARIABLES LIKE 'max_open_cursors';
```

La bonne hygiène reste de toujours `CLOSE` un curseur dès qu'on a fini de le parcourir.
