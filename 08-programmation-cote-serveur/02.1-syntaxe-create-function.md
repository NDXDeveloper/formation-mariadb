# 8.2.1 Syntaxe `CREATE FUNCTION`

La définition d'une fonction reprend l'ossature de `CREATE PROCEDURE` vue en 8.1.1, avec deux ajouts qui lui sont propres : une clause `RETURNS` qui annonce le type de la valeur produite, et une instruction `RETURN` dans le corps qui fournit cette valeur. Cette section détaille ces spécificités ; les éléments communs aux deux types de routines ne sont que rappelés, avec renvoi vers 8.1.1.

## Forme générale

La syntaxe d'une fonction se présente ainsi :

```sql
CREATE [OR REPLACE]
    [DEFINER = { utilisateur | CURRENT_USER | rôle | CURRENT_ROLE }]
    FUNCTION [IF NOT EXISTS] nom_fonction ( [ paramètre [, ...] ] )
    RETURNS type
    [ caractéristique ... ]
    corps_de_la_routine
```

où chaque paramètre prend simplement la forme :

```sql
nom_param type
```

Le début est identique à celui d'une procédure (`CREATE`, éventuellement `OR REPLACE` ; clause `DEFINER` facultative ; mot-clé — ici `FUNCTION` — éventuellement suivi de `IF NOT EXISTS` ; nom et liste de paramètres entre parenthèses). Deux différences apparaissent ensuite : la liste de paramètres ne comporte aucun mode (`IN`/`OUT`/`INOUT`), et une clause `RETURNS type` **obligatoire** s'intercale entre les parenthèses et les caractéristiques.

## Des paramètres en entrée uniquement

Une fonction ne reçoit que des données : ses paramètres sont tous implicitement en entrée. On n'y écrit donc ni `IN`, ni `OUT`, ni `INOUT` — uniquement un nom et un type. C'est la traduction syntaxique de ce qui a été dit en 8.2 : la seule sortie d'une fonction est sa valeur de retour, jamais un paramètre.

## La clause `RETURNS` et l'instruction `RETURN`

Deux constructions, à ne pas confondre, travaillent ensemble.

La clause **`RETURNS type`**, dans l'en-tête, déclare le type de la valeur renvoyée (par exemple `RETURNS DECIMAL(10,2)` ou `RETURNS VARCHAR(20)`). Elle est obligatoire et n'apparaît jamais dans une procédure.

L'instruction **`RETURN expression`**, dans le corps, évalue l'expression, la convertit vers le type déclaré et **quitte immédiatement** la fonction en transmettant cette valeur à l'appelant. Une fonction peut comporter plusieurs `RETURN` — par exemple un par branche d'un `IF` ou d'un `CASE` — le premier atteint mettant fin à l'exécution. En revanche, tout chemin d'exécution doit aboutir à un `RETURN` : si la fonction se termine sans en rencontrer, une erreur est levée à l'exécution.

```sql
DELIMITER $$

CREATE OR REPLACE FUNCTION categorie_prix(p_prix DECIMAL(10,2))
    RETURNS VARCHAR(20)
    DETERMINISTIC
BEGIN
    IF p_prix < 50 THEN
        RETURN 'Économique';
    ELSEIF p_prix < 200 THEN
        RETURN 'Standard';
    ELSE
        RETURN 'Premium';
    END IF;
END $$

DELIMITER ;

SELECT nom, categorie_prix(prix) AS gamme FROM produits;
```

À noter que `RETURN` est spécifique aux fonctions : dans une procédure, on ne s'en sert pas pour renvoyer une valeur, et la sortie anticipée se fait plutôt avec `LEAVE` sur un bloc étiqueté (voir le contrôle de flux en 8.7).

## Éléments partagés avec `CREATE PROCEDURE`

Le reste de la syntaxe est commun aux deux types de routines et a déjà été présenté en 8.1.1 :

- **Le corps et le délimiteur** : le corps est généralement un bloc `BEGIN … END`, et l'on redéfinit temporairement le séparateur avec `DELIMITER` lors d'une saisie en console.
- **`CREATE OR REPLACE` et `IF NOT EXISTS`** : ces deux extensions MariaDB, mutuellement exclusives, s'appliquent aussi aux fonctions (redéploiement atomique ou création conditionnelle).
- **La clause `DEFINER`** : elle fixe le compte de rattachement et, avec `SQL SECURITY DEFINER` par défaut, le contexte de privilèges d'exécution.
- **Les caractéristiques** : elles se placent après la clause `RETURNS`. Pour les fonctions, elles ont une portée concrète (optimiseur, journalisation binaire) et font l'objet de la section 8.2.2 ; en particulier, sous journalisation binaire, déclarer une caractéristique adéquate (telle que `DETERMINISTIC`) peut être nécessaire à la création même de la fonction.

## Un exemple complet

L'exemple suivant rassemble ces éléments : une fonction simple, déterministe, réutilisable dans des requêtes — la fonction `prix_ttc` évoquée en 8.2.

```sql
DELIMITER $$

CREATE OR REPLACE FUNCTION prix_ttc(p_montant_ht DECIMAL(10,2))
    RETURNS DECIMAL(10,2)
    DETERMINISTIC
    COMMENT 'Convertit un montant HT en TTC (TVA 20 %)'
BEGIN
    RETURN p_montant_ht * 1.20;
END $$

DELIMITER ;

SELECT prix_ttc(100);   -- 120.00
```

## Définition en mode Oracle

En mode Oracle (`SET sql_mode = ORACLE`), la syntaxe se rapproche de PL/SQL. La différence la plus visible concerne la déclaration du type de retour : le mot-clé est **`RETURN`** (et non `RETURNS`) dans l'en-tête, tandis que l'instruction du corps reste `RETURN`. S'y ajoutent le mot-clé `AS` (ou `IS`) introduisant le corps et le terminateur `END;`.

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE FUNCTION prix_ttc(p_montant_ht DECIMAL(10,2))
    RETURN DECIMAL(10,2)
AS
BEGIN
    RETURN p_montant_ht * 1.20;
END;
$$

DELIMITER ;
```

D'éventuelles variables locales se déclarent alors entre `AS` et `BEGIN`, sans le mot-clé `DECLARE`, à la manière de PL/SQL.
