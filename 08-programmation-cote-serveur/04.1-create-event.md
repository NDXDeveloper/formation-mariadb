# 8.4.1 `CREATE EVENT` et planification

Créer un event consiste à associer un **traitement SQL** (la clause `DO`) à un **calendrier** (la clause `ON SCHEDULE`). Cette dernière est le cœur de la définition : c'est elle qui décide quand, et à quelle fréquence, le serveur exécutera le traitement.

## Forme générale

La syntaxe d'un event se présente ainsi :

```sql
CREATE [OR REPLACE]
    [DEFINER = { utilisateur | CURRENT_USER }]
    EVENT [IF NOT EXISTS] nom_event
    ON SCHEDULE planification
    [ON COMPLETION [NOT] PRESERVE]
    [ENABLE | DISABLE | DISABLE ON SLAVE]
    [COMMENT 'chaîne']
    DO corps
```

On y retrouve des éléments déjà rencontrés : `CREATE OR REPLACE` et `IF NOT EXISTS` (extensions MariaDB vues en 8.1.1) et la clause `DEFINER`. Le `corps` qui suit `DO` est soit une instruction unique, soit un bloc `BEGIN … END` ; dans ce second cas, on redéfinit temporairement le séparateur avec `DELIMITER` lors d'une saisie en console, comme pour les procédures et les triggers. Les clauses propres aux events — `ON SCHEDULE`, `ON COMPLETION`, l'état d'activation et `COMMENT` — sont détaillées ci-dessous.

## La planification : `AT` et `EVERY`

La clause `ON SCHEDULE` exprime le calendrier selon deux modes :

```sql
    AT date_heure [ + INTERVAL ... ]
  | EVERY quantité unité
        [ STARTS date_heure ]
        [ ENDS   date_heure ]
```

`AT` définit une exécution **ponctuelle**, à un instant précis. On peut combiner un horodatage de référence et un décalage : par exemple, exécuter dans une heure.

```sql
CREATE EVENT ev_rappel_unique
ON SCHEDULE AT CURRENT_TIMESTAMP + INTERVAL 1 HOUR
DO
    INSERT INTO notifications (message, cree_le)
    VALUES ('Rappel ponctuel', NOW());
```

`EVERY` définit une exécution **récurrente**, à intervalle régulier. L'unité peut être `SECOND`, `MINUTE`, `HOUR`, `DAY`, `WEEK`, `MONTH`, etc. Les clauses facultatives `STARTS` et `ENDS` bornent la récurrence (par défaut, elle commence immédiatement et ne se termine jamais).

```sql
CREATE EVENT ev_purge_sessions
ON SCHEDULE EVERY 1 DAY STARTS CURRENT_TIMESTAMP + INTERVAL 1 DAY
DO
    DELETE FROM sessions
     WHERE derniere_activite < NOW() - INTERVAL 30 DAY;
```

Dans ces deux exemples, le corps `DO` est une instruction unique : le point-virgule final termine simplement l'ordre `CREATE EVENT`, et aucun `DELIMITER` n'est nécessaire.

## Conserver ou non : `ON COMPLETION`

La clause `ON COMPLETION` décide du sort de l'event une fois son calendrier épuisé. Par défaut (`NOT PRESERVE`), un event **ponctuel** est **automatiquement supprimé** après son exécution, et un event récurrent l'est lorsqu'il atteint sa date de fin. C'est un comportement parfois déroutant : un event `AT` disparaît de lui-même une fois passé.

Pour conserver la définition après son achèvement — par exemple pour la réactiver ou la consulter ultérieurement — on précise `ON COMPLETION PRESERVE` ; l'event est alors conservé, mais désactivé.

## Activation et réplication : `ENABLE` / `DISABLE`

À sa création, un event est actif (`ENABLE`, l'état par défaut). On peut le définir inactif avec `DISABLE` : il existe alors, mais ne s'exécute pas tant qu'on ne le réactive pas. La variante `DISABLE ON SLAVE` se rapporte à la réplication : elle correspond à l'état dans lequel les events sont placés sur les réplicas, afin qu'ils ne s'exécutent que sur le serveur primaire (voir l'introduction de 8.4).

## Un exemple complet

L'exemple suivant rassemble ces clauses : une maintenance nocturne récurrente, bornée dans le temps, conservée après sa période d'activité, commentée, et dont le corps enchaîne plusieurs instructions dans un bloc `BEGIN … END` (d'où le recours à `DELIMITER`).

```sql
DELIMITER $$

CREATE OR REPLACE EVENT ev_maintenance_nocturne
ON SCHEDULE
    EVERY 1 DAY
    STARTS '2026-07-01 02:00:00'
    ENDS   '2026-12-31 02:00:00'
ON COMPLETION PRESERVE
COMMENT 'Maintenance nocturne : purge et statistiques'
DO
BEGIN
    DELETE FROM logs WHERE cree_le < NOW() - INTERVAL 90 DAY;

    INSERT INTO stats_quotidiennes (jour, nb_commandes)
    SELECT CURDATE(), COUNT(*)
      FROM commandes
     WHERE DATE(cree_le) = CURDATE();
END $$

DELIMITER ;
```

## Gérer un event existant

Un event se modifie avec `ALTER EVENT` (pour changer son calendrier, son corps, son commentaire, ou l'activer/désactiver) et se supprime avec `DROP EVENT`. Il se consulte via `SHOW EVENTS`, `SHOW CREATE EVENT` ou la vue `INFORMATION_SCHEMA.EVENTS` (voir l'introduction de 8.4).

Un dernier rappel, essentiel : un event correctement défini ne s'exécutera **que si le planificateur d'événements est actif**. C'est précisément l'objet de la section suivante.
