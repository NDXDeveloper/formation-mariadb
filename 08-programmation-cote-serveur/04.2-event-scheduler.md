# 8.4.2 Event Scheduler

Définir un event ne suffit pas : encore faut-il qu'un mécanisme se charge de l'exécuter au bon moment. Ce mécanisme est l'*Event Scheduler*, un thread de fond du serveur qui surveille les events et déclenche chacun lorsque son calendrier l'exige. Sans lui, les events définis restent inertes — c'est le point le plus important de cette section.

## La variable `event_scheduler` et ses états

L'état du planificateur est gouverné par la variable système globale `event_scheduler`. Elle accepte principalement trois valeurs :

| Valeur | Effet | Modifiable à chaud ? |
|---|---|---|
| `ON` | planificateur actif ; les events s'exécutent | oui (→ `OFF`) |
| `OFF` *(défaut MariaDB)* | planificateur arrêté ; aucun event ne s'exécute (définitions conservées) | oui (→ `ON`) |
| `DISABLED` | planificateur indisponible | non (fixé au démarrage) |

Détail crucial : dans MariaDB, le planificateur est **désactivé (`OFF`) par défaut**. Un event correctement créé ne s'exécutera donc pas tant qu'on n'aura pas explicitement activé le planificateur — et aucun message d'erreur ne le signale (à la différence de MySQL 8.0 et suivants, où la valeur par défaut est `ON`). La valeur `DISABLED` est particulière : elle rend le planificateur indisponible et ne peut être positionnée qu'au démarrage du serveur. Une fois `DISABLED`, on ne peut plus l'activer à chaud — un redémarrage avec une autre configuration est nécessaire. (Une valeur `ORIGINAL`, plus rare, permet de revenir à l'état défini au démarrage.)

## Activer le planificateur

Pour activer le planificateur à chaud, on règle la variable globale :

```sql
SET GLOBAL event_scheduler = ON;
```

Cette opération requiert le privilège `SUPER` (ou un privilège équivalent de modification des variables globales). Comme il s'agit d'une variable globale, elle doit impérativement être préfixée par `GLOBAL`.

Ce réglage à chaud est toutefois **perdu au redémarrage** : la variable repasse alors à sa valeur par défaut. Pour activer durablement le planificateur, on l'inscrit dans le fichier de configuration :

```ini
[mariadb]
event_scheduler = ON
```

Et, pour mémoire, si le planificateur a été mis à `DISABLED` au démarrage, aucune commande à chaud ne fonctionnera : il faut redémarrer avec une configuration modifiée.

## Vérifier et superviser

On vérifie l'état du planificateur ainsi :

```sql
SHOW VARIABLES LIKE 'event_scheduler';
-- ou
SELECT @@global.event_scheduler;
```

Lorsqu'il tourne, le thread du planificateur apparaît dans `SHOW PROCESSLIST` (sous le compte `event_scheduler`), tout comme les threads des events en cours d'exécution. Le suivi des events eux-mêmes se fait via `SHOW EVENTS` ou la vue `INFORMATION_SCHEMA.EVENTS`, qui expose notamment leur statut et leur dernière exécution :

```sql
SELECT event_name, status, last_executed
  FROM information_schema.events;
```

Enfin, comme un event s'exécute sans client pour recevoir ses messages, les **erreurs survenant pendant son exécution sont consignées dans le journal d'erreurs** du serveur (*error log*) — c'est là qu'il faut regarder en cas de comportement inattendu.
