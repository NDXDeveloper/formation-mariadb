# 8.1.1 Syntaxe `CREATE PROCEDURE`

Cette section présente la structure complète de l'ordre `CREATE PROCEDURE`, élément par élément, ainsi que la question du délimiteur — indispensable pour saisir une procédure dans le client en ligne de commande. Le détail du passage de paramètres et de l'appel est traité aux sections 8.1.2 et 8.1.3 ; on se concentre ici sur la *forme* de la définition.

## Forme générale

La syntaxe d'une procédure se présente ainsi :

```sql
CREATE [OR REPLACE]
    [DEFINER = { utilisateur | CURRENT_USER | rôle | CURRENT_ROLE }]
    PROCEDURE [IF NOT EXISTS] nom_procedure ( [ paramètre [, ...] ] )
    [ caractéristique ... ]
    corps_de_la_routine
```

où chaque paramètre prend la forme :

```sql
[ IN | OUT | INOUT ] nom_param type
```

et où une caractéristique est l'une des clauses suivantes :

```sql
    LANGUAGE SQL
  | [NOT] DETERMINISTIC
  | { CONTAINS SQL | NO SQL | READS SQL DATA | MODIFIES SQL DATA }
  | SQL SECURITY { DEFINER | INVOKER }
  | COMMENT 'chaîne'
```

Concrètement, une définition se lit de gauche à droite : le mot-clé `CREATE`, éventuellement suivi de `OR REPLACE` ; une clause `DEFINER` facultative qui fixe le propriétaire de la routine ; le mot-clé `PROCEDURE`, éventuellement suivi de `IF NOT EXISTS` ; le nom de la procédure et, entre parenthèses, sa liste de paramètres (qui peut être vide, mais les parenthèses restent obligatoires) ; une suite facultative de caractéristiques qui décrivent le comportement de la routine ; et enfin le corps, qui contient le traitement à exécuter.

Le nom peut être qualifié par celui de la base (`nom_base.nom_procedure`) ; sans qualification, la procédure est créée dans la base courante.

## Le corps de la procédure : `BEGIN … END`

Le corps d'une procédure est une *instruction composée*, c'est-à-dire un bloc délimité par `BEGIN` et `END` qui regroupe plusieurs instructions séparées par des points-virgules :

```sql
BEGIN
    instruction_1;
    instruction_2;
    ...
END
```

Lorsque le traitement ne comporte qu'une seule instruction, le bloc `BEGIN … END` n'est pas strictement obligatoire, mais on le conserve presque toujours par lisibilité et pour pouvoir étoffer la procédure par la suite. Les blocs peuvent être imbriqués et nommés par une étiquette, ce qui est utile pour structurer les traitements longs et cibler les instructions `LEAVE` ou `ITERATE` (voir le contrôle de flux en 8.7).

Une règle d'ordre s'applique à l'intérieur du bloc : les déclarations locales (variables, conditions, curseurs puis gestionnaires d'erreurs) doivent figurer en tête, avant les instructions exécutables. Ces déclarations sont détaillées plus loin dans le chapitre (8.5 pour les curseurs, 8.6 pour les gestionnaires, 8.7 pour les variables) ; il suffit ici de retenir qu'elles ouvrent le bloc.

## La question du délimiteur

Un obstacle se présente dès la première définition saisie en console. Par défaut, le client `mariadb` considère le point-virgule comme la fin d'une instruction et l'envoie aussitôt au serveur. Or le corps d'une procédure contient lui-même des points-virgules : le client couperait donc la définition en morceaux et provoquerait une erreur de syntaxe.

La solution consiste à changer temporairement le séparateur d'instructions avec la commande `DELIMITER` (une commande du client, non un ordre SQL), le temps de définir la procédure, puis à le rétablir :

```sql
DELIMITER $$

CREATE PROCEDURE bonjour()
BEGIN
    SELECT 'Bonjour le monde' AS message;
END $$

DELIMITER ;
```

Tant que le délimiteur vaut `$$`, les points-virgules internes sont transmis tels quels au serveur, et c'est `$$` qui marque la fin de la définition. On choisit librement la séquence (`$$`, `//`, etc.) du moment qu'elle n'apparaît pas dans le corps.

Ce détail ne concerne que la saisie interactive ou les scripts exécutés par le client. Lorsqu'une procédure est créée depuis un langage applicatif (via un connecteur JDBC, PDO, etc.), chaque ordre est transmis individuellement : la gestion du délimiteur ne se pose pas.

## Les caractéristiques de la routine

Les caractéristiques se placent entre la liste des paramètres et le corps, dans un ordre quelconque. Elles précisent le comportement de la procédure :

| Caractéristique | Rôle | Valeur par défaut |
|---|---|---|
| `LANGUAGE SQL` | Langage du corps (seul `SQL` est pris en charge) | `SQL` |
| `[NOT] DETERMINISTIC` | Indique si la routine renvoie toujours le même résultat pour les mêmes entrées | `NOT DETERMINISTIC` |
| `CONTAINS SQL` · `NO SQL` · `READS SQL DATA` · `MODIFIES SQL DATA` | Décrit la nature de l'accès aux données | `CONTAINS SQL` |
| `SQL SECURITY { DEFINER \| INVOKER }` | Contexte de privilèges à l'exécution | `DEFINER` |
| `COMMENT 'chaîne'` | Description libre, visible dans les métadonnées | *(aucun)* |

Pour les procédures, `DETERMINISTIC` et les clauses d'accès aux données sont surtout *déclaratives* : elles documentent l'intention et peuvent influencer la journalisation binaire, mais ne contraignent pas le contenu réel du corps. La clause `SQL SECURITY` est en revanche réellement appliquée à l'exécution (voir le contexte d'exécution abordé en 8.1). Renseigner ces caractéristiques de façon honnête reste une bonne pratique de documentation.

## `CREATE OR REPLACE` et `IF NOT EXISTS`

Sans précaution, redéfinir une procédure déjà existante échoue avec une erreur. MariaDB propose deux clauses, mutuellement exclusives, pour gérer ce cas — particulièrement utiles dans les scripts de déploiement rejoués plusieurs fois.

`CREATE OR REPLACE PROCEDURE` supprime la procédure existante puis la recrée, de manière atomique. C'est l'option idéale pour déployer la dernière version d'une routine sans avoir à écrire un `DROP PROCEDURE` préalable. Cette extension est propre à MariaDB et n'existe pas pour les procédures sous MySQL.

`CREATE PROCEDURE IF NOT EXISTS` ne crée la procédure que si elle n'existe pas déjà ; si elle existe, la définition est ignorée et un simple avertissement est émis, sans erreur ni écrasement. On l'utilise lorsqu'on veut garantir la présence d'une routine sans risquer de remplacer une version déjà en place.

## La clause `DEFINER`

La clause `DEFINER` fixe le compte auquel la procédure est rattachée. En son absence, le définisseur est l'utilisateur courant (`CURRENT_USER`). Ce compte est déterminant car, avec le mode par défaut `SQL SECURITY DEFINER`, c'est avec ses privilèges que la procédure s'exécutera, quel que soit l'appelant — le mécanisme de délégation détaillé en 8.1. Désigner un autre définisseur que soi-même requiert un privilège d'administration adapté (historiquement `SUPER`).

## Un exemple complet

L'exemple suivant rassemble ces éléments : une procédure idempotente à redéployer, dotée d'un paramètre d'entrée, d'un paramètre de sortie, d'une caractéristique d'accès aux données et d'un commentaire.

```sql
DELIMITER $$

CREATE OR REPLACE PROCEDURE nb_commandes_client(
    IN  p_client_id INT,
    OUT p_total     INT
)
    READS SQL DATA
    COMMENT 'Renvoie le nombre de commandes d''un client'
BEGIN
    SELECT COUNT(*)
      INTO p_total
      FROM commandes
     WHERE client_id = p_client_id;
END $$

DELIMITER ;
```

Son appel, présenté ici pour situer l'usage (et approfondi en 8.1.3), se fait avec `CALL` et récupère le résultat dans une variable utilisateur :

```sql
CALL nb_commandes_client(42, @total);
SELECT @total;
```

## Définition en mode Oracle

En mode Oracle (`SET sql_mode = ORACLE`), la syntaxe se rapproche de PL/SQL et présente quelques différences notables : l'ordre des paramètres place le nom avant le mode (`nom IN type` plutôt que `IN nom type`), un mot-clé `AS` (ou `IS`) introduit le corps, et le bloc se termine par `END;`. La même procédure s'écrit alors :

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE PROCEDURE nb_commandes_client(
    p_client_id IN  INT,
    p_total     OUT INT
)
AS
BEGIN
    SELECT COUNT(*)
      INTO p_total
      FROM commandes
     WHERE client_id = p_client_id;
END;
$$

DELIMITER ;
```

Dans ce mode, d'éventuelles variables locales se déclarent entre `AS` et `BEGIN`, sans le mot-clé `DECLARE`, à la manière de PL/SQL. Cette compatibilité facilite grandement la reprise de paquetages et de procédures issus d'Oracle vers MariaDB 12.3.
