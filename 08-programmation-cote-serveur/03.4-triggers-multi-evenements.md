# 8.3.4 Triggers multi-événements 🆕

> 🆕 Disponible depuis MariaDB 12.0, donc présent dans la 12.3 LTS. Contrairement à la référence `:OLD`/`:NEW`, cette fonctionnalité ne nécessite **pas** le mode Oracle : elle est disponible en mode standard.

Jusqu'à présent, chaque trigger ne traitait qu'un seul événement : pour appliquer la même logique à l'insertion, à la mise à jour et à la suppression, il fallait écrire trois déclencheurs presque identiques. Depuis MariaDB 12.0, un même trigger peut couvrir plusieurs événements à la fois, dans l'esprit de la syntaxe Oracle. Cette nouveauté réduit la duplication et simplifie la maintenance.

## Combiner les événements avec `OR`

On combine les événements avec le mot-clé `OR`, juste après le timing :

```sql
CREATE OR REPLACE TRIGGER nom_trigger
    { BEFORE | AFTER } événement [ OR événement [ OR événement ] ]
    ON table
    FOR EACH ROW
    corps_du_trigger
```

Ainsi, `AFTER INSERT OR UPDATE OR DELETE` déclenche le même trigger pour les trois opérations. Le corps doit alors savoir distinguer les cas où les événements diffèrent — ce que permettent les prédicats présentés ci-dessous.

## Distinguer l'événement : `INSERTING`, `UPDATING`, `DELETING`

À l'intérieur du corps, trois *prédicats conditionnels* indiquent quel événement a déclenché le trigger : `INSERTING`, `UPDATING` et `DELETING`. Ils renvoient `TRUE` ou `FALSE` et s'emploient partout où une condition est attendue — `IF`, `CASE WHEN`, `WHILE`. L'exemple suivant consolide en un seul trigger les trois déclencheurs d'audit vus en 8.3.2 :

```sql
DELIMITER $$

CREATE OR REPLACE TRIGGER trg_produits_audit
    AFTER INSERT OR UPDATE OR DELETE ON produits
    FOR EACH ROW
BEGIN
    IF INSERTING THEN
        INSERT INTO produits_audit (produit_id, action, prix_avant, prix_apres)
        VALUES (NEW.id, 'INSERT', NULL, NEW.prix);
    ELSEIF UPDATING THEN
        INSERT INTO produits_audit (produit_id, action, prix_avant, prix_apres)
        VALUES (NEW.id, 'UPDATE', OLD.prix, NEW.prix);
    ELSEIF DELETING THEN
        INSERT INTO produits_audit (produit_id, action, prix_avant, prix_apres)
        VALUES (OLD.id, 'DELETE', OLD.prix, NULL);
    END IF;
END $$

DELIMITER ;
```

Un point mérite l'attention : dans chaque branche, on ne référence que le pseudo-enregistrement valide pour l'événement concerné — `NEW` pour `INSERTING` et `UPDATING`, `OLD` pour `UPDATING` et `DELETING` (voir 8.3.3). Les prédicats servent précisément à isoler correctement ces accès.

Un prédicat ne peut viser qu'un événement effectivement déclaré. Dans un trigger `AFTER INSERT OR DELETE`, par exemple, écrire `UPDATING` fait **échouer la création** du trigger (erreur *Event flag 'UPDATING' … is not compatible with the trigger event type*), puisque l'`UPDATE` ne fait pas partie des événements couverts.

## Restreindre aux colonnes : `UPDATE OF`

L'événement `UPDATE` peut être limité à certaines colonnes avec la clause `UPDATE OF` : le trigger ne se déclenche alors que si l'une des colonnes listées est modifiée.

```sql
DELIMITER $$

CREATE OR REPLACE TRIGGER trg_produits_maj_prix
    BEFORE UPDATE OF prix, tva ON produits
    FOR EACH ROW
BEGIN
    SET NEW.prix_modifie_le = NOW();
END $$

DELIMITER ;
```

Cette clause se combine avec les autres événements, à condition de placer `UPDATE OF` **en dernier** dans la liste : la liste de colonnes consomme la fin de la clause, si bien qu'aucun `OR` ne peut la suivre. On écrit donc `BEFORE INSERT OR DELETE OR UPDATE OF prix` (et non `… OR UPDATE OF prix OR DELETE`, qui provoque une erreur de syntaxe).

## Précautions

Quelques points de vigilance accompagnent cette nouveauté :

- Les mots `INSERTING`, `UPDATING` et `DELETING` **ne sont pas réservés**. Si le corps déclare une variable portant l'un de ces noms, c'est la variable qui l'emporte, et non le prédicat — à éviter, sous peine de fausser silencieusement la logique.
- Dans `INFORMATION_SCHEMA.TRIGGERS`, la colonne `EVENT_MANIPULATION` d'un trigger multi-événements devient une **liste séparée par des virgules** (par exemple `INSERT,UPDATE,DELETE`), pas nécessairement dans l'ordre de déclaration.
- La fonctionnalité étant récente, il reste prudent de tester soigneusement les triggers complexes et de les garder lisibles : un déclencheur qui empile trop de logiques conditionnelles peut devenir plus difficile à maintenir que plusieurs triggers distincts. Le gain de la consolidation doit rester un gain de clarté.
