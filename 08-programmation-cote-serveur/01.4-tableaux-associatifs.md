# 8.1.4 Tableaux associatifs Oracle (`DECLARE TYPE … TABLE OF … INDEX BY`) 🆕

> 🆕 Disponible depuis MariaDB 12.1, donc présent dans la 12.3 LTS. Fonctionnalité de compatibilité PL/SQL accessible uniquement en mode Oracle (`SET sql_mode = ORACLE`).

Le mode Oracle de MariaDB introduit les *tableaux associatifs* (associative arrays), aussi appelés *INDEX BY tables* en PL/SQL. Ce sont des structures de données en mémoire, sous forme de paires clé-valeur, que l'on déclare et manipule à l'intérieur d'une procédure, d'une fonction ou d'un bloc. Leur intérêt premier est la **compatibilité** : ils permettent de porter sans réécriture le code Oracle qui s'appuie sur ces collections, typiquement pour des recherches rapides en mémoire, du traitement par lots ou la génération de rapports.

## Qu'est-ce qu'un tableau associatif ?

Un tableau associatif est une collection *creuse* (sparse) et *non bornée* d'éléments, chacun repéré par une clé unique. Contrairement à un tableau classique indexé par des entiers contigus, ses clés peuvent être des entiers *ou* des chaînes de caractères, et n'ont pas besoin de se suivre. Il vit uniquement en mémoire pendant l'exécution du bloc qui le déclare : ce n'est pas un objet de schéma, et son contenu est perdu à la fin du traitement. On accède à un élément par la notation `variable(clé)`.

## Déclaration

La déclaration se fait en deux temps : on définit d'abord un *type* de tableau avec `DECLARE TYPE`, puis on déclare une variable de ce type.

```sql
DECLARE
   TYPE nom_type IS TABLE OF type_element INDEX BY type_cle;
   ma_variable nom_type;
```

Le **type des éléments** (`type_element`) peut être un type scalaire de MariaDB, un type `RECORD` ou un type ancré (`%TYPE`, `%ROWTYPE`). Le **type des clés**, fixé par la clause `INDEX BY`, est soit un type **entier** (`INT`, `INTEGER`), soit un type **chaîne** (`VARCHAR2`, `VARCHAR`). ⚠️ Les types entiers propres à Oracle `PLS_INTEGER` et `BINARY_INTEGER` ne sont **pas reconnus** en 12.3 (erreur *Unknown data type*) : un code Oracle qui s'en sert doit basculer sur un type entier de MariaDB.

## Affecter et lire des éléments

On renseigne une entrée par affectation (l'opérateur `:=` du mode Oracle) : si la clé n'existe pas, l'élément est créé ; si elle existe, sa valeur est remplacée. La lecture emploie la même notation `variable(clé)`.

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE PROCEDURE demo_salaires()
AS
  TYPE t_salaires IS TABLE OF NUMBER INDEX BY VARCHAR2(20);
  salaires t_salaires;
BEGIN
  salaires('Rajnish') := 62000;
  salaires('James')   := 78000;
  salaires('James')   := 80000;          -- met à jour la valeur existante

  SELECT salaires('James') AS salaire_james;   -- 80000
END;
$$

DELIMITER ;
```

## Les méthodes du tableau

Les tableaux associatifs exposent une API de méthodes calquée sur Oracle, invoquées par notation pointée sur la variable :

- `COUNT` — le nombre d'éléments.
- `FIRST` / `LAST` — la première / dernière clé.
- `NEXT(clé)` / `PRIOR(clé)` — la clé suivante / précédente.
- `EXISTS(clé)` — renvoie `TRUE` si la clé existe.
- `DELETE(clé)` — supprime l'entrée correspondante ; `DELETE` (sans argument) vide le tableau.

Point important : les clés sont conservées dans l'**ordre trié**, et non dans l'ordre d'insertion. Le parcours typique combine `FIRST` et `NEXT` pour visiter les entrées dans cet ordre, en s'arrêtant lorsque `NEXT` renvoie `NULL`.

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE PROCEDURE lister_salaires(p_resultat OUT VARCHAR2(200))
AS
  TYPE t_salaires IS TABLE OF NUMBER INDEX BY VARCHAR2(20);
  salaires t_salaires;
  v_cle    VARCHAR2(20);
BEGIN
  salaires('Rajnish') := 62000;
  salaires('James')   := 78000;
  salaires('Sophie')  := 71000;

  p_resultat := salaires.COUNT || ' employés : ';

  v_cle := salaires.FIRST;                  -- première clé (ordre trié)
  WHILE v_cle IS NOT NULL LOOP
    p_resultat := p_resultat || v_cle || '=' || salaires(v_cle) || ' ';
    v_cle := salaires.NEXT(v_cle);          -- clé suivante
  END LOOP;
END;
$$

DELIMITER ;

CALL lister_salaires(@r);
SELECT @r;
-- 3 employés : James=78000 Rajnish=62000 Sophie=71000   (clés dans l'ordre trié)
```

## Éléments composites : `RECORD` et `%ROWTYPE`

Le type des éléments ne se limite pas aux scalaires : il peut s'agir d'un `RECORD` (structure à plusieurs champs) ou du `%ROWTYPE` d'une table, ce qui permet d'indexer des lignes entières par une clé. Par exemple, un répertoire indexé par surnom :

```sql
TYPE personne_t IS RECORD (
  prenom VARCHAR2(64),
  nom    VARCHAR2(64)
);
TYPE t_repertoire IS TABLE OF personne_t INDEX BY VARCHAR2(20);
repertoire t_repertoire;
-- ...
repertoire('Monty') := personne_t('Michael', 'Widenius');
```

Avec un type d'élément ancré, la déclaration `TABLE OF persons%ROWTYPE INDEX BY VARCHAR2(20)` suit automatiquement la structure de la table `persons` : si une colonne change de type, le tableau s'adapte. La **clé**, en revanche, doit rester un type concret — un type ancré (`%TYPE`) n'est pas accepté dans la clause `INDEX BY` en 12.3.

## Différences avec Oracle

L'implémentation de MariaDB reste très proche de celle d'Oracle, à trois nuances près :

- **Constructeur** : lors d'une initialisation par constructeur, les clés doivent être des *littéraux*, là où Oracle accepte des expressions.
- **Tri des clés** : pour contrôler l'ordre des clés chaîne, MariaDB s'appuie sur la clause SQL standard `COLLATE`, et non sur les paramètres Oracle `NLS_SORT` / `NLS_COMP`.
- **Pas d'imbrication** : les tableaux de tableaux (tableaux associatifs imbriqués) ne sont pas pris en charge.

En interne, MariaDB stocke ces tableaux dans un arbre rouge-noir, ce qui explique à la fois le maintien des clés en ordre trié et l'efficacité des recherches, insertions et parcours.

## Portée et limites en 12.3

Quelques limites sont à garder en tête pour la 12.3 :

- Un tableau associatif est **local** au bloc — procédure, fonction ou bloc — où il est déclaré, et purement transitoire (perdu en fin d'exécution).
- En 12.3, les types définis par `DECLARE TYPE` ne s'emploient qu'à l'intérieur des blocs locaux : ils ne peuvent pas servir de **paramètres de routine** ni de **type de retour** de fonction. Cette possibilité, pour les types `RECORD` et `REF CURSOR`, n'apparaît qu'avec MariaDB **13.0.1**, dans la série suivante.
- Les tableaux associatifs imbriqués ne sont pas supportés (voir ci-dessus).
- L'ensemble de ces constructions n'est disponible qu'en mode Oracle.
