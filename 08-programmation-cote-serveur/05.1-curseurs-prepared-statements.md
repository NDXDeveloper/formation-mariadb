# 8.5.1 Curseurs sur prepared statements 🆕

> 🆕 Disponible depuis MariaDB 12.3. Cette fonctionnalité implémente la syntaxe standard SQL du *dynamic declare cursor*.

Le curseur classique présenté en 8.5 est lié à une requête **figée**, écrite en dur dans la déclaration. Depuis MariaDB 12.3, un curseur peut au contraire être lié à une **instruction préparée**, ce qui ouvre la porte au *SQL dynamique* à l'intérieur des routines stockées : la requête parcourue peut alors être construite à l'exécution.

## Le verrou levé : une requête dynamique

Un curseur classique souffre d'une limite : sa requête peut référencer des variables, mais elle ne peut pas **être** une variable ni être composée dynamiquement (et ne peut pas comporter de clause `INTO`). Autrement dit, on ne pouvait pas, jusque-là, décider à l'exécution de la *structure* de la requête — par exemple le nom de la table à parcourir.

Avant la 12.3, le contournement consistait à créer dynamiquement une **vue** (via un `CREATE VIEW` exécuté en instruction préparée), puis à pointer un curseur classique sur cette vue. La 12.3 rend ce détour inutile : on lie désormais le curseur directement à une instruction préparée.

## La syntaxe et le déroulé

La déclaration accepte, à la place d'un `SELECT`, le **nom d'une instruction préparée** :

```sql
DECLARE nom_curseur CURSOR FOR { requête_select | nom_instruction_préparée }
```

Le déroulé impose un ordre précis. Le curseur est déclaré, dans la section de déclaration du bloc, en référençant un nom d'instruction préparée. Ce nom doit ensuite être **préparé avec `PREPARE` avant l'ouverture** du curseur (dans la partie exécutable du bloc). On ouvre alors le curseur, on parcourt les lignes, on le ferme, puis on libère l'instruction avec `DEALLOCATE PREPARE`.

## Un exemple : parcourir une table dont le nom varie

L'exemple suivant illustre le cas typique — une requête dont la *structure* dépend d'un paramètre (ici, le nom de la table), impossible à exprimer avec un curseur figé.

```sql
DELIMITER $$

CREATE OR REPLACE PROCEDURE archiver_table(IN p_table VARCHAR(64))
BEGIN
    DECLARE v_fini INT DEFAULT 0;
    DECLARE v_id   INT;

    -- le curseur est lié au nom d'instruction préparée « stmt_archive »
    DECLARE cur CURSOR FOR stmt_archive;
    DECLARE CONTINUE HANDLER FOR NOT FOUND SET v_fini = 1;

    -- construction dynamique de la requête, puis préparation (avant OPEN)
    SET @sql = CONCAT('SELECT id FROM ', p_table, ' WHERE archive = 0');
    PREPARE stmt_archive FROM @sql;

    OPEN cur;
    boucle: LOOP
        FETCH cur INTO v_id;
        IF v_fini = 1 THEN LEAVE boucle; END IF;

        -- traitement de chaque ligne
        CALL archiver_ligne(p_table, v_id);
    END LOOP;
    CLOSE cur;

    DEALLOCATE PREPARE stmt_archive;
END $$

DELIMITER ;
```

Noter l'ordre : le `DECLARE … CURSOR FOR stmt_archive` précède le `PREPARE`, mais c'est bien le `PREPARE` qui doit s'exécuter **avant** le `OPEN`. La liaison entre le curseur et l'instruction se résout au moment de l'ouverture.

> ⚠️ **Sécurité** : construire une requête en concaténant un identifiant (ici un nom de table) issu d'un paramètre est une porte ouverte à l'injection SQL. Un tel nom doit être validé contre une liste blanche avant d'être inséré dans la requête (voir 17.8 sur la prévention des injections).

## Limites

Quelques restrictions encadrent les curseurs dynamiques. Les déclarations de type ROW (`%ROWTYPE`) ne sont **pas prises en charge** pour ces curseurs : on récupère les colonnes dans des variables scalaires distinctes, comme dans l'exemple. Par ailleurs, seul un `SELECT` est admis, et l'instruction préparée doit exister (être `PREPARE`d) avant l'`OPEN`, faute de quoi l'ouverture échoue.

## Alternative : les paramètres de curseur

Lorsque seules des **valeurs** varient — et non la structure de la requête — il n'est pas nécessaire de recourir au SQL dynamique. MariaDB permet de doter un curseur de **paramètres**, passés à l'ouverture, ce qui est plus simple et sans risque d'injection :

```sql
DECLARE cur CURSOR (p_statut VARCHAR(20)) FOR
    SELECT id FROM commandes WHERE statut = p_statut;
-- ...
OPEN cur('en_attente');
```

On réserve donc les curseurs sur instruction préparée aux cas où la **forme** de la requête doit être déterminée à l'exécution.
