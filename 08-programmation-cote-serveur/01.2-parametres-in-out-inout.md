# 8.1.2 Paramètres `IN`, `OUT`, `INOUT`

Les paramètres sont le canal par lequel une procédure communique avec son appelant : ils lui transmettent des valeurs en entrée et lui renvoient des résultats en sortie. Chaque paramètre est déclaré dans la liste entre parenthèses sous la forme `[ mode ] nom type`, où le mode — `IN`, `OUT` ou `INOUT` — détermine le *sens* de la communication. Si le mode est omis, c'est `IN` qui s'applique.

Dans le corps de la procédure, les paramètres se manipulent exactement comme des variables locales : on lit leur valeur dans les expressions et on la modifie avec `SET` ou `SELECT … INTO`.

## Vue d'ensemble des trois modes

| Mode | Valeur initiale dans le corps | Lecture | Écriture répercutée vers l'appelant | Argument requis à l'appel |
|------|-------------------------------|---------|-------------------------------------|---------------------------|
| `IN` *(défaut)* | celle passée par l'appelant | oui | non | littéral, expression ou variable |
| `OUT` | `NULL` *(valeur de l'appelant ignorée)* | oui (mais part de `NULL`) | oui | **variable obligatoire** |
| `INOUT` | celle passée par l'appelant | oui | oui | **variable obligatoire** |

La logique se résume ainsi : `IN` fait entrer une donnée, `OUT` fait sortir un résultat, `INOUT` fait les deux sur un même paramètre.

## `IN` : paramètre d'entrée

Un paramètre `IN` reçoit une valeur fournie par l'appelant. La procédure travaille sur une *copie locale* : elle peut la modifier librement pour ses besoins internes, mais cette modification reste sans effet à l'extérieur. C'est le mode le plus courant et le mode par défaut.

Comme aucune valeur n'a besoin d'être renvoyée, l'argument correspondant peut être un littéral, une expression ou une variable.

```sql
DELIMITER $$

CREATE PROCEDURE augmenter_prix(IN p_pct DECIMAL(5,2))
BEGIN
    UPDATE produits
       SET prix = prix * (1 + p_pct / 100);
END $$

DELIMITER ;

-- Augmentation de 10 % : on peut passer un littéral
CALL augmenter_prix(10);
```

## `OUT` : paramètre de sortie

Un paramètre `OUT` sert à renvoyer une valeur à l'appelant. Deux points le caractérisent et méritent une attention particulière.

D'abord, l'argument fourni à l'appel **doit être une variable** (variable utilisateur `@var`, ou variable locale lorsqu'on appelle depuis une autre routine) : la procédure a besoin d'un emplacement où écrire son résultat. Passer un littéral ou une expression pour un paramètre `OUT` provoque une erreur.

Ensuite, et c'est un piège classique, la valeur éventuellement passée par l'appelant **n'est pas visible** dans le corps : au démarrage de la procédure, un paramètre `OUT` vaut toujours `NULL`, quelle que soit la variable transmise.

```sql
DELIMITER $$

CREATE PROCEDURE compter_produits(OUT p_nb INT)
BEGIN
    SELECT COUNT(*) INTO p_nb FROM produits;
END $$

DELIMITER ;

CALL compter_produits(@n);
SELECT @n;        -- le nombre de produits
```

L'exemple suivant illustre la réinitialisation à `NULL` : malgré l'affectation préalable de `@x`, le corps part bien de `NULL`.

```sql
DELIMITER $$

CREATE PROCEDURE demo_out(OUT p_val INT)
BEGIN
    -- ici p_val vaut NULL, indépendamment de l'argument fourni
    SET p_val = IFNULL(p_val, -1);
END $$

DELIMITER ;

SET @x = 99;
CALL demo_out(@x);
SELECT @x;        -- -1, et non 99 : la valeur d'entrée a été ignorée
```

Corollaire pratique : un paramètre `OUT` que la procédure n'affecte jamais reste à `NULL` après l'appel.

## `INOUT` : paramètre d'entrée-sortie

Le mode `INOUT` combine les deux précédents : la valeur de l'appelant est transmise au corps, qui peut la lire *et* la modifier, la modification étant ensuite répercutée à l'appelant. Comme pour `OUT`, l'argument doit obligatoirement être une variable.

```sql
DELIMITER $$

CREATE PROCEDURE appliquer_remise(
    INOUT p_montant DECIMAL(10,2),
    IN    p_taux    DECIMAL(5,2)
)
BEGIN
    SET p_montant = p_montant - (p_montant * p_taux / 100);
END $$

DELIMITER ;

SET @total = 200.00;
CALL appliquer_remise(@total, 15);   -- remise de 15 %
SELECT @total;                       -- 170.00
```

## Règles et contraintes à l'appel

Quelques règles encadrent la liaison des arguments, indépendamment du mode :

Le nombre d'arguments fournis à l'appel doit correspondre exactement au nombre de paramètres déclarés. MariaDB ne gère pas les valeurs par défaut de paramètres : il n'est pas possible d'omettre un argument, chacun doit être fourni — une différence notable avec des langages comme PL/SQL.

Pour les modes `OUT` et `INOUT`, l'argument doit être une cible assignable (une variable). À l'inverse, un paramètre `IN` accepte n'importe quelle expression. La valeur de chaque argument est convertie vers le type déclaré du paramètre.

Après l'appel, les résultats portés par les paramètres `OUT` et `INOUT` se lisent dans les variables passées en argument. Lorsqu'on appelle une procédure depuis une autre routine, ces variables peuvent être des variables locales ; la mécanique complète de l'appel est détaillée en 8.1.3.

## Le cas des fonctions

Cette typologie de modes est propre aux procédures. Une **fonction stockée** n'accepte que des paramètres d'entrée : on n'y précise ni `IN`, ni `OUT`, ni `INOUT` (tous les paramètres sont implicitement en entrée), et la valeur produite est renvoyée par l'instruction `RETURN`. C'est l'une des différences structurantes entre procédures et fonctions, approfondie en 8.2.

## En mode Oracle

En mode Oracle (`SET sql_mode = ORACLE`), la déclaration place le **nom avant le mode** (`nom IN type` au lieu de `IN nom type`), et le mode bidirectionnel s'y écrit `IN OUT`, en deux mots, conformément à PL/SQL.

```sql
SET sql_mode = ORACLE;

DELIMITER $$

CREATE OR REPLACE PROCEDURE compter_produits(
    p_categorie IN  VARCHAR(50),
    p_nb        OUT INT
)
AS
BEGIN
    SELECT COUNT(*) INTO p_nb
      FROM produits
     WHERE categorie = p_categorie;
END;
$$

DELIMITER ;
```

Le sens et les contraintes de chaque mode (variable obligatoire pour la sortie, réinitialisation, etc.) restent identiques : seule l'écriture change.
