# 8.1.3 Appel avec `CALL`

Une procédure ne s'exécute pas spontanément : on la déclenche explicitement avec l'instruction `CALL`. C'est par elle qu'on transmet les arguments, qu'on récupère les valeurs de sortie et qu'on reçoit les éventuels jeux de résultats produits par la procédure.

## Syntaxe

L'appel suit une forme très simple :

```sql
CALL nom_procedure([argument[, ...]]);
```

Le nom peut être qualifié par celui de la base (`CALL nom_base.nom_procedure(...)`) ; sans qualification, c'est la procédure de la base courante qui est invoquée. Lorsque la procédure ne prend aucun paramètre, les parenthèses sont facultatives : `CALL maintenance;` et `CALL maintenance();` sont équivalents. Appeler une procédure suppose de détenir le privilège `EXECUTE` (ou d'en être le définisseur), comme vu en 8.1.

## Arguments et récupération des valeurs de sortie

Le nombre d'arguments doit correspondre exactement au nombre de paramètres déclarés. Conformément aux règles vues en 8.1.2, un paramètre `IN` accepte n'importe quelle expression (littéral, calcul, variable), tandis qu'un paramètre `OUT` ou `INOUT` exige une variable, car la procédure doit pouvoir y écrire.

Après l'appel, les valeurs renvoyées par les paramètres `OUT` et `INOUT` se lisent dans les variables passées en argument :

```sql
SET @montant = 200.00;

CALL appliquer_remise(@montant, 15);   -- INOUT en entrée-sortie, IN en entrée

SELECT @montant;                       -- la valeur mise à jour par la procédure
```

## Renvoyer des jeux de résultats

Outre les paramètres de sortie, une procédure dispose d'un second canal, plus puissant pour les données tabulaires : le **jeu de résultats**. Toute instruction `SELECT` exécutée *sans* clause `INTO` à l'intérieur du corps renvoie ses lignes directement au client, exactement comme une requête ordinaire. C'est la manière naturelle, pour une procédure, de retourner un ensemble de lignes — ce qu'une fonction ne sait pas faire.

```sql
DELIMITER $$

CREATE PROCEDURE lister_produits_chers(IN p_seuil DECIMAL(10,2))
BEGIN
    SELECT id, nom, prix
      FROM produits
     WHERE prix >= p_seuil
     ORDER BY prix DESC;
END $$

DELIMITER ;

CALL lister_produits_chers(100);   -- affiche le résultat comme une requête
```

Particularité notable : une procédure peut renvoyer **plusieurs jeux de résultats**, simplement en enchaînant plusieurs `SELECT`. Le client les reçoit successivement.

```sql
DELIMITER $$

CREATE PROCEDURE tableau_de_bord()
BEGIN
    SELECT COUNT(*) AS nb_clients   FROM clients;
    SELECT COUNT(*) AS nb_commandes FROM commandes;
    SELECT SUM(montant) AS ca_total FROM commandes;
END $$

DELIMITER ;

CALL tableau_de_bord();   -- trois jeux de résultats successifs
```

En console, ces jeux s'affichent l'un après l'autre. Depuis une application, ils doivent être parcourus explicitement (voir plus bas).

## Appeler une procédure depuis une autre routine

Une procédure peut en appeler une autre. Dans ce contexte, les paramètres `OUT` et `INOUT` reçoivent des variables locales de la routine appelante plutôt que des variables utilisateur :

```sql
DELIMITER $$

CREATE PROCEDURE traiter_client(IN p_client_id INT)
BEGIN
    DECLARE v_nb INT;

    CALL nb_commandes_client(p_client_id, v_nb);   -- v_nb : variable locale en sortie

    IF v_nb = 0 THEN
        -- traitement spécifique aux clients sans commande
        SELECT CONCAT('Client ', p_client_id, ' sans commande') AS info;
    END IF;
END $$

DELIMITER ;
```

(La déclaration de variables locales avec `DECLARE` et les structures comme `IF` relèvent du contrôle de flux, détaillé en 8.7 ; elles ne sont utilisées ici que pour illustrer l'appel imbriqué.)

## Appel depuis une application et instructions préparées

Depuis un langage applicatif, l'appel passe par le mécanisme dédié du connecteur (par exemple un *callable statement* en JDBC, ou `PDO`/`mysqli` en PHP). Deux aspects sont à gérer côté client : la lecture des paramètres `OUT`/`INOUT` après l'exécution, et le parcours des jeux de résultats lorsqu'il y en a plusieurs (les pilotes exposent une méthode pour passer au jeu suivant). Les détails par langage sont traités au chapitre 17.

Au niveau SQL, on peut aussi préparer un `CALL`, ce qui est utile pour paramétrer dynamiquement l'appel :

```sql
PREPARE appel FROM 'CALL nb_commandes_client(?, @total)';
SET @id = 42;
EXECUTE appel USING @id;
SELECT @total;
DEALLOCATE PREPARE appel;
```

Le `CALL` figure parmi les rares instructions qui, sous forme préparée, prennent en charge le renvoi de plusieurs jeux de résultats. Les instructions préparées sont approfondies en 17.9.

## Récursivité

Une procédure peut s'appeler elle-même, mais la récursivité est **désactivée par défaut** : la variable `max_sp_recursion_depth` vaut `0`, ce qui fait échouer tout appel récursif. Pour l'autoriser, on fixe une profondeur maximale (de `1` à `255`) :

```sql
SET max_sp_recursion_depth = 10;
```

Cette limite protège contre les récursions infinies qui épuiseraient la pile. À noter que cette possibilité ne concerne que les procédures : les **fonctions stockées ne peuvent pas être récursives**, quelle que soit la configuration.
