# 8.3.1 `BEFORE` et `AFTER`

La première dimension d'un trigger est son *moment* de déclenchement : il peut s'exécuter **avant** (`BEFORE`) ou **après** (`AFTER`) la modification qui le déclenche. Ce choix n'est pas une simple question de chronologie : il détermine ce que le trigger a le droit de faire. Cette section introduit la syntaxe `CREATE TRIGGER`, puis détaille les possibilités propres à chaque timing. Le choix de l'événement (`INSERT`/`UPDATE`/`DELETE`) est traité en 8.3.2, et les pseudo-enregistrements `OLD`/`NEW` en 8.3.3.

## La syntaxe `CREATE TRIGGER`

Un trigger se définit ainsi :

```sql
CREATE [OR REPLACE]
    [DEFINER = { utilisateur | CURRENT_USER }]
    TRIGGER [IF NOT EXISTS] nom_trigger
    { BEFORE | AFTER } { INSERT | UPDATE | DELETE }
    ON nom_table
    FOR EACH ROW
    [ { FOLLOWS | PRECEDES } autre_trigger ]
    corps_du_trigger
```

On y reconnaît plusieurs éléments déjà rencontrés : `CREATE OR REPLACE` et `IF NOT EXISTS` (extensions MariaDB vues en 8.1.1), la clause `DEFINER`, et un corps qui est en général un bloc `BEGIN … END` (avec, comme toujours, recours à `DELIMITER` lors d'une saisie en console). Trois clauses sont propres aux triggers : le couple **timing + événement** (`{ BEFORE | AFTER } { INSERT | UPDATE | DELETE }`), la table cible (`ON nom_table`), et `FOR EACH ROW` qui rappelle que le déclenchement est au niveau ligne. La clause facultative `FOLLOWS`/`PRECEDES` permet d'ordonner ce trigger par rapport à un autre déclencheur défini sur la même table pour le même timing et le même événement.

C'est le premier mot-clé, `BEFORE` ou `AFTER`, qui nous intéresse ici.

## `BEFORE` : agir avant l'écriture

Un trigger `BEFORE` s'exécute *avant* que la modification ne soit appliquée à la ligne. Sa particularité décisive est de pouvoir **modifier les valeurs qui vont être écrites** : en affectant `NEW.colonne`, on change ce qui sera réellement stocké. Le `BEFORE` est donc le moment privilégié pour deux familles d'actions : la **normalisation** (ajuster une valeur avant enregistrement) et la **validation** (vérifier les données et rejeter la modification si nécessaire, en levant une erreur avec `SIGNAL`, détaillé en 8.6).

```sql
DELIMITER $$

CREATE OR REPLACE TRIGGER trg_clients_before_insert
    BEFORE INSERT ON clients
    FOR EACH ROW
BEGIN
    -- normalisation : on force l'email en minuscules
    SET NEW.email = LOWER(NEW.email);

    -- validation : on rejette un email manquant
    IF NEW.email IS NULL OR NEW.email = '' THEN
        SIGNAL SQLSTATE '45000'
            SET MESSAGE_TEXT = 'Email obligatoire';
    END IF;
END $$

DELIMITER ;
```

## `AFTER` : réagir après l'écriture

Un trigger `AFTER` s'exécute *après* que la modification a été appliquée. La ligne est alors écrite : il est trop tard pour la changer, et `NEW` y est en **lecture seule** (toute affectation à `NEW` dans un trigger `AFTER` provoque une erreur). En revanche, c'est le moment idéal pour **réagir à un changement accompli** : journaliser l'opération, répercuter la modification sur d'autres tables, ou mettre à jour une donnée dérivée. Les valeurs auto-générées (comme un identifiant `AUTO_INCREMENT`) y sont également disponibles, ce qui n'est pas le cas en `BEFORE`.

```sql
DELIMITER $$

CREATE OR REPLACE TRIGGER trg_clients_after_insert
    AFTER INSERT ON clients
    FOR EACH ROW
BEGIN
    -- NEW.id contient ici l'identifiant généré, disponible en AFTER
    INSERT INTO clients_audit (client_id, action, effectue_le)
    VALUES (NEW.id, 'INSERT', NOW());
END $$

DELIMITER ;
```

## Choisir entre les deux

Le critère de décision est simple : si le trigger doit **influer sur la ligne elle-même**, il doit être `BEFORE` ; s'il doit **réagir à la modification une fois faite**, il doit être `AFTER`.

| | `BEFORE` | `AFTER` |
|---|---|---|
| Moment | avant l'application de la modification | après l'application |
| Modifier `NEW` | **oui** (change ce qui sera écrit) | non (`NEW` en lecture seule) |
| Valeurs auto-générées (`AUTO_INCREMENT`) | pas encore disponibles | disponibles |
| Usages typiques | validation, normalisation, calcul de valeurs | audit, mise à jour d'autres tables, données dérivées |

Pour chaque ligne affectée, l'ordre des opérations est : exécution du trigger `BEFORE`, puis application de la modification et vérification des contraintes, puis exécution du trigger `AFTER`. Ce déroulé explique pourquoi un `BEFORE` peut ajuster des valeurs afin qu'elles satisfassent ensuite les contraintes, et pourquoi la validation par rejet est plus naturelle en `BEFORE` : la donnée invalide est écartée avant même d'être écrite.
