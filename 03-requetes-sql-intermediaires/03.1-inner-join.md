# 3.3.1 INNER JOIN : Intersection

> **Chapitre 3 : Requêtes SQL Intermédiaires** · Section 3.3.1  
> Version de référence : **MariaDB 12.3 LTS**

---

## Définition

`INNER JOIN` est la jointure la plus courante. Elle ne conserve que les lignes pour lesquelles **une correspondance existe dans les deux tables**. Toute ligne qui n'a pas de contrepartie de l'autre côté de la jointure est purement et simplement **écartée du résultat**.

C'est en ce sens que l'on parle d'**intersection** : seules survivent les lignes appariées de part et d'autre. Si l'on reprend l'intuition des diagrammes de Venn évoquée en [section 3.3](03-jointures.md), `INNER JOIN` correspond à la zone commune aux deux ensembles.

---

## Rappel du schéma

Les exemples reposent sur les deux tables introduites en [section 3.3](03-jointures.md) :

**`client`**

| id | nom | ville |
|----|-----|-------|
| 1 | Dupont | Lille |
| 2 | Martin | Marseille |
| 3 | Leroy | Strasbourg |
| 4 | Petit | Lyon |

**`commande`** (`client_id` → `client.id`)

| id | client_id | montant | date_cmd |
|----|-----------|---------|----------|
| 1 | 1 | 120.00 | 2026-01-05 |
| 2 | 2 | 80.50 | 2026-01-07 |
| 3 | 1 | 200.00 | 2026-02-10 |
| 4 | 3 | 80.50 | 2026-02-15 |
| 5 | 2 | 150.00 | 2026-03-01 |

Rappel utile : le client **Petit (id 4) n'a aucune commande**, et **Dupont (id 1) en a deux**.

---

## Syntaxe

```sql
SELECT colonnes
FROM table_gauche
INNER JOIN table_droite
    ON table_gauche.cle = table_droite.cle;
```

Le mot-clé `INNER` est **facultatif** : écrire `JOIN` tout court équivaut à `INNER JOIN`, car c'est le type de jointure par défaut. Les deux formes ci-dessous sont strictement identiques :

```sql
... FROM client c INNER JOIN commande cmd ON c.id = cmd.client_id
... FROM client c       JOIN commande cmd ON c.id = cmd.client_id
```

---

## Exemple de base

Associons chaque commande au nom de son client :

```sql
SELECT c.nom, cmd.id AS commande, cmd.montant
FROM client c
INNER JOIN commande cmd ON c.id = cmd.client_id;
```

| nom | commande | montant |
|-----|----------|---------|
| Dupont | 1 | 120.00 |
| Martin | 2 | 80.50 |
| Dupont | 3 | 200.00 |
| Leroy | 4 | 80.50 |
| Martin | 5 | 150.00 |

Deux observations à faire sur ce résultat, qui sont la signature même de la jointure interne :

- Le client **Petit n'apparaît pas**. N'ayant aucune commande, il n'a pas de correspondance dans la table `commande` : `INNER JOIN` l'élimine.
- **Dupont apparaît deux fois**, une ligne par commande. À chaque correspondance trouvée, une ligne combinée est produite.

Le résultat compte donc 5 lignes — exactement le nombre de commandes — puisque chaque commande possède un et un seul client, tandis qu'un client peut en avoir plusieurs.

---

## Trois comportements à retenir

### Les lignes non appariées sont écartées — des deux côtés

Le cas de Petit illustre l'exclusion côté `client`. La règle vaut **symétriquement** : une commande qui ne référencerait aucun client existant serait elle aussi écartée. Dans notre schéma, la contrainte de clé étrangère interdit cette situation, mais le principe reste : `INNER JOIN` ne garde une ligne que si la correspondance existe **dans les deux sens**.

Si vous avez besoin de conserver les lignes sans correspondance — par exemple lister *tous* les clients, y compris ceux sans commande — ce n'est pas `INNER JOIN` qu'il faut, mais une jointure externe `LEFT JOIN` (voir [section 3.3.2](03.2-left-right-join.md)).

### Une correspondance multiple multiplie les lignes

Le doublon de Dupont n'est pas une anomalie : c'est le comportement attendu d'une relation **un-à-plusieurs**. Un client lié à *n* commandes engendre *n* lignes dans le résultat. Cette « multiplication » est essentielle à comprendre, notamment avant d'enchaîner sur une agrégation (voir plus bas).

### INNER JOIN est commutatif

L'ordre des tables n'a **aucune incidence sur l'ensemble des lignes** renvoyées (il ne change que l'ordre des colonnes si l'on écrit `SELECT *`). Les deux requêtes suivantes produisent les mêmes lignes :

```sql
FROM client c   INNER JOIN commande cmd ON c.id = cmd.client_id
FROM commande cmd INNER JOIN client c   ON c.id = cmd.client_id
```

Cette propriété distingue nettement `INNER JOIN` des jointures externes, où le côté « gauche » et le côté « droit » jouent au contraire des rôles différents.

---

## Équivalence avec la syntaxe implicite

Comme indiqué en [section 3.3](03-jointures.md), une jointure interne peut aussi s'écrire avec la syntaxe historique par virgule, la condition passant dans le `WHERE` :

```sql
-- Équivalent à l'INNER JOIN précédent, mais déconseillé
SELECT c.nom, cmd.id AS commande, cmd.montant
FROM client c, commande cmd
WHERE c.id = cmd.client_id;
```

Le résultat est identique, mais on privilégiera toujours la forme explicite `INNER JOIN ... ON`, plus lisible et moins sujette à l'oubli accidentel de la condition de jointure.

---

## Filtrer une jointure interne : ON ou WHERE ?

On combine très souvent une jointure avec un filtre. Ajoutons par exemple une condition sur le montant :

```sql
SELECT c.nom, cmd.montant
FROM client c
INNER JOIN commande cmd ON c.id = cmd.client_id
WHERE cmd.montant > 100;
```

| nom | montant |
|-----|---------|
| Dupont | 120.00 |
| Dupont | 200.00 |
| Martin | 150.00 |

**Pour une jointure interne**, placer cette condition dans le `WHERE` ou l'adjoindre au `ON` (`... ON c.id = cmd.client_id AND cmd.montant > 100`) donne **exactement le même résultat**. C'est une propriété spécifique à `INNER JOIN`. Attention : pour une jointure *externe*, cette distinction change radicalement le résultat — un point traité en détail en [section 3.3.2](03.2-left-right-join.md).

---

## INNER JOIN et agrégation

L'un des usages les plus fréquents consiste à joindre puis à **regrouper** (voir [section 3.2](02-regroupement-donnees.md)), par exemple pour obtenir le total dépensé par client :

```sql
SELECT
    c.nom,
    COUNT(*)         AS nb_commandes,
    SUM(cmd.montant) AS total
FROM client c
INNER JOIN commande cmd ON c.id = cmd.client_id
GROUP BY c.id, c.nom;
```

| nom | nb_commandes | total |
|-----|--------------|-------|
| Dupont | 2 | 320.00 |
| Martin | 2 | 230.50 |
| Leroy | 1 | 80.50 |

Ici encore, **Petit est absent** : `INNER JOIN` l'ayant écarté en amont, il n'entre dans aucun groupe. C'est un point décisif à garder en tête : une jointure interne suivie d'une agrégation ne peut compter que les entités **ayant au moins une correspondance**. Si l'on voulait faire figurer Petit avec un total de `0`, il faudrait un `LEFT JOIN` (voir [section 3.3.2](03.2-left-right-join.md)).

> ℹ️ On regroupe sur `c.id, c.nom` plutôt que sur `c.nom` seul : la clé primaire garantit l'unicité du groupe, et inclure `c.nom` satisfait la règle SELECT/GROUP BY rappelée en [section 3.2](02-regroupement-donnees.md).

---

## Conditions de jointure particulières

### Clés composites

Lorsque la relation repose sur plusieurs colonnes (clé composite), la clause `ON` combine les égalités avec `AND` :

```sql
... INNER JOIN détail d
    ON  d.commande_id = cmd.id
    AND d.exercice    = cmd.exercice
```

### Le cas des NULL dans la colonne de jointure

Une subtilité utile : si la colonne de jointure contient `NULL`, la correspondance **échoue toujours**, car une comparaison avec `NULL` n'est jamais vraie (voir la logique ternaire en [section 4.6](../04-concepts-avances-sql/06-gestion-valeurs-null.md)). Une commande dont le `client_id` serait `NULL` ne s'apparierait donc à aucun client et disparaîtrait du résultat d'un `INNER JOIN`. Pour la conserver malgré tout, il faudrait un `LEFT JOIN` à partir de `commande` ([section 3.3.2](03.2-left-right-join.md)).

---

## Jointures multi-tables

Une jointure interne s'enchaîne naturellement pour relier trois tables ou plus, chaque `JOIN` apportant sa propre clause `ON` :

```sql
SELECT ...
FROM client c
INNER JOIN commande cmd   ON c.id = cmd.client_id
INNER JOIN ligne_cmd l    ON l.commande_id = cmd.id;
```

Le traitement de requêtes complexes à plusieurs tables est approfondi en [section 4.5](../04-concepts-avances-sql/05-requetes-complexes-multi-tables.md).

---

## Performance

Une jointure interne s'exécute efficacement dès lors que la **colonne de jointure est indexée**. Dans notre schéma, `commande.client_id` l'est automatiquement du fait de sa contrainte de clé étrangère (voir [section 5.5.2](../05-index-et-performance/05.2-index-cles-etrangeres.md)). Pour vérifier qu'une jointure exploite bien les index plutôt que de parcourir les tables intégralement, on inspecte le plan d'exécution avec `EXPLAIN` (voir [section 5.7](../05-index-et-performance/07-analyse-plans-execution.md)).

---

## À retenir

- `INNER JOIN` ne conserve que les lignes **appariées dans les deux tables** ; les lignes sans correspondance sont écartées des deux côtés.
- Le mot-clé `INNER` est facultatif : `JOIN` seul est une jointure interne.
- Une correspondance **multiple** (un-à-plusieurs) produit plusieurs lignes — comportement normal, à anticiper avant une agrégation.
- `INNER JOIN` est **commutatif** : l'ordre des tables ne change pas l'ensemble des lignes.
- Pour une jointure interne, condition dans `ON` ou dans `WHERE` reviennent au même (ce n'est plus vrai en jointure externe).
- Une jointure interne suivie d'un `GROUP BY` ignore les entités sans correspondance : pour les inclure (avec un total nul, par exemple), utiliser `LEFT JOIN`.
- Un `NULL` dans la colonne de jointure empêche toute correspondance.
