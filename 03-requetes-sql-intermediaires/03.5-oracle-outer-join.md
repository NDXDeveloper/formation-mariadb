# 3.3.5 Syntaxe ( + ) pour jointures externes en mode Oracle 🆕

> **Chapitre 3 : Requêtes SQL Intermédiaires** · Section 3.3.5  
> 🆕 Nouveauté de la série **MariaDB 12.x** · Version de référence : **MariaDB 12.3 LTS**

---

## Contexte

Avant la normalisation de la syntaxe `JOIN ... ON` par le standard SQL, Oracle exprimait les jointures externes au moyen d'un opérateur particulier, **`(+)`**, placé dans la clause `WHERE`. Cette notation, antérieure aux jointures ANSI, reste très répandue dans le code Oracle existant.

Pour **faciliter les migrations depuis Oracle**, MariaDB reconnaît désormais cette syntaxe lorsque le **mode Oracle** est activé. Quand le mode Oracle est actif, la syntaxe `(+)` de style Oracle peut être utilisée. Il s'agit d'un dispositif de **compatibilité**, et non d'une syntaxe à privilégier : en code MariaDB natif, on continue d'écrire `LEFT`/`RIGHT JOIN` (voir [section 3.3.2](03.2-left-right-join.md)). L'intérêt est de pouvoir exécuter du SQL Oracle hérité sans le réécrire immédiatement.

---

## Le principe de l'opérateur (+)

L'opérateur `(+)` se place **du côté « déficient » de la jointure** — la table dont les lignes peuvent manquer et qui sera complétée par des `NULL`. Le `(+)` se place sur le côté de la condition de jointure qui manque d'information.

La conséquence, parfois déroutante, est que **la table préservée est celle qui ne porte PAS le `(+)`**. La documentation MariaDB formule cela en termes de parties interne et externe : la table dont les colonnes sont marquées par l'opérateur `(+)` constitue la partie interne ; la table non marquée appartient à la partie externe — c'est-à-dire le côté conservé.

Une lecture mnémotechnique : **`(+)` signifie « ajouter ici les lignes fantômes à `NULL` »**. On marque donc la table optionnelle, et l'autre est intégralement conservée.

---

## Activation du mode Oracle

L'opérateur n'est interprété qu'en mode Oracle, que l'on active pour la session (ou globalement dans la configuration) :

```sql
SET sql_mode = 'ORACLE';
```

En dehors de ce mode, un `(+)` dans une condition provoque une erreur de syntaxe. Le `sql_mode` est traité plus largement en [section 11.3](../11-administration-configuration/03-modes-sql.md).

---

## Exemple : conserver tous les clients

Reprenons le schéma `client` / `commande` de la [section 3.3](03-jointures.md), avec le client **Petit** sans commande. Pour lister tous les clients, y compris ceux sans commande, on marque la table `commande` (la déficiente) du `(+)` :

```sql
-- Mode Oracle. FROM en liste séparée par des virgules, condition dans WHERE.
SELECT c.nom, cmd.id AS commande
FROM client c, commande cmd
WHERE c.id = cmd.client_id(+);
```

| nom | commande |
|-----|----------|
| Dupont | 1 |
| Dupont | 3 |
| Martin | 2 |
| Martin | 5 |
| Leroy | 4 |
| Petit | *NULL* |

Le `(+)` étant du côté `commande`, c'est **`client` qui est préservé** : Petit apparaît avec une commande `NULL`. Cette requête est rigoureusement équivalente à la jointure externe ANSI vue en [section 3.3.2](03.2-left-right-join.md) :

```sql
-- Équivalent ANSI (à privilégier)
SELECT c.nom, cmd.id AS commande
FROM client c
LEFT JOIN commande cmd ON c.id = cmd.client_id;
```

---

## La forme miroir

En déplaçant le `(+)` sur l'autre colonne, on inverse le côté préservé. Marquer `client` conserve alors **toutes les commandes** :

```sql
SELECT c.nom, cmd.id AS commande
FROM client c, commande cmd
WHERE c.id(+) = cmd.client_id;
```

L'équivalent ANSI est une jointure préservant `commande` (`commande LEFT JOIN client`, soit `client RIGHT JOIN commande`). Dans notre schéma, la contrainte de clé étrangère garantissant que toute commande a un client, ce résultat coïncide ici avec celui d'une jointure interne — mais le mécanisme, lui, reste celui d'une jointure externe.

---

## Comment MariaDB l'interprète

En interne, MariaDB ne crée pas un nouveau mode d'exécution : il **réécrit la requête `(+)` en jointures `LEFT JOIN` classiques** en préservant autant que possible l'ordre des tables. La tâche consiste à convertir l'opérateur de jointure externe d'Oracle en `LEFT JOIN` réguliers tout en conservant l'ordre des tables d'origine quand c'est possible.

On peut d'ailleurs visualiser cette traduction : pour « réécrire » une requête utilisant la syntaxe `(+)`, on emploie `EXPLAIN EXTENDED`, dont la sortie donne une approximation de la forme `LEFT JOIN` correspondante. C'est un bon moyen, en phase de migration, de vérifier qu'on a bien compris la sémantique d'une vieille requête Oracle.

---

## Restrictions

La syntaxe `(+)` obéit à des règles strictes, que la documentation MariaDB énumère. Les principales :

- **Uniquement dans le `WHERE`.** L'opérateur `(+)` ne peut être utilisé que dans une clause `WHERE`.
- **Pas de mélange avec les jointures ANSI.** Le `(+)` ne peut pas être combiné avec d'autres méthodes de jointure : la clause `FROM` doit être une liste de tables séparées par des virgules. On ne panache donc pas `(+)` et `JOIN ... ON` dans la même requête.
- **Seulement sur une colonne de table du `SELECT` local** — pas sur une colonne issue d'une requête englobante.
- **Pas à droite d'un `IN`,** ni **dans une opération de ligne** (comparaison de n-uplets).
- **Pas avec `OR`.** L'opérateur n'est pas autorisé dans les expressions comportant un `OR`. Les conditions combinées par `AND` sont en revanche analysées indépendamment, chacune déterminant son propre côté externe.
- **Cohérence par table :** toutes les colonnes d'une même table figurant d'un côté de l'égalité doivent être marquées du `(+)`, sans mélange marqué/non marqué.

Ces contraintes — absentes de la syntaxe `LEFT/RIGHT JOIN` — sont une raison de plus de réserver le `(+)` au code hérité.

---

## Pourquoi l'éviter en code natif

Oracle lui-même décourage l'usage de cet opérateur : Oracle recommande d'employer la syntaxe `OUTER JOIN` de la clause `FROM` plutôt que l'opérateur de jointure d'Oracle. Les raisons sont les mêmes sous MariaDB :

- la syntaxe `(+)` est **moins lisible** et concentre la logique de jointure dans le `WHERE`, mêlée aux filtres ;
- elle est **moins expressive** (les restrictions ci-dessus, l'absence d'équivalent direct du `FULL OUTER JOIN`) ;
- elle est **non portable** hors du mode Oracle.

La règle pratique est donc : **comprendre `(+)` pour lire et migrer du code Oracle, mais écrire les nouvelles requêtes en `LEFT`/`RIGHT JOIN`.** La migration depuis Oracle, qui mobilise aussi d'autres dispositifs de compatibilité (`TO_DATE`, `SYS_REFCURSOR`, tableaux associatifs…), est traitée en [section 19.2.1](../19-migration-compatibilite/02.1-depuis-oracle.md).

---

## À retenir

- `(+)` est la **syntaxe Oracle historique** des jointures externes, reconnue par MariaDB **en mode Oracle** (`SET sql_mode = 'ORACLE'`) à des fins de **compatibilité/migration** 🆕.
- L'opérateur se place du **côté optionnel** (déficient) ; la table **non marquée** est celle qui est **préservée**.
- `WHERE c.id = cmd.client_id(+)` équivaut à `client LEFT JOIN commande ON c.id = cmd.client_id`.
- MariaDB **réécrit** ces requêtes en `LEFT JOIN` ; `EXPLAIN EXTENDED` révèle la forme équivalente.
- Restrictions notables : `WHERE` uniquement, `FROM` en liste de virgules (pas de mélange avec `JOIN`), pas avec `OR`, pas à droite d'un `IN`, pas dans les opérations de ligne.
- En code MariaDB natif, **préférez `LEFT`/`RIGHT JOIN`** : ne gardez `(+)` que pour lire ou migrer du code Oracle.
