# 3.3.2 LEFT/RIGHT JOIN : Jointures externes

> **Chapitre 3 : Requêtes SQL Intermédiaires** · Section 3.3.2  
> Version de référence : **MariaDB 12.3 LTS**

---

## Définition

Là où `INNER JOIN` n'avait conservé que les lignes appariées dans les deux tables ([section 3.3.1](03.1-inner-join.md)), une **jointure externe** conserve **toutes les lignes d'un côté**, même celles sans correspondance. Les colonnes manquantes du côté non apparié sont alors remplies par des **`NULL`**.

Le « côté » préservé dépend du mot-clé :

- **`LEFT JOIN`** conserve toutes les lignes de la table **de gauche** (celle qui précède le `JOIN`) ;
- **`RIGHT JOIN`** conserve toutes les lignes de la table **de droite**.

C'est précisément ce que `INNER JOIN` ne savait pas faire : lister, par exemple, *tous* les clients y compris ceux qui n'ont jamais commandé.

---

## Rappel du schéma

Les exemples reposent toujours sur les deux tables de la [section 3.3](03-jointures.md), avec le client **Petit (id 4) sans aucune commande** :

**`client`** — Dupont (1, Lille), Martin (2, Marseille), Leroy (3, Strasbourg), Petit (4, Lyon)  
**`commande`** — cmd 1→client 1 (120.00), 2→client 2 (80.50), 3→client 1 (200.00), 4→client 3 (80.50), 5→client 2 (150.00)  

---

## LEFT JOIN

### Syntaxe

```sql
SELECT colonnes
FROM table_gauche
LEFT JOIN table_droite
    ON table_gauche.cle = table_droite.cle;
```

Le mot-clé `OUTER` est facultatif : `LEFT JOIN` et `LEFT OUTER JOIN` sont équivalents.

### Exemple

Listons **tous les clients** avec leurs commandes, qu'ils en aient ou non :

```sql
SELECT c.nom, cmd.id AS commande, cmd.montant
FROM client c
LEFT JOIN commande cmd ON c.id = cmd.client_id;
```

| nom | commande | montant |
|-----|----------|---------|
| Dupont | 1 | 120.00 |
| Dupont | 3 | 200.00 |
| Martin | 2 | 80.50 |
| Martin | 5 | 150.00 |
| Leroy | 4 | 80.50 |
| Petit | *NULL* | *NULL* |

La différence avec l'`INNER JOIN` saute aux yeux : **Petit apparaît désormais**, avec des `NULL` dans les colonnes issues de `commande`, puisqu'il n'a aucune commande à apparier. Les autres clients se comportent exactement comme dans une jointure interne.

On peut résumer ainsi : **`LEFT JOIN` = résultat de l'`INNER JOIN` + les lignes de gauche non appariées, complétées par des `NULL`**. Comme pour toute jointure, l'ordre des lignes n'est pas garanti : ajoutez `ORDER BY` si nécessaire.

---

## Trouver les lignes sans correspondance

Le `LEFT JOIN` ne sert pas qu'à « compléter » : c'est aussi l'outil canonique pour **isoler les lignes orphelines**. En filtrant sur un `NULL` du côté droit, on ne garde que les lignes de gauche **sans correspondance** — c'est le motif dit *anti-jointure*.

```sql
-- Les clients qui n'ont jamais commandé
SELECT c.nom
FROM client c
LEFT JOIN commande cmd ON c.id = cmd.client_id
WHERE cmd.id IS NULL;
```

| nom |
|-----|
| Petit |

Le raisonnement : après le `LEFT JOIN`, les clients sans commande ont des `NULL` partout du côté `commande`. Le `WHERE cmd.id IS NULL` ne retient donc que ceux-là.

> 💡 **Bonne pratique.** Testez `IS NULL` sur une colonne **jamais nulle** du côté droit — idéalement sa clé primaire (`cmd.id`) ou la colonne de jointure. Tester une colonne *nullable* mêlerait deux significations différentes du `NULL` : « pas de correspondance » et « valeur réellement nulle ».

---

## RIGHT JOIN

`RIGHT JOIN` est l'exact miroir de `LEFT JOIN` : il conserve toutes les lignes de la table **de droite**. Pour préserver tous les clients en les plaçant à droite, on écrirait :

```sql
SELECT c.nom, cmd.id AS commande
FROM commande cmd
RIGHT JOIN client c ON c.id = cmd.client_id;
```

Petit y figurerait à nouveau avec une commande `NULL`, comme dans l'exemple `LEFT JOIN` précédent. Ce n'est pas un hasard : **`A RIGHT JOIN B` est strictement équivalent à `B LEFT JOIN A`**. Toute jointure droite peut donc se réécrire en jointure gauche en inversant l'ordre des tables.

En pratique, `RIGHT JOIN` est **rarement utilisé**. La convention, largement répandue, est de **toujours raisonner en `LEFT JOIN`** : on place à gauche la table dont on veut conserver toutes les lignes, et l'on lit la requête de gauche à droite. C'est cette habitude que nous recommandons.

---

## ON ou WHERE : la distinction décisive

Voici le point le plus important de cette section — et la grande différence avec l'`INNER JOIN`, pour lequel `ON` et `WHERE` étaient interchangeables ([section 3.3.1](03.1-inner-join.md)). **En jointure externe, placer une condition sur la table optionnelle dans le `ON` ou dans le `WHERE` ne produit pas le même résultat.**

Comparons une même condition (`montant > 100`) selon son emplacement.

### Condition dans le ON

```sql
SELECT c.nom, cmd.id AS commande, cmd.montant
FROM client c
LEFT JOIN commande cmd
    ON c.id = cmd.client_id AND cmd.montant > 100;
```

| nom | commande | montant |
|-----|----------|---------|
| Dupont | 1 | 120.00 |
| Dupont | 3 | 200.00 |
| Martin | 5 | 150.00 |
| Leroy | *NULL* | *NULL* |
| Petit | *NULL* | *NULL* |

La condition fait partie de **l'appariement** : pour chaque client, seules les commandes de plus de 100 sont rattachées. Mais **tous les clients restent présents**. Leroy, dont l'unique commande (80.50) ne satisfait pas la condition, n'a donc aucune correspondance et apparaît avec des `NULL` — au même titre que Petit. Résultat : **5 lignes, les 4 clients représentés**.

### Condition dans le WHERE

```sql
SELECT c.nom, cmd.id AS commande, cmd.montant
FROM client c
LEFT JOIN commande cmd ON c.id = cmd.client_id
WHERE cmd.montant > 100;
```

| nom | commande | montant |
|-----|----------|---------|
| Dupont | 1 | 120.00 |
| Dupont | 3 | 200.00 |
| Martin | 5 | 150.00 |

Ici, la jointure produit d'abord toutes les lignes (y compris les `NULL` de Petit), **puis** le `WHERE` filtre ce résultat. Or `NULL > 100` ne vaut pas vrai : les lignes à `NULL` sont éliminées, tout comme la commande à 80.50 de Leroy. **Leroy et Petit disparaissent** : on retombe sur 3 lignes, c'est-à-dire le résultat d'un `INNER JOIN`.

### La règle à retenir

| Emplacement de la condition (sur la table optionnelle) | Effet |
|---|---|
| Dans le **`ON`** | Restreint l'appariement ; **conserve** toutes les lignes du côté préservé (`NULL` si non apparié) |
| Dans le **`WHERE`** | Filtre le résultat *après* la jointure ; **élimine** les lignes non appariées → annule l'effet « externe » |

Autrement dit : un `WHERE` portant sur une colonne de la table optionnelle **retransforme silencieusement la jointure externe en jointure interne**. Une condition sur la table *préservée* (gauche), elle, peut sans risque rester dans le `WHERE`. C'est une source d'erreurs extrêmement fréquente : si vous obtenez moins de lignes qu'attendu d'un `LEFT JOIN`, vérifiez d'abord vos `WHERE`.

## LEFT JOIN et agrégation

Reprenons le calcul du total par client de la [section 3.3.1](03.1-inner-join.md), mais cette fois en voulant **inclure les clients sans commande**, avec un total de `0`. Le `LEFT JOIN` est la clé — mais il introduit un piège classique.

```sql
SELECT
    c.nom,
    COUNT(cmd.id)              AS nb_commandes,
    COALESCE(SUM(cmd.montant), 0) AS total
FROM client c
LEFT JOIN commande cmd ON c.id = cmd.client_id
GROUP BY c.id, c.nom;
```

| nom | nb_commandes | total |
|-----|--------------|-------|
| Dupont | 2 | 320.00 |
| Martin | 2 | 230.50 |
| Leroy | 1 | 80.50 |
| Petit | 0 | 0.00 |

Deux précautions, qui découlent directement de la [section 3.1](01-fonctions-agregation.md), sont indispensables :

- **`COUNT(cmd.id)` et non `COUNT(*)`.** Pour Petit, le `LEFT JOIN` produit *une* ligne (avec des `NULL`). `COUNT(*)` compterait cette ligne et renverrait `1` — un faux résultat, puisque Petit n'a aucune commande. `COUNT(cmd.id)` ne compte que les valeurs non nulles : il renvoie bien `0`. C'est l'erreur la plus répandue avec les jointures externes agrégées.
- **`COALESCE(SUM(...), 0)`.** `SUM` renvoie `NULL` sur un ensemble vide (rappel de la [section 3.1](01-fonctions-agregation.md)) ; on remplace ce `NULL` par `0` pour Petit.

Avec un `INNER JOIN`, Petit n'apparaissait pas du tout ; avec un `LEFT JOIN` correctement écrit, il figure avec `0` commande et `0.00` de total.

---

## Émuler un FULL OUTER JOIN

MariaDB ne propose pas `FULL OUTER JOIN`, qui conserverait les lignes non appariées **des deux** tables. On l'obtient en réunissant un `LEFT JOIN` et un `RIGHT JOIN` avec [`UNION`](05-operateurs-ensemblistes.md) (qui dédoublonne les lignes communes) :

```sql
SELECT c.nom, cmd.id AS commande
FROM client c
LEFT JOIN commande cmd ON c.id = cmd.client_id

UNION

SELECT c.nom, cmd.id AS commande
FROM client c
RIGHT JOIN commande cmd ON c.id = cmd.client_id;
```

Le premier `SELECT` apporte les clients sans commande, le second apporterait les commandes sans client. Dans notre schéma, la contrainte de clé étrangère interdit les commandes orphelines : ce `FULL OUTER JOIN` se réduit donc, en pratique, à un `LEFT JOIN`. La technique reste néanmoins à connaître pour les schémas où les deux tables peuvent contenir des lignes orphelines. Les opérateurs ensemblistes font l'objet de la [section 3.5](05-operateurs-ensemblistes.md).

---

## Performance

Comme les jointures internes, les jointures externes s'appuient sur les **index des colonnes de jointure** pour être efficaces (voir [section 5.5.2](../05-index-et-performance/05.2-index-cles-etrangeres.md)). On notera toutefois qu'un `LEFT JOIN` contraint davantage l'optimiseur : l'ordre de traitement des tables est en partie imposé par la sémantique (le côté préservé), là où une jointure interne, commutative, laisse plus de liberté au [planificateur de requêtes](../05-index-et-performance/07-analyse-plans-execution.md).

---

## À retenir

- Une jointure externe conserve toutes les lignes d'un côté ; les colonnes non appariées de l'autre côté valent `NULL`.
- `LEFT JOIN` préserve la table de gauche, `RIGHT JOIN` celle de droite ; `OUTER` est facultatif.
- `A RIGHT JOIN B` ≡ `B LEFT JOIN A` : on standardise en pratique sur `LEFT JOIN`.
- Le motif `LEFT JOIN ... WHERE table_droite.cle IS NULL` isole les lignes **sans correspondance** (anti-jointure).
- **`ON` vs `WHERE` (table optionnelle)** : une condition dans le `ON` préserve les lignes non appariées ; la même condition dans le `WHERE` les élimine et transforme la jointure externe en jointure interne.
- En agrégation sur un `LEFT JOIN`, utiliser **`COUNT(colonne_droite)`** (pas `COUNT(*)`) et envelopper `SUM` dans un `COALESCE(..., 0)`.
- `FULL OUTER JOIN` n'existe pas : on l'émule par `LEFT JOIN UNION RIGHT JOIN`.
