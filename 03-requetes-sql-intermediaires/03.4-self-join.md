# 3.3.4 Self-Join : Joindre une table à elle-même

> **Chapitre 3 : Requêtes SQL Intermédiaires** · Section 3.3.4  
> Version de référence : **MariaDB 12.3 LTS**

---

## Définition

Un **self-join** (auto-jointure) consiste à joindre une table **à elle-même**. Ce n'est pas un nouveau *type* de jointure : on emploie un `INNER JOIN` ou un `LEFT JOIN` ordinaire, mais la même table apparaît **deux fois** dans la requête, distinguée par des **alias** différents.

L'idée clé est là : pour relier une ligne à une autre ligne *de la même table*, il faut pouvoir nommer séparément « la ligne courante » et « la ligne associée ». Les alias remplissent ce rôle en donnant deux identités logiques à une seule table physique. C'est pourquoi, contrairement aux jointures vues précédemment, **les alias ne sont pas une commodité mais une nécessité absolue** dans un self-join.

On y recourt principalement dans deux situations : naviguer dans une **relation hiérarchique** que la table entretient avec elle-même (employé → responsable, catégorie → catégorie parente, commentaire → commentaire parent), et **comparer des lignes entre elles** au sein de la même table.

---

## Schéma d'exemple

Le cas d'école est une table d'employés où chaque ligne référence son responsable par une **clé étrangère pointant vers la même table** :

```sql
CREATE TABLE employe (
    id         INT PRIMARY KEY,
    nom        VARCHAR(50),
    manager_id INT,
    salaire    DECIMAL(10,2),
    CONSTRAINT fk_employe_manager
        FOREIGN KEY (manager_id) REFERENCES employe(id)
);
```

| id | nom | manager_id | salaire |
|----|-----|------------|---------|
| 1 | Alice | *NULL* | 6000.00 |
| 2 | Bruno | 1 | 4000.00 |
| 3 | Chloé | 1 | 4200.00 |
| 4 | David | 2 | 4500.00 |
| 5 | Emma | 2 | 3100.00 |

La hiérarchie correspondante : **Alice** dirige (elle n'a pas de responsable, d'où `manager_id` à `NULL`) ; **Bruno** et **Chloé** lui sont rattachés ; **David** et **Emma** dépendent de Bruno.

---

## Exemple de base : associer chaque employé à son responsable

```sql
SELECT e.nom AS employe, m.nom AS manager
FROM employe e
INNER JOIN employe m ON e.manager_id = m.id;
```

| employe | manager |
|---------|---------|
| Bruno | Alice |
| Chloé | Alice |
| David | Bruno |
| Emma | Bruno |

La table `employe` est citée deux fois : l'alias `e` représente l'employé, l'alias `m` représente son responsable. La condition `e.manager_id = m.id` relie le `manager_id` de l'un à l'`id` de l'autre — tous deux issus de la même table.

Notez qu'**Alice n'apparaît pas** : son `manager_id` valant `NULL`, elle n'a aucune correspondance, et `INNER JOIN` l'écarte (rappel de la [section 3.3.1](03.1-inner-join.md) sur les `NULL` dans la colonne de jointure).

---

## Le rôle central des alias

Sans alias, la requête `... employe JOIN employe ON employe.manager_id = employe.id` serait à la fois ambiguë et dénuée de sens : MariaDB ne saurait pas distinguer les deux occurrences de la table. Les alias `e` et `m` lèvent l'ambiguïté en créant deux **rôles** distincts pour la même table :

- `e` — l'employé, « le côté qui pointe vers son responsable » ;
- `m` — le responsable, « le côté référencé ».

Cette distinction par les rôles est tout l'art du self-join. Bien nommer les alias (`e`/`m`, `enfant`/`parent`, `emp`/`mgr`…) rend ces requêtes nettement plus lisibles.

---

## Inclure le sommet de la hiérarchie

Pour faire figurer Alice — la racine, sans responsable — il suffit de remplacer l'`INNER JOIN` par un `LEFT JOIN`, exactement selon la logique de la [section 3.3.2](03.2-left-right-join.md). On complète le `NULL` du manager par un libellé via `COALESCE` :

```sql
SELECT
    e.nom                       AS employe,
    COALESCE(m.nom, '(direction)') AS manager
FROM employe e
LEFT JOIN employe m ON e.manager_id = m.id;
```

| employe | manager |
|---------|---------|
| Alice | (direction) |
| Bruno | Alice |
| Chloé | Alice |
| David | Bruno |
| Emma | Bruno |

Le choix entre `INNER JOIN` et `LEFT JOIN` détermine donc, comme toujours, si la racine de la hiérarchie figure ou non dans le résultat.

---

## Comparer des lignes entre elles

L'autre grand usage du self-join est la **mise en relation de lignes au sein d'une même table** selon une condition.

### Trouver les collègues (même responsable)

```sql
SELECT e1.nom AS collegue_1, e2.nom AS collegue_2
FROM employe e1
INNER JOIN employe e2
    ON  e1.manager_id = e2.manager_id
   AND e1.id < e2.id;
```

| collegue_1 | collegue_2 |
|------------|------------|
| Bruno | Chloé |
| David | Emma |

La condition `e1.manager_id = e2.manager_id` rapproche les employés partageant un responsable. Mais c'est la seconde condition, **`e1.id < e2.id`**, qui est l'idiome essentiel ici ; elle remplit deux fonctions :

- elle **évite d'apparier une ligne avec elle-même** (sans elle, Bruno serait listé comme son propre collègue) ;
- elle **élimine les doublons symétriques** : on obtient la paire (Bruno, Chloé) mais pas aussi (Chloé, Bruno).

Remplacer `<` par `<>` produirait au contraire chaque paire deux fois et conserverait les auto-appariements : la comparaison stricte par l'identifiant est la solution propre. À noter qu'Alice, dont le `manager_id` est `NULL`, n'apparaît dans aucune paire — `NULL = NULL` n'étant pas vrai (voir la logique ternaire en [section 4.6](../04-concepts-avances-sql/06-gestion-valeurs-null.md)).

### Trouver les employés mieux payés que leur responsable

```sql
SELECT
    e.nom     AS employe,
    e.salaire AS salaire_employe,
    m.nom     AS manager,
    m.salaire AS salaire_manager
FROM employe e
INNER JOIN employe m ON e.manager_id = m.id
WHERE e.salaire > m.salaire;
```

| employe | salaire_employe | manager | salaire_manager |
|---------|-----------------|---------|-----------------|
| David | 4500.00 | Bruno | 4000.00 |

Ici, le self-join met en regard le salaire de chaque employé (`e.salaire`) et celui de son responsable (`m.salaire`) sur une même ligne, ce qui permet de les comparer dans le `WHERE`. Seul David gagne davantage que son responsable.

---

## Self-join et hiérarchies : une limite à connaître

Un self-join ne franchit **qu'un seul niveau** de hiérarchie : il relie un employé à son responsable *direct*. Pour remonter ou descendre une chaîne hiérarchique de profondeur **quelconque** — l'ensemble des responsables d'Emma jusqu'au sommet, ou tous les subordonnés (directs et indirects) d'Alice — un seul self-join ne suffit pas. Il faudrait soit enchaîner autant de self-joins que de niveaux (ce qui suppose une profondeur fixe et connue), soit recourir à une **requête récursive** `WITH RECURSIVE`, qui est l'outil adapté à la traversée d'arborescences de profondeur arbitraire. C'est l'objet de la [section 4.1 sur les requêtes récursives](../04-concepts-avances-sql/01-requetes-recursives.md).

En résumé : **self-join pour un niveau, CTE récursive pour une profondeur variable.**

---

## Performance

Un self-join s'optimise comme n'importe quelle jointure : la **colonne d'auto-référence doit être indexée**. Dans notre schéma, `manager_id` l'est automatiquement grâce à sa contrainte de clé étrangère (voir [section 5.5.2](../05-index-et-performance/05.2-index-cles-etrangeres.md)). Pour les self-joins de comparaison (paires), gardez à l'esprit que la condition `e1.id < e2.id` n'empêche pas MariaDB d'examiner de nombreuses combinaisons : sur de grandes tables, vérifiez le plan avec `EXPLAIN` ([section 5.7](../05-index-et-performance/07-analyse-plans-execution.md)).

---

## À retenir

- Un self-join joint une table à elle-même ; ce n'est pas un type de jointure mais une **technique** reposant sur des alias.
- Les **alias sont obligatoires** : ils donnent deux rôles distincts (ex. `e` employé / `m` responsable) à une même table.
- On l'utilise pour les **relations hiérarchiques** (employé → responsable) et pour **comparer des lignes** d'une même table.
- `INNER JOIN` exclut la racine sans parent ; `LEFT JOIN` (+ `COALESCE`) la conserve.
- Pour les comparaisons par paires, la condition **`id < id`** évite les auto-appariements et les doublons symétriques.
- Un self-join ne traverse **qu'un niveau** de hiérarchie ; pour une profondeur quelconque, utiliser une **CTE récursive** ([section 4.1](../04-concepts-avances-sql/01-requetes-recursives.md)).
- Indexer la colonne d'auto-référence (automatique si clé étrangère).
