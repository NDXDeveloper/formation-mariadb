# 11.2.1 — `SHOW VARIABLES` et `SET`

> Première sous-section de 11.2. Elle outille concrètement la consultation et la modification des variables, dans leurs portées globale et de session introduites en 11.2. La frontière entre variables modifiables à chaud et variables nécessitant un redémarrage est traitée séparément en 11.2.2.

---

## Lire les variables : `SHOW VARIABLES`

La commande la plus directe pour consulter les variables est `SHOW VARIABLES`. Sans mot-clé de portée, elle affiche la vue de **session** de la connexion courante :

```sql
-- Variables de la session courante (portée par défaut)
SHOW VARIABLES;
```

Pour cibler explicitement une portée, on précise `GLOBAL` ou `SESSION` :

```sql
-- Valeurs globales (celles héritées par les nouvelles connexions)
SHOW GLOBAL VARIABLES;

-- Valeurs de la session courante (équivalent de SHOW VARIABLES)
SHOW SESSION VARIABLES;
```

La liste complète comptant plusieurs centaines d'entrées, on la filtre presque toujours. La clause `LIKE` accepte les jokers `%` (toute chaîne) et `_` (un caractère), tandis que `WHERE` permet une condition plus expressive :

```sql
-- Une variable précise
SHOW VARIABLES LIKE 'max_connections';

-- Toutes les variables InnoDB liées au buffer pool
SHOW GLOBAL VARIABLES LIKE 'innodb_buffer%';

-- Plusieurs variables nommées
SHOW VARIABLES WHERE Variable_name IN ('sort_buffer_size', 'join_buffer_size');
```

Le résultat se présente sous forme de deux colonnes, `Variable_name` et `Value` — pratique pour une lecture humaine, mais peu adapté à un traitement programmatique, pour lequel on préférera les approches suivantes.

---

## Lire une variable précise : la syntaxe `@@`

Pour récupérer la valeur d'une variable dans une requête (et donc la réutiliser dans une expression), on emploie le préfixe `@@`. Sans qualificateur, il renvoie la valeur de session ; on peut aussi imposer la portée avec `@@global.` ou `@@session.` :

```sql
-- Valeur de session (par défaut)
SELECT @@sort_buffer_size;

-- Portée explicite
SELECT @@global.max_connections;
SELECT @@session.sort_buffer_size;

-- Plusieurs variables d'un coup
SELECT @@global.wait_timeout, @@global.interactive_timeout;
```

Cette forme est idéale dans les scripts et les procédures, où l'on a besoin de la *valeur* elle-même plutôt que d'un tableau d'affichage.

---

## Aller plus loin : `INFORMATION_SCHEMA.SYSTEM_VARIABLES`

MariaDB expose une vue bien plus riche que `SHOW VARIABLES` à travers la table `INFORMATION_SCHEMA.SYSTEM_VARIABLES`. Au-delà de la valeur courante, elle indique notamment **d'où provient** la valeur globale active, ce qui en fait un excellent outil de diagnostic.

```sql
SELECT VARIABLE_NAME,
       SESSION_VALUE,
       GLOBAL_VALUE,
       GLOBAL_VALUE_ORIGIN,
       GLOBAL_VALUE_PATH
FROM   information_schema.SYSTEM_VARIABLES
WHERE  VARIABLE_NAME = 'wait_timeout';
```

La colonne `GLOBAL_VALUE_ORIGIN` précise si la valeur vient d'un `SET` en SQL, d'un fichier de configuration, de la ligne de commande, d'une valeur calculée automatiquement ou de la valeur compilée par défaut ; `GLOBAL_VALUE_PATH` donne, le cas échéant, le chemin du fichier d'où la valeur a été lue. Lorsqu'un réglage ne correspond pas à ce que l'on attendait, cette table répond directement à la question « mais d'où sort cette valeur ? » — un fichier oublié, un `SET` antérieur, ou simplement le défaut du serveur.

En interrogeant deux variables — l'une fixée dans un fichier, l'autre laissée à sa valeur compilée — le diagnostic est immédiat :

```sql
SELECT VARIABLE_NAME, GLOBAL_VALUE, GLOBAL_VALUE_ORIGIN, GLOBAL_VALUE_PATH
FROM   information_schema.SYSTEM_VARIABLES
WHERE  VARIABLE_NAME IN ('max_connections', 'wait_timeout')\G
```

```text
*************************** 1. row ***************************
      VARIABLE_NAME: MAX_CONNECTIONS
       GLOBAL_VALUE: 250
GLOBAL_VALUE_ORIGIN: CONFIG
  GLOBAL_VALUE_PATH: /etc/mysql/conf.d/zz-tuning.cnf
*************************** 2. row ***************************
      VARIABLE_NAME: WAIT_TIMEOUT
       GLOBAL_VALUE: 28800
GLOBAL_VALUE_ORIGIN: COMPILE-TIME
  GLOBAL_VALUE_PATH: NULL
```

On voit immédiatement que `max_connections` provient d'un fichier précis (`CONFIG`, avec son chemin), tandis que `wait_timeout` est resté à sa valeur compilée par défaut (`COMPILE-TIME`, sans chemin). C'est souvent ce qui permet de comprendre, en quelques secondes, pourquoi un réglage « ne prend pas » comme prévu — par exemple parce qu'un *drop-in* oublié le redéfinit (cf. §11.1.3).

## Modifier les variables : `SET`

La commande `SET` change la valeur d'une variable, dans l'une ou l'autre portée. Trois écritures équivalentes existent pour la session, qui reste la portée par défaut :

```sql
-- Modifier la session courante (les trois formes sont équivalentes)
SET sort_buffer_size = 4194304;
SET SESSION sort_buffer_size = 4194304;
SET @@session.sort_buffer_size = 4194304;
```

Pour agir sur la valeur globale — celle qui régit l'instance et que les nouvelles connexions hériteront — on emploie `GLOBAL` ou `@@global.` :

```sql
-- Modifier la valeur globale
SET GLOBAL max_connections = 300;
SET @@global.max_connections = 300;
```

Rappelons le comportement vu en 11.2 : modifier une variable globale ne touche pas les sessions déjà ouvertes ; cela ne change que la valeur de départ des connexions futures (et, pour les variables purement globales, le comportement de l'instance).

On peut enchaîner plusieurs affectations dans une même instruction. Attention à un piège classique : **le mot-clé de portée se propage aux affectations suivantes** tant qu'un autre mot-clé n'apparaît pas. Dans `SET GLOBAL a = 1, b = 2`, `a` **et** `b` sont modifiées globalement — et non, comme on pourrait le croire, `a` en global et `b` en session. Pour mélanger les portées, il faut donc replacer explicitement un mot-clé devant chaque variable qui change de portée :

```sql
-- a ET b sont globales : le mot-clé GLOBAL se propage à long_query_time
SET GLOBAL slow_query_log = ON, long_query_time = 2;

-- Pour mélanger les portées, on l'indique explicitement
SET GLOBAL slow_query_log = ON, SESSION sort_buffer_size = 4194304;

-- La forme @@ qualifiée variable par variable lève toute ambiguïté
SET @@global.slow_query_log = ON, @@session.sort_buffer_size = 4194304;
```

Notez enfin que `SET GLOBAL` ne fonctionne que pour les variables **dynamiques** ; certaines exigent un redémarrage pour changer de valeur. Cette distinction, essentielle, fait l'objet de la sous-section suivante (§11.2.2).

---

## Valeurs et unités : une différence avec les fichiers

Un point de vigilance concerne l'écriture des tailles. Dans un fichier de configuration ou sur la ligne de commande, on peut utiliser les suffixes d'unité `K`, `M`, `G` (par exemple `sort_buffer_size = 256M`). **En SQL, ces suffixes ne sont pas valides** : une instruction `SET` attend une valeur numérique ou une expression.

```sql
-- ✅ En SQL : une valeur numérique ou une expression
SET GLOBAL sort_buffer_size = 256 * 1024 * 1024;   -- 256 Mo, lisible

-- ❌ Le suffixe M est rejeté en SQL (ERROR 1232, « Incorrect argument type »)
-- SET GLOBAL sort_buffer_size = 256M;
```

Écrire `256 * 1024 * 1024` plutôt que `268435456` conserve la lisibilité tout en restant du SQL valide.

---

## Privilèges requis

Modifier une variable de **session** ne demande, dans la grande majorité des cas, aucun privilège particulier : un utilisateur ajuste librement le comportement de sa propre connexion. Modifier une variable **globale**, en revanche, agit sur toute l'instance et requiert donc un privilège élevé (historiquement le privilège `SUPER`). C'est une protection logique : on ne laisse pas n'importe quel compte applicatif redéfinir le fonctionnement du serveur pour l'ensemble des utilisateurs. Les privilèges et leur gestion fine sont traités au chapitre 10.

---

## Persistance : `SET GLOBAL` n'écrit pas sur disque

C'est le point à ne jamais oublier. Une modification par `SET GLOBAL` agit **immédiatement** sur le serveur en marche, mais **n'est pas écrite sur disque** : au prochain redémarrage, la variable reprend la valeur issue des fichiers de configuration (ou, à défaut, sa valeur par défaut). Pour qu'un réglage survive à un redémarrage, il faut le reporter dans un fichier de configuration, comme vu en section 11.1.

Cette mécanique distingue MariaDB de MySQL. Depuis MySQL 8, la commande `SET PERSIST` enregistre la modification dans un fichier dédié (`mysqld-auto.cnf`) pour la rendre persistante sans toucher au `my.cnf`. MariaDB **ne dispose pas de cette commande** et continue de s'appuyer sur les fichiers d'options pour la persistance au démarrage. La conséquence pratique est nette : sur MariaDB, le couple « `SET GLOBAL` pour agir tout de suite + écriture dans un fichier pour pérenniser » reste la marche à suivre, et c'est précisément le réflexe en deux temps évoqué en introduction de la section 11.2.

---

## En résumé

`SHOW VARIABLES` (avec `LIKE` ou `WHERE`) et la syntaxe `@@` permettent de lire les variables dans l'une ou l'autre portée, tandis que `INFORMATION_SCHEMA.SYSTEM_VARIABLES` offre une vue enrichie indiquant l'origine de chaque valeur. La commande `SET`, avec ses mots-clés `GLOBAL` et `SESSION`, les modifie — en gardant à l'esprit que la portée s'applique variable par variable, que les suffixes d'unité y sont interdits, que la portée globale exige un privilège élevé, et surtout qu'un `SET GLOBAL` n'est pas persistant. Reste à savoir *quelles* variables peuvent réellement changer sans redémarrage : c'est la distinction entre variables dynamiques et statiques, objet de la sous-section **11.2.2**.
