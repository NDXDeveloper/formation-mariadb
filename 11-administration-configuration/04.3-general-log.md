# 11.4.3 — General log (journal général des requêtes)

Le journal général enregistre *chaque* requête SQL reçue d'un client, ainsi que chaque connexion et déconnexion. C'est l'outil de débogage le plus complet de MariaDB — et, parce qu'il consigne absolument tout, le plus volumineux et le plus coûteux. Désactivé par défaut, il est destiné à des sessions d'observation ciblées plutôt qu'à un usage permanent.

## Rôle et contenu

Là où le journal des requêtes lentes ne retient que les requêtes coûteuses, le journal général capture l'intégralité du trafic : `SELECT`, modifications, instructions d'administration, *prepared statements*, ouvertures et fermetures de connexions. Une particularité essentielle : il enregistre chaque requête **dès sa réception** par le serveur, et non au moment de la validation. Une requête qui échoue, ou dont la transaction est ensuite annulée, apparaît donc tout de même dans le journal — ce qui en fait un outil précieux pour comprendre exactement ce qu'une application envoie au serveur.

C'est aussi ce qui le distingue du journal binaire (§11.5) : ce dernier n'enregistre que les opérations *modifiant les données*, et seulement *au commit*, tandis que le journal général enregistre tout, immédiatement. Si l'objectif est uniquement de tracer les changements de données, le journal binaire est plus adapté ; si c'est de tracer les accès à des fins d'audit durable, le plugin d'audit (§10.8) est préférable, car plus sélectif et plus robuste.

## Activer le journal

L'activation se fait à chaud, ou dans un fichier de configuration :

```sql
SET GLOBAL general_log = ON;
```

```ini
[mariadbd]
general_log
general_log_file = /var/log/mysql/general.log
```

On contrôle l'état avec :

```sql
SHOW GLOBAL VARIABLES LIKE 'general_log%';
```

## Destination : fichier ou table

Comme les autres journaux configurables, la destination dépend de `log_output`. En mode `FILE` (par défaut), le journal est écrit dans `general_log_file` (par défaut `${hostname}.log` dans le `datadir`). En mode `TABLE`, il alimente la table `mysql.general_log` :

```sql
SET GLOBAL log_output = 'TABLE';
SELECT * FROM mysql.general_log\G
```

Cette table comporte les colonnes `event_time`, `user_host`, `thread_id`, `server_id`, `command_type` et `argument`. À noter : quelle que soit la configuration, les premières entrées écrites au démarrage du serveur le sont toujours dans un fichier.

## Format d'une entrée (mode fichier)

En mode fichier, le journal commence par un en-tête indiquant le binaire, la version, le port et le socket, suivi d'une ligne d'intitulés de colonnes. Chaque entrée se compose ensuite d'un horodatage, de l'identifiant du thread (l'ID de connexion, identique à celui des journaux d'erreurs et de requêtes lentes), du type de commande, puis de l'argument :

```
/usr/sbin/mariadbd, Version: 12.3.2-MariaDB (MariaDB Server). started with:
Tcp port: 3306  Unix socket: /run/mysqld/mysqld.sock
Time                 Id Command    Argument
2026-02-06 18:40:04  312 Connect   app@10.0.0.12 on shop using TCP/IP
2026-02-06 18:40:04  312 Query     SELECT id, total FROM orders WHERE customer_id = 90431
2026-02-06 18:40:05  312 Query     UPDATE orders SET status = 'shipped' WHERE id = 5512
2026-02-06 18:40:07  312 Quit
```

Le type de commande renseigne sur la nature de l'événement : `Connect` (nouvelle connexion, avec l'utilisateur et la base), `Query` (instruction SQL), `Init DB` (changement de base courante), `Prepare`/`Execute` (*prepared statements*), `Quit` (déconnexion). L'identifiant de thread (`Id`) permet de suivre toutes les opérations d'une même connexion. Deux points pratiques : le format de l'horodatage peut varier selon la version (par exemple `yymmdd h:mm:ss` ou ISO 8601), et comme l'argument contient le texte brut de la requête, une entrée peut s'étendre sur plusieurs lignes lorsque la requête elle-même en contient.

## Restreindre la journalisation

Un utilisateur disposant du privilège `SUPER` peut désactiver le journal général pour sa seule connexion, sans affecter les autres :

```sql
SET SESSION SQL_LOG_OFF = 1;
```

Il est par ailleurs possible d'exclure globalement certains types d'instructions via `log_disabled_statements` (disponible depuis MariaDB 10.3.1, réglable au démarrage uniquement) — par exemple les instructions exécutées par le thread de réplication et celles des procédures stockées :

```ini
[mariadbd]
general_log
log_disabled_statements = 'slave,sp'
```

## Sécurité

Le journal général consigne les instructions telles qu'elles sont reçues : il peut donc exposer des données métier confidentielles et, selon les commandes exécutées, des informations d'authentification. Comme le journal des requêtes lentes, il n'est pas chiffré nativement par MariaDB. Il convient donc d'en restreindre strictement les permissions, de le protéger par un chiffrement au niveau du système de fichiers (ou de l'écrire en table, combinée au chiffrement des données au repos — chapitre 18), et de ne pas se reposer sur une éventuelle occultation automatique des secrets. Pour une traçabilité durable et maîtrisée des accès, le plugin d'audit (§10.8) reste la solution recommandée.

## Impact sur les performances

Enregistrer chaque requête a un coût : sur un serveur actif, le journal général génère un volume d'écritures considérable et alourdit le traitement de chaque requête. Il n'est donc pas destiné à rester actif en production. La bonne pratique consiste à l'activer pour une fenêtre de diagnostic courte et ciblée — il est dynamique, donc activable et désactivable sans redémarrage — puis à le couper une fois l'observation terminée.

## Rotation

En mode fichier, la rotation s'appuie sur `logrotate` ; compte tenu de la vitesse à laquelle le fichier grossit, une rotation fréquente est recommandée. Après renommage du fichier courant, on demande la réouverture avec :

```sql
FLUSH GENERAL LOGS;
```

(`FLUSH LOGS` rouvre l'ensemble des journaux.)

## En résumé

Le journal général est l'outil de débogage exhaustif de MariaDB : il enregistre tout le trafic, immédiatement, ce qui le rend idéal pour comprendre le comportement précis d'une application, mais trop coûteux pour un usage permanent. On l'active ponctuellement, après avoir réfléchi à sa destination (fichier ou table) et à sa protection.

Cette sous-section clôt le panorama des trois journaux courants de la section 11.4 : le **journal d'erreurs** (§11.4.1) pour diagnostiquer les incidents, le **journal des requêtes lentes** (§11.4.2) pour optimiser, et le **journal général** pour déboguer en profondeur. Pour le suivi des modifications de données et la réplication, on poursuivra avec le **journal binaire** (§11.5) ; pour l'audit, avec le **plugin d'audit** (§10.8).
