# 11.6.1 — OPTIMIZE TABLE

`OPTIMIZE TABLE` sert à récupérer l'espace inutilisé et à défragmenter une table. Comme annoncé en §11.6, son comportement diffère fortement selon le moteur de stockage — c'est le point le plus important à retenir. L'instruction a en réalité deux fonctions : défragmenter une table, ou réorganiser l'index *fulltext* d'InnoDB.

## Syntaxe

```sql
OPTIMIZE [NO_WRITE_TO_BINLOG | LOCAL] TABLE
    nom_table [, nom_table] ...
    [WAIT n | NOWAIT]
```

L'instruction requiert les privilèges `SELECT` et `INSERT` sur la table. Les options `WAIT n` / `NOWAIT` fixent le délai d'attente d'acquisition du verrou. Par défaut, `OPTIMIZE TABLE` est **journalisé dans le binlog et répliqué** ; le mot-clé `NO_WRITE_TO_BINLOG` (dont `LOCAL` est l'alias) empêche cette journalisation. L'instruction n'est par ailleurs pas écrite au binlog si `read_only` est actif.

```sql
-- Défragmenter une table
OPTIMIZE TABLE commandes;

-- Sans réplication (à lancer alors sur chaque serveur)
OPTIMIZE LOCAL TABLE commandes;

-- Plusieurs tables en une fois
OPTIMIZE TABLE commandes, lignes_commande;
```

## Pourquoi et quand l'utiliser

`OPTIMIZE` se justifie principalement dans deux situations : après avoir supprimé une part importante d'une table, ou après de nombreuses modifications sur une table à lignes de longueur variable (colonnes `VARCHAR`, `VARBINARY`, `BLOB` ou `TEXT`). Les lignes supprimées sont en effet conservées dans une liste chaînée et leurs emplacements sont réutilisés par les `INSERT` ultérieurs, mais la fragmentation s'accumule au fil du temps : espace gaspillé, parcours ralentis. La défragmentation récupère cet espace, améliore la localité des données et rafraîchit les statistiques.

Ce n'est pas une tâche de routine à exécuter trop fréquemment — en particulier sur InnoDB, où l'opération est coûteuse (voir ci-dessous). On la déclenche lorsque la fragmentation est réellement significative. Pour estimer cet espace récupérable, la colonne `data_free` d'`INFORMATION_SCHEMA.TABLES` est un bon indicateur :

```sql
SELECT table_name, data_free
FROM   information_schema.tables
WHERE  table_schema = 'ma_base' AND data_free > 0;
```

## Comportement selon le moteur

### InnoDB

Sur le moteur par défaut, `OPTIMIZE TABLE` se traduit par une **reconstruction complète** de la table (une opération de DDL en ligne, équivalente à un `ALTER TABLE … FORCE`). Cette reconstruction est en grande partie réalisable **en ligne** : les écritures concurrentes restent généralement possibles pendant l'opération, qui ne pose qu'un verrou de métadonnées partagé. Elle reste néanmoins exigeante : sa durée est proportionnelle à la taille de la table et au nombre d'index, elle peut consommer jusqu'au double de la taille de la table en espace temporaire, et elle sollicite fortement le processeur.

Un point essentiel concerne la **restitution de l'espace**. Avec `innodb_file_per_table` (le réglage par défaut), chaque table possède son propre fichier `.ibd` : l'espace libéré est alors rendu au système d'exploitation. Si l'on utilise au contraire le tablespace système partagé, l'espace est récupéré *au sein* d'InnoDB mais **n'est pas rendu à l'OS** (le fichier `ibdata1` peut même grossir). La reconstruction met par ailleurs à jour les statistiques de la table.

Pour de très grandes tables très sollicitées, un outil de modification de schéma en ligne (`pt-online-schema-change`, `gh-ost` — §16.8.3) verrouille souvent moins qu'un `OPTIMIZE` direct.

> **Note historique.** MariaDB a un temps proposé une défragmentation *en place* (sans reconstruction) activée par la variable `innodb_defragment`. Cette fonctionnalité a été dépréciée en MariaDB 11.0 puis **retirée en 11.1** : elle n'existe donc plus en 12.3, où `OPTIMIZE` sur InnoDB reste une reconstruction.

### Aria et MyISAM

Sur ces moteurs, `OPTIMIZE` réorganise physiquement le fichier de données, trie les pages d'index, met à jour les statistiques et récupère l'espace inutilisé. Le moteur Aria propose en outre un suivi de progression de l'opération. Côté MyISAM, un détail mérite l'attention : tant qu'une table fragmentée n'a pas été optimisée, les **insertions concurrentes** y sont désactivées — sauf si la variable `concurrent_insert` est réglée sur `ALWAYS`.

### Autres moteurs

Pour les autres moteurs (par exemple `MEMORY`), `OPTIMIZE` ne fait rien par défaut et renvoie le message « *The storage engine for the table doesn't support optimize* ». Le moteur `ARCHIVE`, lui, est pris en charge.

## Le cas particulier de l'index *fulltext* InnoDB

Lorsque des lignes sont ajoutées ou supprimées dans un index *fulltext* InnoDB, celui-ci n'est pas réorganisé immédiatement (les changements sont stockés à part), car ce serait coûteux. La réorganisation complète n'a lieu qu'à l'exécution d'un `OPTIMIZE TABLE`, et seulement si l'on bascule au préalable la variable `innodb_optimize_fulltext_only` à `1` — un réglage temporaire à remettre à `0` ensuite :

```sql
SET GLOBAL innodb_optimize_fulltext_only = 1;
OPTIMIZE TABLE articles;                          -- réorganise l'index fulltext
SET GLOBAL innodb_optimize_fulltext_only = 0;     -- réglage temporaire
```

La réorganisation pouvant être longue, `innodb_ft_num_word_optimize` limite le traitement à un certain nombre de mots (2000 par défaut) ; on enchaîne plusieurs `OPTIMIZE` pour réorganiser l'index en totalité.

## Partitions

`OPTIMIZE` s'applique aussi aux tables partitionnées, et l'on peut ne traiter que certaines partitions :

```sql
ALTER TABLE ventes OPTIMIZE PARTITION p2025, p2026;
```

On rappellera toutefois que l'utilitaire `mariadb-check` ne fonctionne pas avec les tables partitionnées (§11.6) : pour celles-ci, on passe donc par ces instructions SQL.

## Lire le résultat

L'instruction renvoie un petit jeu de résultats décrivant le déroulement, avec les colonnes `Table`, `Op` (l'opération, `optimize`), `Msg_type` (`status`, `note`, `warning` ou `error`) et `Msg_text`. Une exécution réussie se termine par une ligne de type `status` portant le message `OK`.

## Bonnes pratiques et impact

`OPTIMIZE` est une opération lourde, à planifier pendant une fenêtre de maintenance ou des heures creuses. Sur InnoDB, on s'assure de disposer d'environ deux fois la taille de la table en espace libre avant de lancer la reconstruction. En réplication, on l'exécute généralement serveur par serveur avec `LOCAL` plutôt que de la propager. Pour les très grandes tables, les outils de modification de schéma en ligne sont préférables. Enfin, on évite d'optimiser InnoDB trop souvent : on déclenche l'opération sur constat d'une fragmentation réelle (`data_free`), pas selon un calendrier systématique.
