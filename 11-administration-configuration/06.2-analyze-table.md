# 11.6.2 — ANALYZE TABLE

`ANALYZE TABLE` collecte des statistiques sur la distribution des données — l'information dont l'optimiseur a besoin pour bâtir de bons plans d'exécution : ordre des jointures et choix des index. Contrairement à `OPTIMIZE`, qui reconstruit la table (§11.6.1), `ANALYZE` est une opération légère, fondée sur de l'échantillonnage, et **non bloquante** depuis MariaDB 10.6.16. À ne pas confondre avec l'instruction `ANALYZE` appliquée à une *requête* (de la famille de `EXPLAIN`), traitée en §5.7.

## Syntaxe et principe

```sql
ANALYZE [NO_WRITE_TO_BINLOG | LOCAL] TABLE nom_table [, nom_table ...]
  [PERSISTENT FOR
    { ALL | COLUMNS ([col ...]) INDEXES ([index ...]) }
  ]
```

L'instruction fonctionne sur les tables InnoDB, Aria et MyISAM, et requiert les privilèges `SELECT` et `INSERT`. Elle enregistre la distribution des clés, que MariaDB exploite pour décider de l'ordre dans lequel joindre les tables (lors d'une jointure sur autre chose qu'une constante) et des index à utiliser. Depuis la 10.6.16, l'opération est non intrusive : InnoDB autorise les lectures et écritures pendant l'analyse (lectures et insertions pour MyISAM/Aria), et une connexion utilise les nouvelles statistiques dès la requête suivante.

```sql
-- Recalcule les statistiques de la table
ANALYZE TABLE commandes;

-- Sans réplication (à lancer alors sur chaque serveur)
ANALYZE LOCAL TABLE commandes;
```

Comme les autres opérations de maintenance, `ANALYZE TABLE` est journalisé dans le binlog et répliqué par défaut (sauf `NO_WRITE_TO_BINLOG` / `LOCAL`, ou si `read_only` est actif). Elle s'applique aussi aux tables partitionnées via `ALTER TABLE … ANALYZE PARTITION`.

## Deux familles de statistiques

C'est le point conceptuel central : MariaDB dispose de deux sources de statistiques, complémentaires, que `ANALYZE TABLE` alimente.

| | Statistiques InnoDB | Statistiques EITS |
|--|---------------------|-------------------|
| Niveau | Moteur InnoDB | Serveur (tous moteurs) |
| Données | Limitées (échantillon de pages) | Riches (colonnes non indexées, histogrammes) |
| Mise à jour | Automatique (~10 % de changement) | Manuelle uniquement |
| Stockage | Interne à InnoDB | Tables `mysql.*_stats` (lisibles, modifiables) |
| Activées par défaut | Oui (`innodb_stats_persistent`) | *Utilisées* si présentes, mais *non collectées* |

### Les statistiques persistantes InnoDB

Au niveau du moteur, InnoDB tient des statistiques persistantes (variable `innodb_stats_persistent`, activée par défaut) qui survivent aux redémarrages. Elles sont **mises à jour automatiquement** lorsqu'environ 10 % des lignes d'une table ont changé (`innodb_stats_auto_recalc`), à partir d'un échantillon de pages d'index (`innodb_stats_persistent_sample_pages`) — d'où une précision limitée. On peut les régler table par table avec les options `STATS_PERSISTENT`, `STATS_AUTO_RECALC` et `STATS_SAMPLE_PAGES`. `ANALYZE TABLE` force leur recalcul. Le serveur s'appuyant sur elles par défaut, il dispose donc de statistiques même si `ANALYZE` n'est jamais lancé — ce qui suffit pour la majorité des requêtes.

### Les statistiques indépendantes du moteur (EITS)

Propres à MariaDB, les *Engine-Independent Table Statistics* sont stockées dans trois tables ordinaires de la base `mysql` — `table_stats`, `column_stats` et `index_stats` — que le DBA peut consulter, modifier, voire « figer ». Elles apportent bien plus de données que l'interface des moteurs ne le permet : en particulier, la distribution des valeurs de colonnes **non indexées**, et des **histogrammes**. Leur contrepartie : elles ne sont **jamais mises à jour automatiquement** et leur collecte prend du temps. On les recueille donc à la demande.

Les EITS ne sont collectées par `ANALYZE TABLE` que dans deux cas : si `use_stat_tables` vaut `COMPLEMENTARY` ou `PREFERABLY`, ou si l'instruction comporte la clause `PERSISTENT FOR`. Par défaut, `use_stat_tables` vaut `preferably_for_queries` : les EITS sont *utilisées* si elles existent, mais ne sont *pas collectées* automatiquement.

## Collecter les EITS : la clause PERSISTENT FOR

```sql
-- Toutes les colonnes et tous les index (peut être long sur une très grosse table)
ANALYZE TABLE commandes PERSISTENT FOR ALL;

-- Cibler des colonnes et index précis (recommandé)
ANALYZE TABLE commandes
  PERSISTENT FOR COLUMNS (statut, region) INDEXES (idx_client);
```

La collecte ciblée est préférable. Il est recommandé de la réserver aux colonnes **qui ne débutent pas un index** (si une colonne est en tête d'un index, l'index sert déjà de source de statistiques) et **qui apparaissent dans les conditions `WHERE`** des requêtes. Le ciblage permet aussi d'éviter les colonnes `BLOB`, pour lesquelles aucune statistique n'est collectée. Depuis les versions récentes (présentes en 12.x), les longues colonnes `CHAR`/`VARCHAR` dépassant `analyze_max_length` octets sont par ailleurs ignorées par `ANALYZE TABLE PERSISTENT`, sauf si on les nomme explicitement :

```sql
SET SESSION analyze_max_length = 10000;     -- ignorer les colonnes texte > 10 000 octets
ANALYZE TABLE produits PERSISTENT;
ANALYZE TABLE produits PERSISTENT FOR COLUMNS (description);  -- pour l'analyser malgré tout
```

Autre possibilité : faire collecter les EITS par un `ANALYZE TABLE` ordinaire en ajustant `use_stat_tables`.

```sql
SET SESSION use_stat_tables = 'COMPLEMENTARY';
ANALYZE TABLE commandes;
```

## Les histogrammes

Sous-ensemble des EITS (stockés dans `mysql.column_stats`), les histogrammes décrivent la distribution réelle des valeurs d'une colonne. Ils améliorent nettement les estimations de sélectivité de l'optimiseur, en particulier pour les colonnes non indexées et les conditions de plage. Leur collecte est commandée par `histogram_size` (la valeur `0` la désactive ; elle est non nulle par défaut sur les versions modernes), et leur format par `histogram_type` : height-balanced simple ou double précision, ou — depuis la 10.8 — au format JSON (`JSON_HB`), plus précis. La variable `optimizer_use_condition_selectivity` détermine jusqu'où l'optimiseur exploite ces statistiques. Le réglage fin des histogrammes relève de l'optimisation des requêtes (chapitre 15).

## Quand exécuter ANALYZE

Les statistiques de colonnes évoluant peu dans le temps, même quand la table grossit, il n'y a aucun intérêt à lancer `ANALYZE` trop souvent. On l'exécute plutôt :

- sur une table fraîchement peuplée ;
- après l'ajout de colonnes utilisées dans des clauses `WHERE` ;
- quand une table a doublé de taille ;
- après une variation importante de la distribution des données (chargement massif, suppression d'un grand volume) ;
- lorsqu'une requête ralentit parce que l'ordre des tables a changé, et qu'`EXPLAIN` ou `ANALYZE FORMAT=JSON` révèle une sélectivité mal estimée (§5.7).

À l'inverse, `ANALYZE` n'apporte rien sur des colonnes de type `UNIQUE`, `PRIMARY KEY`, `TIME` ou `CURRENT_TIME`, qu'il vaut mieux exclure.

## Impact, variables utiles et réplication

L'opération restant non bloquante depuis la 10.6.16, son impact en production est limité, mais elle peut consommer de l'espace disque sur de très grandes tables. Quelques variables l'encadrent : `analyze_sample_percentage` fixe le pourcentage de lignes échantillonnées (`0` laisse MariaDB décider), `analyze_max_length` exclut les longues colonnes texte, et `sort_buffer_size` (utilisé par colonne pour compter les doublons) accélère légèrement l'analyse s'il est augmenté. En réplication, on lance fréquemment `ANALYZE` serveur par serveur avec `LOCAL`. Enfin, un `ANALYZE` lent peut être tracé dans le journal des requêtes lentes si `admin` figure dans `log_slow_filter` (§11.4.2).
