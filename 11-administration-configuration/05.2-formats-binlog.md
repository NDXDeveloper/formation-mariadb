# 11.5.2 — Formats du journal binaire : STATEMENT, ROW, MIXED

Le journal binaire peut consigner les modifications sous trois formes, déterminées par la variable `binlog_format` : `STATEMENT`, `ROW` ou `MIXED` (la valeur par défaut). Ce choix n'est pas anodin : il influence la sûreté de la réplication, la taille des journaux et les performances côté réplica. Comme vu en §11.5.1, `binlog_format` est une variable dynamique, modifiable globalement ou par session.

Quel que soit le format, les événements sont toujours stockés sous forme binaire ; on les relit en clair avec l'utilitaire `mariadb-binlog`.

## Vue d'ensemble

| Critère | STATEMENT | ROW | MIXED (défaut) |
|---------|-----------|-----|----------------|
| Ce qui est journalisé | L'instruction SQL telle quelle | Chaque ligne modifiée (le DDL reste en instruction) | Instruction si elle est sûre, sinon lignes |
| Taille du journal | Compacte | Plus volumineuse | Intermédiaire |
| Sûreté de réplication | Risquée (non-déterminisme) | Maximale | Élevée |
| Lisibilité directe | Oui (SQL visible) | Non (sauf annotation) | Mixte |
| Usage typique | Cas contrôlés et déterministes | CDC, moteurs hétérogènes, correction critique | Recommandé par défaut |

## STATEMENT : journalisation par instruction

Dans ce format, les instructions sont écrites dans le journal exactement telles qu'elles ont été exécutées. Son atout est la compacité : une seule instruction est journalisée quel que soit le nombre de lignes affectées, ce qui le rend efficace lorsqu'une requête modifie beaucoup de lignes (le réplica télécharge peu de données).

Ce format n'est toutefois sûr que dans des conditions précises : des fonctions toutes déterministes (mêmes arguments, même résultat), des données identiques entre primaire et réplica, et l'emploi des mêmes moteurs de stockage pour toutes les tables. Les colonnes utilisant un horodatage ou `auto_increment` sont, elles, gérées correctement et restent sûres.

Lorsqu'une instruction est non déterministe, son résultat peut diverger entre serveurs. MariaDB émet alors un avertissement, par exemple pour une requête utilisant une clause `LIMIT` sans ordre garanti, ou des fonctions comme `UUID()` ou `RAND()`. La documentation dédiée aux instructions non sûres en réplication par instruction recense ces cas.

Certaines instructions ne peuvent pas être rejouées fidèlement à partir de leur seul texte SQL :

- une instruction s'appuyant sur une table temporaire créée pendant une phase en mode `ROW` ;
- un `ALTER TABLE` convertissant une table d'un moteur stockant les données à distance (par exemple le moteur S3) vers un autre moteur ;
- l'emploi de `SEQUENCE` dans l'instruction ou dans une définition `CREATE TABLE`.

En mode `MIXED` (le défaut), le serveur **bascule alors automatiquement en `ROW`** pour ces cas (voir plus bas). Mais en mode `STATEMENT` **pur**, cette bascule est impossible : selon le cas, l'instruction est journalisée telle quelle avec un avertissement « non sûr », ou — lorsque la journalisation par ligne est obligatoire, comme avec une `SEQUENCE` — carrément **rejetée** (vérifié sur 12.3.2) :

```
ERROR 1665 (HY000): Cannot execute statement: impossible to write to binary log
since BINLOG_FORMAT = STATEMENT and at least one table uses a storage engine
limited to row-based logging.
```

C'est l'une des raisons pour lesquelles `MIXED` est préférable à `STATEMENT` pur.

## ROW : journalisation par ligne

En mode `ROW`, les instructions de manipulation ne sont pas journalisées en tant que SQL : c'est chaque insertion, mise à jour ou suppression de ligne qui est enregistrée individuellement. Les instructions de définition (DDL) restent, elles, journalisées sous forme d'instructions.

C'est le format le plus sûr, puisqu'il enregistre le résultat exact des modifications, sans dépendre du déterminisme des requêtes — au prix d'un journal plus volumineux, en particulier pour les instructions touchant de nombreuses lignes. Pour retrouver malgré tout la requête d'origine dans la sortie de `mariadb-binlog`, MariaDB active par défaut les *annotated rows events* (`binlog_annotate_row_events`), qui joignent le texte SQL aux événements de ligne.

Le format `ROW` est par ailleurs **requis** par les outils de capture de changements (*CDC*) tels que Debezium, qui s'appuient sur les modifications ligne à ligne (§20.8).

Un comportement mérite l'attention : la suppression totale d'une table avec `DELETE FROM table` (sans `WHERE`). En mode `ROW`, **chaque ligne supprimée est journalisée individuellement**, ce qui peut être long à répliquer et consommer beaucoup d'espace dans le journal ; en mode `STATEMENT` ou `MIXED`, c'est l'**unique instruction `DELETE FROM table`** qui est journalisée, bien plus compacte. MariaDB ne convertit toutefois **pas** ce `DELETE` en `TRUNCATE` : l'auto-incrément n'est pas réinitialisé et l'exécution reste ligne à ligne (vérifié sur 12.3.2). Pour vider réellement une table de façon rapide et compacte, on emploie explicitement `TRUNCATE TABLE`, bien plus efficace qu'un `DELETE` sans `WHERE`.

### Régler le détail des événements ROW

Deux variables ajustent la verbosité des événements de ligne. `binlog_row_image` contrôle quelles colonnes sont journalisées : `FULL` (par défaut, toutes les colonnes dans l'image « avant » et « après »), `MINIMAL` (uniquement la clé d'identification dans l'image avant et les seules colonnes modifiées dans l'image après), `NOBLOB` (évite de journaliser les colonnes BLOB/TEXT non concernées) et, depuis MariaDB 11.4, `FULL_NODUP`. `binlog_row_metadata`, lui, détermine les métadonnées jointes : `NO_LOG` (par défaut), `MINIMAL` ou `FULL` (noms de colonnes et types, indispensables pour décoder le journal sans interroger le schéma vivant). Depuis MariaDB 12.3, les événements de ligne sont rattachés aux colonnes par leur nom, ce qui résout un problème connu de réplication multi-maître lorsque les définitions de tables diffèrent.

### Gros événements ROW : nouveauté 12.3

Un événement `ROW` plus gros que `max_allowed_packet` (dont la valeur maximale est de 1 Go) ne pouvait pas être écrit dans le journal binaire du primaire, ce qui interrompait la réplication avec une erreur fatale (`error 1236`). MariaDB 12.3 lève cette limite : en configurant `binlog_row_event_fragment_threshold` (typiquement à une valeur inférieure ou égale au `slave_max_allowed_packet` des réplicas), le primaire **fractionne** les transactions volumineuses en plusieurs morceaux plus petits (de type `Partial_rows_log_event`), que les réplicas peuvent lire sans difficulté.

## MIXED : le meilleur des deux mondes

Le format `MIXED`, choisi par défaut, combine les deux approches : la journalisation par instruction est utilisée tant qu'elle est sûre, et le serveur bascule en mode ligne dès qu'une instruction présente un risque pour la réplication. Au sein d'une même transaction, certaines instructions peuvent ainsi être journalisées en `STATEMENT` et d'autres en `ROW`.

Ce compromis explique pourquoi `MIXED` est recommandé par défaut : il reste compact comme `STATEMENT` pour les instructions sûres, tout en offrant une sûreté comparable à `ROW` pour les autres, avec une empreinte disque moindre qu'un mode `ROW` intégral.

## Comment choisir

Pour la grande majorité des déploiements, `MIXED` est le bon choix. On privilégiera le mode `ROW` intégral lorsque la correction prime absolument, lorsque les moteurs de stockage diffèrent entre serveurs, ou lorsqu'un outil de CDC l'exige. Le mode `STATEMENT` pur reste réservé à des environnements maîtrisés, déterministes et homogènes où la taille du journal est une contrainte forte. Sur le plan de l'efficacité, on gardera à l'esprit qu'une grosse instruction modifiant beaucoup de lignes se télécharge plus vite en `STATEMENT`, tandis qu'une instruction longue ne modifiant que quelques lignes s'applique plus vite en `ROW`.

## Changer de format

Le format se modifie dynamiquement (privilège `SUPER` requis) ou dans un fichier de configuration :

```sql
SET GLOBAL  binlog_format = 'ROW';   -- pour les nouvelles connexions
SET SESSION binlog_format = 'ROW';   -- pour la session courante
```

```ini
[mariadbd]
binlog_format = MIXED
```

> **Attention en réplication.** Changer le format ne concerne *que* le serveur sur lequel on agit : modifier `binlog_format` sur un primaire ne change pas le format du journal de ses réplicas, et une incohérence peut faire échouer la réplication. De plus, si le serveur est un réplica utilisant la réplication parallèle, un changement de la valeur globale n'affecte pas les threads en cours : il faut réinitialiser `slave_parallel_threads` pour qu'il soit pris en compte (`STOP REPLICA` ; mise à 0 ; changement de format ; remise à la valeur voulue ; `START REPLICA`).

## Comportements particuliers à connaître

Un réplica applique les événements reçus quel que soit leur format ; `binlog_format` ne régit que les mises à jour locales (non répliquées). Le format est par ailleurs ascendant-compatible : la réplication fonctionne tant que le réplica est de version égale ou supérieure à celle du primaire.

La base système `mysql` suit une logique propre. Les modifications **directes** (`INSERT`, `UPDATE`, `DELETE`, `REPLACE`, `TRUNCATE`…) sont journalisées selon `binlog_format`, mais les modifications **indirectes** des comptes et privilèges (`GRANT`, `REVOKE`, `SET PASSWORD`, `RENAME USER`, ainsi que `CREATE`, `ALTER`, `DROP`) le sont toujours en `STATEMENT`. Cas hybride : `CREATE TABLE ... SELECT` journalise la partie `CREATE TABLE` en instruction et la partie `SELECT` selon le format courant.

## Lire les événements avec `mariadb-binlog`

L'utilitaire `mariadb-binlog` présente chaque événement avec un en-tête riche : l'horodatage, le `server id` d'origine, la position de l'événement suivant (`end_log_pos`), le **GTID** (au format `Domaine-ServerID-Séquence`), l'identifiant de transaction `Xid` utilisé pour la validation en deux phases, et l'identifiant de *group commit* (`cid`) partagé par les transactions synchronisées ensemble. À noter : si le journal est chiffré (`encrypt_binlog`), `mariadb-binlog` ne peut pas le déchiffrer directement et doit se connecter au serveur via l'option `--read-from-remote-server`.
