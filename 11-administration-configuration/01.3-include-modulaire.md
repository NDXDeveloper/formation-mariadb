# 11.1.3 — Include et configuration modulaire

> Dernière sous-section de 11.1. On y voit comment éclater la configuration en plusieurs fichiers à l'aide des directives d'inclusion, et comment cette modularité s'inscrit dans l'ordre de lecture étudié en 11.1.2 — un mode d'organisation devenu la norme sur les distributions modernes et dans les déploiements conteneurisés.

---

## Du fichier unique à la configuration modulaire

Concentrer toute la configuration dans un seul `my.cnf` fonctionne pour une instance de test, mais devient vite ingérable en production : le fichier grossit, mélange des préoccupations sans rapport (réglages mémoire, réplication, journalisation…) et se prête mal au travail à plusieurs ou à l'automatisation. La réponse de MariaDB est la **configuration modulaire** : un fichier principal, court, qui délègue le reste à d'autres fichiers grâce à deux directives d'inclusion.

Ces directives se distinguent des options ordinaires par le préfixe `!` en début de ligne. Elles ne configurent pas le serveur directement : elles indiquent *où aller chercher d'autres options*.

---

## `!include` : inclure un fichier précis

La directive `!include` insère le contenu d'un fichier de configuration nommé, à l'endroit exact où elle figure :

```ini
[mariadbd]
datadir = /var/lib/mysql

!include /etc/mysql/tuning.cnf
```

C'est utile pour isoler un bloc de réglages bien identifié — par exemple un fichier de *tuning* mémoire partagé entre plusieurs serveurs — tout en gardant le contrôle précis du fichier importé.

---

## `!includedir` : inclure tout un répertoire

La directive `!includedir`, plus courante, importe **tous les fichiers de configuration d'un répertoire** :

```ini
!includedir /etc/mysql/mariadb.conf.d/
```

Son comportement obéit à trois règles qu'il faut connaître :

- **Seuls certains fichiers sont lus.** Sur les systèmes Unix et Linux, uniquement ceux portant l'extension `.cnf` ; sur Windows, ceux en `.cnf` ou `.ini`. Tout fichier d'une autre extension est ignoré.
- **Les fichiers sont lus dans l'ordre alphabétique** de leur nom. C'est pourquoi les distributions préfixent leurs fichiers par un numéro (`50-server.cnf`, `60-galera.cnf`…) : le préfixe fixe l'ordre de lecture de façon déterministe.
- **L'inclusion n'est pas récursive.** Seuls les fichiers situés directement dans le répertoire sont pris en compte ; les éventuels sous-répertoires ne sont pas explorés.

---

## Place dans l'ordre de lecture

L'inclusion n'introduit pas un ordre parallèle : les fichiers inclus s'insèrent dans la **séquence globale de lecture** (vue en 11.1.2) à la position où apparaît la directive. Un fichier inclus est donc lu *après* tout ce qui précède la directive dans le fichier porteur, et *avant* ce qui la suit.

La conséquence découle directement de la règle « le dernier l'emporte » : une option définie dans un fichier inclus surcharge la même option si elle figurait plus haut, avant la directive `!includedir`. C'est précisément ce qui rend les répertoires d'inclusion si pratiques pour **surcharger** une valeur par défaut. Et puisque les fichiers d'un répertoire sont lus par ordre alphabétique, un `99-overrides.cnf` aura naturellement le dernier mot sur un `50-server.cnf`.

---

## Les conventions des distributions

En pratique, vous rencontrerez rarement un gros `my.cnf` : les paquets MariaDB livrent déjà une arborescence modulaire. Deux familles dominent.

Sur les distributions de la famille **Debian/Ubuntu**, le fichier `/etc/mysql/my.cnf` ne fait quasiment que déléguer :

```ini
[client-server]
!includedir /etc/mysql/mariadb.conf.d/
!includedir /etc/mysql/conf.d/
```

La configuration serveur réelle se trouve alors dans `/etc/mysql/mariadb.conf.d/50-server.cnf`, la configuration du client dans `50-client.cnf`, et ainsi de suite. Dans l'image et les paquets 12.3, `conf.d/` est lu **après** `mariadb.conf.d/` : un *drop-in* placé dans `conf.d/` a donc le dernier mot, ce qui en fait l'emplacement naturel des surcharges (voir plus bas).

Sur les distributions de la famille **Red Hat/Fedora/Rocky**, c'est `/etc/my.cnf` qui pointe vers un répertoire dédié :

```ini
[mysqld]
datadir = /var/lib/mysql
socket  = /var/lib/mysql/mysql.sock

!includedir /etc/my.cnf.d
```

Les fichiers `server.cnf`, `mariadb-server.cnf` et `client.cnf` y prennent place.

Connaître ces conventions évite de chercher au mauvais endroit : sur un serveur inconnu, le bon réflexe est de partir du fichier global, de repérer ses directives `!includedir`, puis d'inspecter le répertoire pointé.

---

## Bonnes pratiques

La modularité n'a d'intérêt que si elle est exploitée avec méthode.

D'abord, **un fichier par préoccupation**. Plutôt que de tout entasser, on regroupe les réglages par thème : un fichier pour le dimensionnement mémoire, un pour la journalisation, un pour la réplication. La configuration devient lisible et chaque sujet se trouve immédiatement.

Ensuite, **maîtriser l'ordre avec des préfixes numériques**. Lorsque des surcharges sont en jeu, nommer les fichiers `00-base.cnf`, `50-server.cnf`, `99-local.cnf` rend l'ordre de lecture explicite et prévisible.

Surtout, **ne modifiez pas les fichiers gérés par le paquet** : une mise à jour de MariaDB risque de les écraser, emportant vos changements. La bonne approche est le *drop-in* : ajouter dans le répertoire d'inclusion un fichier qui vous appartient (par exemple `99-custom.cnf`), lu en dernier, et qui surcharge les valeurs par défaut sans toucher aux fichiers d'origine.

```ini
# /etc/mysql/mariadb.conf.d/99-custom.cnf
[mariadbd]
innodb_buffer_pool_size = 6G
max_connections         = 300
```

Enfin, **désactiver un fichier sans le supprimer** est trivial : puisque seuls les `.cnf` sont lus, il suffit de renommer un fichier (par exemple en `.cnf.disabled`) pour qu'il soit ignoré, ce qui permet de le réactiver plus tard.

---

## Modularité, conteneurs et IaC

Ce modèle de *drop-in* prend tout son sens dans les environnements modernes. L'image Docker officielle de MariaDB lit le répertoire `/etc/mysql/conf.d/` : on y monte un simple fichier `.cnf` pour personnaliser le serveur, sans reconstruire l'image.

```bash
docker run -d \
  -v "$(pwd)/my-custom.cnf:/etc/mysql/conf.d/my-custom.cnf:ro" \
  mariadb:12.3
```

Le même principe s'applique sous Kubernetes, où une `ConfigMap` est montée dans le répertoire d'inclusion, et avec les outils d'*Infrastructure as Code* (Ansible, Terraform), qui génèrent un fichier par rôle ou par préoccupation. La configuration redevient ainsi un artefact versionnable et reproductible — un thème développé au chapitre 16.

---

## En résumé

Les directives `!include` (un fichier) et `!includedir` (un répertoire) permettent d'éclater la configuration de MariaDB en fichiers thématiques. Les fichiers inclus s'insèrent dans l'ordre de lecture à la position de la directive et obéissent à la règle « le dernier l'emporte », ce qui en fait l'outil naturel de la surcharge. Sur les distributions réelles comme dans les conteneurs, le modèle dominant est le *drop-in* : un fichier personnel, lu en dernier, qui surcharge les défauts sans modifier les fichiers du paquet.

Cette section 11.1 a établi *où* et *comment* MariaDB lit sa configuration statique. La suite du chapitre s'intéresse à l'autre versant — la configuration **dynamique** du serveur en cours d'exécution — avec la section **11.2, Variables système et de session**.
