# 11.4.1 — Error log (journal d'erreurs)

Le journal d'erreurs est le plus important des journaux de MariaDB et, en pratique, le premier fichier à consulter face à un incident. Contrairement au journal général ou au journal des requêtes lentes, il est *toujours actif* : seule sa destination est configurable. Il consigne les erreurs critiques, les corruptions de tables ou de données, les informations de démarrage et d'arrêt du serveur, ainsi que des avertissements et des notes dont le niveau de détail se règle finement.

## Rôle et contenu

Le journal d'erreurs rassemble les événements significatifs de la vie du serveur : séquence de démarrage et d'arrêt, initialisation et récupération d'InnoDB, chargement des plugins, erreurs et avertissements de réplication, problèmes de corruption, et — en cas de plantage — la trace de la pile (*stack trace*) du processus. C'est notamment cette dernière qui rend le journal indispensable pour les analyses post-mortem et les rapports de bug.

Trois niveaux de gravité apparaissent dans les messages : `[Note]` pour les messages informatifs, `[Warning]` pour les avertissements et `[ERROR]` pour les erreurs proprement dites.

À noter : les erreurs SQL renvoyées *aux clients* (par opposition aux erreurs internes du serveur) ne figurent pas dans ce journal ; elles peuvent être tracées séparément via le plugin `SQL_ERROR_LOG`, qui est distinct du journal d'erreurs serveur.

## Où le journal d'erreurs est-il écrit ?

La destination dépend de la configuration et du mode de démarrage.

Le cas le plus courant en production est l'écriture dans un fichier, contrôlée par la variable `log_error`. Si aucun nom de fichier n'est précisé, MariaDB écrit dans `${hostname}.err` au sein du répertoire de données (`datadir`). Un nom relatif est interprété relativement au `datadir` ; un chemin absolu est utilisé tel quel :

```ini
[mariadbd]
# Chemin absolu (recommandé en production)
log_error = /var/log/mysql/error.log
```

L'option `log-basename` offre une alternative : elle définit un préfixe commun à tous les journaux (erreurs, général, lent, binaire), le journal d'erreurs prenant alors ce préfixe suivi de l'extension `.err`. Ce préfixe est toujours relatif au `datadir` et ne peut pas être un chemin absolu.

Sur les systèmes Unix, si `log_error` n'est pas défini, les messages partent sur la sortie d'erreur standard (`stderr`), c'est-à-dire généralement le terminal ayant démarré `mariadbd`. Une valeur déjà configurée peut être annulée avec `--skip-log-error`.

Lorsque le serveur est lancé via systemd — cas le plus fréquent sur les distributions modernes — une partie des messages peut être dirigée vers le journal système (`journald`) plutôt que vers un fichier ; on les consulte alors avec `journalctl`. La redirection vers syslog est également possible, que le serveur soit démarré par `mariadbd-safe` ou par systemd. Sur Windows, enfin, les messages sont écrits dans l'Observateur d'événements (source `MariaDB`), ou sur la console si l'option correspondante est active.

> En production, il est fortement recommandé de définir explicitement `log_error` vers un fichier dédié : c'est plus simple à archiver, à analyser et à surveiller que des messages dispersés dans le journal système.

## Localiser le fichier de log

Lorsqu'on ignore où MariaDB écrit ses erreurs, plusieurs méthodes existent. Depuis une session SQL, la variable donne directement le chemin :

```sql
SHOW VARIABLES LIKE 'log_error';
```

Depuis le shell, on peut inspecter les options effectivement appliquées :

```bash
mariadbd --print-defaults

# Depuis MariaDB 10.11
my_print_defaults --mariadbd | grep log-error
```

Si ces commandes ne renvoient rien d'exploitable, c'est probablement que les messages partent vers le journal système ; il faut alors les chercher avec `journalctl -u mariadb`.

## Régler la verbosité : `log_warnings`

Le volume d'avertissements et de notes consignés se règle avec la variable `log_warnings`, dont la valeur par défaut est **2**. Elle est dynamique (modifiable à chaud) :

```sql
SET GLOBAL log_warnings = 3;
```

ou statique, dans un fichier de configuration :

```ini
[mariadbd]
log_warnings = 2
```

Chaque niveau ajoute des catégories de messages aux précédents :

| Niveau | Messages additionnels typiques |
|--------|--------------------------------|
| 0 | Désactive de nombreux avertissements optionnels (les avertissements « cœur » restent toutefois écrits). |
| 1 | Messages de réplication, échecs de résolution DNS, planificateur d'événements, instructions non sûres pour la réplication par instruction (SBR). |
| 2 *(défaut)* | Ajoute les erreurs d'accès refusé, les connexions interrompues (erreurs/timeouts), les erreurs du *handler* de table (clé dupliquée, *lock wait timeout*…), les messages liés aux fichiers d'état de réplication. |
| 3 | Ajoute la progression des opérations *online DDL* d'InnoDB et les avertissements sur d'anciennes options de langue. |
| 4 | Ajoute les connexions tuées, fermées ou libérées (`KILLED`, `CLOSE_CONNECTION`, `RELEASE`). |
| 9 | Très verbeux : trace notamment l'initialisation des plugins. |

Deux points méritent l'attention. D'abord, `log_warnings` n'agit que sur une partie des messages : beaucoup d'avertissements d'InnoDB, par exemple, sont écrits quelle que soit la verbosité. Ensuite, pour éviter qu'un flot d'avertissements identiques sur les instructions SBR ne fasse exploser la taille du fichier, MariaDB suspend automatiquement un même avertissement pendant cinq minutes après dix occurrences identiques.

> MariaDB ne reconnaît pas la variable `log_error_verbosity` introduite par MySQL 5.7 : c'est bien `log_warnings` qu'il faut employer. Ce point est à connaître lors d'une migration depuis MySQL (voir chapitre 19).

## Format d'une entrée

Sur les versions modernes, chaque entrée occupe une ligne et suit le gabarit suivant :

```
YYYY-MM-DD HH:MM:SS  ThreadID  [Niveau]  Message
```

Par exemple :

```
2026-05-28 14:02:13 0 [Note] InnoDB: Buffer pool(s) load completed
2026-05-28 15:17:44 87 [Warning] Aborted connection 87 to db: 'shop' user: 'app' host: '10.0.0.12' (Got timeout reading communication packets)
```

Le premier champ est l'horodatage (`YYYY-MM-DD HH:MM:SS`). Le deuxième est l'identifiant du thread, c'est-à-dire l'ID de la connexion interne ; les messages d'origine système, comme ceux du démarrage, portent l'identifiant `0`. Le troisième est le niveau de gravité entre crochets (`[Note]`, `[Warning]` ou `[Error]`). Le quatrième est le texte du message.

La toute première ligne écrite après un démarrage est particulière : elle indique la version exacte de MariaDB, la révision du code source, un identifiant unique de serveur (`server_uid`, présent dans les séries 11.x et 12.x) et le numéro de processus :

```
2026-05-28 14:02:11 0 [Note] Starting MariaDB 12.3.2-MariaDB source revision … server_uid … as process 1842
```

### Cas particuliers de format

Certaines lignes ne respectent pas ce gabarit, ce qu'il faut garder à l'esprit si l'on analyse les journaux automatiquement. Lors d'un transfert d'état (SST) dans un cluster Galera, les messages des scripts externes (`rsync`, `mariabackup`) sont injectés tels quels et n'ont ni identifiant de thread ni préfixe de niveau (voir §14.2.4). Après un plantage, MariaDB écrit une trace de pile multi-lignes — une ligne d'en-tête (du type `handle_fatal_signal (sig=11)`) suivie d'adresses hexadécimales et de noms de fonctions — précieuse pour un rapport de bug. Enfin, en mode syslog, le préfixe propre à MariaDB est précédé de l'en-tête du système (date, hôte, identifiant, PID).

## Consulter et surveiller le journal

Au quotidien, on suit le journal en temps réel avec les outils habituels du système :

```bash
# Fichier dédié
tail -f /var/log/mysql/error.log

# Journal systemd
journalctl -u mariadb -f
```

Pour un diagnostic, filtrer sur le niveau de gravité fait gagner du temps :

```bash
grep -E '\[ERROR\]|\[Warning\]' /var/log/mysql/error.log
```

En exploitation, la surveillance du journal d'erreurs doit être automatisée : on configure des alertes sur l'apparition d'entrées `[ERROR]` et on centralise les journaux des différents serveurs (chapitre 16).

## Rotation du journal

Contrairement au journal général et au journal des requêtes lentes, le journal d'erreurs ne peut être écrit que dans un fichier (ou vers `stderr`/syslog) — jamais dans une table. Sa rotation s'appuie, sous Linux, sur l'utilitaire `logrotate`, pour lequel MariaDB fournit une configuration par défaut. Après qu'un script de rotation a renommé le fichier courant, le serveur doit rouvrir un nouveau fichier ; c'est le rôle de la commande :

```sql
FLUSH ERROR LOGS;
```

(`FLUSH LOGS` rouvre l'ensemble des journaux ; `mariadb-admin flush-logs` produit le même effet depuis le shell.)

## Messages d'erreur localisés

Les libellés de nombreuses erreurs sont lus dans un fichier de messages localisés (`errmsg.sys`). Si le serveur ne le trouve pas au démarrage, il le signale par une erreur du type `Can't find messagefile …`. On corrige l'emplacement avec `lc_messages_dir`, et l'on choisit la langue des messages avec `lc_messages` :

```ini
[mariadbd]
lc_messages_dir = /usr/share/mysql/
lc_messages     = fr_FR
```

## En résumé et pour la suite

Le journal d'erreurs est toujours actif et constitue le point de départ de tout diagnostic. En production, on prendra soin de le diriger vers un fichier explicite (`log_error`), de conserver la verbosité par défaut (`log_warnings = 2`) quitte à l'augmenter ponctuellement le temps d'une investigation, et de surveiller l'apparition d'entrées `[ERROR]`. Les deux journaux suivants complètent la panoplie : le journal des requêtes lentes (§11.4.2) pour l'optimisation, et le journal général (§11.4.3) pour le débogage exhaustif. Les erreurs spécifiques à la réplication sont par ailleurs traitées au chapitre 13, et la centralisation des journaux au chapitre 16.
