# 11.4.2 — Slow query log (journal des requêtes lentes)

Le journal des requêtes lentes est le principal point d'entrée du travail d'optimisation : il enregistre les requêtes SQL dont l'exécution a dépassé un seuil de durée — et, en option, celles qui n'utilisent aucun index. Pour chaque requête, il consigne notamment le temps d'exécution, le temps passé à attendre des verrous et le nombre de lignes parcourues, autant d'indices pour repérer ce qu'il faut retravailler (voir §15.7 pour l'analyse approfondie).

## Rôle et principe

Désactivé par défaut, ce journal cible les requêtes coûteuses plutôt que l'ensemble du trafic. Une requête y est écrite si sa durée dépasse `long_query_time`, et l'on peut y ajouter d'autres critères : requêtes sans index, requêtes provoquant un tri sur disque ou une table temporaire, etc. Le détail enregistré (plan d'exécution, sortie `EXPLAIN`, statistiques du moteur) est lui-même paramétrable.

> **Sécurité.** Comme le journal général, le journal des requêtes lentes peut contenir des données sensibles, voire des mots de passe apparaissant dans le texte des requêtes. MariaDB chiffre nativement les journaux binaires et les fichiers de données, mais **pas** les journaux *slow* et *general*. Pour les protéger, on combine l'écriture en table avec le chiffrement des données au repos (chapitre 18), ou l'on s'appuie sur un chiffrement de système de fichiers au niveau de l'OS — en complément de permissions restrictives.

## Activer le journal

L'activation se fait à chaud avec `SET GLOBAL`, ou de façon permanente dans un fichier de configuration :

```sql
SET GLOBAL slow_query_log = ON;
```

```ini
[mariadbd]
slow_query_log
slow_query_log_file = /var/log/mysql/mariadb-slow.log
long_query_time     = 1
```

On vérifie l'état et le fichier utilisé avec :

```sql
SHOW GLOBAL VARIABLES LIKE 'slow_query_log%';
```

N'importe quel utilisateur peut par ailleurs désactiver la journalisation pour sa seule connexion :

```sql
SET SESSION slow_query_log = 0;
```

> **Renommage en 10.11.** À partir de MariaDB 10.11, les variables du journal *slow* ont reçu des noms harmonisés en `log_slow_*`, les anciens noms restant valides comme synonymes : `slow_query_log` → `log_slow_query`, `slow_query_log_file` → `log_slow_query_file`, `long_query_time` → `log_slow_query_time`, `min_examined_row_limit` → `log_slow_min_examined_row_limit`. Les exemples ci-dessous utilisent les noms classiques, toujours opérationnels en 12.3.

## Le seuil : `long_query_time`

La durée à partir de laquelle une requête est jugée lente est fixée par `long_query_time`, exprimée en secondes. Sa **valeur par défaut est 10**, ce qui est bien trop élevé pour un usage réel : on l'abaisse généralement à 1 seconde, voire moins.

```sql
SET GLOBAL long_query_time = 0.5;   -- 500 ms
```

Le format du seuil dépend de la destination : en mode FICHIER, on peut donner une valeur fractionnaire avec une résolution à la microseconde (par exemple `0.1` pour 100 ms) ; en mode TABLE, seule une valeur entière en secondes est prise en compte. La variable est aussi modifiable par session, ce qui permet d'instrumenter finement un traitement précis.

## Destination : fichier ou table

Comme pour le journal général, la destination est commandée par `log_output` : `FILE` (par défaut) écrit dans le fichier `slow_query_log_file` (défaut `${hostname}-slow.log` dans le `datadir`) ; `TABLE` écrit dans la table `mysql.slow_log`, interrogeable en SQL.

```sql
SELECT * FROM mysql.slow_log\G
```

Cette table expose des colonnes telles que `start_time`, `user_host`, `query_time`, `lock_time`, `rows_sent`, `rows_examined`, `rows_affected`, `db`, `thread_id` et `sql_text`.

> **Limitation importante.** Les statistiques étendues — `log_slow_verbosity` et la sortie `EXPLAIN` — ne sont **pas** disponibles lorsque `log_output = TABLE`. Pour disposer du plan d'exécution dans le journal, il faut impérativement écrire dans un **fichier**.

## Format d'une entrée (mode fichier)

En mode fichier, chaque enregistrement est multi-lignes. Il débute par une ligne `# User@Host:`, suivie de lignes d'en-tête préfixées par `#`, puis du texte de la requête (qui peut lui-même s'étendre sur plusieurs lignes). L'enregistrement se termine au `# User@Host:` suivant ou à la fin du fichier.

```
# Time: 260206 18:38:23
# User@Host: app[app] @  [10.0.0.12]
# Thread_id: 312  Schema: shop  QC_hit: No
# Query_time: 4.512383  Lock_time: 0.000128  Rows_sent: 25  Rows_examined: 1842301
# Rows_affected: 0
SET timestamp=1770399503;
SELECT * FROM orders WHERE customer_id = 90431 ORDER BY created_at DESC;
```

Les champs les plus utiles sont `Query_time` (durée totale), `Lock_time` (temps d'attente des verrous), `Rows_sent` (lignes renvoyées) et `Rows_examined` (lignes parcourues). Un écart important entre `Rows_examined` et `Rows_sent` est un signal classique de requête à optimiser, souvent par un index.

## Cibler les requêtes : `log_slow_filter`

Par défaut, tous les types de requêtes sont enregistrés (instructions d'administration comprises). La variable `log_slow_filter` permet de restreindre la journalisation à certaines catégories, séparées par des virgules ; une valeur vide équivaut à tout journaliser.

| Valeur | Requêtes journalisées |
|--------|-----------------------|
| `admin` | Instructions d'administration (voir ci-dessous) |
| `full_scan` | Parcours complet de table |
| `full_join` | Jointure sans index sur les tables jointes |
| `filesort` | Tri nécessitant un *filesort* |
| `filesort_on_disk` | *Filesort* débordant sur le disque |
| `filesort_priority_queue` | *Filesort* utilisant une file de priorité |
| `tmp_table` | Création d'une table temporaire |
| `tmp_table_on_disk` | Table temporaire matérialisée sur disque |
| `not_using_index` | Requêtes sans index (ignore `long_query_time`) |
| `query_cache` / `query_cache_miss` | Selon l'usage du cache de requêtes |
| `all` | Active toutes les options |

```sql
SET GLOBAL log_slow_filter = 'filesort,filesort_on_disk,tmp_table,tmp_table_on_disk';
```

Les instructions considérées comme administratives sont `ALTER TABLE`, `ANALYZE TABLE`, `CHECK TABLE`, `CREATE INDEX`, `DROP INDEX`, `OPTIMIZE TABLE`, `REPAIR TABLE` et `ALTER SEQUENCE`. Elles sont journalisées par défaut ; pour les exclure, on retire `admin` de `log_slow_filter`. (La variable dédiée `log_slow_admin_statements` existe encore mais est dépréciée depuis MariaDB 11.0 au profit de cette approche.)

## Enrichir les entrées : `log_slow_verbosity`

Au-delà des en-têtes standard, `log_slow_verbosity` ajoute des informations précieuses pour le diagnostic. Plusieurs valeurs se combinent par des virgules ; le raccourci historique `full` les active toutes.

| Valeur | Information ajoutée |
|--------|---------------------|
| `query_plan` | Caractéristiques du plan pour les `SELECT` (*Full_scan*, *Full_join*, *Tmp_table*, *Filesort*, passes de fusion…) |
| `explain` | Sortie `EXPLAIN` complète de la requête (fichier uniquement) |
| `engine` | Statistiques fournies par le moteur de stockage |
| `innodb` | Conservé pour compatibilité ; équivalent à `engine` (effectif depuis 10.6.15) |
| `warnings` | Notes et avertissements liés à la requête (`log_slow_max_warnings` en limite le nombre, 10 par défaut) |

```sql
SET GLOBAL log_slow_verbosity = 'query_plan,explain,engine';
```

Lorsque `explain` est actif, chaque requête est accompagnée de son plan, avec notamment les colonnes `r_rows` et `r_filtered` qui donnent les valeurs *réelles* observées (comme pour `ANALYZE`) :

```
# explain: id select_type table type possible_keys key key_len ref rows r_rows filtered r_filtered Extra
# explain: 1 SIMPLE orders ALL idx_customer NULL NULL NULL 1850000 1842301.00 100.00 0.05 Using where; Using filesort
```

Rappel : ces statistiques étendues exigent `log_output = FILE`.

## Maîtriser le volume

Sur un serveur très chargé, un seuil bas peut produire un volume d'écritures conséquent. Trois leviers permettent de le contenir.

L'échantillonnage, via `log_slow_rate_limit`, ne journalise qu'une requête lente sur *N* :

```sql
SET GLOBAL log_slow_rate_limit = 5;   -- une requête lente sur cinq
```

L'exclusion des requêtes triviales, via `min_examined_row_limit`, ignore celles qui parcourent moins d'un certain nombre de lignes — utile en complément de `log_queries_not_using_indexes`, qui journalise les requêtes sans index quelle que soit leur durée :

```sql
SET GLOBAL min_examined_row_limit = 100000;
SET GLOBAL log_queries_not_using_indexes = ON;
```

Enfin, `log_slow_disabled_statements` (réglable au démarrage uniquement) exclut certains types d'instructions, par exemple `'admin,call,slave,sp'`.

## Analyser le journal

Un fichier *slow* se lit dans n'importe quel éditeur de texte, mais l'outil `mariadb-dumpslow` en facilite l'exploitation en regroupant et résumant les requêtes similaires. Pour une analyse plus poussée — agrégation, classement par impact cumulé, normalisation des requêtes — l'outil de référence est `pt-query-digest` de la *Percona Toolkit* (voir §15.7.2). Lorsque le journal est écrit en table, une simple requête sur `mysql.slow_log` suffit. Les requêtes ainsi identifiées sont les candidates prioritaires à l'optimisation, par réécriture ou par un meilleur usage des index (chapitres 5 et 15).

## Rotation

Comme le journal général, le journal des requêtes lentes peut être écrit en table ou en fichier ; en fichier, sa rotation s'appuie sur `logrotate` sous Linux. Après renommage du fichier courant, on demande au serveur d'en ouvrir un nouveau avec :

```sql
FLUSH SLOW LOGS;
```

(`FLUSH LOGS` rouvre l'ensemble des journaux.)

## En résumé et pour la suite

Le journal des requêtes lentes est l'outil central de l'optimisation : on l'active avec un seuil réaliste (`long_query_time` de l'ordre de 1 s, abaissé ponctuellement), de préférence en mode **fichier** pour bénéficier du plan d'exécution via `log_slow_verbosity`, et l'on cible le bruit avec `log_slow_filter`, l'échantillonnage et `min_examined_row_limit`. Le journal suivant, le journal général (§11.4.3), adopte une logique inverse : il enregistre *toutes* les requêtes, ce qui en fait un outil de débogage exhaustif mais coûteux. L'exploitation analytique des requêtes lentes est approfondie au chapitre 15.
