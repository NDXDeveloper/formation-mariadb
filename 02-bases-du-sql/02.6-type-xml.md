# 2.2.6 Type XML basique 🆕

> **Chapitre 2 : Bases du SQL — §2.2 Types de données** · Niveau : Débutant  
> Version de référence : **MariaDB 12.3 LTS**  
> 🆕 Nouveauté de la série 12.x

Historiquement, MariaDB n'offrait aucun type dédié au XML : on stockait les documents dans une colonne texte (`TEXT` ou `LONGTEXT`) que l'on manipulait à l'aide de quelques fonctions XPath. La 12.3 comble en partie ce manque en introduisant un type spécialisé, **`XMLTYPE`**. Il s'agit d'un type de données introduit dans MariaDB 12.3 pour stocker des données XML, conçu pour en faciliter le stockage, assurer la compatibilité avec Oracle et préparer de futures capacités de validation et de traitement XML. Comme son nom l'indique, ce premier jet reste **basique**.

## Le type `XMLTYPE`

La déclaration est minimaliste : on écrit simplement `XMLTYPE`, **sans paramètre**. Toute tentative de préciser une longueur, comme `XMLTYPE(100)`, provoque une erreur.

```sql
CREATE TABLE document (
    id       INT,
    contenu  XMLTYPE        -- aucune longueur : XMLTYPE(100) déclencherait une erreur
);

INSERT INTO document VALUES (1, '<livre><titre>SQL</titre></livre>');
```

En interne, le type repose sur un stockage de chaîne : sa capacité maximale est de 4 Go, comme celle de `LONGBLOB`. Il possède d'ailleurs un **jeu de caractères** et une collation, paramétrables comme pour une colonne textuelle, et accepte l'attribut `BINARY` pour une collation binaire :

```sql
CREATE TABLE t (x XMLTYPE CHARACTER SET utf8mb4);   -- jeu de caractères explicite
CREATE TABLE t (x XMLTYPE BINARY);                  -- collation binaire
```

## Une implémentation volontairement minimale

Le qualificatif « basique » n'est pas anodin. Dans cette première version, `XMLTYPE` n'assure qu'un stockage XML élémentaire, sans validation ni fonctionnalité XML spécialisée ; lorsqu'on lui applique des fonctions de chaîne, il est de fait converti en chaîne de caractères et conservé comme tel dans les tables temporaires.

Concrètement, cela signifie deux choses. D'une part, **rien ne garantit que le contenu inséré soit du XML bien formé** : dans l'exemple de la documentation, des valeurs comme `'one'` sont acceptées sans broncher. La validation et le respect d'un schéma ne sont pas pris en charge. D'autre part, le type se comporte essentiellement comme une chaîne pour la plupart des opérations. L'intérêt par rapport à un simple `LONGTEXT` est donc surtout **sémantique** à ce stade — déclarer explicitement qu'une colonne contient du XML — la documentation laissant entrevoir des capacités de traitement plus riches dans le futur.

## Compatibilité Oracle

Le nom `XMLTYPE` n'est pas choisi au hasard : c'est celui du type XML d'Oracle. Le type est explicitement conçu pour être compatible avec le `XMLTYPE` d'Oracle. Cette orientation s'inscrit dans l'effort plus large de compatibilité Oracle qui caractérise la série 12.x (fonctions `TO_DATE`/`TO_CHAR`, tableaux associatifs, `SYS_REFCURSOR`…), détaillé au chapitre 19. La compatibilité reste toutefois partielle : l'attribut `REF_SYSTEM_ID`, présent chez Oracle, n'est pas pris en charge.

## Manipuler du XML

Indépendamment du type `XMLTYPE`, MariaDB fournit depuis longtemps des fonctions XPath pour extraire et modifier des fragments XML, principalement `ExtractValue()` (extraction d'une valeur via une expression XPath) et `UpdateXML()` (remplacement d'un fragment). Elles s'appliquent aussi bien à du XML stocké en texte qu'au nouveau type. À noter que la 12.3 fait évoluer leur intégration : les fonctions `UPDATEXML` et `CAST` renvoient désormais le type `XMLTYPE`. Pour l'import de fichiers XML entiers, l'instruction `LOAD XML INFILE` permet par ailleurs de mapper éléments et attributs vers les colonnes d'une table.

## `XMLTYPE` ou `JSON` ?

Pour des données semi-structurées, le **JSON** est aujourd'hui bien plus abouti dans MariaDB : type validé, large bibliothèque de fonctions, opérateurs raccourcis et indexation via colonnes virtuelles (chapitre 4). Le support du XML, lui, en est à ses débuts. En pratique, on réservera donc `XMLTYPE` aux situations où le XML s'impose : **migration depuis Oracle**, interopérabilité avec des systèmes produisant du XML, ou volonté de typer explicitement une colonne de documents XML. Pour un nouveau projet libre de ses choix, le JSON reste la voie recommandée.

## En résumé

`XMLTYPE`, introduit en 12.3, apporte enfin à MariaDB un type dédié au XML — mais dans une version délibérément **basique** : déclaration sans paramètre, stockage de type chaîne jusqu'à 4 Go, **ni validation ni traitement XML** spécialisé pour l'instant. Sa principale raison d'être actuelle est la **compatibilité avec Oracle** et la préparation de fonctionnalités futures. Le XML se manipule toujours via les fonctions XPath historiques (`ExtractValue`, `UpdateXML`), et pour des données semi-structurées sur un nouveau projet, le `JSON` demeure nettement plus mûr.
