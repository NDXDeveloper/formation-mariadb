# 2.2.1 Types numériques

> **Chapitre 2 : Bases du SQL — §2.2 Types de données** · Niveau : Débutant  
> Version de référence : **MariaDB 12.3 LTS**

MariaDB distingue trois grandes catégories de types numériques : les **entiers**, les **décimaux exacts** (`DECIMAL`) et les **nombres à virgule flottante** (`FLOAT`, `DOUBLE`). La différence entre ces deux dernières — exact contre approché — est sans doute le point le plus important de cette section : la confondre est à l'origine de bugs financiers classiques.

## Les types entiers

Les types entiers se distinguent uniquement par l'espace qu'ils occupent, et donc par l'amplitude des valeurs qu'ils peuvent représenter.

| Type | Octets | Plage signée | Plage non signée (`UNSIGNED`) |
|---|---|---|---|
| `TINYINT` | 1 | −128 à 127 | 0 à 255 |
| `SMALLINT` | 2 | −32 768 à 32 767 | 0 à 65 535 |
| `MEDIUMINT` | 3 | −8 388 608 à 8 388 607 | 0 à 16 777 215 |
| `INT` (`INTEGER`) | 4 | −2 147 483 648 à 2 147 483 647 | 0 à 4 294 967 295 |
| `BIGINT` | 8 | ≈ −9,22×10¹⁸ à 9,22×10¹⁸ | 0 à ≈ 1,84×10¹⁹ |

Le principe de base est de **choisir le type le plus compact** couvrant l'amplitude réellement attendue : inutile de réserver un `BIGINT` (8 octets) pour un âge ou une note sur 20, qu'un `TINYINT` suffit à représenter. Sur une table de plusieurs millions de lignes, ces économies se cumulent et allègent à la fois le stockage et les index.

### L'attribut `UNSIGNED`

L'attribut `UNSIGNED` interdit les valeurs négatives et décale la plage vers le haut, doublant ainsi la valeur positive maximale. Il convient naturellement aux grandeurs qui ne peuvent pas être négatives (quantités, compteurs, identifiants) :

```sql
CREATE TABLE compteur (
    visites INT UNSIGNED NOT NULL DEFAULT 0   -- 0 à ~4,29 milliards
);
```

Attention toutefois à l'arithmétique : une soustraction entre valeurs `UNSIGNED` qui passerait sous zéro peut produire une erreur ou un résultat inattendu selon le mode SQL. On réserve donc `UNSIGNED` aux colonnes dont on est sûr qu'elles ne descendront jamais en dessous de zéro.

### La largeur d'affichage : une fausse contrainte

Vous rencontrerez parfois une syntaxe du type `INT(11)`. Le nombre entre parenthèses est une simple **largeur d'affichage** — une indication du nombre de caractères que le client est invité à réserver, surtout en combinaison avec `ZEROFILL`. Il **ne limite ni la plage des valeurs ni l'espace de stockage** : `INT(4)` accepte parfaitement la valeur `1000000`. Cette notation est aujourd'hui considérée comme obsolète ; il est recommandé de déclarer simplement `INT` sans largeur.

### `AUTO_INCREMENT`

L'attribut `AUTO_INCREMENT`, posé sur une colonne entière (généralement la clé primaire), fait générer automatiquement une valeur croissante à chaque insertion. Le choix entre `INT UNSIGNED` (jusqu'à ~4,29 milliards de lignes) et `BIGINT UNSIGNED` conditionne la capacité maximale de la table : pour un identifiant technique susceptible de croître fortement, `BIGINT UNSIGNED` est le choix prudent.

```sql
CREATE TABLE client (
    id   BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    nom  VARCHAR(100) NOT NULL
);
```

### Dépassement de plage

Par défaut, MariaDB fonctionne en **mode SQL strict** (voir §11.3) : tenter d'insérer une valeur hors plage déclenche une erreur et la ligne est rejetée. En mode non strict, la valeur est au contraire ramenée à la borne la plus proche et un simple avertissement est émis. Connaître le mode actif évite de mauvaises surprises lors d'imports de données.

## Les nombres décimaux exacts : `DECIMAL`

Le type `DECIMAL(M, D)` (synonyme : `NUMERIC`) stocke des nombres décimaux de manière **exacte**, sans erreur d'arrondi. Le paramètre `M` est la **précision** — le nombre total de chiffres significatifs — et `D` l'**échelle** — le nombre de chiffres après la virgule.

```sql
prix DECIMAL(10,2)   -- jusqu'à 99 999 999,99
taux DECIMAL(5,4)    -- ex. 0,1975 pour un taux de 19,75 %
```

La précision maximale est de 65 chiffres (`M`), avec au plus 30 chiffres après la virgule (`D`), et `D` ne peut excéder `M`. En l'absence de paramètres, `DECIMAL` équivaut à `DECIMAL(10,0)`. Le stockage est compact (de l'ordre de 4 octets par tranche de 9 chiffres).

`DECIMAL` est le type de prédilection pour tout ce qui exige une valeur exacte : **montants financiers**, prix, taux, soldes comptables. C'est aussi le type des littéraux décimaux écrits sans notation scientifique (`19.99`), ce qui garantit l'exactitude des calculs courants.

## Les nombres à virgule flottante : `FLOAT` et `DOUBLE`

Les types flottants stockent les nombres de manière **approchée**, au format binaire IEEE 754 :

- `FLOAT` — 4 octets, simple précision, environ 7 chiffres significatifs ;
- `DOUBLE` (synonymes `DOUBLE PRECISION`, `REAL`) — 8 octets, double précision, environ 15 à 16 chiffres significatifs.

Le mot « approché » est ici déterminant. Certaines valeurs décimales simples n'ont pas de représentation binaire exacte, ce qui introduit de minuscules écarts d'arrondi. On peut l'observer en comparant un calcul mené sur des décimaux (exacts) et sur des flottants (notation scientifique avec `E`, donc en `DOUBLE`) :

```sql
SELECT 0.1 + 0.2;        -- 0.3                  (littéraux DECIMAL : exact)
SELECT 0.1e0 + 0.2e0;    -- ≈ 0.30000000000000004 (DOUBLE : approché)
```

Ces écarts sont négligeables pour des **mesures scientifiques ou physiques** (températures, coordonnées, résultats de capteurs), où une tolérance d'arrondi est acceptable. En revanche, ils sont rédhibitoires dès qu'une valeur doit être exacte : **on n'utilise jamais `FLOAT` ou `DOUBLE` pour un montant financier**, sous peine de voir des totaux « faux » d'un centime.

À noter : la syntaxe `FLOAT(M,D)` / `DOUBLE(M,D)` existe encore par compatibilité, mais elle est obsolète et déconseillée.

## `DECIMAL` ou `FLOAT`/`DOUBLE` : que choisir ?

La règle se résume en une phrase. Pour une valeur qui doit être **exacte** (argent, comptabilité, quantités précises), on choisit `DECIMAL`. Pour une valeur **approchée** issue d'une mesure, où la performance et l'amplitude priment sur le dernier chiffre, on choisit `DOUBLE` (ou `FLOAT` si l'on cherche à économiser l'espace au prix d'une précision moindre).

## Types et raccourcis annexes

Quelques types numériques complètent le tableau :

- **`BIT(M)`** — champ de bits de 1 à 64 bits, utile pour stocker des drapeaux ou des masques de manière compacte.
- **`BOOLEAN`** (ou `BOOL`) — synonyme de `TINYINT(1)`. La valeur `0` est interprétée comme *faux*, toute valeur non nulle comme *vrai* ; les littéraux `TRUE` et `FALSE` valent respectivement `1` et `0`. MariaDB ne possède pas de véritable type booléen distinct.
- **`SERIAL`** — non pas un type à part entière, mais un raccourci équivalant à `BIGINT UNSIGNED NOT NULL AUTO_INCREMENT UNIQUE`, pratique pour déclarer rapidement une clé technique.

## En résumé

On retient quelques réflexes : choisir le **type entier le plus compact** suffisant pour les valeurs attendues, et `BIGINT UNSIGNED` pour les clés techniques des tables à fort volume ; ajouter `UNSIGNED` lorsque les valeurs négatives sont impossibles ; réserver `DECIMAL` aux valeurs **exactes** (montants en tête) et `FLOAT`/`DOUBLE` aux mesures **approchées** ; et se souvenir que la largeur d'affichage `INT(M)` est purement cosmétique, sans effet sur les valeurs stockées.
