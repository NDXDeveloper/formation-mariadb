# 2.2.5 Types spécifiques MariaDB

> **Chapitre 2 : Bases du SQL — §2.2 Types de données** · Niveau : Débutant  
> Version de référence : **MariaDB 12.3 LTS**

Au-delà des types standard du SQL, MariaDB propose plusieurs types répondant à des besoins modernes : `JSON` pour les documents semi-structurés, `UUID` pour les identifiants universels, `INET6` (et `INET4`) pour les adresses réseau, et `VECTOR` pour les *embeddings* de l'intelligence artificielle. Chacun apporte un stockage et une validation adaptés là où l'on se contenterait sinon, à grand-peine, d'une chaîne de caractères. Cette section les présente dans leurs grandes lignes ; plusieurs sont approfondis dans des chapitres dédiés.

## `JSON`

Le type `JSON` stocke des documents au format JSON — objets, tableaux, valeurs imbriquées. Particularité importante de MariaDB : `JSON` n'est pas un type binaire dédié, mais un **alias de `LONGTEXT`** auquel s'ajoute automatiquement une contrainte de validation (`JSON_VALID`) garantissant que le contenu est un JSON correct. Les données sont donc stockées sous forme de texte.

```sql
CREATE TABLE produit (
    id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    nom        VARCHAR(100) NOT NULL,
    attributs  JSON          -- ex. {"couleur": "rouge", "tailles": [38, 40, 42]}
);
```

Ce choix d'implémentation diffère de MySQL, qui dispose d'un type JSON binaire au stockage optimisé — un point à connaître lors d'une migration (§19.1). En contrepartie, MariaDB offre un riche écosystème de **fonctions JSON** (`JSON_EXTRACT`, `JSON_VALUE`, `JSON_SET`, `JSON_CONTAINS`…) et l'opérateur raccourci `->>`, ainsi que l'indexation via des **colonnes virtuelles** extraites du document. Tout cela est traité en détail au chapitre 4 (§4.7 à §4.10). La série 12.x y ajoute des apports conformes au standard SQL, comme le prédicat `IS JSON` et la levée de la limite de profondeur (§4.7.4).

## `UUID`

Un UUID (*Universally Unique IDentifier*) est un identifiant de 128 bits, statistiquement unique sans coordination centrale — utile pour des clés générées côté application ou réparties sur plusieurs nœuds. MariaDB dispose d'un type `UUID` **natif**, qui stocke la valeur de façon compacte sur **16 octets** tout en l'affichant sous sa forme canonique de 36 caractères (`6ccd780c-baba-1026-9564-5b8c656024db`).

```sql
CREATE TABLE session (
    id          UUID NOT NULL PRIMARY KEY,
    ouverte_le  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO session (id) VALUES (UUID());   -- UUID() génère la valeur
```

Ce type natif est préférable au stockage manuel : il est plus compact qu'un `CHAR(36)` (16 octets contre 36) et plus lisible qu'un `BINARY(16)`, tout en assurant la validation et l'affichage normalisé. La fonction `UUID()` permet d'en générer une valeur.

Un atout moins visible mérite d'être souligné : le type `UUID` range ses valeurs dans un **ordre favorable à l'indexation**. Pour les UUID de version 1 (ceux que produit `UUID()`, fondés sur l'horloge), MariaDB réagence les octets afin de les stocker dans un ordre proche de l'ordre chronologique. Cela atténue le piège classique de l'UUID employé comme clé primaire sous InnoDB — où des valeurs *aléatoires* éparpillent les insertions dans tout l'index *clustered* et le fragmentent : ici, les nouvelles lignes restent regroupées et l'index demeure plus compact et mieux mis en cache. Revers de la médaille : un `ORDER BY` sur une colonne `UUID` suit cet **ordre de stockage, et non l'ordre lexical** de la forme affichée — à savoir si l'on attend un tri alphabétique. (Les UUID de version ≥ 6, tel l'UUIDv7 déjà conçu pour être ordonné dans le temps, sont stockés sans ce réagencement.)

## `INET6` et `INET4`

Pour stocker des adresses IP, MariaDB propose deux types dédiés : `INET6` pour les adresses **IPv6** (16 octets) et `INET4` pour les adresses **IPv4** (4 octets). `INET6` sait également représenter une adresse IPv4 sous forme mappée, ce qui en fait un choix polyvalent.

```sql
CREATE TABLE connexion (
    id       BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    adresse  INET6 NOT NULL
);

INSERT INTO connexion (adresse) VALUES ('2001:db8::1'), ('192.0.2.10');
```

Comparés au stockage d'une adresse dans un `VARCHAR`, ces types apportent un **stockage compact**, une **validation** à l'insertion, une **forme canonique** normalisée et une comparaison correcte (utile pour trier ou filtrer des plages d'adresses). Les fonctions de conversion historiques (`INET6_ATON`/`INET6_NTOA`, `INET_ATON`/`INET_NTOA`) restent disponibles pour interopérer avec des représentations numériques.

## `VECTOR`

Le type `VECTOR(N)` stocke un **vecteur de `N` nombres** (un *embedding*) : la représentation numérique d'un texte, d'une image ou de tout autre contenu produite par un modèle d'IA. Disponible depuis la 11.8, il est au cœur des usages de **recherche sémantique** et de **RAG** (*Retrieval-Augmented Generation*).

```sql
CREATE TABLE document (
    id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    contenu    TEXT,
    embedding  VECTOR(1536) NOT NULL,   -- ex. un embedding produit par un LLM
    VECTOR INDEX (embedding)
);

INSERT INTO document (contenu, embedding)
VALUES ('…', VEC_FromText('[0.12, -0.03, 0.88, …]'));
```

Le type s'accompagne d'un **index HNSW** pour la recherche des plus proches voisins (§5.3), de fonctions de distance comme `VEC_DISTANCE_EUCLIDEAN` et `VEC_DISTANCE_COSINE`, et de fonctions de conversion `VEC_FromText` / `VEC_ToText`. La 12.3 y apporte des optimisations supplémentaires — calcul de distance par extrapolation et exécution au niveau du moteur de stockage (§15.16, §18.10.7). L'ensemble du sujet, jusqu'à l'intégration avec les LLM, est développé en section 18.10 et au chapitre 20.

## Récapitulatif

| Type | Rôle | Stockage | Approfondi en |
|---|---|---|---|
| `JSON` | Documents semi-structurés | alias de `LONGTEXT` (texte validé) | chapitre 4 |
| `UUID` | Identifiant unique 128 bits | 16 octets, affiché en texte canonique | — |
| `INET6` / `INET4` | Adresses IP (v6 / v4) | 16 / 4 octets | — |
| `VECTOR(N)` | *Embeddings* pour l'IA | binaire, `N` valeurs | §18.10, chapitre 20 |

## En résumé

Ces types spécialisés évitent de détourner un `VARCHAR` pour des données qui méritent mieux : `JSON` (alias de `LONGTEXT` validé) pour des documents souples, accompagné d'un large jeu de fonctions détaillé au chapitre 4 ; `UUID` pour des identifiants 128 bits stockés sur 16 octets et affichés lisiblement ; `INET6`/`INET4` pour des adresses IP compactes, validées et normalisées ; et `VECTOR` pour les *embeddings* d'IA, pierre angulaire de la recherche vectorielle approfondie en section 18.10. Chacun apporte stockage adapté, validation et fonctions dédiées là où une simple chaîne serait fragile et coûteuse.
