# 14.2.1 — Architecture synchrone multi-master

> **Chapitre 14 — Haute Disponibilité** · Section 14.2.1  
> Version de référence : **MariaDB 12.3 LTS**

La section 14.2 a présenté Galera de l'extérieur. Cette sous-section **ouvre le capot** : comment fonctionne concrètement une architecture *synchrone multi-maître*, depuis la pile logicielle jusqu'au cheminement précis d'une transaction d'écriture. C'est ce mécanisme qui permet à Galera d'offrir une cohérence forte (RPO ≈ 0) tout en conservant de bonnes performances. La **détection des conflits** proprement dite est traitée en 14.2.2 ; la **configuration** en 14.2.3 ; les **transferts d'état** en 14.2.4.

---

## 1. Le modèle multi-master : des nœuds tous égaux

Dans une architecture **multi-maître**, il n'existe **aucun nœud privilégié** : chaque nœud est à la fois une **copie complète** des données et un **point d'écriture** à part entière. Galera est donc une architecture **symétrique et sans maître désigné** (*masterless*).

Conséquences directes :

- un client peut se connecter à **n'importe quel nœud** pour lire **et** écrire ;
- il n'y a **pas de promotion** à orchestrer en cas de panne : si un nœud disparaît, les autres continuent sans changement de rôle (*failover* implicite) ;
- les **lectures se répartissent** naturellement sur l'ensemble des nœuds ;
- la coordination ne repose pas sur un coordinateur central, mais sur un **protocole de communication de groupe** qui établit un ordre commun (§3).

> ℹ️ « Multi-maître » ne signifie pas qu'il faut écrire partout en même temps. En pratique, router les écritures d'une même donnée vers un seul nœud (via un proxy, voir 14.4) réduit fortement les conflits de certification — tout en gardant la capacité de basculer instantanément.

---

## 2. La pile de réplication wsrep

Galera s'insère dans MariaDB via une pile de couches bien séparées :

```
┌────────────────────────────────────────────────┐
│           Application / clients SQL            │
└─────────────────────────┬──────────────────────┘
                          │ SQL
┌─────────────────────────▼──────────────────────┐
│               Serveur MariaDB                  │
│         (analyse SQL · moteur InnoDB)          │
└─────────────────────────┬──────────────────────┘
                          │ API wsrep
┌─────────────────────────▼──────────────────────┐
│      Provider Galera (libgalera_smm.so)        │
│   certification · ordre global · GCache        │
└─────────────────────────┬──────────────────────┘
                          │
┌─────────────────────────▼──────────────────────┐
│      GCS — Group Communication System          │
│     diffusion en ordre total (gcomm://)        │
└─────────────────────────┬──────────────────────┘
                          │ réseau
                    (les autres nœuds)
```

| Couche | Rôle |
|--------|------|
| **Serveur MariaDB** | Exécute le SQL, gère les verrous locaux et le moteur InnoDB. |
| **API wsrep** | Interface standardisée par laquelle MariaDB délègue la réplication au provider. |
| **Provider Galera** | La bibliothèque (`libgalera_smm.so`) qui réalise la certification, attribue l'ordre global et gère le GCache (voir 14.2.4). |
| **GCS** | La couche de messagerie qui assure une **diffusion en ordre total** des write-sets entre tous les nœuds. |

## 3. Cycle de vie d'une transaction d'écriture

C'est le cœur de l'architecture. Une écriture traverse une **phase locale** puis une **phase de réplication** au moment du `COMMIT`.

```
Nœud A (origine)                       Nœuds B et C
─────────────────                      ──────────────
1. Exécution locale
   (verrous InnoDB locaux,
    write-set en construction)
        │
2. COMMIT demandé
        │
3. Génération du write-set
   (lignes modifiées + clés)
        │
4. Diffusion en ORDRE TOTAL ─────────► réception, même seqno global
        │                                     │
5. Certification (déterministe) ◄─ même décision ─► Certification
        │  (succès)                            │  (succès)
6. Commit local                          (mise en file d'application)
        │                                     │
7. Retour « OK » au client          8. Application asynchrone
                                        du write-set, puis commit
```

Étape par étape :

1. **Phase locale.** La transaction s'exécute uniquement sur le nœud d'origine, comme sur un serveur autonome : elle acquiert ses **verrous InnoDB locaux** et construit progressivement son **write-set** (l'ensemble de ses modifications).
2. **Au COMMIT**, la phase de réplication démarre.
3. **Génération du write-set** : les lignes modifiées et les **clés** servant à la certification sont rassemblées.
4. **Diffusion en ordre total** : le write-set est envoyé à tous les nœuds via le GCS, qui lui attribue un **numéro de séquence global** unique. Galera identifie chaque transaction par un couple `<uuid_du_cluster>:<seqno>` (visible via `wsrep_last_committed`).
5. **Certification** : sur **chaque** nœud, un contrôle de conflit **déterministe** est appliqué à ce flux totalement ordonné. Comme tous les nœuds traitent les write-sets dans le **même ordre** avec la **même logique**, ils aboutissent **tous à la même décision** (le mécanisme exact est détaillé en **14.2.2**).
6. **Si la certification réussit**, le nœud d'origine **valide localement**…
7. … et **rend la main au client immédiatement**.
8. Sur les nœuds distants, le write-set certifié est **mis en file** puis **appliqué de façon asynchrone**, avec la garantie que l'issue est déjà identique partout.

Si la certification **échoue**, la transaction est **annulée** et le client reçoit une erreur de type *deadlock*, qu'il devra **rejouer** (la robustesse de ce mécanisme côté *applier* est abordée en 14.2.6).

---

## 4. « Synchrone » vs « virtuellement synchrone » : la distinction précise

Galera est qualifié de **virtuellement synchrone** parce que ses deux phases ont un comportement différent :

- **La partie synchrone** = l'**ordonnancement global** et la **certification**, qui ont lieu **avant** que le `COMMIT` ne rende la main au client. C'est ce qui garantit qu'aucune transaction validée ne pourra entrer en conflit indétecté.
- **La partie asynchrone** = l'**application physique** du write-set sur les nœuds distants, **découplée** du retour au client.

Ce découplage est précisément ce qui distingue Galera d'une réplication réellement synchrone (de type *two-phase commit*) :

| Modèle | Ce qu'on attend au COMMIT | Cohérence | Performance |
|--------|---------------------------|-----------|-------------|
| **Synchrone strict (2PC)** | L'application **physique** sur tous les nœuds | Forte | Faible (coûteux, verrous distribués) |
| **Virtuellement synchrone (Galera)** | Seulement l'**ordonnancement + la certification** | Forte (garantie déterministe) | Bonne |
| **Asynchrone (réplication classique)** | Rien (propagation différée) | Éventuelle | Élevée |

Galera obtient ainsi la **cohérence d'un système synchrone** avec une part importante des **performances d'un système asynchrone**.

---

## 5. Concurrence optimiste et BF-abort

Galera applique une **concurrence optimiste à l'échelle du cluster** : une transaction s'exécute **sans coordination inter-nœuds** pendant toute sa phase locale ; les conflits ne sont résolus **qu'au COMMIT**, par la certification. Localement, en revanche, le verrouillage **pessimiste** habituel d'InnoDB reste en vigueur.

Lorsqu'un write-set répliqué (en cours d'application) entre en conflit avec une **transaction locale encore ouverte**, c'est la **réplication qui est prioritaire** : la transaction locale est **annulée de force** — c'est le ***BF-abort*** (*Brute Force abort*). Le principe général est que **la transaction ordonnée la première l'emporte** ; l'autre devra être rejouée. Le détail des règles de conflit relève de 14.2.2, et le réessai automatique côté *applier* de 14.2.6.

---

## 6. Application parallèle des write-sets

Sur chaque nœud, les write-sets ne sont pas forcément appliqués un par un : des **threads d'application** (paramètre `wsrep_slave_threads`) peuvent les traiter **en parallèle** lorsqu'ils portent sur des données **indépendantes**. Cela améliore le débit d'application et limite le retard d'un nœud.

> Ce parallélisme s'appuie sur l'identification des lignes par leur **clé primaire**. C'est l'une des raisons pour lesquelles Galera **exige une `PRIMARY KEY` sur chaque table** (rappelé en 14.2) : sans elle, l'application déterministe et parallèle des write-sets n'est plus garantie.

---

## 7. Flow control : empêcher la divergence

Un cluster « virtuellement synchrone » ne tolère pas qu'un nœud prenne un retard arbitraire. Le **flow control** est le mécanisme qui l'en empêche :

- chaque nœud surveille sa **file de réception** (write-sets reçus mais pas encore appliqués, visible via `wsrep_local_recv_queue`) ;
- si cette file dépasse un seuil (`gcs.fc_limit`), le nœud émet un message de **pause** qui **ralentit l'ensemble du cluster** ;
- lorsque le nœud a suffisamment rattrapé son retard, il signale la **reprise**.

**Conséquence architecturale majeure :** un cluster Galera avance au **rythme de son nœud le plus lent**. Un nœud sous-dimensionné, surchargé ou sur un réseau lent peut donc bridér tout le cluster. Le temps passé en pause se mesure via `wsrep_flow_control_paused` (proportion du temps où le nœud a été bloqué) — une métrique clé de surveillance.

---

## 8. États d'un nœud dans le cluster

Un nœud passe par plusieurs **états** au cours de sa vie, observables via `wsrep_local_state_comment` :

| État (`wsrep_local_state_comment`) | Signification |
|------|---------------|
| **Initialized** | Le nœud est démarré mais **ne fait pas (ou plus) partie d'une composante opérationnelle** — état initial, ou après une **perte de quorum** (voir 14.3). |
| **Joining** (*Joiner*) | Le nœud **reçoit un transfert d'état** pour se synchroniser (SST/IST, voir 14.2.4). |
| **Joined** | Le transfert est terminé ; le nœud **rattrape** les write-sets accumulés entre-temps. |
| **Synced** | Le nœud est **pleinement opérationnel** : il sert lectures et écritures et participe à la certification. |
| **Donor / Desynced** | Le nœud **fournit** un transfert d'état à un *joiner* (il peut être temporairement indisponible selon la méthode SST). |

> ℹ️ Ne pas confondre l'**état local** ci-dessus (`wsrep_local_state_comment`, valeur numérique correspondante dans `wsrep_local_state`) avec l'**appartenance à la composante primaire**, exposée **séparément** par `wsrep_cluster_status` (`Primary` ou `non-Primary`, voir 14.3). Un nœud peut être `Synced` **et** `wsrep_cluster_status = Primary` ; après une perte de quorum, il repasse `Initialized` côté état local et `non-Primary` côté cluster.

Seuls les nœuds **Synced** appartenant à la **composante primaire** (`wsrep_cluster_status = Primary`) traitent les transactions normalement.

---

## 9. Lectures multi-maître et cohérence causale

Les **lectures sont servies localement** par le nœud auquel le client est connecté. En multi-maître, il existe donc une fenêtre où un write-set vient d'être **certifié** mais **pas encore appliqué** sur un nœud donné : une lecture sur **ce** nœud pourrait ne pas refléter une écriture tout juste validée **ailleurs**.

Pour garantir une **cohérence causale** (lire ses propres écritures, ou les écritures déjà validées dans le cluster), Galera propose le paramètre **`wsrep_sync_wait`** (qui remplace l'ancien `wsrep_causal_reads`) : le nœud **attend** alors d'avoir appliqué tous les write-sets en attente **avant** de répondre. Le compromis est une **latence de lecture** légèrement accrue, à n'activer que pour les requêtes qui l'exigent.

---

## 10. Implications architecturales (synthèse des compromis)

Ce fonctionnement explique directement les forces et limites vues en 14.2 :

| Propriété de l'architecture | Conséquence pratique |
|-----------------------------|----------------------|
| Diffusion en ordre total à chaque COMMIT | Le **débit en écriture** est borné par la latence réseau (RTT) et le nœud le plus lent. |
| Tous les nœuds appliquent toutes les écritures | **Écrire** plus de nœuds n'augmente **pas** la capacité d'écriture (mais celle de lecture, oui). |
| Aucun nœud privilégié | **Pas de SPOF** pour les écritures ; bascule implicite. |
| Concurrence optimiste + certification | Les **points chauds** (mêmes lignes modifiées simultanément) génèrent des **conflits** à rejouer. |
| Write-set diffusé en entier | Les **grosses transactions** pèsent sur tout le cluster. |
| Composante primaire requise | En cas de partition, la minorité **cesse de servir** — Galera privilégie la **cohérence (C)** du théorème CAP (voir 14.1 et 14.3). |

---

## À retenir

- L'architecture **multi-maître** de Galera est **symétrique et sans maître** : tout nœud lit et écrit, le *failover* est implicite, les lectures se répartissent.
- La réplication s'appuie sur une **pile wsrep** (serveur → API wsrep → provider Galera → GCS) et une **diffusion en ordre total** des write-sets.
- Une écriture s'exécute **localement** (concurrence optimiste), puis, au `COMMIT`, son write-set est **diffusé en ordre global**, **certifié de façon déterministe** sur tous les nœuds, validé localement et **appliqué en différé** ailleurs.
- « **Virtuellement synchrone** » = l'**ordonnancement et la certification** sont synchrones (avant le retour au client), l'**application distante** est asynchrone — d'où cohérence forte **et** bonnes performances.
- Le **flow control** empêche la divergence au prix d'un cluster qui avance au **rythme du nœud le plus lent**.
- L'application parallèle (`wsrep_slave_threads`) et la **clé primaire obligatoire** vont de pair ; la cohérence causale en lecture s'obtient via `wsrep_sync_wait`.

La sous-section suivante détaille le mécanisme qui rend tout cela cohérent : la **certification**.
