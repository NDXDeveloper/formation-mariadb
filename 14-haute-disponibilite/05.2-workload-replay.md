# 14.5.2 — Workload Replay

> **Chapitre 14 — Haute Disponibilité** · Section 14.5.2  
> Version de référence : **MariaDB 12.3 LTS** · MaxScale **25.01**

Deuxième outil du module **WCAR**, le **rejeu de charge** (*Workload Replay*) **reproduit le trafic capturé** (voir 14.5.1) contre un système cible, afin de **comparer son comportement et ses performances** — typiquement une version actuelle face à une version cible, avant une montée de version. Cette sous-section détaille le flux complet, de la restauration de la base jusqu'à la **visualisation** comparée des résultats, au moyen de la chaîne d'outils `maxplayer`. C'est une approche **hors ligne**, à distinguer de la comparaison **en direct** du Diff Router (14.5.3).

---

## 1. Le principe : rejouer le trafic capturé

Le rejeu prend les **fichiers de capture** produits en 14.5.1 et les **réexécute** contre un serveur cible, à partir d'une base **restaurée dans l'état de départ** de la capture. On obtient ainsi une exécution **réaliste et reproductible**, que l'on peut lancer **deux fois** — contre une **référence** (*baseline*) puis contre une **nouvelle configuration** — pour **comparer**.

---

## 2. Le flux complet

Le rejeu s'inscrit dans une séquence en six temps :

```
1. Restaurer la base de rejeu au point de départ  (sauvegarde de 14.5.1)
2. (Transformer les fichiers de capture — automatique au 1er rejeu)
3. Rejouer contre la version « baseline »   →  baseline-result.csv
4. Rejouer contre la version cible          →  comparison-result.csv
5. Post-traiter (canonicals + maxpostprocess) →  *-summary.json
6. Visualiser (maxvisualize)                →  comparaison dans le navigateur
```

---

## 3. Prérequis et installation

Le rejeu s'installe sur le serveur MaxScale qui l'exécutera. Une dépendance à installer manuellement est **Python 3.9 ou plus récent** ; sur la plupart des distributions, l'interpréteur Python par défaut convient. Il faut généralement aussi les **paquets de développement Python** (`python3-devel` sur les systèmes RHEL, `python3-dev` sur Debian) ainsi qu'un **compilateur C++**.

Pour RHEL 8, Rocky Linux 8 et Alma Linux 8, on installe une version plus récente de Python et ses en-têtes (`dnf install python39 python39-devel gcc-c++`) et on la définit comme implémentation Python par défaut via `alternatives`.

---

## 4. Préparer la base de rejeu

Le rejeu **restaure la base au point dans le temps où la capture a démarré**. On commence donc par restaurer le serveur de rejeu dans cet état, puis on y copie les fichiers de capture.

La **position GTID du premier *commit*** de la capture est visible dans la sortie de la commande de résumé :

```bash
maxplayer summary /chemin/vers/capture.cx
```

> Si les données capturées n'ont pas encore été transformées au format de rejeu, la commande `summary` réalise d'abord la transformation avant d'afficher le résumé.

Le rejeu requiert par ailleurs un **compte utilisateur** ayant accès à toutes les tables concernées. Pour des tests, le plus simple est :

```sql
CREATE USER 'maxreplay'@'%' IDENTIFIED BY 'replay-pw';
GRANT ALL ON *.* TO 'maxreplay'@'%';
```

**Variantes de restauration.** Pour un rejeu en **lecture seule**, la position GTID exacte importe moins — il peut même être avantageux que les serveurs soient dans l'état d'**après** la capture. Le rejeu prend aussi en charge le mode **écriture seule** : effectuer la **restauration complète** puis un rejeu **write-only** prépare le serveur à exécuter **plusieurs rejeux read-only** successifs, sans nouvelle synchronisation — utile pour **affiner les réglages** du serveur.

---

## 5. La transformation des fichiers

Lors du **premier rejeu**, les fichiers de capture sont **transformés sur place** (l'étape peut aussi être lancée séparément). Selon la taille et la structure de la capture, la **transformation peut utiliser jusqu'à deux fois l'espace** du fichier `capture.ex` — rappel : les fichiers `.ex` contiennent l'essentiel des événements (voir 14.5.1). Il faut donc prévoir l'**espace disque** correspondant.

---

## 6. Rejouer la charge : référence puis comparaison

On rejoue d'abord contre l'installation **de référence**, puis contre la **nouvelle** installation à comparer :

```bash
# Rejeu contre la version de référence (baseline)
maxplayer replay --user maxreplay --password replay-pw \
  --host <hôte:port> --output baseline-result.csv /chemin/vers/capture.cx

# Rejeu contre la nouvelle version (comparaison)
maxplayer replay --user maxreplay --password replay-pw \
  --host <hôte:port> --output comparison-result.csv /chemin/vers/capture.cx
```

Chaque rejeu produit un fichier de résultats (`.csv`) qui servira au post-traitement.

---

## 7. Options clés de `maxplayer`

La commande `maxplayer` propose plusieurs commandes (`summary`, `replay` — par défaut —, `convert`, `canonicals`, `dump-data`, `show`) et de nombreuses options pour le rejeu :

| Option | Rôle |
|--------|------|
| `--host` | Adresse du serveur de rejeu (`IP:port`). |
| `--user` | Utilisateur de rejeu (cette version **ne prend pas en charge** les noms d'utilisateurs d'origine de la capture). |
| `--password` | Mot de passe (**seuls les mots de passe en clair** sont pris en charge pour l'instant). |
| `--output` | Fichier de résultats (ex. `baseline.csv`). |
| `--speed` | Multiplicateur de vitesse (`2.5` = ×2,5, `0.5` = moitié, `0` = aussi vite que possible). |
| `--idle-wait` | Gère les **longues périodes d'inactivité** de la capture : permet d'avancer le temps de simulation pour **sauter les « trous »** (défaut : 1 s). |
| `--query-filter` | `none` / `write-only` / `read-only` : rejouer **seulement** les écritures ou les lectures. |
| `--analyze` | Suit le `Rows_read` du serveur par requête (**ralentit** le rejeu). |
| `--report` | Rapporte périodiquement des statistiques (prend une durée). |

> Les notes « noms d'utilisateurs d'origine non pris en charge » et « mots de passe en clair seulement » reflètent l'état de l'outil à cette version ; on vérifiera la documentation pour les évolutions.

---

## 8. Post-traitement et visualisation

Les résultats bruts doivent être **post-traités en résumés** avant d'être visualisés. On génère d'abord le fichier des **formes canoniques**, nécessaire au post-traitement :

```bash
maxplayer canonicals /chemin/vers/capture.cx > canonicals.csv
```

On post-traite ensuite les résultats de référence et de comparaison avec `maxpostprocess` :

```bash
maxpostprocess canonicals.csv baseline-result.csv   -o baseline-summary.json
maxpostprocess canonicals.csv comparison-result.csv -o comparison-summary.json
```

Enfin, on **visualise** la comparaison avec `maxvisualize`, qui **ouvre un navigateur** (URL par défaut `http://localhost:8866/`) :

```bash
maxvisualize baseline-summary.json comparison-summary.json
```

Pour écouter sur toutes les interfaces réseau, on ajoute `--Voila.ip='0.0.0.0'`.

---

## 9. Limites

Plusieurs comportements sont à connaître lors du rejeu :

- les commandes **`KILL`** ne fonctionnent pas correctement et peuvent **tuer la mauvaise session** (MXS-5056) ;
- un **`COM_STMT_EXECUTE` utilisant un curseur** est rejoué **sans curseur** (MXS-5059) ;
- l'exécution d'un **`COM_STMT_SEND_LONG_DATA`** ne fonctionne pas (MXS-5060) ;
- pour les tables **MyISAM/Aria**, le verrou de table est tenu moins longtemps.

En outre, le mode `read-only`/`write-only` a ses limites : les **résultats des requêtes** peuvent différer sensiblement de ceux observés pendant la capture.

---

## 10. Cas d'usage : valider une montée de version

Le scénario phare réunit tout ce qui précède :

1. on **capture** le trafic de production (14.5.1) avec sa sauvegarde de départ ;
2. on **restaure** ce point de départ sur un serveur de test ;
3. on **rejoue** la charge contre la version **actuelle** (*baseline*) — par exemple 11.8 ;
4. on **rejoue** la même charge contre la version **cible** — par exemple 12.3 (voir 19.10) ;
5. on **compare** via `maxpostprocess`/`maxvisualize`.

On obtient ainsi une **évaluation chiffrée et réaliste** de l'impact de la migration **avant** de l'appliquer en production. Le mode **write-only puis read-only répété** permet en outre de **fignoler les réglages** du serveur cible sur la même base de départ.

---

## À retenir

- Le **rejeu** réexécute le **trafic capturé** (14.5.1) contre un serveur cible, à partir d'une base **restaurée à l'état de départ** — approche **hors ligne** (vs Diff Router, 14.5.3).
- Le flux : **restaurer** la base → **transformer** (auto au 1er rejeu, jusqu'à ~2× l'espace du `.ex`) → **rejouer** *baseline* puis *comparaison* → **post-traiter** → **visualiser**.
- Prérequis : **Python 3.9+**, paquets de développement et **compilateur C++** sur le serveur de rejeu ; un **compte de rejeu** avec accès aux tables ; la **position GTID** s'obtient via `maxplayer summary`.
- La chaîne `maxplayer replay` → `maxplayer canonicals` → `maxpostprocess` → `maxvisualize` produit une **comparaison visuelle** (navigateur, `localhost:8866`).
- Options clés : `--host`, `--user`, `--speed`, `--idle-wait` (sauter les temps morts), `--query-filter` (`read-only`/`write-only`).
- ⚠️ Limites : `KILL`, curseurs et `COM_STMT_SEND_LONG_DATA` mal gérés ; en read/write-only, les résultats peuvent différer de la capture.
- Cas d'usage majeur : **valider une montée de version** (ex. 11.8 → 12.3, voir 19.10) avec une **charge réaliste**, avant la production.
