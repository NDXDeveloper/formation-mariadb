# 14.2.3 — Configuration et déploiement

> **Chapitre 14 — Haute Disponibilité** · Section 14.2.3  
> Version de référence : **MariaDB 12.3 LTS**

Après l'architecture (14.2.1) et la certification (14.2.2), place à la mise en œuvre concrète : **comment configurer et déployer un cluster Galera**. Cette sous-section détaille les prérequis, les ports réseau, les variables `wsrep_*` essentielles, l'amorçage du premier nœud (*bootstrap*) et la vérification de l'état du cluster. Les **mécanismes de transfert d'état** (SST/IST) sont détaillés en 14.2.4, et les spécificités du **packaging 12.3** en 14.2.5.

---

## 1. Prérequis de déploiement

| Prérequis | Recommandation |
|-----------|----------------|
| **Nombre de nœuds** | **Impair**, **3 au minimum**, pour préserver le quorum (voir 14.3). |
| **Version MariaDB** | **Identique sur tous les nœuds** (un cluster ne mélange pas les versions majeures). |
| **Domaines de défaillance** | Répartir les nœuds sur des racks / zones distincts (voir 14.1). |
| **Paquet Galera** 🆕 | Installer le paquet **`mariadb-server-galera`**, désormais séparé en 12.3 (voir 14.2.5). |
| **Réseau** | Faible latence entre nœuds (Galera est sensible au RTT). |

> Le paquet `mariadb-server-galera` fournit la bibliothèque Galera (`libgalera_smm.so`) et les scripts associés. Son emplacement exact dépend de la distribution ; on s'appuie sur le chemin fourni par le paquet plutôt que sur une valeur codée en dur.

---

## 2. Les ports réseau de Galera

Un cluster Galera utilise quatre ports, qui doivent être **ouverts entre tous les nœuds** (et seulement entre eux) :

| Port | Usage |
|------|-------|
| **3306** /tcp | Connexions clientes SQL (comme un serveur autonome). |
| **4567** /tcp + udp | Trafic de **réplication** et communication de groupe (GCS). |
| **4568** /tcp | **IST** — *Incremental State Transfer* (voir 14.2.4). |
| **4444** /tcp | **SST** — *State Snapshot Transfer* (voir 14.2.4). |

> ⚠️ Un pare-feu (ou un *security group* cloud) bloquant l'un de ces ports est la cause **n°1** d'un cluster qui refuse de se former ou d'un nœud qui ne parvient pas à rejoindre.

---

## 3. Les variables de configuration essentielles

La configuration Galera se place généralement dans un fichier dédié (par exemple `/etc/mysql/mariadb.conf.d/60-galera.cnf`), sous une section `[galera]`. Voici les variables incontournables :

| Variable | Rôle |
|----------|------|
| `wsrep_on` | Active la réplication wsrep (`ON`). |
| `wsrep_provider` | Chemin de la bibliothèque Galera (`…/libgalera_smm.so`). |
| `wsrep_cluster_name` | Nom **logique** du cluster — **identique** sur tous les nœuds. |
| `wsrep_cluster_address` | Liste des membres : `gcomm://hôte1,hôte2,hôte3`. |
| `wsrep_node_name` | Nom **propre** au nœud. |
| `wsrep_node_address` | Adresse IP **propre** au nœud. |
| `wsrep_sst_method` | Méthode de transfert d'état (voir 14.2.4) ; `mariabackup` recommandé. |
| `wsrep_sst_auth` | Identifiants utilisés par le SST (`utilisateur:motdepasse`). |
| `wsrep_slave_threads` | Nombre de threads d'application parallèle (voir 14.2.1). |
| `binlog_format` | **Doit** valoir `ROW`. |
| `default_storage_engine` | `InnoDB`. |
| `innodb_autoinc_lock_mode` | **Doit** valoir `2` (mode entrelacé, voir §4). |

---

## 4. Gestion de l'auto-incrément en multi-maître

Un piège propre au multi-maître : si deux nœuds insèrent **en même temps** dans une table à clé auto-incrémentée, ils risquent de générer la **même valeur** de clé primaire, provoquant un conflit. Galera résout ce problème automatiquement.

Avec **`wsrep_auto_increment_control = ON`** (la valeur par défaut), Galera ajuste sur chaque nœud :

- **`auto_increment_increment`** = nombre de nœuds du cluster (le pas entre deux valeurs) ;
- **`auto_increment_offset`** = position du nœud (le décalage de départ).

Sur un cluster à 3 nœuds, les valeurs générées se répartissent ainsi :

```
Nœud 1 (offset 1) →  1, 4, 7, 10, …
Nœud 2 (offset 2) →  2, 5, 8, 11, …
Nœud 3 (offset 3) →  3, 6, 9, 12, …
```

Chaque nœud produit donc des séquences **disjointes** : aucune collision possible, même lors d'insertions concurrentes. C'est aussi la raison pour laquelle **`innodb_autoinc_lock_mode = 2`** (mode entrelacé) est requis.

> ℹ️ Effet de bord à connaître : les identifiants auto-incrémentés ne sont **pas strictement consécutifs** à l'échelle du cluster (des « trous » apparaissent). C'est normal et sans incidence.

---

## 5. Un fichier de configuration type

```ini
[galera]
# --- Activation de la réplication wsrep ---
wsrep_on                 = ON
wsrep_provider           = /usr/lib/galera/libgalera_smm.so

# --- Identité du cluster (IDENTIQUE sur tous les nœuds) ---
wsrep_cluster_name       = "production_cluster"
wsrep_cluster_address    = "gcomm://10.0.0.11,10.0.0.12,10.0.0.13"

# --- Identité du nœud (PROPRE à chaque nœud) ---
wsrep_node_name          = "node1"
wsrep_node_address       = "10.0.0.11"

# --- Transfert d'état (voir 14.2.4) ---
wsrep_sst_method         = mariabackup
wsrep_sst_auth           = "sstuser:mot_de_passe_solide"

# --- Application parallèle des write-sets (voir 14.2.1) ---
wsrep_slave_threads      = 4

# --- Réglages InnoDB requis par Galera ---
binlog_format            = ROW
default_storage_engine   = InnoDB
innodb_autoinc_lock_mode = 2

# --- Accepter les connexions réseau ---
bind-address             = 0.0.0.0
```

Seules les lignes `wsrep_node_name` et `wsrep_node_address` changent d'un nœud à l'autre ; le reste est commun.

## 6. Amorçage du cluster (*bootstrap*)

C'est l'étape la plus délicate, et la source la plus fréquente d'erreurs. Un cluster Galera ne peut pas « démarrer tout seul » : il faut désigner **un premier nœud** qui **crée** le cluster et constitue la **composante primaire** initiale (voir 14.3). Les autres nœuds **rejoignent** ensuite ce cluster existant.

**Séquence de déploiement initial :**

```
1. Nœud 1 : amorçage (bootstrap)  ──►  crée le cluster (composante primaire)
                                          │
2. Nœud 2 : démarrage normal      ──►  rejoint  →  transfert d'état (SST/IST)
                                          │
3. Nœud 3 : démarrage normal      ──►  rejoint  →  transfert d'état (SST/IST)
```

Sur un système systemd, l'amorçage du premier nœud se fait avec le script fourni :

```bash
# Sur le PREMIER nœud uniquement
galera_new_cluster
```

(En interne, cela revient à démarrer le serveur avec l'option `--wsrep-new-cluster`, c'est-à-dire avec une adresse `gcomm://` vide.) Les nœuds suivants démarrent ensuite **normalement** :

```bash
# Sur chaque AUTRE nœud
systemctl start mariadb
```

> ⚠️ **Règle d'or : on n'amorce qu'UNE seule fois, sur UN seul nœud, à la création du cluster.** Amorcer un nœud censé rejoindre un cluster existant crée un **second cluster indépendant** — c'est-à-dire un **split-brain** (voir 14.3). Pour un nœud qui rejoint, on utilise **toujours** un démarrage normal.

---

## 7. `grastate.dat` et `safe_to_bootstrap`

Chaque nœud conserve dans son répertoire de données un fichier **`grastate.dat`** qui mémorise notamment :

- l'**UUID** de l'état du cluster ;
- le **dernier seqno** validé (la position de progression du nœud) ;
- un indicateur **`safe_to_bootstrap`** (`0` ou `1`).

Ce fichier devient crucial lors d'un **redémarrage complet du cluster** (tous les nœuds arrêtés). La règle est d'**amorcer depuis le nœud le plus avancé** — celui qui a le **seqno le plus élevé**, donc les données les plus récentes. Lors d'un arrêt propre, Galera positionne `safe_to_bootstrap: 1` sur le **dernier nœud à s'être arrêté**, qui est précisément celui par lequel ré-amorcer.

> Si un nœud a subi un **arrêt brutal**, son seqno peut être indéterminé (`-1`) ; on le récupère alors via une exécution en mode récupération (`--wsrep-recover` / le script `galera_recovery`) avant de décider du nœud d'amorçage. Les procédures de reprise complètes sont traitées en 14.8.

---

## 8. Vérifier l'état du cluster

Une fois le cluster démarré, ces variables d'état confirment qu'il est sain :

```sql
SHOW STATUS LIKE 'wsrep_%';
```

Les indicateurs clés à contrôler :

| Variable d'état | Valeur attendue |
|-----------------|-----------------|
| `wsrep_cluster_size` | Le **nombre de nœuds** attendu (ex. `3`). |
| `wsrep_cluster_status` | **`Primary`** (le cluster a le quorum). |
| `wsrep_connected` | **`ON`** (le nœud est connecté au groupe). |
| `wsrep_ready` | **`ON`** (le nœud accepte les requêtes). |
| `wsrep_local_state_comment` | **`Synced`** (le nœud est pleinement opérationnel). |

Si `wsrep_cluster_size` est inférieur au nombre attendu, un nœud n'a pas rejoint (vérifier les ports et les logs). Si `wsrep_cluster_status` n'est pas `Primary`, le nœud a perdu le quorum (voir 14.3).

---

## 9. Pièges de déploiement courants

- **Amorcer plusieurs nœuds** → plusieurs clusters distincts (split-brain).
- **Pare-feu** bloquant l'un des ports 4567 / 4568 / 4444.
- **`wsrep_cluster_name` différent** d'un nœud à l'autre → les nœuds refusent de se joindre.
- **`innodb_autoinc_lock_mode` absent ou ≠ 2**.
- **Tables sans clé primaire** (voir 14.2.2).
- **SELinux / AppArmor** empêchant le processus SST d'écrire ou de se connecter.
- **Versions MariaDB hétérogènes** entre nœuds.

---

## 10. Topologie et arbitre (garbd)

Un cluster sain repose sur un **nombre impair de nœuds** pour que le quorum soit toujours tranchable. Lorsqu'on ne dispose que de **deux nœuds de données** (ou d'un nombre pair), on ajoute un **arbitre Galera (`garbd`)** : un processus léger qui **participe au vote** sans stocker de données ni recevoir le trafic SQL. Il rétablit l'imparité nécessaire au quorum à moindre coût. Le rôle de l'arbitre dans la prévention du split-brain est détaillé en 14.3.

---

## À retenir

- Un déploiement Galera demande un **nombre impair de nœuds** (3 minimum), une **version MariaDB homogène** et l'installation du paquet **`mariadb-server-galera`** (séparé en 12.3, voir 14.2.5).
- Quatre **ports** doivent être ouverts entre nœuds : **3306** (SQL), **4567** (réplication), **4568** (IST), **4444** (SST).
- Les variables clés sont l'**identité du cluster** (`wsrep_cluster_name`, `wsrep_cluster_address`), l'**identité du nœud** (`wsrep_node_name`, `wsrep_node_address`), la **méthode SST** et les **réglages InnoDB requis** (`binlog_format=ROW`, `innodb_autoinc_lock_mode=2`).
- En multi-maître, **`wsrep_auto_increment_control`** répartit automatiquement les valeurs auto-incrémentées pour éviter les collisions de clés.
- L'**amorçage** crée le cluster sur **un seul nœud** (`galera_new_cluster`) ; les autres **rejoignent** par un démarrage normal. **Ne jamais amorcer deux fois** sous peine de split-brain.
- Pour un **redémarrage complet**, on amorce depuis le nœud au **seqno le plus élevé** (`safe_to_bootstrap: 1` dans `grastate.dat`).
- L'état se vérifie via `wsrep_cluster_size`, `wsrep_cluster_status` (= `Primary`) et `wsrep_local_state_comment` (= `Synced`).

La sous-section suivante détaille la mécanique par laquelle un nœud se synchronise en rejoignant le cluster : les **transferts d'état (SST, IST)**.
