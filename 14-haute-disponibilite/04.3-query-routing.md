# 14.4.3 — Query Routing

> **Chapitre 14 — Haute Disponibilité** · Section 14.4.3  
> Version de référence : **MariaDB 12.3 LTS**

Au-delà de la répartition par connexion (14.4.1) et du partage lecture/écriture (14.4.2), MaxScale offre un **contrôle plus fin du routage des requêtes** (*query routing*) : forcer explicitement une destination, router selon un **motif**, selon le **schéma** visé (sharding), ou vers le backend le plus performant. Cette sous-section présente ces mécanismes et la façon de **reprendre la main** sur les décisions automatiques. Le **filtrage/blocage** des requêtes est traité séparément en 14.4.4.

---

## 1. Le routage, du général au fin

Les deux routeurs vus précédemment couvrent la majorité des besoins :

- **`readconnroute`** — routage **par connexion** (14.4.1) ;
- **`readwritesplit`** — routage **par instruction**, écritures/lectures (14.4.2).

Le *query routing* désigne la **boîte à outils complémentaire** qui permet d'aller plus loin : indications explicites, routage par motif, par schéma, ou routeurs spécialisés. On y recourt lorsque la logique automatique ne suffit pas ou doit être **affinée**.

---

## 2. L'analyseur de requêtes (*query classifier*)

Toutes les décisions de routage fondées sur le **contenu** reposent sur l'**analyseur de requêtes** de MaxScale. Pour chaque instruction, il **parse** le SQL afin de déterminer :

- s'il s'agit d'une **lecture** ou d'une **écriture** ;
- si elle utilise des **fonctions**, des **variables**, des **verrous** ;
- quelles **tables** et quel **schéma** sont visés.

C'est cette analyse qui alimente, par exemple, les décisions de `readwritesplit` (14.4.2). L'analyseur est **modulaire**.

---

## 3. Forcer le routage : les indications (*hints*)

Lorsqu'on **sait mieux** que l'analyseur où une requête doit aller, on peut le **forcer** au moyen d'**indications** (*hints*) : des **commentaires spéciaux** insérés dans le SQL, interprétés par MaxScale.

Les formes principales :

```sql
-- Forcer cette lecture vers le primaire (données fraîches garanties)
SELECT solde FROM comptes WHERE id = 42; -- maxscale route to master
```

```sql
-- Forcer vers un serveur nommé précis
SELECT ... ; -- maxscale route to server dedie_reporting
```

Les destinations usuelles sont `route to master`, `route to slave` et `route to server <nom>`. Les indications peuvent être écrites sous différentes formes de commentaires (`--`, `#`, `/* … */`).

> ⚙️ Pour que ces indications soient prises en compte, le **`hintfilter`** doit figurer dans la chaîne de filtres du service. MaxScale propose aussi des **indications nommées et réutilisables** (définies une fois, appliquées à plusieurs requêtes) ; on se réfère à la documentation pour leur syntaxe exacte.

> ⚠️ **À manier délibérément.** Une indication **contourne** la logique de sûreté de `readwritesplit`. Forcer une écriture ou une lecture sensible vers le mauvais serveur peut introduire des incohérences. On réserve les *hints* aux cas **maîtrisés** (par exemple forcer une lecture critique vers le primaire pour garantir sa fraîcheur).

---

## 4. Routage par motif vers des serveurs nommés (`namedserverfilter`)

Le filtre **`namedserverfilter`** route les requêtes **correspondant à une expression régulière** vers des **serveurs nommés** précis. On définit des paires *motif → serveur cible*.

Cas d'usage typiques :

- diriger toutes les requêtes touchant une certaine table ou un certain motif vers un **serveur dédié** ;
- isoler des requêtes lourdes de *reporting* sur un nœud spécifique pour ne pas perturber le trafic transactionnel.

C'est un routage **basé sur des règles déclaratives**, complémentaire de l'analyse automatique.

---

## 5. Routage par schéma : le sharding (`schemarouter`)

Le routeur **`schemarouter`** prend ses décisions selon le **schéma (base de données)** auquel accède la requête. Il permet de mettre en place un **sharding horizontal** : différentes bases résident sur **différents serveurs**, et `schemarouter` dirige chaque requête vers le serveur **détenant** le schéma concerné, tout en présentant aux applications un **point d'accès unifié**.

- ✅ **Intérêt** : répartir les données sur plusieurs serveurs (distribution horizontale) derrière une seule connexion.
- ⚠️ **Limite** : les **requêtes inter-shards** (jointures entre bases situées sur des serveurs différents) sont **restreintes**.

Ce mécanisme se rattache aux thèmes du **sharding** (15.11) et des **architectures distribuées** (chapitre 20).

---

## 6. Routeurs spécialisés (`smartrouter`)

MaxScale propose des routeurs **spécialisés** pour des besoins particuliers. Le **`smartrouter`** dirige une requête vers le backend **le plus à même de l'exécuter rapidement**, selon le **type** de requête. Il est typiquement utilisé dans une architecture **HTAP** mêlant :

- un moteur **transactionnel** (InnoDB) pour l'OLTP ;
- un moteur **analytique** (ColumnStore, voir 7.5 et 20.3) pour l'OLAP.

Les requêtes transactionnelles sont alors routées vers l'un, les requêtes analytiques vers l'autre, de façon transparente.

---

## 7. Au-delà du routage : réécriture et filtrage

Le routage s'accompagne souvent d'autres traitements dans le **pipeline de filtres** :

- la **réécriture de requêtes** (filtres à base d'expressions régulières) peut **transformer** une requête avant qu'elle n'atteigne le serveur (sans relever à proprement parler du routage) ;
- le **pare-feu de base de données** peut **bloquer** des requêtes indésirables — c'est l'objet de la sous-section suivante, 14.4.4.

---

## 8. Bonnes pratiques

- **Privilégier le routage automatique** (`readwritesplit`) ; n'utiliser les **indications** que de manière **ciblée et maîtrisée**.
- Soigner l'**ordre des filtres** dans la chaîne (le pipeline s'applique séquentiellement).
- **Vérifier** les décisions de routage en pratique (CLI `maxctrl`, journaux) plutôt que de les supposer.
- Pour le **sharding** (`schemarouter`), tenir compte d'emblée de la limite sur les requêtes inter-shards dans la conception du modèle de données.

---

## À retenir

- Le **query routing** complète `readconnroute` (14.4.1) et `readwritesplit` (14.4.2) par un **contrôle fin** du routage.
- L'**analyseur de requêtes** parse chaque instruction (lecture/écriture, fonctions, schéma) et fonde les décisions de routage par contenu.
- Les **indications** (`-- maxscale route to master/slave/server <nom>`) permettent de **forcer** une destination ; elles nécessitent le **`hintfilter`** et **contournent** la logique de sûreté — à utiliser délibérément.
- Le **`namedserverfilter`** route par **motif (regex)** vers des serveurs nommés ; le **`schemarouter`** route par **schéma** et permet le **sharding** (avec une limite sur les requêtes inter-shards).
- Le **`smartrouter`** dirige vers le backend le plus performant selon le type de requête (architectures **HTAP** InnoDB + ColumnStore).
- Le routage cohabite avec la **réécriture** de requêtes et le **filtrage/blocage** (pare-feu, voir 14.4.4).

La sous-section suivante traite la dernière fonction cœur de MaxScale présentée ici : le **pare-feu de base de données** (*database firewall*).
